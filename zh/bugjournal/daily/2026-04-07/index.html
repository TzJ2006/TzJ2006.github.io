问题与解决方案#

关键问题#

1. Hugo 未安装，脚本在 Step 5 以 ‘hugo: command not found’（exit 127）终止#

解决方案: 用户通过系统包管理器安装 Hugo extended v0.154.5

关键洞察: set -e 会将任何命令失败直接转化为脚本终止，Hugo 缺失让整个 7 步流程无法到达关键构建阶段

2. themes/PaperMod/ 和 public/ 目录均不存在，Hugo 构建和部署无法运行#

解决方案: 用户克隆 PaperMod 主题和 tzj2006.github.io 部署仓库

关键洞察: Hugo 站点部署依赖三个独立组件（hugo 二进制、主题仓库、部署 git 仓库），任何一个缺失都会静默或终止式失败，应在文档中明确前置条件

一般问题#

3. static/videos 目录不存在，find 命令返回错误，set -e 导致 Step 4 意外终止#

解决方案: 在 find 前添加目录存在性检查（[ -d “$DIR” ]），目录不存在时跳过该步骤

关键洞察: 对可选/空目录的 find 操作在 set -e 脚本中必须做防护，否则正常的"目录为空"场景会触发非预期的脚本终止

4. HandBrakeCLI 内联回退分支的 PID 被重复加入 pids 数组，且压缩后未替换原文件（输出为 .hb.mp4）#

解决方案: 将 pids+=($!) 和 echo 语句移入各自 if/else 分支；在 HandBrakeCLI 命令后加 && mv 替换原文件

关键洞察: 并行后台任务的 PID 追踪逻辑需和 if/else 控制流边界精确对齐，把公共代码放在分支外部容易引入重复执行和静默数据丢失 Bug

5. git push 因 GitHub 认证未配置失败（could not read Username for ‘https://github.com’）#

解决方案: 用户配置了 SSH key 或 gh auth 后重新运行，脚本完整通过

关键洞察: HTTPS remote 在非交互 shell 中需要预配置凭据，部署仓库应改用 SSH remote 以避免环境差异

人类思路 vs AI 思路#

战略层面#

脚本验证策略#

角色 思路
人类 用户提出验证任务但未指定方式，期望知道脚本是否能正常工作
AI AI 选择静态分析优先：先读脚本和 CLAUDE.md 了解架构，再逐一检验外部依赖、目录结构和逻辑正确性，最后才实际运行脚本验证

差异分析: AI 的分层验证在运行前就发现了所有致命问题，避免了多次盲目运行；但 AI 无法自主解决 sudo 级别的环境问题，最终依赖用户手动操作

实现层面#

Hugo 安装方式选择#

角色 思路
人类 用户选择系统包管理器（ubuntu apt）安装，得到 v0.154.5
AI AI 建议 snap 或从 GitHub releases 下载预编译二进制（v0.147.4），但因 sudo 权限缺失无法执行

差异分析: 用户选择的系统包方式更简洁，版本也比 AI 建议的更新；AI 的建议偏向版本控制和跨发行版兼容，对本地开发环境略显过度

AI 局限性#

一般局限#

  • 无法执行需要 sudo 的命令（安装 Hugo、pngquant 等），只能生成脚本由用户手动执行，无法端到端完成环境搭建
  • 无法配置 GitHub 认证（SSH key 或 gh auth），阻断了首次完整部署运行，需要用户手动介入
  • 并行工具调用（parallel tool call）在本次会话中多次因竞态取消而失败，被迫改为串行调用，显著影响检查效率

今日收获#

核心收获#

  • Hugo 站点部署有三个必须同时到位的独立依赖（hugo 二进制、主题仓库、部署 git 仓库），任何一个缺失都会导致不同阶段的失败，应在项目文档中明确列出这些前置条件

实践收获#

  • bash set -e 脚本中对可选目录的 find 操作必须加目录存在性检查，否则正常的"目录尚未创建"场景会触发非预期的脚本终止
  • 并行后台任务（& + pids 追踪）的代码必须和 if/else 控制流边界精确对齐，公共代码放在分支外部容易引入重复 PID 追踪和静默的文件替换遗漏 Bug

问题与解决方案#

关键问题#

1. Linux下FloatBall窗口恒为400x400，Tauriset_size()完全无效#

**解决方案：**GTK自下而上的尺寸协商取max(window_requested, widget_natural_size)，WebKitGTK自然尺寸约为200x200导致窗口锁死。修复方法：通过with_webview()在GTK主线程调用widget.set_size_request()，移除set_resizable切换（with_webview是异步派发，set_resizable(false)在回调执行前入队会锁死旧尺寸），Linux永久保持可调整大小。

**关键洞察：**Tauri的set_size/set_min_size/set_max_size遵循GTK窗口级别的提示，GTK自下而上的协商会忽略这些提示以满足widget自然尺寸。with_webview返回时回调尚未执行，后续TauriAPI调用仍然是FIFO异步入队。

2. Linux收起时视觉跳变：两阶段序列暴露中间状态+WMclamp旧宽度导致坐标系统性偏移#

**解决方案：**两阶段序列：MacOS有250ms动画缓冲，Linux禁用过渡后中间状态刺眼，改为Linux路径先Rust原子缩窗口到56px，再设置CSSexpanded=false（此时100%=56px，变化不可见）。WMclamp问题：GTK重力在该WM下无效（反而引入系统性偏移），probe+correctivereapply产生双帧闪烁，根本解决方案是固定大尺寸窗口+内部CSS（待实施）。

**关键洞察：**禁用CSS过渡的同时必须同步调整JS/Rust时序逻辑；GTK重力并非所有WM都支持；事后补救本身就是可见闪烁的来源，根本解决方案是让WM一次性接受正确坐标。

3. Life-CopilotMCPServer三层断裂：有Calendartoken但agent仍报无法连接#

**解决方案：**三处同时断裂：FastMCP只被实例化未调用run()、.claude/mcp_servers.json缺失、butler指令（含MCPtools列表）从未传给ClaudeCLI子进程。新建__main__.py启动stdiotransport、新建mcp_servers.json、ClaudeAdapter添加–append-system-prompt，19个MCPtools全部恢复正常。

关键洞察：“代码写了”和“代码被调用”是两回事；工厂模式创建对象不等于服务启动；外部程序需要配置文件才能发现服务；App验证层和CLI工具上下文注入是两个独立层，token验证通过不代表CLI工具收到了正确的上下文。

4. LinuxScreenTimeTracker初始方案使用xdotool，不支持Wayland（用户机器运行GNOME50Wayland）#

**解决方案：**可行性探测确认Wayland环境，改用GNOMEIntrospectD-Bus（org.gnome.Shell.Introspect.GetWindows()）作为主路径，xprop作为XWayland回退，PID→cmdline作为最终降级，所有subprocess调用加5秒超时+proc.kill()。

**关键洞察：**GNOME41+提供标准IntrospectD-Bus接口，是Wayland下获取活跃窗口的正确方式；此问题由Critic子代理对抗审查主动发现，若遗漏会导致功能在用户机器上完全不可用。

5. AionUI无法通过URL以外方式（SSH）连接远程headless服务器上的ClaudeCodeCLI#

**解决方案：**AionUI远程Agent架构假定Agent暴露HTTP端口，CLI进程无HTTP端点，从根本上不兼容。建议使用tmux/screen管理多会话作为临时替代，并向AionUI团队提功能请求。

**关键洞察：**AionUI数据库已有remote_agents表（含allow_insecure字段），说明开发团队正在构建远程Agent能力，SSH直连可能是未来roadmap方向。

6. X11WM拒绝半出屏窗口定位，导致吸附和收起位置漂移#

**解决方案：**Mutter/GNOME等X11WM将无装饰窗口夹在可见区域内，拒绝x<0或x>screen_width - window_width的定位。Linux使用edge_offset=0，MacOS/Windows使用edge_offset=ball_size/2。

**关键洞察：**MacOS和Windows允许无装饰窗口部分超出屏幕边界，X11WM通常不允许，两个平台需要不同的边缘偏移量。

7. ClaudeCLIAdapter.is_available()缺少async导致Claude静默降级到Codex并超时#

**解决方案：**await普通方法不报编译错误，运行时TypeError被exceptException吞掉，Claude被判定不可用。添加async关键字后Claudeavailable:True恢复正常。

**关键洞察：**Python中await同步方法产生TypeError，若被宽泛exceptException静默捕获，会导致逻辑产生错误的False结果，表现为功能降级而非报错，极难排查。

8. 多轮对话中bot无法理解跨轮指代（DualVerifier参数接线断裂）#

**解决方案：**DualVerifier、SemanticRouter、KeywordScorer都已实现last_assistant_message和last_intent参数，但_process_input()调用时从未传递。修复接线即可，无需改任何分类逻辑。关键洞察：“功能已实现参数接口”并不等于“参数已被传递”；集成层是最常见的故障点，验证时应检查调用端而不仅仅是实现端。

9. Codex CLI 会话恢复存在漏洞，三个 CLI 会话机制差异太大，无法统一处理#

**解决方案：**放弃依赖 CLI 原生会话机制，在适配器层维护一个轻量级的历史缓冲区（最多存储20条记录），当 CLI 切换时，将历史记录作为提示前缀注入，从而实现跨 CLI 的透明对话历史传递。

**关键洞察：**在抽象层统一历史记录注入比适配每个 CLI 的会话机制更为可靠，会话机制的不一致性是多 CLI 架构的主要隐患。

10. lark-oapi WebSocket 客户端运行方式会导致阻塞，无法与 FastAPI asyncio 主循环共享#

**解决方案：**在独立的守护线程中启动飞书机器人，通过 asyncio.run_coroutine_threadsafe() 将消息回调连接到主事件循环。

**关键洞察：**lark-oapi v1.5.3中的Client.start()本质上是同步阻塞的，必须通过线程隔离来处理，而不是在asyncio任务中直接等待；这个问题是由Critic代理在对抗审查过程中发现的。

11. WebKitGTK内部的ScrolledWindow滚动条无法通过CSS进行抑制#

**解决方案：**WebKitGTK的滚动条属于gtk::ScrolledWindow层，无法通过CSS伪元素进行控制。在with_webview回调中遍历Widget的祖先链，对每个ScrolledWindow调用set_policy(Never,Never)。

**关键洞察：**WebKitGTK的UI Chrome属于GTK widget层，无法通过CSS选择器进行选择，必须在GTK代码层进行处理。

12. OpenReview API对于未经认证的访问会返回403错误，NeurIPS D&B Track的审稿意见获取失败#

**解决方案：**识别出两个叠加的问题：OpenReview最近收紧了匿名访问策略（需要账号认证）；NeurIPS D&B Track使用了独立的venueID而不是主会议的子路径，这两个问题的叠加导致了完全的失效。目前处于阻塞状态，需要使用OPENREVIEW_USERNAME/OPENREVIEW_PASSWORD环境变量。

**关键洞察：**在OpenReview的venueID体系中，卫星track使用了独立的venue而不是parent会议路径，需要单独确认ID；最近API策略已经全面收紧。

  • 在引入新机制之前，未验证平台支持：GTK gravity未确认WM是否支持（引入后反而产生了系统性偏移）；透明安全边距方案引入后立即破坏了拖拽和WM clamp两个问题，未在修改前预判副作用。
  • 对大型代码库全局依赖图感知不完整：ccplan Critic agent发现了AI规划时遗漏的4个隐藏依赖（BackgroundCoordinator、slash命令等）；并发/竞态修复容易遗漏边界情况（pointerUp在异步初始化前触发的场景需追问才发现）。

学习成果#

关键学习成果#

  • GTK窗口尺寸控制机制的完整图谱：GTK自下而上的协商取max(window_requested,widget_natural_size)；WebKitGTK自然尺寸约为200x200逻辑值，突破必须调用widget.set_size_request()，Tauri的set_size/set_min_size/set_max_size对WebKitGTK窗口无效；with_webview()是异步派发，后续的TauriAPI调用会在回调执行前入队，FIFO下可能锁死旧尺寸；WebKitGTK滚动条属于GTK widget层，必须用set_policy(Never,Never)在GTK代码层禁用。
  • Linux透明无边框Tauri/WebKitGTK浮窗的架构性约束：WM、GTK、WebView内容区三者尺寸/位置不会自动同步，这是平台约束而非bug。正确的架构应该是固定的大尺寸native窗口+内部纯CSS展开/收起，不依赖频繁resize外层窗口；GTK gravity并非所有WM都支持，用右边界坐标move会造成系统性偏移；probe + corrective reapply无法消除用户可见的过渡帧；在native坐标层做可见rect和窗口rect分离会使拖拽起点和鼠标漂移。
  • 运行时日志是底层系统调试的根本手段：当出现GTK、WM、OS等底层系统行为时，应优先通过日志/探针证实假设，而非从代码推断；跨平台动画策略需整体一致，禁用CSS过渡的同时必须同步调整对应的JS/Rust时序逻辑；X11WM将无装饰窗口强制夹在可见区域内（edge_offset需平台差异化），macOS/Windows允许半出屏。
  • NeurIPS D&B Track投稿策略：“领域纠偏者”（通过量化现有基准具体缺陷，如57%正确率，触发评估范式重构）比“新方法提出者”更容易被接受且影响力更持久；审稿人最重视多维度评估、公平基线对比、人工验证机制（κ≥0.68）和反直觉负面发现；反事实实体替换可消除词汇重叠偏差；符号化验证闭环（SPARQL/SQL执行对比+N次自动修复）可将数据正确率从57%提升至96%+；数据清洗可重构为无监督异常检测（IsolationForest+Z分数跨语言标准化）。
  • 激进简化（更换引擎）往往比渐进修补更有效：ThinAdapter+MCPServer+Skills三层模式替代专有路由逻辑，获得跨CLI通用性；大型重构应在编码前进行对抗性审查（Criticsubagent），可提前发现隐藏依赖和架构风险，本日多项目中Criti阶段共发现Wayland兼容性、lark-oapi线程阻塞、ThoughtStore非单例、4个隐藏代码依赖等高价值问题，ROI极高。
  • “代码已实现参数接口”并不等于“参数已被传递”：集成层是最常见的故障点，验证时应检查调用端而不仅仅是实现端；FastMCP的“创建”和“serve”是两步，同时需要mcp_servers.json声明外部程序才能发现；Pythonawait同步方法的TypeError如果被exceptException吞掉会导致功能静默降级而非报错。
  • 现代Linux桌面（Ubuntu22.04+）已默认使用Wayland，跨平台窗口追踪方案必须以GNOMEIntrospectD-Bus为主路径，xdotool/xprop只能作为XWayland回退；后台采样任务不应全部依赖CronScheduler（最小tick60s），需要非整分钟周期的任务应用独立asyncio.Task循环；ECL文档将设计决策结构化持久化，是多阶段复杂功能开发的关键工程实践。
  • 多CLI系统设计最佳实践：不同的CLI工具（ClaudeCode/Codex/Gemini）各有约定的指令文件（CLAUDE.md/AGENTS.md/GEMINI.md），利用约定文件比运行时注入更简单可靠；Skills本质是Markdown提示词，跨平台迁移只需工具名称映射；App状态验证与CLI工具上下文注入是两个独立层，token验证通过不代表CLI工具收到了正确的上下文；统一historyringbuffer比适配各CLI会话机制更可靠。
  • AionUI远程Agent仅支持URL连接（CLI进程无HTTP端点，架构层面根本限制），但AionUI数据库已有remote_agents表，SSH直连可能是未来的roadmap；AI在快速迭代的产品领域不应以确定语气陈述技术参数（命令、模型规格），而应声明知识截止时间并建议用户验证。
  • lark-oapiWebSocket的正确架构：Client.start()是阻塞式的，必须用独立daemon线程+asyncio.run_coroutinethreadsafe()桥接；GoogleCalendarOAuthscope精细划分，验证token应用events.list(calendarId=‘primary’)而非calendarList.list()；日志级别是零成本可观测性杠杆，将provider错误从DEBUG改为WARNING立即显著改善运维体验。

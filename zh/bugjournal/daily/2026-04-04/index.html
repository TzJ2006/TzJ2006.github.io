问题与解决方案#

关键问题#

1. Linux 下 FloatBall 窗口恒为 400x400，Tauri set_size() 完全无效#

解决方案: GTK bottom-up 尺寸协商取 max(window_requested, widget_natural_size)，WebKitGTK natural size ≈200x200 导致窗口锁死。修复：通过 with_webview() 在 GTK 主线程调用 widget.set_size_request()，移除 set_resizable 切换（with_webview 是异步派发，set_resizable(false) 在回调执行前入队会锁死旧尺寸），Linux 永久保持 resizable。

关键洞察: Tauri 的 set_size/set_min_size/set_max_size 走 GTK window 级别 hint，GTK bottom-up 协商会无视这些 hint 去满足 widget natural size。with_webview() 返回时回调还未执行，后续 Tauri API 调用依然是 FIFO 异步入队。

2. Linux 收起视觉跳变：两阶段序列暴露中间态 + WM clamp 旧宽度导致坐标系统性偏移#

解决方案: 两阶段序列：macOS 有 250ms 动画缓冲，Linux 禁用过渡后中间态刺眼，改为 Linux 路径先 Rust 原子缩窗口到 56px，再设 CSS expanded=false（此时 100%=56px，变化不可见）。WM clamp 问题：GTK gravity 在该 WM 不生效（反而引入系统性偏移），probe + corrective reapply 产生双帧闪烁，根本解法为固定大尺寸窗口+内部 CSS（待实施）。

关键洞察: 禁用 CSS 过渡的同时必须同步调整 JS/Rust 时序逻辑；GTK gravity 并非所有 WM 支持；事后补救本身就是可见闪烁的来源，根本解法是让 WM 一次性接受正确坐标。

3. Life-Copilot MCP Server 三层断裂：有 Calendar token 但 agent 仍报无法连接#

解决方案: 三处同时断裂：FastMCP 只被实例化未调用 run()、.claude/mcp_servers.json 缺失、butler 指令（含 MCP tools 列表）从未传给 Claude CLI 子进程。新建 main.py 启动 stdio transport、新建 mcp_servers.json、ClaudeAdapter 添加 –append-system-prompt，19 个 MCP tools 全部恢复正常。

关键洞察: “代码写了"和"代码被调用"是两回事；factory 模式创建对象不等于服务启动；外部程序需要配置文件才能发现服务；App 验证层和 CLI 工具上下文注入是两个独立层，token 验证通过不代表 CLI 工具收到了正确的上下文。

4. Linux Screen Time Tracker 初始方案使用 xdotool，不支持 Wayland（用户机器运行 GNOME 50 Wayland）#

解决方案: 可行性探测确认 Wayland 环境，改用 GNOME Introspect D-Bus（org.gnome.Shell.Introspect.GetWindows()）作为主路径，xprop 作为 XWayland 回退，PID→cmdline 作为最终降级，所有 subprocess 调用加 5 秒超时 + proc.kill()。

关键洞察: GNOME 41+ 提供标准 Introspect D-Bus 接口，是 Wayland 下获取活跃窗口的正确方式；此问题由 Critic 子代理对抗审查主动发现，若遗漏会导致功能在用户机器上完全不可用。

5. AionUI 无法通过 URL 以外方式（SSH）连接远程 headless 服务器上的 Claude Code CLI#

解决方案: AionUI 远程 Agent 架构假定 Agent 暴露 HTTP 端口，CLI 进程无 HTTP 端点，根本上不兼容。建议 tmux/screen 管理多会话作为临时替代，并向 AionUI 团队提 feature request。

关键洞察: AionUI 数据库已有 remote_agents 表（含 allow_insecure 字段），说明开发团队正在构建远程 Agent 能力，SSH 直连可能是未来 roadmap 方向。

6. X11 WM 拒绝半出屏窗口定位，导致吸附和收起位置漂移#

解决方案: Mutter/GNOME 等 X11 WM 将无装饰窗口夹在可见区域内，拒绝 x < 0 或 x > screen_width - window_width 的定位。Linux 使用 edge_offset=0，macOS/Windows 使用 edge_offset=ball_size/2。

关键洞察: macOS 和 Windows 允许无装饰窗口部分超出屏幕边界，X11 WM 通常不允许，两个平台需要不同的边缘偏移量。

7. ClaudeCLIAdapter.is_available() 缺少 async 导致 Claude 静默降级到 Codex 并超时#

解决方案: await 普通方法不报编译错误，运行时 TypeError 被 except Exception 吞掉，Claude 被判定不可用。添加 async 关键字后 Claude available: True 恢复正常。

关键洞察: Python 中 await 同步方法产生 TypeError，若被宽泛 except Exception 静默捕获，会导致逻辑产生错误的 False 结果，表现为功能降级而非报错，极难排查。

8. 多轮对话中 bot 无法理解跨轮指代（DualVerifier 参数接线断裂）#

解决方案: DualVerifier、SemanticRouter、KeywordScorer 均已实现 last_assistant_message 和 last_intent 参数，但 _process_input() 调用时从未传递。修复接线即可，无需改任何分类逻辑。

关键洞察: “功能已实现参数接口"不等于"参数已被传递”；integration layer 是最常见的接线断裂点，验证时要检查调用侧而不只是实现侧。

9. Codex CLI session resume 有 bug，三个 CLI session 机制差异过大无法统一#

解决方案: 放弃依赖 CLI 原生 session，在 Adapter 层维护轻量历史 ring buffer（last-20 条），CLI 切换时将历史注入为 prompt 前缀，实现透明的跨 CLI 对话历史传递。

关键洞察: 统一抽象层（history injection）比适配各 CLI 的 session 机制更可靠，session 机制的不一致性是多 CLI 架构的主要隐患。

10. lark-oapi WebSocket client 阻塞式运行，无法与 FastAPI asyncio 主循环共用#

解决方案: 在独立 daemon 线程中启动飞书 bot，消息回调通过 asyncio.run_coroutine_threadsafe() 桥接到主 event loop。

关键洞察: lark-oapi v1.5.3 的 Client.start() 本质是同步阻塞的，必须线程隔离而非在 asyncio 任务中直接 await；此问题由 Critic agent 对抗审查预先发现。

11. WebKitGTK 内部 ScrolledWindow 滚动条无法用 CSS 抑制#

解决方案: WebKitGTK 滚动条属于 gtk::ScrolledWindow 层而非 WebKit 渲染树，CSS 伪元素无法触及。在 with_webview 回调中遍历 widget 祖先链，对每个 ScrolledWindow 调用 set_policy(Never, Never)。

关键洞察: WebKitGTK 的 UI chrome 属于 GTK widget 层，CSS 选择器无法选中，必须在 GTK 代码层处理。

12. OpenReview API 对未认证访问全面返回 403，NeurIPS D&B Track 审稿意见获取失败#

解决方案: 识别两个叠加问题：OpenReview 近期收紧匿名访问策略（需账号认证）；NeurIPS D&B Track 使用独立 venue ID 而非主会议子路径，两者叠加导致完全失效。当前阻塞，需 OPENREVIEW_USERNAME/OPENREVIEW_PASSWORD 环境变量。

关键洞察: OpenReview venue ID 体系中，卫星 track 使用独立 venue 而非 parent conference 路径，需单独确认 ID；近期 API 策略已全面收紧。

13. Research pipeline 重复执行导致 overview.md 重复写入，缓存命中旧结果导致依赖修复后仍使用空结果#

AI 局限性#

重要局限#

  • 缺乏运行时实证时，错误假设底层 API（Tauri set_size、GTK gravity、Google Calendar calendarList scope）按预期工作，需要日志证据或用户报错才能纠正方向；多次修复几何算法而未质疑底层 OS 机制是否生效。
  • 多轮修复失败后未能主动识别"继续修补不能解决架构性问题"，每次看到新日志就继续尝试下一个技术方案，需要用户明确要求停止才会进行根因总结。

  • 对大型代码库全局依赖图感知不完整：ccplan Critic agent 发现了 AI 规划时遗漏的 4 个隐藏依赖（BackgroundCoordinator、slash 命令等）；并发/竞态修复容易遗漏边界情况（pointerUp 在异步初始化前触发的场景需追问才发现）。
  • 在快速迭代的产品领域提供过时或编造信息：编造了 AionUI 的 /model 和 /effort 斜杠命令（用户验证后失败），以确定口吻陈述 Claude 200k context window（实际已 1M），不应以确定口吻陈述快速更新的技术参数。
  • 对工具链外部 API 限制（OpenReview 403、lark-oapi 线程阻塞、Codex session bug）无法在规划阶段预见，需实际运行或 Critic 对抗审查才能发现；无法主动提供 OpenReview 认证墙的工作替代方案。

一般局限#

  • 调查路径分散：面对多层断裂问题（日历无法连接、Discord 上下文异常）时从各个独立症状分别入手，未能快速从架构层面整体定位根因，需要多轮工具调用才汇聚到结论。

今日收获#

核心收获#

  • GTK 窗口尺寸控制机制全貌：GTK bottom-up 协商取 max(window_requested, widget_natural_size)；WebKitGTK natural size ≈200x200 logical，突破必须调用 widget.set_size_request()，Tauri 的 set_size/set_min_size/set_max_size 对 WebKitGTK 窗口无效；with_webview() 是异步派发，后续 Tauri API 调用会在回调执行前入队，FIFO 下可能锁死旧尺寸；WebKitGTK 滚动条属于 GTK widget 层，必须用 set_policy(Never, Never) 在 GTK 代码层禁用。
  • Linux 透明无边框 Tauri/WebKitGTK 浮窗的架构性约束：WM、GTK、WebView 内容区三者尺寸/位置不会自动同步，这是平台约束而非 bug。正确架构应为固定大尺寸 native window + 内部纯 CSS 展开/收起，不依赖频繁 resize 外层窗口；GTK gravity 并非所有 WM 支持，用右边界坐标 move 会造成系统性偏移；probe + corrective reapply 无法消除用户可见的过渡帧；在 native 坐标层做可见 rect 和窗口 rect 分离会使拖拽起点和鼠标漂移。
  • 运行时日志是底层系统调试的根本手段：当出现 GTK、WM、OS 等底层系统行为时，应优先通过日志/探针实证确认假设，而非从代码推断；跨平台动画策略需整体一致，禁用 CSS 过渡的同时必须同步调整对应的 JS/Rust 时序逻辑；X11 WM 将无装饰窗口强制夹在可见区域内（edge_offset 需平台差异化），macOS/Windows 允许半出屏。
  • NeurIPS D&B Track 投稿策略：“领域纠偏者”（通过量化现有基准具体缺陷，如 57% 正确率，触发评估范式重构）比"新方法提出者"更容易被接受且影响力更持久；审稿人最重视多维度评估、公平基线对比、人工验证机制（κ≥0.68）和反直觉负面发现；反事实实体替换可消除词汇重叠偏差；符号化验证闭环（SPARQL/SQL 执行对比 + N 次自动修复）可将数据正确率从 57% 提升至 96%+；数据清洗可重构为无监督异常检测（Isolation Forest + Z-score 跨语言标准化）。
  • 激进简化（换掉引擎）往往比渐进修补更有效：Thin Adapter + MCP Server + Skills 三层模式替代专有路由逻辑，获得跨 CLI 通用性；大型重构应在编码前进行对抗性审查（Critic subagent），可提前发现隐藏依赖和架构风险，本日多项目中 Critic 阶段共发现 Wayland 兼容性、lark-oapi 线程阻塞、ThoughtStore 非单例、4 个隐藏代码依赖等高价值问题，ROI 极高。
  • “代码已实现参数接口"不等于"参数已被传递”：integration layer 是最常见的接线断裂点，验证时要检查调用侧而非只看实现侧；FastMCP 的"创建"和"serve"是两步，同时需要 mcp_servers.json 声明外部程序才能发现；Python await 同步方法的 TypeError 若被 except Exception 吞掉会导致功能静默降级而非报错。
  • 现代 Linux 桌面（Ubuntu 22.04+）已默认使用 Wayland，跨平台窗口追踪方案必须以 GNOME Introspect D-Bus 为主路径，xdotool/xprop 只能作为 XWayland 回退；后台采样任务不应全部依赖 CronScheduler（最小 tick 60s），需要非整分钟周期的任务应用独立 asyncio.Task 循环；ECL 文档将设计决策结构化持久化，是多阶段复杂功能开发的关键工程实践。
  • 多 CLI 系统设计最佳实践：不同 CLI 工具（Claude Code/Codex/Gemini）各有约定指令文件（CLAUDE.md/AGENTS.md/GEMINI.md），利用约定文件比运行时注入更简单可靠；Skills 本质是 Markdown 提示词，跨平台迁移只需工具名称映射；App 状态验证与 CLI 工具上下文注入是两个独立层，token 验证通过不代表 CLI 工具收到了正确的上下文；统一 history ring buffer 比适配各 CLI session 机制更可靠。
  • AionUI 远程 Agent 仅支持 URL 连接（CLI 进程无 HTTP 端点，架构层面根本限制），但 AionUI 数据库已有 remote_agents 表，SSH 直连可能是未来 roadmap；AI 在快速迭代的产品领域不应以确定口吻陈述技术参数（命令、模型规格），应声明知识截止时间并建议用户验证。
  • lark-oapi WebSocket 正确架构：Client.start() 是阻塞式的，必须用独立 daemon 线程 + asyncio.run_coroutine_threadsafe() 桥接；Google Calendar OAuth scope 精细划分，验证 token 应用 events.list(calendarId=‘primary’) 而非 calendarList.list()；日志级别是零成本可观测性杠杆，将 provider 错误从 DEBUG 改为 WARNING 立即显著改善运维体验。

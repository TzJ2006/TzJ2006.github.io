问题和解决方案#

关键问题#

1. Openpi-moe训练在norm_stats.json缺少prevActions键时不会出错，导致轨迹VAE中的尺度不匹配（未归一化的历史动作与归一化动作合并）#

**解决方案：**根本原因是transforms.py的apply_tree()默认设置为strict=False，默默地跳过缺失键的归一化处理。修改了compute_norm_stats.py来动态检测批处理中的prevActions并计算其统计信息，解决了这个问题。

**关键洞察：**数据预处理过程中的静默失败是危险的。apply_tree(strict=False)的松散行为允许在训练过程中出现尺度不匹配的问题。始终要验证所有预期键是否具有归一化的统计信息，特别是对于合并的特征。

2. services/init.py中的 eager导入会导致循环依赖（services ↔ core.scheduler），使得无法导入任何服务模块#

**解决方案：**将所有导入改为懒加载，只保留__all__列表和注释，要求用户直接导入（例如，from srcServices.calendar_service import CalendarService），避免在包初始化过程中产生循环依赖。

**关键洞察：**Python包__init__.py不需要立即提供导入语句；对于具有循环依赖的包，结合文档注释的懒加载是一个更好的选择；在包级别暴露接口并不等于必须急切地加载这些模块。

3. 16个危险的异常错误：忽略所有错误会导致状态不完整，但系统仍然可以正常工作，无法在调试过程中定位错误的来源解决方案： 将代码替换为 except Exception as e: logger.error/warning(f"Description: {e}")，同时保留错误信息以供诊断，区分错误（关键功能）和警告（次要功能）#

关键洞察： 切勿使用空处理程序来处理异常；如果必须忽略错误，至少记录错误信息；在状态跟踪层（situation_monitor/life_context）中隐藏失败会导致“薛定谔态”——无法确定是真正的零处理还是加载失败

4. 如何在不产生顺序瓶颈的情况下高效实施大规模31项升级计划？#

解决方案： 使用并行后台代理执行模式：同时启动4个独立代理来执行context_detector、checkin_service、learning_service和energy_projection增强功能，同时直接修改另外8个文件。所有代理均成功完成，没有发生冲突

关键洞察： 基于波的并行执行可以在正确隔离依赖关系的情况下显著加速大规模重构。关键在于识别可以真正并发运行的独立工作单元

5. AI反复尝试进入规划模式，但用户更喜欢直接执行#

解决方案： 用户拒绝了ExitPlanMode工具调用，直接将规划内容提供给AI执行。AI读取之前研究代理在规划模式下的结果，然后直接写入文档中，无需重新规划

关键洞察： 用户更倾向于快速迭代而非长期规划；AI应自适应减少规划模式的触发频率，或在多次用户拒绝后了解这一偏好

6. CalendarPro有6个完全未经测试的代码文件，12个新的修改方法，而BackgroundCoordinator虽然实现了但从未启动过#

解决方案： 创建了系统的测试基础设施：使用conftest.py共享测试夹具，根据现有模式创建20个新的测试文件（pytest、AsyncMock、patch装饰器），修复on_ready()函数以实际调用coordinator.start_all()，添加9个缺失的Discord意图处理程序

关键洞察： 大规模功能通常伴随着不完整的集成。后台服务需要明确的启动调用，而不仅仅是类定义。系统的测试需要共享测试夹具和所有测试文件的一致性

7. 测试夹具中的补丁路径不正确（使用修补点而不是定义点），导致懒加载后出现模拟失败，多个测试报告了AttributeError错误#

解决方案： 所有用于修补的单例工厂函数（get_settings/get_calendar_service）必须位于源模块中（src.config/src.services.calendar_service），而不是调用点处（例如srcservices.gap_analyzer），因为懒加载会延迟名称绑定

关键洞察： 使用unittest.mockpatch的黄金规则是——在查找位置进行修补，而不是在定义位置；懒加载意味着“查找”位置总是在源模块中，而不是在导入位置创建新的绑定

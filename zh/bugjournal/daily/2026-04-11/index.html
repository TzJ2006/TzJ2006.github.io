Problems & Solutions#

Critical Issues#

1. MimicGen增广验证全部失败（0/10 scenes），cubeA未被抬起（高度0.82桌面）#

Solution: 多轮诊断发现根本原因：subtask fragmentation（6个subtask=5个独立warp边界，误差累积）+无post-grasp buffer+gripper检测bug（找第一次而非持续闭合）+retract subtask未warp+路径解析bug。逐个修复：简化为2-subtask、添加15帧buffer、改进gripper检测、合并retract进re_acquire、修复路径解析

Key Insight: 原版MimicGen设计智慧：最小化subtask数量（2个）+ post-grasp buffer + 从home position开始。每个subtask边界都是误差源，应尽量减少

2. success_demos.hdf5无法作为MimicGen source——缺少env_interface_name和env_interface_type属性#

Solution: 添加ensure_env_interface_attrs() helper，在filter_valid_source、augment_success_demos、generate_success_from_error三处自动补全属性；补丁修复已有7个task的HDF5文件

Key Insight: MimicGen DataGenerator依赖特定datagen_info attributes识别环境接口，缺失这些属性导致get_env_interface_info_from_dataset失败；需在生成链每个输出点确保完整性

3. BetterSSH前端按钮无响应——WebSocket消息被静默丢弃#

Solution: 追踪到index.ts中onMessage()占位符返回null。实例化SessionManager、加载SSH/BetterSSH配置、连接完整消息路由（get_sessions、create_session、destroy_session、send_input、resize）

Key Insight: 后端有SessionManager完整实现（500+ LOC）但从未实例化——骨架代码陷阱，接口存在但从未连接

4. Token使用数据在删除JSONL日志时消失#

Solution: 在{app_data_dir}/usage-archive/下实现仅追加的每小时聚合归档，使用时间边界分区（archive覆盖[0..frontier]，live source覆盖(frontier..now]）防止重复计数

Key Insight: 时间边界分区消除归档和实时数据重叠，确保零重复统计，存储开销最小（约1KB/天/提供商）

5. Claude Code在探索性项目中频繁要求权限确认，影响开发效率#

Solution: 将逐条枚举的80+条Bash命令权限简化为一个工具级通配符"Bash"，涵盖所有Bash命令

Key Insight: Claude Code支持工具级通配符（“Bash"匹配所有Bash命令），可用极简配置实现宽松权限，同时通过ask数组为危险操作保留确认机制

6. 对抗性审查发现WebSocket PTY写入无认证（ATK-007）#

Solution: 新增REQ-009：127.0.0.1绑定+256-bit会话token（启动时随机生成并打印），所有WS upgrade验证query string token

Key Insight: 安全问题在需求阶段修复成本最低，红蓝对抗应在Phase 8（架构确定后、编码前）强制执行

7. OpenReview API返回403 Forbidden阻止访问NeurIPS论文评审数据#

Solution: 诊断双层阻止：(1) OpenReview API现强制所有请求需认证token（禁用匿名访问）(2) 频繁探测触发nginx速率限制。使用用户OpenReview凭证（tongtongtot@163.com）通过openreview-py SDK登录，成功检索61条评审（7 Oral论文：29评审+7 AC决定；8 Spotlight论文：32评审）。请求间添加延迟避免速率限制

Key Insight: OpenReview的认证策略已改变——即使公开内容API访问也需登录token。/login端点保持可访问（返回400验证错误）而所有数据端点返回403，表明应用级认证强制执行。Web前端工作因Next.js RSC服务端渲染使用内部凭证

8. Terminal渲染质量差——字体模糊、滚动慢、中键粘贴失效#

Solution: 识别xterm.js是行业标准（VS Code/Replit使用），问题是缺少插件。添加WebGL渲染器（GPU加速）、Unicode11（CJK）、改进主题（Catppuccin Mocha）、更好字体栈（Cascadia Code回退链）

Key Insight: 仅xterm.js不足——生产终端需完整插件生态。WebGL提供10倍渲染提升和清晰字体。Unicode11防止CJK宽度错误计算

9. GLM5.1和DeepSeek-Reasoner显示$0.00定价，尽管已添加OpenRouter集成#

Solution: 向PricingCache结构添加version字段（CACHE_VERSION=2），使陈旧缓存（仅包含LiteLLM数据）在引入新定价源时自动失效

Key Insight: 缓存系统需显式版本控制处理模式演变——仅依赖TTL允许陈旧数据在代码更改后持续存在

10. Terminal切换会话丢弃所有终端内容——无回滚历史#

Solution: 改变终端生命周期：在useEffect([], [])中创建xterm实例一次，跨会话重用。切换时：term.clear()→订阅新会话→后端发送RingBuffer回填→term.write()

Key Insight: Terminal实例昂贵且保存回滚状态——切换会话时不应销毁。后端RingBuffer（2MB）提供重放缓冲区

General Issues#

11. tianhe集群git pull失败，报’GnuTLS recv error (-110): The TLS connection was non-properly terminated'#

Solution: 诊断代理配置不匹配：git全局配置指向172.16.31.200:3138，而环境变量（来自proxy_on）使用127.0.0.1:10087。使用git -c http.proxy=http://127.0.0.1:10087 -c https.proxy=http://127.0.0.1:10087 pull覆盖每命令代理设置，成功拉取

Key Insight: Git有两个独立代理配置层（全局配置vs环境变量）可能分歧。使用-c标志的每命令覆盖提供即时解决方案而不修改持久配置

Strategic Level#

Role Approach
Human 人类识别当前逻辑本质问题：对每个target scene重新error injection获得多样性，不符合原始MimicGen设计；应把一个error scene当固定基础场景，通过env.reset()物体随机化生成variants
AI AI最初只看到代码实现细节（_replay_with_mimicgen_targets），未从’增广策略’层面质疑设计；人类明确指出问题后，AI才理解应将原始MimicGen的reset逻辑搬到error scene上

Difference Analysis: 人类从系统设计层面思考’什么是合理增广方式’，AI倾向接受现有实现并在其框架内优化；这是战略vs战术层面差异

AI Limitations#

Critical Limitations#

  • 未主动提议技术调研：AI设计完成后准备直接实施，未主动建议’先调研现有解决方案’，需人类提醒。工程师应具备的’复用优于重写’意识缺失
  • 隐含需求挖掘不足：用户说’不愿意切换窗口’时，AI只关注远程SSH场景，未推理出本地CLI也需统一管理。需更强场景完整性推理
  • 需要6轮诊断才定位所有MimicGen bug：subtask fragmentation→retract未warp→gripper检测错误→插值步数不够→并行化路径bug→缓存清理。AI无法一次性预见所有问题
  • 初次访问OpenReview API失败，未意识到近期认证策略变化。所有30+次尝试（API v1/v2、不同端点、访客登录、各种headers）返回403，直到用户提供凭证。系统缺乏2026年OpenReview认证要求的最新知识
  • 未能在初始架构扫描时检测到SessionManager从未实例化——花费大量时间调试端口冲突和WebSocket问题后才发现根本原因是空onMessage处理程序
  • AI最初分析时未能主动质疑’重复error injection’的设计合理性，需人类明确指出’应固定error scene + reset randomization’才开始重构
  • 路径假设错误：Agent最初假设文件在recovery_collection_portable/下编辑，但用户已移出一层。所有编辑丢失，需在新路径重新应用
  • 未最初识别陈旧定价缓存会阻止OpenRouter数据加载——需通过litellm.rs缓存逻辑显式代码追踪才发现版本控制差距

General Limitations#

  • 触发OpenReview nginx速率限制，通过诊断阶段约30次连续请求导致临时IP封禁（152.3.43.42）。应更早实施指数退避或诊断认证要求
  • 子agent执行过程不透明：5个后台agent并行运行时，用户看不到中间进度（只有’等待完成通知’），缺乏实时可见性。大型任务进度感知是UX痛点

Learnings#

Key Learnings#

  • Superpowers技能链威力：brainstorming（需求分析）→ccplan（ECL驱动实施）→optimize（质量提升）形成完整软件生命周期。ECL文档作为跨会话持久化’设计契约’，使多轮对话、多个agent、甚至不同模型都能保持上下文一致性
  • 对抗性审查价值：Critic agent发现的9个问题中，2个CRITICAL（无认证PTY写入、SSH参数注入）和4个HIGH（token暴力破解、原型污染）如在生产环境暴露将造成安全灾难。需求阶段修复成本«代码阶段«生产阶段

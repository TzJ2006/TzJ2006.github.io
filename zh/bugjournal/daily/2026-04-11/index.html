问题与解决方案#

关键问题#

1. 所有MimicGen增强验证均失败（0/10场景），cubeA未解除限制（桌面高度0.82）#

解决方案：经过多轮调试发现根本原因：子任务碎片化（6个子任务=5个独立边界，累积错误）+没有后抓取缓冲区+夹持器检测漏洞（寻找连续闭合而不是首次阈值超过）+收回子任务未形成边界+路径解析漏洞。逐一修复问题：将子任务简化为2个，增加了15帧缓冲器，改进了夹持器检测，将收回功能合并到重新获取中，修复了路径解析问题

关键洞察：原始MimicGen设计理念：尽量减少子任务数量（2个）+后抓取缓冲区+从初始位置开始。每个子任务边界都是错误来源，因此应尽可能减少

2. success_demos.hdf5不能作为MimicGen源文件——缺少env_interface_name和env_interface_type属性#

解决方案：添加了ensure_env_interface attrs()辅助函数，自动在filter_valid_source、augment_success_demos、generate_success_from_error中完成属性设置；修复了7个任务的HDF5文件

关键洞察：MimicGen DataGenerator依赖于特定的datagen_info属性来识别环境接口。缺少这些属性会导致get_env_interface_info_from_dataset失败；必须在生成链的每个输出点确保完整性

3. BetterSSH前端按钮无响应——WebSocket消息被静默丢弃#

解决方案：追溯到index.ts中onMessage()占位符的null返回问题。实例化SessionManager，加载SSH/BetterSSH配置，连接到完整的消息路径（get_sessions、create_session、destroy_session、send_input、resize）

关键洞察：后端有完全实现的SessionManager（500多行代码），但从未实例化它——这是一个代码陷阱，接口存在但从未连接

4. 删除JSONL日志时Token使用数据消失#

解决方案：在{app_data_dir}/usage-archive/下实现每小时追加归档，使用时间边界分割（归档覆盖[0..前沿]，实时源覆盖（前沿..现在））以防止重复计数

关键洞察：时间边界分割消除了归档和实时数据之间的重叠，确保了零重复统计，同时减少了存储开销（约1KB/天/提供者）

5. Claude代码在探索性项目中频繁请求权限确认，影响开发效率#

解决方案：将80多项逐项枚举Bash命令权限简化为工具级通配符“Bash”，覆盖所有Bash命令关键洞察： Claude Code支持工具级通配符（“Bash"匹配所有Bash命令），允许在最小配置下实现宽松的权限管理。危险操作仍然需要通过ask数组进行确认机制

6. 对抗性审查发现WebSocket PTY写作没有进行认证验证 (ATK-007)#

解决方案： 增加了NEWREQ-009：绑定到127.0.0.1 + 256位会话令牌（随机生成并在启动时打印），所有WS升级都需要一个查询字符串令牌

**关键洞察：**在需求阶段，解决安全问题所需的成本最低。在第八阶段应实施红蓝对抗机制（在确定架构后，在开始编码之前）

7. OpenReview API返回403禁止访问，导致无法访问NeurIPS论文评审数据#

解决方案： 诊断出双层阻塞问题：(1) OpenReview API现在强制所有请求都需要认证令牌（匿名访问被禁用），(2)频繁探测触发了nginx的速率限制。使用用户OpenReview凭据(tongtongtot@163.com)通过openreview-py SDK登录，成功获取61篇评审记录（7篇口头论文：29篇评审 + 7项AC决策；8篇Spotlight论文：32篇评审）。在请求之间增加延迟以避免速率限制

关键洞察： OpenReview的认证策略发生了变化——即使是公开内容的API访问也需要登录令牌。/login端点仍然可访问（返回400验证错误），而所有数据端点都返回403错误，表明应用程序级认证被强制执行。由于Next.js RSC服务器端渲染，网页前端使用内部凭据进行工作

8. 终端渲染质量较差——字体模糊，滚动缓慢，中间键粘贴失败#

解决方案： 确定xterm.js是行业标准（被VS Code/Replit使用），问题在于缺少插件。增加了WebGL渲染器（GPU加速）、Unicode11（中文语言支持）、改进的主题设置（Catppuccin Mocha）、更好的字体集（Cascadia Code备用）

**关键洞察：**仅使用xterm.js是不够的——生产环境终端需要完整的插件生态系统。WebGL提供了10倍的渲染提升和清晰的字体。Unicode11防止了中文宽度错误

9. GLM5.1和DeepSeek-Reasoner显示的价格为$0.00，尽管已添加了OpenRouter集成#

解决方案： 在PricingCache结构中增加了版本字段（CACHE_VERSION=2），使得过时的缓存（仅包含LiteLLM数据）在新定价源引入时自动失效

**关键洞察：**缓存系统需要明确的版本控制来处理模式演变——仅依赖TTL允许旧数据在代码更改后仍然存在

10. 终端会话切换会丢弃所有终端内容——没有回滚历史记录#

解决方案： 改变了终端的生命周期：在useEffect([], [])中创建xterm实例一次，并在多个会话之间重复使用。在切换时：使用term.clear() →订阅新会话→后端发送RingBuffer回填→term.write()

**关键洞察：**终端实例成本较高，需要保存回滚状态——在切换会话时不应销毁它们。2MB的后端RingBuffer提供了重放缓冲区

一般问题#

11. Tianhe集群git pull失败，报告“GnuTLS recv错误（-110）：TLS连接终止不当”**解决方案：**代理配置不正确：全局Git设置指向172.16.31.200:3138，而环境变量（来自proxy_on）使用127.0.0.1:10087。使用git -c http.proxy=http://127.0.0.1:10087 -c httpsproxy=http://127.0.0.1:10087 pull命令可以覆盖每个命令的代理设置，成功拉取仓库。#

**关键洞察：**Git有两个独立的代理配置层（全局设置和环境变量），这两者可能会冲突。使用-c标志来覆盖每个命令的设置可以提供即时的解决方案，而无需修改持久化配置

人类与人工智能的方法论#

战略层面#

角色 方法
人类 人类识别当前逻辑中的基本问题：针对每个目标场景重复错误注入会导致缺乏多样性，这与原始的MimicGen设计不符。错误场景应被视为固定的基础场景，而变体应通过使用env.reset()随机化对象来生成
人工智能 最初，人工智能只关注代码实现的细节（），而不从增强策略的角度质疑设计。只有在人类指出问题后，人工智能才理解应该从原始MimicGen重置逻辑转移到错误场景中

**差异分析：**人类从系统设计的层面考虑“合理的增强方式”，而人工智能倾向于接受现有的实现并在其框架内进行优化。这代表了战略与战术层面的区别。

**差异分析：**人类表现出工程上的务实性（重用比重写更好），而人工智能在没有提示的情况下往往会陷入“从头开始重新创造”的习惯。这是人工智能的一个盲点。

终端用户体验质量的判断#

角色 方法
人类 基于直觉的直观评估：“它看起来根本不像一个终端，有些地方不对劲”——使用真实的终端进行即时模式识别
人工智能 技术检查清单：识别缺失的WebGL渲染器、Unicode11、合适的主题、尺寸调整同步——系统诊断根本原因

差异分析： 人类擅长评估整体用户体验，而人工智能则擅长识别技术上的缺陷。人类能够立即发现问题，而人工智能则提供诊断和解决方案。

人工智能的限制#

关键限制#

  • 没有主动建议研究：在人工智能的设计完成后，它直接投入使用，而没有建议“先研究现有的解决方案”。这需要人类的提醒。工程师应该意识到“重用比重写更好”。
  • 需求挖掘不足：当用户表示“不愿意切换窗口”时，人工智能仅关注远程SSH场景，未能推断出本地CLI也需要统一管理。需要强大的场景完整性推理能力。
  • 花了6轮诊断才找到所有MimicGen错误：子任务碎片化→撤回不会变形→抓取器检测错误→插值步骤不足→并行路径错误→缓存清理。人工智能无法一次性预测所有问题。
  • 第一次尝试时未能访问OpenReview API，没有意识到最近的认证政策变化。所有30多次尝试（API v1/v2、不同的端点、访客登录、各种头部信息）都返回了403错误，直到用户提供凭据。系统缺乏2026年OpenReview认证要求的最新知识。
  • 在最初的架构扫描过程中未能检测到SessionManager从未被实例化——花费大量时间调试端口冲突和WebSocket问题，最终发现根本原因是空的onMessage处理程序。
  • 人工智能在分析时未能质疑“重复错误注入”设计的合理性，需要人类明确指出“修复的错误场景+重置随机化”才能开始重构。
  • 路径假设不正确：代理最初假设文件是在recovery_collection_portable/下编辑的，但用户已删除了一个层级。所有编辑内容都丢失了，需要在新的路径中重新应用。
  • 最初没有意识到过时的定价缓存可能会阻止OpenRouter数据加载——是通过使用litellm.rs缓存逻辑进行显式代码追踪才发现的版本控制差异。

一般限制#

  • 触发了OpenReview nginx速率限制，导致在诊断阶段连续请求约30次后，临时IP被封禁（152.3.43.42）。应该更早地实施指数退避或诊断认证要求。
  • 子代理执行过程不透明：当5个后端代理同时运行时，用户无法看到中间进度（只能看到“等待完成通知”），缺乏实时可见性。对于大型任务来说，进度感知是一个用户体验问题。

经验教训#

关键经验- 超级技能链的强大之处：头脑风暴（需求分析）→ ccplan（ECL驱动的实现）→ 优化（质量提升）构成了完整的软件生命周期。ECL文档作为跨会话的持久“设计合同”，确保了在多次对话、多个代理甚至不同模型之间的上下文一致性。#

  • 对抗性评审的价值：由Critic代理发现的9个问题中，有2个是严重的（如无身份验证的PTY编写、SSH参数注入），4个是较严重的（如令牌暴力破解、原型污染），如果在生产环境中暴露这些问题可能会导致安全灾难。在需求阶段进行修复的成本远低于代码阶段和生产阶段的成本。
  • 以对象为中心的变形原理：EEF相对于物体运动模式是一个不变量。这个不变量通过坐标变换T_eef_new = T_obj_target × T_obj_src⁻¹ × T_eef_source来保持。两步理解：（1）消除源物体的绝对位置，（2）用新物体的位置重新映射。
  • MimicGen设计背后的智慧：（1）最小化子任务数量（仅2个，用于Stack）（2）使用10-20帧的后抓取缓冲区以确保稳定捕获（3）通过env.reset()回到初始位置来减少姿态差异（4）以准闭环方式执行（逐步计算目标_pose_to_action的增量）。
  • MimicGen的核心设计是源演示+ env.reset()随机化：应将错误场景视为固定的“初始状态”，然后使用placement_initializer生成变体，而不是为每个目标反复注入错误。
  • NeurIPS DB Track评审的优先级（基于61次评审）：最受重视的品质包括“全面性”（提及17次）、“社区性”（17次）、“新颖性”（15次）、“清晰性”（28次）以及“设计良好”（13次）。最常见的批评是“缺乏比较”（基线不足）、“细节不够”（方法不明确）和“范围有限”（应用受限）。最佳论文（Artificial Hivemind）在Oral论文方面的平均得分为4.75——这超过了AC确定的评审分数。Oral的平均得分为5.00，而Spotlight的平均得分为5.16，这表明评分并不是唯一考虑的因素。
  • 并行代理调度模式：将独立模块（Config/Parser/LocalPTY）分配给不同的子代理进行并行实现，然后收敛并验证，使开发效率提高3-5倍。关键在于识别DAG中的非依赖项，以最大化并行性。
  • Claude Code –output-format stream-json：通过研究发现，Claude原生支持结构化输出（换行分隔的JSON），无需对PTY进行常规解析。这提高了Parser的可靠性，从“启发式”变为“确定性”。总是在重新发明之前先进行研究。
  • TypeScript严格模式的投资回报：195次测试+0个类型错误=生产就绪代码。类型系统在重构时特别有效（在提取了解析状态Utils之后，所有调用点都会自动进行类型检查）。在早期阶段多花20%的时间编写类型，可以节省80%的后期调试时间。
  • xterm.js是行业标准浏览器终端（被VS Code、Replit、Vercel、ttyd使用）。其质量取决于完整的插件堆栈：用于GPU渲染的WebGL、用于CJK的Unicode11、用于调整大小的Fit、SearchAddon、WebLinksAddon。仅基础的xterm.js不足以满足生产环境的需求。
  • 准备MimicGen的数据集需要特定的元数据结构：每个demo_*/datagen_info必须包含env_interface_name和env_interface_type，否则DataGenerator无法正确初始化环境。
  • 在收集阶段之前应对人类演示数据的质量进行验证：仅仅拥有远程操作成功是不够的；还必须验证重放稳定性和可扩展性。否则，将收集大量无法用于训练的数据。
  • 时间边界划分（归档完成小时数、当前小时数的实时源覆盖）巧妙地解决了重复删除问题，而无需在记录级别显式跟踪——零重叠意味着零重复计数。
  • Claude Code的权限系统支持工具级通配符（如“Bash”匹配所有Bash命令），允许最少配置，而不是数十条逐项规则。
  • 子任务碎片化的成本：每个子任务边界都是一个独立的“检查物体姿态→重新映射→5步插值”，错误会累积。6个子任务=5个错误来源，远远超过原来的一个。
  • OpenReview API认证架构（2026年）：所有数据端点都需要使用令牌登录；只有/login端点允许匿名访问（返回验证错误）。网页前端通过Next.js RSC使用服务器端凭据。限流在nginx级别进行（基于IP），在大约30次请求后触发。学术论文评审API越来越严格，需要机构或个人凭证。
  • 终端大小调整同步需要全链路协调：ResizeObserver（前端）→ term.onResize → WebSocket消息→SessionManager→PTy.resize()。不匹配会导致错误列中出现错误。需要阻尼机制来防止拖放调整时导致 flooding问题。
  • 在会话切换期间不应销毁终端实例——它们保留回滚缓冲区（10,000行）。正确的模式是：创建一次，然后用clear()填充+后端RingBuffer（2MB）进行跨会话重用。
  • 当数据源可以演变时，缓存系统必须包含明确的版本字段——仅TTL到期不足以处理模式/来源的变化。
  • 权限配置允许/拒绝/询问三层机制可以在安全性和效率之间取得平衡：允许实现自动化，询问保留危险操作的确认，拒绝用于绝对禁止。
  • Claude Code配置不会自动继承（每个项目都需要独立配置），但可以通过自动化脚本实现标准化的初始化过程（如init-claude.sh）。
  • 复杂的机器人系统文档应区分开发者和操作员：通过结构化问题明确定义目标受众、结构、语言和内容保留，以避免歧义。
  • React闭包陷阱参考模式：在useEffect中注册第三方库回调以捕获xterm.js onData中的过期props是一个常见错误。一般的解决方案是：props→ref，useEffect与ref.current同步，回调读取ref.current。这是React生态系统中的最佳实践。
  • 实现WebSocket稳健性的三个关键要素：重新拨号（指数级回退）、压力测试（检查bufferedAmount）和心跳（未实现但应考虑）。分布式系统的网络层必须假设“连接可能随时断开”，而不是“连接应保持稳定”。- 持续闭合检测的重要性：机器人捕获可能会尝试多次。如果第一次尝试时gripper_closed大于阈值而失败，则必须检测到连续闭合（≥10帧）以确认成功捕获。
  • 多进程调试的复杂性：（1）每个工作线程都需要一个独立的环境扩展器实例；（2）路径解析更容易出错（相对路径可能会遗漏层级）；（3）像demos_dir这样的绝对路径必须明确传递；（4）需要在工作线程函数中单独抑制Robosuite警告。
  • 配额计数逻辑应反映实际可用性：仅可增强模式比仅重放模式更严格但更合理，因为不可增强的演示即使可以重放也无法扩大训练集。
  • 最先进的ASR模型（2026年）：真正的流式模型很少见。Moonshine v2（73-107毫秒延迟，34-245百万参数）和Voxtral-Mini-4B-Realtime（480毫秒，4亿参数）是最先进的。Moonshine使用基于回调的API，适合实时转录；Voxtral需要手动分块处理，但准确性更高。两者都支持Apple Silicon原生加速（Moonshine使用CoreML，Voxtral使用MLX）。
  • 探索性项目和生产项目的权限要求不同：探索性项目优先考虑效率（宽松的权限），而生产项目则优先考虑安全性（严格的确认）。
  • 逐渐积累的配置可能导致不一致性（例如，80条以上长的规则，不同的项目风格），但定期使用模板进行标准化可以提高可维护性。

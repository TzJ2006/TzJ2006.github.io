问题与解决方案#

严重问题#

1. Claude代码JSONL格式的变化导致成本增加了2.4倍：包含多个内容块的助手消息被写成单独的行，且相同内容的重复数据在请求ID缺失时被跳过。#

解决方案：修改了create_claude_unique_hash()函数，当请求ID为空时，仅使用message.id作为备用方案。现在，同一消息在不同行上的重复数据可以通过单个API调用正确去重。

关键洞察：新的Claude代码格式在许多条目中缺乏顶级请求ID；由于内容块共享相同的message.id，因此message.id足以作为唯一的标识符。

2. SSH远程同步导致数据膨胀了10-25倍：提取脚本产生了重复的内容块行，而本地追加操作使用了-newer文件过滤规则，导致在任何更新时都会重新提取整个文件，且在追加之前没有进行去重处理。#

解决方案：采取两层修复措施：(1) 远程Python脚本在提取过程中保持seen={}基于message.id。(2) 本地sync_host()函数加载现有缓存，从(ts, model, in, out)元组构建密钥集，在追加之前过滤掉新记录。

关键洞察：基于文件的增量同步（-newer mtime）并不意味着记录级别的去重。每一层（提取、摄取、聚合）都需要显式的重复检测。

3. ccusage报告的令牌使用量比TokenMonitor高出约1.75倍，这导致了实际消耗量的混淆。#

解决方案：在ccusage的createUniqueHash()函数中发现了根本原因，需要同时使用message.id和requestId。当请求ID缺失时（20,116个条目中的13,960个），不会进行去重处理，导致重复数据被多次计算。此外，ccusage使用了“先胜”策略，保留了早期令牌数量较低的快照。

关键洞察：流式API每次响应都会生成多个JSONL快照，并增加输出令牌数；“先胜”去重策略保留了早期令牌数量较低的快照，而TokenMonitor的“最大输出令牌数”策略则能够准确捕捉最终值。

4. Apple Speech字幕一直处于灰色状态，从未变为白色状态，新文本覆盖了旧文本而不是追加到旧文本上。#

解决方案：修改了_on_result回调函数的条件，从‘if text and text != self._last_text’改为‘if text and (is_final or text != self._last_text)’，以确保最终回调不会被去重逻辑跳过。

5. Agent的claude_parser.rs提取代码使ssh.rs回到了旧版本，从而丢失了Task 3的device_aggregation重构工作。#

解决方案：人工检查了文件行数，将ssh.rs的行数恢复到了1536行。手动重新应用了所有对ssh.rs的修改：移除了聚合函数，更新了导入语句，修复了所有调用者。并通过cargo检查+测试进行了验证。

6. Apple Speech音频捕获在将音频样本复制到AVAUDIO_PCM_BUFFER时停止工作，出现了‘TypeError: wrong type’错误。#

解决方案：通过直接使用ctypes调用objc_msgSend函数来绕过PyObjC桥接，使用sel_registerName(‘floatChannelData’)，获取与ctypes兼容的浮点指针，而不是使用PyObjC包装的对象。

7. SSH远程提取只能找到~/.claude/projects和~/.config/claude/projects文件夹，缺少自定义安装，如~/.claude-code和不同模型的文件夹。#

解决方案：用glob模式替换了硬编码的路径，即‘for _cd in $HOME/.claude*; do [ -d $_cd/projects ] && …’来自动发现所有以.claude开头的目录。同时，还修改了shell的rc文件以获取CLAUDE_CONFIG_DIR环境变量。

8. GitHub Actions的缓存接近10GB的限制，因为缓存了整个目标目录，包括增量构建工件和文档。#

解决方案：用Swatinem/rust-cache@v2替换了actions/cache@v5，该工具只缓存依赖项。在CI作业中添加CARGO_INCREMENTAL=0，从而减少了80%的缓存空间。

9. 图表线的Y坐标与工具提示的值不匹配：51.45点的数值显示在0附近，而工具提示正确地显示了51.45。#

解决方案：将lineData的段查找方式从find()改为filter().reduce((sum, s) => sum + s.cost, 0)，以便对每个桶中的所有匹配段进行求和。关键洞察： 聚合数据（工具提示）和视觉编码（行位置）必须使用相同的求和逻辑。设备级别的细分意味着一个模型可以在每个时间区间内有多个部分

10. DevicesView 和主图表显示的设备细分总数不同。DevicesView 使用了存档和前沿过滤，而主页面则直接读取 raw remote-cache usage.jsonl 文件，没有排除存档时间段的记录#

解决方案： 统一两种代码路径，都调用 build_device_summary_merged() 函数，并添加 archive.frontier() 过滤器，跳过已被存档条目覆盖的紧凑记录

关键洞察： 存档系统创建双重数据源（存档的 JSONL 文件和实时缓存数据）。所有消费者都必须使用前沿截止时间来避免重复计算过渡期的数据

常见问题#

11. CI 测试失败，提示“无法找到类型 FloatBallSizes”——Linux 环境下使用了 #[cfg] 块来引用 FloatBallSizes，但 mod.rs 在 Linux 环境下没有导入该模块#

解决方案： 添加了 #[cfg(target_os = “linux”)] 条件，并使用 layout::FloatBallSizes；同时为 FloatBallAnchor 添加了 #[allow(unused_imports)] 条件（仅在非 Linux 环境下的 #[cfg] 块中使用）。通过 cargo clippy 进行了验证

关键洞察： 平台特定的编译需要平台特定的导入语句。未使用的导入警告可能取决于平台；应在 #[cfg] 文档中使用 #[allow(unused_imports)] 条件

12. 图表 Y 轴缩放过于粗略：从 $50 到 $100 的跳跃使得 $51-75 的数据看起来“平坦”（只使用了图表高度的一半）#

解决方案： 为 niceMax() 序列添加了中间阈值：10-50 之间的阈值为 15/30，50-100 之间的阈值为 75，100-500 之间的阈值为 150/300，以此类推。现在 $51 对应 $75 的上限，而不是 $100

关键洞察： 常见的数值刻度需要在常用范围内有更密集的步长。对于通常价格范围在 $10-100 的成本数据来说，2 倍的跳跃幅度过大

人工智能的局限性#

关键局限性#

  • 代理驱动的文件操作可能会默默地撤销未提交的变化，如果它们是从系统提醒的 linter 保存的状态中读取的，而不是实际的工作树。在代理完成操作后，没有主动验证之前会话的工作是否保存下来
  • 没有主动怀疑 SSH 同步管道是重复的来源；需要用户的明确假设来调查远程提取和追加逻辑。首先关注本地解析器，而不是基于假设的调查
  • 最初将 Apple Speech 问题误诊为文件写入问题，而不是回调被去重逻辑过滤掉；根据用户对“文本保持灰色”的描述，应该先分析回调链
  • 在 ccusage 分析中，最初怀疑是模式验证问题，并花费了多轮时间来探索时间戳正则表达式、使用字段结构——实际上根本原因是 createUniqueHash 返回了 null。应该先检查去重逻辑
  • 最初声称用户的 requestId 字段“完全缺失”，但实际上数据显示有 30.6% 的条目存在 requestId

一般局限性#

  • 在修复折线图 Y 轴时，最初只改变了段聚合，但忽略了条形图（堆叠）和折线图（非堆叠）需要不同的最大计算。需要视觉分析才能意识到最大成本应该根据图表模式有所不同
  • 直到 CI 测试失败之前，才注意到平台特定的导入问题（仅需要在 Linux 上使用的 FloatBallSizes）。Rust 的 #[cfg] 条件编译创建了依赖于平台的符号可见性，这在本地 macOS 构建中未被捕捉到

经验教训#

关键经验- 多层数据管道需要在每个边界处进行防重复处理：提取（远程脚本）、导入（本地追加）、聚合（解析器）和归档（前沿过滤）。基于文件的增量同步无法保证记录级别的唯一性。#

  • 对于流式API，正确的JSONL去重方式应该是：相同的message.id被多次写入，但output_tokens会递增。必须保留最后一条记录或最多输出_tokens数量。如果仅依赖requestId（可能缺失），会导致去重失败。采用“先入者胜”策略会低估输出令牌的数量。
  • 开源工具的正确性验证：ccusage有两个未解决的漏洞——#797（缺少requestId导致重复计数）和#705（“先入者胜”策略低估了输出令牌的数量）。通过跨工具比较和搜索GitHub问题，可以有效验证算法的正确性。
  • Apple的SFSpeechRecognizer在isFinal=True时返回相同的文本；去重逻辑必须明确检查is_final标志，以避免跳过最终化的回调。
  • 使用具有共享可变状态的工厂闭包模式（例如，resizeOrchestrator有19个状态变量）不应该强行重构。成本效益分析表明，只有6行是纯函数；提取功能需要传递19个参数或引入状态对象，从而增加认知负担。
  • 对于类似的功能（如汇率），重复使用经过验证的架构模式可以确保一致性并减少错误。两种实现方式都是：获取API → 磁盘缓存（时间容忍）→ OnceLock<RwLock<» → 启动加载 + 异步刷新 + 后台循环检查。
  • GitHub Actions的缓存策略：增量编译在持续集成中没有任何好处（每次运行都需要冷构建）。Swatinem/rust-cache能够智能地只缓存依赖项而不是整个目标目录，从而将缓存量减少80%，同时保持速度。
  • 用户对系统行为的直觉（如SSH同步重复）通常比AI的代码路径分析更快，能够更快地找到根本原因。当症状有清晰的模式时，假设驱动的调试比穷尽式的追踪更为有效。
  • 流式API的去重方法应该采用“最后获胜”或“最大值”策略，而不是“先入者胜”，因为早期的数据块包含不完整累积值（尤其是输出令牌）。
  • PyObjC桥接对象与ctypes不兼容；可以使用ctypes objc_msgSend结合sel_registerName来实现直接C API互操作，以获取原生指针。
  • 基于glob的路径发现（~/.claude*）比硬编码特定目录名称更易于维护，特别是当用户可能有并行安装且配置不同的情况时。
  • 对于大型Rust文件，可以采用模块提取模式：目录模块（foo/带有mod_rs和layout_rs）可以将纯逻辑与状态化/平台特定的代码分开。这一模式已成功应用于float_ball_rs（几何图形→layout_rs，IPC/窗口→mod_rs）和parser_rs（Claude解析→claude_parser_rs）。
  • 图表视觉编码和工具提示聚合必须使用相同的数据管道。如果工具提示汇总了所有设备段，但某行只使用了第一个段，那么即使两者在自己的上下文中都是“正确”的，这种不匹配也是显而易见的。
  • jq作为无状态JSON处理器无法实现去重——需要像Python/awk这样的有状态工具。多路径提取逻辑（jq/python/grep）必须保证行为的一致性。
  • 归档系统创建双重数据源（归档后的JSONL和实时缓存）。所有消费者都必须使用前沿截止点来避免重复计算过渡期的数据。
  • MimicGen增强功能需要深入的语义理解：任务阶段编码了操作进度，对象锚定防止无效场景，变形保留了相对姿态，而成功验证是必不可少的。没有上下文感知的适应性，纯轨迹重放是无法成功的。
  • 去重键的构建应该使用回退策略（例如，当requestId缺失时使用message.id），而不是返回null，以避免意外地将重复项视为唯一条目。

实际经验#

  • 好的数值规模需要根据领域典型的值范围进行经验性调整。通用的几何级数（1/2/5/10/20/50/100）对于集中在10-100美元的成本数据来说过于粗糙；中间步骤（15/30/75）则能改善数据密度比例。

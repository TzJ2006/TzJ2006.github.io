错误信息：2026-03-03

2026年3月3日 · 3 分钟

问题及解决方案#

主要问题#

  1. 在显示视频时，SwiftUI .ultraThinMaterial 显示的不是透明的玻璃效果，而是固定的颜色。

    • 解决方案： 将 SwiftUI 材料替换为 NSViewRepresentable，包裹 ，并在 CATransaction.setDisableActions(true) 中指定 .inWindow 模式以及 clear backgroundColor。
    • 关键洞察： SwiftUI 的抽象层隐藏了渲染过程的细节。对于窗口内部透明等复杂的视觉效果，需要在 NS 级别进行控制。

  2. 在 151676 中，所有 STAIG 融合嵌入结果都为零值（嵌入相似度为 0，Spearman r 为 NaN），表明模型完全失败。

    • 解决方案： 在 load_fused_embeddings 中添加对 np.allclose(embeddings, 0) 的检查。如果满足此条件，则打印警告并自动触发 --recompute_fusion，从缓存中重新加载基因/视觉嵌入数据并进行 STAIG 重新训练（约 300 个周期，在 RTX 2080 Ti 上需要大约 50 秒）。修复后，嵌入结果恢复正常（64 维，标准差为 1.0）。
    • 关键洞察： 由于训练不稳定，第二阶段保存的 STAIG 融合嵌入可能会变为零。需要在基准测试脚本中主动检测并自动重新计算，以避免手动干预。

  3. 在 NSOpenPanel 中选择一个视频后切换其他视频时，旧视频仍然播放。

    • 解决方案： 添加机制来检测书签 URL 之间的不匹配情况，如果没有匹配则返回原始 URL，并在 ensureFileAccess 中添加保护。
    • 解决方案： 确保 FileAccess 根据屏幕上的书签解析旧视频 URL，而不是请求新的 URL。添加了比较解析路径与请求路径的保护机制；如果它们不同，则返回原始 URL（该 URL 由 NSOpenPanel 提供临时沙箱权限）。

关键洞察： 每个屏幕上存储的书签与多视频历史记录之间存在冲突；需要保护机制来检测书签指向的过期文件。

Coffee 任务的成功率仅为 0%，怀疑是评估错误。#

  • 解决方案： 检查了训练日志，确认 Coffee BC-RNN 训练本身在所有周期中（600 个周期）的 SR 为 0%。这不是评估错误；该任务的训练确实失败了。
  • 关键洞察： 在调试评估过程之前，始终需要验证训练指标。下游为零值可能反映了上游训练的失败。

an53 节点在 08:30 崩溃，导致 tmux 会话丢失，任务 46798 得以执行。#

  • 解决方案： 通过 sacct 日志进行了调查，确定崩溃是由资源过载引起的（8 个 GPU 的使用率达到了 100%，且 CPU 使用率很高）。排除了 AI 操作作为原因（最后一次完成是在 05:07，即崩溃前 3 小时）。通过使用 ssh-keygen -R an53 解决了 SSH 主机密钥不匹配的问题。
  • 关键洞察： Slurm 报告了关于崩溃的少量信息，但没有对系统日志进行 root 访问。srun 命令中的“请求节点繁忙”错误表明节点过载，而不是命令失败。使用 Slurm 时，即使没有 root 访问权限，也可以获得最少的崩溃信息。

P100 GPU 的 CUDA 内核与 PyTorch（sm_70+ 版本）不兼容；torch.cuda.is_available() 返回 True，但异步操作失败。#

  • 解决方案： 最初尝试使用 try/except 来捕获 torchzeros(1).cuda() 的异常，但由于 P100 上的异步操作无法捕捉错误。用户在切换到 RTX 2080 Ti（sm_75）后成功运行了该程序。最后，添加了 CUDA_VISIBLE_DEVICES="" 环境变量，以强制在脚本中使用 CPU。

LLM 生成的 JSON 被截断（…截断标记），导致 daily_summary.py 在 2026-02-14 及其他日期无法解析。#

  • 解决方案： 将截断的原始文本作为单独的提示发送给 AI，仅请求修复/提取有效的 JSON 部分；AI成功恢复了完整的 JSON 结构。
  • 关键洞察： 当 LLM 生成非常长的 JSON 时，存在截断的风险（尤其是在多个片段合并的情况下）；应在合并阶段添加截断检测和自动 JSON 修复重试逻辑，而不是完全依赖手动发现和干预。

在更新屏幕保护时钟文本后，有 0.2 秒的残影。#

  • 解决方案： CALayer 对于内容属性的更改具有大约 0.25 秒的隐式动画。在 CATransaction 中更新 wrapped blurView.layer?.mask 时，设置 setDisableActions(true) 以强制立即替换。

关键洞察： 默认情况下，CALayer 几乎对所有属性更改都有动画效果；需要显式的事务控制来实现即时视觉更新。

BC-RNN 策略评估因“无法处理数据类型 <class ’numpy.float64>’”而崩溃。#

  • 解决方案： Robomimic 的 TensorUtils.to_tensor() 无法处理 numpy 标量类型。修改了_to_robosuite_obs，只将包含数组/图像值的键传递给 robomimic 策略。

关键洞察： Robomimic 期望一个包含数组值的结构化 obs 字典；在传递给 TensorUtils 之前，必须过滤掉 robosuite 返回的 numpy 标量（如 gripper_width_float64）。

可视化脚本多次失败：Bash heredoc 不符合 != 语法错误，文件名前缀不匹配（151673_filtered_feature_bc_matrix.h5 和 filtered Feature_bc_matrix.h5），空间/数据路径错误（位于 Dataset/DLPFC，而不是直接在部分目录中），load_dlpfc_data() 返回了值错误（返回了 1 而不是 2），read_10x_h5 缺少 uns[‘spatial’]，导致 sc.pl.spatial() 失败。#

  • 解决方案： 使用独立的脚本/_viz_rm_ideal.py 文件代替 heredoc；使用 glob “*filtered_feature_bc_matrix.h5” 来查找正确的文件名；直接加载 sc.read_visium("/path/to/Dataset/DLPFC/151673")（自动处理空间子目录）；修复了 load_dlpfc_data 的未解包结果是一个单一返回值的问题；放弃了使用 load_dlpfc_data，改用 sc.read_visium 来确保存在 uns[‘spatial’]。关键洞察： Bash的heredoc对特殊字符的转义非常敏感；复杂的脚本应该作为文件来编写。DLPFC数据文件的名称有章节前缀；空间数据位于Dataset/DLPFC/{section}/spatial而不是{section}/spatial；load_dlpfc_data使用read_10x_h5代替read_visium，导致缺乏空间元数据；sc.pl.spatial严重依赖uns[‘spatial’]，必须使用read_visium来加载。

常见问题#

屏幕保护程序的时钟文本在边缘显示灰色渐隐效果。#

解决方案： 根本原因是ScreensaverClockHighlight（.light）和renderTextMask（.regular）之间的字体权重不匹配，导致掩码变厚，从而暴露了hudWindow的磨砂玻璃边缘。通过统一字体权重为.pdfigma解决了这个问题。

关键洞察： 视觉效果层必须保持像素级对齐；即使轻微的权重差异也可能暴露底层内容。

处理13天的历史数据需要频繁切换日期参数并手动确认导出/合并状态，这既繁琐又容易出错。#

解决方案： 每天手动执行daily_summary.py的export –date YYYY-MM-DD和merge命令，单独处理每个日期，并通过文件系统确认输出文件的完整性。

关键洞察： summarize工具缺乏批处理模式（例如，–date-range 2026-02-07..2026-02-19会自动迭代）；未来应添加对批处理和进度恢复机制的支持。

通过SSH从项目目录而不是主目录启动时，使用无hup的相对路径会失败。#

解决方案： 将命令从’ssh an53 “nohup bash scripts/…"‘改为’ssh an53 “cd $PROJ && nohup bash scripts/…"’，确保在脚本执行前正确输入了工作目录。

差异分析： 人类利用关于模型架构的领域知识发现了基本的训练配置错误。人工智能忽略了这一战略性问题，而专注于操作指标。

诊断零嵌入问题#

角色 方法
人类 在发现嵌入相似度为0后，立即询问“能给我展示一下可视化结果吗？”从数值检查转向视觉诊断；找到根本原因后，请求了跨样本补丁查询的方法。

差异分析： 人类从数值检查到可视化，再到方法总结，积极推动了整个过程；人工智能倾向于逐步解决单个问题，缺乏全局视角和主动性。

an53节点故障的原因分析#

角色 方法

人工智能 人工智能系统地分析了Slurm日志（sacct），跟踪所有srun命令（04:55-05:07，全部成功完成），在8:30时识别出NODE_FAIL问题（AI活动后3小时），并诊断出可能是由于资源过载。

差异分析： 人类使用间接证据和同行报告来形成假设。人工智能利用系统日志和时间戳来构建清晰的时间线，证明操作在故障前3小时结束。

基准测试的设计决策#

角色 方法
人类 人类明确指定只测试PCA+UNI2+STAIG_FUSION组合，目标章节为151673和151676。首先，他们创建单独的脚本来验证可行性，然后将它们整合到后续过程中。他们反复拒绝ExitPlanMode，期望在方向明确时直接执行。
人工智能 最初的设计涵盖了多种融合方法和更多的章节；在每个计划创建后，它尝试使用ExitPlanMode进行批准；由于多次被拒绝，它未能理解用户的执行风格偏好。

差异分析： 人类做出了战略决策（范围控制、优先级排序），而人工智能倾向于全面覆盖一切；人类更喜欢直接执行模式，并且人工智能的保守批准过程与用户期望的执行效率之间存在紧张关系。

差异分析： 人类具有批处理和时间规划能力的全局意图，而人工智能只能看到单个调用；人工智能缺乏主动识别重复模式并建议自动化改进的能力。

人工智能的局限性#

主要局限#

  • 在多次评估过程中，未能主动检测到pi0.5基础模型中的配置错误（libero与基础模型之间的差异），直到用户明确提出了这个问题。
  • 缺乏主动性和全局视角：①没有主动提出可视化解决方案，这需要用户的明确请求；②没有主动建议撰写方法论文，这也需要用户的请求；③在多次ExitPlanMode失败之后，仍然未能理解用户的执行风格偏好（直接实施而不是等待批准）；④在发现零个嵌入问题后，未能主动诊断根本原因（为什么在第二阶段保存的嵌入会导致崩溃）。
  • compute Embedding_similarity函数的设计错误：它在run_cross_section_benchmark中错误地调用了compute Embedding_similarity(embeddings_ref, niche_mask_query=None)，参数不正确且逻辑混乱；只有在重新创建compute_cross_embedding_similarity函数之后才得以修复。这表明对交叉部分相似性计算的理解不足。
  • Bash heredoc和路径处理中的多次失败：①Bash heredoc != 用!=转义（语法错误）；②多次错过带有部分前缀的DLPFC文件名称；③误认为spatial/直接位于子目录中，而不是在Dataset/DLPFC/{section}/spatial中；④滥用load_dlpfc_data()导致缺少uns[‘spatial’]。这表明对DLPFC数据集文件组织和scanpy API的理解不足。
  • CUDA兼容性检测策略的错误：torch.cuda.is_available()仅检查驱动程序和库，而不检查GPU计算能力的兼容性；try/except无法捕获CUDA异步失败；在P100异步失败的情况下，多层CPU回溯无效。这表明对PyTorch CUDA机制和异步错误传播的理解不足。 -作为每日报告分析师，需要反复处理类似任务（分析历史对话→生成结构化报告），但未能识别为批量处理场景，也没有主动建议“是否需要批量处理模式”或“自动化脚本”以提高效率。 -在子代理探索任务中，API令牌多次过期，阻碍了对checkpoints和zhaoganlong文件夹的调查。需要手动干预/登录。 -在尝试解决BC-RNN numpy dtype错误之前，进行了多次调试迭代（尝试跟踪回溯路径、ssh重试、检查点），最终确定to_robosuite_obs是过滤标量类型的重点。 -当看到咖啡评估中的SR为0%时，未能主动检查训练日志；在调查上游训练失败之前，需要明确的问题陈述。

问题与解决方案#

主要问题#

1. SwiftUI .ultraThinMaterial在视频中显示的是固定颜色的玻璃，而不是透明玻璃#

**解决方案：**SwiftUI材料使用.behindWindow混合模式（在窗口后面显示桌面），而不是.integralWindow模式（在窗口内显示内容）。使用NSVisualEffectView替换它，并在NSHostingView中明确设置.integralWindow模式并清除backgroundColor。

**关键洞察：**SwiftUI的抽象性隐藏了复杂视觉效果的细节；对于像窗口级透明度这样的复杂视觉效果，控制需要放在NSView级别。

2. STAIG融合嵌入的151676个样本全部为零（所有嵌入相似度值均为0，Spearman r为NaN），导致模型完全崩溃#

**解决方案：**在load_fused Embeddings中添加np.allclose(embeddings, 0)检测；如果为真，则打印警告并自动触发–recompute_fusion路径，从缓存中重新加载基因/视觉嵌入并重新训练STAIG（300个周期，大约50秒在RTX 2080 Ti上）。修复后，嵌入值恢复正常（64个维度，范数为1.0）。

**关键洞察：**由于训练不稳定，第二阶段保存的STAIG融合嵌入可能会降为零；在基准测试脚本中需要主动检测和自动重新计算，以避免手动干预。

3. 在NSOpenPanel中选择书签后切换视频时，旧视频仍然播放#

**解决方案：**确保FileAccess将每个书签解析为旧视频URL，而不是请求新的URL。添加了一个守卫来比较解析出的URL与请求的URL；如果不同，则返回原始URL（该URL由NSOpenPanel提供临时沙箱权限）。

**关键洞察：**书签存储与多个视频历史记录冲突；需要有一个守卫来检测书签指向的文件已过期。

4. BC-RNN在Coffee任务上的成功率率为0%，怀疑是评估错误**解决方案：**检查了训练日志，确认BC-RNN在咖啡任务上的训练在所有阶段都达到了0%的成功率（进行了600个周期的训练）。这不是评估错误；任务在训练过程中确实失败了。#

**关键洞察：**在调试评估过程之前，验证训练指标是非常重要的。下游指标中的零值可能表明上游训练过程出现了失败。

Node an53在08:30崩溃，导致tmux会话丢失，作业46798失败。#

**解决方案：**通过sacct日志进行了调查，确定崩溃是由资源过载引起的（8个GPU的使用率达到了100%，且CPU使用率很高）。排除AI操作作为原因（最后一次完成是在05:07，即崩溃前3小时）。通过使用ssh-keygen -R an53解决了SSH主机密钥不匹配的问题。

**关键洞察：**Slurm报告了崩溃的详细信息，但它无法访问root系统的系统日志。

P100 GPU上的CUDA内核（sm_60）与PyTorch不兼容（需要sm_70+版本）。torch.cuda.is_available()返回True，但异步操作失败。#

**解决方案：**最初尝试使用try/except来捕获torch zeros(1).cuda()异常，但在P100上无法捕捉到异步失败。用户成功地在脚本中替换为RTX 2080 Ti（sm_75）后运行脚本。最后，在脚本中添加了一个环境变量CUDA_VISIBLE_DEVICES=““来实现强制使用CPU的逻辑。

**关键洞察：**torch.cuda.is_available()仅检查驱动程序和库是否存在，而不检查GPU的计算能力是否与当前的PyTorch兼容。CUDA操作可能会异步失败，无法被Python的try/except捕获。最可靠的方法是更换GPU或使用环境变量来强制使用CPU。

LLM生成的JSON被截断（…截断标记），导致daily_summary.py在2026-02-14及其他日期无法解析。#

**解决方案：**将截断的原始文本作为单独的提示发送给AI，只要求修复/提取所需的有效JSON部分。AI成功恢复了完整的JSON结构。

**关键洞察：**LLM生成的极长JSON有被截断的风险（特别是在多个片段合并的情况下）。应在合并阶段添加截断检测和自动JSON修复重试机制，而不是仅仅依赖手动发现和干预。

屏幕保护程序时钟文字在更新超时文本时会产生0.2秒的模糊现象。#

**解决方案：**CALayer默认动画效果约为0.25秒，用于更改内容属性。在CATransaction中，blurView.layer?.mask被更新，并且setDisableActions(true)被设置为强制立即替换。

**关键洞察：**CALayer几乎对所有属性更改都有动画效果。需要明确的交易控制来实现即时视觉更新。

BC-RNN策略评估因“无法处理数据类型<class ’numpy.float64>’”而崩溃。#

**解决方案：**Robomimic的TensorUtils.to_tensor()无法处理numpy标量类型。to_robosuite_obs被修改，以过滤掉raw_obs字典中只包含数组/图像值的键，这些值会被传递给robomimic策略。

**关键洞察：**Robomimic期望一个包含数组值的结构化obs字典。必须从robosuite中过滤出NumPy标量（例如gripper width float64），然后再将其传递给TensorUtils。

可视化脚本多次失败：Bashheredoc不等于转义字符!=（语法错误），文件名前缀不匹配（151673_filtered_feature_bc_matrix.h5与filtered_feature_bc_matrix.h5），空间/数据路径错误（位于Dataset/DLPFC，而不是直接在章节目录中），load_dlpfc_data()返回值解析错误（返回1而不是2），read_10x_h5缺少uns[‘spatial’]，导致sc.pl.spatial()失败。解决方案： 使用独立的脚本/_viz_rm_ideal.py文件代替heredoc方法。使用glob “*filtered_feature_bc_matrix.h5"来查找正确的文件名。直接调用sc.read_visium("/path/to/Dataset/DLPFC/151673”)进行加载（自动处理空间/子目录）。解决了load_dlpfc_data函数无法将数据拆解为单个返回值的问题。放弃了load_dlpfc_data函数，直接使用sc.read_visium来确保uns[‘spatial’]的存在。#

关键洞察： Bash的heredoc对特殊字符转义敏感。复杂的脚本应该写成单独的文件。DLPFC数据文件的命名包含章节前缀。空间数据位于Dataset/DLPFC/{section}/spatial/而不是{section}/spatial/。load_dlpfc_data使用read_10x_h5代替read_visium，导致缺少空间元数据。sc.pl.spatial严重依赖于uns[‘spatial’]，因此必须使用read_visium进行加载。

常见问题#

屏幕保护时钟文字的边缘有灰色的幽灵阴影。#

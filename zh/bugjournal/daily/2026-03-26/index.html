问题和解决方案#

关键问题#

1. ccplan的自动执行阶段（4-6）由于浅层分析而进展过快，这是由于“连续执行指令”所驱动的#

**解决方案：**增加了明确的深度要求：第4阶段扩展到10个维度，设置了最小检测阈值；第5阶段变成了4级分析（链/交叉切割/流程/契约）；第6阶段至少需要一个检测和结构化风险评估。

**关键洞察：**自动执行阶段需要定量深度要求和最小输出阈值。像“彻底”这样的定性指令不足以推动效率提升。

2. 当WebSearch/WebFetch工具返回标签时，ccplan工作流程会被中断，导致技能列表重新加载并丢失阶段上下文#

**解决方案：**增加了工具调用状态保存协议，包括3步恢复机制：在工具调用之前记录当前阶段，在工具返回后重新锚定到ECL状态字段，忽略系统提醒元数据；在第0/3/7阶段添加了内联注释，这些阶段可以进行搜索。关键洞察： 复杂的多阶段工作流程需要清晰的状态保存协议，尤其是在调用返回大量元数据的外部工具时。隐式的上下文连续性无法依赖。

3. 修复过程中的函数回归问题 - Claude Code由于上下文压缩而忘记了某些功能，导致意外删除#

解决方案： 实现了具有功能保护部分的持久性ECL文档。开发了一个安全的guard-check.py CLI工具（带有命令执行的交互确认），该工具实现了Phase 10的自动生成，并增加了预修改检查协议。

关键洞察： 内存和上下文是临时的，但磁盘数据是永久性的。写入ECL文件的 Guards可以在会话和压缩过程中保持不变。便携性（与技能目录一起携带）比特定项目的解决方案更好。

4. 浮球拖动失败 - startDragging()函数在setTimeout回调中被调用，导致Windows所需的指针事件上下文丢失#

解决方案： 手动捕获指针：在onPointerDown事件中捕获指针并记录起始位置，在onPointerMove事件中检查180毫秒的阈值，并调用move_float_ball_toIPC函数；在onPointerUp事件中根据阈值决定是否扩展窗口。

关键洞察： Windows要求拖动操作必须在原始指针上下文中启动。setTimeout/async会破坏这一规则。

5. 在CSS max-height过渡期间调整窗口大小会导致底部边缘失去任务栏对齐，这是由于多次快速的ResizeObserver→IPC调用造成的#

解决方案： 将细节的CSS过渡改为’opacity 0.8s ease, max-height 0s linear 0.8s’ —— 首先使不透明度逐渐消失而不改变布局，然后在延迟后立即折叠高度，触发一次大小调整。

关键洞察： 影响布局大小的CSS过渡会每帧都触发ResizeObserver；将布局更改推迟到动画完成之后可以防止IPC调用风暴并保持窗口原子重新定位。

6. 当浮动窗口接近边缘时，其展开的面板会溢出屏幕；Windows的窗口调整总是以底部为参考，无论实际位置如何。#

解决方案： 添加了EXPAND_MARGIN=8.0常量，以确保在展开时至少距离边缘8像素；实现了动态锚点检测，阈值设为5像素——当接触顶部边缘时使用顶部锚点，当接触底部边缘时使用底部锚点；添加了SWP_NOCOPYBITS标志以防止复制旧的客户端区域内容。

关键洞察： 展开/折叠状态需要不同的定位策略；锚点应根据当前窗口位置动态确定，而不是硬编码的；Windows默认的调整行为会保留旧内容，导致视觉漂移。

7. 由于两种窗口系统的共存（主窗口全局CSS + 浮动窗口页面），浮动窗口的透明框架会产生伪影，悬停延迟也会造成视觉问题。#

解决方案： 从浮动窗口页面中移除了全局CSS重用，添加了Windows原生区域裁剪功能（SetWindowRgn），并使用统一的矩形计算方法重写了窗口几何状态机，以确保底部边缘始终与工作区域的底部对齐。

关键洞察： 窗口系统的共存会导致样式和容器堆叠冲突；需要完全分离，而不是修补。

8. 图表悬停细节覆盖解决方案导致元素之间的视觉重叠，违反了用户体验要求。#

解决方案： 使用块布局并通过onDetailToggle回调协调窗口高度动画（遵循现有的AccordionToggleDetail模式）。在App.svelte中实现了handleChartDetailToggle功能，并添加了观察者抑制机制。

关键洞察： 在处理原生窗口操作时，应在协调动画过程中抑制被动观察，以防止双重触发和视觉伪影。

9. 当前的成本计算使用了一个本地硬编码的定价表，这与ccusage计算出的值不匹配。#

解决方案： 增加了ccusage CLI适配器（ccusage.rs），在Windows上实现了无提示执行，使用CREATE_NO_WINDOW；实现了基于优先级的机制（先尝试ccusage，如果失败则回退到解析器）；UsagePayload包含了usage_source和usage_warning字段，并在前端显示了明确的警告。

关键洞察： 透明的回退机制非常重要：不要将降级后的结果伪装成正常状态；向用户清楚地说明数据来源和限制。

10. 在非MacOS平台上，由于glassEffect被应用于所有平台而没有原生模糊支持，视觉效果变得过于透明。#

解决方案： 在applyGlass()中添加了isMacOS()检查，强制在非MacOS平台上实现不透明度效果；将默认值改为false并更新了测试快照。

关键洞察： 不同平台之间的差异（macOS具有原生模糊功能，Windows/Linux没有）会导致相同配置下的不同视觉结果；需要在应用程序级别进行平台检测和补偿。

一般问题#

11. 编辑工具的replace_all函数需要先读取文件；跨多个文件的批量路径更新失败，导致“文件尚未读取”的错误。#

解决方案： 使用sed进行跨文件的批量替换（crate::parser → crate::usage::parser用于多个.rs文件）。

关键洞察： 对于跨多个文件的批量路径更新，像sed和grep这样的shell工具比单独调用工具在预读编辑限制下表现更好。

人工智能的局限性### 主要限制#

  • 无法主动识别ccplan中的“连续执行要求”，导致在自动执行阶段的分析不够深入。只有在用户注意到症状后，才会意识到设计缺陷（在第4-6阶段过于迅速）。
  • 对窗口大小调整行为的理解最初是不完整的：没有意识到默认行为会保留并复制旧客户端内容到左上角，从而导致视觉漂移。用户必须指示“内容向上移动”，之后才能添加SWP_NOCOPYBITS。
  • 没有预料到处理多个意图将是一个需求，尽管这是现实世界中的常见场景。理论上的设计考虑忽略了实际的使用模式。
  • 可移植解决方案（随技能目录一起携带）并没有立即被认识到在架构上优于特定项目的可复用组件解决方案。
  • 最初的解决方案（涵盖细节）对于图表悬停在技术上是合理的，但未能满足隐含的用户体验要求（无重叠），这些要求在实施后才被发现。
  • 在实施固定浮动球行为后，对菜单设置的依赖一直未被注意到，直到用户指出了逻辑一致性问题。

一般限制#

  • 在ccplan第0阶段意图校准过程中，需要使用多个AskUserQuestion来消除歧义。无法直接从上下文中推断用户的确切意图。
  • 过度依赖规划文档，而用户真正的担忧（如“无终端弹出窗口”）是更简单的技术细节。当用户需要直接验证约束时，解释MCP的原因需要时间。

学习成果#

关键收获 - 复杂的多阶段工作流程在调用返回大量元数据结果的外部工具时需要明确的状态保存协议。不能依赖隐含的上下文连续性。标签的工具可能会损害工作流的状态。- 工作流程的自动执行阶段要求有定量的深度需求（至少N个发现点，M层分析，至少一次探测）。这样可以避免“快速但浅显”的行为。像“彻底”这样的定性指示对于效率激励来说是不够的。#

  • 持久性保护机制（ECL feature_guard）和自动化检查（guard-check.py）可以防止功能在会话之间下降。内存和上下文是临时的，但磁盘是持久的。测试是理想的，但保护机制为增量开发提供了更轻的安全网。
  • 在Windows GUI应用程序中，为了无声地执行子进程，可以使用std::os::windows::process::CommandExt来创建标志，并设置CREATE_NO_WINDOW（0x0800_0000）以避免控制台窗口弹出。
  • 在调整Windows窗口大小时控制视觉锚点：默认行为会将旧客户端内容复制到左上角，导致视觉漂移。需要使用SWP_NOCOPYBITS来禁用复制并强制完全重绘，同时需要动态检测_virtual_anchor以修复正确的边缘。
  • 影响布局大小的CSS过渡效果会在每一帧触发ResizeObserver。延迟布局变化直到视觉过渡完成（例如，“opacity 0.8s ease, max-height 0s linear 0.8s”）可以防止IPC调用风暴，并保持窗口重新定位的原子性。
  • 将基于波形的执行与验证检查点结合起来（cargo check、cargo test、每个波形后的vitest），可以在大规模重构早期发现问题，从而避免级联故障。
  • 使用目录mtime进行智能缓存无效化比无条件无效化更高效。现有的基础设施（DirectoryStamp）可以用于低成本的变化检测。
  • 在处理原生窗口操作时，在协调动画过程中必须抑制被动观察（ResizeObserver），以防止双重触发和视觉伪影。
  • 透明的回退机制是必不可少的：不要将降级的结果伪装成正常结果。通过数据模型字段（source/warning）和UI警告向用户清晰说明。
  • 提示工程技能应该在提取意图之前分解多意图。在实际应用中，假设单一意图是一种常见的错误模式。第一步：分解多意图可以让每个跟踪独立处理。
  • 采用研究优先的方法并设置质量限制（例如，>1k GitHub stars）比孤立的发明更能产生好的设计。利用成熟的解决方案可以减少盲点。
  • Tauri v2功能系统要求对所有窗口API进行明确的权限声明。缺少权限会导致无声失败（返回null），而不是可见的错误。所需的功能包括：core:window:allow-outer-position、允许-scale-factor和允许-current-monitor，以实现拖动功能。
  • 大文件分解应该保持每个模块的单一责任，同时将测试代码保留在同一文件中。使用987行的Query.rs是可以接受的，因为其中大部分是测试代码。
  • 混合工具策略（并行代理用于复杂逻辑，sed/bash用于批量字符串替换，直接编辑用于针对性修复）可以在代码库内优化重构的正确性和速度。
  • 遵循现有的代码库模式（例如，AccordionToggleDetail回调机制）可以带来可维护的解决方案，而不是引入新的通信范式。
  • 跨平台UI一致性问题通常源于平台API的差异（macOS上的模糊效果与Windows/Linux上没有模糊效果）。需要应用程序级别的平台检测和补偿。
  • 固定的UI行为不应具有运行时配置依赖关系。如果行为是恒定的，代码应该通过去除条件逻辑、提高清晰度和可维护性来反映这一点。

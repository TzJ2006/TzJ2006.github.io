问题与解决方案#

关键问题#

1. macOS非沙箱应用程序无法持久化文件访问权限，历史记录中的旧视频出现“操作不允许”的错误。#

解决方案： 启用App Sandbox权限（com.apple.security.app-sandbox和files.user-selected.read-write）。修复了ensureFileAccess以添加非限定性书签回滚解析。为showVideo/showImage添加了NSOpenPanel重新授权机制。

关键洞察： macOS TCC需要App Sandbox来创建安全级书签。非限定性书签只能解析URL，但不能授予访问权限，导致旧文件无法访问。需要结合沙箱和降级路径来解决此问题。

2. torch 2.7.1和torchvision 0.21.0的ABI不兼容，导致注册torchvision::nms操作符失败。#

解决方案： 将torchvision升级到0.22.1+cu126，以匹配torch 2.7.1版本。

关键洞察： 在编译torchvision中的自定义操作符时，会链接到torch的C++ API。不匹配的版本可能导致操作符注册失败。torchvision 0.21.0对应于torch 2.6.x，不能与torch 2.7.x一起使用。

3. curobo的预编译.so文件与当前PyTorch ABI不兼容，JIT编译无法找到cuda_runtime.h。#

解决方案： 删除旧的.so文件，安装ninja编译器，设置CUDA_HOME为/path/to/conda.env，并将CPATH设置为/path/to/targets/x86_64-linux/include。使用pip install -e重新编译。

关键洞察： 升级PyTorch版本后，需要重新编译C++扩展。通过pip在conda环境中安装的nvidia包将CUDA头文件放置在非标准路径(targets/x86_64-linux/include/)中，环境变量需要正确配置以使JIT编译能够找到它们。

一般问题#

4. GPU监控工具的输出宽度过大且过程显示重复，难以阅读。#

解决方案： 将输出限制为80列，聚合每个过程的GPU，并过滤掉多进程等工作过程的噪声。

关键洞察： 原始实现按设备分组GPU，导致在多台GPU上重复显示多个GPU的过程。需要聚合过程并显示GPU列中使用的所有卡片号。

经验教训#

关键经验#

  • macOS TCC权限机制：非沙箱应用程序无法创建安全范围的书签；只能创建非安全范围的书签（仅解析无需授予访问权的URL），导致应用程序重启后历史文件访问失效。必须启用应用沙箱以保持文件访问权限。
  • PyTorch C++扩展ABI兼容性：torchvision中的自定义操作符在编译时会链接到torch的C++ API。版本必须完全匹配（torchvision 0.21→torch 2.6，torchvision 0.22→torch 2.7）。版本不匹配可能导致操作符注册失败。
  • Condna环境CUDA工具链配置：通过pip安装的Nvidia软件包将CUDA头文件放置在targets/x86_64-linux/include/中，而不是标准的include/目录。对C++扩展的JIT编译需要设置CUDA_HOME指向conda环境，并设置CPATH包含目标路径。
  • 防御性编程原则：沙箱和重新授权相结合比单一解决方案更为稳健。沙箱解决了根本原因（持久权限问题），而NSOpenPanel提供了降级途径（临时授权）。结合使用可以确保用户始终能够访问文件。

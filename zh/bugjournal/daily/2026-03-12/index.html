问题与解决方案#

关键问题#

1. macOS非沙箱应用程序无法持久化文件访问权限。历史记录中的旧视频会显示“操作不允许”的错误。#

**解决方案：**启用了App Sandbox权限（com.apple.security.app-sandbox和files.user-selected.read-write）。修复了ensureFileAccess函数，添加了对非安全范围书签的回退解析。为showVideo/showImage添加了NSOpenPanel重新授权机制。

**关键洞察：**macOS TCC需要App Sandbox来创建安全范围的书签。非安全范围的书签只能解析URL而不授予访问权限，从而导致无法打开旧文件。需要结合沙箱降级路径来解决此问题。

2. torch 2.7.1和torchvision 0.21.0不兼容，导致torchvision::nms的操作员注册失败。#

**解决方案：**将torchvision升级到0.22.1+cu126，以匹配torch 2.7.1。

**关键洞察：**torchvision中的自定义操作符在编译时会链接到torch的C++ API。不兼容的版本会导致操作员注册失败。torchvision 0.21.0与torch 2.6.x兼容，而不是2.7.x。

3. 预编译的curobo.so文件与当前PyTorch ABI不兼容。JIT编译无法找到cuda_runtime.h。#

**解决方案：**删除了旧的.so文件，安装了ninja编译器，设置CUDA_HOME为/path/to/conda.env，并将CPATH设置为/path/to/targets/x86_64-linux/include/。使用pip install -e重新编译。

**关键洞察：**在升级PyTorch版本后，需要重新编译C++扩展。通过conda环境安装的Nvidia包会将CUDA头文件放在非标准路径（targets/x86_64-linux/include/）中，因此需要正确设置环境变量以便JIT编译能够找到它们。

一般问题#

4. GPU监控工具的输出过于宽泛，多个进程被重复显示，难以阅读。#

**解决方案：**将输出限制为80列，按进程聚合GPU列表，并过滤掉多处理工作线程和其他子进程的噪声。

**关键洞察：**原始实现将GPU分组在一起，导致多GPU进程在多个GPU上多次出现。有必要按进程聚合，并在GPU列中显示所有使用的卡片号。

经验教训#

关键经验#

  • macOS TCC权限机制：非沙盒应用程序无法创建安全范围的书签。它们只能创建非安全范围的书签（仅解析URL而不授予访问权限），导致应用程序重启后历史文件访问失败。必须启用应用沙盒才能持久保存文件访问权限。
  • PyTorch C++扩展ABI兼容性：torchvision中的自定义操作符在编译时会链接到torch的C++ API。版本必须完全匹配（torchvision 0.21 → torch 2.6，torchvision 0.22 → torch 2.7）。不兼容的版本会导致操作符注册失败。
  • Condna环境CUDA工具链配置：通过pip安装的Nvidia包将CUDA头文件放置在targets/x86_64-linux/include/而不是标准的include/目录。C++扩展的JIT编译需要将CUDA_HOME设置为conda环境的路径，并将CPATH包含目标路径。
  • 防御性编程原则：沙盒和重新授权相结合比单一解决方案更为稳健。沙盒解决了根本原因（持久的权限问题），而NSOpenPanel提供了备选方案（临时授权）。两者结合使用可以确保用户始终能够访问文件。

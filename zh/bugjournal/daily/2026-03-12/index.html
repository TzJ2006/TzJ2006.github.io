Problems & Solutions#

Critical Issues#

1. macOS非沙盒应用无法持久化文件访问权限,历史记录中旧视频报Operation not permitted错误#

Solution: 启用App Sandbox权限(com.apple.security.app-sandbox和files.user-selected.read-write),修复ensureFileAccess添加non-scoped bookmark回退解析,showVideo/showImage添加NSOpenPanel重新授权机制

Key Insight: macOS TCC机制要求App Sandbox才能创建security-scoped bookmark。Non-scoped bookmark只能解析URL但不授予访问权限,导致旧文件无法打开。需要沙盒+降级路径的组合方案

2. torch 2.7.1与torchvision 0.21.0 ABI不兼容导致torchvision::nms算子注册失败#

Solution: 升级torchvision到0.22.1+cu126以匹配torch 2.7.1版本

Key Insight: torchvision的自定义算子编译时链接torch的C++ API,版本不匹配会导致算子注册接口失效。torchvision 0.21.0对应torch 2.6.x,不能用于torch 2.7.x

3. curobo预编译.so文件与当前torch版本ABI不兼容,JIT编译又找不到cuda_runtime.h#

Solution: 删除旧.so文件,安装ninja编译工具,设置CUDA_HOME=/path/to/conda/env和CPATH=/path/to/targets/x86_64-linux/include,通过pip install -e .重新编译

Key Insight: PyTorch版本升级后C++扩展需要重新编译。conda环境中pip安装的nvidia包将CUDA headers放在非标准路径(targets/x86_64-linux/include/),需要正确配置环境变量才能让JIT编译找到

General Issues#

4. GPU监控工具输出宽度过大且进程重复显示,难以阅读#

Solution: 限制输出为80列,按进程聚合GPU列表,过滤multiprocessing worker等子进程噪声

Key Insight: 原始实现按GPU分组导致一个多卡进程在多个GPU下重复出现。需要按进程聚合并在GPU列显示所有使用的卡号

AI Limitations#

General Limitations#

  • 未能主动解释技术背景(macOS TCC/沙盒机制),需要人类追问才给出完整上下文。AI应在提出解决方案时主动说明’为什么'
  • 误解GPU监控需求,将进程-GPU的多对多关系简化为一对多聚合,导致信息丢失。对复杂数据关系的可视化需求理解不准确
  • 在checkpoint路径问题分析时,开始过度深入代码细节(连续Read操作)而未先与用户确认问题范围。应该先high-level理解参数映射逻辑

Learnings#

Key Learnings#

  • macOS TCC权限机制: 非沙盒应用无法创建security-scoped bookmark,只能创建non-scoped bookmark(仅解析URL不授予权限),导致历史文件访问在app重启后失效。必须启用App Sandbox才能持久化文件访问权限
  • PyTorch C++扩展ABI兼容性: torchvision的自定义算子(如torchvision::nms)编译时链接torch的C++ API,版本必须精确匹配(torchvision 0.21→torch 2.6, torchvision 0.22→torch 2.7)。版本不匹配会导致算子注册失败
  • conda环境CUDA工具链配置: pip安装的nvidia包将CUDA headers放在targets/x86_64-linux/include/而非标准的include/。JIT编译C++扩展时需要设置CUDA_HOME指向conda env,CPATH包含targets路径
  • 防御性编程原则: 沙盒+重新授权的组合方案比单一方案更robust。沙盒解决根因(持久化权限),NSOpenPanel提供降级路径(临时授权),两者结合保证用户总能访问文件

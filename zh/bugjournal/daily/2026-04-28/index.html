问题与解决方案#

关键问题#

1. 当悬停在图表条上时，由于细节面板扩展导致窗口底部从任务栏偏移#

**解决方案：**引入了SCROLL_THRESHOLD_CAP = 500的常量；修改了resolveScrollThresholdHeight，使其返回基于比例的计算和阈值之间的最小值；现在内容会在固定高度的窗口内滚动，而不是调整窗口大小

**关键洞察：**用户指定的固定阈值（500px）可以在所有显示器尺寸下提供可预测的行为，而现有的滚动机制可以处理溢出情况；基于比例的方法（0.75 * 工作区域）在较大显示器上过于宽松

2. 最初的v0.12.1版本在cargo fmt和clippy的CI检查中失败#

**解决方案：**对window.rs应用cargo fmt，以修复导入顺序和行尾填充问题；删除了config_rs中的不必要的return语句；删除了旧标签并重新发布以修复问题

**关键洞察：**预发布的本地验证应包括完整的CI检查套件（fmt、带有-D警告的clippy）；rustfmt和clippy规则在CI流水线中是不可协商的

一般问题#

3. 尝试更新版本号时，编辑工具出现“文件尚未读取”的错误#

**解决方案：**在尝试编辑之前，先读取所有三个版本文件（package.json、Cargo.toml、tauri.conf.json）；工具要求在写入之前先进行读取操作

**关键洞察：**Claude Code Edit工具有严格的读前写要求；工作流程必须是：读取→编辑，而不是直接编辑

经验教训#

关键经验#

  • 预发布验证必须包括在本地完整的CI检查套件中；cargo fmt –check和cargo clippy – -D警告可以防止标签变动并保持干净的发布历史记录
  • 硬编码的UI阈值比基于比例的计算更可预测，适用于跨设备的一致性；500px限制确保了在1080p、1440p和4K显示器上的统一行为
  • 公共发布的git标签管理需要删除/重新创建标签的流程（git tag -d、git push –delete、重标签、推送），而不是强制推送或修改，以保持完整性

实际经验#

  • MCP工具（mcp__hex-line__edit_file）和标准编辑工具有不同的读取要求；MCP工具在读取操作中接受edit_ready标志，以便简化工作流程

问题与解决方案#

关键问题#

1. Linux/Wayland 上 Tauri 主窗口 set_position() 被合成器完全忽略，窗口出现在屏幕中央#

解决方案: 在 run() 入口检测 WAYLAND_DISPLAY 环境变量，若在 Wayland 会话下强制设置 GDK_BACKEND=x11，使应用通过 XWayland 运行

关键洞察: Wayland 协议设计层面禁止客户端控制窗口绝对位置，任何重试策略都无效；必须回退到 X11 协议层

2. 浮球展开布局以屏幕边界而非球本身位置为锚点，导致点击展开后球视觉跳位#

解决方案: 修改 expanded_rect_from_ball 为 ball-anchored 模式：Left 锚时展开 x = ball_rect.x，Right 锚时展开 x = ball_rect.right - expanded_width，配合 clamped_y 保持 y 稳定

关键洞察: 展开窗口的参考点应锚定在球本身位置，而非屏幕安全区边界，才能保证展开前后球的视觉位置一致

3. ssh_remote.rs 中 host_cache_dir() 使用 assert!() 在路径校验失败时 panic 崩溃整个程序#

解决方案: 将函数签名改为 Result<PathBuf, String>，所有调用链同步传播 Result，调用点用 unwrap_or_default() 处理

关键洞察: assert!() 在生产环境等同于定时炸弹；接受外部输入的路径上必须用 Result 传播错误

4. Linux GTK 窗口设置 resizable(false) 后，编程式 set_size() 被 WM 拒绝，导致浮球无法收起#

解决方案: 在 Linux 分支中，set_size 前先 set_resizable(true) 临时解锁，操作完成后再 set_resizable(false) 重新锁定

关键洞察: resizable(false) 的 WM hint 同时阻止用户拖拽和编程式调用，不限于用户交互

5. FloatBallState 不存储窗口几何，每次从 WM 读取 outer_position/outer_size，在展开/收起时序中可能读到过时值导致布局错误#

解决方案: 添加 FloatBallState.last_rect，创建 apply_and_store_rect 辅助函数同步写窗口和记录状态；新增 get_float_ball_position 命令供前端使用

关键洞察: 应用层维护权威几何状态，完全绕过 WM 读取的时序不确定性

6. sync_ssh_host 在持有 RwLock 读锁时跨越长达数秒的 SSH I/O await 点，阻塞所有读者#

解决方案: 给 SshCacheManager 添加 #[derive(Clone)]，在 await 前克隆 manager，提前释放锁

关键洞察: 锁跨异步边界是 Rust 异步编程中常见隐患；解法是在 await 前 Clone 出所需数据

7. provider 参数未校验可被用于路径穿越攻击（如传入 ‘../../../etc’ 构造恶意路径）#

解决方案: 添加 validate_provider() 白名单函数，仅允许 ‘claude’ 和 ‘codex’，其他值返回 Err

关键洞察: 任何拼接到文件路径的用户输入都必须在系统边界做白名单校验，白名单比黑名单更可靠

8. load_cached_records 对所有 I/O 错误（含权限拒绝）静默返回空 Vec，掩盖真实错误#

解决方案: 改为返回 Result，明确区分 ErrorKind::NotFound（合法，返回 Ok(vec![])）与其他 I/O 错误（返回 Err）

关键洞察: 文件不存在和无法读取是本质不同的状态，混为一谈会使调试极其困难

9. ssh_command 在 SSH 退出码非零但有部分 stdout 时视为成功，接受错误结果#

解决方案: 严格先检查 exit code，仅在 success() 时接受 stdout；非零退出时记录 stderr 到日志

关键洞察: partial stdout + non-zero exit 是 SSH 错误的常见模式，不能以 stdout 非空作为成功标准

10. Svelte 5 组件中同时使用 onMount 和 $effect 触发数据请求，导致挂载时双重请求#

解决方案: 删除 onMount 调用，保留 $effect（Svelte 5 中 $effect 在挂载时自动执行一次）

关键洞察: Svelte 5 的 $effect 与 Svelte 4 的响应式语句不同，挂载时立即执行，不需要配合 onMount

11. FloatBall 组件 onMount 内异步注册 Tauri 事件监听器存在竞态：组件可能在 listen() 解决前被销毁，导致清理函数不被调用，造成内存泄漏#

解决方案: 引入 destroyed 布尔标志，Promise 解决时若已标记销毁则立即调用 unlisten()

关键洞察: 前端组件中所有异步获取清理句柄的模式都需要考虑’组件比 Promise 更短命’的情况

一般问题#

12. sync.py pull 时 6 个目录报 ‘directory not found’，用户误认为同步失败#

解决方案: 确认根因为这些子目录从未 push 到远端，远端不存在。应在 sync 逻辑中将远端不存在视为成功（无数据可拉）而非错误

关键洞察: 远端目录不存在不等于同步失败，需区分’远端有数据但拉取失败’与’远端本就没有此目录’

今日收获#

核心收获#

  • Wayland 合成器在协议层面禁止客户端设置绝对窗口位置，任何 set_position() 重试策略都无效；正确解法是检测 WAYLAND_DISPLAY 后强制 GDK_BACKEND=x11 回退到 XWayland
  • Rust 生产代码中 assert!() 和 unwrap() 在接受外部输入（文件路径、用户参数、SSH 输出）的路径上都是潜在崩溃点；统一模式是将函数签名改为 Result 并用 ? 传播，调用边界用 unwrap_or_default() 处理
  • GTK 的 resizable(false) WM hint 同时阻止用户拖拽和编程式 set_size()；对需要动态调整尺寸的透明浮窗，必须在 resize 前临时解锁 resizable
  • 透明叠加窗口的几何状态应由应用层权威管理（last_rect），而非每次从 WM 重新读取；WM 的 outer_position/outer_size 在 resize 后可能存在时序延迟，展开/收起时序中读到过时值会导致布局错误
  • 浮窗展开布局应以球本身为锚点（ball-anchored），而非以屏幕安全区边界为锚点，才能保证展开前后球的视觉位置一致，用户感知无跳动
  • Rust 异步代码中锁跨 await 点是难以发现的性能/死锁隐患；解法是在 await 前 Clone 或 Arc::clone 出所需数据，提前释放锁
  • 任何拼接到文件路径的用户输入都必须在系统边界做白名单校验；‘只允许已知值’比’拒绝危险字符’的防护更可靠，可有效避免路径穿越攻击
  • Svelte 5 中 $effect 在组件挂载时自动执行一次，与 onMount 同用会导致双重请求；前端组件异步获取清理句柄时，需引入 destroyed 标志处理’组件比 Promise 短命’的竞态场景

实践收获#

  • git reset –soft 是整理 commit 历史的高效方法：无需交互式 rebase，可将任意数量的 commit 压缩为一个语义提交，适合 CI 环境和自动化工具
  • rclone sync 配置中新增目录结构后，若从未 push 数据远端不会自动创建这些目录；pull 时应区分’远端不存在’（正常，视为成功）和’拉取失败’（错误），避免误报

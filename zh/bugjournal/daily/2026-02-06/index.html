问题与解决方案#

关键问题#

1. 公众用户无法将结果推送到仓库，但需要贡献基准测试结果。#

**解决方案：**设计了三层架构：用户在中继侧提交 → 中继验证并将结果转发到repository_dispatch → GitHub Actions进行队列/拉取/发布。中继持有写入令牌，用户没有访问仓库的权利。

**关键洞察：**众包数据管道需要不受信任的拉取路径；带有验证工作流程的GitHub Actions可以提供审计跟踪和滥用控制，同时不授予贡献者写入权限。

2. 条形图显示倒序长度（较高的分数对应的条形较短），尽管CSV中的数值是正确的。#

**解决方案：**根本原因是二进制编码的图表数据与CDN版本不匹配。修复措施包括：(1)调整Plotly JS以匹配安装的Python版本，(2)将所有图表x/y/文本转换为普通Python列表而不是Pandas Series，(3)在加载时强制flops_gflops为数值类型，(4)设置yaxis rangemode=‘tozero’。

**关键洞察：**客户端绘图库中的二进制序列化格式可能会在CDN版本变化时导致无声的渲染错误；显式的列表转换和版本固定确保了可视化的准确性。

3. 在没有用户同意的情况下自动上传基准测试引发了伦理问题。#

**解决方案：**在保存运行后添加了交互式提示（默认禁用），为自动化操作添加了明确的–upload/–no-upload标志，要求在提示之前配置中继URL，允许非交互式运行跳过提示，除非指定了–upload选项。

**关键洞察：**遥测和数据收集需要明确的选择和透明的披露；禁用提示并要求使用标志可以尊重用户的偏好。

4. HOSTNAME_RE正则表达式过于宽泛，导致有效数据中的时间戳和版本字符串被损坏。#

**解决方案：**缩小了HOSTNAME_RE的匹配范围，从匹配任何点分隔的令牌改为要求TLD后缀（.[A-Za-z]{2,}\b），以防止误判为'13.0’或'2026-02-07’模式。

**关键洞察：**隐私净化规则需要精确的边界；过于激进的正则表达式模式可能会损坏与PII具有相似表面结构的有效技术元数据。

一般问题#

5. 趋势图没有被彻底移除，留下了导致AttributeError错误的过时方法调用。#

**解决方案：**为create_trend_chart()函数添加了兼容性后备措施，返回空图表，以便旧的或过时的调用路径可以优雅地失败，而不会引起AttributeError错误。

**关键洞察：**通过防御性编程防止在重构过程中删除功能时出现的脆弱故障；兼容性层平滑了迁移过程。

经验教训#

关键要点#

  • 众包数据管道需要一个三层架构（不可信提交 → 可信网关 → 受保护存储），并配备验证/审计追踪功能，以防止滥用，同时不授予写入权限。
  • 伦理数据收集需要明确同意，不应默认进行；自动化场景需要明确的标志，而不是依赖交互式检测。
  • 客户端可视化错误通常源于序列化格式不匹配，而非数据正确性问题；明确的数据列表转换和版本固定可以防止无声的渲染失败。
  • 隐私清理规则需要精确的边界条件；过于激进的正则表达式模式可能会损坏与PII具有相似表面语法的合法技术元数据，从而导致PII泄露。

实用经验#

  • 功能移除重构应包括逐步迁移的兼容性存根；非操作性的回退可以防止因过时调用路径导致的脆弱的AttributeError崩溃。

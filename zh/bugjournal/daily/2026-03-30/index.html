问题与解决方案#

关键问题#

1. Teddy gene-only ARI（0.186）不如 PCA（0.255），Foundation Model 在聚类任务上未能超越简单基线#

解决方案: 分析根本原因：Teddy 以疾病分类为优化目标而非方差最大化；1024d 高维空间使 KMeans 表现变差；训练数据为 dissociated scRNA-seq，与空间转录组分布不同

关键洞察: 基础模型 embedding 的"泛化表示"与"聚类适配性"存在根本张力：PCA 直接最大化数据方差，天然适配 KMeans；而 Foundation Model 学习疾病区分性表示，在下游聚类任务上存在系统性迁移损失

一般问题#

2. evaluation_planner.py 对 151676 的 staig_gene_feat 缓存检查误报，导致 Phase 2 以 sys.exit(1) 终止#

解决方案: 修改 has_staig_gene_feat 检查逻辑，动态解析切片配置中的实际 top_genes 数量（151676 使用 5000 而非默认 3000），正确识别缓存存在

关键洞察: 切片特定配置（top_genes 来自 staig_utils.py 的 num_gene 字段）与 planner 的硬编码默认值不一致，这类"配置隐式分叉"容易造成难以察觉的假阴性检查

3. STAIG 融合评估中 151676 切片出现 model collapse，无法产生有效聚类结果#

解决方案: 跳过该切片，以 10/11 切片的平均值报告结果（ARI=0.495）

关键洞察: 151676 是已知的 STAIG 困难切片，其 STAIG collapse 与 gene encoder 无关，是该切片数据特性导致的系统性问题

4. HPC 上无法构建 TokenMonitor（缺少 Rust、webkit2gtk4.1-devel 等 Tauri 依赖，无 sudo 权限）#

解决方案: 建议三条路径：在本地有权限机器构建后拷贝二进制、请 HPC 管理员安装依赖、通过 GitHub Actions CI 构建 Linux 发布包

关键洞察: Tauri 应用依赖 GTK/WebKit 系统库，这类 GUI 依赖在 HPC 环境中通常不可用，且 module 系统也不提供此类库；HPC 本质上不是 GUI 应用的目标部署环境

今日收获#

核心收获#

  • STAIG 融合框架对 gene encoder 具有高度鲁棒性：Teddy 1024d（ARI=0.495）与 PCA 50d（ARI=0.497）在 STAIG 框架下性能几乎相同，说明 STAIG 的空间图对比学习机制主导了聚类性能，gene embedding 质量的影响被大幅稀释
  • Foundation Model 在跨域下游任务上的系统性迁移损失：Teddy 在疾病分类上 SOTA（ACC=0.72），但在空间转录组聚类上不如 PCA（0.186 vs 0.255），根本原因是优化目标（区分性 vs 方差最大化）的不匹配
  • HVG1500（原始高变基因直接拼接，无降维）在 151673 单切片上 ARI=0.330，优于所有 Foundation Model；这提示高维原始特征在某些切片上可能因保留更多局部方差信息而受益

实践收获#

  • 切片特定配置（如 staig num_gene）与 pipeline 全局默认值之间的隐式分叉是一类高风险 bug，应在缓存检查逻辑中动态解析而非硬编码

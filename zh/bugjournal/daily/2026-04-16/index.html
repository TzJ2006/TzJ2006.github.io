问题与解决方案#

关键问题#

1. revalidate_demos.py中的第96行存在错误：obj_orientations四元数数据错误地写到了obj_positions字典中，导致数据丢失#

解决方案： 将第96行的代码从’obj_positions[key[4:-5]]‘修改为’obj_orientations[key[4:-5]]’，以正确地将四元数写入字典

关键洞察： 数据加载代码中的复制粘贴错误可能会默默地破坏多字段数据结构

2. determine_counts_toward_target函数在字段存在时返回缓存的counts_toward_target值，导致在新的配额规则下无法重新评估#

解决方案： 在重新验证过程中调用determine_counts_toward_target之前，清除counts_toward_target和quota_reason字段

关键洞察： 需要重新计算时，检查预计算值的函数需要显式地清除缓存

一般问题#

3. NumpyEncoder类在4个不同的文件中重复存在（resegment_recovery_demos.py、3_mimicgen_recovery_augment.py、2_collect_recovery_demos.py、revalidate_demos.py）#

解决方案： 将NumpyEncoder移动到共享的scriptUtils.py文件中，并更新所有消费者以从该文件导入

关键洞察： 共享的JSON编码逻辑应该放在共同的实用程序中，以防止漂移和维护负担

4. 对正确的CLI语法存在初步的不确定性——仓库README提到了/plugin命令，但实际上CLI使用的是’claude plugin’结构#

解决方案： 在执行命令之前，使用’claude –help’和’claude plugin –help’来验证实际的CLI功能，确认正确的语法是’claude plugin marketplace add’和’claude plugin install’

关键洞察： 外部文档可能与实际的本地CLI语法不符；在执行安装命令之前，务必先查看当地的帮助文档

5. tag_npz_validation和save_manifest函数在revalidate_demos.py和2_collect_recovery_demos.py之间重复存在#

解决方案： 将两个函数移动到共享的recovery_collection_validation.py模块中，并更新消费者以从该模块导入

关键洞察： 验证和持久化逻辑应该集中在验证框架模块中

学习成果#

关键学习点- 错误恢复演示验证支持四种配额规则（仅成功、仅重放、仅可扩展、重放与可扩展），这些规则具有不同的严格程度；更改规则需要批量重新验证现有数据。#

  • agent-skills提供了20种生产级工程技能，这些技能分布在六个开发阶段（定义：规范/细化；计划：架构/测试驱动开发；构建：增量编码；验证：测试；评审：代码审查；部署）：这些技能代表了成熟的工程工作流程。
  • 跨多个文件进行系统化的代码去重可以显著减少代码行数（本例中减少了80行），同时提高可维护性。
  • 需要缓存计算值的函数必须在重新计算时显式地使缓存失效；determinate_counts_toward_target功能会检查是否存在pre-existing counts_toward_target字段。
  • Claude Code插件安装遵循市场优先模式：使用’claude plugin marketplace add ‘添加插件源，然后使用’claude plugin install @‘全局安装插件。

实际学习成果#

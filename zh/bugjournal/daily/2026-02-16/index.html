人工智能的关键局限性#

  • “完成”的定义过于乐观——单元测试可能等同于运行准备状态，而忽略了在VLA集成中实际进行模型检查点和API集成测试的需求，而不仅仅是模拟测试。
  • 最初，在openpi配置中没有注意到使用repack_transform来重命名键，导致对输入键格式的错误假设。这个问题是在查看了LiberoInputs的源代码和make_libero示例之后才发现的。
  • 在编写VLA集成代码时，并没有预料到需要包含图像这一需求；这个问题是通过运行时错误才发现的。此外，对于VLA策略输入的格式要求，也缺乏事先的知识检查。
  • 生成了一个维度消除计划，但用户拒绝执行该计划，因此放弃了它。人工智能不清楚为什么该计划被拒绝，也不明白用户希望如何继续操作。

经验教训#

关键教训 - 嵌入维度瓶颈可能导致模型崩溃：STAIG的64维嵌入可以将1536维视觉特征压缩96%，产生的嵌入的标准差仅为0.12，几乎无法区分。空间细化可以消除一些簇，而多数投票可以解决其他问题。其他融合方法（如注意力机制或qformer：256-512维）不会出现这些问题，这表明64维嵌入本身就不够充分。- VLA端到端集成的关键要求：确保政策观察输入格式正确，特别是在摄像头图像传输部分。在state_extractor.extract()函数中必须明确使用include_images=True参数，并为所有依赖摄像头的政策提供camera_names信息（如VLA和VisuoMotor BC）。#

  • 在集成第三方机器学习库（如openpi和robomimic）时，必须使用inspect.signature()检查实际的API签名，并阅读来自源仓库的示例代码。文档往往缺乏完整性或已经过时。openpi库的create_trained_policy()提供了高级的封装功能，但手动使用低级API可能会容易出错。
  • Pi0（LIBERO检查点）在robosuite的PickPlace任务中的泛化能力较差（失败率为100%）。主要的失败模式包括joint_limit_approach（47%）和grasp_wrong_pose（30%）。这表明VLA的跨任务泛化仍然存在挑战，但这些失败数据可以为基准测试提供有价值的自然错误数据。
  • 捕捉自然错误比注入错误更有效：50次迭代产生了150种场景，而注入方法只产生了3种场景。自然错误模式更加多样化和真实。
  • 基准系统中的插件式架构模式：检测器、分类器和注入器都使用基本的ABC+注册表模式来实现可扩展性。错误分类系统使用CLASSIFIER_REGISTRY注册了19个分类器，使得无需修改核心逻辑即可添加新的错误类型。
  • 文档验证需要检查实际的代码注册表和枚举值，而不仅仅是阅读文档字符串或之前的文档。DETECTOR_REGISTRY、CLASSIFIER_REGISTRY和ErrorType枚举是真正的参考标准。通过系统的搜索，可以避免残余错误。
  • 不同Conda环境之间的通信架构：VLA服务器使用TCP+pickle协议来分离模型推理（openpi/openpi05环境）和基准测试框架（mimicgen_env）。长度前缀的帧可以防止消息碎片化，而动作分块可以减少推理调用的次数。由于存在冲突的依赖关系，环境隔离对于VLA/机器人项目至关重要。
  • 观测数据的预处理必须能够优雅地处理各种输入格式：StateExtractor（内部框架）、raw env.step()（robosuite）以及模型特定的预处理（VLA服务器）。一个强大的键查找机制加上回退机制可以防止无声失败。观测数据的字典格式因来源不同而有所不同，因此需要统一的预处理层。
  • 空间多数投票细化（最近邻半径为15）可能会放大而不是修复聚类错误，特别是当底层嵌入已经压缩的情况下。细化应该基于嵌入质量指标（例如，如果轮廓分数低于某个阈值，则应禁用细化）。
  • 双VLM后端设计（Gemini API + Claude Code CLI）提供了灵活性：API适用于批量自动化操作，而CLI适用于交互式深度分析和调试。两者相辅相成，可以满足不同的使用需求。
  • 为了生成自然错误，VLA集成需要进行架构调整：（1）在不同Conda环境中采用策略服务器模式；（2）支持环境包装器中的摄像头观测；（3）预处理观测数据（将四元数转换为角度坐标，BGR转换为RGB，进行动作分块）；（4）双迭代模式（合成注入和自然捕获）。这与演示回放中使用的合成扰动方法非常不同。
  • 系统的代码库探索策略（从基础设施检查开始，到模型文件位置，再到配置参数的确认，最后到用户决策确认）可以有效支持复杂的多步骤项目规划，避免忽视关键依赖关系。
  • 在数据结构设计中，向后兼容性是一个重要的考虑因素：通过引入可选的、优雅的降级新conversation_summaries字段，确保了旧报告仍然可以正常使用。这种方法适用于不断演变的数据格式。
  • 在错误分类系统中，有24种错误类型，但在实际VLA迭代中只触发了5种：joint_limit_approach、grasp_wrong_pose、misalignment_pre_grasp、overshoot和premature_release。这表明某些错误类型需要特定的场景才能显现出来。

实践经验#

  • 使用GPU进行批量处理以进行可视化（5个GPU，20个视频，零失败）证明了并行基础设施的价值。subprocess.Popen + GPU分配模式非常适合并行任务。
  • 在Python中处理虚假值的微妙之处：0 or default会被评估为default，因为0是一个虚假值，这会导致以0为索引的计数器出现错误。PrematureReleaseClassifier在步骤0开始捕获时总是返回0。解决方案是明确使用is None来检查可选的0索引值。

问题与解决方案#

1. GTK bottom-up尺寸协商导致FloatBall窗口恒为400x400，Tauri set_size()完全无效 [TokenMonitor] (2026-04-04)#

解决方案: 通过with_webview()回调在GTK主线程调用widget.set_size_request()突破WebKitGTK natural size约束；移除set_resizable切换（with_webview是异步派发，后续API调用会在回调执行前入队会锁死旧尺寸），Linux永久保持resizable

2. Linux透明无边框窗口动态resize导致WM系统性+305px/+113px位置偏移，probe/corrective机制产生可见闪烁，WebView与native window尺寸不一致 [TokenMonitor] (2026-04-05)#

解决方案: 架构重构为固定expanded尺寸窗口+GDK input_shape_combine_region：collapsed时仅球区域（56×56）接受事件，展开及拖拽时全窗口接受，彻底消除resize操作，删除150行补丁代码

3. Linux/Wayland上Tauri主窗口set_position()被合成器完全忽略，窗口出现在屏幕中央 [TokenMonitor] (2026-04-02)#

解决方案: 在run()入口检测WAYLAND_DISPLAY后强制设置GDK_BACKEND=x11，使应用通过XWayland运行，绕过Wayland协议层面禁止客户端设置绝对位置的根本限制

4. ssh_remote.rs中assert!()在路径校验失败时panic崩溃整个程序，provider参数未校验可用于路径穿越攻击 [TokenMonitor] (2026-04-02)#

5. sync_ssh_host在持有RwLock读锁时跨越长达数秒的SSH I/O await点，阻塞所有读者 [TokenMonitor] (2026-04-02)#

解决方案: 给SshCacheManager添加#[derive(Clone)]，在await前克隆manager提前释放锁

学习收获#

架构 (architecture)#

  • Linux透明无边框Tauri/WebKitGTK浮窗的正确架构是固定大尺寸native window+内部纯CSS展开/收起，不依赖频繁resize外层窗口。GTK bottom-up协商取max(window_requested, widget_natural_size)，必须通过with_webview()回调调用widget.set_size_request()才能突破WebKitGTK natural size约束；WM、GTK、WebView内容区三者尺寸/位置不会自动同步，这是平台约束而非bug
  • Wayland协议在设计层面禁止客户端控制窗口绝对位置，任何set_position()重试策略都无效；正确解法是检测WAYLAND_DISPLAY后强制GDK_BACKEND=x11回退到XWayland。X11 WM将无装饰窗口强制夹在可见区域内，边缘偏移量需平台差异化（Linux edge_offset=0，macOS/Windows可半出屏）
  • 激进简化往往比渐进修补更有效：Thin Adapter+MCP Server+Skills三层模式替代3170行专有路由逻辑，净删2823行并获得跨CLI通用性。多CLI系统应利用约定文件（CLAUDE.md/AGENTS.md/GEMINI.md）而非运行时注入，统一history ring buffer比适配各CLI session机制更可靠
  • Claude CLI -p非交互模式的MCP完整配置需两步：(1)显式传–mcp-config加载项目级工具；(2)添加–permission-mode bypassPermissions绕过权限确认。–allowedTools通配符在此模式下因与MCP server初始化存在竞争条件而不可靠，bypassPermissions是后端服务调用的正确模式

调试 (debugging)#

  • 运行时日志是底层系统调试的根本手段：当出现GTK、WM、OS等底层系统行为时，应优先通过日志/探针实证确认假设（如backend.log显示+305px系统性偏移）而非从代码推断。多轮修复失败后应主动识别’继续修补不能解决架构性问题’的边界，停止局部修复转向根因总结
  • ‘代码已实现参数接口’不等于’参数已被传递’：integration layer是最常见的接线断裂点，验证时要检查调用侧而非只看实现侧。Python中await同步方法产生TypeError，若被宽泛except Exception静默捕获，会导致功能静默降级而非报错，极难排查
  • 调试嵌套进程CLI工具（主进程→Claude CLI→MCP server）时，先记录实际传递的命令行参数再逐一隔离验证参数效果，比直接猜测参数行为更高效。pipeline的skip_completed机制无法检测内容无效（gt_labels全为-1）的文件，需主动校验内容合法性

工具 (tools)#

  • ccplan技能的发散探索（多方案）+Critic对抗审查+可行性探测流程能有效将复杂平台架构问题转化为可执行计划。本周Critic阶段共预先发现Wayland兼容性、lark-oapi线程阻塞、ThoughtStore非单例、4个隐藏代码依赖等高价值问题，ROI极高，是处理不确定性高任务的关键工程实践
  • GDK input_shape_combine_region在X11层拦截事件，setPointerCapture不能越过它；拖拽场景需在Rust侧临时扩展input shape到全窗口。GTK的resizable(false) WM hint同时阻止用户拖拽和编程式set_size()，需要动态调整尺寸时必须临时解锁
  • Rust生产代码中assert!()和unwrap()在接受外部输入的路径上是潜在崩溃点，统一改为Result并用?传播。锁跨await点是异步编程中常见隐患，解法是在await前Clone出所需数据提前释放锁。任何拼接到文件路径的用户输入必须做白名单校验（比黑名单更可靠）
  • structlog包装stdlib logging时需显式添加ExtraAdder() processor，否则extra={}字段静默丢失且无报错。git reset –soft是整理commit历史的高效方法，可将任意数量commit压缩为语义清晰的提交，适合CI环境和自动化工具

领域知识 (domain)#

  • MIHD核心实验结论：高质量embedding本身比聚类方法选择更重要（STAIG+mclust ARI=0.546 vs STAIG+Leiden=0.513）；STAIG融合框架对gene encoder具有高度鲁棒性（Teddy ARI=0.495 vs PCA ARI=0.497），空间图对比学习机制主导聚类性能，gene embedding质量的影响被大幅稀释
  • Foundation Model在跨域下游任务上存在系统性迁移损失（Teddy gene-only ARI=0.186不如PCA=0.255，优化目标不匹配）；HD维度不平衡问题（concat融合1586d=50d PCA+1536d UNI2反而差于单模态）；数据规模是关键变量（adaln在DLPFC有效但在HD 17502 spots完全崩溃）
  • 多模态融合论文叙事需同等重视负面结果与正面结果。论文Setting应以研究空白定义（‘当前没有zero-shot FM融合的系统研究’）而非方法性能；单模态也能做检索这一反驳纳入核心论点可形成更强论证链（简单融合是陷阱→空间图学习是桥梁→正确融合才能超越单模态）
  • NeurIPS D&B Track投稿策略：‘领域纠偏者’（量化现有基准具体缺陷触发评估范式重构）比’新方法提出者’更容易被接受；审稿人最重视多维度评估、公平基线对比、人工验证机制（κ≥0.68）和反直觉负面发现。Nature Methods Analysis级别要求每个方法跑10次报mean±std、4维评估
  • 现代Linux桌面（Ubuntu 22.04+）已默认Wayland，跨平台窗口追踪必须以GNOME Introspect D-Bus为主路径，xdotool/xprop仅作XWayland回退。后台采样任务不应依赖最小tick 60s的CronScheduler，非整分钟周期任务需独立asyncio.Task循环

AI 使用备注#

有效模式:

  • ✓ Critic subagent对抗审查：在规划阶段主动发现Wayland兼容性、lark-oapi线程阻塞、ThoughtStore非单例、4个隐藏代码依赖等高价值架构风险，避免事后大规模返工，是本周多项目成功交付的关键
  • ✓ 并行subagent批量生成代码：同时创建多个分析脚本（跨切片Query汇总、Fig可视化、HD Marker Gene可视化），大幅缩短多任务串行时间
  • ✓ 系统化多层排查：从日志证据出发（backend.log的+305px偏移、permission_denials字段），逐层确认假设定位根因，在MCP三层断裂、GTK尺寸协商等复杂问题上表现突出
  • ✓ ccplan结构化规划将FloatBall Linux修复等复杂平台问题分解为6个备选方案，配合可行性探测后收敛最优解
  • ✓ ECL文档跨会话上下文持久化：FloatBall多天连续修复中，通过/compact恢复上下文，减少重复分析成本

局限性:

  • ✗ 缺乏运行环境时错误假设底层API（Tauri set_size、GTK gravity、Google Calendar calendarList scope）按预期工作，多次在错误层面迭代修复，需用户实测反馈或日志证据才能纠正方向
  • ✗ 多轮修复失败后未能主动识别’继续修补不能解决架构性问题’的边界，每次看到新日志就继续尝试下一方案，需用户明确要求才进行根因总结
  • ✗ 在快速迭代产品领域提供过时或编造信息（AionUI斜杠命令、Claude context window规格），不应以确定口吻陈述快速更新的技术参数
  • ✗ 缺乏本地环境感知，默认尝试从GitHub克隆已存在于本地的仓库，未优先搜索本地文件系统
  • ✗ git操作范围判断失误：在用户要求’回退本会话改动’时误用git stash影响所有未提交改动，险些造成历史工作丢失
  • ✗ 对大型代码库全局依赖感知不完整，integration layer的接线断裂（参数未传递）等问题需多轮排查才能定位

学习成果#

架构（architecture）#

  • STAIG融合框架对基因编码器的鲁棒性很高：Teddy 1024d（ARI=0.495）与PCA 50d（ARI=0.497）在STAIG框架下的性能几乎相同，这表明空间图对比学习机制主导了聚类性能，而基因嵌入质量的影响则大大减弱。高质量的嵌入本身比聚类算法选择更重要：STAIG+mclust（0.546）>STAIG+Leiden（0.513），Leiden有助于弱嵌入（+29%），但对于强嵌入来说收益递减（+6%）。
  • Linux X11窗口管理器无法保证透明无边界窗口大小的调整，不同的窗口管理器对同一坐标的解释不同，导致系统性的偏移；完全消除窗口大小调整是唯一可靠的解决方案，局部补丁只是解决了症状问题。Linux透明无边界Tauri/WebKitGTK浮动窗口的架构限制是：窗口管理器、GTK和WebView的内容区域三个层的大小/位置不会自动同步，这是平台限制而不是漏洞。正确的架构应该是使用大型原生窗口和内部纯CSS来实现展开/折叠，而不是依赖频繁的外窗大小调整。
  • Life-Copilot的彻底简化（替换引擎）通常比渐进式修补更有效：Thin Adapter + MCP Server + Skills三层模式取代了专有的路由逻辑，获得了跨CLI的通用性。大型重构应该在编码之前进行对抗性审查，可以提前发现隐藏的依赖关系和架构风险。本周Critic阶段发现了Wayland兼容性、lark-oapi线程阻塞、ThoughtStore非单例、4个隐藏的代码依赖等问题，这些问题的价值非常高，投资回报率极高。

调试（debugging）### 下周展望#

  • GTK窗口大小控制机制的全貌：GTK自下而上的协商方式，取最大值(window_requested, widget_natural_size)；WebKitGTK的自然尺寸约为200x200，需要调用widget.set_size_request()，Tauri的set_size/set_min_size/set_max_size在WebKitGTK窗口中无效；使用webview()是异步调度，后续的Tauri API调用会在回调执行前排队，按照FIFO原则可能会锁定旧尺寸；WebKitGTK的滚动条属于GTK控件层，必须在GTK代码层使用set_policy(Never, Never)来禁用它们。
  • 运行时日志是进行低级系统调试的基本手段：当发生GTK、WM、OS级别的系统行为时，应该优先通过日志/探针来确认假设，而不是从代码中推断；跨平台的动画策略需要整体一致性，在禁用CSS过渡时，必须同步调整相应的JS/Rust计时逻辑；X11 WM会强制将未装饰的窗口限制在可见区域内（edge_offset需要针对不同平台进行差异化处理），macOS/Windows则允许将窗口置于屏幕一半之外。
  • Claude CLI中的-p模式完成MCP配置需要两个步骤：(1)明确传递–mcp-config以加载项目级工具；(2)添加–permission-mode参数以绕过权限确认，在非交互模式下无法实现此功能；–allowedTools参数在这种模式下不可靠，因为与MCP服务器初始化存在竞争条件，bypassPermissions参数更适合用于后端服务调用。
  • 在调试嵌套进程CLI工具时（主进程→Claude CLI→MCP服务器），首先记录实际传递的命令行参数，然后逐一隔离并验证参数的效果，这比直接猜测参数行为更高效。三个主要优先事项：(1) MIHD论文执行——完成高清对比融合实验，解决4个代码漏洞（莱顿结果恢复、图表生成流程、空间连续性指标、多种子支持），开始使用实际数据撰写结果部分；(2) TokenMonitor在Linux生产环境中的验证——在真实的Linux X11窗口管理系统中测试固定窗口+输入形状解决方案，处理所有剩余的边缘情况，考虑对Wayland原生支持的调查；-(3) Life-Copilot的生产稳定性提升——监控结构化日志和审计记录，了解Discord机器人的实际使用模式，验证Screen Time Tracker在用户Wayland桌面上的准确性，研究OpenReview认证解决方案以解锁NeurIPS评审的访问权限。次要任务：将NIPS研究数据从DCC同步到DesktopLinux，探索AionUI远程代理功能，以实现统一的管理面板。

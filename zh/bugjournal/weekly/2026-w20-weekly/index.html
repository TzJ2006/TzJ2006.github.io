问题与解决方案#

1. 用户提示过于笼统，导致AI误解意图，超出范围并忽略约束条件 [统一代理工作流程系统] (2026-05-11)#

解决方案： 设计了包含五个必填字段的强制性规范模板（目标、范围、约束条件、成功标准、非目标），要求在实施前进行结构化规范。代理必须完成并获得用户确认。

2. 缺乏验证机制，导致不确定代码更改是否真正解决了问题 [统一代理工作流程系统] (2026-05-11)#

解决方案： 实现了验证门控，其中成功标准（可执行命令）在规范中定义。完成后，自动运行验证：通过则进入审查阶段，失败则暂停到调试模式，并提供结构化的假设。

3. PyTorch CUDA包（torch+cu130、torchvision+cu130）不在标准PyPI上，导致pip安装失败 [Condana环境迁移] (2026-05-11)#

解决方案： 在所有yaml文件中添加了–extra-index-url https://download.pytorch.org/whl/cu130，以访问PyTorch的CUDA特定包索引。

4. 源安装的包（mimicgen、robosuite、gadget）不在PyPI上，导致conda环境创建过程中pip阶段失败 [Condana环境迁移] (2026-05-11)#

解决方案： 实施了两阶段策略：首先创建仅包含conda的yaml文件（删除pip部分），然后安装conda包，最后从过滤后的requirements.txt文件中安装pip，排除不可用的包。

5. AI在测试失败时盲目重试或继续修改，没有推理过程或用户确认 [统一代理工作流程系统] (2026-05-11)#

解决方案： 设计了调试模式，在验证失败时暂停，生成结构化假设（因为X → 因此Y，附带证据和置信度），输出终端摘要和HTML详细报告，等待用户确认方向。

6. 缺少跨会话上下文记录，导致下一个会话或其他代理无法知道已经做了什么以及为什么 [统一代理工作流程系统] (2026-05-11)**解决方案：**实现了双格式审查日志：人类版本为.md格式（适合审查/PR描述的叙述性文件），代理版本为.json格式的结构化JSON文件（包含文件修改信息、理由以及程序读取的推理链）。#

7. 未部署到GitHub Pages的报告——72天内的报告未出现在 staging 中 [总结部署流程] (2026-05-12)#

学习成果#

架构 (architecture)#

  • 使用带有必填字段的结构化规范（目标、范围、约束条件、成功标准、非目标），比使用自然语言提示更能防止AI误解。成功标准应该是“成功是什么”的可执行定义——概念上与验证门无关，而是单独的步骤。
  • 社区成熟的工具（如Spec Kit拥有95K星数，Entire CLI拥有4.3K星数，git-cliff拥有11.8K星数）通过大规模验证比自定义构建更可靠。应优先采用这些工具，仅在需要控制时才构建自定义版本（如调试模式）和粘合层（如AGENTS.md协议）。
  • 跨会话上下文传递需要双格式：人类版本（适合审查/PR的叙述性文件）和代理版本（适合程序阅读的结构化JSON文件）。单一格式无法同时满足这两种需求。在审查日志中记录原始计划，可以方便地进行“意图与结果”比较，以便反思和改进。
  • 子代理驱动开发擅长并行执行独立任务：为每个任务派遣新的子代理（具有独立上下文），在完成后再进行两阶段审查（规范符合性→代码质量），避免主会话上下文污染。
  • AGENTS.md作为跨代理协议比SKILL.md更通用，可以被Codex/Cursor/Claude/Copilot读取。.cursorrules是特定于Cursor的补充文件——结合两者可以覆盖所有代理。

调试 (debugging)#

  • AI的所有判断都必须有透明的推理链（“因为X，所以Y”，并附带证据和置信度水平）。在失败时暂停等待用户确认比盲目重试更高效。结构化的假设迫使AI明确表达推理过程。
  • 在调试部署流程时，需要验证整个基础设施链（代码逻辑+文件路径+Git仓库+远程端点），而不仅仅是代码逻辑。AI可以找到代码错误，但可能错过基础设施层面的问题，除非进行端到端验证。
  • 应该作为独立Git子模块管理的被忽略目录在调试过程中需要明确的验证。在预期位置找不到.git目录是无声部署失败的重要诊断信号。

领域知识 (domain)#

  • PyTorch CUDA包需要显式的索引URL配置（–extra-index-url [[[HYMTPH_1]]）。标准的PyPI只提供CPU版本，而+cu130版本字符串无法通过pip解析。

工具 (tools)#

  • 使用conda和pip混合依赖关系的Conda环境迁移非常脆弱。采用两阶段方法（先使用conda包，然后使用pip进行单独安装，并具有容错能力）比集成conda环境创建提供了更多的控制和恢复选项。源安装的包需要手动排除并重新安装。
  • 在bash/sed中处理Windows路径时，由于反斜杠转义的复杂性，容易出现错误。对于复杂的字符串替换，委托给本地的PowerShell命令（Get-Content | .Replace() | Set-Content）更为可靠。

AI使用注意事项#

有效模式：

  • ✓ 子代理驱动开发，用于具有独立上下文的并行任务
  • ✓ 双格式审查日志（人类版本的.md格式和代理版本的.json格式），用于跨会话上下文传递
  • ✓ 在调试模式下使用结构化假设推理，强制使用透明的“因为→因此”推理链
  • ✓ 通过系统化的代码追踪结合基础设施验证进行根本原因分析
  • ✓ 两阶段问题解决：首先识别代码错误，然后验证基础设施层面

局限性：

  • ✗ 后台任务输出（run_in_background: true）经常会被截断或丢失，需要回退到前台执行
  • ✗ 最初低估了社区工具的成熟度，倾向于使用自定义构建以获得控制，而不是优先考虑经过验证的解决方案
  • ✗ 倾向于过度分层设计，将概念上统一的概念拆分为独立的组件
  • ✗ 最初的调试仅集中在代码逻辑层面，直到有人观察到实际部署状态后才发现基础设施问题（缺少Git仓库）
  • ✗ 在宣布修复完成后，没有主动验证端到端部署路径
  • ✗ 没有在审查日志中主动包含原始计划，以便于进行意图与结果的比较

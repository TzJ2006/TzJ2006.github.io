问题与解决方案#

1. VLA服务器初始观测中缺少摄像头图像，导致“observation/image” KeyError [错误恢复基准测试] (2026-02-16)#

解决方案： 修改rollout_generator.py中的_get_current_obs()函数，添加include_images参数；根据PolicyServerAdapter类型，在摄像头名称列表中添加include_images=True参数。

2. VLA服务器使用了错误的openpi API——手动加载norm_stats导致模型初始化失败 [错误恢复基准测试] (2026-02-16)#

解决方案： 用openpi的create_trained_policy()函数替代手动加载方式，该函数会自动处理norm_stats、transform和模型架构等问题。添加了 --config_name命令行参数，并提供了正确的摄像头键映射方案。

3. Pi0 (LIBERO检查点)在自然捕获过程中在PickPlace任务上实现了100%的失败率 [错误恢复基准测试] (2026-02-16)#

解决方案： 识别出失败模式分布：joint_limit_approach占47%，grasp_wrong_pose占30%，misalignment_pre_grasp占17%，overshoot占5%，premature_release占1%。将这些数据记录为自然错误数据，并确定需要针对PickPlace的特定检查点或领域适应措施。

4. VLA集成看似已完成（测试通过），但在实际运行中发现无法正常工作 [错误恢复基准测试] (2026-02-16)#

解决方案： 深入调查后发现以下问题：openpi API使用错误、norm_stats.json处理缺失、检查点路径不完整、--policy vla_server标志不受支持。制定了详细的修复计划，并系统地实施了所有修正措施。

5. 当StateExtractor返回的状态信息字典与raw robosuite观测数据格式不同时，_preprocess_obs()失败 [错误恢复基准测试] (2026-02-16)#

解决方案： 重构_preprocessobs()函数，使其能够识别输入格式并将其规范化为常见的结构{“state”: […], “images”: {…}。增加了强大的键查找机制，并改进了浮点数到uint8图像的转换方法。

6. 由于组织位置列表文件中的X/Y坐标顺序错误，导致从转置后的图像位置中提取到的视觉嵌入数据出现错误 [MIHD] (2026-02-18)#

解决方案： 将列名改为pxl_row/pxl_col（语义名称），交换坐标顺序为(X, Y) = (pxl_col, pxl_row)，删除视觉缓存，重新进行实验。

7. Pi0_libero检查点在PickPlace任务上的成功率仅为0%，而在Coffee/Stack任务上则达到96-100% [错误恢复基准测试] (2026-02-18)解决方案： 确定了根本原因：Pi0_libero是针对LIBERO领域进行微调的，缺乏针对PickPlace任务的训练数据（只有10个源演示，而其他任务则有500个演示）；提出了MimicGen→LeRobot→LoRA的流程。#

8. 无法直观查看VLA在PickPlace任务上的策略行为，因此无法诊断0%的成功率 [错误恢复基准测试] (2026-02-18)#

解决方案： 构建了visualize_policy_rollout.py脚本，该脚本支持EGL屏幕渲染、逐帧捕捉、HUD叠加显示（策略/步骤/奖励/成功情况）以及按轮次导出MP4文件。

9. 如何证明错误恢复基准测试的价值，而不仅仅是“能够生成场景”——需要证据表明数据确实能提升模型性能，以用于论文贡献 [错误恢复基准测试] (2026-02-19)#

解决方案： 设计了G4验证方法，作为四步实验：将2000多个场景转换为LeRobot格式，对错误场景进行LoRA微调，评估保留的测试数据集，测量改进效果（预计有15-30%的性能提升，并通过自助法统计检验和Wilcoxon检验来验证其显著性）。

10. UNI2基础模型的特征在DLPFC空间域聚类任务上的准确率仅为0.11-0.16 [MIHD] (2026-02-19)#

解决方案： 发现了5个根本原因：训练数据领域差异、预训练任务不匹配、DLPFC结构的细微差异、特征与基因之间的冗余问题，以及缺乏空间上下文。提出了6种适应方法（STEGO、SCAN、BYOL、DINOv2-GAT、TTT-MAE、SFDA），按照解决顺序进行实施：特定领域的预训练→测试时的适应→特征优化。

11. 错误恢复基准测试的数据集不平衡问题——90%的场景属于pre_reach/reach阶段，而transport/place阶段则完全为空（0个场景）[错误恢复基准测试] (2026-02-19)#

解决方案： 采用BC-RNN策略作为关键手段，以达到后续阶段的目标，将检测器的最小步长阈值从100降低到5-50，以便捕捉lift/transport阶段的失败情况。同时使用MimicGen生成200多个演示变体，以增加轨迹多样性。

12. 如何严格验证24类错误分类法是合理的、完整的且不会重复，以便用于学术出版 [错误恢复基准测试] (2026-02-19)#

解决方案： 采用三支柱验证方法：(1)将8个类别映射到ISO 15066/10218/IEC 61508标准的FMEA故障模式；(2)通过6个阶段×3个家族=18个单元格矩阵来证明完整性；(3)使用Cohen’s/Fleiss’ kappa进行统计验证（目标是在同一类别内的κ>0.6，跨类别的κ<0.3）。

13. M14基线评估（5448个剧集，运行时间为5-15小时）如果VLA服务器在运行过程中崩溃，可能会导致所有进度丢失 [错误恢复基准测试] (2026-02-19)#

解决方案： 在collector.py中实现恢复功能：启动时读取现有的episodes.jsonl文件，构建已完成的剧集列表，然后在三重循环中跳过这些剧集，从而保留摘要的正确性。添加了–resume标志。现在可以自动恢复，避免数据丢失。

14. VS Code远程SSH连接至HPC集群时遇到“client_loop: send disconnect: Bad file descriptor”和“tried to write to a nonexistent pipe”的错误 [基础设施] (2026-02-19)#

解决方案： Windows OpenSSH不支持ControlMaster/ControlPath/ControlPersist所使用的Unix域套接字。在~/.ssh/config文件中注释掉了这3个选项，并删除了过期的套接字文件。连接立即恢复。

15. run_benchmark.py每次运行都会重新实例化编码器并提取嵌入数据，浪费了大量时间来处理已经计算过的UNI2/SCAN特征 [MIHD] (2026-02-20)#

解决方案： 将CacheManager与has/load/save_*_embedding()方法集成在一起；在编码器实例化之前进行缓存检查，并支持变体（标准/频率/staig_strict）；将覆盖标志传递给两个调用点。

16. BC-RNN策略期望的输入大小为65，但环境提供的维度为37，导致 RuntimeError [错误恢复基准测试] (2026-02-20)#

解决方案： 根本原因是对象数量不匹配（训练时使用8个对象=56维，而评估环境中有4个对象=28维）。修复了_to_robosuite_obs()函数，将其中的’object-state’键替换为’object’，并从env._env._get_observations()中获取raw robosuite观测数据，而不是使用StateExtractor。同时修复了注入和自然捕获路径的问题。

19. Pi0.5 VLA服务器在加载模型时出现GPU内存不足问题（分配失败150MB），导致无法对7,800个案例进行M14基线评估[错误恢复基准测试] (2026-02-22)#

经验教训#

架构方面#

  • VLA端到端集成需要明确的摄像头观测处理：在state_extractor.extract()函数中需要包含images=True参数，并且为所有依赖摄像头的策略指定camera_names。政策观测格式的正确性至关重要。
  • 嵌入维度瓶颈会导致模型崩溃：STAIG的64维嵌入可以将1536维的视觉特征压缩96%，使得嵌入向量几乎无法区分（标准差为0.12）。通过空间细化可以消除少数簇。为了避免这种情况，需要使用256-512维的嵌入。
  • 医学影像中的空间坐标系需要明确的语义命名（pxl_row=Y, pxl_col=X），并通过物理单位进行验证（像素×缩放因子=微米）。通用的pixel_x/pixel_y命名方式容易出错，可能会降低性能。
  • 错误生成的相位不平衡源于成功率分布造成的可达性问题：需要具有60-70%的成功率的政策才能到达后期阶段，同时产生错误。如果随机错误过早发生，Pi0的成功率为0%——只有BC-RNN能够成功完成运输/放置任务。
  • 两阶段融合（自我监督细化→多模态整合）创造了新的实验维度，但改进的嵌入并不总是优于原始嵌入。优化目标的一致性至关重要：自我监督聚类优化了可分离性，但这可能与多模态融合的互补性目标不一致。
  • 缓存变体（标准/频繁/staig_strict）可以防止由于预处理标志不同而产生相同嵌入时的冲突。比单独使用编码器名称更为可靠。嵌入缓存基础设施消除了重复创建编码器的需求，从而节省了许多运行时间。
  • 长时间数据收集的恢复逻辑必须从JSONL中重建摘要状态，而不仅仅是跳过剧集。正确的方法是：启动时读取JSONL，重新构建all_episodes字典，将完成的剧集添加到跳过集合中，然后继续收集。这样可以确保幂等性。
  • 基线模型的选择对实验的有效性至关重要。使用不匹配的领域检查点会降低有效性声明，因为这是由于分布变化而非数据集质量导致的。必须明确验证或微调基线模型，以确保公平评估。
  • 多路径代码库中的观测管道错误需要彻底分析——在一个路径中修复的问题并不能保证在所有路径中都有效。需要全局搜索所有使用该模式的用例。
  • 依赖关系拓扑图应该在执行计划之前呈现。清晰地可视化M14作为关键路径障碍，有助于决策资源分配和时间线估计，从而实现战略性调度。
  • 基于符号链接的检查点管理可以实现路径引用的版本控制，同时保留原始数据位置。这对于共享文件系统的协作非常重要。

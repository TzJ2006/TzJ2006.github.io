问题与解决方案#

1. HD 免疫分析发现 T cell marker 极稀疏（7.3% spots），原「多模态优于单模态捕捉免疫浸润」论点不成立，R6 P3 无法按原计划撰写 [MIHD 论文写作] (2026-04-06)#

解决方案: 将 R6 P3 叙述改为「聚类-检索权衡」：STAIG 检索结果 96% 集中在上皮域，PCA 基因表达 T cell 富集 3.8x 优于 STAIG 的 2.2x，Attention 融合在边界处最均衡（60/40）；发现这是比原预期更有理论价值的新论点

2. benchmark_rm_ideal.py 默认将结果写入同一 summary.csv，多 pair 实验依次运行时后一个 pair 结果覆盖前一个 [MIHD 论文写作] (2026-04-06)#

解决方案: 每个 pair 实验完成后立即 cp 备份为独立文件（summary_151673_151674.csv 等），已覆盖的 151673↔151669 结果用 heredoc 手动重建

3. conda activate 在后台命令中不生效，三个实验进程输出文件始终为 0 字节 [MIHD 论文写作] (2026-04-06)#

解决方案: 改用 conda run -n General python ... 替代 conda activate General && python ...，确保后台命令在正确环境中执行

4. Hugo 未安装导致 update.sh 在 Step 5 以 exit 127 终止，整个 7 步流程无法完成 [website 基础设施] (2026-04-07)#

解决方案: 用户通过系统包管理器安装 Hugo extended v0.154.5

5. themes/PaperMod/ 和 public/ 部署仓库均缺失，Hugo 构建和 git push 阶段均失败 [website 基础设施] (2026-04-07)#

解决方案: 用户克隆 PaperMod 主题和 tzj2006.github.io 部署仓库到对应目录

6. static/videos 目录不存在时 find 命令报错，set -e 导致 Step 4 意外终止 [website 基础设施] (2026-04-07)#

解决方案: 在 find 前添加目录存在性检查 [ -d "$DIR" ]，目录不存在时跳过该步骤

7. HandBrakeCLI 回退分支的 PID 被重复加入 pids 数组，且压缩后输出为 .hb.mp4 未替换原文件，造成静默数据丢失 [website 基础设施] (2026-04-07)#

学习收获#

领域知识 (domain)#

  • 聚类与检索的最优融合策略存在分岔：STAIG 空间图对比学习最适合全局域聚类（Avg ARI 0.546），但强空间同质化特性导致边界检索过度集中（96% Epithelium）；Attention 融合在边界微环境检索上更均衡（60/40）——这是 MIHD 论文 R6 的核心新论点
  • 三组新 RM-IDEAL 实验：同 donor 跨切片（151673↔151674）Layer 3 Spearman ρ 高达 0.86；跨 donor（151673↔151669）Layer 3 出现负相关（ρ=-0.73），提示不同 donor 间 Layer 3 形态存在系统性差异
  • HD 结肠癌数据中 T cell marker（CD3D/CD3E 等）在 8μm spots 中极度稀疏（7.3% 有表达），PCA 基因表达检测 T cell 富集能力（3.8x）高于视觉+基因融合（STAIG 2.2x），说明 T cell 信号主要由基因表达驱动而非组织形态
  • 论文 Introduction 开头策略：直接给出最核心的数值发现（ARI 0.26→0.55）比泛泛背景铺垫更有说服力；每段对应一个 Results section 标题可强化论文结构一致性

工具 (tools)#

  • Hugo 站点部署有三个必须同时到位的独立依赖（hugo 二进制、主题仓库、部署 git 仓库），任何一个缺失都会在不同阶段以不同错误码失败，应在项目文档中明确列出这些前置条件
  • Claude Code 的后台 bash 命令不继承 shell 的 conda activate 状态，后台运行 Python 脚本必须用 conda run -n ENV python ... 而非 conda activate ENV && python ...

调试 (debugging)#

  • bash set -e 脚本中对可选目录的 find 操作必须加目录存在性检查（[ -d "$DIR" ]），否则正常的「目录尚未创建」场景会触发非预期的脚本终止
  • 并行后台任务（& + pids 追踪）的 PID 记录语句必须与 if/else 控制流边界精确对齐，公共代码放在分支外部容易引入重复 PID 追踪和静默文件替换遗漏 Bug
  • 长时间分批运行的实验若共享输出文件（如 summary.csv），需在脚本层面设计 per-run 输出路径，或在每批完成后立即备份；后台进程输出为空时应先用 ps aux 确认进程状态，而不是直接重新提交

AI 使用备注#

有效模式:

  • ✓ 多子 Agent 并行探索代码库 + 实验结果，在执行前完成可行性验证，避免基于错误前提写作
  • ✓ 静态分析优先策略（读代码→检依赖→检目录→实际运行）在运行前就识别出所有致命问题，减少盲目试错
  • ✓ 实际数据驱动修正论点：AI 运行分析后发现原论点不成立，主动提出更有理论价值的替代论点（聚类-检索权衡）

局限性:

  • ✗ 生成可视化图后未主动读取图像验证质量（query 点可见性），需用户反馈才发现问题，缺乏主动视觉 QA
  • ✗ 未能提前识别 benchmark_rm_ideal.py 的 CSV 覆盖问题，导致实验结果丢失需手动重建
  • ✗ 无法执行 sudo 级命令（安装 Hugo、pngquant 等），环境搭建需用户手动介入，无法端到端完成
  • ✗ 并行工具调用多次因竞态取消失败，被迫改为串行调用，影响检查效率

问题与解决方案#

关键问题#

1. Q-Former与LLaVA MLP是两个不同的架构，用户最初将二者混淆为一个方法#

解决方案: AI主动澄清：Q-Former来自BLIP-2，使用可学习查询+交叉注意力；LLaVA实际使用的是2层MLP连接器。用户确认两者都要实现。

关键洞察: 文献名称与实际架构不对应是常见误区，需在规划阶段澄清以避免实现方向错误。

2. MIHD融合管道存在嵌入归一化不一致：UNI2/scGPT已L2归一化，但HIPT/ResNet50/MLP/GCN未做归一化，融合后也无归一化，影响下游聚类质量#

解决方案: 设计三点归一化方案（post-encoder、pre-fusion、post-fusion），通过 utils/normalization.py 统一封装，并通过config控制各点的归一化方法。

关键洞察: 对已L2归一化向量再做L2归一化是幂等操作（安全），因此统一归一化不会损害已正确处理的编码器输出。

一般问题#

3. 原始实施过程中AI立即开始执行而未先写计划文件，用户需要中断并要求先将计划写入文件#

解决方案: 用户明确提出「Wait, please write your plan into a file first」，AI随即将计划写入 docs/ENHANCEMENT_PLAN.md。

关键洞察: 对于大型多阶段任务，AI倾向于直接执行，但用户更希望先有可审查的书面计划再开始实施。

人类思路 vs AI 思路#

战略层面#

Q-Former与LLaVA MLP的选择#

角色 思路
人类 用户提出「实现Qformer（用于LLaVA中融合视觉和语言嵌入）」，将两种架构合并为一个需求描述。
AI AI通过研究发现Q-Former（BLIP-2）和LLaVA MLP是不同方法，主动澄清并询问用户的选择，最终用户选择两者都实现。

差异分析: AI具备更精确的文献知识，识别出用户描述中的技术歧义并主动澄清；用户则具有更清晰的研究意图（两种都要探索）。

超参数配置化的范围#

角色 思路
人类 用户提出「polish the pipeline so that changing the config changes all the hyperparameter」，希望全流程配置化。
AI AI将其拆解为两阶段：Phase 1先建立config基础（新增dataclass），Phase 7再完成全部接线，并设计了覆盖编码器/融合/归一化/Q-Former/批次校正/QueST的完整config结构。

差异分析: 用户提出高层需求，AI负责技术分解和实施路径规划，两者协作明确。

实现层面#

计划先行还是直接实施#

角色 思路
人类 用户在AI开始直接执行后中断，要求先将计划写入文件以便审查。
AI AI的默认行为是立即开始执行任务（创建Task、读取文件、开始修改），未主动提出先写计划文件。

差异分析: 用户更重视可审查性和可控性；AI偏向行动导向。这反映了大型任务管理中人机协作的典型张力。

AI 局限性#

重要局限#

  • AI在处理用户「实现Q-Former（LLaVA中使用的）」这类存在技术歧义的描述时，最终通过提问解决，但若不加澄清直接实现可能会走错方向，说明AI对用语歧义的主动容错能力仍有限。

一般局限#

  • 在接到大型多阶段实施任务时，AI倾向于立即开始执行（创建任务、读取文件、修改代码），而未主动提出先输出书面计划供用户审查，需要用户主动中断才转为计划模式。

今日收获#

核心收获#

  • 三点归一化设计（post-encoder / pre-fusion / post-fusion）是处理多模态嵌入空间不一致的系统性方案，比逐个修复编码器输出更具可维护性，且对幂等操作安全。
  • Q-Former（BLIP-2）使用固定数量的可学习查询通过交叉注意力从视觉特征中提取信息；LLaVA MLP使用2层线性投影直接对齐视觉和语言空间——两者设计哲学截然不同，选择取决于计算预算和下游任务性质。
  • QueST通过k-hop子图池化实现niche表示，通过对抗性批次判别器实现批次效应校正——两种思路均可解耦并移植到MIHD管道中，无需引入QueST全部依赖。

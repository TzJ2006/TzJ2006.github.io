日报 — 2026-02-01#

今日概览#

  • 做了什么： 诊断并修复了 STAIG Fusion 的 ARI 从 0.4+ 跌至 0.13 的回归 bug，并新增 GCN 基因编码器对 CLI 的支持
  • 怎么做的： 通过 git diff 与实验日志对比定位了两处根本原因——错误传入 3000 维原始基因表达代替 50 维 PCA embeddings、边权重方法被错误改为 softmax_log——并逐一修复
  • 有什么用： ARI 从 0.13 恢复至 0.28~0.33（kmeans），为后续使用 mclust 聚类恢复至 0.4+ 奠定了基础，同时完善了 GCN 实现文档与对齐原始 STAIG 的代码

在 HPC 集群上调试 MIHD 项目 STAIG Fusion 的 ARI 性能回归问题，从 0.13 恢复至 0.28~0.33，并对 STAIG 与 MIHD 的 GCN 实现进行系统性对比分析

今日任务#

架构与策略#

  • ✅ 修复 STAIG Fusion ARI 回归 bug — 定位并修复两处导致 ARI 从 0.4+ 跌至 0.13 的根本原因：① run_benchmark_core.py 错误将 3000 维原始基因表达传入 STAIG（应用 50 维 PCA embeddings）；② STAIGTrainer.py 默认 edge_weight_method 被改为 softmax_log（应为 softmax_neglog）
  • ✅ STAIG vs MIHD GCN 实现对比分析 — 对比了 STAIG net.py 与 MIHD GCNGeneEncoder / STAIGTrainer 的激活函数应用位置、图构建算法效率、边权重方向、温度参数 τ、默认层数五个维度的差异，并将 GCNGeneEncoder 默认行为对齐原始 STAIG

实现与修复#

  • ✅ 运行 GCN + UNI2 + STAIG_fusion benchmark — 在 Section 151676 上测试了 GCN + UNI2 + STAIG_fusion、PCA + UNI2 + STAIG_fusion、UNI2 only、PCA only 四种配置，验证 GCN 路径端到端可运行
  • ✅ 更新文档与 CLI 支持 — 在 CLAUDE.md 中补充了 STAIG 实现注意事项（gene embedding 维度要求、边权重方法）；为 run_benchmark.py CLI 添加了 –gene_encoder gcn 选项；为 GCNGeneEncoder 增加了 apply_activation_to_last_layer 参数

问题与解决方案#

关键问题#

1. ARI 从历史最高 0.37~0.53 跌至 0.13，复现失败#

解决方案: 通过 git diff 对比历史成功版本，发现两处代码引入了错误"修复"：① STAIG 应使用 PCA gene embeddings（50维），而非原始基因表达（3000维）；② edge_weight_method 默认值被改为 softmax_log（距离大权重大），应为 softmax_neglog（距离小权重大）。移除 raw_gene_expr 逻辑并恢复正确的边权重方法后，ARI 回升至 0.28~0.33

关键洞察: 之前某次"对齐原始 STAIG"的代码修改实际上破坏了性能——raw_gene_expr 逻辑和 softmax_log 改动都是伪修复，真正的高 ARI 还依赖 mclust 聚类（需要 General 环境的 rpy2）

2. AIR 修复过程中多次尝试均未见效（epoch 数、tau、num_layers 逐一排查）#

解决方案: 最终通过 git stash + checkout 历史版本，逐步 diff 对比，才精确定位到 gene feature 维度和边权重方法这两个真正关键的变量

关键洞察: 当参数空间大时，应优先通过 git bisect / diff 历史成功版本来缩小搜索范围，而非逐参数试探

一般问题#

3. 运行测试时 mclust 聚类失败（无 rpy2）#

解决方案: 改用 kmeans 聚类完成验证，后续切换至 General 环境（已有 rpy2 3.5.11）可恢复 mclust 以获得更高 ARI

关键洞察: 环境差异（base vs General）是最初 ARI 差距的核心原因之一，需在 CI / benchmark 流程中固定运行环境

人类思路 vs AI 思路#

战略层面#

ARI 回归根本原因的识别#

角色 思路
人类 人类在会话开始时就提供了完整的根因分析计划，明确指出环境（General vs base）、聚类方法（mclust vs kmeans）、epoch 数（550 vs 300）、tau（26 vs 35）的差异，并指定了解决步骤
AI AI 按计划执行，但在实际测试中发现结果仍不对，开始自主排查 epoch 数、num_layers 等参数，花费多轮才定位到 gene feature 维度和边权重方法这两个真正原因

GCN 对比分析的深度#

角色 思路
人类 人类仅提出了分析需求，并通过提前准备好的 STAIG 代码访问权限（/add-dir）创造了条件
AI AI 并行探索两个代码库，系统性总结了五个维度的差异，并主动修改代码使 MIHD 对齐 STAIG

差异分析: AI 能快速跨越两个大型代码库做比较分析，人类的价值在于知道"要做这件事"和"这件事的意义"

AI 局限性#

重要局限#

  • AI 在修复过程中进行了多次错误的"修复"（先改 num_epochs，再改激活函数，再改 edge_weight_method），每次都需要重新测试才能发现无效，缺乏在修改前通过更系统分析来缩小搜索空间的能力

一般局限#

  • AI 无法判断 conda 环境上下文——当前 shell 是 base 环境，conda activate 在 subprocess 中无效，需要使用 conda run -n General 才能正确切换，AI 多次用了错误的环境调用方式
  • AI 在用 git stash pop 时因本地修改冲突导致失败，随后选择了 git stash drop 来解决，丢失了部分中间状态；在复杂 git 操作中容易产生状态管理错误

今日收获#

核心收获#

  • STAIG Fusion 高 ARI（0.4+）依赖两个关键条件：① 输入 gene feature 必须是 PCA embeddings（50维），而非原始基因表达（3000维）；② 聚类必须用 mclust（需要 General 环境的 rpy2），kmeans 会导致显著下降
  • STAIG 原始实现的边权重方向是 softmax(log(距离))（距离大→权重大），这与直觉相反；MIHD 使用 softmax(-log(距离))（距离小→权重大）则更符合直觉，两者会产生不同的聚类结果
  • 在排查性能回归时，优先使用 git diff 历史成功版本比逐参数试探效率高一个数量级，尤其当改动涉及多个文件时
  • STAIG 原始 GCN 对所有层（包括最后一层）应用激活函数（tau=0.5），而 MIHD GCNGeneEncoder 默认最后层不加激活；两者架构差异在于 MIHD 更适合作为特征提取器，STAIG 更适合端到端对比学习

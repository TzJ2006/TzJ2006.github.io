问题与解决方案#

关键问题#

1. Docker容器的bare except隐藏Render Error真实原因；NVIDIA apt包版本（535.288.01）与内核驱动版本（535.104.12）不匹配，常规安装后Vulkan仍无法识别GPU#

解决方案: 将bare except改为except Exception as e:并打印完整traceback暴露根因；从NVIDIA官网下载535.104.12驱动.run文件，–extract-only提取全套匹配库文件，通过VK_ICD_FILENAMES+LD_LIBRARY_PATH+__EGL_VENDOR_LIBRARY_FILENAMES三变量注入；批量复制所有libnvidia-*.so避免逐一追踪依赖链

关键洞察: 受限容器中版本不匹配时可直接从驱动包提取用户态库文件通过环境变量绕过系统路径，无需root权限；bare except是调试黑洞，排查阶段应第一步暴露完整traceback

2. Discord Bot将追问（‘你安排在几点了呀’）误判为通用问候，返回’我不太确定您在问什么’#

解决方案: 修复三层：①缩小general_keywords范围或改为精确全词匹配；②LLM Prompt FIRST CHECK加入对话历史感知条件；③优化语义路由GENERAL_ROUTE utterances精确度

关键洞察: 多层意图分类防御机制中任一层过于激进都会导致上下文对话失败；有对话历史时应整体降低general类触发敏感度

3. FLARE warmup训练在OpenPI config.py中缺少coffee_D1、stack_three_D1、three_piece_assembly_D1三个任务的配置#

解决方案: 在config.py的_CONFIGS列表中分别添加add_finetune_config和add_inference_config调用，共补充6个配置项

关键洞察: 训练配置扩展时需系统性检查全任务×全难度的完整性，遗漏的均为D1难度版本，说明扩展时应从矩阵视角而非逐条添加

4. Phoenix目录复制后占用78GB（train_datasets 77GB + ckpt 1GB），远超预期代码体积#

解决方案: 删除已复制的大目录，改为符号链接指向zhaoganlong原始数据，代码体积从78GB降至37MB

关键洞察: HPC上大型数据集和checkpoint应通过符号链接跨项目共享，不应随代码库复制；应在项目初始化阶段就建立shared_deps/共享依赖层

一般问题#

5. 大型h5ad文件直接读取+toarray()导致OOM（exit code 137），且初始分析目标文件定位错误（不同处理阶段的h5ad特征完全不匹配）#

解决方案: 改用backed=‘r’模式（内存映射）加载文件只切片读取；用户提供processdata.ipynb路径确认正确目标文件为all_region/adata_8um.h5ad

关键洞察: anndata支持backed模式实现lazy loading，大型稀疏矩阵应默认使用此模式；同一项目下不同处理阶段的h5ad差异极大，应先确认文件路径再分析

6. norm_stats路径重复：load_norm_stats()将asset_id（文件名）当作子目录名拼接，产生assets/norm_stats.json/norm_stats.json#

解决方案: 当robotwin_repo_id有值时直接从checkpoint_dir/assets目录加载，跳过asset_id子目录层级

关键洞察: 复用通用路径拼接函数时需验证其对目录结构的假设是否与实际一致

7. MulticoreTSNE构建失败：cmake版本不兼容（需要≤3.18，系统3.26.4）#

解决方案: （未解决）建议注释掉该可选依赖（仅用于TSNE可视化），用户拒绝。备选：pip install cmake==3.18.4或用scikit-learn TSNE替代

关键洞察: 旧版C++扩展包的cmake版本绑定是常见HPC安装痛点，可选依赖应在requirements中明确标注

问题与解决方案#

关键问题#

1. H&E image-only 方法在 DLPFC 精细空间域识别（7 层分层）上 ARI 仅 0.11-0.16，与多模态方法（0.45-0.64）差距巨大，且文献中几乎无精确 image-only 数值可参考#

解决方案: 通过 SpaConTDS 论文找到仅有的两个 image-only 数字（stLearn ARI=0.11，SpaConTDS ARI=0.16）；系统拆解 5 个 Foundation Model 失效维度（训练数据 domain mismatch、预训练任务不匹配、层间形态差异小、特征冗余、缺乏空间上下文）；提出 STEGO/SCAN/BYOL 适配路径，并在 MIHD 中验证 SCAN 可将 ARI 从 0.251 提升至 0.303。

关键洞察: patch-level 独立编码缺乏空间位置上下文（无法知道 spot 距皮层表面的相对深度），这一根本限制无法通过提升 encoder 解决，必须引入空间结构建模（图注意力或特征 distillation）。绝大多数多模态 ST 论文 ablation 设计为「去掉 image」而非「只用 image」，导致 image-only 基线数据极难从文献获取。

2. STEGO 训练全程 loss=NaN：dense InfoNCE 在大 n（3639 spots）+ 小 temperature（0.07）下 float32 数值溢出#

解决方案: 在 train_stego_refine() 入口对 image_emb 做 sklearn L2 归一化；在 STEGOSpatialLoss.forward() 内部对投影向量做 F.normalize，使用数值稳定的 log_softmax（减去 row max 避免 exp 溢出）。

关键洞察: dense 相似度矩阵在 float32 下可超出表示范围；必须在损失函数内部显式 F.normalize + log_softmax，不能仅依赖上游输入归一化。

3. transport/place 相位场景数量为零：Pi0 在 PickPlace 成功率 0%，无法靠 VLA rollout 生成后期相位场景#

解决方案: 三路并行方案：MimicGen 扩增 demo 增加轨迹多样性；BC-RNN（~60-70% SR）作为唯一能到达后期相位的策略；降低 detector min_step 从 100 到 50 覆盖 lift/transport 相位。

关键洞察: BC-RNN 是解决 transport/place 空白的核心——它是唯一有足够成功率能到达任务后期阶段的策略，VLA 在未微调时 SR=0% 完全无法提供后期错误数据。

4. G4 实验需要恢复轨迹，但 BC-RNN 在错误场景上成功率接近 0%，自动采集可行性低#

解决方案: 用户决策采用人工遥操作收集 ≥200 条恢复轨迹，按 error_type 分层抽样，先采集 50 条验证趋势后再扩大规模。

关键洞察: 人工遥操作数据质量高、覆盖所有错误类型，比 Oracle 自动化数据更能体现「人类示范恢复策略」的核心价值，对 Foundation Model 微调效果更有论文说服力。

5. VLA 评估进程崩溃后已完成 episode 数据全部丢失，重跑需 5-15 小时#

解决方案: 在 collector.py 中添加 resume 逻辑：读取已有 episodes.jsonl 构建 completed 集合，三重循环跳过已完成的 (scene_id, policy_name, seed)，同时重建 all_episodes 内存字典保证 summary 正确。

关键洞察: JSONL 的 append-only 写入天然支持 resume，无需数据库或 checkpoint 机制，只需启动时扫描一遍文件。

一般问题#

6. Windows OpenSSH 不支持 Unix domain socket，SSH config 中 ControlMaster/ControlPath/ControlPersist 导致连接在握手阶段崩溃（Bad file descriptor / nonexistent pipe）#

解决方案: 注释掉三个连接复用选项，清理 ~/.ssh/sockets/ 目录下残留 socket 文件。

关键洞察: ControlMaster 连接复用依赖 Unix domain socket，在 Windows 上根本无法工作；换 SSH 路径没有意义，真正的修复是去掉不兼容配置选项。

7. eval_scan_fusion.py 中 load_dlpfc_data 返回值解包错误，以及 basic_contrastive/qformer/staig_fusion 因 coords 格式不兼容而失败#

解决方案: 直接用 adata = load_dlpfc_data(...) 取单一返回值，从 adata.obsm[‘spatial’] 获取坐标，绕过 load_spatial_coordinates 的 barcode 匹配逻辑；后者坐标传参 bug 仍在修复中。

关键洞察: load_dlpfc_data 已在内部存入 adata.obsm[‘spatial’]，无需再单独调用坐标加载函数；apply_fusion 中不同策略对 coords 要求不同，需统一预处理。

8. MimicGen 启动报 EGL 设备 ID 断言失败（CUDA_VISIBLE_DEVICES=3 但 MUJOCO_EGL_DEVICE_ID=0）#

解决方案: 将 MUJOCO_EGL_DEVICE_ID=0 改为 MUJOCO_EGL_DEVICE_ID=3，与物理 GPU 索引一致。

关键洞察: MuJoCo EGL 的 MUJOCO_EGL_DEVICE_ID 必须使用物理 GPU 绝对索引，而非 CUDA_VISIBLE_DEVICES 中的相对编号。

9. H&E 面板与 scanpy 面板坐标系不对齐（比例和偏移均不匹配），以及 151510 section 缺失 tissue_lowres_image.png#

解决方案: H&E 面板改用 sc.pl.spatial(color=None, ax=axes[0]) 渲染（利用 scanpy 内置 scalefactor 处理），spot 透明度设为 0；151510 创建 hires → lowres 的 symlink。

关键洞察: scanpy 的 sc.pl.spatial 内置坐标变换，比手动计算 scalefactor 更可靠，三面板自动对齐；symlink 比复制文件更轻量且不影响原始数据完整性。

10. 单元测试 assert result['significant'] is True 失败（numpy.bool_ vs Python bool），以及 from scripts import _import_script_module ImportError（scripts/ 非 Python 包）#

解决方案: 将 is True 改为 == True（或隐式 bool 判断）；改用 importlib.util.spec_from_file_location 按路径直接加载脚本模块。

关键洞察: numpy.bool_ 不是 Python bool 单例，is 做对象身份检查会失败；scripts/ 是脚本目录而非 Python 包，不能像包一样 import。

人类思路 vs AI 思路#

战略层面#

实验边界主动延伸：SCAN 融合实验与 image-only 搜索约束#

角色 思路
人类 SCAN 结果出来后，用户主动提出「用 SCAN embedding 和 PCA + 所有 fusion 方法一起计算」，将两个独立模块连接起来；在 image-only 调研中多次明确约束「只要 image-only，不要加入 gene expression」。
AI AI 在完成计划内任务后输出结果并等待指示，未主动延伸到跨模块组合实验；在 image-only 搜索中反复将多模态对比数据纳入回答，需要用户多次纠正。

差异分析: 人类具有更强的实验设计意识，能将两个独立模块关联起来验证跨模块假设；而 AI 在执行有明确边界约束的搜索任务时倾向于「帮忙补全背景」而非严格执行约束。

跨框架类比与先验文献认知#

角色 思路
人类 用户直接指出「task1 不就是 clustering 吗？」将 CRC-100K 分类与 DLPFC 空间域识别等同；用户主动提示「我记得有个方法用了 BYOL」引导 AI 找到 STAIG-BYOL 的具体连接；用户发现 UNI2 聚类图中棕色重复 patch 现象并判断值得深入分析。
AI AI 描述评测任务时未主动建立跨框架类比；未主动发现 STAIG 使用 BYOL 这一关键连接；初次分析棕色 patch 时将问题误判为 scGPT（基因模态），需用户更正。

差异分析: 人类具有更强的跨框架类比能力和先验文献认知，能识别不同包装下本质相同的问题并提供关键观察起点；AI 需要用户提示才能建立联系，且在模态归属判断上有误差。

计划执行模式：Plan Mode 被多次中断与「人规划+AI执行」分工#

角色 思路
人类 在 M12、M13、M14、文档精简、G1-G4 等多个场景中断 AI 的 ExitPlanMode，最终直接提供完整计划文本给 AI 执行，完全绕过 AI 规划阶段。
AI AI 倾向于完成完整的「探索→规划→ExitPlanMode」流程，在被多次拒绝后未主动询问用户是否已有现成计划，而是继续推进同一路径。

差异分析: 「人规划+AI执行」比「AI规划+AI执行」更高效——用户在外部已备好计划，AI 的规划步骤在这种情况下是冗余的。AI 应在首次被拒绝后立即询问用户的具体疑虑或是否有计划可直接提供。

研究方案关键决策：G4 数据采集方式与 benchmark 论证框架#

角色 思路
人类 直接选择人工遥操作（非 AI 推荐的 Oracle 自动化方案），优先考虑数据质量和论文说服力；用简洁的大目标描述核心研究逻辑，要求 AI 细化到可操作小目标；精简文档时明确指出「保留所有大目标/中目标/小目标清单」。
AI AI 将 Oracle Recovery 标注为「推荐」方案（倾向工程便利性）；将论证框架展开为 G1-G4 三层体系并补充统计验证技术细节；文档精简时初始方案将目标清单作为可压缩内容。

差异分析: 人类从论文说服力角度做战略决策，能清楚区分「可精简的叙述性内容」和「不可精简的可操作 checklist」；AI 负责将战略转化为可执行路线图，但在推荐倾向上偏向工程便利性而非研究价值。

SSH 根因定位的实证方法#

角色 思路
人类 基于通用经验提出三步修复方向（useLocalServer/换 Git SSH/关闭 useExecServer）。
AI 通过实际测试错误信息直接定位到 ControlMaster socket 不兼容，找到更精准根因，跳过用户建议的三步方案。

差异分析: 用户的经验框架提供了正确方向，AI 的测试驱动方法找到了更精确的根因位置。两者互补：用户提供方向，AI 通过实证找到最小修复路径。

AI 局限性#

重要局限#

  • 多次 ExitPlanMode 被用户拒绝后，AI 未主动询问用户的具体疑虑或是否已有现成计划，而是继续推进同一路径。应在首次被中断后立即询问，而非等待用户再次中断。
  • 缺乏主动的跨框架类比意识：描述 UNI 评测任务时未将 CRC-100K 与 DLPFC 聚类等同；初次分析棕色 patch 时将问题误判为 scGPT 而非视觉编码器，需用户更正才切换框架。
  • 在研究方案推荐中倾向工程便利性（将 Oracle Recovery 标注为推荐），而非从论文说服力角度评估，需用户主动纠正偏好方向。
  • 未预见 dense InfoNCE 在大 n + 小 temperature 下的 float32 数值溢出（STEGO NaN），以及新建脚本中的运行时 bug（numpy.bool_ 比较、scripts 包导入、load_dlpfc_data 返回值解包），均需实际运行才暴露，未在编写时预判。

一般局限#

  • 在执行有明确边界约束的搜索任务时（「只要 image-only」），AI 反复将多模态对比数据纳入回答，需用户多次纠正才收敛，对约束指令的执行一致性不足。
  • 仅凭标题和摘要将 MILWRM 误归类为 image-only 方法（实际基于基因表达），以及无法从 PDF 图片中直接提取数值（图表数字依赖文本提取），反映文献预判准确性的局限。

今日收获#

核心收获#

  • Vision-only 空间域识别在文献中几乎无系统独立评测：DLPFC image-only ARI 仅 0.11-0.16（SpaConTDS/stLearn），而多模态方法可达 0.45-0.64；但粗粒度 WSI 组织分类（肿瘤/基质/坏死）无监督准确率可超 0.93。这一「粗粒度强、细粒度弱」特性决定了适用边界，MIHD 的 vision-only baseline 处于文献空白区，具有发表价值。
  • 监督评估（Linear probe/KNN accuracy）与无监督聚类（KMeans ARI）是根本不同的评估范式——UNI2 在 CRC-100K 上 95.7% 准确率不能直接预测 ARI 表现；benchmark 结果的跨范式迁移需谨慎解读。CRC-100K 9 类组织分类与 DLPFC 空间域识别在问题结构上本质相同，都是聚类问题。
  • SCAN 自监督聚类（离线特征 k-NN 挖掘+聚类一致性损失）在冻结 UNI2 特征上效果最优（ARI +20.6%），优于 STEGO 和 BYOL；SCAN 优化的视觉嵌入可系统性提升多模态融合效果（llava_mlp+SCAN ARI 0.409，+20% vs UNI2_raw），说明自监督视觉增强与多模态融合相互增益而非独立优化。
  • Foundation Model 失效于 DLPFC 空间域识别的核心原因：patch-level 独立编码缺乏空间位置上下文（无法知道 spot 距皮层表面的相对深度），这一限制无法通过单纯提升 encoder 解决，必须引入空间结构建模（STEGO 特征 distillation 或图注意力）。
  • 机器人研究 benchmark 价值论证需四层递进：大规模数据（G1）→ 分类合理性（G2）→ 性能基线（G3）→ 对下游 Foundation Model 的提升效果（G4）。省略任何一层都削弱论文说服力；G4 是最核心但最难实现的一层，人工遥操作数据（200条高质量）比自动化数据（数千条低质量）更有说服力。
  • 「数据少+无标注+类间差异小」场景下最有效的无监督适配路径：冻结 Foundation Model 提取特征 → STEGO/SCAN 等方法做域内 refinement，而非从头训练或全量微调。BYOL 在 ST 场景的优势：不需要负样本、小 batch 友好、对 H&E 染色变异不敏感。这一结论在 FGVC、医学影像、遥感、显微镜四个领域均有印证。
  • transport/place 相位覆盖策略：BC-RNN（~60-70% SR）是后期相位错误的唯一可靠来源；VLA 未微调时 SR=0%；FailSafe 和 VLA-SCT 是最易集成的 error recovery 方法（training-free 插件），适合作为 M15 微调对比的基线。
  • HEST-1k 方法论核心：224×224 patch + 冻结 encoder + PCA 降到 256 维 + Ridge 回归预测 50 个高变基因；PCA 降维是公平比较不同维度 encoder 的关键设计。spEMO 研究结论：GPFM/CHIEF 在空间域识别最优，UNI2 在 spot 检索最优——同一 Foundation Model 在不同下游任务的排名可能完全不同，需针对具体任务类型验证。
  • 大规模 InfoNCE 损失（n>1000 spots）的数值稳定性需在损失函数内部显式 F.normalize + log_softmax（减去 row max），仅依赖输入归一化不足以防止 float32 溢出。adaln_attention 对输入特征维度敏感，使用 SCAN embeddings（256 维）替代 UNI2（1536 维）后 ARI 骤降至 0.064，不适合处理大幅压缩的嵌入。
  • 文档精简的关键不是逐行删除，而是先建立「哪个章节是该内容权威来源」的映射关系，再删除所有非权威重复，保留独特内容并合并到权威位置。本次将 1554 行文档压缩 51% 至 763 行，同时保留了所有 87 个可操作 checklist 项目。

实践收获#

  • Windows OpenSSH 不支持 Unix domain socket，SSH config 中任何使用 ControlMaster/ControlPath/ControlPersist 的配置都会在 Windows 上导致连接失败；MuJoCo EGL 的 MUJOCO_EGL_DEVICE_ID 必须使用物理 GPU 绝对索引，而非 CUDA_VISIBLE_DEVICES 中的相对编号。

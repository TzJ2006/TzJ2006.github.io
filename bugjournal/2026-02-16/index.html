问题与解决方案#

关键问题#

1. VLA rollout 第一步报错 ‘observation/image’ KeyError，VLA policy 无法获取图像导致 pipeline 中断#

解决方案: 修改 _get_current_obs() 添加 include_images 参数，在 _generate_from_single_rollout 和 capture_natural_errors 中检测到 PolicyServerAdapter 时传入 include_images=True

关键洞察: 两种 obs 来源（state_extractor.extract 和 env.step 原始 obs）格式不同；初始 obs 来自 state_extractor，默认不包含相机图像；VLA 模式必须显式请求图像，此 bug 只在 VLA 模式首次运行时才会暴露

2. Pi0（LIBERO checkpoint）在 PickPlace 任务上 100% 失败，50 个 rollout 全部 success=False#

解决方案: 确认为预期行为——Pi0 在 LIBERO 数据集上训练，不能泛化到 robosuite PickPlace；natural_capture 模式利用这种失败来采集错误场景

关键洞察: 用于 natural_capture 的策略不需要成功执行任务，高失败率反而提供更丰富的错误样本；平均每 rollout 48 个错误事件说明 Pi0 在 PickPlace 上的失败模式具有多样性

一般问题#

3. conda run –no-banner 参数在 HPC 节点的 conda 版本中不被支持，VLA 服务器启动失败（exit code 2）#

解决方案: 去掉 –no-banner 标志直接使用 conda run，接受 banner 输出混入 stderr

关键洞察: HPC 环境的 conda 版本可能落后于开发环境，需测试兼容性；conda run 的 stdout 被 shell 缓冲，无法实时看到输出，应改用端口监听轮询判断服务就绪

4. VLA 服务器目标端口 5555 被其他进程占用，服务器无法启动#

解决方案: ss -tlnp 检查端口占用后改用 5556 端口，rollout 脚本的 –vla_port 参数同步修改

关键洞察: 在共享 HPC 节点上启动服务前应先检查端口可用性；轮询 ss -tlnp 比等待固定时间更可靠

5. Edit 工具在多步骤编辑 daily_summary.py 时积累了重复行（重复的 if 语句、注释、dict key）#

解决方案: 通过 Read 工具检查当前文件状态，定位并逐一删除所有重复行，最终用 ast.parse 做语法验证

关键洞察: Edit 工具需要精确的 old_string 匹配，当文件中存在已有重复内容时易引入新重复；每次编辑后应立即用 Read 验证

人类思路 vs AI 思路#

战略层面#

日报格式设计：用户主导的多轮需求升级#

角色 思路
人类 用户在第一次实现后连续提出三项具体升级：summary 应为散文而非 bullet list、需要 level/importance 评分系统（high=战略决策，1-10 分）、daily_overview 需要分全局和设备两层
AI AI 在第一次实现 conversation_summaries 时只做了最基础的字段结构，未预见优先级评分和多层 overview 这些使用场景

差异分析: 人类对日报作为"可检索知识库"的长期使用需求有深层理解；AI 倾向于最小可行实现，需要用户明确描述使用场景才能做出合适的设计决策

hidden_dim 消融实验：用户设计架构，AI 负责实现#

角色 思路
人类 用户预先设计了完整的 5 文件修改计划，包括 CLI 参数层级、三级注入链、exp_name 命名规范（仅 hidden_dim!=64 时加后缀）和 pipeline_config 消融组定义
AI AI 按照计划执行实现，在单元测试验证时主动覆盖了 exp_name 后缀、EvaluationPlanner hidden_dim 提取等边界情况

差异分析: 人类负责架构设计和关键决策（如默认值 64 不加后缀），AI 负责转化为可运行代码并主动验证；这是高效人机协作的典型模式

实现层面#

VLA 执行节奏：人类拒绝了 AI 准备直接运行的 rollout#

角色 思路
人类 在 AI 准备运行第一次（修复前的）rollout 时，用户直接拒绝了工具调用，随后中断并重新给出计划
AI AI 在服务器端口监听就绪后立即准备运行 rollout，未等待用户确认

差异分析: 人类对执行时机有更谨慎的判断；AI 在技术条件满足后倾向于立即执行，缺乏对"是否应该先确认"的主动询问

AI 局限性#

重要局限#

  • 在实现 conversation_summaries 时未预见 level/importance 评分、prose summary 格式要求和两层 daily_overview 等高价值需求，需要用户明确提出后才能逐一添加

一般局限#

  • Edit 工具在多步骤编辑中积累了重复行，未能在每次编辑后主动调用 Read 验证文件完整性
  • 在 VLA 服务器就绪后准备直接运行 rollout，未主动询问用户是否确认，被拒绝后才意识到需要等待

今日收获#

核心收获#

  • VLA rollout 初始 obs 必须显式请求相机图像：state_extractor.extract() 默认不包含图像，VLA 模式下必须传 include_images=True，否则 VLA server 在第一步因 ‘observation/image’ KeyError 而失败；这个 bug 只在 VLA 模式首次运行时才会暴露
  • natural_capture 模式下策略的高失败率是数据优势：Pi0 在 PickPlace 上 100% 失败，但每 rollout 平均产生 48 个错误事件、3 个场景；不能完成任务的策略反而能高效采集错误场景，这是 natural_capture 设计的核心逻辑
  • 日报工具加入 level/importance 评分是从"日志"到"知识库"的关键升级：评分使未来可以按重要性筛选和检索，将日常工作记录转化为可用的学习资产
  • GCN hidden_dim 通过三级注入链（CLI flag → extra_config → fusion_config）传递的设计模式，使消融实验可以在不修改模型代码的情况下进行；这种"外部化超参数"的配置注入模式值得在类似实验框架中推广

问题与解决方案#

关键问题#

1. vla_server.py 使用了不存在的 openpi 函数名及错误参数名（config 而非 train_config），以及错误的 pi0_libero 输入键名格式（动态路径 vs 固定键名 observation/image / observation/wrist_image）#

解决方案: 改用 create_trained_policy(train_config=config, checkpoint_dir=…)，由 openpi 库自动处理所有初始化；添加 IMAGE_KEY_MAP 字典映射相机键名；通过 inspect.signature 和阅读 libero_policy.py 源码验证正确格式。

关键洞察: openpi create_trained_policy() 封装了所有初始化细节，手动处理是多余且易错的；VLA 模型输入格式需与训练数据格式严格匹配，必须读源码而非猜测；跨 conda 环境代码应先通过 conda run 实际探测 API 再写代码。

2. adaln_attention 缓存结果 ARI=0.066 异常（早期未收敛版本）；STAIG GCN hidden_dim=64 在高复杂度 section（151672）引发严重坍塌（std=0.12，只有 2 个有效 cluster）#

解决方案: 使用 –override 强制重新运行 adaln_attention（ARI=0.159）；诊断根因（hidden_dim=64 bottleneck + spatial majority vote 放大），添加 –hidden_dim CLI 参数设计 64/128/256/512 消融实验。

关键洞察: 新策略实现后应始终 –override 验证，缓存可能保留早期 bug 版本；STAIG 64 维 bottleneck（信息损失 94-96%）与 spatial majority vote 双重叠加是坍塌根因。

3. injection 模式首次 rollout 即报 KeyError ‘observation/image’：_generate_from_single_rollout 初始 obs 通过 _get_current_obs() 获取，未传 include_images=True，相机图像缺失#

解决方案: 为 _get_current_obs 添加 include_images 参数，检测到 PolicyServerAdapter 时自动启用；同步修复 capture_natural_errors 中相同问题。

关键洞察: injection 和 natural_capture 两个模式各自独立维护 obs 获取逻辑，任何格式变更必须在两处同步修改，是持续的维护风险点。

4. PrematureReleaseClassifier 中 self._grasp_start_step=0 导致 hold_duration 恒为 0（Python falsy：0 or step 在 grasp_start=0 时返回 step），分类器永不触发#

解决方案: 将初始化改为 None，用 is None 判断是否已抓取，消除 falsy 歧义。

关键洞察: Python 中 0 是 falsy 值，步数等从 0 起计的变量不能用整数 0 作未初始化哨兵，应使用 None 并用 is None 判断。

5. SpatialAttentionBias 预计算完整 (H, N, N) 偏置矩阵（N=4384）导致 CUDA OOM；修复为惰性局部计算后，QFormer Enhanced 逐 spot Python 循环 + spatial bias 在 4384 spots × 200 epochs 下训练仍超过 1.5 小时#

解决方案: 将全局预计算改为按 spot 惰性计算（forward_single() 内仅提取当前 spot 邻居坐标），解决 OOM；但逐 spot 循环的根本性能瓶颈未解决，一次长时运行被用户中断。

关键洞察: Spatial bias 只需 (n_heads, num_queries, ctx_len) 局部子矩阵，无需全局 N×N 矩阵；QFormer 的逐 spot Python 循环本身是性能瓶颈，spatial bias 进一步放大，需向量化才能规模化使用。

一般问题#

6. 多 benchmark 并行导致 GPU OOM，OOM 进程长时间占用 CPU 但无 GPU 活动，被误判为正常运行等待超过 100 分钟；conda run 子进程输出缓冲导致无法通过文件内容监控进度#

解决方案: 改为串行运行，用 nvidia-smi + ps aux CPU 时间而非文件内容监控进程状态；通过历史 experiment_comparison.csv 补全 plain qformer 已有结果（ARI=0.344）。

关键洞察: OOM 进程不一定立即退出，需同时检查 GPU 显存和 CPU 时间；conda run 将子进程 stdout 全部缓冲至结束，长时任务应通过进程/GPU 状态监控。

7. scGPT embeddings 缓存路径不匹配（legacy vs pipeline 两套路径）；register tokens 与 spatial bias 同时启用时 bias 形状未考虑 context 长度从 k 扩展为 2+2k#

解决方案: 创建 symlink 修复路径（两套 npz key 均为 ’embeddings’，兼容）；在 _build_context() 中为 register token 位置填充零偏置对齐总长度。

关键洞察: Register tokens 插入后 context 长度变化必须同步更新所有依赖 context 长度的张量；symlink 是最低成本的路径兼容修复方案。

8. VLA 服务器启动障碍：port 5555 被占用、conda –no-banner 参数版本不兼容、openpi 包安装在其他用户目录（editable 安装），应使用 openpi05 而非 openpi conda 环境#

解决方案: 切换至 port 5556 并用 ss -tlnp | grep 5556 轮询就绪；去掉 –no-banner 参数；通过 python -c 'import openpi; print(openpi.__file__)' 定位实际路径，统一使用 openpi05 环境。

关键洞察: 共享 HPC 环境包路径可能不符合预期，应用运行时验证；对于 conda run 启动的长驻服务，应通过端口/进程状态而非日志判断就绪状态。

9. AI 初次搜索仅限项目目录，错误判断 VLA 检查点未下载；BC-RNN checkpoint Stanford 旧链接 404，无公开替代来源#

解决方案: 用户提示后扩大搜索至 zhaoganlong 目录和 HuggingFace 缓存，找到 Pi0（12GB）等完整检查点；BC-RNN PickPlace checkpoint 尚未解决，可能需从头训练。

关键洞察: 大型文件往往存放在共享用户目录；auto memory 存储跨会话操作记录，是定位历史下载任务的有效途径；外部资源链接应事先验证有效性。

10. CLAUDE.md、项目全景总结.md 和 error_taxonomy.py 中多处数字错误（检测器、分类器、测试用例、场景数等），且跨文档同步扩散#

解决方案: 以代码注册表（DETECTOR_REGISTRY、CLASSIFIER_REGISTRY）和枚举定义为 source of truth 逐一核实，grep 全文搜索确认无残留，79 个测试验证通过。

关键洞察: 文档统计数字应从代码自动生成而非手动维护；错误数字被其他文档引用后会放大传播。

AI 局限性#

重要局限#

  • 跨 conda 环境 API 猜测：无法直接 import 验证时，AI 猜测了 openpi 的 API 名称（不存在的函数）和参数名（config 而非 train_config），生成了无法运行的服务器代码；VLA 模型输入格式也凭推断实现，均需事后修正。跨 conda 环境代码应先通过 conda run 实际探测 API 再编写。
  • 内存设计疏漏：SpatialAttentionBias 实现时预计算了完整 (H, N, N) 全局注意力偏置矩阵（N=4384），未在设计阶段考虑大规模空间数据的内存上限，导致 CUDA OOM；多 benchmark 并行时也未预判 GPU 内存竞争风险。
  • 研究框架盲区：FM 融合方案 A（均值投影作为目标）存在循环定义缺陷，AI 提出时未主动揭示；adaln_attention 的设计目标是多 section 联合训练，AI 在实现时未充分警告其单 section 场景的局限性。
  • 搜索范围保守：AI 初始仅搜索项目目录，错误判断 VLA 检查点未下载；对大型分布式 HPC 文件系统（共享用户目录、HuggingFace 缓存）的搜索需要人类明确提示才能展开。

一般局限#

  • 监控与计算效率判断受限：无法实时读取 conda run 后台任务输出，QFormer Enhanced 超时和 OOM 进程均需通过 ps/nvidia-smi 间接判断；未主动估算长时任务完成时间并告知用户；计划被拒绝后未主动分析拒绝原因（范围、前置条件、优先级等）。

今日收获#

核心收获#

  • 使用真实 VLA 策略自然失败生成错误场景，比合成扰动（impulse/friction）更具研究价值——Pi0 在 PickPlace 上 50 次 rollout 成功率 0%，自然产生的 joint_limit_approach / grasp_wrong_pose 错误类型与 injection 互补，共同构成更完整的错误场景库（121→271）并超越 M5 目标。
  • QFormer Enhanced（register tokens + spatial bias）是今日非 STAIG 最优方案：ARI=0.401，较 plain qformer +17%（0.344→0.401），仅需 50 epochs。核心贡献来自 spatial bias 而非 register tokens，印证「空间先验 > 复杂融合架构」原则。
  • 编码器质量决定融合上限：PCA-only（ARI=0.288）远超 scGPT-only（0.115）；scGPT+UNI2 concat（0.028）低于 scGPT 单独基线，弱基因编码器与视觉特征的简单融合是有害的；训练型融合在强基线下存在过拟合风险。
  • Flow Matching 与 embedding 融合存在根本性不匹配：FM 需要明确的目标分布 B，融合目标是「创造新表示」而非「逼近已知分布」。最有原则性的路径是方案 C（跨模态预测）和方案 E（OT 中间点），但均未经验证。
  • openpi 的 create_trained_policy(train_config, checkpoint_dir) 自动处理 norm_stats、架构和输入输出变换，手动处理是多余且易错的；pi0_libero 输入格式固定为 state（8D）+ observation/image + observation/wrist_image；action chunk 应取第一行执行，剩余行入队缓冲以减少推理延迟。
  • Python falsy 陷阱：整数 0 不能用作「未初始化」哨兵。步数等从 0 起计的变量，0 or step 在 step 任意值时会因 0 是 falsy 而返回 step，应使用 None 初始化并用 is None 判断。
  • 跨 conda 环境 VLA 模型调用最优方案：pickle + TCP socket（Python stdlib）实现进程隔离通信，无需统一依赖树；最小侵入式集成（collector.py 仅 3 行注入，enabled 标志保持向后兼容）是维护多框架并存系统的有效模式。
  • STAIG GCN hidden_dim=64 在高复杂度 section 会引发严重坍塌（信息损失 94-96%），spatial majority vote 进一步放大效应——模型容量是空间组学 GCN 融合中被低估的关键超参数，为后续消融实验提供了明确动机。
  • zhaoganlong 的 Gemini VLM 实现使用 ChatAnywhere 代理，解决了国内网络访问限制，是可直接复用的 VLM 推理基础设施；与 Claude Code CLI 脚本模式互补，共同构成视频/帧级错误分类方案。

实践收获#

  • element_wise_sum 作为零训练基线（ARI=0.199），略胜 concat（0.181），可作为快速实验合理起点；文档统计数字应从代码注册表自动生成而非手动维护，否则随开发进展快速过时并在多文档间同步扩散错误。

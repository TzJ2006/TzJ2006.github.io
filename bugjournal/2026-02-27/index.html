日报 — 2026-02-27#

在天河超算集群上进行存储分析、Slurm 环境探索，并查看 M14 基线评估结果（Pi0/Pi0.5/BC-RNN 在错误恢复场景上全部接近零成功率）。

今日任务#

架构与策略#

  • ✅ M14 基线评估结果查看 — 查看之前对 649 个错误恢复场景运行的 Pi0、Pi0.5、BC-RNN 和 Random 策略的评估结果，确认各策略在 error recovery 任务上均接近零成功率。
  • ✅ 天河集群 Slurm 环境探索与使用指南 — 探索 tianhexy-a 集群的 Slurm 命令、分区信息、GPU 节点状态，发现 pam_slurm_adopt 限制，整理出完整的 GPU 节点申请与稳定连接方案（salloc + tmux）。

实现与修复#

  • ❌ 查找训练集评估代码修改 — 用户记得曾要求将评估切换为 evaluate on training set，但 AI 未能在代码库中找到相关修改记录，需要用户进一步澄清具体需求。
  • ✅ HDD_POOL 存储空间分析 — 分析 /XYFS01/HDD_POOL/sysu_gbli2/sysu_gbli2xy_1/ 下所有用户目录的占用情况，识别空间大户。采用并行 du 扫描解决大目录超时问题，最终获得全组各用户的磁盘使用概况。

问题与解决方案#

关键问题#

1. M14 评估中所有学习策略的恢复进度均低于 Random 策略#

解决方案: 暂无解决方案，这是预期结果——error injection 将机器人推到训练分布之外，未经针对性训练的策略无法泛化；M15（LoRA finetune Pi0.5）是后续解法

关键洞察: Random 策略的 recovery progress 反而最高（0.0199），说明当前学习策略在非分布数据上会做出有害动作，而非中性随机动作

2. 集群配置了 pam_slurm_adopt，无法直接 SSH 到计算节点（测试 an53 返回 Access denied）#

解决方案: 必须先通过 salloc 获得 job allocation，然后才能 SSH 或通过 srun 连接到节点；使用 tmux + salloc 保证断线后 job 不被杀

关键洞察: 这是天河集群的安全机制，不是错误；正确工作流是 tmux → salloc → srun –pty bash 或 ssh

一般问题#

3. GPU GRES 在 sinfo 中显示为 (null)，不知道是否需要 –gres=gpu 参数#

解决方案: GRES 为 null 说明 GPU 未在 Slurm 中注册为 GRES 资源，申请节点时不需要 –gres=gpu:X 参数，分配到节点后 GPU 自动可用

关键洞察: 部分集群直接按节点分配 GPU，不走 GRES 机制

4. 大目录 du 扫描反复超时（miniconda3、VLA_data、chenshiyu 等目录 5 分钟内无法完成扫描）#

解决方案: 切换为并行后台任务（所有目录同时扫描），并对每个目录设置独立 timeout，最终 VLA_data（248GB）等大部分目录成功扫描完成

关键洞察: Lustre 文件系统 inode 元数据访问延迟高，串行扫描会被单个大目录阻塞；并行化是必要的，但 5 个超极大目录（>2TB）仍无法在限制时间内完成

AI 局限性#

重要局限#

  • 跨会话记忆缺失：用户记得曾经要求修改评估为 training set 模式，但 AI 在当前会话中无法找到任何证据，只能搜索代码库而非回忆对话历史

一般局限#

  • 大文件系统扫描效率低：AI 多次启动串行扫描任务，反复超时后才切换到并行策略，浪费了较多时间（至少 3 次失败尝试）
  • GPU 资源探测不够精准：对于 GRES 显示 null 的情况，AI 没有立即给出明确结论，而是继续尝试多种检查命令，增加了不必要的来回

今日收获#

核心收获#

  • M14 核心发现：BC-RNN、Pi0、Pi0.5 在 649 个错误恢复场景上成功率全部接近零（BC-RNN 0.28%，VLA 0%），且学习策略的 recovery progress 低于 Random，说明策略在非分布数据上会主动做出有害动作。这强烈支持 M15（有针对性的 LoRA finetune）的必要性。
  • 天河集群使用要点：tianhexy-a 集群有 ai 和 xy-a800 两个 GPU 分区，当前有 9 个空闲节点（an16/21/23/28-29/31/34/39-40）；pam_slurm_adopt 阻止无 job 的 SSH；正确流程是 tmux → salloc → srun –pty bash；GPU 不需要 –gres 参数。

实践收获#

  • 组存储现状：共享文件系统总量 7.5PB，已用 81%；单组 sysu_gbli2xy_1 已确认使用约 14.7TB+（含超时未扫完目录估计 25-35TB），最大用户 chenjunye 约 5.1TB，用户自身 tangzijia 只有 28GB，处于最小量级。
  • Lustre 文件系统扫描技巧：对 HPC 上的大型 Lustre 存储，du 串行扫描会被大目录阻塞，需要并行后台任务配合 timeout；lfs quota 可能因配置问题数据不准确。

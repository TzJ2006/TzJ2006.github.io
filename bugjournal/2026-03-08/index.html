问题与解决方案#

关键问题#

1. RoboTwin 关键区域标注采集后 critical_region 字段全为 0，τ=0.10m 阈值无效#

解决方案: 新增 get_ee_target_distances() 为 10 个 task 逐帧记录 EE 与各目标点实时距离，待分析真实分布后重新校准阈值

关键洞察: 几何距离阈值不能凭配置参数经验估计——机器人末端执行器在实际工作空间的接近距离可能远大于 pre_grasp_dis；正确工作流是’先记录分布再定阈值’，而非直接实施标注再事后修正

2. Injection 模式在 three_piece_assembly 上生成 0 个场景；stack 任务 rejection 率极高，部分 rollout 生成 0 scenes#

解决方案: three_piece_assembly 转而可视化已有 158 个自然错误场景；stack 任务异常继续观察中

关键洞察: 脉冲力注入有效性强依赖操作对象的物理稳定性；稳定底座类任务需要专属注入策略（位姿扰动或夹爪偏置）而非通用脉冲力；stack 任务错误触发条件比 pick_place 更严格，策略 rollout 失败分布可能不均匀

3. Research Scout 缺乏按研究者维度搜索论文的能力，用户需要追踪特定老师的最新工作#

解决方案: 利用 arXiv API 的 au: 查询语法新增 –author 参数，镜像 –conference 模式完整集成到 search/report 两个子命令，与 –conference 互斥

关键洞察: arXiv au: 查询对姓名格式敏感（全名 vs 缩写），现有 –conference 代码是高质量参考模板；作者搜索与会议搜索的关键差异是无需 comment 字段后过滤

一般问题#

4. 服务器 headless GPU 渲染环境配置不全：MuJoCo EGL 设备 ID 未与 CUDA_VISIBLE_DEVICES 对齐，以及 collect_data.sh 缺少 Vulkan ICD 文件路径导致 SAPIEN 渲染失败#

解决方案: 将 MUJOCO_EGL_DEVICE_ID 显式对齐 CUDA_VISIBLE_DEVICES；在 collect_data.sh 中补充 eval.sh 已有的三个 Vulkan/EGL 环境变量

关键洞察: headless 服务器需要显式指定 Vulkan ICD 文件路径和 EGL 设备 ID，两者相互独立、不自动同步；应在项目中统一维护渲染环境配置模板，避免新建脚本时遗漏

5. 目录重组后 sys.path parent 链深度计算错误，以及 Phase 6 Agent 引入了已被删除的符号链接路径引用#

解决方案: 逐文件重新计算 file 到项目根的实际 parent 调用次数；主 Agent 事后验证并修正 FLARE 脚本中的 deps 路径引用

关键洞察: 多层目录迁移时所有基于 Path(file).parent 链的路径推导必须逐文件验证深度，此类错误不在静态检查中暴露；并行 Agent 各阶段的耦合点需主 Agent 事后统一验证交叉影响

6. SSH 远程执行脚本时反复报 ‘can’t open file’ 错误（同一命令失败 5 次以上）#

解决方案: 改用完全绝对路径指定 Python 脚本，并在命令头部显式 cd 到项目根目录

关键洞察: SSH 默认落地在 home 目录，不继承本地 cwd；所有远程脚本调用必须使用绝对路径或 cd <dir> && 前缀，永远不要依赖远程 cwd 与本地一致

人类思路 vs AI 思路#

战略层面#

VLA 关键区域概念的提出与数据驱动的阈值校准#

角色 思路
人类 从对 VLA 模型失败案例的直接观察（按钮没按准、抓取偏差）提炼出’关键区域’核心概念，明确了 CLS token 分类器的研究路径；发现 critical_region 全为 0 后凭对机器人工作空间的直觉推断’可能是阈值设小了’
AI 在用户给定概念框架后设计启发式标注规则，但初始阈值 τ=0.10m 仅凭 pre_grasp_dis 参数经验估计；接受人类诊断方向后，提出通过记录全程距离建立实证基础再确定阈值（比直接猜测调大更稳健）

差异分析: 问题定义、研究动机和异常诊断来自人类直觉；AI 在修复策略上体现了系统性实验设计思维，但初始参数选取缺乏领域直觉，缺乏对核心假设的主动验证意识

可视化策略：从研究价值角度主动设计 vs 满足技术完成度#

角色 思路
人类 用户两次主动推动可视化策略升级：①直接质问’你现在跑到不能生成 visualization 吗’，指出 AI 陷入分析-等待模式而非利用空闲 GPU 行动；②指出可视化不应只从错误瞬间开始，而应回放到错误前 5 秒或 rollout 起点，以便观察 BC-RNN 的完整行为模式
AI AI 初始倾向于罗列选项（查看旧结果 vs 等待）或仅在错误点前后截取固定帧数，聚焦于技术实现完成度，未从数据分析角度主动质疑可视化设计的研究价值

差异分析: 用户从研究需求出发（理解策略在错误前的行为比单看错误本身更重要），AI 停留在技术完成面；两次推动分别引导 AI 从’分析’切换为’行动’，从’展示错误片段’升级为’完整行为轨迹回放’

大规模重构的宏观计划制定 vs 执行层衍生问题修复#

角色 思路
人类 提供精确的 9 阶段实施计划，包括每个文件的具体修改内容、全局替换模式和并行执行策略
AI 负责执行，并在执行中发现约 4-5 处计划未覆盖的衍生问题（sys.path 深度错误、FLARE deps 路径问题、输出路径拼接 bug）并进行补充修复

差异分析: 宏观计划和架构决策由人类完成；AI 在执行层能发现并修复计划未明确覆盖的细节问题，但需要人类提供系统性框架才能高效工作

AI 局限性#

重要局限#

  • 缺乏主动验证核心假设的意识：设计关键区域标注时未提议先收集少量样本验证 τ=0.10m 阈值合理性再全量实施，导致全量采集后发现标注全为 0；面对可视化需求时默认进入’查找已有结果’模式，未主动评估能否立即利用空闲 GPU 生成新结果，两者均需用户明确纠正才切换策略
  • 可视化设计仅聚焦技术完成度，未从研究价值角度主动质疑设计——仅截取错误发生瞬间，没有考虑展示策略完整行为轨迹对分析失效根因的必要性，需要用户提醒才触发改进

一般局限#

  • 远程执行环境问题反复犯错：SSH 路径问题失败 5 次以上才彻底修正，MuJoCo EGL 设备 ID 未预判需与 CUDA_VISIBLE_DEVICES 显式对齐，浪费用户等待时间
  • 并行 Agent 执行中未预防跨阶段耦合问题：Phase 6 Agent 在 FLARE 脚本引入了 Phase 7 将删除的 deps 符号链接路径，需主 Agent 事后修正，说明不能完全信任独立 Agent 的局部正确性

今日收获#

核心收获#

  • 错误场景可视化的研究价值在于展示策略完整行为轨迹（含预错误上下文），而非仅截取错误瞬间；VLA 失败模式集中于特定任务阶段，可通过关键区域标注 + CLS token 分类器进行针对性精细化训练，比全程均匀训练更高效
  • 机器人操作数据标注中几何距离阈值必须从实际轨迹分布中确定；正确工作流是’先记录全程距离分布→再定阈值→再实施标注’，而非直接实现标注逻辑再事后修正——这一原则适用于所有依赖阈值的启发式标注方案
  • 脉冲力注入有效性强依赖操作对象的物理稳定性；稳定底座类任务（如 three_piece_assembly）需专属注入策略（位姿扰动或夹爪偏置），通用脉冲力命中率为零
  • 大规模并行 Agent 重构中各阶段耦合点需主 Agent 事后统一验证交叉影响；多层目录迁移时 Path(file).parent 链深度必须逐文件验证，此类错误不在静态检查中暴露，只在运行时以路径找不到的形式出现

实践收获#

  • arXiv API 支持 au: 查询语法用于作者维度搜索，与关键词/会议搜索互补；headless 服务器 GPU 渲染（Vulkan/EGL 配置）应在项目中统一维护环境配置模板，避免新建脚本时遗漏

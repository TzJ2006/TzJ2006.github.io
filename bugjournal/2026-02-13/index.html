日报 — 2026-02-13#

AI 局限性#

重要局限#

  • 系统性路径处理失误：MIHD 调度器未设置 subprocess cwd 导致 58 个实验文件全部写入错误目录（浪费约 1 小时 GPU 计算）；SSH 命令中 10+ 次重复遗漏 cd，缺乏「我一直在犯同一错误」的元认知和系统性策略调整
  • 缺乏领域常识驱动的数据验证：接受物理上不合理的 EEF-物体距离数据（0.22m 最小距离）作为既成事实而不质疑；未主动验证 MuJoCo cvel 内存布局，依赖了本身就错误的代码注释
  • 调参导向的问题诊断：面对零场景问题时反复调整阈值而非先构建诊断工具；面对 demo 回放异常时放宽 proximity 阈值（0.05→0.30）让测试通过，属于症状治疗，掩盖了真正的根因
  • 计划文档范围理解偏差：将「整个 MIHD 项目计划文档」理解为「当前会话完成总结」；对计划文件需要承担跨会话「状态持久化」功能的重要性认识不足，多次 ExitPlanMode 被拒后才调整

一般局限#

  • 急于实施而未充分理解用户意图：多次在用户希望讨论机制根因时直接进入 ExitPlanMode；未能从反复被拒中学习用户「直接执行」的偏好风格
  • 代码生成质量问题集中：生成了不存在的模块引用（MIHD utils.clustering_utils）；新文件未遵守项目 lint 规则（多次 ruff 修复）；mock 路径指向错误层级（多次重现）；JSON 提取首版使用贪婪 regex 而非平衡括号算法

今日收获#

核心收获#

  • MuJoCo 关键细节：① cvel 布局是 [angular(3), linear(3)]，与直觉（先线速后角速）相反；② set_sim_state_flat() 只恢复动态状态（qpos/qvel），不恢复 model 属性（geom_friction）和 xfrc_applied；③ 持续性注入器（Friction/GripperBias）必须在所有退出路径主动调用 clear()，否则物理效果跨候选叠加；④ 初始化后存在残余震动（~1.0 rad/s），需等待约 100 步（5s@20Hz）才能消散
  • 机器人仿真 demo 回放正确性：必须从 HDF5 data.attrs[’env_args’] 读取完整环境配置（含 controller_configs），而非依赖固定任务名；MimicGen 任务名含变体后缀（_D0/_D1）需剥离后映射回基础任务名；controller 参数（kp/damping/ramp_ratio）根本性地改变 action 效果，配置不匹配时相同 action 产生完全不同轨迹
  • HPC 批量实验调度关键实践：① conda run 在非 TTY 环境缓冲输出，直接使用 conda env Python 二进制（condaenv/XXX/bin/python）；② subprocess 必须显式设置 cwd=project_root，所有路径在 main() 入口转为绝对路径；③ SSH 多步骤命令用绝对路径而非依赖 cd；④ 可恢复性（checkpoint/resume）是第一优先级工程约束，SLURM job 有时间限制，任何不支持断点续跑的方案在实际使用中都会失败
  • 调试原则：先验证「输入条件是否正确」（环境配置、数据格式、参数来源），再调整算法参数；放宽约束让测试通过是技术债，根因修复后应立即恢复正确约束；近距离门控等物理参数必须先测量实际数据分布再设置，不能凭直觉假设
  • Google Calendar API：singleEvents=True 将循环事件展开为独立实例（ID 格式 parentId_YYYYMMDDTHHMMSSZ）；用 instance ID 调用 update 并移除 recurrence 字段，API 自动创建 exception 只修改该实例，不影响整个系列
  • 双重验证（SR+LLM）冲突解决策略：语义路由基于向量相似度，易受关键词干扰（“leetcode” 误匹配 manage_recurring），缺乏上下文理解；LLM 理解完整语义。mismatch 时应信任 LLM，SR 的价值在于低延迟的一致确认
  • 两阶段工具架构（export/merge）的核心优势：将「数据收集」与「AI 分析」解耦，export 无需 API key（纯本地操作），merge 统一调用 LLM；使工作流可跨设备、可延迟、可重试。这是分布式数据收集场景的标准模式
  • 注册表模式（Registry Pattern）构建可扩展插件系统：新增注入器/验证器/检测器只需在 init.py 注册，核心 rollout_generator 不需要修改。跨系统接口契约（检测器生成 spec → 注入器消费 spec）字段命名必须精确一致，且应有端到端单元测试覆盖，否则命名漂移只在实际运行时才会暴露
  • API/Service 层错误处理原则：service 层不应 print() 静默吞噬异常（return None/False），应 raise 自定义异常携带完整错误信息，让 interface 层决定如何向用户展示，既保持 service 层纯粹性，又确保错误信息不丢失
  • Q-Former 在空间组学场景的适配方式：每个 spot 以 k-NN 邻居的 (image+gene) 拼接嵌入为 context 序列，learnable queries 通过 cross-attention 聚合多模态邻域信息，输出固定维度的 niche 表示；extra_config 字典模式是跨多层函数调用传递配置的有效方式，比修改大函数签名或依赖全局变量更灵活且便于测试

实践收获#

  • asyncio 最佳实践三则：① Task 异常默认静默丢失，必须用 add_done_callback 主动注册；② 两层超时（连接级 httpx.Timeout + 协程级 asyncio.wait_for）才能完整防止卡死；③ JSON 提取用平衡括号算法而非贪婪 regex（贪婪匹配在多 JSON 块场景会从第一个 { 到最后一个 }，产生无效结果）
  • Python 类型与 mock 陷阱合集：① bool(“False”) == True，JSON 反序列化不自动推断布尔类型，需显式 isinstance 判断；② dataclass 在 f-string 中格式化需实现 format，str 只处理无格式规格；③ lazy import 的 mock 必须 patch 方法所在模块的局部命名空间，而非 import 链顶层

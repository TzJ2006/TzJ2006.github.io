问题与解决方案#

关键问题#

1. MuJoCo data.cvel 中 linvel/angvel 顺序读反：代码将 cvel[:3] 当作线速度，实际布局为 [angular(3), linear(3)]，导致 instability detector 系统性误触发#

解决方案: 交换 env_wrapper.py 中的 :3 和 3: 赋值，修正注释，同步修改 state_extractor.py/filters.py/impulse.py

关键洞察: angvel=0.445 rad/s 被误判为 linvel=0.445 m/s 超过 0.3m/s 阈值触发检测，是影响整个 scene 生成流程有效性的系统性 bug；MuJoCo cvel 布局与直觉相反，需对照官方文档验证。

2. Demo 回放中物体完全不动，EEF 与物体距离始终 ≥22cm，所有依赖 EEF 近距离的检测器全部失效#

解决方案: 发现根本原因：demo 录制使用 PickPlace_D0 + OSC_POSE controller_configs，但生成脚本使用裸 PickPlace 且无 controller_configs。修改 create_env() 从 HDF5 data.attrs[’env_args’] 读取完整环境配置，将 MimicGen 变体名映射回基础名，pop 不兼容的 kwargs

关键洞察: 相同 action 在不同 controller 参数下产生完全不同的轨迹；只有 instability detector（依赖物体角速度而非 EEF 位置）能生成场景，proximity/collision/grasp_precond 全部失效。必须从 HDF5 metadata 读取完整 env_args 重建环境。

3. subprocess 调用子进程时 CWD 不确定，导致 58 个实验执行成功但 npz 文件未写入预期路径（MIHD benchmark）#

解决方案: 在 build_command() 中将脚本路径改为基于 file 的绝对路径，在 subprocess.run() 中显式设置 cwd=project_root，main() 入口处将 output_dir 转为绝对路径

关键洞察: subprocess 继承父进程 CWD，跨目录调用或后台运行时相对路径输出目录会失效；调度脚本层面必须将所有路径转为绝对路径，断点续跑依赖文件系统状态时尤为关键。

4. 可视化视频三阶段画面几乎相同，看不到错误效果，缺少 demo 回放上下文#

解决方案: 从 HDF5 加载 demo actions，Phase 1 从 init_state 回放到 trigger_step，Phase 3 全程捕获每一步渲染帧；彻底废弃 collect_rollout_stats（该函数执行 step 但不返回渲染帧，吞掉关键帧）

关键洞察: 错误注入可视化必须展示完整因果链：demo 回放（手臂接近物体）→ detector 触发 → 注入错误 → 物理效果；仅展示注入时刻静态快照无法建立因果理解。

5. rollout_generator.py 硬编码只使用 ImpulseInjector，3 个新注入器/验证器完全未被使用；检测器生成的 error_type（‘friction_scale’）与注入器注册表键（‘friction’）不匹配导致 KeyError；检测器/注入器参数名不一致#

解决方案: 将硬编码替换为基于 INJECTOR_REGISTRY/VALIDATOR_REGISTRY 的动态加载；统一将 error_type 改为 ‘friction’，同步修正参数名和 base_detector.py 的 family_map

关键洞察: 注册表模式的调用方也必须同步改为 registry 查找；检测器（生产者）和注入器（消费者）之间的字符串类型协议应在统一常量文件中定义，「写好但未接入」是代码库中常见隐性风险。

6. FrictionInjector/GripperBiasInjector 修改 model.geom_friction/data.xfrc_applied，这些修改在 set_sim_state_flat() 状态恢复后持续存在，导致跨候选物理污染#

解决方案: 在所有三个退出路径（验证失败、稳定性失败、成功保存）均添加 injector.clear(env, spec) 调用

关键洞察: MuJoCo 的 set_sim_state_flat() 只恢复运动学状态（qpos/qvel），不恢复 model.geom_friction 或 data.xfrc_applied；持久性物理注入必须显式清除。

7. HPC 环境中 conda activate 在非交互式 subshell 中失效；Windows CMD 下 conda activate 同样无法正常工作#

解决方案: HPC：直接使用 /path/to/condaenv/bin/python 完整路径替代 conda activate + python。Windows：创建 start.ps1 在 PowerShell 中调用 conda activate，start.bat 代理调用 PowerShell

关键洞察: conda activate 需要 shell 的 interactive mode；batch/background 调用必须用环境 Python 的绝对路径；Windows CMD 不原生支持 conda activate，PowerShell 是 conda 的原生支持 shell。

8. MIHD DLPFC 多个 section 因空间坐标缺失、频域滤波空 patch 崩溃（边界 spot）、STAIG 模型坍缩（mclust 无法收敛）导致无法运行#

解决方案: 从 Dataset.zip 提取缺失的空间数据文件（tissue_positions_list.csv 等）；频域滤波后检查 patch 高宽是否为 0，是则返回零矩阵；mclust 调用外层加 try/except，失败时 fallback 到 KMeans

关键洞察: benchmark 框架应对数据缺失和模型失败鲁棒：记录警告并继续而非 crash；遇到数据缺失时应优先检查同目录下的压缩包。

9. daily_summary.py 只能在单台设备上运行，无法汇总多台设备的对话记录#

解决方案: 重构为 export/merge 两阶段：export 在各设备本地解析输出 JSON log（无需 API key），merge 在汇总机器上读取所有 log 生成最终日报

关键洞察: 解耦「解析」（纯本地操作）和「AI 总结」（需要 API），以 JSON 作为中间媒介，任意设备都可独立完成解析，最终汇总只需在一台机器上运行一次。

10. SR/LLM 意图分类不一致时使用了 SR 的 intent（如「晚点做 leetcode」被误识别为 manage_recurring 而非 update）#

解决方案: 修改 dual_verify.py，SR/LLM 不一致时改用 LLM 的 intent，日志记录「→ using LLM intent」

关键洞察: 语义路由器做嵌入相似度匹配时被关键词牵引，而 LLM 理解完整句义；mismatch 场景下 LLM 更可信。

11. _extract_json 初始实现使用贪婪正则 {[\s\S]*}，在多 JSON 块场景下合并两个对象导致解析失败；AI API 调用无超时保护导致请求卡死#

解决方案: 改用字符级平衡括号计数算法精确定位最外层 JSON；为三个 AI provider 添加 asyncio.wait_for(timeout=30) + httpx.Timeout(30.0) 双层超时保护

关键洞察: 正则无法处理嵌套结构；AI API 调用需要协程级（防止 await 卡死）和连接级（防止 TCP 握手卡死）两层超时，缺一不可。

12. ResourceMonitor 记录的 GPU 利用率和显存峰值始终为 0#

解决方案: 改用 threading.Thread daemon 线程每 1 秒采样一次峰值，stop_monitoring() 通过 threading.Event 通知线程后再返回统计

关键洞察: 原来的 stop_monitoring() 只在 GPU 操作完成后取一次快照，此时 GPU 已空闲读数为 0；需要在操作进行期间持续采样才能捕获峰值。

13. mock patch 路径错误：lazy import（方法体内部 import）使名字绑定位置改变，直接 patch 模块路径无效，导致测试失败#

解决方案: 改为 patch 函数内部实际执行导入的位置（即执行模块内的查找路径），或直接 patch 已绑定的实例方法

关键洞察: Python mock 的 patch 路径应指向「名字被查找（查找发生）的地方」而非「函数定义的地方」；lazy import 会将名字绑定到执行模块的命名空间。

一般问题#

14. 更新循环事件时修改了整个系列；日历操作失败时错误信息被静默吞掉#

解决方案: update_event 检测 recurringEventId，移除 recurrence 字段后提交让 Google API 只为该实例创建 exception；HttpError 改为 raise CalendarServiceError，各 handler catch 后返回详情给用户

关键洞察: Google Calendar singleEvents=True 返回带 instance ID 的独立事件，更新时不携带 recurrence 字段 API 自动只创建 exception；所有外部 API 调用必须在边界层抛出携带上下文的自定义异常而非静默返回 None。

15. smoke test 中 JSONL 的 success 字段被加载为字符串 ‘False’ 而非布尔值 False；MetricsWithCI 对象在 f-string 格式化时抛出 TypeError#

解决方案: EpisodeSummary 构造时添加显式 bool 转换（bool(str(v).lower() == ’true’)）；为 MetricsWithCI 添加 format 方法将格式化符代理给 .value

关键洞察: JSONL 序列化时若使用 Python str() 会将 False→‘False’，bool(‘False’)=True；Python dataclass 不自动实现 format，需要显式实现才能在 f-string 数值格式化中使用。

16. SSH 远程执行时 AI 反复忘记 cd 到项目目录，在 MIHD、error recovery benchmark 等多个项目中共计失败 20+ 次#

解决方案: 改用绝对路径调用 Python 脚本，或在 SSH 命令开头嵌入 cd $PROJECT &&；将规范写入项目 CLAUDE.md

关键洞察: SSH 登录后默认工作目录在 home；HPC 集群所有节点挂载同一共享文件系统，路径与登录节点完全一致，使用绝对路径最可靠。

AI 局限性#

重要局限#

  • 接受异常数据作为既成事实，未主动质疑数据合理性：面对 linvel=0.445 m/s 和 EEF 距物体 22cm 等明显异常数值，AI 从代码逻辑角度合理化解释，需要用户追问才触发根因调查。正确做法是「数据看起来不合理」时优先验证数据来源，而非调整阈值。
  • 未预见 subprocess CWD 继承问题导致 58 个实验数据丢失；引用了不存在的模块路径（utils.clustering_utils），基于计划文档假设而非实际代码结构；在大型函数内误用了 args 对象（仅存在于 main() 作用域）。共同特征：实现时未先验证运行时假设。
  • 对用户可视化反馈连续误解两次：将「视频三阶段画面相同」误诊为帧捕获技术问题，被用户纠正后才理解是设计层面的因果链缺失。AI 在处理设计层面的抽象反馈时倾向于先降维到技术实现层面。
  • ExitPlanMode 被多次拒绝后未主动询问原因（MIHD、CalendarPro 等项目共出现 5+ 次），而是被动等待用户新指令；未意识到应主动确认用户期望。
  • 初始实现选择不够稳健的方案：_extract_json 使用贪婪正则（测试失败后才改为括号计数）；mock patch 路径判断错误（测试失败后才修正）；在设置阈值前未先测量实际数据分布，导致多轮无效参数调整。共同特征：未先分析问题约束就选择了最直觉的方案。

一般局限#

  • SSH 命令反复失败同一错误（缺少 cd 或绝对路径）：在 MIHD、error recovery benchmark 等多个项目中共计失败 20+ 次，未能在前 2-3 次失败后识别问题本质并切换到绝对路径方案。
  • 对本地环境隐性知识缺乏预判：conda 环境名称需要用户告知才使用正确 Python 解释器；Windows 平台特殊行为（nul 设备名、CMD 不支持 conda activate）未能提前识别风险；生成视频后未主动考虑可读性标注需求。

今日收获#

核心收获#

  • MuJoCo data.cvel 布局为 [angular_velocity(3), linear_velocity(3)]，与直觉相反。错误读取会导致 linvel/angvel 完全颠倒，对依赖物体速度的 detector 产生系统性误判；需对照官方文档验证，不能依赖注释或直觉。
  • robosuite 环境复现的关键：必须从 HDF5 数据集的 data.attrs[’env_args’] 读取完整环境配置（包含 controller_configs），不能仅凭任务名创建环境。MimicGen 变体任务名（PickPlace_D0）需去掉后缀映射回基础名，并 pop 掉不兼容的 kwargs（z_rotation、replacement_probs 等）。同一任务不同控制器参数（kp、阻尼比、ramp_ratio）会产生完全不同的轨迹。
  • 调试前应优先验证底层数据的合理性（如物体是否运动、EEF 是否在预期范围），而非直接进入参数调优。「数据看起来不合理」通常意味着存在更深层的配置问题，而非参数需要调整。
  • 错误注入可视化必须展示完整因果链：demo 回放（手臂靠近物体）→ detector 触发 → 注入错误 → 物理效果。仅展示注入时刻静态快照无法建立因果理解；collect_rollout_stats 等执行 step 但不返回渲染帧的函数，可视化时必须逐步手动渲染替代。
  • MuJoCo 持久性物理参数（model.geom_friction、data.xfrc_applied）不随 set_sim_state_flat() 状态恢复而重置；所有持久性物理注入必须在退出前显式调用 injector.clear()，否则产生跨候选的隐性物理污染。
  • 双重验证分类器（embedding SR + LLM）不一致时应优先信任 LLM：SR 做嵌入相似度匹配，容易被 utterance 中的名词牵引；LLM 理解完整句意，mismatch 场景下准确率更高。
  • 从混合文本中提取 JSON 的正确方案是字符级平衡括号计数，不能使用贪婪正则 {[\s\S]*}——后者在多 JSON 块时会合并相邻对象产生无效 JSON。
  • Python mock 中 patch 路径必须指向「名字被查找（查找发生）的模块」，而非「函数定义的模块」；函数内部的 lazy import 会将名字绑定到执行模块的命名空间，对使用 lazy import 的代码进行单元测试时应 patch 已绑定的实例方法而非模块级路径。
  • HPC/服务器环境两个关键规范：①conda activate 在非交互式 bash 中失效，必须用环境 Python 的完整路径；②subprocess 调用子进程时需要显式设置 cwd，且输出路径必须是绝对路径，断点续跑依赖文件系统状态时尤为关键。
  • 多设备工作流应解耦「解析」（本地无网络操作）和「AI 汇总」（需要 API key），以 JSON 作为中间媒介；任意设备可独立完成解析，只需在一台机器上运行一次 AI 汇总，大幅降低使用门槛。
  • AI API 调用需要双层超时保护：asyncio.wait_for（协程级，防止 await 卡死）+ httpx.Timeout（连接级，防止 TCP 握手卡死）；asyncio.create_task 创建的后台任务必须设置 done_callback 做异常监控，否则任务静默死亡外部无法感知。
  • 深度学习 benchmark 管道应对聚类步骤的失败保持鲁棒：模型坍缩导致 mclust 崩溃时，通过 try/except fallback 到 KMeans 并记录警告；GPU 监控必须使用持续采样（后台线程）而非开始/结束时快照。
  • 注入器注册表键（INJECTOR_REGISTRY）必须与检测器生成的 ErrorSpec.type 字符串严格一致；最佳实践是在统一常量文件中定义错误类型枚举，避免散落各处字符串；「写好但未接入」是代码库常见隐性风险，需端到端集成测试才能发现。
  • EEF 近距离门控（max_eef_object_distance=5cm）是保证错误场景语义正确性的关键过滤条件：只有机械臂真正靠近物体时注入错误才具有实验意义；Google Calendar singleEvents=True 展开循环事件，更新时不携带 recurrence 字段 API 自动只为该实例创建 exception。

实践收获#

  • 定时后台任务用 last_scheduled_date 字段做每日幂等性保护是简洁有效的防重复策略；JSONL 格式（每行一条 JSON）是快速追加写入场景的最佳存储选择：无需加锁、天然支持流式读取。
  • CLAUDE.md 应只包含「需要读多个文件才能理解的大局架构」和「会造成困惑的运行时约束」，不应重复 README 内容或列出工具可直接发现的文件结构；过长的 CLAUDE.md 反而降低使用效率。

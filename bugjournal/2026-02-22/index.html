问题与解决方案#

关键问题#

1. VLA评估多重阻塞：Pi0.5 OOM（12GB模型显存不足）、Pi0 server port 5555被占用（模型加载2.5分钟后bind失败），以及checkpoint战略选择失误（pi0_libero为LIBERO任务微调版，pi05_base为通用基础模型，均未针对PickPlace/RoboMimic任务微调）#

解决方案: Pi0.5 OOM需更换大显存节点（A800）或探索量化方案；port占用需换用空闲端口；checkpoint问题由用户主动识别后中断，需明确评估意图（通用基线 vs 任务特定微调）后再决定是否继续

关键洞察: VLA基线评估的checkpoint选择是影响论文结论有效性的战略性决策，必须在实验设计阶段明确而非按默认配置执行；服务启动前应先检查端口占用和资源可用性

2. Vision Refinement首个实验（scan_cluster refine + concat）ARI=0.313，低于无refine的直接concat ARI=0.387，两步pipeline未能提升性能#

解决方案: 调整精炼方向：增大hidden_dim（默认256可能过小）、尝试attention等非concat融合策略、对比byol_spatial/stego_refine精炼效果

关键洞察: 为单模态聚类优化的视觉表征（scan_cluster 256d）与为多模态融合优化的表征存在目标不对齐——自监督聚类损失并不最大化跨模态互补信息；image-only ARI的提升不等于多模态融合质量的提升

3. tianhe集群SSH连接Slurm GPU节点不稳定（Bad file descriptor、pam_slurm_adopt策略冲突），以及srun直接调用命令超时（不加–overlap时新命令会排队无限挂起）#

解决方案: 改用srun –jobid= –overlap附着到已有交互式Slurm作业；在CLAUDE.md中明确禁止SSH直连，规范化为三步工作流（source set-XY-I.sh → squeue → srun –jobid –overlap）

关键洞察: HPC集群SSH直连计算节点在pam_slurm_adopt策略下本质不稳定；Slurm交互式job上追加命令需要–overlap标志，否则会排队挂起

4. ExitPlanMode审批流程与用户偏好摩擦：用户在MIHD整理和Phoenix微调方案两次会话中均拒绝了AI主导的计划审批环节#

解决方案: 用户在后续会话中通过’Implement the following plan:‘直接提交完整计划文本，跳过审批流程立即进入执行阶段

关键洞察: 用户偏好「我定计划、你执行」的直接协作模式；AI主导的plan-first审批工作流在用户已有明确计划时产生不必要摩擦，应识别并适应这种协作模式

一般问题#

5. Pi0.5 Phoenix数据管线兼容性：HuggingFace Python客户端遭代理503拦截，以及datasets 4.4.1与lerobot 0.1.0不兼容（torch.stack()收到Column对象而非Tensor tuple）#

解决方案: 改用wget+hf-mirror.com镜像站下载18.4GB数据集；将datasets从4.4.1降级至3.6.0，清空旧缓存后重新转换

关键洞察: lerobot 0.1.0要求datasets<4.0；datasets 4.x将列访问从list改为Column对象，破坏了lerobot的假设；工具链安装时应pin依赖版本

6. 实验进度监控可靠性：conda run批量实验stdout被缓冲（/tmp/output文件始终0字节），以及在后台转换任务未完成时（3247/4500 episodes）误判timestamp校验为真bug#

解决方案: 通过检查实验结果目录中文件是否存在来判断完成状态（比依赖stdout更可靠）；后台任务完成后重新验证即可确认timestamp是假警报

关键洞察: 文件系统原子写入比进程stdout更可靠作为实验完成标志；验证数据集前应先确认后台生产任务已完成

今日收获#

核心收获#

  • VLA基线评估的checkpoint选择是影响论文结论有效性的战略性决策：domain-specific微调版（pi0_libero）与通用基础模型（pi05_base）对PickPlace任务的SR结果有根本性差异，必须在实验设计阶段明确意图，不能按默认配置执行
  • 单模态优化表征不等于有利于多模态融合：scan_cluster为单模态聚类优化的256维表征降低了多模态融合ARI（0.313 vs 0.387），自监督聚类损失并不最大化跨模态互补信息；image-only ARI的提升不代表该表征能提升多模态融合质量，两者优化目标存在根本性差异
  • Slurm集群GPU访问规范：SSH直连计算节点在pam_slurm_adopt策略下本质不稳定；正确工作流为source set-XY-I.sh → squeue查看活跃job → srun –jobid= –overlap附着执行；–overlap是在已有job上追加命令的必要标志，不加时新srun会排队无限挂起
  • Pi0.5/VLA大模型资源规划：Pi0.5需要约12GB显存（通用A800节点可满足）；MimicGen HDF5图像分辨率为84×84（Phoenix训练通过OpenPI ResizeImages上采样到224×224）；OpenPI已有convert_mimicgen_data_to_lerobot.py可直接复用，无需从头编写
  • OpenPI/LeRobot工具链关键约束：JAX自动检测所有可见GPU（CUDA_VISIBLE_DEVICES设置即可），batch_size整除GPU数即可线性加速；lerobot 0.1.0要求datasets<4.0（4.x将列访问从list改为Column对象破坏torch.stack调用）；hf-mirror.com在代理拦截时是可靠的HF镜像
  • M14 pipeline性能基准：每场景约36秒（Random 15s + BC-RNN 21s），649场景全量评估约6.5小时；sanity check SR=0%完全符合预期（tip_over错误场景难以自动恢复）

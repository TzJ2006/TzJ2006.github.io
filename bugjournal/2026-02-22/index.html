问题与解决方案#

关键问题#

1. scan_cluster refine+concat（ARI=0.313）低于plain concat（ARI=0.387），两阶段融合预期效果未实现，几乎回退到image-only水平#

解决方案: 分析根因：scan_cluster将视觉embedding从1536d压缩至256d，以自监督聚类判别性为优化目标，丢失了与基因互补所需的细粒度空间纹理信息；原始高维embedding+concat保留了更多可融合信息

关键洞察: 自监督方法的优化目标（图像空间聚类）与多模态融合所需特征（与基因的互补信息）存在根本冲突；refine压缩后的embedding更适合image-only使用，多模态融合应直接使用原始高维特征

2. Pi0.5 VLA双重阻塞：2026-02-21因GPU OOM（150M bytes分配失败）无法启动，当前使用的checkpoint（pi0_libero/pi05_base）均非PickPlace任务专版，评估结果有效性存疑；Pi0 server因port 5555被占用启动失败#

解决方案: Pi0.5 OOM暂未修复，等待更大显存GPU；checkpoint匹配问题需决策：直接以通用预训练做零样本基线，或先在目标任务微调（Phoenix管线已就绪）；Pi0 port冲突需换端口（如5553）或释放占用进程

关键洞察: VLA大规模评估需在规划阶段提前确认显存需求（Pi0.5约12GB）和checkpoint任务匹配性；零样本评估与微调后评估是两种不同实验设计，应在计划阶段明确区分

一般问题#

3. datasets 4.4.1与lerobot 0.1.0不兼容（torch.stack()收到Column对象而非Tensor，抛出TypeError）；HuggingFace官方接口通过集群代理返回503#

解决方案: 降级datasets至3.6.0（pip install ‘datasets>=2.19.0,<4.0’）并重新转换；改用hf-mirror.com镜像站+集群代理（export http_proxy=http://172.16.31.200:3138）稳定下载，速度约40-200 MB/s

关键洞察: lerobot 0.1.0依赖datasets<4.0的列访问语义（>=4.0改变了返回类型）；集群代理对部分域名有效，hf-mirror.com是HuggingFace的稳定备用

4. Slurm集群节点访问问题：SSH直连pam_slurm_adopt节点不稳定；srun并发使用超时；多个GPU分区（xy-a800/ai/all/lava/temp）申请均权限不足#

解决方案: SSH改为srun –jobid进入已有作业（source set-XY-I.sh → squeue查JOBID → srun –jobid=）；并发操作添加–overlap标志允许同一allocation上并发多个srun；无新分区权限时通过–overlap复用已有job节点资源执行辅助命令

关键洞察: Slurm共享集群应以srun –jobid复用已分配资源为主要访问模式；–overlap是并发执行辅助命令（如监控、一次性命令）的关键标志

5. 后台任务监控困难：conda run输出被缓冲至进程退出，输出文件始终0字节；ExitPlanMode计划审批两次被拒导致规划流程反复中断#

解决方案: 改为基于文件系统检查（结果文件是否生成+ps aux进程状态）替代标准输出监控；用户通过直接提供完整计划文本（“Implement the following plan:"）绕过plan mode UI，AI直接进入实施阶段

关键洞察: conda run不适合实时后台监控，结果文件检查是更可靠的策略；用户偏好自主把控计划内容后直接提供执行，AI在plan mode中的探索工作仍有价值但呈现机制需与用户工作流对齐

AI 局限性#

重要局限#

  • 未能预判scan_cluster refine会降低多模态融合效果：实现两阶段融合时未分析自监督聚类目标与多模态融合信息需求的潜在冲突，只有实验结果出来后才做事后解释，缺乏先验推断能力
  • 未主动核查VLA checkpoint与目标任务的匹配性，直接按Makefile配置执行；未提供零样本评估vs微调后评估两种策略的权衡分析，需用户追问才暴露问题

一般局限#

  • 后台任务工具使用盲点：conda run输出缓冲问题、datasets与lerobot版本兼容性、数据转换未完成时验证产生误报——均在运行时才被发现，未能在任务启动前主动预判规避
  • 集群配置不透明（sinfo无输出）时无法推断正确操作方式，依赖用户补充关键参数；GPU资源分配倾向保守固定分配，未主动优化多GPU利用率
  • 日报批量生成产生元层次结构歧义：日报管线本身工作与被分析日期的研究工作混层，存在将历史日期内容误归入当日的风险；ExitPlanMode两度被拒反映计划呈现方式与用户工作流未对齐

今日收获#

核心收获#

  • 两阶段融合的信息损失陷阱：自监督精炼后的低维embedding（256d）以聚类判别性为优化目标，多模态融合时丢失与基因互补的细粒度信息；concat是最稳健的多模态融合基线（151673: ARI=0.387），当前数据规模下特征多样性优于特征质量
  • Phase II关键路径：M14（基线评估）→ M15.1（人工轨迹采集，8-20h，主瓶颈）→ M15.2（pi0.5 LoRA微调）→ M15.3（最终评估）；VLA评估前需提前确认显存需求（Pi0.5约12GB）、checkpoint任务匹配性和端口可用性
  • Pi0.5 Phoenix微调全流程验证：MimicGen HDF5（84×84，7D动作）→ LeRobot（LeRobotLiberoDataConfig可直接复用，无需自定义DataConfig）→ JAX 4卡训练（CUDA_VISIBLE_DEVICES=0,1,3,4，无需修改config，batch_size整除GPU数）→ 9任务评估；lerobot 0.1.0必须固定datasets<4.0
  • M14评估基础设施端到端验证通过：episodes.jsonl、detected_errors.jsonl、summary.json格式正确，SR/SPL/RP/Efficiency + Bootstrap 95% CI + Fisher test分析pipeline全部可用；649 scenes全量评估基础设施就绪
  • 151673模态互补分析：image-only天花板约ARI=0.303与gene-only PCA（0.312）相当；有效融合（concat=0.387）约+25%，但大多数学习型融合（attention/mean）无法稳定超越单模态基线，两模态互补信息需要精心融合设计才能被利用

实践收获#

  • 天河集群最佳实践：GPU访问优先用srun –jobid进入已有作业，并发操作加–overlap；HuggingFace数据下载用hf-mirror.com镜像+集群代理（export http_proxy=http://172.16.31.200:3138）；VLA server启动前检查端口占用和显存余量
  • 研究项目文档管理四层体系：战略规划（plans.md）/操作指南（BENCHMARK_GUIDE.md）/研究资料（research/）/历史归档（archived/）；追加式文档演化导致冗余，定期重构合并优于持续追加；新增工具需同步更新顶层README和CLAUDE.md

问题与解决方案#

关键问题#

1. QFormer forward()用Python for循环逐spot处理，2700 spot × 200 epoch = 54万次独立transformer forward，GPU利用率仅4-5%#

解决方案: 预构建padded邻居索引tensor，用torch.index_select批量构建context，通过key_padding_mask处理变长邻居，一次性forward全部spots

关键洞察: 变长序列batching的标准做法是padding+mask；预计算padded tensor只需一次性开销，训练循环中无需重建

2. pi0.5全量微调在4块A800 80GB上持续OOM：训练状态需要62GB（参数37GB+优化器15GB+EMA10GB），JAX FSDP init_train_state初始化阶段使用replicated_sharding，每卡仍需持有完整参数#

解决方案: 切换LoRA微调（gemma_2b_lora + gemma_300m_lora），可训练参数量减少90%，单卡仅需22GB

关键洞察: JAX FSDP分片仅在训练步骤中生效；初始化阶段的OOM无法通过增加FSDP设备数解决，必须减少模型参数量

3. STAIG边权重计算使用O(n²) Python嵌套循环，2700 spots时遍历约730万次，初始化耗时2-5秒#

解决方案: 用scipy.cdist一次性计算完整距离矩阵，NumPy高级索引直接提取边距离，scipy.softmax向量化per-row softmax

关键洞察: cdist底层为C实现，批量矩阵运算远快于Python循环；NumPy高级索引可直接从稠密矩阵提取稀疏COO格式边属性，无需额外循环

4. Pi0.5训练job 46551约580步后以exit code 0退出，日志显示step 498，但实际step 1000 checkpoint已保存；日志未实时刷出导致误判训练状态#

解决方案: 加入PYTHONUNBUFFERED=1和stdbuf -oL解决输出缓冲；set -eo pipefail和ERR trap捕获错误；从实际已保存的step 3000 checkpoint用–resume提交新job 46553

关键洞察: 输出缓冲掩盖了真实运行状态；checkpoint保存正常，只是日志未及时刷写；HPC训练脚本必须加unbuffered日志输出

5. 所有三条M14评估管线在scene 122/649处崩溃：EnvironmentMismatchError（xml_hash不一致）#

解决方案: 在collector.py场景循环中加入try-except捕获EnvironmentMismatchError，记录警告并break跳过该场景剩余seeds

关键洞察: 649场景数据库含约519个impulse/augmented场景（兼容）和约130个natural capture场景（不同xml_hash）；strict_version_check=true导致直接raise；需在collector层而非config层处理场景级不兼容

6. UNI2（ViT-H/14，patch_size=14）与STAIG strict模式256×256输入不兼容（256不能被14整除），触发AssertionError#

解决方案: 提取UNI2 embedding时自动检测STAIG strict模式并resize到224×224（224=14×16）

关键洞察: STAIG原始代码基于UNI v1（patch_size=16，支持dynamic_img_size），UNI2使用patch_size=14；扩展新编码器时必须验证patch_size兼容性

一般问题#

7. stale GPU进程（nvitop、失败训练残留进程）持续占用70-80GB显存，多次导致新训练尝试在看似空闲GPU上OOM#

解决方案: 使用fuser -k和nvidia-smi –query-compute-apps定位僵尸进程PID并kill -9清理

关键洞察: nvitop会创建CUDA上下文持有GPU内存；每次训练前必须验证GPU内存实际占用，不能仅看nvidia-smi显示的空闲显存

8. leRoBot数据集转换及pi0.5训练启动时三个兼容性问题并发：datasets 4.4.1→Column类型不兼容、LeRobotDataset.prev_delta_indices属性缺失、swanlab无法连接api.swanlab.cn导致训练启动超时阻塞#

解决方案: datasets降级至3.6.0并重新转换；添加prev_delta_indices=None初始化；注释掉import swanlab和sync_wandb()两行

关键洞察: leRobot依赖datasets API在4.x发生破坏性变更，应固定版本范围；swanlab.sync_wandb()注册的回调在wandb.init()时执行，需注释整行而非仅捕获ImportError

9. M13分析脚本运行时三个小bug：tabulate包未安装、episodes.jsonl中success字段为字符串’False’/‘True’导致int()转换失败、scipy.stats.kruskal在所有SR=0时抛出异常#

解决方案: pip install tabulate；加入string-to-bool转换；kruskal调用外加try-except，退化情况返回H=0, p=1.0

关键洞察: 分析脚本依赖未加入requirements.txt；当前所有策略SR=0，kruskal结果无意义，待VLA数据后重新运行才有意义

今日收获#

核心收获#

  • scan_cluster vision refine对弱融合方法（mean/attention/basic_contrastive）有显著提升（+0.04~+0.09 ARI），但对强学习型融合（qformer -0.054，concat -0.074）反而有害——1536d→256d的降维丢失了learnable fusion能利用的判别性特征
  • QFormer baseline（151673，ARI=0.483）是当前PCA+UNI2所有组合的最高点，超过所有refine+fusion组合；直接使用原始高维embedding配合learnable fusion是当前最优策略，无需额外refine步骤
  • pi0.5（2B PaLiGemma + 300M action expert）全量微调训练状态需要~62GB，超出单卡A800 80GB实际可用容量；JAX FSDP init_train_state初始化阶段使用replicated_sharding（每卡持有完整参数），FSDP分片仅在训练步骤生效，无法解决初始化OOM；切换LoRA是有限GPU资源下的必要选择（~22GB）
  • Fleiss’ kappa=-0.02（poor）且最大跨类kappa=0.71，说明M13的14种错误类型中存在高度共现，分类体系需要重构或合并——这是M13的核心发现；当前区分度分析（SR=0）需VLA数据后重新运行才有意义
  • 评估数据库包含不同来源场景（impulse vs natural capture）时，必须在collector层做场景级错误恢复；单场景崩溃不应终止整条评估管线，依赖config层strict=false是不够的
  • HPC训练脚本最佳实践：①PYTHONUNBUFFERED=1+stdbuf -oL防止日志缓冲掩盖实际进度；②sbatch独立作业比srun交互式作业更稳定（避免步骤超时）；③每次训练前用nvidia-smi –query-compute-apps验证GPU实际占用（nvitop等监控工具会持有CUDA上下文）
  • Slurm srun –overlap允许在同一已分配job内叠加多个步骤，无需排队等待新GPU资源，对调试和并行任务极有价值
  • staig_fusion strict模式下gene_encoder=none与gene_encoder=pca产生完全相同的结果（STAIG使用独立HVG特征路径，是需要修复的接口设计隐患）；扩展新vision encoder时必须验证patch_size兼容性（UNI v1 patch_size=16支持256×256，UNI2 patch_size=14不支持）
  • STAIG fusion在10个有效DLPFC sections上均值ARI=0.546，在151671（0.810）和151672（0.770）上突出，在151669（0.254）和151676（坍塌）上较差，效果与section组织结构清晰度强相关

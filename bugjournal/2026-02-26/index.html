问题与解决方案#

关键问题#

1. Pi0.5 在 5 个 MimicGen 任务上训练，却在 PickPlace 上评估，导致 0% success rate#

解决方案: 改变策略：不重新训练，而是扩展评估框架，在已训练的 5 个 task 上评估；同时 BC-RNN 也采用相同策略（训练和测试 task 保持一致）

关键洞察: 训练-评估任务不一致是最常见但最难察觉的错误之一，需要在项目初期就明确 task mapping

2. BC-RNN 评估时维度不匹配：Expected 65, got 37#

解决方案: 根本原因是 create_env() 剥离了 _D0 后缀，导致使用 base robosuite 环境而非 MimicGen 增强环境，物体观测维度不足。修复：添加 import mimicgen 触发环境注册，保留 _D0 后缀。

关键洞察: MimicGen 的 _D0 环境注册是隐式的，需要 import mimicgen 才能激活；随意截断 env_name 会破坏 observation space

3. BC-RNN 训练配置为 low_dim 模式，与 MimicGen 论文主结果（image 模式）不一致#

解决方案: 全部重写为 image 模式：移除 object 观测键，添加 agentview_image 和 robot0_eye_in_hand_image，配置 ResNet18Conv + SpatialSoftmax + CropRandomizer

关键洞察: MimicGen 论文主结果基于 image 模式（84×84 像素输入），low_dim 模式成功率通常更低且不可比较

4. 当前登录节点（天河集群）无 GPU，无法直接运行 VLA server 或执行评估#

解决方案: 通过 Slurm 作业调度系统提交到配备 GPU 的计算节点（如 an49），或使用已有的交互式 GPU 作业（JOBID 45628）进行评估

关键洞察: 集群上的 VLA 评估需要分离登录节点（代码探索/调度）与计算节点（模型推理），评估流程需通过 SLURM 作业提交或利用已有的交互式 GPU 会话

一般问题#

5. SLURM 分区权限不足：xy-a800 分区不可用#

解决方案: 切换到 ai 分区；但 sbatch 仍然失败，最终通过 srun –jobid=45628 –overlap 复用已有交互式 job，直接在 an49 节点启动训练进程

关键洞察: 在权限受限的 HPC 环境下，复用已有 interactive job 是绕过分区权限的有效方法

6. srun 命令使用了系统默认路径（/usr/bin/srun）而非正确的 SLURM 环境路径#

解决方案: source /APP/u22/ai_x86/toolshs/set-XY-I.sh 后使用 /usr/local/slurm.24051/bin/srun

关键洞察: 多版本 SLURM 共存的环境需要明确指定 SLURM 版本路径

7. 评估结果目录不存在，尚无任何 phoenix checkpoint 的已有评估数据#

解决方案: 需要按照 evaluate_mimicgen.py 中的两阶段流程（先启动 VLA server，再运行评估脚本）从零开始执行评估

关键洞察: 这是首次在 phoenix 训练 checkpoint 上运行评估，无历史基准可参考

AI 局限性#

重要局限#

  • 未主动验证训练-评估 task 一致性：分析 Pi0.5 failure 时，AI 假设评估 task 与训练 task 一致，直接归因于 action clipping 和 OOM，需要人类提问才发现真正的根因
  • 初次实现 BC-RNN 配置时选择了 low_dim 模式而非与论文一致的 image 模式，需要用户明确指定才更正
  • AI 在发现当前节点无 GPU 后，未主动询问用户是否有可用的 GPU 节点或提供具体的 Slurm 提交命令来推进评估，停留在规划阶段而非帮助用户直接采取下一步行动

一般局限#

  • SLURM 环境问题需要多次尝试才解决：先尝试 sbatch（权限不足），再尝试 srun（路径错误），最终通过用户环境探索找到正确路径
  • 多个子 Agent（Task tool）的探索结果以 ToolResult 形式出现在对话中，说明 AI 使用了多层嵌套 agent 调用，增加了复杂性，但最终评估脚本仍未实际运行
  • Write 工具要求先 Read 才能覆写，AI 多次因此失败（bc_rnn_stack.json 等），影响了工作效率

今日收获#

核心收获#

  • 任何机器学习评估实验开始前必须验证：训练任务 ⊆ 评估任务，且两者使用完全相同的 observation space 和 action space；否则 0% success rate 几乎是必然结果
  • MimicGen image 模式 BC-RNN 关键超参：ResNet18Conv + SpatialSoftmax(32 kp, 64-dim) + CropRandomizer(76×76)，RNN hidden_dim=1000，actor_layer_dims=[]，batch_size=16，hdf5_cache_mode=‘low_dim’（只缓存低维，图像按需读取）
  • MimicGen 的 _D0 环境注册机制：必须 import mimicgen 才能激活 robosuite 中的 Coffee_D0/Stack_D0 等环境；不可随意剥离 _D0 后缀，否则 observation space 会缩减导致维度不匹配
  • MimicGen 不提供预训练 checkpoint，只提供 HDF5 数据集；BC-RNN 是官方唯一 benchmark 算法，需要从头训练。论文报告 image 模式 D0 成功率：Stack 100%，Coffee 100%，Threading 98%，Three Piece Assembly 82%
  • Pi0.5 phoenix_comparison checkpoint 已训练至 60000 步（步数：10000/15000/20000/25000/30000/35000/40000/45000/50000/55000/60000），存储于 zhaoganlong 的框架目录下，评估需通过 websocket 协议连接 VLA server（openpi05 conda 环境）与评估客户端（mimicgen_env conda 环境）
  • 天河集群的 VLA 评估采用两环境分离架构：模型推理端使用 openpi05 环境启动 serve_policy.py（websocket server），评估端使用 mimicgen_env 运行 evaluate_mimicgen.py（websocket client），两者通过 localhost 端口通信

实践收获#

  • 在 HPC 权限受限环境下的训练方案：当 sbatch 分区权限不足时，可通过 srun –jobid=<existing_job_id> –overlap 复用已有 interactive job 的资源，配合 CUDA_VISIBLE_DEVICES 实现多 GPU 并行训练

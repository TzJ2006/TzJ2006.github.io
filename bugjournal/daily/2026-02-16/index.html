Lessons Learned#

Key Lessons - Embedding Dimensionality Bottlenecks Can Cause Model Crashes: STAIG’s 64-Dimensional Embeddings Could Compress 1536-Dimensional Visual Features by 96%, producing embeddings with a standard deviation of 0.12, which are almost indistinguishable. Spatial refinement could eliminate a few clusters, while majority voting could resolve other issues. Other fusion methods (such as attention mechanisms or qformer: 256-512 dimensions) wouldn’t have these problems, indicating that 64-dimensional embeddings are inherently insufficient.#

  • Key requirements for VLA end-to-end integration: Ensure that policy observation input formats are correct, especially in the camera image transmission section. It’s necessary to explicitly use the include_images=True parameter in the state_extractor.extract() function and provide camera_names information for all policies that rely on cameras (such as VLA and VisuoMotor BC).
  • When integrating third-party machine learning libraries (such as openpi and robomimic), it’s essential to check the actual API signatures using inspect.signature() and read the sample code from the source repository. Documentation often lacks completeness or is outdated. The openpi library’s create_trained_policy() provides advanced encapsulation, but manually using the low-level API can be error-prone.
  • Pi0 (LIBERO checkpoint) has poor generalization ability in the robosuite’s PickPlace task (failure rate of 100%). The main failure modes include joint_limit_approach (47%) and grasp_wrong_pose (30%). This indicates that VLA’s cross-task generalization still presents challenges, but these failure data can provide valuable natural error data for benchmark testing.
  • Capturing natural errors is more efficient than injecting them: 50 iterations generated 150 scenarios, while the injection method only generated 3 scenarios. The natural error mode is more diverse and realistic.
  • Plugin-based architecture pattern in the benchmark system: Detectors, classifiers, and injectors all use the basic ABC+ registry pattern to achieve scalability. The error classification system uses CLASSIFIER_REGISTRY to register 19 classifiers, making it easy to add new error types without modifying the core logic.
  • Document verification requires checking the actual code registry and enumeration values, not just reading the document strings or previous documents. DETECTOR_REGISTRY, CLASSIFIER_REGISTRY, and ErrorType enumeration are real reference standards. Through systematic searching, residual errors can be avoided.
  • Communication architecture across Conda environments: VLA servers use TCP+pickle protocol to separate model inference (openpi/openpi05 environment) and benchmark framework (mimicgen_env). Length-prefixed frames prevent message fragmentation, and action chunking reduces the number of inference calls. Environment isolation is crucial for VLA/robot projects due to conflicting dependencies.
  • Preprocessing of observation data must handle various input formats gracefully: StateExtractor (internal framework), raw env.step() (robosuite), and model-specific preprocessing (VLA servers). A robust key lookup mechanism plus a fallback mechanism can prevent silent failures. The dictionary format of observation data varies depending on the source, so a unified preprocessing layer is needed.
  • Spatial majority voting refinement (nearest neighbor with a radius of 15) may amplify rather than fix clustering errors, especially when the underlying embeddings are already compressed. Refinement should be based on embedding quality metrics (for example, if the contour score is below a threshold, refinement should be disabled).
  • The dual VLM backend design (Gemini API + Claude Code CLI) offers flexibility: The API is suitable for batch automation operations, while the CLI is suitable for interactive deep analysis and debugging. Both complement each other and can meet different usage needs.
  • To generate natural errors, VLA integration requires architectural adjustments: (1) Adopt a policy server mode across Conda environments; (2) Support camera observations in environment wrappers; (3) Preprocess observation data (convert from quaternions to angular coordinates, BGR to RGB, action chunking); (4) Dual iteration mode (synthetic injection and natural capture). This is very different from the synthetic perturbation method used in demonstration replay.
  • The system’s code library exploration strategy (starting from infrastructure checks, to model file locations, to confirmation of configuration parameters, and finally to user decision confirmation) can effectively support complex multi-step project planning, avoiding overlooking key dependencies.
  • In data structure design, backward compatibility is an important consideration: By introducing optional and graceful downgrading of the new conversation_summaries field, it’s ensured that old reports can still be used normally. This approach applies to evolving data formats.
  • Out of the 24 error types in the error classification system, only 5 were triggered in actual VLA iterations: joint_limit_approach, grasp_wrong_pose, misalignment_pre_grasp, overshoot, and premature_release. This indicates that certain error types require specific scenarios to manifest.

Practical Experience#

  • Using GPUs for batch processing for visualization (5 GPUs, 20 videos, zero failures) demonstrated the value of parallel infrastructure. The subprocess.Popen + GPU allocation pattern is ideal for parallel tasks.
  • The subtleties of handling false values in Python: 0 or default would be evaluated as default because 0 is a false value, which would cause errors with counters indexed by 0. PrematureReleaseClassifier always returns 0 when starting to capture at step 0. The solution is to explicitly use is None to check for optional 0-indexed values.

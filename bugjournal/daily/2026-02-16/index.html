Problems and Solutions#

Key Issues#

1. STAIG fusion produced only 2 clusters instead of 5, ARI of 0.1149 indicates severe crashesSolution: Identified the root cause: a 64-dimensional embedding bottleneck caused 1536-dimensional UNI features to compress by 96% (standard deviation of 0.12), and then spatial refinement algorithm’s radius-15 majority voting mechanism eliminated minority clusters. Proposed dimension ablation studies to test alternative solutions with 128/256/512 dimensions, so as to compare with other fusion methods.#

Key Insight: Cascading failures exacerbated the problem: embeddings generated by GCN training collapsed almost completely, and subsequent spatial smoothing completely eliminated remaining minor clusters. The 64-dimensional bottleneck was crucial—other fusion methods using 256-512 dimensions did not suffer from such problems.

2. VLA server initially lacked camera images for observation data, leading to “observation/image” KeyError#

Solution: Fixed the get_current_obs() function in rollout_generator.py by adding the include_images parameter; detected the PolicyServerAdapter type in generate_from_single_rollout() and capture_natural_errors() functions, and passed the include_images=True parameter when camera_names included [‘agentview’,‘robot0_eye_in_hand’].

Key Insight: VLA policies require visual input, but state_extractor.extract() does not include images by default; it was necessary to explicitly identify policy types and set the include_images flag. This was a general issue affecting all policies that relied on cameras (VLA, VisuoMotor BC).

3. VLA server used incorrect openpi API—manual loading of norm_stats caused model initialization failure#

Solution: Replaced manual loading with openpi’s create_trained_policy(train_config=get_config(config_name), checkpoint_dir=…) method, which can automatically handle norm_stats, transformation, and model architecture. Added the –config_name CLI parameter and IMAGE_KEY_MAP to achieve correct camera key mapping.

Key Insight: The openpi library provides an advanced create_trained_policy() function that encapsulates all model loading complexities. Manual use of low-level APIs is error-prone and problematic with library upgrades.

4. Pi0 (LIBERO checkpoint) achieved a 100% failure rate during natural_capture deployment process#

Solution: Identified the failure pattern distribution: joint_limit_approach was 47%, grasp_wrong_pose was 30%, misalignment_pre_grasp was 17%, overshoot was 5%, premature_release was 1%; recorded these situations as natural error data; determined that it was necessary to produce solutions specific to PickPlace checkpoints or domain adaptability.

Key Insight: Pre-trained VLA has limited generalization ability; performance declined significantly across tasks (LIBERO→robosuite PickPlace). However, these failure patterns are valuable natural error data and meet the benchmarks’ objectives for natural error capture.

5. VLA integration seemed complete (code and tests passed), but failed to run properly in actual deployment#

Solution: In-depth investigation revealed obstacles: incorrect openpi API usage, lack of norm_stats.json processing, incomplete checkpoint paths, failure to support –policy vla_server flag. A detailed fix plan was developed and all corrections were systematically implemented.

Key Insight: “Tests passed” does not equal “system is running properly”—before claiming completion, integration testing with actual dependencies (model checkpoints, external APIs) was required. Synthetic unit tests could verify code correctness but could not guarantee end-to-end functionality.

6. _preprocessobs() failed because the state information dictionary returned by StateExtractor differed from the raw robosuite observation data#

Solution: Reconstructed the preprocessobs() function to detect input formats and standardize them into common structures {“state”: […], “images”: {…}. Added a powerful key lookup function and implemented correct floating-point to uint8 image conversion.

Key Insight: The format of observation dictionaries depends on the source: StateExtractor.extract() returns keys centered around EEF, while env.step() returns raw robosuite observation data with robot0 prefix. A unified preprocessing method was needed.

7. Documentation contained more than 15 factual errors that contradicted the actual code repository status, involving multiple files#

Solution: Conducted a systematic registry-based verification (DETECTOR_REGISTRY, CLASSIFIER_REGISTRY, ErrorType enumeration), corrected all errors through targeted edits to 3 files, and ran test suites to confirm no failures (79/79 passed).

Key Insight: Even the code’s documentation strings had errors. The real source was the registered code, not the documentation itself. Verification workflow: used grep old values after editing to ensure thorough cleanup.

  • Natural error capture is more efficient than injection-based methods: 50 iterations generated 150 scenarios (about 3 scenarios per iteration), while injection-based methods produced only 3 scenarios in 10 iterations (about 0.3 scenarios per iteration). Natural error patterns are more diverse and realistic.

The error classification system implements registration of 19 classifiers through CLASSIFIER_REGISTRY, allowing for easy addition of new error types without modifying the core logic.
  • Documentation verification requires checking the actual code registry and enumeration values, rather than just reading document strings or previous documentation. The DETECTOR_REGISTRY, CLASSIFIER_REGISTRY, and ErrorType enumerations are reliable references. Systematic editing can prevent residual errors.
  • Communication architecture across conda environments: VLA servers use TCP+pickle protocol to separate model inference (openpi/openpi05 environment) from the benchmark framework (mimicgen_env). Length-prefixed frame format prevents message fragmentation, and action chunking reduces the number of inference calls. Environment isolation is crucial for VLA/robot projects due to dependency conflicts.
  • Observation data preprocessing must handle multiple input formats gracefully: StateExtractor (internal framework), raw env.step() (robosuite), and model-specific preprocessing (VLA server). A robust key lookup mechanism with fall-back strategies can prevent silent failures. The dictionary format of observation data varies depending on the source, so a unified preprocessing layer is required.
  • Spatial majority voting refinement operations (nearest neighbor with radius 15) may amplify rather than solve collapsed clustering errors. Refinement should be based on metrics of embedding quality (e.g., disabling refinement if contour scores are below a certain threshold).
  • The dual VLM backend design (Gemini API + Claude Code CLI) provides flexibility: The API is suitable for batch automation, while the CLI is ideal for interactive depth analysis and debugging. They complement each other to meet different usage scenarios.
  • Adjustments were made to the architecture for generating natural errors in VLA integration: (1) policy server mode across conda environments, (2) camera observation support in environment包装器, (3) observation data preprocessing (quaternions → axis-angle conversion, BGR→RGB, action chunking), and (4) dual iteration mode (synthetic injection + natural capture). This differs significantly from the demonstration-play synthetic perturbation method.

Practical Experience#

  • Using GPUs for batch processing to achieve visualization results (5 GPUs, 20 videos, zero failures) demonstrated the value of parallel infrastructure. The subprocess.Popen + GPU allocation pattern is suitable for tasks with parallelism.
  • The subtleties in handling null values in Python: 0 or default would be evaluated as default, as 0 is a null value, which could lead to errors with counters indexed by 0. The PrematureReleaseClassifier always returns 0 when starting capture at step 0. The solution is to explicitly use is None to check for optional values indexed by 0.

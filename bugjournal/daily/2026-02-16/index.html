Problems & Solutions#

Critical Issues#

1. STAIG fusion produces only 2 clusters instead of 5, with ARI=0.1149 indicating severe collapse#

Solution: Root cause identified: 64-dim embedding bottleneck compresses 1536-dim UNI features by 96% (std=0.12), then spatial refinement’s radius-15 majority voting eliminates minority clusters. Proposed dimension ablation study to test 128/256/512-dim alternatives matching other fusion methods.

Key Insight: Cascading failures compound: GCN training produces near-collapsed embeddings, then spatial smoothing eliminates remaining small clusters entirely. The 64-dim bottleneck is critical—other fusion methods using 256-512 dims don’t exhibit this pathology.

2. VLA server initial obs missing camera images, causing ‘observation/image’ KeyError#

Solution: Fixed rollout_generator.py _get_current_obs() to add include_images parameter; detect PolicyServerAdapter type in _generate_from_single_rollout() and capture_natural_errors() to pass include_images=True with camera_names=[‘agentview’,‘robot0_eye_in_hand’]

Key Insight: VLA policies require visual input, but state_extractor.extract() doesn’t include images by default; need explicit policy type detection and include_images flag. This is a generic issue affecting all camera-dependent policies (VLA, VisuoMotor BC).

3. VLA server used wrong openpi API - manually loaded norm_stats causing model initialization failures#

Solution: Replaced manual loading with openpi’s create_trained_policy(train_config=get_config(config_name), checkpoint_dir=…) which handles norm_stats, transforms, and model architecture automatically. Added –config_name CLI arg and IMAGE_KEY_MAP for correct camera key mapping.

Key Insight: The openpi library provides high-level create_trained_policy() function that encapsulates all model loading complexity. Using low-level APIs manually is error-prone and breaks when the library evolves.

4. Pi0 (LIBERO checkpoint) achieved 100% failure rate on PickPlace task during natural_capture rollouts#

Solution: Identified failure mode distribution (joint_limit_approach 47%, grasp_wrong_pose 30%, misalignment_pre_grasp 17%, overshoot 5%, premature_release 1%); recorded as natural error data; determined need for PickPlace-specific checkpoint or domain adaptation for production use.

Key Insight: Pretrained VLA generalization is limited; cross-task distribution (LIBERO→robosuite PickPlace) causes significant performance degradation. However, these failure modes are valuable natural error data, aligning with benchmark’s natural error capture goals.

5. VLA integration appeared complete (code + tests passing) but discovered non-operational when attempting actual rollout#

Solution: Deep exploration revealed blockers: wrong openpi API usage, missing norm_stats.json handling, incomplete checkpoint paths, unsupported –policy vla_server flag. Created detailed fix plan and systematically implemented all corrections.

Key Insight: ‘Tests passing’ ≠ ‘system operational’—integration testing with real dependencies (model checkpoints, external APIs) essential before claiming completion. Synthetic unit tests validate code correctness but not end-to-end functionality.

6. _preprocess_obs() failed when StateExtractor returned state_info dict (different format from raw robosuite obs)#

Solution: Refactored _preprocess_obs() to detect input format and normalize to common structure {“state”: […], “images”: {…}}. Added robust key lookup with fallback patterns and proper float-to-uint8 image conversion.

Key Insight: Observation dict format changes depending on source: StateExtractor.extract() returns state_info with EEF-centric keys, while env.step() returns raw robosuite obs with robot0_* prefixes. Need unified preprocessing.

7. Documentation contained 15+ factual errors contradicting actual codebase state across multiple files#

Solution: Performed systematic registry-based verification (DETECTOR_REGISTRY, CLASSIFIER_REGISTRY, ErrorType enum), corrected all instances via targeted edits across 3 files, ran test suite to verify no breakage (79/79 passing).

Key Insight: Even code’s own docstrings were wrong. Truth source is the registered code, not docs. Verification workflow: grep all old values after edits to ensure complete cleanup.

Learnings#

Key Learnings#

  • Embedding dimension bottlenecks cause catastrophic model collapse: STAIG’s 64-dim output compresses 1536-dim vision features by 96%, producing embeddings with std=0.12 that are nearly indistinguishable. Spatial refinement then eliminates minority clusters via majority voting. Other fusion methods (attention/qformer: 256-512 dim) don’t exhibit this pathology, suggesting 64-dim is fundamentally insufficient.
  • VLA end-to-end integration critical requirements: ensure policy observation input format is correct, especially camera image passing; need explicit include_images=True in state_extractor.extract() and camera_names specification for all camera-dependent policies (VLA, VisuoMotor BC).
  • When integrating third-party ML libraries (openpi, robomimic), always check actual API signatures via inspect.signature() and read example code in source repo. Documentation is often incomplete or outdated. The openpi library’s create_trained_policy() provides high-level encapsulation; using low-level APIs manually is error-prone.
  • Pi0 (LIBERO checkpoint) demonstrates poor generalization on robosuite PickPlace task (100% failure rate); dominant failure modes: joint_limit_approach (47%) and grasp_wrong_pose (30%). This reveals VLA cross-task generalization remains challenging, but these failures provide valuable natural error data for benchmark purposes.
  • Natural error capture (natural_capture mode) is more efficient than injection mode: 50 rollouts generated 150 scenes (~3 per rollout), while injection mode 10 rollouts generated only 3 scenes (~0.3 per rollout); natural failure modes are more diverse and realistic.
  • Plugin-based architecture pattern in benchmark systems: detectors, classifiers, injectors all use base ABC + registry pattern for extensibility. Error classification system demonstrates this with 19 classifiers registered via CLASSIFIER_REGISTRY, enabling easy addition of new error types without modifying core logic.
  • Documentation verification requires checking actual code registries and enums, not just reading docstrings or prior docs. DETECTOR_REGISTRY, CLASSIFIER_REGISTRY, and ErrorType enum are ground truth. Systematic grep after all edits prevents residual errors.
  • Cross-conda-env communication architecture: VLA server uses TCP+pickle protocol to decouple model inference (openpi/openpi05 env) from benchmark framework (mimicgen_env). Length-prefix framing prevents message fragmentation, action chunking reduces inference calls. Environment isolation is critical for VLA/robotics projects due to conflicting dependencies.
  • Observation preprocessing must handle multiple input formats gracefully: StateExtractor (framework-internal), raw env.step() (robosuite), and model-specific (VLA servers). Robust key lookup with fallbacks prevents silent failures. The observation dict format changes depending on source—need unified preprocessing layer.
  • Spatial majority voting refinement (radius=15 nearest neighbors) can amplify rather than fix clustering errors when underlying embeddings are already collapsed. Refinement should be conditional on embedding quality metrics (e.g., disable if silhouette score < threshold).
  • Dual VLM backend design (Gemini API + Claude Code CLI) provides flexibility: API suits batch automation, CLI suits interactive deep analysis and debugging; complementary coverage of different use cases.
  • VLA integration for natural error generation requires architectural shifts: (1) policy server pattern for cross-conda-env execution, (2) camera observation support in environment wrapper, (3) observation preprocessing (quaternion→axis-angle, BGR→RGB, action chunking), (4) dual rollout modes (synthetic injection + natural capture). Fundamentally different from demo-replay synthetic perturbation approach.
  • Systematic codebase exploration strategy (infrastructure check → model file location → config parameter confirmation → user decision confirmation) effectively supports complex multi-step project planning, avoiding missing critical dependencies.
  • Backward compatibility is key consideration in data structure design: new conversation_summaries field through optionality and graceful degradation (missing field → no section rendering) ensures old reports continue working. This pattern applies broadly to evolving data formats.
  • Error classification system’s 24 error types only triggered 5 types in actual VLA rollouts (joint_limit_approach, grasp_wrong_pose, misalignment_pre_grasp, overshoot, premature_release), indicating some error types require specific scenarios to manifest.

Practical Learnings#

  • GPU batch processing for visualization (5 GPUs, 20 videos, zero failures) demonstrates value of parallel infrastructure. The subprocess.Popen + GPU assignment pattern scales well for embarrassingly parallel tasks.
  • Falsy value handling subtlety in Python: 0 or default evaluates to default because 0 is falsy, causing bugs in 0-indexed counters. PrematureReleaseClassifier had step - (self._grasp_start_step or step) always returning 0 when grasp started at step 0. Solution: explicit is None checks for optional 0-indexed values.

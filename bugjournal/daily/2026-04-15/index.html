Problems & Solutions#

Critical Issues#

1. Robosuite missing critical patches (z_rotation parameter, get_bounding_box_half_size, total_size, locations_relative_to_corner) required by MimicGen, causing environment creation failures#

Solution: Reapplied shared/mimicgen_workspace/robosuite_patches/robosuite_recovery.patch to installed robosuite package using patch -p1; verified all parameters exist in target class signatures

Key Insight: MimicGen requires patched robosuite with custom APIs not present in official releases; patched dependencies must be re-patched after pip install overwrites them; cluster uses local patched copies in shared/mimicgen_workspace/ rather than upstream versions

2. script_utils.py create_env function silently swallowed exceptions with ’except Exception: continue’, showing only generic ‘Could not create env’ error#

Solution: Modified exception handling to capture last_error and re-raise with original traceback context in RuntimeError, revealing the actual MuJoCo PNG loading error

Key Insight: Exception handling with bare ’except: continue’ or ’except Exception: pass’ makes debugging extremely difficult and should never be used in diagnostic/setup code; always capture the actual exception, log it, and include it in re-raised errors

3. Shared/ directory with critical dependencies was gitignored, making environment non-reproducible for other users or cluster deployment#

Solution: Tracked modified mimicgen directly in repo (13MB); created patch file system for robosuite to avoid 587MB commit; updated setup_env.sh to auto-apply patches after upstream installation

Key Insight: External dependencies with local modifications should be tracked in-repo (if small) or via patch files (if large) rather than blindly gitignored, especially for reproducible research environments; unified diff patches minimize repository size while preserving reproducibility

4. Pip editable install of mimicgen shadowed by residual site-packages/mimicgen/ directory from previous install, causing import failures#

Solution: Removed stale site-packages/mimicgen/ directory; pip editable install then correctly mapped to src/mimicgen

Key Insight: Editable pip installs on NTFS cross-filesystem mounts can leave residual site-packages directories that take precedence over editable path mappings; always verify import resolution matches pip show location by checking path and file attributes

5. 776 file differences between installed robosuite and upstream turned out to be mostly CRLF vs LF line ending noise, obscuring 13 real code changes#

Solution: Used diff –strip-trailing-cr to filter line ending differences, revealing only 13 files with actual code modifications

Key Insight: Cross-platform development creates false positives in file comparison due to line ending differences; always use –strip-trailing-cr or -w flags when comparing across Windows/Linux platforms

General Issues#

6. Pip-installed mimicgen missing XML assets (coffee_pod.xml, coffee_body.xml) required by environment definitions#

Solution: Cloned mimicgen repo and manually copied assets/ directory to site-packages

Key Insight: Setup.py may not package all necessary data files; always verify critical non-Python assets are included in distribution

7. Missing robomimic dependency caused import failures in mimicgen config modules#

Solution: Installed robomimic with pip install robomimic –no-deps to bypass egl_probe build failure (EGL rendering not needed for SpaceMouse collection)

Key Insight: Use –no-deps to install packages when optional rendering dependencies fail to build but core functionality remains intact

8. Robosuite installation upgraded mujoco to 3.7.0, breaking API compatibility with robosuite 1.4.1#

Solution: Force reinstalled mujoco 2.3.2 with –no-deps flag to prevent transitive upgrades

Key Insight: Use –no-deps when reinstalling pinned dependencies to prevent package managers from upgrading transitive dependencies

9. MuJoCo failed to load ceramic.png texture (PNG file load error) because shared/mimicgen_workspace/mimicgen/ lacked assets/textures/ directory present in src/mimicgen upstream#

Solution: Copied ceramic.png from src/mimicgen/mimicgen/models/robosuite/assets/textures/ to shared/mimicgen_workspace/mimicgen/mimicgen/models/robosuite/assets/textures/

Key Insight: When MuJoCo environments fail with asset loading errors, check both Python import paths and relative asset paths (meshes, textures) referenced in XML model definitions

Learnings#

Key Learnings#

  • MimicGen requires patched robosuite with custom APIs (get_bounding_box_half_size, total_size, locations_relative_to_corner, z_rotation) not present in official releases; version pinning in setup scripts may refer to local patched copies in shared/ directories rather than upstream commits
  • Editable pip installs on NTFS cross-filesystem mounts can leave residual site-packages directories that shadow editable path mappings; always verify import resolution matches pip show location by checking both path and file attributes; patched dependencies must be re-patched after pip install overwrites them
  • Exception handling with bare ’except: continue’ or ’except Exception: pass’ makes debugging extremely difficult and should never be used in diagnostic/setup code; always capture the actual exception, log it, and include it in re-raised errors for proper diagnosis
  • External dependencies with local modifications should be tracked in-repo (if small) or via patch files (if large) rather than blindly gitignored; unified diff patches are the correct tool for tracking modifications to large upstream projects - they minimize repository size, document exact changes, and enable automated application
  • Cross-platform line ending differences (CRLF vs LF) can create massive false positives in file comparisons; always use –strip-trailing-cr or -w flags when comparing across Windows/Linux platforms to reveal actual code changes
  • When robosuite environments fail to load, check three layers: Python import paths (pip show, sys.path), robosuite registration (REGISTERED_ENVS), and MuJoCo asset file paths (textures, meshes, XML relative references)
  • Pip packages may exclude data files (XML assets, configs); always verify critical non-Python files are packaged correctly in distribution; setup.py may miss necessary asset directories

Practical Learnings#

  • Use –no-deps when reinstalling pinned dependencies or installing packages with optional rendering dependencies that fail to build but aren’t needed for core functionality
  • Presence of .orig and .rej files indicates a patch was attempted, but empty diffs mean the patches may have already been successfully applied (artifacts weren’t cleaned up)

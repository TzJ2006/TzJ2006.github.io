Problems and Solutions#

Key Issues#

1. scan_cluster refine + concat (ARI=0.313) actually shows lower results than direct concatenation (ARI=0.387). The two-phase fusion effect did not meet expectations.#

Solution: Continue testing other 7 fusion strategies; or adjust the refinement hidden_dim to its current default of 256; or try stego_refine/byol_spatial as the refining method.

Key Insight: SCAN lost the original feature diversity required for multimodal fusion during the process of compressing 1536d to 256d. The self-supervised clustering optimization goals were inconsistent with the downstream fusion tasks, and self-supervised compression did not always enhance subsequent fusion.

2. VLA evaluation used non-target task-specific checkpoints (pi0_libero fine-tuning version and pi05_base pre-training version, not PickPlace-exclusive models), which affected the validity of the experiment.#

Solution: Clearly define it as a cross-domain generalization evaluation rather than an in-task evaluation. State the rationality of the experimental setup in the paper. Add a comparison experiment after fine-tuning in M15 to fully demonstrate the usefulness of the dataset.

Key Insight: Using non-target task-specific checkpoints tests VLA’s zero-shot/cross-domain recovery capabilities. In scientific experiments, key assumptions should be actively exposed and stated; otherwise, credibility may be questioned by reviewers.

3. Lerobot 0.1.0 is not compatible with datasets >=4.0 (TypeError: torch.stack argument ’tensors’ must be tuple of Tensors, not Column).#

Solution: Downgrade the datasets from version 4.4.1 to 3.6.0 (<4.0). Reason: In dataset 4.0, the return value of dataset[‘column’] was changed from a list to a Column object, and Lerobot expected a list/tuple of tensors.

Key Insight: Lerobot 0.1.0 has strict requirements for dataset versions. It needs to be pinned to version <4.0. When configuring the environment, it is necessary to check the dependency compatibility matrix rather than installing the latest version by default.

4. Pi0.5 encountered OOM on tianhe cluster (failure to allocate 150MB of GPU memory), resulting in complete blockage of the baseline evaluation.#

Solution: Not yet resolved. It is necessary to apply for larger memory GPUs or reduce the batch size. It is recommended to test the memory peak value with a small batch first.

Key Insight: The memory requirement should be verified in advance before conducting large model experiments. Since the scale of Pi0.5 model exceeds the current GPU configuration, it should be included in the resource planning stage during experimental design.

General Issues#

5. Unstable access to Slurm GPU nodes: SSH connections are unreliable; srun without the –overlap parameter causes commands to hang; multiple partition submissions are rejected.#

Solution: Standard workflow: source set-XY-I.sh → squeue → srun –jobid= –overlap. Directly connecting to GPU nodes via SSH should be prohibited. Always add –overlap when submitting new tasks.

Key Insight: In a shared HPC environment, –overlap is a key parameter for running new commands on top of existing interactive jobs. Issues with partition permissions should be quickly addressed by asking the user for the correct account and partition information rather than trying everything out.

6. HuggingFace official downloads failed due to proxy issues (Squid 503), and Python download scripts could not connect.#

Solution: Use hf-mirror.com + wget as an alternative download source. URL format: https://hf-mirror.com/datasets/{repo_id}/resolve/main/{file_path}，通过集群HTTP代理可达40-200MB/s

Key Insight: The standard solution for accessing HuggingFace on Chinese mainland HPC clusters is the hf-mirror.com mirror site, which even has a faster speed than the official API. It should be used as the default solution for cluster downloads rather than an alternative.

7. Pi0 VLA server port conflicts and SLURM node permissions caused GPU task blockages.#

Solution: For port conflicts: check lsof -i:5555 in advance, and include a logic to detect and automatically switch ports in the startup script. For SLURM permissions: quickly ask the user for the correct account and partition information without trying everything out.

Today’s Achievements#

Key Achievements#

  • Two-stage fusion is not necessarily better than one-stage fusion: After compressing visual embeddings from 1536d to 256d with scan_cluster, the multi-modal ARI of 0.313 was lower than direct concatenation of 0.387, indicating that self-supervised compression reduces the original feature diversity required for fusion tasks. There are inconsistencies between the objectives of self-supervised clustering optimization and those of downstream fusion tasks.
  • For VLA baseline evaluation, it is necessary to clearly state the source of checkpoints: Models fine-tuned with non-target tasks (pi0_libero, pi05_base) were evaluated for zero-shot cross-domain recovery capabilities rather than task-specific performance. The paper must clearly specify this experiment setup and add comparisons after fine-tuning in subsequent experiments to fully demonstrate the usefulness of the dataset.
  • Lerobot 0.1.0 is strictly incompatible with datasets >=4.0: In datasets 4.0, the value returned by dataset[‘column’] was changed from a list to a Column object, causing TypeError when using torch.stack(). It is necessary to pin datasets <4.0 (recommended version 3.x). When configuring the environment, it is necessary to check the dependency versions in advance.
  • MimicGen and LIBERO have fully compatible obs/action formats (84×84 images, 8D states, 7D actions), allowing direct reuse of OpenPI’s LeRobotLiberoDataConfig. During training, the ResizeImages transform is used to dynamically resize to 224×224 without the need for pre-resizing or custom data loaders.
  • Multi-modal performance pattern in section 151673: The best ARI for image-only was approximately 0.303 (scan_cluster), for gene-only it was approximately 0.31 (PCA), and for multi-modal concatenation it was 0.387. The two modalities complement each other, with benefits coming from non-overlapping information. It is necessary to verify the memory requirements in advance before large model experiments to avoid OOM causing delays in the entire evaluation process.
  • OpenPI’s JAX training naturally supports multi-GPU data parallelism. Just specify a list of GPUs using CUDA_VISIBLE_DEVICES, and JAX will automatically build a 2D mesh parallel without any modifications to TrainConfig. The batch_size should be divisible by the number of GPUs. If memory is insufficient, you can add –fsdp-devices.
  • The standard way to access HuggingFace on Chinese mainland HPC clusters is through the hf-mirror.com mirror site. URL format: https://hf-mirror.com/datasets/{repo_id}/resolve/main/{file_path}，通过wget+集群HTTP代理可达40-200MB/s，应作为默认方案而非备选。

Practical Achievements#

  • Slurm’s –overlap allows running new commands over existing interactive jobs, making it a key technique for calling cluster GPU nodes in Claude Code environment. The project’s scenario library has increased from 454 to 649 (+43%) since documentation records were created.

Problems & Solutions#

Critical Issues#

1. Pi0.5 VLA server hits GPU OOM when loading model (150MB allocation failure logged on 2026-02-21 04:17), blocking M14 baseline evaluation of 7,800 episodes (649 scenes × 4 policies × 3 seeds)#

Solution: Identified blocker but did not fix during session - requires either switching to GPU with larger memory or reducing batch size. Documented issue for follow-up; also initiated alternative path of fine-tuning pi05_base on MimicGen tasks to create domain-matched baseline

Key Insight: M14 baseline infrastructure complete and ready but cannot start until Pi0.5 memory issue resolved. Created parallel path (fine-tuning) to establish appropriate baseline while debugging OOM issue

2. VLA checkpoint selection ambiguity: pi0_libero (LIBERO-tuned) and pi05_base (generic pretrained) both domain-mismatched for PickPlace task, potentially weakening dataset usefulness claims#

Solution: User paused execution to question checkpoint relevance. Identified need for explicit decision: use existing checkpoints as-is for generic baseline or fine-tune on PickPlace/Phoenix data first. Initiated fine-tuning pipeline as solution path

Key Insight: Baseline evaluation requires careful model selection: domain-mismatched checkpoints may underperform due to distribution shift rather than dataset quality, making usefulness claims invalid. Need task-aligned baseline for fair evaluation

3. Vision Refinement scan_cluster method achieved ARI 0.313, lower than direct concat (0.387), indicating 256d compression loses information valuable for multimodal fusion despite improving single-modality clustering (0.303)#

Solution: Continue testing remaining 6 fusion methods (attention, adaln_attention, llava_mlp, qformer, etc.) to determine if certain strategies better utilize refined 256d embeddings; consider adjusting hidden_dim or testing alternative refinement methods (stego_refine, byol_spatial)

Key Insight: Self-supervised clustering methods optimize for cluster separability, which may not align with multimodal fusion’s information complementarity objectives. Dimensionality reduction can filter out features valuable for gene-vision complementarity even if less relevant for image-only clustering

4. GPU allocation strategy unclear: initially assumed only GPU 0 available despite 8 A800 GPUs existing on Slurm node#

Solution: Human clarified to check all GPUs via nvidia-smi and use any with 50%+ free memory. Updated plan to dynamically check memory usage and allocate CUDA_VISIBLE_DEVICES accordingly, enabling parallel task execution

Key Insight: In shared HPC environments, availability ≠ idle. Need memory-based criteria (50% threshold) rather than binary idle check. Multiple tasks can share node if memory allows, maximizing cluster utilization

5. SSH connections to GPU nodes (an49, an53, etc.) unstable and disconnect frequently, interrupting long-running experiments#

Solution: Switched to Slurm-based workflow documented in CLAUDE.md: use squeue to find active jobs, then srun –jobid= to attach to allocated node. Avoids direct SSH and leverages Slurm’s job management layer

Key Insight: Slurm job attachment (srun –jobid) more stable than direct SSH because designed for HPC workloads and respects cluster’s resource management layer. Job IDs persist across sessions enabling reattachment

9. QFormer training shows high memory usage (19.4GB) but extremely low GPU utilization (4%, 55W), causing slow training progress#

Key Insight: High memory occupancy ≠ high GPU load. Learnable query mechanisms result in training bottlenecked by CPU preprocessing while GPU cores remain mostly idle, with VRAM used for storage rather than computation

Learnings#

Key Learnings#

  • Two-stage fusion (self-supervised refinement → multimodal integration) creates new experimental dimension bridging image-only and multimodal methods; but initial testing shows refined embeddings not universally superior to raw embeddings across all fusion strategies - optimization objective alignment critical
  • Baseline model selection critically affects experimental validity. Using domain-mismatched checkpoints (LIBERO-tuned on PickPlace task) weakens usefulness claims. Must explicitly validate or fine-tune baselines to ensure fair evaluation
  • Vision Refinement pipeline implementation proves feasibility of inserting intermediate processing steps in run_benchmark.py: adding ~50 lines at appropriate location (post-vision encoding, pre-fusion) enables entirely new experimental paradigm while maintaining full compatibility with existing caching/naming/logging systems
  • HPC cluster GPU access via Slurm job attachment (srun –jobid) more stable than direct SSH because leverages cluster’s resource management layer and designed for long-running scientific workloads. Job IDs persist across sessions enabling reattachment even after disconnection
  • Phase I/Phase II research project structure (infrastructure building → validation experiments) benefits from clear milestone tracking. Error Recovery Benchmark’s M1-M11 completion (649 scenes) provides solid foundation for M14 baseline evaluation, even though M14 currently blocked by Pi0.5 OOM
  • In HPC clusters, GPU availability should be judged by memory headroom (50%+ free), not idle status. Multiple jobs can coexist on same node if memory allows, enabling parallel task execution and maximizing cluster utilization
  • Dependency topology diagrams should precede execution plans. Clear visualization of M14 as critical path blocker shaped resource allocation decisions and timeline estimates, enabling more strategic scheduling
  • Collector resume feature enables fault-tolerant long-running evaluations. M14 sanity check validated that interrupted evaluations can restart from last checkpoint, critical for 6+ hour full runs
  • Project maintenance ‘cleanup day’ pattern effective: use 3 parallel Explore agents for comprehensive project survey, then systematic cleanup (delete redundancy, relocate logs, reorganize docs, update references/Memory), resulting in simplified root directory, clear documentation hierarchy, traceable plans
  • Systematic documentation audit via /init skill highly effective at catching long-standing gaps accumulated during iterative development, especially for multi-tool projects where each tool documented independently
  • Error detection pipeline (20 classifiers, PolicyErrorMonitor) works reliably at scale. Detected 37-74 errors per episode in sanity check, proving taxonomy implementation production-ready
  • MimicGen-to-LeRobot conversion pipeline requires careful handling of flat vs nested HDF5 layouts. Phoenix paper uses 84×84 images resized to 224×224 for training, must preserve this preprocessing in conversion
  • JAX auto-detects all visible GPUs for multi-GPU training when batch_size divisible by GPU count. CUDA_VISIBLE_DEVICES=0,1,3,4 with batch_size=64 enables efficient 4-GPU parallelism without explicit configuration
  • Explore agents excel in large project organization tasks: can recursively traverse directories, read file headers for quick content identification, count code lines, compare documents for redundancy - order of magnitude more efficient than manual exploration
  • In large project cleanup efforts, user-controlled planning (bypassing ExitPlanMode approval) can be more efficient when user has clear objectives and willing to own plan quality. AI’s role becomes execution rather than design

Practical Learnings#

  • When projects accumulate substantial work history, batch historical report generation valuable for establishing searchable archive, even though meta-work rather than primary development. gadget project now has consolidated reports covering 3 weeks of multi-project activity
  • QFormer and other learnable-query-based fusion methods exhibit ‘high VRAM + low GPU utilization’ during training: parameters and intermediate results occupy memory but actual computation (cross-attention) is lightweight, with most time in CPU-intensive k-NN graph construction and loss calculation. This is architectural characteristic, not performance bug

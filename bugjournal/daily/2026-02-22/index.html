Issues and Solutions#

Key Issues#

1. Pi0.5 VLA server encountered a GPU memory shortage issue when loading models (150MB allocation failed on 2026-02-21 04:17), hindering the M14 baseline evaluation for 7,800 episodes (649 scenarios × 4 strategies × 3 seeds).#

Solution: The issue was identified, but could not be resolved during the meeting—it is necessary to switch to GPUs with more memory or reduce the batch size. This issue has been recorded for further processing; at the same time, fine-tuning of pi05_base on the MimicGen task has begun to create a baseline that matches the domain.

2. There was ambiguity in the VLA checkpoint selection: both pi0_libero (LIBERO fine-tuning version) and pi05_base (general pre-trained version) were not suitable for this domain in the PickPlace task, which may weaken the validity statement of the dataset.#

Solution: Users paused the execution to confirm the relevance of the checkpoints. It is necessary to decide whether to use the existing checkpoints directly for the general baseline, or to first fine-tune the PickPlace/Phoenix data. The fine-tuning process has been started as a solution. Key insight: The baseline evaluation requires careful selection of models: due to distribution bias rather than dataset quality issues, checkpoints that do not match the domain may perform poorly, thereby invalidating statements about the model’s usefulness. A baseline optimized for the task needs to be used to evaluate the model’s performance.

3. The ARI value obtained from the Vision Refinement scan clustering method was 0.313, which is lower than the 0.387 value of the direct concatenation method. This indicates that while 256-dimensional compression improved the performance of single-modal clustering (0.303), it lost valuable information for multimodal fusion.#

Solution: Continue testing the remaining 6 fusion methods (attention mechanism, adln_attention, llava_mlp, qformer, etc.) to determine if certain strategies can better utilize the improved 256-dimensional embeddings; consider adjusting the hidden dimension or trying other optimization methods (such as stego_refine, byol_spatial).

Key insight: Self-supervised clustering methods optimize the independence of clustering, but this may not align with the goal of multimodal fusion. Reducing the dimensionality can filter out features that are valuable for gene-visual complementation, even if these features are not important for image-based clustering.

4. The GPU allocation strategy was unclear: initially assumed that only GPU 0 was available, but in reality, there were 8 A800 GPUs on the Slurm nodes.#

Solution: Manually verified the status of all GPUs, used the nvidia-smi tool to check memory usage, and selected GPUs with memory usage exceeding 50%. The updated plan involves dynamically checking memory usage and accordingly allocating CUDA_VISIBLE_DEVICES to ensure parallel task execution.

Key insight: In a shared HPC environment, the availability of resources does not necessarily mean they are idle. Memory-based criteria (such as a 50% threshold) should be used instead of simple idle detection. If memory allows, multiple tasks can share a node, maximizing cluster utilization.

5. The SSH connection to the GPU node was unstable and often disconnected, causing long-running experiments to be interrupted.#

Solution: Adopted the workflow recorded in CLAUDE.md, using squeue to find running tasks, and then using srun –jobid= to connect to the allocated node. This avoids direct SSH use and leverages Slurm’s task management capabilities.

Key insight: Using srun –jobid for task connections is more stable than using SSH directly, as Slurm is designed for HPC workloads and can manage cluster resources effectively. Task IDs can be reused across different sessions, allowing for reconnections.

  • Despite having 8 A800 GPUs available, artificial intelligence still assumed the use of a single GPU for allocation, requiring humans to correct it to effectively utilize cluster resources.   • When organizing MIHD files, the location of ENHANCEMENT_PLAN.md was misjudged initially (thought it was in the doc/ directory), and it was necessary to verify with the ls command. In large projects, file paths should be verified more carefully.

It should have been identified earlier that output redirection failed, and instead, checking the existence of .npz files should have been done instead of relying on logs.

Lessons Learned#

Key Lessons#

  • The two-stage fusion (self-supervised refinement → multi-modal integration) created a new experimental dimension, connecting methods based solely on images and multi-modal methods. However, preliminary tests showed that among all fusion strategies, finely processed embeddings were not always better than the original embeddings. It is crucial to maintain consistency in the optimization goals.
  • The choice of baseline model has a significant impact on the effectiveness of the experiment. Using checkpoints from different domains (such as LIBERO adjusted for the PickPlace task) will weaken its effectiveness. The baseline model must be clearly validated or fine-tuned to ensure fair evaluation.
  • The implementation of the visual refinement process demonstrated the feasibility of adding intermediate processing steps in run_benchmark.py: adding approximately 50 lines of code at the appropriate position (after visual encoding and before fusion) could create a completely new experimental paradigm, while being fully compatible with existing cache/naming/logging systems.
  • Accessing GPUs in the HPC cluster through Slurm job attachments (srun –jobid) is more stable than direct SSH, as it allows utilization of the cluster’s resource management system and is suitable for long-running scientific workloads. The job ID remains unchanged between sessions, allowing reattachment even after disconnection.
  • The research project structure for the first or second phase (infrastructure building → verification experiments) benefits from clear milestone tracking. The completion of error recovery benchmark tests M1-M11 (649 scenarios) provided a solid foundation for the M14 baseline evaluation, although currently it is hindered by insufficient memory in Pi0.5.
  • In a HPC cluster, the availability of GPUs should be determined based on memory space (at least 50% or more idle), not just based on the idle state. If memory allows, multiple jobs can coexist on the same node, enabling parallel task execution and maximizing cluster utilization.
  • The dependency topology should be presented before the execution plan. Clearly visualizing M14 as a key path blocker helps in resource allocation and timeline estimation, leading to more strategic scheduling.
  • The collector resume function makes fault-tolerant long-term evaluation possible. The rationality check of M14 verifies that interrupted evaluations can restart from the last checkpoint, which is important for full runs lasting up to 6 hours.
  • The “cleanup day” mode for project maintenance is very effective: using 3 parallel exploration agents for a comprehensive project survey, followed by systematic cleanup (deleting redundant files, moving logs, reorganizing documents, updating references/memory), thus simplifying the root directory and making the document hierarchy clear and traceable.
  • The systematic documentation audit carried out through the /init skill is very effective in identifying long-term gaps accumulated during iterative development, especially in multiple tool projects where each tool is recorded independently.
  • The error detection process (20 classifiers, PolicyErrorMonitor) runs reliably in large-scale situations. During the rationality check, 37-74 errors were detected per set, proving that the classification method is ready for production use.
  • The MimicGen-to-LeRobot conversion process requires careful handling of flat and nested HDF5 layouts. The 84×84 images used in the Phoenix paper were resized to 224×224 for training, and this preprocessing must be retained during the conversion process.
  • JAX can automatically detect all visible GPUs for multi-GPU training, when batch_size can be divided evenly by the number of GPUs. Setting CUDA_VISIBLE_DEVICES=0,1,3,4 and batch_size=64 allows for efficient 4-GPU parallelism without explicit configuration.
  • Exploration agents perform well in large project organization tasks: they can recursively traverse directories, read file headers to quickly identify content, calculate line counts, and compare redundant parts in documents—their efficiency is several orders of magnitude higher than manual exploration.
  • In the cleanup work of large projects, if the user has clear goals and is willing to take responsibility for the quality of the plan, then user-controlled planning (avoiding the approval of ExitPlanMode) may be more efficient. The role of AI becomes one of execution, rather than design.

Practical Experience#

  • When a project accumulates a large amount of work history, generating historical reports in batches is valuable for creating searchable archives. Although these reports are mainly meta-work rather than main development, the gadget project now has a comprehensive report covering more than 3 weeks of project activities.
  • QFormer and other fusion methods based on learnable queries exhibit the characteristics of “high VRAM + low GPU utilization” during training: parameters and intermediate results consume memory, but the actual computation (cross-attention mechanism) is relatively simple, with most of the time spent on CPU-intensive k-NN graph construction and loss calculation. This is an architectural feature, not a performance issue.

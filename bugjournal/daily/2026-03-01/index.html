Problems & Solutions#

Critical Issues#

1. VisiumHD dataset paths incompatible with DLPFC-oriented pipeline: image file mismatches, spatial coordinate loading failures, mclust not scaling to HD bin counts (17,502 bins vs DLPFC 4,992 spots)#

Solution: 5-round debugging: fixed path resolution logic, refactored coordinate loading, implemented mclust→KMeans fallback for large datasets, added tqdm + PYTHONUNBUFFERED for visibility. HD crop10large now runs through complete STAIG pipeline with 360x speedup (2 min vs 40+ min per section)

Key Insight: DLPFC and VisiumHD differ by >3x scale; mclust’s EM algorithm doesn’t scale (40 min for 17,502 bins), KMeans provides acceptable quality tradeoff (ARI ~0.17 vs 0.20-0.25) for exploratory spatial transcriptomics

2. Coffee BC-RNN achieved 0% success rate across all 600 epochs while Stack reached 98%#

Solution: User diagnosed root cause: all 9 task configs from same template only included robot-centric observations (eef_pos, eef_quat, gripper_qpos), missing 57-dim object state (pod/machine/holder positions). Coffee multi-stage precision operations require object info; Stack simple task can rely on vision. Fixed by adding extra_low_dim=[‘object’] to all tasks, regenerating configs

Key Insight: Observation space deficiencies have vastly different impacts across task complexities. Simple tasks can compensate via vision; complex tasks completely fail. Template-based config generation needs per-task observation customization

3. MIHD refactoring Phase 2 dependency order: pipeline/ modules massively import from run_benchmark.py, preventing direct deletion#

Solution: Executed Phase 2 first (extract shared functions to models/staig_utils.py, models/vision_extractors.py, models/Fusion.py, utils/clustering.py and update all pipeline/ imports), then Phase 3 (delete run_benchmark.py). Verified via grep for residual references and pipeline dry run

Key Insight: Large refactoring must follow gradual dependency decoupling: migrate depended functions first, then delete source files, avoiding intermediate import failures. Topological ordering is critical

4. Desktop Video PowerSave/PowerSave+ modes used per-screen updatePlayState(for:) logic, causing new window creation to only affect current screen while others continued playing#

Solution: Modified SharedWallpaperWindowManager.updatePlayState(for:) to delegate .powerSave and .powerSavePlus cases to AppDelegate.shared?.updatePlaybackStateForAllScreens(), ensuring global re-evaluation on window events

Key Insight: Per-screen state methods must explicitly delegate to global evaluators when mode semantics require system-wide decisions; local+global duality requires careful routing

5. JAX crashes with ‘No GPU/TPU found’ when CUDA_VISIBLE_DEVICES="" set for CPU-only norm_stats#

Solution: Added JAX_PLATFORMS=cpu environment variable alongside CUDA_VISIBLE_DEVICES="" to force JAX CPU backend initialization

Key Insight: JAX requires explicit backend selection via JAX_PLATFORMS, not just CUDA device masking. CUDA_VISIBLE_DEVICES alone causes backend initialization errors

6. Pathologist annotation coordinates (μm scale on full slide) needed mapping to 8um bin grid for VisiumHD crop10large comparison#

Solution: Discovered r_big//4 mapping rule through empirical testing, achieved 100% barcode match (17,502/17,502 bins), validated via spatial overlay inspection

Key Insight: VisiumHD’s multi-resolution grid (2um bins with 4×4 aggregation to 8um) requires integer division coordinate transform; Zenodo metadata lacked explicit mapping documentation

7. Desktop Video Automatic mode lacked battery state awareness, always using PowerSave logic even when macOS was in Low Power Mode#

Solution: Added NSProcessInfoPowerStateDidChange observer in applicationDidFinishLaunching; implemented powerStateDidChange handler; updated shouldPauseAllVideos() to check ProcessInfo.processInfo.isLowPowerModeEnabled: normal power → allOverlaysCompletelyCovered(), low power → anyOverlayCompletelyCovered()

Key Insight: macOS Low Power Mode queryable via ProcessInfo.isLowPowerModeEnabled and observable via NSProcessInfoPowerStateDidChange; adaptive behaviors benefit from both query + observation

8. GPU VRAM exhausted - all 8 GPUs near full, blocking new training jobs#

Solution: Identified Phoenix serve_policy.py (PID 2413737) consuming ~407GB across 7 GPUs with 0% utilization (idle server). Killed it plus two other idle VLA servers (PIDs 2046779, 2138978, ~61GB each). Freed GPUs 0-6 completely

Key Insight: Idle model serving processes can monopolize GPU memory for days if not monitored. Regular nvidia-smi checks and process audits prevent resource starvation. Multi-GPU models especially wasteful when idle

9. Training hangs indefinitely on wandb.init() ProxyError (no proxy configured on compute node)#

Solution: Added WANDB_MODE=disabled to training command environment

Key Insight: HPC compute nodes often lack internet access; disable cloud logging services for batch jobs (W&B, MLflow, Comet) or configure offline mode

10. Gadget merge –sync downloaded all logs and processed as single batch with just a warning about multiple dates, requiring manual iteration for backfilling#

Solution: Modified cmd_merge() to check if args.date is None when args.sync is True, then delegate to _cmd_merge_sync_all() which groups files by date (YYYY-MM-DD_device.json pattern), processes each date in separate subprocess, skips dates with existing reports, batch deploys at end

Key Insight: Unifying –sync behavior to match –sync-all eliminated confusion and made batch backfilling efficient (eliminated 50+ manual commands for February history)

11. Pi0.5 training jobs 46551 and 46553 both failed/cancelled#

Solution: Analyzed training logs: both jobs manually cancelled via scancel command. Job 46551 only reached ~579 steps of 100k target. Root cause: user intervention, not system/code error

Key Insight: Manual job cancellation is different from training failure - distinguish between system errors and user-initiated stops when diagnosing ‘failed’ runs

12. Norm_stats path mismatch: script checks checkpoints//norm_stats/ but compute_norm_stats.py writes to assets//benchmark/mimicgen_/norm_stats.json#

Solution: Updated both skip-check (Step 1) and prerequisite-check (Step 2) to use correct assets/ path structure

Key Insight: Tool output paths don’t always match intuitive checkpoint directory structure; verify actual output locations via dry runs

13. Bash tool completely failed: all commands (echo hello, true) returned exit code 1 with no output#

Solution: User insisted on retries (’try again’); tool eventually self-recovered. Root cause unknown, possibly sandbox binary or NFS filesystem temporary issue

Learnings#

Key Learnings#

  • mclust probabilistic clustering scales poorly beyond ~5000 observations (40+ min for 17,502 bins); KMeans provides 360x speedup (2 min vs 6+ hours for 11 sections) with acceptable quality tradeoff (ARI ~0.17 vs 0.20-0.25) for exploratory spatial transcriptomics analysis
  • Cannot safely delete ’legacy’ code without comprehensive dependency analysis via grep - new modular pipelines often still import utilities from old monolithic files during transition periods. ‘Legacy’ files may still be critical infrastructure if new code imports from them
  • Large refactoring best practices: analyze dependencies (grep all imports) → migrate in topological order (dependees first, dependents later) → delete source files → global verification (grep residual references + dry run tests), ensuring every intermediate state is runnable
  • Observation space design has nonlinear impact on task performance: missing object key caused Coffee 0% SR, but Stack simple task still achieved 98%. Different tasks have vastly different dependencies on observation information; config templates cannot be one-size-fits-all
  • BC-RNN significantly outperforms Pi0.5 on normal MimicGen tasks (Stack D0: 98% vs 24%) but both fail at error recovery (~0% success rate). Suggests behavior cloning from demonstrations works well for in-distribution tasks but doesn’t generalize to error scenarios without explicit training on recovery trajectories
  • Configuration bugs can be silent killers - Coffee BC-RNN trained for 600 epochs with 0% success because config omitted ‘object’ observation key (57 dims of task-critical state). Loss decreased normally but policy learned nothing useful. Always validate observation keys match environment capabilities
  • VisiumHD’s multi-resolution bin system (2um native with 4×4 aggregation to 8um) requires integer division coordinate transforms (r_big//4) for pathologist annotation alignment; coordinate mapping empirically derived, not documented in Zenodo metadata
  • HD/VisiumHD datasets require dataset-aware path handling throughout entire pipeline (vision extraction, gene extraction, fusion, clustering) - cannot assume DLPFC-style directory structure. Different datasets need different path structures; HD uses cropped regions with non-standard naming requiring explicit handling
  • Per-screen vs. global state duality requires explicit routing: local methods must delegate to global evaluators when semantics are system-wide. Multi-file coordinated edits benefit from constant extraction, centralized orchestration (AppState.didSet → global re-eval), comprehensive localization
  • JAX CPU-only execution requires both CUDA_VISIBLE_DEVICES="" (device masking) and JAX_PLATFORMS=cpu (explicit backend selection); former alone causes ‘No GPU/TPU found’ errors
  • Long-running operations on HPC must have visibility (tqdm + PYTHONUNBUFFERED=1) - buffered output creates confusion and wastes time checking process status
  • Finalize marking two-phase propagation semantics: export phase sets _finalized (when date<today and no new conversations) → merge phase reads all source log _finalized → only when all sources finalized AND date<today, merge report sets _finalized=True, ensuring completeness check
  • Multi-device development environments require systematic documentation workflows; two-phase export-then-merge architecture enables asynchronous log collection from DCC/MacBook/tianhe with centralized consolidation, critical for HPC cluster setups where devices have limited network access
  • macOS adaptive behaviors (Low Power Mode response) implementable via ProcessInfo query + NotificationCenter observation for responsive UX. ProcessInfo.processInfo.isLowPowerModeEnabled queryable and observable via NSProcessInfoPowerStateDidChange
  • BC-RNN training on MimicGen data shows task-dependent convergence rates: stack tasks achieve 44-64% success at epoch 20, coffee tasks remain 0% (higher difficulty confirmed); validates training pipeline integrity before VLA comparison experiments
  • GPU memory leaks from idle model servers are major operational hazard in shared clusters. Phoenix server consumed ~407GB across 7 GPUs at 0% utilization for hours. Need automated monitoring/cleanup or strict resource quotas
  • GPU resource monitoring needs multi-dimensional metric cross-validation: high memory occupation ≠ high GPU utilization. Norm_stats data preprocessing steps may occupy GPU memory due to model eager loading but actual computation on CPU
  • XLA_PYTHON_CLIENT_MEM_FRACTION controls JAX GPU memory pre-allocation: defaults to 0.75, can safely increase to 0.95 for dedicated training workloads to maximize available memory (from 61GB to 77.7GB per GPU)
  • HPC compute nodes typically lack direct internet access; always disable cloud logging services (W&B, MLflow, Comet) via WANDB_MODE=disabled or configure offline mode for batch training jobs
  • Codebase cleanup systematic approach: v4.17 successfully cleaned error_recovery_benchmark by (1) archiving legacy code (v1-v3 mimicgen_workspace), (2) creating external dependency index (checkpoints/datasets/repos), (3) merging duplicate docs (VLM tutorials), (4) establishing .gitignore rules. Cleaned structure more maintainable
  • Experiment results documentation completeness requirements: not just code modifications (what/how), but also experiment metrics tables (ARI/NMI/Silhouette) and visualization image links (60+ PNGs), providing complete reference for future research
  • Batch historical report generation benefits from day-by-day subprocess isolation rather than single-batch processing: independent subprocesses prevent single date’s API errors from blocking entire backlog, enable parallel execution on future multi-core systems, allow incremental progress tracking
  • SwiftUI Liquid Glass on macOS Tahoe follows conventions: .regular glass for readability, geometric containers (Capsule/RoundedRectangle), reduced shadows, locale-aware formatting via DateFormatter templates (EEEEMMMMd), consistent spacing
  • Shared HPC conda environments may point to arbitrary users’ package installations (openpi05→lipeida/openpi) while training configs reside elsewhere (zhaoganlong repo); explicit verification via python -c 'import pkg; print(pkg.__file__)' mandatory before assuming code-config alignment
  • Slurm srun –overlap flag allows multiple commands to execute on same job allocation without launching new jobs, enabling incremental debugging on reserved nodes
  • Black-box tool failure pragmatic response: when Bash tool completely failed and diagnosis yielded nothing, simple retry strategy (user’s ’try again’) ultimately solved problem. In production environments, fast recovery may be higher priority than root cause analysis
  • Project documentation maintenance requires disciplined version tracking: when adding features, must synchronously update version number, statistics (LOC counts, file counts), file listings, and version history. Documentation completeness prevents knowledge decay

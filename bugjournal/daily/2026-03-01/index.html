Issues and Solutions#

Key Issues#

1. VisiumHD Dataset Path and DLPFC-Directed Process Compatibility Issues: Image files do not match, spatial coordinate loading fails, mclust cannot adapt to HD interval counting (17,502 intervals, while DLPFC has only 4,992 positions) Solution: 5 rounds of debugging: fixed path parsing logic, reconstructed coordinate loading method, implemented a backtrack mechanism from mclust to KMeans for large datasets, added tqdm and PYTHONUNBUFFERED for better visibility. HD crop10large can now quickly complete the entire STAIG process, with speed increased by 360 times (2 minutes vs. over 40 minutes)#

Key Discovery: DLPFC and VisiumHD differ by more than 3 times in scale; mclust’s EM algorithm cannot handle large-scale data processing (17,502 intervals took 40 minutes), while KMeans provides acceptable quality trade-offs in exploratory spatial transcriptomics (ARI was about 0.17 vs. 0.20-0.25)

2. Coffee BC-RNN had a success rate of 0% across all 600 cycles, while Stack achieved 98%#

Solution: Users identified the root cause: all 9 task configurations came from the same template, containing only robot-centered observation data (eef_pos, eef_quat, gripper_qpos), while missing 57-dimensional object state information (pod/machine/holder positions). Coffee’s multi-stage precise operations required object information; while Stack’s simple tasks could rely on visual recognition. The solution was to add an additional low_dim=[‘object’] parameter to all tasks, re-generating the configurations

Key Discovery: The lack of observation space had a different impact on task complexity. Simple tasks could be compensated by visual recognition; while complex tasks could not complete the tasks at all. Template-based configuration generation required customization of observation data for each task

3. MIHD Refactoring Second Phase Dependency Order: Pipeline/Modules were imported from run_benchmark.py in large quantities, making it impossible to directly remove themSolution: First, perform the second stage of work (extracting shared functions into models/staig_utils.py, models/vision_extractors.py, models/Fusion.py, utils/clustering.py, and updating all pipeline imports). Then, carry out the third stage of work (deleting run_benchmark.py). Verify the results by using grep to check for remaining references and running the pipeline in a dry run.#

Key Findings: Large-scale refactoring must involve gradually separating dependencies: first, migrate dependent functions, then delete the source files to avoid failures in intermediate imports. Topological sorting is crucial.

4. The Desktop Video PowerSave/PowerSave+ modes use the logic of updating PlayState(for:) for each screen, resulting in new windows only affecting the current screen, while other screens continue to play.#

Solution: Modify SharedWallpaperWindowManager.updatePlayState(for:) to delegate the .powerSave and .powerSavePlus cases to AppDelegate.shared?.updatePlaybackStateForAllScreens(), ensuring a global re-evaluation during window events.

Key Findings: When the mode requires system-level decisions, the state method for each screen must be explicitly delegated to a global evaluator; the combination of local and global approaches requires careful routing.

5. When CUDA_VISIBLE_DEVICES="" is set to CPU-only for norm_stats, JAX crashes, indicating “No GPU/TPU found.”#

Solution: Added the JAX_PLATFORMS=cpu environment variable, as well as CUDA_VISIBLE_DEVICES="", to force JAX to use the CPU backend for initialization.

Key Findings: JAX needs to explicitly choose the backend through JAX_PLATFORMS, not just the CUDA device mask. Using only CUDA_VISIBLE_DEVICES would lead to backend initialization errors.

6. Pathologist annotation coordinates (micrometers across the entire slice) need to be mapped to the 8-micrometer interval grid when comparing with VisiumHD crop10large.#

Solution: Through experimental testing, a mapping rule of r_big//4 was found, achieving 100% barcode matching (17,502/17,502 intervals), which was verified through spatial overlay checks.

Key Findings: VisiumHD’s multi-resolution grid (2-micrometer intervals, 4×4 aggregations to 8 micrometers) requires integer division for coordinate conversion; Zenodo metadata lacks a clear mapping document.

7. The Desktop Video Auto mode does not have a battery status awareness function, always using the PowerSave logic even when macOS is in low-power mode.#

Solution: Added an observer for NSProcessInfoPowerStateDidChange in applicationDidFinishLaunching; implemented a powerStateDidChange handler; updated shouldPauseAllVideos() to check ProcessInfo.processInfo.isLowPowerModeEnabled: use allOverlaysCompletelyCovered() in normal power consumption, anyOverlayCompletelyCovered() in low-power mode.

Key Findings: macOS low-power mode can be queried through ProcessInfo.isLowPowerModeEnabled, and observed through NSProcessInfoPowerStateDidChange; adaptive behavior requires both querying and observing the two scenarios.

8. GPU memory exhaustion—all 8 GPUs are nearing full capacity, preventing new training tasks from being executed.#

Solution: It was found that Phoenix serve_policy.py (PID 2413737) consumed approximately 407GB of memory on 7 GPUs, with a utilization rate of zero (idle server). This process and two other idle VLA servers (PID 2046779, 2138978, each about 61GB) were killed. GPUs 0-6 were released.

Key Findings: If not monitored, idle model service processes may occupy GPU memory for days. Regular use of nvidia-smi for checking and process auditing can prevent resource shortages. Multi-GPU models waste resources especially when idle.

9. During training, wandb.init() encountered ProxyError, causing the training task to hang indefinitely (no proxy configured on the computing node).#

Solution: Added WANDB_MODE=disabled in the training command environment.

Key Findings: HPC computing nodes often lack internet connectivity; for batch tasks, disable cloud logging services like W&B, MLflow, Comet, or configure an offline mode.

10. Gadget merging—sync downloaded all logs and processed them in one batch, but it only indicated multiple dates, requiring manual iteration to complete the filling process.#

Solution: Modified the cmd_merge() function to check if args.sync was True, and then delegate the task to the dirgent_merge_sync_all() function. This function groups files by date (in the format YYYY-MM-DD_device.json), processes each date in a separate sub-process, skips already reported dates, and finally performs batch deployment.

Key Findings: Unified the –sync behavior to match –sync-all, eliminating confusion and making the batch filling process more efficient (eliminating over 50 manual commands to handle February records).

11. Training tasks 46551 and 46553 for Pi0.5 both failed/cancelled.#

Solution: Analyzed the training logs: both tasks were manually canceled using the scancel command. Task 46551 only completed approximately 579 steps, while the target was 100k steps. The root cause was user intervention, not a system or code error.

Key Findings: Manually canceling a task is different from training failure—when diagnosing a “failed” runtime, it’s necessary to distinguish between system errors and user-initiated stops.

12. The path of norm_stats does not match: the script checks checkpoints//norm_stats/, but compute_norm_stats.py writes the output to assets//benchmark/mimicgen_/norm_stats.json.#

Solution: Updated the skip checks (step 1) and prerequisite checks (step 2), using the correct asset/path structure.

13. The Bash tool completely failed: all commands (echo hello, true) returned exit code 1, with no output.#

Solution: The user insisted on a retry (“try again”); the tool eventually recovered on its own. The root cause was unclear, possibly due to a sandbox binary or NFS file system issue.

Key Findings: Tool-level failures sometimes only require simple retries, rather than deep diagnosis. During Bash failures, read/write/wildcard tools can still work properly, serving as a backup for partial tasks.

14. The complex multi-project dates (February 15th, 16th, 17th, 28th) exceeded the Claude API’s 32K output token limit during the merge operation. Solution: No immediate fix needed; this limitation was accepted. Due to the daily workload distribution among devices, most reports did not exceed the limit.#

Key Insights: High cross-project activities on multiple devices may result in daily reports exceeding the API output limit. Future mitigation measures include block-based merging, processing by device, or adopting hierarchical summarization methods (day → week → month).

Common Issues#

15. The training script encountered parsing errors when running “–overwrite/–no-overwrite: ignoring explicit parameter False”.#

Solution: Change –overwrite=False to –no-overwrite (boolean flag syntax, not key=value).

Key Insights: Python CLI boolean flags typically use –flag/–no-flag mode, not the –flag=value syntax.

Limitations of Artificial Intelligence#

Key Limitations - Before explicitly using grep for verification, it was not possible to detect the dependency of the pipeline on run_benchmark.py - initially thought this was old code that could be removed after functional comparisons.#

  • It was necessary to perform 5 separate debugging sessions to fix the issues with the Pi0.5 training launcher, rather than comprehensively analyzing potential failure modes (JAX backend, paths, CLI syntax, network dependencies).
  • When attempting to generate comprehensive multi-day reports, due to the token output limit reaching 32,000, an API error occurred, and no graceful degradation or automatic chunking was implemented.
  • Initially, it was not known that JAX required setting JAX_PLATFORMS=cpu in CPU-only mode, rather than simply disabling CUDA_VISIBLE_DEVICES; this was learned through repeated experimentation.
  • Before the pre-training check failed, it was not noticed that the expected norm_stats path in the script did not match the actual output location of the tool.
  • No proactive identification of GPU resource usage anomalies was made. The user needed to explicitly prompt “8 GPUs are occupied but the utilization is only 0%,” before further analysis of norm_stats that did not require GPU inference could be conducted. There was a lack of proactive multi-indicator cross-validation methods.
  • During the implementation plan formulation, AI called ExitPlanMode multiple times without user approval, resulting in multiple rejections; lacked patience to wait for the “execute” signal.
  • When users started generating multiple consecutive historical reports, no proactive suggestion was made for daily batch processing, and the relevant functionality was implemented only after explicit requests from users. If a more proactive approach had been taken, the filling pattern could have been identified after 2-3 manual merge commands.
  • Although this is a common cluster configuration pattern, the dependency issue of W&B networks on HPC computing nodes was not anticipated.
  • Over-reliance on Agent exploration. In the project cleanup task, 3 parallel Agents were initiated for deep exploration (current files, code merging opportunities, document status), but this was quickly interrupted by the user. The user might have expected direct execution rather than re-planning.
  • Despite sharing the infrastructure, separate repair plans were proposed for 3 desktop video playback vulnerabilities, indicating that system-level architectural awareness was still limited before explicit exploration.

Overall Limitations#

  • When extracting high-definition patches, it was initially impossible to see the buffered Python output. The user needed to ask “What’s wrong here?” before adding tqdm and PYTHONUNBUFFERED to view the output.
  • When reaching the 32K output token limit during report generation, no other strategies were provided (dividing by device, reducing details, segmenting output), and the failure outcome was accepted instead.
  • When monitoring and evaluating progress, task output polling with a long delay of 600 seconds was used, although this process clearly took a long time (50 episodes × 9 tasks). A more efficient approach would be to perform non-blocking checks regularly and use exponential backoff strategies.

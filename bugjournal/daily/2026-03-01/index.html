Issues and Solutions#

Key Issues#

1. VisiumHD HD dataset path adaptation: vision encoder image paths, spatial coords loading, and multiple failures during fusion due to assuming a DLPFC directory structure.#

Solution: Three fixes were made: add cropped_fullres.tif mode recognition in find_spatial_image(); pass crop_dir.parent as data_root when calling vision encoder; load coordinates directly from adata.obsm[‘spatial’] during fusion phase, bypassing path lookup.

Key insight: The pipeline design assumes a flat DLPFC directory structure (data_root/section_id/), but HD data needs to be mapped at the call level rather than modifying the encoder’s internal logic. The path abstraction layer should provide interfaces for adapting to different dataset structures.

2. Invisibility of long-term task progress: GPU utilization was 0%, but VRAM was fully loaded, making it impossible to determine if it was a CPU bottleneck or model eager loading causing wasted time.#

Solution: Set PYTHONUNBUFFERED=1 and add tqdm for critical loops; set compute_norm_stats.py to require CUDA_VISIBLE_DEVICES="" + JAX_PLATFORMS=cpu (root cause: PaligemmaTokenizer is eagerly loaded into all visible GPUs during get_config().model call, but norm_stats only requires dataset transforms).

Key insight: GPU utilization of 0% could indicate a CPU preprocessing bottleneck or eager loading without inference calculations. The difference between the two can be distinguished by monitoring VRAM usage with nvidia-smi. Norm stats doesn’t require model inference and should run in CPU mode by default.

3. Coffee BC-RNN loss converged normally (-7.66→-15.4), but SR was always 0%, while Stack achieved 64% at epoch 20 on equivalent configurations.#

Solution: By comparing actual HDF5 dataset observation key structures, we found that Coffee requires 57-dimensional object states (multi-step precise operations: Pod/Machine/Holder position + relative orientation + hinge angle). The configuration template was missing this information. We added extra_low_dim=[‘object’] to all nine tasks and re-generated the configuration.

Key insight: BC-RNN’s observation keys must match task complexity. Coffee requires multi-step precision operations, and 84×84 images don’t provide enough spatial accuracy information. The performance of configuration bugs (normal loss but SR=0) is highly similar to insufficient model capabilities. This can only be distinguished by comparing dataset structures.

4. Robomimic BC-RNN image observation integration failure: four consecutive bugs (enable camera, checkpoint config format, resolution, image format).#

Solution: ① Detect rgb modality automatic activation from checkpoint config fields (JSON strings, not dict access); ② Read actual image size (84×84) from shape_metadata and pass it to create_env; ③ Manually perform HWC→CHW transformation and uint8→float32/255 normalization in _prepare_image_obs().

Key insight: Robomimic checkpoint config is stored as a JSON string, which doesn’t follow intuition. When bypassing the standard rollout process, the format conversion automatically completed by ObsUtils.process_obs needs manual supplementation.

5. Pi0.5 LoRA training had four consecutive obstruction bugs: JAX CPU forced failure, norm_stats path error, boolean parameter format issues, and W&B proxy blocking.#

Solution: ① Set CUDA_VISIBLE_DEVICES="" + JAX_PLATFORMS=cpu for JAX CPU mode (the latter cannot be omitted); ② Output norm stats to assets//benchmark//norm_stats.json, not checkpoints/; ③ Use –no-overwrite for argparse boolean flags, not –overwrite=False; ④ HPC nodes have no outbound network, so set WANDB_MODE=disabled.

Key insight: JAX XLA backend detects independently of CUDA environment variables. Openpi norm stats path is strongly bound to dataset_name and written to assets/. HPC nodes should disable all log systems that rely on external networks by default.

6. MIHD run_benchmark.py was heavily imported by pipeline, making it impossible to delete directly; BasicContrastive.py had reversed drop_feature logic.Solution: Conduct a systematic grep to identify all reference points and migrate in the order of “create new modules → update references → delete old files.” Fix the issue with drop_feature: change >= drop_prob to < drop_prob, and import the corrected version from staig_utils.#

Key insights: The order of large-scale refactoring migrations is crucial. Functions with the same name may have three variants in three different files. Careful comparison of semantics before merging is necessary (differences in > / >= / < boundary conditions can lead to completely opposite functionality).

General Issues#

7. SLURM srun commands hang on nodes with interactive sessions; orbax checkpoint resume conflicts (Destination …/5000 already exists)#

Solution: Add –overlap to srun to allow shared execution. For orbax resume, set overwrite=True explicitly or clear the existing step directory in advance.

Key insights: SLURM interactive jobs are assigned exclusively by default. Orbax resume=True does not automatically overwrite existing step directories, which is a common pitfall in JAX training frameworks.

8. Bash tools fail multiple times on tianhe nodes (all echo/true/pwd have exit code 1), making it impossible to execute any shell commands.#

Solution: Use Read/Write/Edit/Glob tools to perform all file operations. For tasks that require a shell, clearly mark them as tasks for user execution.

Key insights: Claude Code toolset can replace most Bash file operations. When Bash fails, switch strategies proactively and inform users instead of waiting in vain.

Human Thinking vs AI Thinking#

Strategic Level#

Active discovery and optimization decisions regarding GPU resources#

Role Thoughts
Humans Actively discover anomalies where GPU util=0 but VRAM is fully utilized (VisiumHD patch causes issues with CPU, norm_stats loads the full model, an49 leaves 530GB VRAM idle). Proactively stop and investigate root causes. Based on knowledge of JAX memory mechanisms, request XLA_PYTHON_CLIENT_MEM_FRACTION=0.95, and expand training GPUs from 1-4 to 1-6 (considering global cluster resources).
AI Tend to wait for tasks to complete without actively auditing resources. Passively respond to GPU anomalies. In the norm_stats scenario, provide precise root cause analysis through agent-tracking of source code (PaligemmaTokenizer eager loading). Analyze the advantages and disadvantages of memory optimization passively, but do not proactively suggest solutions.**

Difference analysis: Humans actively discover system-level anomalies using nvidia-smi and prior knowledge to provide optimization directions. AI has an advantage in root cause analysis depth but lacks proactive monitoring awareness. It fails to detect GPU resources available from 5-6.

Execution permissions and rhythm control (multiple times rejected ExitPlanMode)#

Role Thoughts
Humans Reject AI’s automatic execution requests for irreversible operations such as large-scale file operations, batch submissions of GPU cluster tasks, and training submissions. Require reviewing the plan or starting with a single task for gradual verification.**
AI Tend to request execution authorization immediately after planning. Expect to complete all work at once, maximizing parallelism and automation.**

Difference analysis: Humans maintain strict review rhythm for irreversible operations. Gradual verification (running a single task first and then batching) is more reasonable in HPC resource-constrained environments. AI systematically underestimates the need for review and gradual verification.

Independent discovery of BC-RNN defects#

Role Thoughts
Humans Discover anomalies where Coffee loss is normal but SR=0%. Submit a diagnostic request without providing specific hypotheses.**
AI Systematically compare Coffee/Stack configuration files. Check the actual obs key structure in HDF5 datasets (find object: 57-dimensional). Analyze differences in task complexity and independently conclude that “object keys are missing” (without any human guidance).**

Difference analysis: AI has a stronger ability to identify root causes through systematic data/code comparisons compared to human intuition. Humans discover problems based on anomalies, while AI discovers root causes through structured exploration.

Coffee simulator bug vs identification of model learning failures#

Role Thoughts
Humans Immediately recognize that the problem was an object misalignment or freezing issue during rollout video playback, rather than a model problem.**
AI Initially attribute the 0% SR to model failure. Tend to attribute problems from data/parameters/task complexity perspectives.**

Difference analysis: Human intuition and simulator experience allow humans to quickly distinguish between physical errors and learning failures. AI lacks direct perception of simulator visual anomalies and tends to attribute simulator bugs to model capabilities.

Pragmatic engineering decisions (tool replacement and diagnostic scope)#

Role Thoughts
Humans Directly decide to replace mclust with KMeans under environmental constraints, as it is more efficient than fixing dependencies. For Pi0.5 training failures, choose “only diagnosing without repairing” (since the third attempt has reached 99999 steps). First, evaluate the results before deciding whether to retrain.**
AI Tend to repair existing tools (install rpy2) or provide complete diagnostic and repair solutions. Do not prioritize the key constraint of “existing checkpoints are evaluable”.**

Difference analysis: Humans make more pragmatic resource prioritization decisions. AI tends to focus on completeness and repair, sometimes ignoring key constraints like “third attempt has been successful.”

AI Limitations#

Major limitations#

  • Lack of active long-term task health checks: Relying on users to discover anomalies such as GPU idling or process freezing instead of regular checks. Set up active monitoring mechanisms for tasks that take more than 5 minutes, rather than waiting for output or responding passively. -Lack of preventive verification before integrating external tools/libraries: No pre-checking the availability of mclust/rpy2, external script CLI interfaces (compute_norm_stats.py parameter format), memory consumption during large-scale operations (LeRobot handling segfaults), or image format conventions (robosuite HWC vs robomimic CHW). This led to multiple failures before fixing problems. The bug in the 4th layer of image observation is a concentrated example of systematic lack of foresight. -Lack of specificity in path and environment assumptions: MIHD pipeline hard-codes the DLPFC directory structure. Script errors assume norm_stats output goes to checkpoints/. JAX CPU forcibly omits JAX_PLATFORMS=cpu. H5 does not actively anticipate that HPC nodes have no outbound network (W&B), both of which led to failures during initial execution before fixing problems. -Assumption bias: Misjudge the Coffee simulator environment initialization bug (object misalignment/freezing) as a lack of model learning ability until users watch videos. Lack of means to perceive simulator visual anomalies. -Tendency to automate batch executions: In large pipelines, do not actively ask if users need gradual verification. Users need multiple interventions to control execution粒度。Killing old processes without realizing that the launcher has restarted child processes with a new PID requires 3 rounds of killing operations.

General limitations#

-Lack of ability to diagnose root causes when Bash tools fail. Can only bypass them passively. SubAgent exploration reports sometimes rely on document inferences rather than actual file system scans, resulting in inferred conclusions that do not match the actual directory structure.

Today’s Takeaways#

Key takeaways#

  • VisiumHD coordinate mapping: 2um bin r_big//4 yields an 8um grid row (similar for col). Annotation barcode format is s_008um_{row:05d}_{col:05d}-1. MIHD on VisiumHD has pca×uni2×staig_fusion Silhouette=0.343, which is significantly better than pure PCA’s 0.086. This verifies the effectiveness of multimodal fusion for HD data.
  • Key configuration for robomimic BC-RNN inference: Checkpoint config stored as a JSON string (needs json.loads, not direct dict access). shape_metadata records actual image sizes. When bypassing standard rollout, manual HWC→CHW transformation and uint8→float32/255 normalization are required (robomimic’s standard rollout automatically handles ObsUtils.process_obs).
  • Key configuration for JAX/Pi0.5 training: ① Force CPU to set CUDA_VISIBLE_DEVICES="" + JAX_PLATFORMS=cpu simultaneously; ② XLA_PYTHON_CLIENT_MEM_FRACTION=0.95 is effective for A800 with 80GB (from 61 to 77.7GB); ③ HPC nodes must preset WANDB_MODE=disabled; ④ argparse boolean flag should use –no-overwrite, not –flag=False; ⑤ openpi norm_stats path is assets//benchmark//norm_stats.json (not checkpoints/).
  • BC-RNN configuration matches task complexity: Coffee requires a 57-dimensional object state (multi-stage precise operations are necessary); Stack only requires simple stacking of images. Orbax resume=True does not automatically overwrite existing step directories; explicit overwrite=True is required (a common JAX ecosystem trap). -Methodology for large repository refactoring: Systematically use grep to verify all imported dependencies, then execute in the order of “create new → update references → delete old”. Carefully compare boundary conditions before merging variants with the same name (> vs >= vs < can lead to completely opposite functionality). -Comparison of BC-RNN and Pi0.5 capabilities: BC-RNN achieves 64-100% SR after 22 epochs for simple tasks (Stack D0/D1, Threading D0); complex multi-step tasks (Coffee, ThreePieceAssembly D1) have clear capacity limits (600 epochs still close to 0%); Pi0.5 base model has a SR of 4.2%, and the effectiveness of LoRA fine-tuning remains to be verified. -Accelerating norm_stats calculations: Only dataset transforms are needed; no model inference required. In CPU mode, with –max-frames set to 10000 and 9 tasks in parallel, execution time is reduced from 10+ minutes to 2.5 minutes (156 batches are sufficient for stable statistics).

Practical takeaways#

-Tips for SLURM HPC debugging: srun –overlap allows additional commands to be executed on nodes with interactive sessions (direct SSH is blocked by pam_slurm_adopt). Nohup bash script child processes do not exit with the parent process. When killing old processes, ensure that the launcher and child processes are handled separately.

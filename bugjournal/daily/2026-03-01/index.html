Issues and Solutions#

Key Issues#

1. VisiumHD Dataset Path Compatibility with DLPFC-Directed Processes: Image file mismatches, spatial coordinate loading failures, mclust unable to handle HD interval counting (17,502 intervals vs 4,992 positions on DLPFC) Solution: 5 rounds of debugging: fixed path parsing logic, refactored coordinate loading mechanism, implemented mclust-to-KMeans backtracking for large datasets, added tqdm and PYTHONUNBUFFERED for better visibility. HD crop10large can now complete the entire STAIG process quickly, with speed increased by 360 times (2 minutes vs over 40 minutes/partial processes)#

Key Insight: DLPFC and VisiumHD differ in scale by more than 3fold; mclust’s EM algorithm cannot handle large datasets (17,502 intervals requires 40 minutes), KMeans provides acceptable quality trade-offs when handling exploratory space transcriptomics (ARI is about 0.17 vs 0.20-0.25)

2. Coffee BC-RNN failed to achieve any success rate across all 600 cycles, while Stack achieved 98%#

Solution: Users identified the root cause: all 9 task configurations from the same template contained only robot-centered observation data (eef_pos, eef_quat, gripper_qpos), missing 57-dimensional object state information (container/holder positions). Coffee’s multi-stage precise operations required object information; simple tasks could rely on visual processing. This issue was fixed by adding extra_low_dim=[‘object’] to all tasks and regenerating the configuration

Key Insight: Observation space defects have a significant impact on task complexity. Simple tasks can compensate with visual processing, while complex tasks cannot complete them at all. Configurations based on templates need to be customized for each task’s observation data

3. MIHD Refactoring Phase Dependency Order: Pipeline/Modules Imported from run_benchmark.py in Large Numbers, Preventing Direct DeletionSolution: First, perform the second stage (moving shared functions to models/staig_utils.py, models/vision_extractors.py, models/Fusion.py, utils/clustering.py, and updating the import statements for all pipelines). Then, carry out the third stage (deleting run_benchmark.py). Verify the remaining references and pipeline operations using grep.#

Key Insight: Large-scale refactoring must follow the principle of gradually decoupling dependencies: first migrate dependent functions, then delete source files to avoid intermediate import failures. Topological sorting is very important.

4. The Desktop Video PowerSave/PowerSave+ mode uses a per-screen updatePlayState(for:) logic, resulting in the creation of new windows only affecting the current screen, while other screens continue playing.#

Solution: Modify SharedWallpaperWindowManager.updatePlayState(for:) to delegate handling for .powerSave and .powerSavePlus cases to AppDelegate.shared?.updatePlaybackStateForAllScreens(), ensuring a global re-evaluation during window events.

Key Insight: When mode semantics require system-level decisions, per-screen state methods must be explicitly delegated to the global evaluator. The dichotomy between local and global needs careful routing.

5. When CUDA_VISIBLE_DEVICES="" is set to only CPU, JAX crashes due to “No GPU/TPU found”.#

Solution: Added the JAX_PLATFORMS=cpu environment variable, while setting CUDA_VISIBLE_DEVICES="" to force JAX CPU backend initialization.

Key Insight: JAX needs to explicitly choose the backend through JAX_PLATFORMS, not just mask the CUDA devices. Using only CUDA_VISIBLE_DEVICES will lead to backend initialization errors.

6. In VisiumHD crop10large comparison, pathologist annotation coordinates (micrometer-level on full slices) need to be mapped to a 8um bin grid.#

Solution: Through experimentation, it was found that the mapping rule of r_big//4 achieved 100% barcode matching (17,502/17,502 intervals), and this was verified through spatial overlay checks.

Key Insight: VisiumHD’s multi-resolution grid (2um intervals, 4×4 aggregated to 8um) requires integer division coordinate conversion. Zenodo metadata lacks a clear mapping document.

7. The Desktop Video Auto mode has no battery state awareness function, and always uses the PowerSave logic even when macOS is in low-power mode.#

Solution: Added an observer for NSProcessInfoPowerStateDidChange to applicationDidFinishLaunching; implemented a powerStateDidChange handler; updated shouldPauseAllVideos() to check ProcessInfo.processInfo.isLowPowerModeEnabled: normal power → allOverlaysCompletelyCovered(), low power → anyOverlayCompletelyCovered().

Key Insight: macOS low-power mode can be queried through ProcessInfo.isLowPowerModeEnabled, and observed through NSProcessInfoPowerStateDidChange. Adaptive behavior needs to be handled by combining queries and observations.

8. GPU VRAM depletion - All 8 GPUs were nearly full, resulting in the inability to execute new training tasks.#

Solution: It was found that Phoenix serve_policy.py (PID 2413737) consumed approximately 407GB of memory on 7 GPUs, but the utilization was only 0% (idle server). The script and two other idle VLA servers (PID 2046779, 2138978, each consuming about 61GB of memory) were shut down. GPUs numbered 0-6 were released.

Key Insight: If not monitored, idle model service processes may monopolize GPU memory for days. Regular use of nvidia-smi for checks and process auditing can prevent resource exhaustion. Multi-GPU models waste resources especially when idle.

9. Training tasks are indefinitely stuck in the wandb.init() proxy error (no proxy configured on the computing node).#

Solution: Added WANDB_MODE=disabled to the training command environment.

Key Insight: HPC computing nodes usually lack internet access; for batch jobs, disable cloud logging services like W&B, MLflow, Comet, or configure an offline mode.

10. Gadget merging –sync downloaded all logs and processed them in a single batch, with only one warning regarding multiple dates that needed manual iteration to fill in missing data.#

Solution: Modified the cmd_merge() function to check if args.date was None when args.sync was True, then delegate it to _cmd_merge_sync_all(). This function groups files by date (in YYYY-MM-DD_device.json format), processes each date in a different sub-process, skips dates already reported, and finally deploys them in batches.

Key Insight: Unifying the –sync behavior with –sync-all eliminates confusion and makes batch filling more efficient (eliminating over 50 manual commands for historical data in February).

11. Both Pi0.5 training tasks 46551 and 46553 failed/canceled.#

Solution: Analyzed the training logs: both tasks were manually canceled using the scancel command. Task 46551 only reached about 579 steps, while the target was 100k steps. The root cause was user intervention, not a system or code error.

Key Insight: Manually canceling tasks and training failures are different - when diagnosing “failed” runs, it’s necessary to distinguish between system errors and manual stops.

12. norm_stats path does not match: The script checks checkpoints//norm_stats/, but compute_norm_stats.py writes data to assets//benchmark/mimicgen_/norm_stats.json.#

Solution: Updated the skipped checks (step 1) and prerequisite checks (step 2) to use the correct asset/path structure.

Key Insight: The output path of tools doesn’t always match the intuitive checkpoint directory structure; verify the actual output location through dry runs.

13. Bash tool completely failed: All commands (echo hello, true) returned exit code 1 with no output.#

Solution: The user insisted on a retry (“try again”); the tool eventually recovered on its own. The root cause was unknown, possibly a temporary issue with the sandbox binary or the NFS file system.

Key Insight: Tool-level failures sometimes only require simple retries rather than deep diagnostics. During Bash failures, read/write/Glob tools can still work properly and can serve as a backup for partial tasks.

14. Complex multi-project dates (February 15th, 16th, 17th, and 28th) exceeded the Claude API’s 32K output token limit during merging operations.#

Solution: No immediate fix needed; accept this limitation. Since the daily workload distribution among devices is natural, most reports remain within the limit.

Key Insight: Daily reports with high activity across multiple devices may exceed the API output limit. Future mitigations include chunk merging, processing by device, or using a hierarchical summary approach (day→week→month).

Common Issues#

Learning Outcomes#

Key Learnings - mclust probability clustering performance significantly decreased after over 5000 observations (at least 40 minutes for 17,502 intervals); KMeans provided a 360-fold acceleration (2 minutes compared to over 6 hours for 11 parts), while the quality loss was acceptable (ARI was about 0.17 compared to 0.20-0.25) suitable for exploratory spatial transcriptomics analysis.- It is impossible to safely delete the “old” code without a comprehensive dependency analysis using grep. During the transition period, the new modular process usually still imports tools from the old monolithic files. If the new code is imported from these “old” files, they may still be critical infrastructure.#

  • Best practices for large-scale refactoring: Analyze dependencies (use grep to check all import statements) → Migrate in topological order (migrate dependencies first, then the items that depend on them) → Delete the source file → Global verification (check remaining references and run tests), ensuring that every intermediate state is runnable.
  • The design of the observation space has a non-linear impact on task performance: Missing object keys can result in a 0% SR for Coffee, but Stack’s simple tasks can still achieve 98%. Different tasks have varying dependencies on observation information; configuration templates cannot be generalized.
  • BC-RNN performs significantly better than Pi0.5 on normal MimicGen tasks (Stack D0: 98% vs 24%), but both perform poorly in terms of error recovery (success rate is approximately 0%). This indicates that cloning behavior from demonstrations works well for tasks within the distribution, but it cannot be generalized to error scenarios without a clear training recovery trajectory.
  • Configuration errors can be a silent killer – Coffee BC-RNN still has no successful cases after 600 cycles of training because the configuration omitted the “object” observation key (57 dimensions of the task’s critical state). Although the loss decreases, the strategy does not learn any useful information. Always verify whether the observation keys match the environment capabilities.
  • VisiumHD’s multi-resolution interval system (native 2 microns, reaching 8 microns through a 4×4 aggregation) requires integer division for coordinate conversion (r_big//4) to ensure consistency with pathologist annotations; the coordinate mapping is empirically determined and not recorded in Zenodo metadata.
  • The HD/VisiumHD dataset requires consideration of dataset path handling throughout the process (visual extraction, gene extraction, fusion, clustering) – it cannot be assumed that the directory structure is similar to that of DLPFC. Different datasets require different path structures; HD uses cropped areas, and non-standard naming requires special handling.
  • The duality between each screen and the global state requires clear routing: When semantics are system-wide, local methods must be delegated to the global evaluator. Multi-file coordinated editing benefits from extraction, centralized orchestration (AppState.didSet → global re-evaluation), and comprehensive positioning.
  • JAX only CPU execution requires specifying CUDA_VISIBLE_DEVICES="" and JAX_PLATFORMS=cpu simultaneously; using the former alone will cause an error stating “GPU/TPU not found”.
  • Long-running operations running on HPC must have visibility (tqtd + PYTHONUNBUFFERED=1) – buffering output can lead to confusion and wasted time checking process status.

Created evaluation pipeline scripts for pi0.5 and mimicgen baseline models but blocked by cluster GPU access permissions

Tasks#

Implementation & Fixes#

  • ❌ Execute evaluation runs — Cannot submit SLURM jobs or SSH to GPU nodes (an49, an16, etc.) - user group ‘sysu_gbli2’ lacks permissions on all partitions, and pam_slurm_adopt blocks SSH without active jobs
  • ✅ Create evaluation pipeline scripts — Developed run_full_eval.sh (3-step SLURM pipeline), launch_eval_tmux.sh (tmux-based parallel execution), and monitor_eval.sh (status checker) for running pi0.5 and BC-RNN evaluations

Problems & Solutions#

Critical Issues#

1. SSH to GPU nodes (an49, an16, etc.) blocked by pam_slurm_adopt requiring active jobs, and an49 not present in any SLURM partition despite user’s history showing prior direct access#

Solution: Verified an49 is completely absent from sinfo output; recommended admin intervention while scripts remain ready to execute

Key Insight: Historical bash_history (repeated ‘ssh an49’ commands) confirms previous access method, but node removal from cluster management requires admin restoration

2. All SLURM job submissions fail with ‘User’s group not permitted to use this partition’ across xy-a800, ai, lava, and all other partitions#

Solution: Created alternative tmux-based launcher (launch_eval_tmux.sh) that works with direct SSH or srun once access is restored; documented the blocker for admin escalation

Key Insight: When cluster access is fundamentally broken, prepare self-contained workarounds and clear documentation for when access returns rather than trying endless permission variants

General Issues#

3. SLURM sbatch failed with ‘Invalid generic resource (gres) specification’ when using –gres=gpu:1 and –gres=gpu:2#

Solution: Discovered cluster has no GRES configuration (sinfo shows null for all nodes) - removed –gres flags and requested GPUs via node allocation only

Key Insight: Not all SLURM clusters configure GRES; some expose GPUs directly on compute nodes without resource scheduling

AI Limitations#

General Limitations#

  • Spent extensive time trying SLURM permission variants (different partitions, explicit –account flags, sacctmgr queries) despite early evidence that no partitions were accessible; should have pivoted to alternative approaches sooner
  • Attempted to use –gres=gpu:1/2 without first verifying cluster GRES configuration, wasting one submission cycle before discovering GRES was unconfigured (sinfo showed null for all GRES fields)

Learnings#

Key Learnings#

  • HPC cluster access patterns vary widely - always check both SLURM (sbatch/srun/squeue) and direct SSH access, verify GRES configuration before using –gres flags, and examine bash_history to understand historical access methods
  • When infrastructure blockers are fundamental (no partition access, SSH blocked by pam_slurm_adopt), deliver ready-to-run automation scripts with clear documentation rather than partial workarounds, so work resumes immediately once admin restores access

Issues and Solutions#

Key Issues#

1. Panda robot arm gripper action polarity contradicts code assumptions: Panda requires action=+1 to close, but the code hardcodes -1, resulting in all injections failing when the gripper is open, causing failures for coffee/stack/stack_three/throttling injections.#

Solution: In EnvWrapper.init, check the robot arm type through env.robots[0].gripper.class.name. Set PandaGripper’s close_action to +1.0 and RethinkGripper’s close_action to -1.0; add get_gripper_action_close/open() helper and replace all hardcoded actions [-1].

Key insight: The comments stating “-1=open, +1=closed” indicate different directions of internal multipliers for Panda and Sawyer, leading to opposite behavior. It cannot be determined by comments; a diagnostic script must be written to measure qpos changes.

2. Three_piece_assembly scan count decreased from 23 to 4: Out of 887 opportunities, 879 phases were filtered out as pre_reach due to valid_phases.#

Solution: Root cause analysis: get_target_object() and InteractionSegmenter both selected targets based on nearest distance among all objects. The base fixture’s lift_height could never reach 0.84 due to z≈0.80; the repair solution is to change both to _get_graspable_objects() priority, with all objects being returned if necessary. Code changes are pending next session due to sandbox restrictions.

Key insight: In multi-object assembly tasks, fixtures will consistently win the ’nearest object’ competition. Target_object selection must be based on grasp_geoms rather than pure distance.

3. wrong_object generated 0 scenarios in coffee/three_piece_assembly: Skills attempted to grasp fixed devices without grasp_geoms (e.g., coffee machines), where EEF could not reach the center and grippers failed to close.#

Solution: Filter out non-target candidate objects that have non-empty grasp_geoms in the task configuration’s objects list, excluding fixed devices.

Key insight: If both grabble objects and fixed devices exist in the task configuration, skills must filter based on grasp_geoms rather than treating all objects as wrong targets.

4. drop_with_interaction generated 0 scenarios across all 6 tasks: The original implementation only released objects when they were moved 1-3cm above the current position, with object spacing of 10-20cm ensuring no contact.#

Solution: Rewrite the inject logic: move the object directly above the non-target object (+0.15m) and then release it to fall freely onto the non-target; track object-object contact throughout the settle process in rollout_utils.py.

Key insight: Object interaction with other objects requires active movement to the collision point. Random motion after a small offset cannot expect to hit neighbors perfectly.

5. Hugo list template section condition filtering experienced three iterations of failure (Section casing issues, lower(.RelPermalink) still failed), before converging.#

Solution: Use eq .Title “bugJournal” as the condition to completely avoid differences in case handling within Hugo.

Key insight: Hugo has inconsistent casing handling for .Section and .RelPermalink on different versions/platesforms. .Title is the most stable identifier for section list pages.

6. PaperMod theme header.html does not support nested sub-menus.#

Solution: Create a custom header.html in website/layouts/partials to override the theme default template, add has-submenu/submenu class rendering logic, and implement hover dropdown effects using bugjournal-menu.css.

Key insight: Hugo supports overriding themes with project-level layouts/partials. The parent field in config.yml declares sub-menu relationships, eliminating the need to fork or modify the theme source code.

7. Static report directory static/benchmark/ conflicts with content page content/benchmark.md, causing /benchmark/index.html not to be generated; wrapper frontmatter using datetime.now() was recognized as future content and not published.#

Solution: Renamed the static report directory to benchmark-report/, updating internal links in the packaging pages; removed dynamic timestamps and replaced them with fixed past dates.

Key insight: Hugo uses the same name for static and content directories, causing the static directory to overwrite the content rendering results. Automatically generating frontmatter should not use the current system time as the date.

General Issues#

8. Damaged PaperMod theme pack index (._pack*.idx) caused git pull failures (non-monotonic index), and Hugo v0.157.0 required a new partial but the old theme was missing, leading to build failures.#

Solution: Delete macOS resource fork files (._pack*.idx and ._pack*.pack), then use git stash + git pull origin master + git stash pop; update the theme to the latest version, manually resolve head.html merge conflicts by keeping local MathJax and checkbox scripts.

Key insight: macOS-generated._prefix resource fork files can contaminate the.git/objects/pack directory; themes need to be updated when Hugo major versions are upgraded, and locally customized layouts need to be manually retained during merges.

9. bwrap sandbox was unavailable on Tianhe node (Unknown option –argv0), resulting in about 10 tool calls failing.#

Solution: Use the parameter sandbox_permissions=‘require_escalated’ to request elevated privileges, bypassing bwrap restrictions and allowing file reads to proceed normally.

Key insight: The Codex environment’s sandbox is unavailable on certain HPC nodes by default. Escalated permissions are the only viable solution; the apply_patch tool also suffers from this limitation.

10. update.sh fails when executed in set -e mode, returning non-zero exit codes for find cleanup of public/ and committing without changes, causing script termination.#

Solution: Add 2>/dev/null || true at the end of the find command; use git diff –cached –quiet before commit to check for changes, skipping commit and push steps if there are no changes.Key Insights: All cleanup operations in the shell script set -e mode are protected by || true; the deployment script should gracefully exit without errors when there are no changes (idempotency design).

  • When dealing with the internal behavior of underlying libraries (robosuite), AI cannot draw conclusions through static code analysis. It requires testing the diagnostic script to confirm. The initial design of drop_with_interaction had fundamental flaws (small offsets rather than active handling). This flaw was not questioned by the AI during multiple rounds of repairs until it was pointed out by the user.
  • Codex bwrap sandbox limitations caused approximately 10 tool calls to be invalid (on tianhe node). The AI failed to recognize this earlier and needed elevated privileges. The apply_patch tool also had limitations. The complete code patch (envwrapper + InteractionSegmenter double fixes) was not implemented, and the conversation was interrupted at critical points.
  • Hugo template fields lack sufficient understanding of semantics: section conditional statements required three iterations to converge; failed to anticipate conflicts between static directories and content paths; failed to anticipate future date filtering caused by datetime.now(). These are errors that can be avoided in the design phase.

Core Gains#

  • In robosuite, PandaGripper and RethinkGripper have opposite action polarities: Panda action = +1 for closing, Sawyer action = -1 for closing. The code comments are not reliable, and it must be confirmed through testing qpos changes. When diagnosing unknown behaviors, writing a minimal test script is more reliable and efficient than static code analysis, especially when dealing with the internal behavior of third-party libraries.
  • In multi-object assembly tasks, the selection of target_object must be based on grasping_geoms filtering out fixtures. The pure “nearest object” strategy will systematically select the wrong target when fixtures are close to EEF, resulting in all phase_labels being pre_reach. Non-target candidates for error skills also need to distinguish between graspable objects and fixed devices.
  • The architecture pattern in outputs/site staging area: tools write to outputs/site → website/sync_staging.py synchronization → Hugo build → deployment. This decoupling allows each tool’s release logic to be independent of the website directory structure, with sync_staging executing incremental updates as a single entry point.
  • In physical simulation, making objects interact with other objects requires active handling to the collision position, not relying on natural movement after random offsets (object spacing is usually much larger than the offset). The ungenerable cases caused by benchmark verification thresholds (collision detection at 1cm displacement, position error of 2cm) are physical constraint problems due to insufficient injection strength. It is necessary to adjust injection parameters or verify thresholds instead of running the pipeline again.
  • Hugo best practices collection: .Title is the most reliable field for identifying specific sections (.Section and .RelPermalink have inconsistent capitalization across platforms); static directory names that match the basic name of content pages will cause path conflicts (static files overwrite content rendering results); automatic frontmatter generation should not use the current system time; dropdown menus can be implemented through project-level layouts/partials/safety covering themes without having to fork themes.

Practical Gains#

  • macOS resource fork files with a._ prefix can contaminate git pack directories, causing non-monotonic index errors. After Hugo major version upgrades, themes need to be updated synchronously. Cleanup operations in shell script set -e mode should be protected by 2>/dev/null || true. The deployment script should gracefully skip commits when there are no changes (idempotency design).
  • Slurm ai partitions must specify –gres=gpu:a800:[N] even if the task does not use a GPU. MUJOCO_GL=disabled allows running physical simulations (CPU-only) without GPU rendering. Batch archiving strategies (generating multiple weeks’ worth of weekly reports at once) are efficient methods for creating retrievable knowledge bases.

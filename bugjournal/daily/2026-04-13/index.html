Problems and Solutions#

Key Problems#

1. Teddy+UNI2 multi-modal fusion led to decreased white matter recognition ability: ARI when using only Teddy was 0.186, merged to 0.151 (-19%), average 0.153 (-18%). Root cause: Teddy reached L2 normalization before, its variance was 4.3e-5 for each dimension (460 times smaller than 2.0e-2 achieved by PCA in 50 days). After L2 normalization, gene signals contributed only 13.3% of total variance during integration, while visual factors accounted for 86.7%.#

Solution: Background agent analyzed embedded data from section 151673 and found inconsistent variance distribution. Fix: Reduce Teddy’s variance before integration, use weighted variance merging, or use standard scaling instead of L2 normalization.

Key Insight: Pre-normalized high-dimensional embeddings lose signal strength after L2 normalization. Fusion strategy must consider input variance distribution. High dimensionality itself does not guarantee discrimination ability—variance for each dimension is crucial.

2. STAIG fusion showed no difference between Teddy and PCA encoders (Teddy average was 0.495, PCA average was 0.467, difference was +0.028, noise standard deviation was 0.119). All gene encoders using STAIG fusion were ineffective.#

Solution: Code inspection revealed line 1253 in model/Fusion.py: staig_fusion passed gene_feat=staig_gene_feat_3000.npz (original HVG), completely ignoring gene_emb parameter. Teddy+STAIG and PCA+STAIG used the same input. Observed ARI difference was due to random GCN training noise, not systematic gene encoder effects.

Key Insight: STAIG fusion architecture has flaws: gene encoder parameters are non-operational functions. Regardless of which encoder used, all gene encoders simplified to original HVG input. This makes gene encoder comparisons using STAIG as a fusion method ineffective.

3. Single cursor (_current_line_start) in MeetingHelper could not track two independent local lines (microphone and system audio) simultaneously, causing text corruption and overwriting issues.#

Solution: Region-based architecture was re-designed: subtitle window divided into two areas (MIC/SYS), each with independent cursor, file handle, local line buffer, and finalization logic. Opus sub-agent identified this fatal concurrency issue before implementation.

Key Insight: Concurrency transcription output requires independent state management for each source. Shared variable states (cursors, buffers) are the root of concurrency issues. CCPlan adversarial audit phase can prevent fundamentally infeasible architectures from being implemented.

4. Portable collection script hard-coded 6 tasks, requiring manual updates in multiple files to accommodate new tasks. Solution: Task definition was centralized in task_registry.yaml, allowing all scripts to dynamically retrieve task list. Portable registry was generated during packaging, with asset verification mechanism for quick failure (including task_config, HDF5, scenarios).#

Key Insight: Centralized configuration and dynamic derivation avoided multi-file consistency issues, making it easier to scale than manually syncing scattered hard-coded lists.

5. Five interrelated errors in error recovery process led to 100% failure rate of MimicGen enhancement (success rate 0/10): controller cache was asynchronous, quaternion format mismatch, missing leading zero action frames, lack of stability mechanism, and demonstration verification order issues.#

Solution: These five errors were systematically fixed: 1) Controller synchronization after state modification; 2) Quaternion wxyz→xyzw conversion; 3) Removal of leading zero action frames; 4) Addition of stability mechanism; 5) Demonstration verification before enhancement. Success rate increased to 40% (4/10). These fixes applied to the recovery mainline, but only partially to the training source data process.

Key Insight: Complex system failures are often not single-point issues, but a chain of minor errors. It’s necessary to systematically identify and fix the entire causal chain. When reusing code across dependent paths, it’s essential to clearly track to ensure fixes are applied to all locations.

6. libero2lerobot_parallel.py performance dropped significantly after processing many episodes (performance became O(n²) as dataset size increased).#

Solution: The line dataset.hf_dataset = dataset.create_hf_dataset() in per-episode loop was removed (line 224). This operation re-scans all existing parquet files, resulting in O(n²) performance decline as dataset size increases. save_episode() can now persist data; only re-building hf_dataset is needed when dataset will be read in the future.

Key Insight: Incremental data operations should avoid completely rebuilding the entire dataset in loops. Always perform performance analysis on long-running processes to detect hidden O(n²) patterns—especially those that re-scans from scratch with each iteration.

7. On RTX 2080 Ti (11GB VRAM), CUDA illegal memory access and OOM errors occurred during HD basic_contrastive and QFormer fusion. RTX 2080 Ti had insufficient memory for 17,502 positions × 550 cycles.#

Solution: Job resubmitted to gpu-common partition with explicit GPU constraints gres=gpu:5000_ada:1 (24GB VRAM), memory increased to 64GB, time limit extended to 2 days. QFormer separated for dedicated experimental configuration to isolate memory-intensive fusion operations.

Key Insight: Large-scale spatial data (17K+ positions) requires at least 24GB VRAM for contrastive learning. SLURM GPU type syntax: –gres=gpu::: (e.g., gres=gpu:5000_ada:1) to request specific GPU models in heterogeneous partitions.

8. LeRobot 0.4.0 updated add_frame() function signature to require tasks as separate position parameters, rather than embedding them in frame dict, which disrupted existing conversion code.#

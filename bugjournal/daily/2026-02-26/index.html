Problems & Solutions#

Critical Issues#

1. Pi0.5 achieved 0% success rate despite completing 99,999 training steps, BC-RNN had input dimension mismatch (Expected 65, got 37)#

Solution: Root cause analysis revealed Pi0.5 was trained on 5 MimicGen tasks (Coffee, Stack, Threading, Assembly, Stack Three) but evaluated on PickPlace which wasn’t in training data. BC-RNN dimension mismatch was due to create_env() stripping ‘_D0’ suffix from env names, causing base robosuite env to return fewer object observables than MimicGen variant. Solution: import mimicgen to register _D0 envs, use full env names, and align training/evaluation tasks

Key Insight: Task distribution mismatch is a critical but often overlooked failure mode in robot learning evaluation; environment naming conventions can silently break observation consistency between training and deployment

2. BC-RNN training configs were in low_dim mode (state-based) but MimicGen paper uses image mode (pixel-based) as primary benchmark#

Solution: Rewrote all 5 BC-RNN configs to image mode: removed ‘object’ from obs.low_dim, added agentview_image and robot0_eye_in_hand_image to obs.rgb, configured ResNet18Conv+SpatialSoftmax encoder, increased RNN hidden_dim from 400 to 1000, reduced batch_size from 64 to 16, set hdf5_cache_mode to ’low_dim’

Key Insight: Image-based BC-RNN requires fundamentally different architecture (ResNet18 visual encoder, larger RNN, smaller batches) compared to low_dim mode; official MimicGen benchmark uses image mode with 600 epochs

3. Evaluation framework only supported PickPlace task, couldn’t evaluate on the 5 tasks Pi0.5 was actually trained on#

Solution: Built comprehensive multi-task framework: created 5 task YAML configs with task-specific objects, phases, and thresholds; created task_registry.yaml mapping task names to datasets/checkpoints/prompts; modified 1_generate_scenes.py, 3_collect_data.py, 4_analyze_results.py to accept –task parameter with auto-resolution; rewrote run_full_eval.sh to loop over tasks

Key Insight: Evaluation infrastructure needs to be task-agnostic from the start; retrofitting multi-task support requires coordinated changes across configs, scripts, and orchestration layers

4. Unclear whether to use official BC-RNN checkpoints or self-train, and which observation modality to use#

Solution: Investigated MimicGen repo and confirmed they provide only datasets, not pre-trained checkpoints. User clarified preference for image mode (pixel-based with ResNet18, 600 epochs) to match paper’s primary results, rather than low_dim mode (state-based, 2000 epochs)

Key Insight: Benchmark papers often don’t provide model weights, only datasets; explicitly confirming configuration choices (observation modality, training duration) prevents wasted effort on wrong setup

5. Needed to understand complex multi-repo evaluation infrastructure spanning error_recovery_benchmark and Motion-based-Self-Reflection-Framework with interdependent components#

Solution: Launched 3 parallel exploration agents to map out training setup, eval infrastructure, and pipeline simultaneously, then synthesized findings

Key Insight: Parallel exploration of independent aspects dramatically speeds up understanding of complex systems compared to sequential investigation

Learnings#

Key Learnings#

  • Task distribution mismatch between training and evaluation is a critical failure mode that manifests as 0% success rate. In this case, Pi0.5 trained on {Coffee, Stack, Threading, Assembly, Stack Three} but evaluated on PickPlace, causing complete failure despite successful training.
  • MimicGen paper provides only demonstration datasets (HDF5 files), not pre-trained model checkpoints. Researchers must train BC-RNN models themselves using robomimic, which is the official benchmark protocol.
  • BC-RNN has two distinct observation modes with fundamentally different architectures: low_dim (state-based, 2000 epochs, larger MLP) vs image (pixel-based, ResNet18 encoder, 600 epochs, larger RNN). MimicGen paper’s primary results use image mode.
  • Environment naming conventions in robosuite/MimicGen have semantic implications: stripping ‘_D0’ suffix changes which env class is instantiated, which alters object observables returned. Base robosuite envs return fewer object states than MimicGen ‘_D0’ variants, causing dimension mismatches (65→37 dims).
  • VLA policy evaluation requires coordinating multiple components: checkpoint location, policy server (websocket/TCP based), evaluation script, conda environment management, and GPU resource allocation via Slurm - missing any component blocks the entire pipeline
  • Multi-task evaluation infrastructure requires coordinated design across layers: task configs (YAML), registry mapping (datasets/checkpoints/prompts), script parameters (–task flag with auto-resolution), and orchestration (task loops in bash scripts). Retrofitting is more expensive than upfront task-agnostic design.
  • Image-based robot learning policies need camera rendering enabled during both training data collection and evaluation rollouts. Camera resolution must match training (84×84 for BC-RNN image mode) to avoid visual encoder input dimension mismatches.
  • Parallel exploration agents are highly effective for understanding multi-component systems - exploring training setup, evaluation infrastructure, and pipeline code independently then synthesizing enables faster comprehension than sequential investigation
  • The Pi0.5 evaluation stack uses a two-process architecture: policy server (openpi05 conda env) serves model via websocket, evaluation script (mimicgen_env) runs environments and queries server over network

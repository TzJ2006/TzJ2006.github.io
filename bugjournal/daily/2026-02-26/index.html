Problems and Solutions#

Key Problems#

  • 1. Pi0.5 completed 99,999 steps but had a 0% success rate. The input dimension of BC-RNN did not match expectations (65 instead of 37). Solution: It was found that Pi0.5 was trained on 5 MimicGen tasks (coffee, stack, thread, assembly, stack three), but the evaluation was done on PickPlace, which was not in the training data. The dimension mismatch in BC-RNN occurred because the create_env() function removed the ‘_D0’ suffix from the environment name, resulting in fewer object observations returned by the base robosuite environment compared to the MimicGen variant. Solution: Imported mimicgen to register the _D0 environment, used the full environment name, and adjusted the training/evaluation tasks. Key Insight: Task distribution mismatch is a critical but often overlooked issue in robot learning evaluation; environment naming conventions can disrupt the consistency of observations between training and deployment.

  • 2. BC-RNN training configurations were in low-dimensional mode (state-based), while the MimicGen paper used image mode (pixel-based) as the main benchmark. Solution: Rewrote all 5 BC-RNN configurations to use image mode: removed ‘object’ from obs.low_dim, added agentview_image and robot0_eye_inhand_image to obs.rgb, configured ResNet18Conv+SpatialSoftmax encoder, increased the RNN hidden layer dimension from 400 to 1000, reduced the batch size from 64 to 16, and set hdf5_cache_mode to ’low_dim’. Key Insight: Image-based BC-RNN requires a completely different architecture (ResNet18 visual encoder, larger RNN, smaller batch size) compared to low-dimensional mode. The official MimicGen benchmark uses image mode, with 600 cycles.

  • 3. The evaluation framework only supported the PickPlace task, making it impossible to evaluate the 5 tasks actually trained by Pi0.5. Solution: Built a comprehensive multi-task framework: created 5 task YAML configuration files containing task-specific objects, stages, and thresholds; created task_registry.yaml to map task names to datasets/checkpoints/prompting; modified 1.generate_scenes.py, 3.collect_data.py, and 4.analyze_results.py to accept the –task parameter and automatically parse it; rewritten run_full_eval.sh to loop through tasks. Key Insight: The evaluation infrastructure needs to be independent of tasks from the beginning; adding multi-task support requires coordinating changes in configurations, scripts, and orchestration layers.

  • 4. It was unclear whether to use the official BC-RNN checkpoints or train independently, and which observation mode to use. Solution: Investigated the MimicGen repository and confirmed that they only provided datasets, not pre-trained checkpoints. Users explicitly expressed their desire to use image mode (pixel-based ResNet18 model, 600 cycles) to match the main results of the paper, rather than low-dimensional mode (state-based model, 2000 cycles). Key Insight: Benchmark papers usually do not provide model weights, only datasets; clarifying the configuration choices (observation mode, training duration) can prevent wasting time on incorrect settings.

  • 5. There was a need to understand the complex multi-repository evaluation infrastructure, including error_recovery_benchmark and Motion-based-Self-Reflection-Framework, which contain interdependent components. Solution: Started 3 parallel exploration agents, mapped training settings, evaluation infrastructure, and flowcharts, and then synthesized the findings.

Lessons Learned#

Key Lessons#

  • The mismatch between training and evaluation tasks was a serious failure mode, resulting in a 0% success rate. In this case, Pi0.5 was trained on {Coffee, Stack, Threading, Assembly, Stack Three}, but evaluated on PickPlace, leading to a system failure despite successful training.
  • The MimicGen paper only provided demonstration datasets (HDF5 files) and did not provide pre-trained model checkpoints. Researchers had to train the BC-RNN model themselves using robomimic, which is the official benchmark protocol.
  • BC-RNN has two different observation modes, and its architecture is also completely different: low-dimensional mode (state-based, 2000 cycles, larger MLP) and image mode (pixel-based, ResNet18 encoder, 600 cycles, larger RNN). The main results of the MimicGen paper used the image mode.
  • The environment naming convention in robosuite/MimicGen has semantic meaning: removing the ‘_D0’ suffix would change the environment category instantiated, thereby changing the object properties returned. The base robosuite environment returns fewer object states than the MimicGen ‘_D0’ version, resulting in a dimension mismatch (65 → 37 dimensions).
  • The VLA strategy evaluation requires coordination of multiple components: checkpoint locations, strategy servers (based on websocket/TCP), evaluation scripts, conda environment management, and GPU resources allocated through Slurm—any one of these components can hinder the entire process.
  • The multi-task evaluation infrastructure needs to be designed with coordination at different levels: task configuration (YAML), registry mapping (datasets/checkpoints/prompting), script parameters (with the –task flag for automatic parsing), and orchestration (task loops in bash scripts). Later adjustments are much more expensive than designing tasks without them.
  • Image-based robot learning strategies require camera rendering capabilities during both training data collection and evaluation processes. The camera resolution must match the training resolution (84×84 for BC-RNN image mode) to avoid mismatches in input dimensions for the visual encoder.
  • Parallel exploration agents are very effective for understanding multi-component systems—exploring training settings, evaluation infrastructure, and process code separately, and then synthesizing this information to understand the system faster than sequential investigations.
  • The Pi0.5 evaluation stack uses a dual-process architecture: the strategy server (openpi05 conda environment) serves the model through websocket, and the evaluation script (mimicgen_env) runs the network environment and queries the server.

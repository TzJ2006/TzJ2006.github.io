Problems and Solutions#

Key Issues#

1. Pi0.5 evaluation had a success rate of 0%. Initially, it was suspected to be a model training problem or action clipping issue.#

Solution: The root cause was a complete task mismatch: Pi0.5 was trained on 5 MimicGen tasks, but the evaluation used PickPlace (not part of the training set). This problem was solved by changing the evaluation method to use the training tasks.

Key Insight: Ensure that the training dataset named local/mimicgen_phoenix_9tasks actually contains 5 MimicGen tasks. PickPlace represents a completely different task system, and a 0% success rate is inevitable. Aligning the evaluation framework with the training tasks is crucial for successful experiments.

2. RuntimeError in BC-RNN inference: input.size(-1) must equal input_size. The expected value was 65, but it was actually 37.#

Solution: In create_env(), Coffee_D0 was removed, causing additional observation data to be lost in the MimicGen environment. This problem was solved by registering the _D0 environment with import mimicgen, retaining all environmental details.

Key Insight: The _D0 environment provides additional object states through the robosuite registration mechanism. Removing the suffix causes the observation dimension to differ from that during training.

3. Challenges in Pi0.5 evaluation infrastructure: No available GPUs on the login node, unclear relationship between two codebases (tangzijia vs zhaoganlong), and an empty evaluation result directory.#

Solution: Use squeue to confirm that the user has a reusable interactive GPU node (an49, task number 45628). Clarified the relationship between two codebases: prioritize using tangzijia’s evaluate_mimicgen.py (adapted for 9 MimicGen tasks, using encapsulated TCP protocol), while zhaoganlong’s serve_policy.py uses WebSocket protocol and requires compatibility checks. An empty evaluation result directory indicates this is the first evaluation. Key Insight: The two codebases use different communication protocols (TCP vs WebSocket), which needs to be confirmed before evaluation. Having an existing SLURM interactive job can avoid reapplying for GPU resources.

4. BC-RNN training configuration was initially set to low_dim mode, which did not match the settings in the MimicGen paper.#

Solution: Reviewed the official MimicGen configuration generation script and confirmed that the paper used image mode (84×84, ResNet18 encoder). The configurations of all 5 tasks were modified.

Key Insight: The main results in the MimicGen paper are based on image observation data. low_dim is not a benchmark comparison metric. The main parameters include: hidden_dim=1000, actor_layer_dims=[], batch_size=16, CropRandomizer 76×76.

General Issues#

5. SLURM sbatch submission failed: The user’s group was not allowed to use this partition (xy-a800).#

Solution: An existing interactive job (45628, an49) was found running. Using srun –jobid=45628 –overlap allowed running 5 training processes on the existing allocated resources, thus bypassing the partition permission restriction.

Key Insight: srun can be attached to an existing SLURM interactive job allocation. When sbatch permissions are unavailable, this is a viable alternative.

6. LaTeX resume publication section: Long titles were not formatted correctly using line breaks.#

Solution: Changed the single-line inline format to a two-line format using resumeSubheading: The first line was the title, and the second line was the author role and impact factor.

Key Insight: In LaTeX, long text should be used in two lines to avoid overflow. The appropriate command should be used based on the text length.

Today’s Summary#

Key Conclusions#

  • Ensuring the evaluation framework is consistent with training tasks is crucial for successful experiments: When designing the evaluation process, it is necessary to confirm the specific tasks covered by the training set. Any abnormal success rate should prioritize checking the consistency of task coverage issues.
  • MimicGen does not provide pre-trained checkpoints but only offers HDF5 demonstration datasets. The benchmark algorithm BC-RNN needs to be trained independently. The main results in the paper use image observation data (ResNet18 + SpatialSoftmax 84×84), rather than low_dim. _D0 environment must be registered with import mimicgen to robosuite; it is not allowed to remove the suffix, otherwise the observation dimension will differ, leading to BC-RNN inference failure. The main parameters include: hidden_dim=1000, actor_layer_dims=[], batch_size=16, CropRandomizer 76×76, hdf5_cache_mode=low_dim.
  • Pi0.5 evaluation adopted a two-stage architecture: ① Start the WebSocket policy server on GPU nodes using the openpi05 conda environment; ② Execute evaluate_mimicgen.py at the evaluation end to connect to the server and repeat the execution of 9 MimicGen tasks, with each task iterating 20 times. The phoenix_comparison checkpoints contain complete checkpoints from step 10,000 to 100,000, allowing for multi-step comparisons during evaluation.
  • Two existing codebases (tangzijia’s error_recovery_benchmark and zhaoganlong’s motion self-reflection framework) use different communication protocols: the former uses TCP sockets, while the latter uses WebSocket. Compatibility needs to be confirmed before use; prioritize using tangzijia’s version (adapted for handling 9 MimicGen tasks).

Practical Summary#

  • srun –jobid= –overlap can be attached to an existing SLURM interactive job to run additional processes, making it a viable alternative when sbatch partition permissions are unavailable.

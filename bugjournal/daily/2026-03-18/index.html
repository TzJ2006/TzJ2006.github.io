Problems and Solutions#

Key Issues#

1. π₀.₅ Pi0WithCompletionHead uses a combination method (self.pi0 = Pi0(…)), causing all parent module parameter paths to have an extra pi0/ prefix. When merging pytree, it throws a ‘2 children vs 1 child’ ValueError, making checkpoint loading impossible.#

Solution: Changed to inheritance method (class Pi0WithCompletionHead(Pi0)). Pi0 parameters were directly aligned with the checkpoint path at the top level; completion_head retained random initialization through missing_regex.

Key Insight: The checkpoint path for Flax NNX was determined by the module tree structure: Combination added a prefix before all parent module parameter paths, while inheritance did not—requiring loading of existing checkpoints’ extended models. Inheritance was the only feasible solution.

2. Robotwin output transform silently discards manipProgress: RobotwinOutputs.call only returns {“actions”:…}. Even if the model correctly predicts the progress field, it cannot be accessed externally, and the eval txt file remains empty.#

Solution: Modified the output transform’s return dict to include the manipProgress field, re-establishing data flow.

Key Insight: The output transform is an implicit filter in the reasoning chain; any field not in the return dict is silently discarded. To troubleshoot such bugs, trace upstream from the end of the data flow instead of assuming correct output upstream.

3. CalendarPro system returns ‘I’m not sure what you want to do’ when users report batch task status, rather than processing the request.#

Solution: Analyzed the entire root cause chain: Low similarity between semantic routing and LLM classifier (≈0.20 vs 0.50 threshold) → LLM classifier had no batch_update intent → GENERAL handler clarification branch discarded AI response output hard-coded prompt. Added BATCH_UPDATE support at the routing-classification-processing full linkage stage, with keyword counting rules (2+completion verbs → +0.30 boost) to compensate for embedding blind spots.

Key Insight: The single-intent architecture naturally has low similarity in embedding multi-task messages; pure semantic routing is insufficient. GENERAL fallback should first determine whether the LLM response is substantial content (>20 words and not a template sentence) before deciding whether to use it.

4. In all_aligned mode of MIHD, the –alignment parameter was not injected into the evaluate stage, resulting in EvaluationJob.alignment being always None, and Harmony alignment results being completely ignored.#

Solution: Added three lines of code before the run_pipeline.py evaluate phase: When phase == ‘all_aligned’ and args.alignment has a value, inject the alignment into extra_config[‘alignment’] for each experiment.

Key Insight: The isolation design between CLI parameters and experimental configurations in multi-stage pipelines can cause data flow disruptions when introducing new stages. Each new parameter across stages requires checking the entire data flow path.

5. The semantics of prefix_output in pi0.5 are unclear: The documentation does not explain the physical meaning of prefix/suffix. Current code discards prefix_output, making it difficult to determine the appropriate feature source for classification.#

Solution: Confirmed through code exploration that prefix=[BOS]+image+language global token, suffix=action expert and VLM interaction token; mean pooling semantics of prefix_output is clear for classification tasks.

Key Insight: Analyzing the feature boundaries of VLM-Action expert is a key prerequisite for designing classification headers. It’s necessary to read the entire forward call chain.

6. Explanation ambiguity in ‘multiple forward passes’ during training with from_hidden mode: Users cannot understand the difference between training and reasoning calculations multiple times.#

Solution: Compared and analyzed to clarify the mechanism: Reasoning always involves two steps; during training, Exp1-4 can use teacher forcing (teacher labels known in advance) to combine prefix+suffix into a single joint forward optimization; from_hidden mode introduces cyclic dependencies due to conditioning dependent on model output, making this optimization impossible. An additional prefix-only forward is required.

Key Insight: The calculation diagrams for training and reasoning are different: Teacher forcing makes conditioning a constant that can be used for joint forward optimization; from_hidden mode introduces cyclic dependencies, requiring returning to the same two-step process as reasoning.

7. STAIG original batch effect metrics (batch_kl, batch_entropy) depend on R package nabor and cannot run in a general Python environment.#

Solution: Replaced R nabor with sklearn.neighbors.NearestNeighbors, rewriting all kNN query logic in pure Python.

Key Insight: Language-dependent dependencies (Python↔R) are common friction points in ML projects. Python alternatives are usually available and easier to maintain.

General Issues#

8. Multi-path contamination in LIBERO caused registration repairs not to take effect: There were three LIBERO paths in PYTHONPATH, the wrong path was modified, but what actually loaded was openpi/third_party/libero.#

Solution: Used python -c 'import libero.libero.benchmark; print(__file__)' to confirm the actual loading path at runtime. Modified and copied the bddl file to that path.

Key Insight: When multiple versions of packages coexist, the editable install .pth file may not work because MAPPING is empty. The actual loading order is determined by PYTHONPATH and must be confirmed at runtime.

9. RepackTransform reported a KeyError. In the dataset, the task completion label actually has the key name observation.task_completed instead of task_completed.Solution: Change the mapping in config.py from ’task_completed’:’task_completed’ to ’task_completed’:‘observation.task_completed’#

Key Insight: The LeRobot dataset key names use dot-separated nested paths, and the value field of RepackTransform must exactly match the original dataset path

10. When manually exporting CUDA_VISIBLE_DEVICES=0,1 in K8s containers, JAX reports a cuInit failure with no visible GPU devices#

Solution: Remove the manual setting and use the container’s default allocation: The container actually only allocates physical GPU 1, forcing the inclusion of GPU 0 results in a complete failure of cuInit due to lack of access rights

Key Insight: CUDA_VISIBLE_DEVICES within K8s containers can only be remapped within the allocated devices. Use jax.devices() instead of nvidia-smi to verify the actual visible devices at the framework level

  • Insufficient prediction of framework-specific behaviors: Failure to anticipate the impact of Flax NNX inheritance and composition on checkpoint parameter paths during the design stage (only discovered through runtime pytree errors); underestimating the reuse potential of JAX/NNX’s existing TrainConfig mechanism (creating unnecessary new Config classes)

Today’s Achievements#

Core Achievements#

  • π₀.₅ training computation graph characteristics: Inference naturally has two steps (VLM prefix forward → action expert denoising). During training, Exp1-4 can combine prefix+suffix into a single joint forward using teacher forcing (conditioning=GT labels are known in advance). The from_hidden mode introduces cyclic dependencies due to conditioning relying on model output, requiring an additional prefix-only forward. This maintains the same two-step process as inference
  • π₀.₅ gradient flow mechanism: Action loss is transmitted back to VLM backbone through cross-attention of action expert (K/V comes from VLM prefix tokens). stop_gradient strictly limits aux loss to MLP parameters. Since both losses act on completely different parameter sets, a large difference in the ratio between action/aux losses does not indicate an imbalance during training
  • When extending Flax NNX with trained models, inheritance (class Child(Parent)) is the only way to maintain compatibility with checkpoint paths. Composition will result in a complete failure of checkpoint merge because adding a prefix before all parent module parameter paths causes issues
  • Harmony cross-slice alignment actually improves PCA embedding (batch_entropy 0.33→0.52, batch_kl 0.39→0.25), but the improvement is limited - The batch effect of per-section PCA mainly stems from inconsistencies in the feature space itself. Post-processing corrections are just a relief measure
  • Data flow transparency principle: In multi-stage pipelines (extract → align → evaluate), each stage’s configuration must be explicitly passed to downstream stages and cannot rely on implicit sharing. Output transform is an implicit filter in the inference chain, and fields that need to be transparently maintained must be clearly defined - “The model predicts something, but nothing can be seen from outside” is the most hidden type of bug, which requires tracking upstream from the end of the data flow
  • VLM’s prefix_output (global[BOS] + image + language token mean pooling) is more suitable for task completion classification than action expert’s suffix_output because it encodes global state understanding rather than step-by-step action information
  • Batch intent detection requires a specialized architecture: Embed similarity naturally has a low value for “multi-task state messages”. Keyword counting rules (2+completion verb→+0.30 boost) are effective supplements to fill in the blanks of embedding. GENERAL fallback should first determine whether the LLM response contains substantial content before deciding whether to use it or not. It should not discard it unconditionally
  • The freezing mechanism in existing frameworks (such as NNX’s freeze_filter + nnx.DiffState) is usually designed with scalability in mind. Reuse is preferred over creating new ones, which can significantly reduce the amount of code changes

Practical Achievements#

  • K8s container GPU isolation: All physical GPUs are visible to nvidia-smi, but CUDA runtime is isolated by the container’s cgroup. Manually setting CUDA_VISIBLE_DEVICES beyond the allocated range will cause a complete cuInit failure, not just limiting visibility of cards. It should always be verified using jax.devices() or torch.cuda.device_count() for the number of visible devices at the framework level
  • When multiple versions of packages coexist, the editable install .pth file may not take effect if MAPPING is empty. The actual loading path is determined by the PYTHONPATH order. It is necessary to confirm the actual loading path using python -c 'import pkg; print(pkg.__file__)' during runtime
  • Orbax checkpoint directory hierarchy: Steps directory (29999/) stores metadata and assets, while params/subdirectory stores actual parameters. The weight_loader path must point exactly toparams/subdirectory

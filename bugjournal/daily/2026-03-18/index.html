问题与解决方案#

关键问题#

1. MIHD all_aligned模式下alignment参数未传播到evaluate阶段：align阶段生成了对齐embedding，但evaluate阶段EvaluationJob.alignment始终为None，对齐工作被完全忽略。#

解决方案: 在run_pipeline.py evaluate阶段前，当phase==all_aligned且args.alignment有值时，将alignment注入各实验的extra_config，使EvaluationPlanner正确生成带alignment字段的EvaluationJob（3行代码修复）。

关键洞察: 多阶段pipeline中CLI参数需显式注入到每个阶段的配置对象，不能依赖上下文隐式传递；任何影响后续阶段行为的参数都应作为配置对象的显式字段。

2. CalendarPro批量任务状态更新（如「A查看好了，B和C已解决」）被系统判定为GENERAL clarification，返回「我不太确定您想做什么」。#

解决方案: 新增BATCH_UPDATE意图，在语义路由（21条话语示例）、关键词计分（2+完成动词+0.30 boost）、LLM提示（updates数组schema）三层同时修复；另将GENERAL clarification硬编码模板改为优先使用>20字的有效LLM回复。

关键洞察: 根因是三层级联失败：嵌入分低→枚举缺失→handler硬编码覆盖LLM有效回复；任一层遗漏都会导致问题重现，同时GENERAL fallback一直在静默丢弃LLM的有效判断结果。

3. RoboBrain Pi0WithCompletionHead使用组合模式导致所有Pi0参数路径加了pi0/前缀与checkpoint树形结构不匹配，PyTree merge失败；且checkpoint路径指向step目录而非params/子目录，Orbax restore失败。#

解决方案: 改为继承（class Pi0WithCompletionHead(Pi0)），Pi0参数直接在顶层与checkpoint对齐；checkpoint路径末尾加/params指向29999/params/子目录。

关键洞察: Flax NNX中加载预训练checkpoint必须用继承：组合会在参数路径中插入子模块名前缀；Orbax step目录只含元数据，实际参数数据在params/子目录，restore路径必须精确指向params/。

4. MIHD批次效应指标原STAIG实现依赖R包（kBET、lisi、nabor）在General环境不可用，无法直接移植。#

解决方案: 完全用Python重新实现：sklearn.NearestNeighbors替代R nabor，scipy.sparse.csgraph.connected_components替代R igraph，KL divergence用numpy手算，无需rpy2依赖。

关键洞察: 原始R依赖链（rpy2→nabor→kBET）是脆弱的；纯Python实现依赖更稳定且可移植性更强，指标结果等价。

5. RoboTwin output_transform（RobotwinOutputs）将manip_progress字段静默丢弃，导致progress txt日志无法生成，_last_infer_result中永远没有该键。#

解决方案: 在RobotwinOutputs.__call__中检测data中是否存在manip_progress，若存在则透传到返回dict。

关键洞察: output_transform是显式whitelist设计：任何非actions字段必须手动透传，框架不自动保留额外模型输出，这是一个容易造成静默数据丢失的设计模式。

6. RoboTwin progress conditioning中从标量直接linear投影到高维表示能力弱，且预测值存在超出[0,1]范围的情况。#

解决方案: 将linear投影替换为sinusoidal positional encoding（类NeRF/Transformer PE），输出后加clip(0,1)。

关键洞察: Sinusoidal encoding提供多频率分量，让模型在不同粒度感知进度值，比单一linear映射表达能力显著更强；clip是必要的推理稳定性保障。

一般问题#

7. openpi容器环境三重阻塞：socket.gaierror（容器DNS解析失败）、libero_object_com KeyError（三处libero副本路径冲突导致正确注册被覆盖）、手动设置CUDA_VISIBLE_DEVICES=0,1与K8s容器GPU映射冲突导致cuInit完全失败。#

解决方案: socket问题：UDP探测+127.0.0.1 fallback；libero冲突：inode比对确认路径关系后在openpi/third_party/libero完成注册并复制bddl；CUDA冲突：去掉手动设置使用容器默认kubelet注入值。

关键洞察: K8s容器的CUDA_VISIBLE_DEVICES由kubelet注入，手动覆盖会导致访问未授权设备；Python多editable install场景下venv的sys.path决定实际加载路径，需通过print(file)和inode比对确认。

今日收获#

核心收获#

  • Flax NNX加载预训练checkpoint必须用继承而非组合：组合在参数路径中插入子模块名前缀导致树形结构不匹配；Orbax step目录只含元数据，实际参数在params/子目录；凡是需要精确对齐checkpoint路径的场景，继承是唯一正确选择。
  • 多阶段pipeline的CLI参数传递陷阱：在extract→align→evaluate这类多阶段流程中，任何影响后续阶段行为的参数必须显式注入到每个阶段的配置对象中，不能假设上下文隐式传递；「对齐了但结果完全没用」这类静默错误比崩溃更难发现。
  • Teacher forcing在扩散模型计算图中的影响：Exp1-4用GT标签conditioning使训练可单次联合forward；from_hidden模式因conditioning依赖VLM输出，训练必须拆成prefix-only forward + 联合forward两步；推理时因本来就是两步所以无差异，差异只在训练计算图。
  • 批量状态汇报是对话AI的高频场景盲区：单意图架构无法覆盖「多任务×多动作」输入模式，必须同时在语义路由（多话语示例）、关键词计分（完成动词计数）、LLM提示（结构化schema）三层支撑，缺一都会导致检测失效；同时LLM fallback回复不应被硬编码模板覆盖，意图分类失败时LLM可能已给出合理回复。
  • 批次效应指标可完全用Python替代R实现：原STAIG的R依赖链（rpy2→nabor→kBET）可用sklearn.NearestNeighbors+numpy完全替代，不需要rpy2桥接，依赖更稳定且可移植；实验结论：Harmony对batch_entropy有明显改善（0.21→0.32），但graph_connectivity轻微下降（0.98→0.93），需在下游ARI指标上进一步验证实际收益。
  • output_transform的显式whitelist设计：框架将模型输出映射到用户格式时，任何非预期字段会被静默丢弃；所有需要透传的辅助输出（如manip_progress）都必须在transform中手动处理，这是框架层面常见的「静默数据消失」陷阱。
  • pi0.5的prefix_out（VLM最后层图像+语言token输出）是任务完成检测的最佳特征来源：不受flow-matching时间步污染、语义干净、不需要运行action expert即可获取，是优于action hidden state的分类输入选择。
  • K8s容器GPU分配机制：CUDA_VISIBLE_DEVICES由kubelet注入，手动覆盖为更大范围会导致cuInit访问未授权设备而失败；nvidia-smi绕过CUDA隔离可看到所有物理GPU，但JAX/PyTorch只能访问容器申请的GPU；EGL设备编号跟随CUDA_VISIBLE_DEVICES值，不跟随物理编号。
  • Flax NNX参数冻结完整链路：nnx.Not(PathRegex(…))作为freeze_filter → trainable_filter = nnx.All(nnx.Param, nnx.Not(freeze_filter)) → nnx.DiffState自动只对可训练参数计算梯度，无需手写任何梯度截断代码；可直接复用而非重新设计。
  • 「先写量化工具，再实现被量化的功能」的工程顺序：batch metrics作为对齐质量的客观依据，在对齐实现完成后立即给出可解释数字，避免黑箱状态；CLAUDE.md中最有价值的内容是「多文件变更模式」指南，而非文件枚举——新开发者最需要了解的是「新增某类功能需要改哪几个文件、按什么顺序改」。
  • Sinusoidal positional encoding用于连续标量条件化：相比单一linear projection，PE提供多频率分量让模型在不同粒度感知进度值，是NeRF、Transformer中的标准做法；BCE loss初始值0.25（非0.693）在任务完成标签严重不均衡的机器人数据集中是正常的。

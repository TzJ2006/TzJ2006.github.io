Problems and Solutions#

Key Issues#

1. BC-RNN had SR=0% on all normal tasks, which did not match the training rollout results#

Solution: Through code exploration, it was identified that an observation key bug in baseline_accuracy evaluation led to empty observations being received by the strategy. This was not a model capability issue but a bug in the evaluation script.

Key insight: When comparing training results with evaluations, it is necessary to check whether the observation space is consistent (key names, dimensions) rather than assuming that the model did not learn anything.

2. There are multiple discrepancies between the documented pipeline and the actual code implementation (VLA model type errors, outdated code statistics, M14 data being only early results)#

Solution: Systematically launch 3 parallel agents to explore various aspects of the code library, collect actual values (number of files, functions, evaluation results), and precisely edit the document line by line.

Key insight: Large project documents tend to drift from code. Regular alignment between “document” and “code” is crucial for maintaining readability, and the code should be used as the authority.

General Issues#

3. The data preparation script still has pdb.set Tracy() debugging breakpoints activated for only one task, coffee_d1, making it impossible to run all 9 tasks in batch mode#

Solution: By reading 4 scripts line by line, identify the location of pdb calls and the list of commented-out tasks, then use Edit tools to uncomment them and remove breakpoints.

Key insight: Debugging breakpoints left behind in shared code libraries can cause silent hangs in automated pipelines. This should be checked as a priority during implementation planning.

4. arXiv PDFs cannot be directly parsed (binary stream returned), and version 3 has no HTML version available (404 error)#

Solution: First, use WebSearch to find the v1 HTML version to obtain the core architecture, then use Moonlight literature review site and GitHub repository to supplement with differences in version 3.

Key insight: Not all arXiv papers have HTML rendering. Using secondary literature review platforms can help access updates to summaries.

5. The project overview.md contains over 25,000 tokens, and a single reading tool cannot fully capture the content#

Solution: Use offset+limit segmentation for reading, combine with Grep to precisely identify lines that need modification, avoiding full loading.

Key insight: For large documents, segmented reading combined with precise Grep is more efficient than full loading. Key changes should be identified by Grep before editing.

Today’s Takeaways#

Core Takeaways#

  • QueST’s core idea for cross-sample evaluation: When there is no niche-level ground truth, use cell type annotation + WWL Graph Kernel to construct pseudo ground truth. Then, use Pearson correlation coefficient to quantify similarity between the two rankings - this is a general method for establishing comparable benchmarks in unsupervised embedding space.
  • The performance gap between Pi0.5 Per-Task LoRA and the Global Model is significant (Stack_D0: ~95% vs ~24%). This indicates that the Foundation Model requires task-level fine-tuning to be effective on specialized tasks. The 58.9% average SR of the Global Model masks severe task imbalance.
  • A clear distinction between Normal SR and Error Recovery SR is a key design element in error recovery benchmarks. M14’s close to 0% Error SR does not indicate model failure but rather a core point that current strategies lack error recovery capabilities. This can be demonstrated by comparing it with Normal SR.
  • The three-module design of the zhaoganlong framework: The Motion Prediction Module (MPM) predicts future motion directions and encodes them into a 37-dimensional codebook → The Motion-Conditioned Diffusion Policy receives the codebook to generate actions → The Motion Correction Module (MCM) detects execution deviations and generates corrective instructions, forming a closed-loop self-reflection mechanism.

Practical Takeaways#

  • Large project documentation alignment strategy: Use code as the authority. Use parallel agents to collect actual status from multiple dimensions (error frameworks, scripts, outputs), then systematically edit documents section by section - this is 3-5 times more efficient than manual line-by-line comparison.

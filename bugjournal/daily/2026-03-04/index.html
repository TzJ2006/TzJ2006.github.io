Lessons Learned#

Key Lessons - QueST Evaluation Framework: When there are no existing real labels, use the Wasserstein Weisfeiler-Lehman graph to verify synthetic real labels constructed from cell type annotations, and then use the Pearson correlation coefficient to verify the similarity ranking of the models.#

  • Building multi-task machine learning processes through incremental prototype design (starting with one task) retains decentralized single-task configurations (annotations removed, hardcoded paths). To scale to production environments, it is necessary to systematically review data preparation scripts, dataset mapping JSON files, checkpoint directories, and evaluation configurations.
  • Document synchronization mode: Use parallel exploration proxies (one for each main directory), verify that all numbers come from actual files (summary.json, file system statistics), and then systematically update document content section by section. Do not rely on statements in documents—always check the actual situation of code and data.
  • Debugging checklist for training and evaluation mismatch: If a strategy performs well during training but fails during evaluation, systematically check: (1) observation dictionary keys; (2) environment initialization parameters; (3) action space boundaries; (4) data standardization; (5) camera rendering settings. BC-RNN had a recovery rate of 0% on normal tasks, but 0.28% on error recovery tasks, indicating environmental mismatch rather than strategy failure.
  • Cross-sample label query evaluation requires: (1) obtaining batch-invariant embeddings through adversarial training; (2) using graph kernels to provide real labels when labels are unavailable; (3) using dual metrics to evaluate the highest accuracy and overall ranking correlation.
  • For spatial transcriptomics, the Wasserstein Weisfeiler-Lehman graph kernel can convert discrete cell type labels into continuous label similarity scores by calculating the graph-level Wasserstein distance between subgraph representations.
  • Large-scale HPC training scheduling requires clear dependency mappings (stage 0 → stage 1+2 parallelism → stage 3 → stage 4 → stage 5), GPU allocation planning for each stage, and persistent session management (tmux) to handle multi-day training activities and SSH disconnections.
  • Multi-source data collection architecture should be clearly stated in documents: Generation of error scenarios requires data from different sources (for reproducible demonstration injections, for realistic policy deployment injections, for realistic natural fault capture). Flowcharts must show all sources as first-class citizens, not hidden in script options.
  • M14 evaluation results showed baseline performance: 6,474 scenarios involving 4 strategies. In 649 error scenarios, the random strategy had a recovery rate of 0%, while the BC-RNN had a recovery rate of 0.28%. Pi0 had a recovery rate of 0%, while Pi0.5 had a recovery rate of 0%. Nearly zero error recovery rates validated the difficulty of the benchmark tests and provided a basis for future improvements (expecting M15 human recovery data and LoRA fine-tuning to show positive improvement effects).
  • Best practices for checkpoint inventory: Record the full path, size, last modification date, and availability status of all model checkpoints. A large checkpoint directory (180GB) indicates many saved steps—it is necessary to record the specific steps used for evaluation. Date-based naming helps track the training timeline, but it is necessary to clearly associate steps with dates.
  • Expansion of error framework: There are 53 Python files (approximately 1.47 thousand lines of code), including 6 detectors, 4 injectors, 4 validators, and 20 classifiers, covering 3 domains (operation, planning, progress). The plugin architecture is implemented through DETECTOR_REGISTRY, INJECTOR_REGISTRY, VALIDATOR_REGISTRY, CLASSIFIER_REGISTRY. The cooling time of each detector (20 steps) prevents detecting junk behavior.

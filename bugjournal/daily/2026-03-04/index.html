Learnings#

Key Learnings#

  • QueST evaluation framework: when no ground truth niche labels exist, construct synthetic ground truth using Wasserstein Weisfeiler-Lehman Graph Kernel over cell type annotations, then validate model’s similarity rankings with Pearson correlation
  • Multi-task ML pipelines built via incremental prototyping (start with 1 task) leave scattered single-task configurations (commented-out task lists, hardcoded paths). Scaling to production requires systematic audit of: data prep scripts, dataset mapping JSONs, checkpoint directories, and evaluation configs.
  • Documentation synchronization pattern: Use parallel exploration agents (one per major directory), verify all numbers from actual files (summary.json, filesystem stats), then systematically update document section by section. Don’t trust document claims - always check code/data ground truth
  • Training-vs-evaluation mismatch debugging checklist: When policy performs well in training but fails in evaluation, systematically check: (1) observation dictionary keys, (2) environment initialization parameters, (3) action space bounds, (4) data normalization, (5) camera rendering setup. BC-RNN 0% SR on normal tasks + 0.28% on error recovery suggests env mismatch, not policy failure
  • Cross-sample niche query evaluation requires: (1) batch-invariant embeddings via adversarial training, (2) ground truth from graph kernels when labels unavailable, (3) dual metrics for top-1 accuracy and overall ranking correlation
  • For spatial transcriptomics, WWL (Wasserstein Weisfeiler-Lehman) kernel can convert discrete cell type labels into continuous niche similarity scores by computing graph-level Wasserstein distance between subgraph representations
  • Large-scale HPC training orchestration requires explicit dependency mapping (Stage 0 → Stage 1+2 parallel → Stage 3 → Stage 4 → Stage 5), per-stage GPU allocation planning, and persistent session management (tmux) to survive multi-day training campaigns and SSH disconnections
  • Multi-source data collection architectures should be explicit in documentation: Error scene generation benefits from diverse sources (demo injection for reproducibility, policy rollout injection for realism, natural failure capture for authenticity). Pipeline diagrams must show all sources as first-class citizens, not buried in script options
  • M14 evaluation results revealed baseline performance: 6,474 episodes across 4 policies on 649 error scenes shows Random (0% SR, 1.9% RP), BC-RNN (0.28% SR, 0.3% RP), Pi0 (0% SR, 0.2% RP), Pi0.5 (0% SR, 0% RP). Near-zero error recovery rates validate benchmark difficulty and establish baseline for future improvement (M15 human recovery data + LoRA finetuning expected to show positive delta)
  • Checkpoint inventory best practices: Record full path, size, last modified date, and usability status for all model checkpoints. Large checkpoint directories (180GB) indicate many saved steps - document which step to use for evaluation. Date-based naming helps track training timeline but needs explicit step-to-date mapping
  • Error framework scale: 53 Python files (~14.7K LOC) with 6 detectors, 4 injectors, 4 validators, 20 error classifiers across 3 families (manipulation, planning, progress). Plugin architecture with DETECTOR_REGISTRY, INJECTOR_REGISTRY, VALIDATOR_REGISTRY, CLASSIFIER_REGISTRY enables extensibility. Per-detector cooldown (20 steps) prevents detection spam

Practical Learnings#

  • /init command triggers comprehensive codebase analysis + CLAUDE.md generation/improvement workflow, designed for onboarding future Claude Code sessions. Effective for documentation audits even when CLAUDE.md already exists.

Learning Points#

Key Learnings - QueST Evaluation Framework: Using Wasserstein Weisfeiler-Lehman Graphs to verify synthetic base truth in cell type annotations, then using Pearson correlation coefficients to validate model similarity rankings#

  • Multi-task machine learning processes built through incremental prototype design (starting with one task) maintain a dispersed single-task configuration (annotated task lists, hardcoded paths). To scale to production environments, systematic reviews of data preparation scripts, dataset mapping JSON files, checkpoint directories, and evaluation configurations are required
  • Document synchronization pattern: Use parallel exploration proxies (one for each main directory), verify all numbers from actual files (summary.json, file system statistics), then systematically update document content section by section. Don’t trust claims in documents—always check the reality of code and data
  • Debugging checklist for training vs evaluation mismatch: When a strategy performs well in training but fails in evaluation, systematically check: (1) observation dictionary keys, (2) environment initialization parameters, (3) action space boundaries, (4) data normalization, (5) camera rendering settings. BC-RNN had 0% recall on normal tasks and 0.28% recall on error recovery tasks, indicating environmental mismatch rather than strategy failure
  • Niche query evaluation across samples requires: (1) obtaining batch-invariant embeddings through adversarial training; (2) using graph kernels to build base truth data without labels; (3) using dual metrics to evaluate Top-1 accuracy and overall ranking correlation
  • For spatial transcriptomics, Wasserstein Weisfeiler-Lehman graph kernels can convert discrete cell type labels into continuous niche similarity scores by calculating graph-level Wasserstein distances between subgraph representations
  • Large-scale HPC training orchestration requires clear dependency mappings (Stage 0→Stage 1+2 parallel→Stage 3→Stage 4→Stage 5), GPU allocation planning for each stage, and persistent session management (tmux) to ensure continuous operation during multi-day training activities and to handle SSH disconnections
  • Multi-source data collection architecture should be clearly stated in documentation: Generation of error scenarios benefits from diverse sources (for reproducibility demonstrations, for realistic policy deployment, for real-world natural fault capture). Flowcharts must show all sources as first-class citizens, not hidden in script options
  • M14 evaluation results showed baseline performance: 6,474 episodes involving 4 strategies against 649 error scenarios. Random method had recall of 0%, accuracy of 1.9%; BC-RNN had recall of 0.28%, accuracy of 0.3%; Pi0 had recall of 0%, accuracy of 0.2%; Pi0.5 had recall of 0%, accuracy of 0%. Near-zero error recovery rate validated the difficulty of the benchmark and laid the foundation for future improvements (future M15 human restoration data and LoRA fine-tuning are expected to show positive effects)
  • Best practices for checkpoint inventory: Record complete paths, sizes, last modification dates, and availability status of all model checkpoints. A large checkpoint directory (180GB) indicates many saved steps—steps used for evaluation should be noted in documentation. Date-based naming helps track the training timeline, but a clear mapping from steps to dates is required
  • Extensibility of error frameworks: 53 Python files (about 14.7K lines of code), including 6 detectors, 4 injectors, 4 validators, and 20 error classifiers, covering 3 domains (operation, planning, progress). Through a plugin architecture, scalability was achieved using DETECTOR_REGISTRY, INJECTOR_REGISTRY, VALIDATOR_REGISTRY, CLASSIFIER_REGISTRY. A cooling time of 20 steps for each detector prevents the detection of junk information

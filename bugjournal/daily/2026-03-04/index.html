问题与解决方案#

关键问题#

1. QueST 跨样本 niche 查询任务不存在现成 ground truth（niche 相似性无标注数据集）#

解决方案: 论文通过 WWL Graph Kernel（以 cell type label 为节点初始特征）计算子图间 Wasserstein 距离并转化为相似度，构造伪标签 ground truth；用 Pearson 相关系数量化模型预测排序与 WWL 排序的一致性

关键洞察: 无监督评估场景下，利用已有结构化标注（cell type）+ 图核方法构造伪标签是可行范式，将「niche 查询准确性」转化为「模型相似度排序与图结构相似度排序的一致性」，为 MIHD 等类似评估场景提供方法论模板

2. M14 评估中 BC-RNN SR≈0%，用户认为与训练时 rollout 表现不符#

解决方案: 明确区分 Error SR（错误场景下恢复成功率）与 Normal SR（正常任务完成率）：M14 测的是前者，对无恢复能力的现有策略接近 0% 是预期结果；BC-RNN Normal SR 也为 0% 是独立的 object 观测键 bug

关键洞察: 评估语境（error recovery vs normal task）的严格区分是理解基线数据的关键；现有策略 error recovery SR≈0% 恰好是本项目要论证的核心观点，不应视为异常

3. zhaoganlong 数据准备脚本中所有 9 个任务均被注释（仅保留 coffee_d1 开发测试条目）、含 pdb 调试断点、origin_datasets/ 中仅有 3 个任务的 HDF5 文件#

解决方案: 系统性取消注释 4 个脚本中全部 9 任务配置、删除所有 pdb.set_trace()、修正测试路径后缀；通过 ls 确认 tangzijia/mimicgen_datasets/core 存有全部 9 个任务文件，将缺失 6 个文件的 cp 命令写入编排脚本 Stage 0a 作为前置步骤

关键洞察: 实验代码批量化部署前需系统检查四类遗留问题：单任务硬编码配置、pdb 调试断点、测试路径后缀、数据文件完整性；这些问题在单任务开发阶段不可见，批量化时会造成静默失败或进程挂起

一般问题#

4. AI 响应超出 32K output token 上限，导致 JSON 在 conversation_summaries 中段截断无法解析#

解决方案: 将截断文本单独发送给 LLM 并使用专用 JSON 修复 prompt，从残缺文本中成功提取有效 JSON；月度总结采用「分段生成+合并」两阶段模式规避单次超限

关键洞察: 针对超长 AI 日报需预设 token 超限应对策略；JSON 修复 prompt 是有效应急手段，根本解决方案是按天分批或限制 conversation_summaries 数量

5. arXiv PDF 二进制 FlateDecode 无法解析，目标 v3 版本无 HTML（404 Not Found）#

解决方案: 通过 WebSearch 定位 arXiv ID，优先请求 /html/ 路径；v3 无 HTML 时可通过 GitHub 仓库 README 或第三方综述获取更新摘要

关键洞察: arXiv 论文调研应优先请求 /html/ 路径且需验证目标版本可用性；不同版本 HTML 支持不一致，存在版本信息盲区，需主动维护版本追踪意识

6. zhaoganlong 框架 LLaVA MPM/MCM checkpoint 路径不存在#

解决方案: 通过 find 命令确认路径不存在，在训练指南中明确标注为「已清理/不可用」

关键洞察: 服务器空间管理可能导致中间训练产物被清理，文档应如实记录当前可用状态而非假设存在；启动训练前需完整清点 checkpoint 依赖

今日收获#

核心收获#

  • QueST 评估核心范式：cross-sample niche query 无 ground truth 时，可用 WWL 图核（以 cell type annotation 为节点特征）计算子图 Wasserstein 距离作伪标签，再用 Pearson 相关系数量化模型预测与 WWL 排序的一致性——将「无监督 niche 相似性评估」转化为「有监督排序一致性检验」，可直接移植到 MIHD cross-sample query 实验设计
  • Error SR（错误场景下恢复成功率）与 Normal SR（正常任务完成率）是完全不同的指标：现有策略 error recovery SR≈0% 是预期结果，恰好支持了 ErrorRecoveryBenchmark 的核心论点——现有策略缺乏错误恢复能力；不应将其视为数据异常
  • HPC 实验代码批量化部署前需系统检查四类遗留问题：单任务硬编码配置、pdb 调试断点、测试路径后缀、数据文件完整性；这些问题在单任务开发阶段不可见，批量化时会造成静默失败或进程挂起
  • Error Recovery Benchmark 的场景来源应为三路并行：Demo 回放注入 / Policy Rollout 注入 / 自然失败捕获，三者共同构成更丰富的错误场景分布；仅记录单一入口是结构性文档缺失
  • Pi0.5 per-task LoRA 对 MimicGen 任务的正常 SR 远优于全局模型（Stack_D0 ~95% vs ~24%，整体 ~58.9%），说明 per-task 微调对结构化操作任务至关重要
  • 复杂多阶段训练流程（如 zhaoganlong 的 6 阶段依赖链）最适合由人类先设计依赖关系图和资源分配，再交给 AI 脚本化实施；月度总结及超长日报适合「分段生成+合并」两阶段模式规避单次 token 超限

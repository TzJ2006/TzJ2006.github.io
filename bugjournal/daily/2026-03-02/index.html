Issues and Solutions#

Key Issues#

1. Desktop Video: 100% failure rate of video/image restoration after application restart, code 257 “operation not allowed” error#

Solution: Fixed in four steps: 1) Added com.apple.security.files.bookmarks.app-scope permission to permission files (originally empty dictionary). 2) Added alternative solution in restoreFromBookmark: If withSecurityScope fails, try using options:[] to restore old non-security scope bookmarks, then save as security scope bookmarks again. 3) Track active URLs in activeSecurityScopedURLs dictionary and release them in clear(). 4) Automatically resolve bookmark issues before file I/O.

Key Insight: macOS sandbox bookmark system requires compile-time permissions declarations and runtime URL.startAccessingSecurityScopedResource() lifecycle management. When permission files are empty, bookmark creation implicitly reverts to non-security scope, causing restore with withSecurityScope to fail, resulting in code 257 error. Due to concurrent FileHandle race conditions, errorLog() never recorded errors, thus hiding the root cause for months.

2. MIHD: STAIG two-stage fusion ignored gene encoder embedding, making scGPT+UNI2+STAIG equivalent to PCA+UNI2+STAIG#

Solution: Checked models/Fusion.py and pipeline/runner.py: staig_fusion used staig_gene_feat (from cached original HVG matrix) instead of gene_emb parameter. This was an architectural issue, not a mistake. Recorded this key finding in weekly reports as it affected interpretation of all STAIG results.

Key Insight: Gene encoder (PCA/scGPT) output was only used by non-STAIG fusion methods. For graph-based fusion like STAIG, pre-trained spatial embeddings did not enter GCN node features. This explains why scGPT performance was poor: it was designed for cross-organizational generalization rather than maximizing variance within the dataset like PCA did.

3. VLA Evaluation: Although Pi0.5 had 100% GPU utilization during training, evaluation GPU utilization was only low (about 10%), leading to slower evaluation#

Solution: Identified root cause: Action block architecture reduced Pi0.5 output to 50-step action sequences, reducing inference calls to about 8 per 400 steps of experimentation (8 × 2.5 sec = 20 sec GPU usage/total 200 sec = 10% utilization). Although a solution was developed, due to network research and requirements for batch evaluation, the plan was temporarily suspended.

Key Insight: Bottleneck in VLA evaluation wasn’t GPU computing power, but rather pipeline serialization issues: action block reduced GPU call counts, while each inference cycle included CPU preprocessing, WebSocket I/O, and simulation steps, which accounted for most of the total time. Using timer-based request collection could improve throughput.

4. GPU nodes an49/an53 couldn’t access SSH due to PAM slurm_adopt blocking#

Solution: Used Slurm’s srun command to execute commands directly, bypassing SSH restrictions, i.e., using –jobid flag to execute commands directly within the assigned task context: srun --jobid=45628 bash -c 'command'

Key Insight: Slurm’s srun command allowed execution of commands within the assigned task context, even if SSH connection was blocked by PAM, which was useful for clusters with strict access controls.

General Issues#

5. When using srun on heavily loaded GPU nodes, background tasks timed out for more than 60 seconds, and command response times were long#

Solution: Added –overlap flag and reduced timeout expectations. When nvitop was unavailable or slow, switched to using nvidia-smi instead. Higher-loaded nodes would experience higher command delays. Key Insight: Management commands executed via srun might encounter significant delays when GPU nodes ran at 100% utilization; monitoring scripts should handle timeouts gracefully and use lightweight alternative commands.

6. MIHD: mclust clustering program hung indefinitely during visualization of scGPT (512 dimensions) and UNI2 (1536 dimensions) embeddings#

Solution: In visualize_from_cache.py, replaced default mclust with kmeans using –cluster_method flag. High-dimensional data made mclust’s EM algorithm difficult to compute.

Key Insight: Gaussian mixture model clustering performed poorly beyond 100 dimensions. For base model embeddings (512 dimensions - 1536 dimensions), kmeans or spectral methods needed to be used for actual operation.

7. Desktop video log files contained over 1550 garbage entries per session (updateClockLabels × 879, startScreensaverTimer × 671)#

Solution: Removed dlog from updateClockLabels (called once per screen saver period). Moved dlog from startScreensaverTimer to after all early exit protections (only recorded during actual scheduling timer scenarios). Removed dlog from idle timer callbacks.

Key Insight: Frequent dlog calls in timer callbacks generated large log files, masking real errors. Logging should be based on state transitions rather than periodic timing.

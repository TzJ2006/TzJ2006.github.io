Problems and Solutions#

Key Problems#

1. STAIG’s two-stage architecture completely ignores gene encoder output, using the original HVG expression matrix as input for GCN, resulting in scGPT+UNI2+STAIG≡PCA+UNI2+STAIG#

Solution: Through three-layer code tracing (models/Fusion.py:1246, pipeline/runner.py:445-446, STAIGTrainer.py), it was confirmed that the staig_gene_feat input path has nothing to do with gene_emb.

Key Insight: STAIG’s gene encoder and GNN training are decoupled designs. Improvements should start from the GNN structure rather than replacing the gene encoder.

2. Pi0.5 evaluation client reports ‘keepalive ping timeout’, causing each episode to fail immediately, with SR always being 0%#

Solution: Added ping_timeout=None in the connect() call of WebSocket_client_policy.py, disabling the default 20s timeout.

Key Insight: JAX first inferences require JIT compilation (30-60s), exceeding the WebSocket library’s default 20s keepalive timeout. All JAX applications that require long first inferences must explicitly set ping_timeout=None.

3. ’no close frame received’ caused server crashes when 5 JAX servers started simultaneously; 2 clients with one server were actually 24% slower than serial (1771s vs 1428s)#

Solution: Adopted staggered startup (v4): each server was started one by one, waiting for JIT warmup before starting the next. Abandoned single-server multi-client setup and replaced it with one server per GPU + one client per GPU.

Key Insight: Multiple JAX processes compete for CPU and memory bandwidth during JIT compilation. VLA server inferences are strictly serializable. Multi-client concurrency cannot improve throughput; the only correct parallel approach is independent deployment of multiple GPUs.

4. GPU utilization in Pi0.5 evaluation pipeline was only about 10%, with a lot of GPU time idle#

Solution: Confirmed that the root cause was action chunking: approximately 400 steps per trial, inference every 50 steps = 8 GPU calls × 2.5s = 20s GPU usage / 200s total time; the optimization direction was batch inference (BatchedVLAServer) rather than increasing concurrent clients.

Key Insight: Action chunking reduces inference calls. Real optimization should focus on aggregating multiple inference requests into batches, rather than trying to concurrency existing serial inference paths.

5. Killing the launcher PID caused all 6 training processes (including those started with nohup) to be terminated by Slurm cgroup together#

Solution: Used independent background srun –overlap commands to start each training task. Srun processes do not depend on the launcher and can survive after SSH disconnects.

Key Insight: Subprocesses launched within srun by nohup are still terminated by Slurm cgroup even after srun exits. Each long-running task must be treated as an independent srun process.

General Problems#

6. SSH to compute nodes was rejected by pam_slurm_adopt; srun to an49 timed out and GPU status could not be directly obtained#

Solution: Nodes with active jobs (an53) can be logged in directly via SSH. Nodes without active jobs use srun –jobid=XXXXX –overlap. Training logs can also be read directly through the shared file system, bypassing node access restrictions.

Key Insight: pam_slurm_adopt requires an active job on the target node for SSH access. Cluster access policy should be “priority SSH → fallback srun –overlap”. The shared file system can serve as a backup source of information.

7. AI directly created new visualization scripts, ignoring existing visualize_from_cache.py; mclust got stuck on high-dimensional embeddings (512d/1536d)#

Solution: After the user corrected it, used the existing script. Added –cluster_method kmeans parameter to solve mclust’s deadlock problem.

Key Insight: Before starting a new task, it is better to search for existing project tools. mclust is not suitable for high-dimensional data and should default to using kmeans.

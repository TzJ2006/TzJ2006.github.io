Issues and Solutions#

Key Issues#

1. MIHD patch query returned incorrect physical location: “Patch does not match the current patch”#

Solution: The root cause was inconsistent barcode sorting between the cache (sc.read_10x_h5 order) and visualization (sc.read_visium order). This was solved by explicitly mapping barcodes to cache indexes: {str(b): i for i, b in enumerate(cache_barcodes)}. Also fixed data path and coordinate convention issues. Key Insight: Index-based searches fail when the order of data sources is different. Always use a key-based alignment method (in this case, barcodes) when connecting cached data and reloading at runtime.

2. K8s container PID namespace isolation prevented nvidia-smi from displaying process information#

Solution: Used /proc//fd/ to build a custom GPU monitoring tool, scanning /dev/nvidia* device files and CUDA_VISIBLE_DEVICES environment variables. Filtered out monitoring tools that used all GPUs (nvitop, gpustat, Claude). Created a rich interface for the nvitop-style user interface. Key Insight: Containerized GPU monitoring requires checking the proc file system. /proc/fd symbolic links can display opened devices; CUDA_VISIBLE_DEVICES is the actual information. Monitoring tools that use all GPUs are false positives—can be determined by checking CUDA_VISIBLE_DEVICES.

3. Despite identical hardware and configuration, training times for the same commands differed by 33% between the two projects (openpi 15 hours vs pi05 20 hours)#

Solution: Parallel exploration agents identified differences in JAX versions (0.5.0 vs 0.5.3) as the main cause. Systematically aligned 6 dependency versions: JAX 0.5.0→0.5.3, torch>=2.5.1→2.7.1, orbax-checkpoint 0.11.1→0.11.13, numpy, beartype, transformers. Added override-dependencies to resolve lerobot’s torch<2.7 constraint conflict. Key Insight: A slight upgrade of the JAX version (0.5.0→0.5.3) can bring about a 20-30% performance improvement, as the XLA compiler has optimization capabilities. Multi-project dependency changes are a hidden performance issue that requires regular alignment audits. uv’s override-dependencies can solve conflicts without splitting upstream code.

4. MIHD file structure became chaotic due to inconsistent naming (benchmark_results, hd_results, rm_ideal were distributed in different directories)#

Solution: Implemented a systematic 8-step migration plan: created new hierarchies (DLPFC, HD, rm_ideal/{cross_section,single_section}), migrated files, updated 14 code files and documents, and cleared old directories. Used bash mv and Edit tools for batch path updates. Key Insight: Large-scale refactoring requires planning (create directories first), then file migration, followed by code updates, and finally document modifications. For historical documents, sed can be used for batch archiving.

5. Pi05 training failed at startup: ‘RuntimeError: torchvision::nms does not exist’#

Solution: Identified a mismatch between torch 2.7.1 + torchvision 0.21.0 (0.21.0 was compiled for torch 2.6.x). Added torchvision==0.22.1 explicitly in pyproject.toml, then used uv sync for the upgrade. After verification, torch was set to 2.7.1+cu126, and torchvision was set to 0.22.1+cu126. Key Insight: PyTorch and torchvision have strict version pairing requirements (2.7.x requires 0.22.x). Indirect dependencies resolved by uv/pip may choose incompatible versions—always specify critical binary coupling dependencies explicitly.

Automating noise filtering (monitoring processes) can improve the signal-to-noise ratio.

Strict Version Pairing in PyTorch/torchvision#

For example, torch 2.7.x requires torchvision 0.22.x, while 2.6.x requires 0.21.x. If the versions do not match, C++ operator registration will fail. When using torchvision, it is necessary to explicitly specify the dependencies of both components, as indirect solutions may fail.

Barcode Sorting Issues in Spatial Transcriptomics#

The order of barcodes in the cache created by sc.read_10x_h5 may differ from the one used in sc.read_visium during runtime. Index-based searches may incorrectly map to the wrong positions. Always use a clear mapping from barcodes to indices: {str(b): i for i, b in enumerate(cache_barcodes)}.

User Barriers When Plans Are Repeatedly Rejected#

When plans are repeatedly rejected, users may face insurmountable obstacles such as unavailable environments, changes in priorities, or technical limitations. After 2-3 rejections, the focus should shift from “Let me fix the plan” to “What is preventing us from moving forward?”

RM-IDEAL Center Point Calculation#

For a given query domain, the center point is calculated by taking the embedding vectors of all points in that domain and calculating the average vector. Then, the matching between the center point and all reference points is compared using cosine similarity. This reduces the comparison from N pairs of M to 1 pair of M, improving efficiency and reliability.

uv’s Overlapping Dependency Tool#

uv’s overlapping dependency tool is a precise tool for resolving transitive dependency conflicts without splitting or patching upstream packages. This tool is particularly useful when fixing git revisions related to incompatible constraints.

Practical Experience Summary#

Before assuming a repair solution, it is necessary to track all source configuration parameters (code defaults, CLI parameters, configuration files, environment variables). This is because the actual values during runtime may differ from the results of code checks.

Issues and Solutions#

Key Issues#

1. MIHD patch query returned incorrect physical locations: “Patch mismatch with current patch”#

Solution: The root cause was inconsistent barcode sorting between the cache (sc.read_10x_h5 order) and visualization (sc.read_visium order). This issue was resolved by using an explicit barcode-to-cache index mapping: {str(b): i for i, b in enumerate(cache_barcodes)}. Also fixed data path and coordinate consistency issues.

Key Insight: Index-based searches fail when the data source order is different. Always use explicit keys for alignment (in this case, barcodes) when bridging cache data with runtime reloading.

2. K8s containers’ PID namespace isolation prevented nvidia-smi from displaying process information#

Solution: Built a custom GPU monitoring tool that scanned /dev/nvidia* device files using /proc//fd/ and the CUDA_VISIBLE_DEVICES environment variable. Filtered out monitoring tools that ran all GPUs (nvitop, gpustat, claude). Implemented a rich-toolkit nvitop-style TUI interface.

Key Insight: Containerized GPU monitoring requires access to the proc file system. /proc/fd symbolic links can display open devices; CUDA_VISIBLE_DEVICES is actual. Monitoring tools that run all GPUs are false positives - filter these tools by checking CUDA_VISIBLE_DEVICES.

3. Despite identical hardware and configuration, training times for the same commands in both projects differed by 33% (openpi 15 hours vs pi05 20 hours)#

Solution: Parallel exploration agents determined that JAX version differences (0.5.0 vs 0.5.3) were the main cause. Systematically aligned 6 dependency versions: JAX 0.5.0→0.5.3, torch>=2.5.1→2.7.1, orbax-checkpoint 0.11.1→0.11.13, numpy, beartype, transformers. Added override dependencies to resolve Lerobot torch<2.7 constraints.

Key Insight: A slight JAX version upgrade (0.5.0→0.5.3) can bring about a 20-30% performance improvement because the XLA compiler offers good optimization benefits; multi-project dependency drift is a hidden performance issue that requires regular alignment audits. uv override dependencies provide an non-invasive way to resolve conflicts.

4. MIHD file structure became chaotic due to inconsistent naming (benchmark_results, hd_results, rm_ideal were distributed in different directories)#

Solution: Executed a systematic 8-step migration plan: created new hierarchies (DLPFC, HD, rm_ideal/{cross_section,single_section}), migrated files, updated 14 code files and documents, cleaned up old directories. Used bash mv and Edit tools for batch path updates. Key Insight: Large-scale refactoring requires upfront planning (create directories first), then file migration, followed by code and document updates. For historical archive documents, sed batch processing can be used.

5. Pi05 training failed at startup: ‘RuntimeError: operator torchvision::nms does not exist’#

Solution: Detected incompatibility between torch 2.7.1 + torchvision 0.21.0 (0.21.0 was compiled for torch 2.6.x). Explicitly specified torchvision==0.22.1 in pyproject.toml, then ran uv sync to upgrade. After verification: torch 2.7.1+cu126, torchvision 0.22.1+cu126.

Key Insight: There are strict version pairing requirements between PyTorch and torchvision (2.7.x requires 0.22.x). Indirect dependencies resolved by uv/pip may choose incompatible versions - always explicitly specify critical binary coupling dependencies.

Strict Version Pairing in PyTorch/torchvision#

For example, torch 2.7.x requires torchvision 0.22.x, and torch 2.6.x requires 0.21.x. If the versions do not match, C++ operator registration will fail. When using torchvision, it is essential to explicitly specify these versions in the dependencies, as indirect solutions may fail.

Barcode Sorting Issues in Spatial Transcriptomics#

Caches created by sc.read_10x_h5 may have a different barcode order compared to runtime sc.read_visium. Index-based searches will unconsciously map to incorrect locations. Always use a clear mapping from barcodes to indices: {str(b): i for i, b in enumerate(cache_barcodes)}.

User-Unintended Barriers During Plan Rejection#

When the planning mode is repeatedly rejected, there may be unspoken barriers (environmental availability, priority changes, technical limitations). After two or three rejections, the approach should shift from “Let me fix the plan” to asking “What is preventing us from moving forward?”

Center Calculation in RM-IDEAL#

For a given query niche, take the embedding vectors of all points in that niche and calculate the average vector to obtain the centroid. Compare the centroid with all reference points using cosine similarity. This can compress N pairs of M comparisons into 1 pair of M, thereby improving efficiency and reliability.

uv’s Coverage Dependency as an Exact Tool#

uv is a precise tool for resolving transitive dependency conflicts without splitting or patching upstream packages. This tool is particularly useful when you need to fix git versions with incompatible constraints.

Practical Learning#

  • Before assuming there is a solution, it is necessary to trace the configuration parameters from all sources: code defaults, CLI parameters, configuration files, environment variables. The values at runtime may differ from the results of code checks.
  • Refactoring large-scale codebases should be done in stages: 1) Create new structures; 2) Migrate files; 3) Update code references; 4) Update documentation (for archived documents, batch editing is sufficient); 5) Verify the final structure; 6) Clean up old directories. Performing code updates before file migration can lead to data corruption.

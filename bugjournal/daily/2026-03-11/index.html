日报 — 2026-03-11#

今日概览#

  • 做了什么： 为 K8s 容器环境开发了仿 nvitop 风格的 GPU 进程监控工具，修复了 torchvision 版本不兼容导致的训练启动报错，并将 pi05 项目的依赖版本完整对齐至 openpi
  • 怎么做的： 通过读取 /proc//fd/ 和 CUDA_VISIBLE_DEVICES 解决 K8s PID 隔离问题；通过 pyproject.toml 版本锁定和 uv override-dependencies 解决依赖冲突；对比两项目 wandb 日志和 uv.lock 定位速度差异根因
  • 有什么用： GPU 监控工具使容器内 GPU 使用情况可见化；依赖对齐有望将 pi05 训练时长从 20h 缩短至接近 openpi 的 15h，节约约 25% 算力

在算力服务器上开发 GPU 监控工具，修复 PyTorch 版本兼容问题，并将训练环境依赖对齐以消除训练速度差异

今日任务#

架构与策略#

  • ✅ 开发 K8s 容器内 GPU 监控工具 gpumon.py — 在 Kubernetes 容器环境中（PID 命名空间隔离，nvidia-smi 无法显示进程），通过扫描 /proc//fd/ 和 CUDA_VISIBLE_DEVICES 建立 GPU-进程映射，开发仿 nvitop 风格的实时监控工具，支持按 GPU 分组显示进程
  • ✅ pi05 训练环境依赖对齐（对齐至 openpi） — 将 pi05 的 jax、orbax-checkpoint、torch、transformers、numpy、beartype 6 个依赖版本对齐至 openpi；添加 [tool.uv] override-dependencies 锁定 ml-dtypes 和 tensorstore；因 lerobot 旧 rev 限制 torch<2.7，添加 torch==2.7.1 到 override-dependencies 解决冲突；uv lock + uv sync 成功

实现与修复#

  • ✅ 修复 torchvision::nms 版本不兼容报错 — torch 2.7.1 与 torchvision 0.21.0 不兼容，在 pyproject.toml 中添加 torchvision==0.22.1，执行 uv sync 完成修复
  • ❌ v5 误差场景 720p 视频重渲染 — 用户希望将已有的 480p v5 错误场景视频重渲染为 720p，但 AI 在 Plan Mode 循环中多次被拒绝，未能成功退出并执行，任务搁置

问题与解决方案#

关键问题#

1. Kubernetes PID 命名空间隔离导致 nvidia-smi 无法在容器内看到 GPU 进程归属#

解决方案: 扫描 /proc//fd/ 中打开的 /dev/nvidia* 设备文件，结合 CUDA_VISIBLE_DEVICES 环境变量，建立进程到 GPU 的映射；过滤 nvitop、claude、gpustat 等监控工具进程避免误报

关键洞察: CUDA_VISIBLE_DEVICES 比 fd 扫描更精确，fd 扫描可作为补充；打开全部 GPU 设备且无 CUDA_VISIBLE_DEVICES 的进程通常是监控工具而非训练进程

2. pi05 的 lerobot 旧版本限制 torch<2.7，与 torch==2.7.1 约束冲突，导致 uv lock 失败#

解决方案: 将 torch==2.7.1 也加入 [tool.uv] override-dependencies，强制覆盖 lerobot 的版本约束

关键洞察: uv 的 override-dependencies 可以绕过间接依赖的版本限制，适合在不修改上游包的情况下强制指定版本

一般问题#

3. RuntimeError: operator torchvision::nms does not exist —— torch 2.7.1 与 torchvision 0.21.0 C++ 算子不兼容#

解决方案: 在 pyproject.toml dependencies 中显式添加 torchvision==0.22.1，运行 uv sync 更新

关键洞察: torchvision 版本与 PyTorch 版本有严格对应关系（2.7.x 对应 0.22.x），被 lerobot/transformers 间接引入时容易出现版本不匹配

4. AI 在 Plan Mode 循环中反复被用户拒绝，无法退出执行渲染任务#

解决方案: 最终通过 AskUserQuestion 工具直接询问用户意图，确认用户希望直接执行而非继续规划

关键洞察: Plan Mode 的 ExitPlanMode 工具需要 allowedPrompts 参数，多次格式不当或意图不明导致被拒绝；应更早询问用户偏好

AI 局限性#

重要局限#

  • 在 Plan Mode 循环中无法准确判断用户拒绝的原因，反复用不同格式尝试 ExitPlanMode 而非直接询问用户意图，浪费了大量交互轮次

一般局限#

  • GPU 监控工具初版未能预判监控工具进程（如 claude 自身）会打开少量 GPU 设备文件的边界情况，需要用户反馈后才修复
  • 在训练速度分析中，AI 对 wandb 日志中 num_workers=16 的来源（CLI 参数覆盖）需要多步排查才能确认，未能第一时间指出最可能的来源

今日收获#

核心收获#

  • 在 Kubernetes 容器中实现 GPU 进程监控的核心方法：优先读取 CUDA_VISIBLE_DEVICES 环境变量，备选扫描 /proc//fd/ 中的 /dev/nvidia* 设备，结合进程名过滤监控工具。这是解决 K8s PID 命名空间隔离的实用工程方案
  • JAX 版本对 XLA JIT 编译优化影响显著，相邻小版本（0.5.0 vs 0.5.3）可造成约 25% 的训练速度差异；在 JAX 生态中维护精确的版本锁定对性能复现性至关重要
  • uv 的 override-dependencies 机制可以强制覆盖传递依赖的版本约束，是解决"上游包版本限制与项目需求冲突"问题的有效工具，但需要明确这会绕过兼容性检查

实践收获#

  • PyTorch 与 torchvision 的版本对应关系严格（2.7.x 对应 0.22.x），通过间接依赖引入时容易出现版本漂移，建议在 pyproject.toml 中显式锁定

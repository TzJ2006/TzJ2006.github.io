Problems & Solutions#

Critical Issues#

1. MIHD patch query returned wrong physical spots: ‘queried patches don’t match current patch’#

Solution: Root cause: barcode ordering mismatch between cache (sc.read_10x_h5 order) and visualization (sc.read_visium order). Fixed by explicit barcode-to-cache-index mapping: {str(b): i for i, b in enumerate(cache_barcodes)}. Also fixed data path and coordinate conventions.

Key Insight: Index-based lookups fail when data sources have different orderings. Always use explicit key-based alignment (barcodes in this case) when bridging cached data and runtime reloads.

2. K8s container PID namespace isolation prevents nvidia-smi from showing process info#

Solution: Built custom GPU monitor using /proc//fd/ to scan /dev/nvidia* device files and CUDA_VISIBLE_DEVICES env var. Filter out monitoring tools (nvitop, gpustat, claude) that open all GPUs. Implemented nvitop-style TUI with rich library.

Key Insight: Containerized GPU monitoring requires proc filesystem inspection. /proc/fd symlinks reveal opened devices; CUDA_VISIBLE_DEVICES is the ground truth. Monitoring tools opening all GPUs are false positives - filter by checking CUDA_VISIBLE_DEVICES absence.

3. Training time for identical commands differed by 33% across two projects (openpi 15h vs pi05 20h), despite same hardware and configurations#

Solution: Identified JAX version difference (0.5.0 vs 0.5.3) as primary cause through parallel exploration agents. Systematically aligned 6 dependency versions: JAX 0.5.0→0.5.3, torch>=2.5.1→2.7.1, orbax-checkpoint 0.11.1→0.11.13, numpy, beartype, transformers. Added override-dependencies to resolve lerobot’s torch<2.7 constraint conflict.

Key Insight: Minor JAX version upgrades (0.5.0→0.5.3) can yield 20-30% performance gains due to XLA compiler optimizations; multi-project dependency drift is a stealth performance bug requiring periodic alignment audits. uv’s override-dependencies provides surgical conflict resolution without forking upstream code.

4. MIHD file structure was cluttered with inconsistent naming (benchmark_results, hd_results, rm_ideal split across dirs)#

Solution: Executed systematic 8-step migration plan: create new hierarchy (DLPFC, HD, rm_ideal/{cross_section,single_section}), migrate files, update 14 code files + docs, cleanup old dirs. Used bash mv + Edit tool for bulk path updates.

Key Insight: Large-scale refactoring requires upfront planning (create dirs first), then file migration, then code updates, then doc updates. Batch sed for archived docs acceptable since they’re historical.

5. Pi05 training failed at startup: ‘RuntimeError: operator torchvision::nms does not exist’#

Solution: Diagnosed torch 2.7.1 + torchvision 0.21.0 version mismatch (0.21.0 compiled for torch 2.6.x). Added explicit torchvision==0.22.1 to pyproject.toml dependencies, ran uv sync to upgrade. Verified: torch 2.7.1+cu126, torchvision 0.22.1+cu126.

Key Insight: PyTorch and torchvision have strict version pairing (2.7.x needs 0.22.x). Indirect dependencies resolved by uv/pip may select incompatible versions - always pin critical binary-coupled dependencies explicitly.

Key Insight: Configuration parameters can originate from multiple sources (code defaults, CLI args, config files); full provenance tracing is necessary before making code changes

Key Insight: Terminal UI design requires multiple iterations. Reference existing tools (nvitop) for user expectations. Auto-filtering noise (monitoring processes) improves signal-to-noise ratio.

Learnings#

Key Learnings#

  • K8s PID namespace isolation prevents nvidia-smi process detection. Solution: /proc//fd/ inspection for /dev/nvidia* device files + CUDA_VISIBLE_DEVICES env var. Monitoring tools opening all GPUs without CUDA_VISIBLE_DEVICES are false positives.
  • JAX/XLA minor version upgrades (0.5.0→0.5.3) can deliver 20-30% training speedups through compiler optimizations; even patch-level framework updates merit performance audits in compute-intensive ML projects. Multi-project dependency drift is a stealth performance bug requiring periodic alignment.
  • PyTorch/torchvision strict version pairing: torch 2.7.x requires torchvision 0.22.x, 2.6.x needs 0.21.x, etc. C++ operator registration breaks if versions mismatch. Always explicitly pin both in dependencies when using torchvision, as indirect resolution can fail.
  • Barcode ordering pitfall in spatial transcriptomics: Cache created with sc.read_10x_h5() may have different barcode order than runtime sc.read_visium(). Index-based lookups silently map to wrong spots. Always use explicit barcode-to-index mapping: {str(b): i for i, b in enumerate(cache_barcodes)}.
  • When plan mode repeatedly rejected, user may have unspoken blocker (environment unavailable, changed priority, technical constraint). After 2-3 rejections, switch from ’let me fix the plan’ to ‘what’s preventing us from proceeding?’.
  • RM-IDEAL centroid computation: For a query niche, take embeddings of all spots in that niche, compute mean vector → centroid. Compare centroid to all ref section spots via cosine similarity. This compresses N-to-M comparison into 1-to-M, improving efficiency and robustness.
  • uv’s override-dependencies is a precise tool for resolving transitive dependency conflicts without forking or patching upstream packages; particularly useful when pinning git revisions with incompatible constraints.

Practical Learnings#

  • Configuration parameter provenance must be traced across all sources (code defaults, CLI args, config files, environment variables) before assuming a fix; the actual runtime value may differ from code inspection.
  • Large-scale codebase refactoring requires phase separation: 1) Create new structure, 2) Migrate files, 3) Update code references, 4) Update docs (bulk sed acceptable for archived docs), 5) Verify final structure, 6) Cleanup old dirs. Doing code updates before file migration causes breakage.

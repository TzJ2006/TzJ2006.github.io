Problems and Solutions#

Key Issues#

1. SpaceMouse collected demonstrations cannot be enhanced by MimicGen, while demonstrations generated by MimicGen can.#

Solution: Identified that MimicGen requires complete datagen_info metadata (eef_pose, object_poses, target_pose, gripper_action, subtask_term_signals) and semantic assumptions about a clean initial state. SpaceMouse demonstrations usually start from an error state, lacking the required metadata. Two different demonstration processes were identified: recovery demonstrations (simplified format) and successful demonstrations (MimicGen-compatible format).

Key Insight: Format compatibility is not just a matter of file structure—it also includes semantic assumptions (clean states versus error states) and metadata integrity. The enhancement process for recovery demonstrations differs from the standard MimicGen source demonstrations.

2. Confusion regarding the term “training source demonstration” in different process contexts#

Solution: Made it clear that in the context of recovery, “training source demonstration” refers to “Training Recovery Demonstration” (RecoveryDemo + AugmentedRecovery), which is different from MimicGen’s success_demos.hdf5. Created a clear document mapping: for manually collected RecoveryDemos, use outputs/recovery/demos/; for MimicGen-style enhanced demonstrations, use outputs/recovery/augmented/.

Key Insight: Technical terms may have specific meanings within a project. In multi-process projects, it’s necessary to have a clear context to avoid mapping errors. The prepared_source/*.hdf5 files are dependencies of the enhancement functions, not the training source demonstration itself.

General Issues#

3. Documentation tone not suitable for the target audience (team guidance)#

Solution: Rewrote the training_source_demo_dual_path.md file twice. The first version used a “rectification” tone because it was generated during debugging of conceptual misunderstandings. The second version redefined it as “explaining the process to team members”, providing clear processes, file paths, and current state snapshots.

Key Insight: Documentation produced during problem-solving often inherits a debugging narrative style. It’s necessary to clearly inform the audience that technical references need different frameworks and levels of detail than team introductions.

Learning Outcomes#

Key Takeaways#

  • MimicGen’s source demonstrations require complete datagen_info metadata (eef_pose, object_poses, target_pose, gripper_action, subtask_term_signals) and semantic assumptions (clean initial state, complete task trajectory, valid subtask transitions). Recovery demonstrations from error states require separate enhancement processes.
  • Format compatibility includes three levels: file structure (NPZ vs HDF5), metadata integrity (datagen_info fields), and semantic assumptions (starting state, task coverage). A file may pass structural checks but may not meet semantic requirements.
  • In multi-pipeline projects, technical terms need clear definitions. In the recovery pipeline, “source demonstration” has a different meaning from standard MimicGen’s success_demos.hdf5. Documents should establish clear namespaces.
  • The LiPM paper demonstrated physical information-based models for battery tasks: MMAE for cross-modal coupling, CIR for coulomb integration, dual-scale encoder for irregular sampling. Key insights are that when physical constraints are correctly encoded, domain-specific base models perform better than general time series models.

Practical Learning#

  • Documentation generated during problem-solving inherited a debugging narrative style. For team onboarding documents, clear adjustment was needed: explaining the current design rather than finding its path. Audience awareness must be externally injected, not spontaneous.

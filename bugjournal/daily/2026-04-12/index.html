Problems and Solutions#

Key Problems#

1. Experiments collected by SpaceMouse could not be enhanced by MimicGen, while experiments generated by MimicGen could#

Solution: It was found that MimicGen requires complete datagen_info metadata (eef_pose, object_poses, target_pose, gripper_action, subtask_term_signals) and clear initial state semantics. SpaceMouse experiments usually start from an error state, lacking the required metadata. Two different demonstration processes were identified: recovery demonstration (simplified format) and successful demonstration (format compatible with MimicGen).

Key Insight: Format compatibility is not just about file structure—it also includes semantic assumptions (clear and error states) and metadata integrity. The enhancement process for recovery demonstrations is different from the standard MimicGen source demonstrations.

2. Confusion of the term “training source demonstration” in different process contexts#

Solution: It was clarified that in the context of recovery, “training source demonstration” refers to “training recovery demonstration” (RecoveryDemo + AugmentedRecovery), which is different from MimicGen’s success_demos.hdf5. A clear document mapping was created: for manually collected RecoveryDemos, use outputs/recovery/demos/; for MimicGen-style enhanced demonstrations, use outputs/recovery/augmented/.

Key Insight: Technical terms have specific meanings within a particular project. In multi-process projects, it’s necessary to have clear context to avoid mapping errors. The prepared_source/*.hdf5 files are dependencies for enhancement, not the training source demonstration itself.

General Problems#

3. Document tone not suitable for the target audience (team-guided)#

Solution: The training_source_demo_dual_path.md document was rewritten twice. The first version had a “fixative” tone because it was generated during debugging of conceptual misunderstandings. The second version was reworded as “explaining the process to team members”, including clear process flow, file paths, and current state snapshots.

Key Insight: Documents generated during problem-solving often have a debugging narrative style. It’s necessary to clearly inform the audience that technical references and team guidance require different levels of detail.

Lessons Learned#

Main Lessons#

  • MimicGen’s source demonstrations require complete datagen_info metadata (eef_pose, object_poses, target_pose, gripper_action, subtask_term_signals) and semantic assumptions (clean initial state, complete task trajectory, valid subtask transitions). Recovery demonstrations from an error state require a separate enhancement process.
  • Format compatibility includes three levels: file structure (NPZ vs HDF5), metadata integrity (datagen_info fields), and semantic assumptions (starting state, task coverage). A file may pass structural checks but may not meet semantic requirements.
  • In multi-pipeline projects, technical terms require explicit project-specific definitions. “Source demonstration” may mean different things in the recovery pipeline (RecoveryDemo) versus the standard MimicGen pipeline’s success_demos.hdf5. Documents should establish clear namespaces.
  • The LiPM paper demonstrated physical information-based training for battery tasks: MMAE for cross-modal coupling, CIR for Coulomb integration, dual-scale encoder for irregular sampling. Key insight is that when physical constraints are correctly encoded, domain-specific baseline models outperform general time series models.

Practical Experience#

  • Documents generated during problem-solving inherited a debugging narrative style. For team onboarding documents, explicit adjustment was needed: describing the current design rather than the path to discovery. Audience awareness must be externally injected, rather than spontaneously generated.

Problems & Solutions#

Critical Issues#

1. SpaceMouse-collected demos cannot be augmented by MimicGen while MimicGen-generated demos can#

Solution: Revealed MimicGen requires complete datagen_info metadata (eef_pose, object_poses, target_pose, gripper_action, subtask_term_signals) and clean initial state semantics. SpaceMouse demos typically start from error states and lack required metadata. Identified two separate demo pipelines: recovery demo (simplified format) vs success demo (MimicGen-compatible format).

Key Insight: Format compatibility is not just about file structure - it includes semantic assumptions (clean state vs error state) and metadata completeness. Recovery demos follow a different augmentation pipeline than standard MimicGen source demos.

2. Confusion about ’training source demo’ terminology across different pipeline contexts#

Solution: Clarified that in recovery context, ’training source demo’ means ’training recovery demo’ (RecoveryDemo + AugmentedRecovery), distinct from MimicGen’s success_demos.hdf5. Created clear documentation mapping: outputs/recovery/demos/ for human-collected RecoveryDemo, outputs/recovery/augmented/ for MimicGen-style augmented demos.

Key Insight: Technical terminology can have project-specific meanings. In multi-pipeline projects, explicit context is required to avoid mapping errors. The prepared_source/*.hdf5 files are dependencies for augmentation, not the training source demos themselves.

General Issues#

3. Documentation tone inappropriate for target audience (team onboarding)#

Solution: Rewrote training_source_demo_dual_path.md twice. First version had ‘correction-style’ tone because it emerged from debugging a conceptual misunderstanding. Second version reframed as ’explaining pipeline to teammates’ with clear flow, file paths, and current status snapshot.

Key Insight: Documentation generated during problem-solving tends to inherit the debugging narrative. Need explicit audience-awareness: technical reference vs team introduction require different framing and level of detail.

Learnings#

Key Learnings#

  • MimicGen source demos require complete datagen_info metadata (eef_pose, object_poses, target_pose, gripper_action, subtask_term_signals) and semantic assumptions (clean initial state, complete task trajectory, valid subtask transitions). Recovery demos from error states need separate augmentation pipeline.
  • Format compatibility encompasses three layers: file structure (NPZ vs HDF5), metadata completeness (datagen_info fields), and semantic assumptions (starting state, task coverage). A file can pass structural checks but fail semantic requirements.
  • In multi-pipeline projects, technical terminology needs explicit project-specific definition. ‘Source demo’ can mean different things in recovery pipeline (RecoveryDemo) vs standard MimicGen pipeline (success_demos.hdf5). Documentation should establish clear namespace.
  • LiPM paper demonstrates physics-informed pretraining for battery tasks: MMAE for cross-modal coupling, CIR for coulombic integration (charge conservation), dual-scale encoders for irregular sampling. Key insight: domain-specific foundation models outperform generic time series models when physical constraints are properly encoded.

Practical Learnings#

  • Documentation generated during problem-solving inherits the debugging narrative. For team onboarding docs, need explicit reframing: state current design, not the path to discovering it. Audience awareness must be externally injected, not emergent.

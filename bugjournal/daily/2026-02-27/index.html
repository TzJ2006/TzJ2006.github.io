Problems and Solutions#

Key Problems#

1. Direct SSH access to compute nodes is denied by pam_slurm_adopt (Access denied: you have no active jobs on this node)#

Solution: You must first allocate resources through salloc to obtain a JOBID, then enter the node using srun –jobid= –pty bash, or use SSH after there are active jobs. It is recommended to use tmux + salloc to maintain a stable session.

Key Insight: This cluster uses the pam_slurm_adopt security policy. Compute nodes completely disable SSH for users without any jobs, which is different from the behavior of ordinary HPC clusters.

General Problems#

2. AI cannot find the modifications mentioned by the user regarding “evaluate on training set”#

Solution: Not resolved. AI suggests that users should further describe the specific content of the modification (is it fixed initial conditions? Or train/test split?). This would help in retracing the process.

Key Insight: Cross-session conversation history is invisible to AI. If the modification does not lie in the code or MEMORY.md, it is completely impossible to trace.

3. The du command repeatedly times out when scanning超大 directories (miniconda3, chenshiyu, VLA_data, etc.), failing to complete within 120 seconds and 300 seconds#

Solution: switched to using tmpdir + background parallel processes (forking each directory separately), set the timeout to 600 seconds. Eventually, we obtained the sizes of VLA_data (248GB) and other directories. There are still 5 directories that cannot be scanned within a reasonable time due to their large size (>2TB per directory).

Key Insight: The speed of du on shared Lustre file systems is limited by both the number of inodes and network latency.超大 directories can only be estimated using parallel + timeout strategies.

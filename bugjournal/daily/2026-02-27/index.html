Problems and Solutions#

Key Problems#

1. All learning strategies (BC-RNN/Pi0/Pi0.5) had almost zero success rates in error recovery scenarios.#

Solution: This is a expected result, not something that needs fixing. The recovery progress of the random baseline was 0.0199, indicating that these strategies were even worse than random operations in error scenarios.

Key Insight: Error injection pushed the environment beyond the training distribution, causing the strategy to get stuck in local optima. This confirms the need for specialized error recovery training data (goal of M15).

General Problems#

2. The scanning speed in the Lustre distributed file system was extremely slow, with multiple user directories failing within 5-10 minutes.#

Solution: Switch to parallel bash background tasks (each directory runs independently with a timeout and runs in the background), allowing for partial results to be obtained within 120 seconds. For超大目录(VLA_data), use –max-depth=1 to limit the depth of traversal.

Key Insight: Metadata operations in Lustre FS are costly. Combining parallel scans with step-by-step depth limits is the only real-time estimation method.

3. The pam_slurm_adopt mechanism prevents direct SSH access to compute nodes (an53 test returned “access denied”).#

Solution: First, use salloc to request resources and obtain a JOBID, then use srun –pty bash to enter the node. Only after the task is allocated can SSH access be used.

Key Insight: This is a cluster security policy, not a failure. It is recommended that srun be used as the standard interface for interacting with nodes.

Summary for Today#

Key Conclusions - BC-RNN and Pi0/Pi0.5 had almost zero success rates in error recovery scenarios (649 PickPlace errors). The recovery progress of the random baseline was higher than all strategies, indicating that specialized error recovery training data is needed for M15. Practical Takeaways#

  • When using pam_slurm_adopt on the tianhexy-a cluster, you need to first obtain a JOBID through salloc before entering the node. tmux+sloc+srun is a standard workflow for stable GPU sessions.
  • The cost of Du scanning in the Lustre distributed file system is very high. Combining parallel background tasks with step-by-step depth limits allows for real-time estimation of the size of large directories.

Problems and Solutions#

Key Issues#

1. All ML strategies suffered catastrophic failures in M14 error recovery evaluations, while the random baseline had higher recovery progress#

Solution: Results indicate the need for an M15 phase—fine-tuning Pi0.5 through manual recovery demonstrations to address distribution bias

Key Insight: Models trained only on successful task demonstrations suffered catastrophic failures when errors were injected (success rate <1%), indicating severe distribution bias; error recovery requires specialized training for recovery paths, not just normal task execution

2. Unable to connect directly to GPU compute nodes via SSH (pam_slurm_adopt access denied)#

Solution: Resources must be allocated via salloc or sbatch first, then connected to active job IDs using srun or SSH; it is recommended to use tmux + salloc workflow for stable interactive sessions

Key Insight: HPC clusters use job-based access control to prevent unauthorized resource usage; direct connection to compute nodes before allocating Slurm jobs is prohibited, requiring a two-step process (allocation → connection)

3. When analyzing HDD_POOL storage, large directory scans timed out for more than 10 minutes (for directories larger than 2TB)#

Solution: Implemented parallel directory scans with a 600-second timeout per directory, enabling most scans while handling very large directories gracefully

Key Insight: Shared HPC file systems require parallel execution with clear timeout limits for directory traversal operations; some user directories (e.g., miniconda3, VLA_data, cpx1/2) are too large to be scanned using lfs quotas or –max-depth=1

Lessons Learned#

Key Lessons#

  • Severe distribution bias in error recovery tasks: Models trained on successful task demonstrations had almost no success rate (<1%) when errors were injected, while the random baseline had better recovery progress than learned strategies, indicating the need for specific training data collection for error scenarios (M15 phase)
  • For large shared HPC file systems, parallel execution with task-level timeout limits is required as individual user directories may exceed 2TB; sequential scanning is not feasible for production-scale analysis
  • HPC clusters use pam_slurm_adopt to enforce job-based SSH access control—resources must be allocated via salloc/sbatch, then connected to compute nodes via ssh/srun; tmux + salloc provides a stable interactive workflow that can withstand connection interruptions
  • Tianhe-XY cluster has 9 idle A800 GPU nodes (an16, 21, 23, 28-29, 31, 34, 39-40), each with 56 CPUs and 1TB of memory, suitable for tasks ranging from 4 to 72 hours without specifying GPU GRES specifications

Problems & Solutions#

Critical Issues#

1. All ML policies (BC-RNN 0.28%, Pi0 0%, Pi0.5 0%) show catastrophic failure on M14 error recovery evaluation, with Random baseline achieving higher recovery progress#

Solution: Results validate need for M15 phase - finetune Pi0.5 with human recovery demonstrations to address distribution shift

Key Insight: Models trained exclusively on successful task demonstrations fail catastrophically on OOD error-injected scenarios (<1% success), demonstrating severe distribution shift; error recovery requires explicit training on recovery trajectories, not just normal task execution

2. Cannot SSH directly to GPU compute nodes (pam_slurm_adopt access denied)#

Solution: Must allocate resources via salloc or sbatch first, then use srun or SSH with active job ID; recommended tmux + salloc workflow for stable interactive sessions

Key Insight: HPC clusters enforce job-based access control to prevent unauthorized resource usage; direct SSH to compute nodes is blocked until a Slurm job is allocated, requiring two-step workflow (allocate → connect)

3. Large directory scanning operations timing out (>10 minutes for directories >2TB) when analyzing HDD_POOL storage#

Solution: Implemented parallel directory scanning with per-directory 600-second timeouts, allowing completion of most scans while gracefully handling extremely large directories

Key Insight: Shared HPC filesystems require parallel execution with explicit timeouts for directory traversal operations; some user directories (miniconda3, VLA_data, cpx1/2) are so large they require alternative approaches like lfs quota or –max-depth=1 scanning

Learnings#

Key Learnings#

  • Severe distribution shift in error recovery tasks: models trained on successful demonstrations achieve near-zero success (<1%) on injected error scenarios, with Random baseline outperforming learned policies on recovery progress, validating need for error-specific training data collection (M15 phase)
  • Parallel execution with per-task timeouts is essential for scanning large shared HPC filesystems where individual user directories may exceed 2TB; sequential scanning is infeasible for production-scale analysis
  • HPC clusters use pam_slurm_adopt to enforce job-based SSH access control - must allocate resources via salloc/sbatch before SSH/srun to compute nodes; tmux + salloc provides stable interactive workflow resistant to connection drops
  • Tianhe-XY cluster has 9 idle A800 GPU nodes (an16,21,23,28-29,31,34,39-40) in ai partition with 56 CPUs and 1TB memory each, available for 4-72 hour jobs without GPU GRES specification

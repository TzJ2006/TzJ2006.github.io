日报 — 2026-02-08#

在 HPC 上为 QueST 和 MIHD 两个项目实现了 RM-Ideal（基于 WWL 图核的空间 niche 匹配分数）的完整集成，并完成了 DLPFC 151508 切片的首次 niche 查询评估实验。

今日任务#

架构与策略#

  • ✅ 在 QueST src/utils.py 中实现 RM-Ideal 计算 — 新增四个函数：wwl_node_features（WWL消息传递）、_wasserstein_distance（最优传输，POT 优先 scipy 回退）、rm_score（单对子图 RM 分数）、compute_rm_ideal_scores（批量计算），全部通过8/8个单元测试
  • ✅ 在 QueSTTrainer 中添加 compute_rm_scores() 方法 — 在 trainer.py 中集成批量 RM 分数计算，遍历所有查询 niche 和参考样本，结果写入 adata.obs，与 Tutorial 2 命名规范一致
  • ✅ 为 MIHD 创建 RM-Ideal 核心模块 utils/rm_ideal.py — 从 QueST 移植 WWL 算法，去除 squidpy/torch_geometric 依赖，新增 RMIdealEvaluator 类，支持 k-hop 子图预计算和 .npy 缓存
  • ✅ 创建 MIHD scripts/evaluate_rm_ideal.py 评估 CLI 脚本 — 包含 NicheQueryEvaluator 类，支持按标签/坐标/索引定义 niche，计算 Spearman 相关、Precision@K、SameLabel@K，生成 scanpy 3面板空间可视化

实现与修复#

  • ✅ 对 151508 切片 Layer_3 进行 RM-Ideal 评估实验 — 使用 uni_staig_fusion 嵌入运行评估：Spearman r=0.424（p~0），SameLabel@50=1.0，说明嵌入能捕获细胞类型但不能精确捕获 niche 级结构
  • ✅ 扩展 MIHD utils/spatial_utils.py 以支持 niche 图构建 — 新增 build_grid_connectivity（基于 sklearn NearestNeighbors 返回 scipy 稀疏矩阵）和 bfs_k_hop（纯 networkx BFS k-hop 子图提取），替代 squidpy 依赖
  • ✅ 修复 SameLabel@K 恒为 0 的 bug — SameLabel 指标错误地从 eval_mask 中排除了所有 niche spots，导致同标签 spot 为零。修复为在全部有效 spots 上计算同标签检索率

问题与解决方案#

关键问题#

1. SameLabel@K 指标恒为 0：niche_label=‘Layer_3’ 将所有 Layer_3 spots 选为查询 niche，eval_mask 排除这些 spots 后无同标签 spots 可检索#

解决方案: 将 SameLabel 指标改为在全部 valid_mask 上计算（含 niche spots），仅 Spearman/Precision@K 使用排除 niche 的 eval_mask

关键洞察: 检索指标设计需区分「避免自匹配偏差」（Precision@K）和「类型检索能力」（SameLabel@K）两类目标

一般问题#

2. POT（最优传输库）未安装在目标环境中，无法使用 ot.emd2()#

解决方案: 设计 POT → scipy.optimize.linprog 两级回退机制，无需新增强制依赖

关键洞察: 科研环境中依赖不完整很常见，关键算法需设计回退路径

3. node ID 冲突：查询样本和参考样本的节点 ID 空间重叠，导致 WWL 标签 dict 混淆#

解决方案: 对参考样本节点施加偏移（offset = adata_q.n_obs），确保两图节点 ID 不重叠

关键洞察: 合并来自不同样本的图时，节点命名空间需显式隔离

4. AI 最初在 MIHD 目录下搜索 QueST 源文件（src/utils.py、trainer.py），但这些文件实际在 /hpc/group/yizhanglab/zt81/QueST/#

解决方案: 用 Glob 广泛搜索后发现正确路径，后续所有操作改为直接引用 QueST 目录

关键洞察: 多仓库项目中，工作目录不等于目标仓库路径，需先定位再操作

5. conda 环境不存在：尝试激活 ‘quest’ 环境运行测试，但该环境未安装#

解决方案: 改用系统 python3 直接运行（内联函数绕过重依赖导入），或在 ‘General’ 环境中执行

关键洞察: HPC 上环境管理复杂，应先用 conda env list 确认环境存在

人类思路 vs AI 思路#

战略层面#

SameLabel@K = 0 的异常识别#

角色 思路
人类 用户在看到实验结果后立即察觉 SameLabel@50=0 不合理，主动追问原因
AI AI 在初次展示结果时未察觉该指标异常，未进行合理性检验

差异分析: 领域直觉（相同标签检索率不应为0）是人类发现 bug 的关键，AI 缺乏对「结果合理性」的先验预期

实现层面#

graphcompass 包查询的必要性#

角色 思路
人类 用户打断 AI 对 graphcompass 包的搜索，直接问「为什么要搜这个，有什么用」，并最终确定直接从 RM_Ideal_implementation_notes.md 伪代码实现
AI AI 主动提出查看 graphcompass 是否已有现成 WWL kernel 实现，计划复用

差异分析: 人类判断伪代码已足够清晰、无需依赖外部包，避免了一次无谓探索；AI 的做法是防御性调研，方向正确但执行时机过早

集成范围决策：utils.py 还是 utils.py + trainer.py#

角色 思路
人类 用户明确选择同时集成到两个文件（utils.py + trainer.py），扩大了集成范围
AI AI 给出了两种选项，等待用户决策

差异分析: 架构决策由用户主导，AI 提供备选方案后执行

AI 局限性#

重要局限#

  • 未在展示评估结果时对 SameLabel@K=0 进行合理性检验，缺乏「结果是否符合领域预期」的主动审查，需用户提醒后才发现 bug

一般局限#

  • 在多仓库 HPC 环境中，未能在首次搜索时正确推断 QueST 源文件路径，需经过两轮错误搜索后才定位到正确目录
  • 启动了多个后台任务（find、conda run 等）但随即发现不再需要，造成资源浪费和通知干扰；conda run 启动慢导致主流程阻塞

今日收获#

核心收获#

  • RM-Ideal（WWL图核 + Wasserstein距离）与嵌入余弦相似度是两类互补的 niche 相似度度量：RM-Ideal是基于细胞类型标签的结构真值，余弦相似度是学习到的表征空间距离；Spearman r=0.424 说明 uni_staig_fusion 嵌入仅中等程度地捕获了 niche 级结构
  • 同一 niche 评估中，「避免自匹配偏差」（Spearman/Precision@K 排除 niche spots）和「类型检索能力」（SameLabel@K 保留 niche spots）应使用不同的评估集，混用会导致指标失真
  • 在将 QueST RM-Ideal 移植到 MIHD 时，去除 squidpy/torch_geometric 依赖并改用 scipy sparse + pure networkx，证明了核心算法（WWL + OT）可在轻量依赖环境下独立运行

实践收获#

  • k-hop 子图预计算 + .npy 缓存是 RM-Ideal 在大规模数据（4384 spots）下可用的关键：首次计算约8分钟，缓存后重复运行近乎即时

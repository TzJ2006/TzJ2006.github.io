Problems and Solutions#

Key Issues#

1. SameLabel@K returns 0.0, even though the embedding clearly retrieves the same cell type#

Solution: The root cause was using ~niche_mask to exclude all specific query points, resulting in no points remaining in the Layer_3 area. This issue was resolved by calculating SameLabel using all_valid_mask while retaining Precision@K on the eval_mask.

Key Insight: Different evaluation metrics require different exclusion masks—retrieval rate needs a complete valid set, while ranking consistency requires excluding the query area itself.

2. The initial plan required implementing RM-Ideal from scratch, but code review showed it was already implemented#

Solution: Manual verification of the existing implementation instead of duplicating work; conducted comprehensive testing to confirm that the same/subdifferentiated subgraphs returned 1.0/0.0 respectively, as expected.

Key Insight: Always verify the status of the existing code library before implementing—documents may not match the actual code.

General Issues#

3. There is a risk of node ID conflicts when combining query subgraphs with shared label dictionaries#

Solution: Apply an id offset of datacq.n_obs to all reference node IDs before creating the merged label dictionary to avoid key conflicts.

Key Insight: Node indices overlap when merging graphs from different AnnData objects—use deterministic offsets rather than relying on non-overlapping ID spaces.

4. Dependency issue with Wasserstein distance solver—POT library not installed in the quest conda environment#

Solution: Implemented a two-stage solver: first tried importing ot.emd2() (fast C backend), and if failed, used scipy.optimize.linprog and HiGHS solver (no additional dependencies required, suitable for small subgraphs with around 30-100 nodes).

Key Insight: Always provide an alternative solution without dependencies for optional performance optimizations—scipy is very common, while POT is not.

Lessons Learned#

Key Lessons#

  • RM-Ideal (area matching with ideal labels) uses WWL graph kernel: K iterations neighbor tag hashing → Hamming cost matrix (normalized by K+1) → Wasserstein distance → Similarity = 1-W. It can capture spatial-specific structures independently of the learned embeddings.
  • Careful design of masks for specific query evaluation metrics: Precision@K uses eval_mask (excludes specific query points to avoid self-matching), SameLabel@K uses all_valid_mask (includes specific query points to obtain retrieval rate), Spearman uses eval_mask (conducts ranking correlation analysis on non-self-matching points)

Practical Experience - Migrating Graph Algorithms Between Frameworks: s squidpy.gr.spatial_neighbors() → sklearn NearestNeighbors + scipy sparse, torch_geometric.k_hop_subgraph() → networkx BFS. Pattern: Convert to the equivalent solution with minimal dependencies.#

  • Cache strategy for expensive calculations: RMIdealEvaluator pre-calculates all k-hop subgraphs during initialization (about 8 minutes, suitable for 4384 positions), saves them to .npy cache files, which can be loaded immediately during subsequent runs. Cache keys should include spatial graph parameters (n_neighbors, k_hop).

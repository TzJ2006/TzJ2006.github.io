Problems and Solutions#

Key Problems#

1. Footer jitter during window height animation: The flex-end layout cannot solve this problem because the screen position of the Footer equals window_top+content_height, there is an inherent 1-frame delay between SetWindowPos and CSS re-layout.#

Solution: Change the Footer to position:fixed;bottom:0, directly anchoring it to the bottom of the viewport, completely bypassing the CSS layout frame delay.

Key Insight: The 1-frame delay between Win32 SetWindowPos and browser CSS re-layout is an inherent architecture problem. Any positioning scheme that relies on CSS layout flow cannot eliminate this delay. Position:fixed relative to the viewport is the only way to bypass it.

2. Windows anchor point detection depends on a 5px absolute threshold, resulting in incorrect anchor direction judgment for non-edge windows; bottom anchor uses work.bottom, causing the window to jump to the screen working area bottom when resized instead of keeping the current bottom edge.#

Solution: Change to comparing the relative sizes of top_gap and bottom_gap to determine the anchor point; change bottom anchor to current_rect.bottom.min(work-bottom) - height, keeping the current bottom edge unchanged.

Key Insight: Anchor point judgment should be based on the relative distance of ‘which side is closer’, not an absolute threshold. Keeping the bottom edge fixed means calculating a new top using ‘current bottom position - new height’, rather than using the screen bottom.

3. #app bottom edge has a temporary transparent gap when the window expands: #app’s min-height:100% needs to wait for CSS re-layout updates, while SetWindowPos has already been executed, and the newly added area of the window becomes transparent during this period.#

Solution: Execute appEl.style.minHeight=nextHeight+‘px’ synchronously before invoking IPC, allowing the browser to immediately layout #app to the target height in the same frame.

Key Insight: JS directly setting style properties is 1-frame earlier than waiting for CSS re-layout, matching the IPC sending timing, ensuring that #app always stays synchronized with the window size.

4. SWP_NOCOPYBITS flag causes WebView2 to discard all old content after each SetWindowPos, making the window area completely transparent before repainting, resulting in flickering.#

Solution: Remove SWP_NOCOPYBITS from resize_window_pos_flags(), keeping only the pixels of the old content for the newly added areas.

Key Insight: The design of SWP_NOCOPYBITS is intended for scenarios where content changes completely. It’s harmful for Tauri transparent windows, where content needs continuous rendering.

General Problems#

5. Chart hover panel flickers when switching between bars:{#key displayedIdx} causes DOM destruction and reconstruction, with old nodes fading out in 500ms and new nodes fading in in 500ms simultaneously.#

Solution: Remove the block enclosed by{#key displayedIdx} to allow Svelte to update content responsively; shorten DETAIL_FADE_MS to 150ms; merge scheduleSlotCleanup into beginDetailHide to eliminate intermediate states.

Key Insight: The {#key} block will completely destroy and reconstruct the DOM every time its value changes, suitable for scenarios that need state reset, but causing unnecessary double animations for scenarios that only require content updates.

Today’s Gains#

Core Gains#

  • There is an inherent 1-frame delay between Win32 SetWindowPos and browser CSS re-layout, which is a core constraint in Tauri desktop application development. Solution: Use position:fixed to bypass CSS layout for UI elements at the bottom; use JS to pre-synchronize style.minHeight for the background container; avoid SWP_NOCOPYBITS to preventWebView2 from repainting throughout the frame.
  • Practical verification of cchypothesis-style assumption-driven debugging: Turning intuitive guesses into verifiable hypotheses and investigating them concurrently effectively shortened the debugging cycle. Key constraint: The investigation phase of parallel subagents must be read-only, and writing operations (editing code, running tests) must be serialized; otherwise, file write conflicts will occur.
  • Visual quality verification of Tauri applications must rely on manual visual testing. Unit tests and type checks can only verify logical correctness and cannot detect native window-level visual defects such as frame delays and transparent gaps.

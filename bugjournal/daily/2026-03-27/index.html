Problems and Solutions#

Key Issues#

1. The “Xs ago” text in the footer moves up/down when the window size is adjusted, despite the anchoring fix.#

Solution: Set the position to position:fixed; bottom:0, so it stays fixed on the viewport instead of relying on layout.

Key Insight: SetWindowPos and CSS relayout have an inherent 1-frame delay in Win32+WebView2 architecture; only position:fixed can bypass CSS layout timing.

2. SWP_NOCOPYBITS causes WebView2 content to disappear every time SetWindowPos is called.#

Solution: Removed the SWP_NOCOPYBITS flag; Windows retains the old pixel content, just making the new area ineffective.

Key Insight: SWP_NOCOPYBITS optimization for native windows has a reverse effect in WebView2 — a blank window appears every frame until rendering is completed.

3. The chart hover detail panel flickers and jumps when the mouse moves between bars.#

Solution: Removed the DOM destruction of #key displayedIdx, reduced the fadeout duration from 500ms to 150ms, and simplified the two-stage hiding process to a single stage.

Key Insight: DOM destruction and recreation during transitions creates visual repetition; reactive content updates are smoother than reloads triggered by keys.

4. The border radius edge of the #app bottom frame shows a transparent gap when the size is adjusted.#

Solution: Pre-set appEl.style.minHeight in JS before calling IPC, so the browser can present the #app to the target height in the same frame as SetWindowPos.

Key Insight: IPC latency between JS and Rust creates visual artifacts; synchronizing CSS changes can eliminate these artifacts.

5. Window anchoring detection fails at any position (not just when 5px away from the edge).#

Solution: Changed detect_vertical_anchor to a distance-based comparison check (top_gap vs bottom_gap); used current_rect.bottom instead of workBottom for bottom anchors.

Key Insight: Threshold-based detection creates dead zones where window behavior is unpredictable; distance-based comparisons work fine in any case.

Learning Outcomes#

Key Learnings#

  • Win32+WebView2 architecture has an inherent 1-frame delay between SetWindowPos (native) and CSS relayout; only position:fixed can bypass this timing dependency.
  • Hypothesis-based debugging is more effective than parallel investigations; simultaneously investigating H1/H2/H3 by 3 sub-agents saves time and accumulates evidence.
  • Evolving Constraint Planning’s adversarial review (stage 5) uncovered key issues: identified 10 problems before implementation, including sub-agent writing conflicts, lack of success criteria, context exhaustion, etc.
  • SWP_NOCOPYBITS optimized native window redrawing but disrupted WebView2 rendering — a blank window appears every frame until the browser re-renders all content.
  • Observable success criteria (like “Xs ago” text position) are better for debugging UI issues than abstract indicators (window coordinates).

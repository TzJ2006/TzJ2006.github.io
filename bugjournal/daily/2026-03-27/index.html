Problems & Solutions#

Critical Issues#

1. Footer ‘Xs ago’ text moves up/down during window resize despite anchor fixes#

Solution: Applied position:fixed; bottom:0 to footer, making it viewport-anchored rather than layout-dependent

Key Insight: SetWindowPos and CSS re-layout have inherent 1-frame delay in Win32+WebView2 architecture; only position:fixed bypasses CSS layout timing

2. SWP_NOCOPYBITS causes WebView2 content to vanish during every SetWindowPos frame#

Solution: Removed SWP_NOCOPYBITS flag; Windows will preserve old pixel content and only invalidate new regions

Key Insight: SWP_NOCOPYBITS optimization for native windows backfires with WebView2 - each frame shows empty window until re-render completes

3. Chart hover detail panel flickers and jumps when mouse moves between bars#

Solution: Removed {#key displayedIdx} DOM destruction, reduced fade duration 500→150ms, simplified two-phase hide flow to single flow

Key Insight: DOM destruction/recreation during transitions creates visual doubling; reactive content updates are smoother than key-triggered remounts

4. #app bottom border-radius edge shows transparent gap during resize#

Solution: Preemptively set appEl.style.minHeight in JS before invoke() IPC call, so browser lays out #app to target height in same frame as SetWindowPos

Key Insight: IPC latency between JS and Rust creates visual artifacts; pre-applying CSS changes synchronously eliminates gaps

5. Window anchor detection fails at arbitrary positions (not just 5px from edges)#

Solution: Changed detect_vertical_anchor from threshold-based (5px) to distance-comparison (top_gap vs bottom_gap); bottom anchor uses current_rect.bottom instead of work.bottom

Key Insight: Threshold-based detection creates dead zones where windows behave unpredictably; distance comparison works everywhere

Learnings#

Key Learnings#

  • Win32+WebView2 architecture has inherent 1-frame delay between SetWindowPos (native) and CSS re-layout (web); only position:fixed bypasses this timing dependency
  • Hypothesis-driven debugging with parallel investigation scales better than sequential trial-and-error; 3 subagents investigating H1/H2/H3 concurrently saves time and accumulates evidence
  • Evolving Constraint Planning’s adversarial review (Phase 5) catches critical issues: 10 findings identified subagent write conflicts, missing success criteria, context exhaustion - all before implementation
  • SWP_NOCOPYBITS optimizes native window repaints but breaks WebView2 rendering - each frame shows blank window until browser re-renders all content
  • Observable success criteria (like ‘Xs ago’ text position) are better than abstract metrics (window coordinates) for debugging UI issues

Problems & Solutions#

Key Problems#

1. Footer jitters during window height animation: flex-end layout cannot solve the problem because Footer’s screen position = window_top + content_height, and there is an inherent 1-frame delay between SetWindowPos and CSS re-layout#

Solution: Changed the Footer to position:fixed;bottom:0, anchoring it directly to the viewport bottom and entirely bypassing the CSS layout frame delay.

Key Insight: The 1-frame delay between Win32 SetWindowPos and browser CSS re-layout is architecturally inherent. Any positioning approach that depends on the CSS layout flow cannot eliminate this delay; position:fixed relative to the viewport is the only viable bypass.

2. Windows anchor detection relied on an absolute 5px threshold, causing non-edge windows to fail to correctly determine the anchor direction; bottom anchor used work.bottom, causing the window to jump to the screen’s work area bottom on resize instead of keeping the current bottom edge#

Solution: Changed to comparing the relative sizes of top_gap and bottom_gap to determine the anchor; bottom anchor changed to current_rect.bottom.min(work.bottom) - height to keep the current bottom edge unchanged.

Key Insight: Anchor detection should be based on “which edge is closer” using relative distance, not an absolute threshold. Keeping the bottom edge still means calculating the new top as “current bottom position − new height,” not calculating from the screen bottom.

3. The #app bottom edge briefly shows a transparent gap when the window expands: #app’s min-height:100% needs to wait for CSS re-layout to update, but SetWindowPos has already executed, leaving the newly expanded window area transparent in the interim#

Solution: Synchronously execute appEl.style.minHeight = nextHeight + 'px' before invoking the IPC, letting the browser immediately lay out #app to the target height in the same frame.

Key Insight: Setting a style attribute directly in JS is 1 frame ahead of waiting for CSS re-layout, matching the timing of the IPC send, which ensures #app stays in sync with the window size at all times.

4. The SWP_NOCOPYBITS flag causes WebView2 to discard all old content after every SetWindowPos call, making the window area completely transparent before repainting and producing flicker#

Solution: Removed SWP_NOCOPYBITS from resize_window_pos_flags(), preserving old pixel content and only invalidating newly added regions.

Key Insight: SWP_NOCOPYBITS is designed for scenarios where content will change completely. For scenarios like a Tauri transparent window that requires continuous rendering, it is harmful.

General Problems#

5. Chart hover panel shows double-animation flicker when switching between bars: {#key displayedIdx} triggers full DOM destruction and recreation; the old node’s 500ms fade-out and the new node’s 500ms fade-in occur simultaneously#

Solution: Removed the {#key displayedIdx} wrapper block to let Svelte reactively update content; shortened DETAIL_FADE_MS to 150ms; merged scheduleSlotCleanup into beginDetailHide to eliminate intermediate state.

Key Insight: A {#key} block fully destroys and recreates the DOM on every value change. This is appropriate when state needs to be reset, but for scenarios that only need to update content, it causes unnecessary double animations.

Today’s Key Learnings#

Core Learnings#

  • There is an inherent 1-frame delay between Win32 SetWindowPos and browser CSS re-layout — this is a core constraint of Tauri desktop application development. Solutions: use position:fixed for bottom-anchored UI elements to bypass the CSS layout frame; use JS to synchronously pre-set style.minHeight for background containers; avoid SWP_NOCOPYBITS to prevent WebView2 full-frame repaints.
  • Practical validation of cchypothesis-style hypothesis-driven debugging: converting intuitive guesses into falsifiable hypotheses and investigating them in parallel effectively shortens debugging cycles. Key constraint: the investigation phase of parallel subagents must be read-only; write operations (editing code, running tests) must be serialized, otherwise file write conflicts will occur.
  • Visual quality validation for Tauri applications must rely on manual visual testing. Unit tests and type checks can only verify logical correctness; they cannot detect frame delays, transparent gaps, or other visual defects at the native window level.

Practical Learnings#

  • Window anchor detection should be based on relative distance comparison rather than absolute thresholds: directly compare the sizes of top_gap and bottom_gap. The window at any position can correctly choose the nearest edge as its anchor without requiring special thresholds for edge cases.

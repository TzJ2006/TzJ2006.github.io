Problems and Solutions#

Key Problems#

1. The “Xs ago” text in the footer moves up and down when the window size is adjusted, despite the anchor fix.#

Solution: The position was set to fixed; bottom:0, so it stays fixed in the viewport instead of relying on layout.

Key Insight: There is a 1-frame delay between SetWindowPos and CSS re-layout in Win32+WebView2 architecture. Only position:fixed can bypass CSS layout timing.

2. SWP_NOCOPYBITS causes the WebView2 content to disappear every time SetWindowPos is called.#

Solution: The SWP_NOCOPYBITS flag was removed. Windows retains the old pixel content and only invalidates the new area.

Key Insight: SWP_NOCOPYBITS optimization works for native windows, but causes problems with WebView2—it shows a blank window every frame until the rendering is complete.

3. The chart hover details panel flickers and jumps when the mouse moves between bars.#

Solution: The DOM destruction at #key displayedIdx was removed. The fade-out duration was reduced from 500 milliseconds to 150 milliseconds. The two-stage hiding process was simplified to a single stage.

Key Insight: DOM destruction and restoration during transitions cause visual repetition. Responsive content updates are smoother than button clicks to reload.

4. The border radius edge of the #app bottom border has transparent gaps when resized.#

Solution: Before calling IPC, appEl.style.minHeight was set in JS. This allows the browser to layout the #app to the target height in the same frame as SetWindowPos.

Key Insight: IPC latency between JS and Rust creates visual artifacts. Synchronizing CSS changes can eliminate gaps.

5. Window anchor detection fails at any location (not just 5px from the edge).#

Solution: detect_vertical_anchor was changed from threshold-based detection to distance comparison (top_gap vs bottom_gap). For bottom anchors, current_rect.bottom was used instead of workBottom.

Key Insight: Threshold-based detection creates dead zones where window behavior is unpredictable. Distance comparison works in all cases.

Learning Outcomes#

Key Learnings#

  • There is a 1-frame delay between SetWindowPos (native) and CSS re-layout. Only position:fixed can bypass this temporal dependency.
  • Hypothesis-based debugging is more effective than parallel investigations. Three sub-agents investigating H1/H2/H3 simultaneously saves time and accumulates evidence.
  • The adversarial review of evolutionary constraint programming (5th stage) uncovered key issues: 10 issues were identified before implementation, including sub-agent writing conflicts, lack of success criteria, context exhaustion, etc.
  • SWP_NOCOPYBITS optimized native window repaints but damaged WebView2 rendering. Every frame showed a blank window until the browser re-rendered everything.
  • Observable success criteria (such as “Xs ago” text position) are better for debugging UI issues than abstract indicators (window coordinates).

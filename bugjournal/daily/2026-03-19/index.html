Issues and Solutions#

Key Issues#

1. Post-fusion Harmony improves batch mixing metrics but reduces expertise retrieval quality across samples#

Solution: After analyzing the results, it was concluded that Harmony’s linear correction cannot handle non-linear dimensional meanings, resulting in inconsistent training of individual parts. It is recommended to use Joint STAIG multi-part training in the next iteration.

Key Insight: Batch correction methods like Harmony assume linear offsets, but these corrections fail when independent training produces embeddings with inconsistent dimensional semantics. Batch mixing and semantic alignment are two different properties. Linear correction on independently trained embeddings cannot maintain semantic consistency across samples.

**2. CalendarPro can only handle a single intent per message, leading to loss of subsequent actions like " scheduling a meeting at 3 PM and changing dinner time to 7 PM "#

Solution: Replaced the dual-validation process with Claude CLI’s multi-intent analyzer, which returns a series of intentions. Updated the _process_message_queue function to iterate through intentions and combine responses. Removed the 7-layer keyword coverage system.

Key Insight: Single-intent classification is a fundamental architectural limitation, not a parsing issue. Keyword techniques (always generic, context-dependent, 7-layer coverage) are symptoms of confusion in the abstraction layer. Switching to LLM-native multi-intent functionality eliminates the need for all heuristic methods.

3. MimicGen augmentation failures: calculated pos_delta but never applied to action deformation, resulting in success rates below 5%#

Solution: Fixed the _replay_with_warping function by applying pos_delta to warpedactions[:, :3] before adding noise. The expected success rate should be above 50% for D0 translation errors.

Key Insight: MimicGen augmentation requires actual coordinate transformation, not just noise injection. The 10→1000 path depends on: 10 seeds × 200 targets × 50% success rate = 1000 demonstrations. Without pos_delta fixes, the success rate will drop significantly.

4. Yuke Zhu’s initial deep extensive analysis using a single 5MB prompt exceeded the LLM’s processing capacity, resulting in failure#

Solution: Implemented a three-tier solution: ① Added a threshold of _MAX_PROMPT_CHARS=150K to automatically trigger segmentation; ② Used a greedy algorithm that groups by year and segments each paper when the data volume exceeds a certain limit; ③ Used the LLM to merge 47 data blocks into a complete report.

Key Insight: The key to large-scale text analysis lies in “divide and conquer + semantic merging”: technically, dynamic segmentation can control the burden of a single LLM, while cognitively guiding the LLM to extract consistency across segments rather than simply concatenating prompts. 150K characters correspond to approximately 15-20 medium-length papers, which is the optimal point empirically.

5. Task completion model training stalled at a loss of about 0.035, despite grad_norm=0.7-1.0 and a positive sample ratio of 4%, with negative samples accounting for 96%#

Solution: Applied a positive sample weighting of pos_weight=10 to the BCE loss to balance gradient contributions, effectively increasing the positive sample ratio from 4% to about 30%.

Key Insight: In cases of low loss, the recall rate of positive samples in an imbalanced dataset does not guarantee good performance. Focusing solely on gradient magnitude is misleading, as negative samples dominate. Clear gradient balancing is required for category imbalance.

6. CalendarPro immediately returns “No events today” for queries like “This week” and “Tomorrow” after creating events#

Solution: Adopted a three-tier solution: (1) Adding intent classification to BASE_SYSTEM_PROMPT (instead of only using INTENT_PROMPT_FRAGMENTS in multi_intent.py), (2) Adding keyword backtracking detection in handle_query() to handle LLM failures, (3) Fixing get_today_events() to display already scheduled events in get_today_events(), (4) Adding Pending_warnings for Discord error feedback.

Key Insight: INTENT_PROMPT_FRAGMENTS and BASE_SYSTEM_PROMPT serve different code paths—the main LLM classification uses BASE_SYSTEM_PROMPT. Keyword backtracking is a reliable LLM safety net. Past events must remain visible to prevent re-creating them.

7. The code repository contains 8 fully implemented modules (approximately 2400 lines of code), which have comprehensive tests but never import during runtime, causing maintenance difficulties#

Solution: Adopted a systematic four-stage refactoring approach: (1) Listing all imported modules and identifying isolated modules, (2) Archiving unused modules and explaining their purpose in README files, (3) Merging duplicate modules into the specification, (4) Deleting truly useless code. After 373 tests, all passed.

Key Insight: For fully implemented modules, archiving is preferable to deletion. Isolated but still usable code should be archived in an archive directory along with relevant context information for future integration. Audit findings indicated discrepancies in configuration and errors in import statements.

13. Multiple analysis failures due to LLM timeouts (the initial 300-second timeout was insufficient for a merge of 187K, and 420 seconds was still insufficient).#

Key Insight: The timeout design should be linearly related to the data size, rather than being a fixed value. Large models have a relatively stable processing speed (about 200-300 characters per second), and it is a reliable strategy to estimate the time based on the prompt length plus 50% buffer time.

14. There are systematic issues with escaping double quotes in the long JSON text fields generated by the LLM, resulting in JSON parsing failures (more than 12 errors).#

Solution: Identify and manually escape special symbols in fields like trajectory_summary and breakthrough[].description (e.g., “let the robot…” → “let the robots…”), and for truncated JSON objects, supplement the missing right parentheses and array closing symbols based on the context.

Key Insight: When the LLM generates long JSON fields, the handling of escaped double quotes is unstable, especially in academic texts that contain frequently cited terms (e.g., “image understanding” becomes “moving while looking”). This leads to formatting errors. Solutions require: (1) explicitly requiring escaped special characters in prompts; (2) setting length limits for overly long text fields, or using multi-round segmentation; (3) adding JSON schema validation steps after generation to automatically fix common formatting errors.

15. Long JSON outputs generated by the LLM have formatting errors: truncation (especially in the last few fields), unclosed quotes, unescaped Chinese quotes (""), and missing closing brackets in nested structures, especially when containing technical descriptions, Chinese text, and deep nesting (greater than 3 levels).#

Solution: A four-step repair process was implemented: (1) directly attempting json.loads; (2) extracting json code blocks; (3) repairing unclosed structures based on depth-based bracket matching; (4) fixing complex errors through model upgrades, from haiku to tanka to epic poetry. Special JSON repair prompts instruct the AI to extract and repair JSON, returning only pure JSON without explanations. All failures are recorded in outputs/logs/json_repair_failed/ for schema analysis and future optimization.

Key Insight: Long JSON files generated by the LLM show systematic failure patterns in different sessions, even with the same prompts (e.g., string escape issues, truncation issues, nesting errors). A single repair strategy is insufficient to address these issues; a layered fault tolerance mechanism is needed. Chinese quotes """ are semantically valid, but must be escaped to ASCII double quotes in JSON string values. Production systems need to validate the generated JSON and support automatic repair retry functions.

16. For highly productive researchers (with more than 130 papers), analyzing the entire corpus at once results in prompt length errors or insufficient analysis. The analysis of Yuke Zhu’s 206 papers failed due to “too long prompts.”#

Solution: An iterative analysis strategy based on year segmentation was adopted: papers were grouped by publication year, and 1-3 representative papers were analyzed in each round to uncover their breakthroughs (descriptions, why previous methods were not used, impacts, technical insights). The complexity of each call was controlled while maintaining the depth of analysis. Paper selection: based on year grouping + citation count ranking + conference award recognition (best paper/top pick/focus/oral presentation recognized by LLM), with a maximum of 10 papers per year.

Key Insight: The year-based segmentation strategy is better than analyzing the entire corpus at once (too long prompts/insufficient analysis) or randomly sampling (missing temporal evolution). It maintains the integrity of time dimensions while considering paper quality. The “few but deep” iterative approach is more suitable for academic analysis than the “many but shallow” one.

17. Reusing the VLM prefix forward pass is required for completing the head reasoning (once in sample_actions, once in predict_task_completion).#

Solution: The sample_actions_with_completion() function was created, which captures the result of the first VLM’s forward pass and reuses it for completing the head, returning a tuple of actions and completion_probability.

Key Insight: Stream matching denoising is cost-effective, but unnecessary VLM forward passes increase measurable overhead. The unified method eliminates redundancy with minimal restructuring.

19. Daily report merging task: How to avoid copying all tasks individually and instead create a comprehensive view.#

Solution: Specific operations include: (1) combining similar tasks into one entry, retaining the latest status; (2) integrating human-machine analysis for the same topics; (3) combining multiple sessions for the same project into one conversation summary; (4) retaining content with higher importance during deduplication. This requires the LLM to understand the causal relationships between tasks (e.g., “implementing a feature” → “fixing an error” → “verifying the fix” is considered a stage of the same work process) as well as topic similarity.

Key Insight: Direct concatenation leads to information redundancy. Synthesis requires identifying causal relationships between tasks (whether sequential or truly parallel) as well as topic similarity (repeated debugging of the same module should be treated as one task), not just string matching.

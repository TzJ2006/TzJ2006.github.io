Problems and Solutions#

Key Issues#

1. Post-fusion Harmony improved batch hybrid metrics but reduced the quality of cross-sample niche retrieval#

Solution: After analyzing the results, it was concluded that Harmony’s linear correction couldn’t handle non-linear dimensional meanings, leading to inconsistencies in the independent training of each part. It is recommended to use Joint STAIG multi-part training in the next iteration.

Key Insight: Batch correction methods like Harmony assume a linear offset, but fail when the independently trained embeddings have inconsistent dimensional semantics—batch mixing and semantic alignment are different attributes. Linear correction of independently trained embeddings can’t maintain semantic correspondence between samples.

2. CalendarPro can only handle a single intent in each message, missing subsequent actions such as “scheduling a meeting at 3 PM and changing dinner time to 7 PM”#

Solution: Replace the single-intent double-verification process with Claude CLI MultiIntentAnalyzer, which returns a series of intents. Updated _process_message_queue to iterate through intents and combine responses. Removed the 7-tier keyword coverage system.

Key Insight: Single-intent classification is a fundamental architectural limitation, not a parsing issue. Keyword techniques (always generic, context-dependent, 7-tier coverage) are symptoms of an abstract failure. Switching to LLM-native multi-intent processing eliminates the need for all heuristics.

3. MimicGen enhancement failed: calculated pos_delta but never applied to warped actions, resulting in a success rate below 5%#

Solution: Fixed _replay_with_warping so that it applies pos delta to warped actions[:, :3] before adding noise. Expected success rate: For D0 translation errors, the success rate should be above 50%.

Key Insight: MimicGen enhancement requires actual coordinate transformation, not just noise injection. The path from 10 to 1000 depends on: 10 seeds × 200 targets × 50% success rate = 1000 demonstrations. Without pos delta fix, the success rate will drop significantly.

4. Yuke Zhu’s initial depth-width analysis used a single 5MB prompt (containing 100 papers), which far exceeded the LLM’s processing capacity, resulting in failure#

Solution: Implemented a three-tier solution: ① Added _MAX_PROMPT_CHARS=150K threshold to automatically trigger segmentation; ② Used a greedy algorithm with year-based grouping and separate splitting for annual papers; ③ Used an LLM to merge 47 segmented analyses into a complete report.

Key Insight: The key to super-large text analysis lies in “divide and conquer + semantic merging”: Technically, it controls the single LLM load through dynamic segmentation, while cognitively guides the LLM to extract consistency across segments, rather than simply combining prompts for simple connection. 150K characters is roughly equivalent to about 15-20 medium-length papers, which is the ideal point in experience.

5. Despite grad_norm=0.7-1.0, task completion model training still stalled with a loss of about 0.035, while the positive sample ratio was 4% and the negative sample ratio was 96%#

Solution: Applied positive sample weighting (pos_weight=10) to the BCE loss to balance gradient contributions, effectively increasing the positive sample ratio from 4% to about 30%.

Key Insight: In datasets with low imbalance, low loss doesn’t guarantee good positive sample recall. It’s misleading just based on the magnitude of gradients, as negative samples are often learned well. Clear gradient balancing is needed for class imbalance.

6. CalendarPro returned “No events today” immediately after creating an event for queries like “this week” and “tomorrow”#

Solution: Adopted a three-tier repair approach: (1) Added intent classification to BASE_SYSTEM_PROMPT in (2) _handle_query(), to cover LLM failures; (3) Fixed get_today_events to include_past=True to show events that have already started; (4) Added _DPending_warnings for Discord error feedback.

Key Insight: INTENT_PROMPT_FRAGMENTS and BASE_SYSTEM_PROMPT serve different code paths—the main LLM classification uses BASE_SYSTEM_PROMPT. Keyword fallback is a reliable LLM safety net. Past events must remain visible to prevent re-creating them.

7. The code library contained 8 fully implemented modules (about 2400 lines of code), which never were imported during runtime, causing maintenance burdens#

Solution: Systematic 4-stage refactoring: (1) Mapping all import statements and identifying isolated modules; (2) Archiving unused modules with a README file attached; (3) Merging duplicate parse_datetime_with_timezone functions into the specification location; (4) Deleting truly useless code. Verified through 373 successful tests.

Key Insight: For full implementations, archiving > deletion approach was used. Isolated but still running code was saved in archives and attached with context to be re-integrated when cleaning up active tree structures. Audit found changes in configuration and import statements errors.

8. Due to always failing checks, 19 breakfast events, 19 lunch events, and 18 dinner events were repeatedly created#

Solution: Adjusted the priority of the date parsing function from display string (‘08:00’) to the ISO date field. Changed the meal event visibility so that ongoing meals remained visible.

Key Insight: The order of field parsing is crucial—ISO format fields should always be processed first to avoid silent parsing failures. To prevent repeated creation, past events must remain visible.

9. Deadlock occurred in the summarization mechanism: historical log files with finalized=False remained permanently, preventing LLM merge from completing#

Solution: The root cause was identified: the two-tier finalization mechanism (logs + reports) requires all sources to be finalized before marking the report. The reason for keeping historical logs in a False state was due to: (a) only one export was performed, thus maintaining the False state; (b) changes in the number of dialogues after the first export led to writing False. The finalized flag is embedded in the JSON content and synchronized using rclone. Solution: Re-export the affected devices on these dates—since the date has passed and the number of dialogues is stable, it will be automatically marked as True. Another method is to manually edit the finalized field in the JSON to True, then upload it using rclone.

Key Insight: The two-tier finalization mechanism requires all sources to be finalized before marking the report. The flag in the JSON content synchronized by rclone ensures consistency across devices. However, historical False entries can lead to permanent deadlocks. The key value is to save LLM API resources (merge phase), rather than rclone download time. Re-exporting through the “data repair” method is more in line with the existing architecture than the “logical change” method—since removing all checks like all_sources_finalized would break consistency.**

When using a block size of 60K for the first split, 100 papers were divided into 102 blocks (almost one paper per block), resulting in excessive LLM calls. Single papers lacked context.**#

Solution: Increase the block size from 60K to 150K, adjust the splitting logic to prioritize year integrity, and ultimately reduce the number of blocks from 102 to 47. This significantly reduces the number of LLM calls while maintaining time context.**

Key Lessons - Batch correction methods like Harmony can achieve global distribution mixing, but they cannot maintain semantic consistency when independent training produces embeddings with inconsistent dimensionality meanings. This requires shared training (Joint STAIG) or a base model. Batch mixing and semantic consistency are independent attributes.#

  • The enhancement in MimicGen style is essentially about coordinate transformation, not noise injection. It is necessary to apply pos_delta to the action trajectory (the offset between the source object and the target object). Without this, the enhancement from 10 to 1000 fails because the success rate is less than 5%.
  • The engineering paradigm for ultra-large text analysis: ① Automatic segmentation (150K characters, about the optimal granularity for 15-20 papers); ② Semantic merging (merging driven by LLM instead of splicing); ③ Dynamic timeout mechanism (linear scaling with prompt length, baseline is 300 seconds, extended to 120 seconds at 50K characters); ④ Caching mechanism (to avoid retry costs). These four are key for the availability of deep-wide patterns.
  • The multi-angle verification method for academic trajectory analysis: By iterating analyzing subsets of different papers by the same researcher (from 1 paper to 100 papers), a three-dimensional cognition can be constructed from the micro level (in-depth analysis of a single representative work) to the macro level (descriptions of complete publication list trajectories).

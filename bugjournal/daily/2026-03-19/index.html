Problems and Solutions#

Key Problems#

1. Post-fusion Harmony alignment improved batch mixing metrics but reduced expertise retrieval quality across samples#

Solution: After analyzing the results, it was concluded that Harmony’s linear correction could not handle non-linear dimensional meanings, resulting in inconsistent training of individual parts. It is recommended to use Joint STAIG multi-part training in the next iteration.

Key Insight: Batch correction methods like Harmony assume linear offsets, but these corrections fail when independent training embeddings have inconsistent dimensional semantics. Batch mixing and semantic alignment are two different attributes. Linear correction on independently trained embeddings cannot maintain semantic consistency across samples.

2. CalendarPro can only handle a single intention per message, resulting in missed subsequent actions, such as " scheduling a meeting at 3 PM and changing dinner time to 7 PM "#

Solution: The Claude CLI multi-intention analyzer was replaced with a dual-validation process, allowing the analyzer to return multiple intentions. The process_message_queue function was updated to handle multiple intentions. The 7-layer keyword coverage system was removed.

Key Insight: Single-intention classification is a fundamental architectural limitation, not a parsing issue. Keyword techniques (always generic, context-dependent, 7-layer coverage) are symptoms of confusion at the abstraction level. Switching to LLM-native multi-intention capabilities eliminates the need for all heuristic methods.

3. MimicGen enhancement failed: Pos_delta was calculated but never applied to action deformation, resulting in a success rate of less than 5%#

Solution: The replay_with_warping function was corrected. pos_delta was applied to warped_actions[:, :3] before adding noise, enabling correct object-centered trajectory transformation. The expected success rate was 50% or higher for D0 translation errors.

Key Insight: MimicGen enhancement requires actual coordinate transformation, not just noise injection. The 10→1000 path depends on: 10 seeds × 200 targets × 50% success rate = 1000 demonstrations. Without pos_delta correction, the success rate will drop significantly.

4. Yuke Zhu’s preliminary deep-wide analysis using a single 5MB prompt (covering 100 full papers) far exceeded the LLM’s capacity, resulting in failure#

Solution: A three-tier solution was implemented: ① Adding a threshold of _MAX_PROMPT_CHARS=150K to automatically trigger segmentation; ② Using a greedy algorithm that groups by year and breaks down each paper, segmenting larger years; ③ Using the LLM to merge 47 data chunks into a complete report.

Key Insight: The key to large-scale text analysis lies in “divide and conquer + semantic merging”: Technically, dynamic segmentation can control the burden of a single LLM, while cognitively guiding the LLM to extract consistency across segments, rather than simply concatenating prompts. 150K characters correspond to about 15-20 medium-length papers, which is the optimal point empirically.

5. Task completion model training stalled at a loss of approximately 0.035, despite grad_norm=0.7-1.0 and a positive sample ratio of 4%, with negative samples accounting for 96%#

Solution: A positive sample weighting mechanism (pos_weight=10) was applied to the BCE loss, effectively increasing the positive sample recall rate from 4% to approximately 30%.

Key Insight: In cases of low loss, the recall rate of positive samples in an imbalanced dataset does not guarantee good performance. Focusing only on gradient magnitude is misleading, as negative samples dominate. Clear gradient balancing is required for category imbalance.

6. CalendarPro immediately returns “No events today” for queries regarding “this week” and “tomorrow” after creating events#

Solution: A three-tier solution was adopted: (1) Adding intent classification to BASE_SYSTEM_PROMPT (instead of just using INTENT_PROMPT_FRAGMENTS in multi_intent.py), (2) Adding keyword backoff detection in handle_query(), (3) Fixing get_today_events() to display already scheduled events in get_today_events(), (4) Adding Pending_warnings for Discord error feedback.

Key Insight: INTENT_PROMPT_FRAGMENTS and BASE_SYSTEM_PROMPT serve different code paths—the main LLM classification uses BASE_SYSTEM_PROMPT. Keyword backoff is a reliable LLM safety net. Past events must remain visible to prevent re-creation.

7. The codebase contains 8 fully implemented modules (approximately 2400 lines of code), which have comprehensive tests, but never imported during runtime, causing maintenance difficulties#

Solution: A systematic four-stage refactoring process was adopted: (1) Listing all imported modules and identifying isolated modules; (2) Archiving unused modules and documenting their purpose in the README file; (3) Merging duplicate modules into the specifications; (4) Deleting truly useless code. After 373 tests, all modules passed.

Key Insight: For fully implemented modules, archiving rather than deleting them is preferred. Isolated but still usable code should be archived and accompanied by relevant context information for future integration. Audit findings indicated deviations in configuration and errors in import statements.

9. Deadlock caused by the finalization mechanism of the widget: Historical log files marked with [[[HYMTPH_3]] exist indefinitely, preventing [[[HYMTPH_4]] from ever becoming True. As a result, [[[HYMTPH_5]] continuously calls the LLM merge function, wasting API costs.#

Solution: Identified the root cause: The two-level finalization mechanism (logs and reports) requires all source logs to be finalized before marking the report as completed. Only when [[[HYMTPH_6]] occurs can the export setting [[[HYMTPH_7]] be applied. The reason historical logs remain in the False state is: (a) When [[[HYMTPH_8]] occurs, only one export is performed, thus keeping it in the False state; (b) After the first export, the number of conversations changes, leaving it in the False state. The [[[HYMTPH_9]] flag is embedded in the JSON content and synchronized using rclone. Solution: Re-run the export operation on the affected device, as now the date has passed and the number of conversations has stabilized, so it will automatically be marked as True. Key Insight: The two-level finalization mechanism requires all source data to be finalized before marking the report as completed. The [[[HYMTPH_11]] flag in the JSON content is synchronized using rclone, ensuring consistency across devices. However, historical False entries can lead to permanent deadlocks. The key value is to save LLM API costs (during the merging phase), not the download time of rclone. Re-exporting through “data repair” can better adapt to the existing architecture without requiring “logical changes” (removing all all_sources_finalized checks would break consistency.)

10. When using a block size of 60K for the first time, 100 papers were split into 102 blocks, with almost one paper per block. This led to a large number of LLM calls, and each paper lacked context information.#

Solution: Increase the block size from 60K to 150K and adjust the splitting logic to prioritize the integrity of years. Eventually, reduce the number of blocks from 102 to 47, significantly reducing the number of LLM calls while maintaining temporal context.

Key Insight: The granularity of splitting needs to be balanced between “LLM processing capacity limitations” and “semantic coherence”: too fine a splitting will lose temporal relationships, while too coarse a splitting will cause timeout issues. 150K roughly corresponds to 15-20 medium-length papers, which is the optimal point based on experience.

Confusion of researchers with the same name in academic databases: Semantic Scholar merges multiple researchers with the same name into a single profile, leading to misleading trajectory analysis (for example, Yuke Zhu involves medical imaging, computer vision, and astrophysics—clearly different individuals).#

Solution: The AI system actively identifies confusion signals in the analysis: excessive domain spans, fragmented co-author networks, inconsistencies between citation counts and paper numbers. Warnings are marked at the beginning of the trajectory summary, suggesting manual splitting or analysis at the paper level. For Yuke Zhu, he is distinguished from robot researchers at UT Austin/NVIDIA, researchers at Mybank Technology, mathematicians at Zhejiang University, and researchers at Ant Group based on institutional affiliations and research domains.

Key Insight: Automated researcher analysis systems must include multi-dimensional signal detection modules: co-author clustering, topic clustering, temporal continuity, and consistency of institutional affiliations. Relying solely on the database’s identity recognition capabilities is not sufficient.

13. Multiple analysis failures due to LLM timeouts (the initial 300-second timeout was insufficient for a 187K merge prompt, and 420 seconds was still insufficient).#

Solution: A dynamic timeout formula was designed: [[[HYMTPH_14]]), with 300 seconds as the baseline, and an additional 120 seconds for every 50K characters added. The final merge step was successfully completed within about 660 seconds of timeout.

Key Insight: The timeout design should be linearly related to the data size, rather than being a fixed value. Large models have a relatively stable processing speed (about 200-300 characters per second), so it is a reliable strategy to estimate time based on the prompt length plus 50% buffer time.

14. There are systematic issues with escaping double quotes in the long JSON text fields generated by the LLM, resulting in JSON parsing failures (more than 12 errors).#

Solution: Identify and manually escape special symbols in fields like trajectory_summary and breakthrough[].description (e.g., “let the robot…” → “let the robot…”, single quotes in technical terms). For truncated JSON objects, add missing right parentheses and array closing symbols based on context.

Key Insight: When the LLM generates long JSON fields, the handling of embedded quotes is unstable, especially in academic texts containing frequently cited terms (e.g., “image understanding” becomes “viewing while moving”). This leads to formatting errors. Solutions require: (1) explicitly requiring escape of special characters in prompts; (2) setting length limits for overly long text fields, or using multi-round segmentation; (3) adding JSON schema validation steps after generation to automatically fix common formatting errors.

15. Long JSON outputs generated by the LLM contain format errors: truncation (especially in the last few fields), unclosed quotes, unescaped Chinese quotes (""), and missing closing brackets in nested structures, especially when containing technical descriptions, Chinese text, and deep nesting (greater than 3 levels).#

Solution: A four-step repair process was implemented: (1) attempting json.loads directly; (2) extracting json code blocks; (3) repairing unclosed structures based on depth-based bracket matching; (4) fixing complex errors through model upgrades, from haiku to sonnet to epic poetry. Special JSON repair prompts instruct the AI to extract and repair JSON, returning only pure JSON without explanations. All failures are recorded in outputs/logs/json_repair_failed/ for schema analysis and future optimization.

Key Insight: Long JSON files generated by the LLM show systematic failure patterns across different sessions, even with the same prompts (e.g., string escape issues, truncation issues, nesting errors). A single repair strategy is insufficient to address these issues; hierarchical fault tolerance mechanisms are needed. Chinese quotes "" are semantically valid, but they must be escaped to ASCII double quotes in JSON string values. Production systems need to validate generated JSON and support automatic repair retries.

16. For highly productive researchers (130 or more papers), analyzing the entire corpus at once results in prompts that are too long or lack depth. The analysis of Yuke Zhu’s 206 papers failed due to “too long prompts.”#

Solution: An iterative analysis strategy based on year segmentation was adopted: papers were grouped by publication year, and 1-3 representative papers were analyzed each round to uncover their breakthroughs (descriptions, why previous methods weren’t used, impact, technical insights). The complexity of each call was controlled while maintaining analysis depth. Paper selection: year-based grouping + citation count ranking + conference award identification (best paper/top pick/key point/identified by LLM oral presentation), with a maximum of 10 papers per year.

Key Insight: Year-based segmentation analysis is better than analyzing the entire corpus at once (too long prompts/insufficient analysis) or randomly sampling (missing temporal evolution). It maintains temporal consistency while focusing on paper quality. The “few but deep” iterative approach is more suitable for academic analysis than the “many but shallow” one.

17. Reusing the VLM prefix forward pass is required for completing the head reasoning (once in sample_actions, once in predict_task_completion).#

Solution: The sample_actions_with_completion() function was created, which captures the result of the first VLM forward pass and reuses it for completing the head, returning a tuple of actions and completion_probability.

Key Insight: Stream matching denoising is cost-effective, but unnecessary VLM forward passes increase measurable overhead. The unified method eliminates redundancy with minimal restructuring.

18. Despite a normal physical connection, the SpaceMouse device still couldn’t be opened because of the exclusive driver lock.#

Solution: The 3Dconnexion driver process was terminated, and the macros_private.py file in robosuite was modified to set SPACEMOUSE_PRODUCT_ID to 0xc63a, allowing access to the physical device.

Key Insight: macOS systems can exclusively lock HID devices. The macros_private.py file overrides the mode instead of modifying the main configuration.

19. Daily report merging task: How to avoid copying all tasks individually and instead create a comprehensive view.#

Solution: Specific operations included: (1) combining similar tasks into one entry with the latest status; (2) integrating human-machine analysis for the same topics; (3) combining multiple sessions for the same project into one conversation summary; (4) retaining content of higher importance during deduplication. This requires the LLM to understand the causal relationships between tasks (e.g., “implementing a feature” → “fixing an error” → “verifying the fix”) as well as topic similarity (repeated debugging of the same module should be treated as one task).

Key Insight: Direct concatenation leads to information redundancy. Synthesis requires identifying causal relationships between tasks (sequential stages or truly parallel situations) as well as topic similarity (repeated debugging of the same module should be treated as one task). This is not just string matching.

  • Misunderstanding of domain terminology: “By title” and “By summary” may be interpreted as choices based on LLM logic, while the user’s actual intention is to “only summarize titles” rather than “summarize summaries”. This needs to be clearly stated.
  • When dealing with a large amount of historical daily report data, it is not possible to effectively track the list of dates that have been processed and the tasks pending. Users need to provide a clear task list and a mechanism for verification step by step.

Lessons Learned#

Key Lessons#

  • Batch correction methods like Harmony can achieve global distribution mixing, but they cannot maintain semantic alignment when embeddings generated from independent training have inconsistent dimensional meanings. This requires Joint STAIG or a base model. Batch mixing and semantic alignment are independent attributes.
  • The enhancement style of MimicGen is essentially about coordinate transformation, not noise injection. It is necessary to apply pos_delta to the action trajectory (the offset from the source object to the target object). Without this, the enhancement effect of 10→1000 will fail because the success rate is less than 5%.
  • Super-large text analysis engineering paradigm: ① Automatic segmentation (optimal granularity: 150,000 characters per paper) ② Semantic merging (merging driven by LLM instead of concatenation) ③ Dynamic timeout mechanism (linear scaling based on prompt length, baseline: 300 seconds, increasing by 120 seconds every 50,000 characters) ④ Caching mechanism (to avoid retry costs). These four elements are key to the availability of deep-wide mode.
  • Multi-angle verification method for academic trajectory analysis: By iterating through different subsets of papers by the same researcher (from 1 paper to 100 papers), a three-dimensional understanding can be built from the micro-level (in-depth analysis of individual representative works) to the macro-level (trajectory features description of the complete publication list). Zhu YuZhe’s case shows that the evolution of Visual Genome → Embodied Intelligence (AI2-THOR) → RoboSuite/RoboTurk ecosystem construction → LLM-driven Embodied Intelligence (MineDojo/Voyager) → Geometric intermediate representation and VLM alignment clearly follows a trajectory. Only with a complete list of papers can this evolution process be fully demonstrated.
  • Zhu YuZhe’s research trajectory shows a clear evolution from visual reinforcement learning at the beginning of 2012 to the DreamDojo world model in 2026. The core proposition remains “enabling robots to efficiently learn from data and generalize to the real world”. This represents a development process from “data infrastructure builder” to “base model pioneer” and then to “world model architect”.
  • Single-intent classification is an architectural constraint that cannot be solved through keyword techniques. CalendarPro’s 7-layer keyword coverage only addresses the symptom of lack of abstraction. Multi-intent LLM analysis eliminates the need for all heuristics.
  • In binary classification, explicit gradient balancing is required when category imbalance occurs. When 96% of samples belong to the same category, low overall loss may mask the learning problems of minority categories. pos_weight can effectively rebalance the gradient contributions.
  • Key dimensions of analysis pattern design: ① Data size (papers/year) ② Analysis depth (summary vs full text) ③ Computing cost (number of LLM calls). The five-pattern system combines two boolean dimensions with one numerical dimension, covering a complete range from “quick overview” to “in-depth exploration”.
  • Basic elements of structured data generation prompts: For fixed-pattern JSON outputs, all required fields, their types (string/array/object), length limits, default values (placeholder text if no data), and special character escape rules must be explicitly stated in the prompt. The evolution of the daily overview hierarchical format of the gadget summarize tool proves that relying solely on example reasoning cannot guarantee output consistency. Enumerative specifications must be used to clarify the generation logic for each field.
  • Path of research infrastructure construction in the field of robot learning: Zhu YuZhe’s academic trajectory reveals an important pattern—breakthroughs in the field often come with work in infrastructure classes (such as AI2-THOR simulation environment, robosuite standard benchmark tests, RoboTurk crowdsourcing platform, MineDojo internet-scale dataset). Although the number of citations for these works does not exceed the number of algorithm innovation papers, their actual impact on the development of the entire field may be greater. For any emerging research direction, prioritizing the establishment of replicable evaluation benchmarks and data collection toolchains is more strategically valuable than publishing single-item algorithm papers.
  • The two-tier finalization mechanism (logs + reports) requires all source logs to be finalized before marking the completion of the report. The _finalized flag in the JSON content is synchronized using rclone to ensure consistency across devices. However, historical error entries can lead to permanent deadlocks. The core value lies in saving LLM API costs during the merging phase, rather than rclone download time. The solution is to re-execute the identity operation (export) to automatically correct historical metadata, while respecting the architectural design.
  • The batch iteration workflow (processing daily reports daily and conducting academic analysis annually) is more reliable than a one-time batch processing. This can effectively avoid issues such as JSON truncation and insufficient analysis depth. For the gadget tool, adding an automatic retry mechanism and JSON validation is very crucial.
  • The problem of ambiguity in names among researchers is a high-frequency and significant quality issue in automated researcher profiling. It is necessary to establish a module for detecting name conflicts through multi-dimensional signals: co-author clustering, topic clustering, temporal continuity, and consistency of institutional affiliations.
  • In the machine learning process, the separation of training/validation data is non-negotiable. The validation set must never leak into the training set. Different data purposes require different generation quotas and strategies.
  • Even if orphaning is achieved, the complete and tested implementations should be archived and deleted. The archive/pattern combined with the README can retain optionsality (for future integration), while clearing the active tree structure.
  • The default values in the data processing workflow must match the semantics of each stage: export (missing = incomplete → False) vs merge (missing = stable → True). Minor inconsistencies may lead to silent failures.
  • The parsing order of date-time fields is crucial—ISO format fields (start_iso) should always be processed first, rather than display format strings (start_time ‘08:00), to avoid implicit parsing failures that could break existence checks.
  • The daily report generation tool for gadget projects demonstrates strong information extraction and structuring capabilities in complex multi-device, multi-project, cross-time scenarios. It can accurately distinguish between parallel tasks across devices, identify task causal relationships, synthesize dialogue summaries, and verify production readiness.
  • The research analyzer can continuously identify Zhu YuZhe’s core research directions and methodological evolutions in different subsets of papers over years, proving the tool’s ability to extract academic trajectories.
  • Voyager (2023) demonstrates the paradigm of LLMs as autonomous agents: In Minecraft, the number of unique items increased by 3.3 times, and the unlocking speed of the technical tree improved by 15.3 times. Skills are stored in the form of executable JS code, rather than neural weights, avoiding catastrophic forgetting.
  • Open X-Embodiment (2023): 22 robots, 21 institutions, and 160,000 tasks demonstrate positive transfer across embodiments, establishing an ImageNet-moment for robot data integration.
  • Hierarchical information density of researcher profiles: First layer = h-index/citation count (quantitative metrics), second layer = list of representative works (enumerateable facts), third layer = narrative of research trajectory (requires LLM to extract implicit evolutionary logic). These three layers correspond to output complexities of fast/normal/deep modes.
  • The fault tolerance mechanism for LLM-generated JSON outputs requires multi-layer processing: direct parsing → code block extraction → bracket matching → LLM repair (model upgrade). A single strategy cannot cover all failure modes. All failures should be recorded for manual review and further optimization.
  • The framework for analyzing breakthrough paper features (why not do this + impact + technical insights) can effectively reveal the historical inevitability of technological innovations, domain impacts, and core insights, which is more valuable than simple abstract summaries.
  • The selection strategy for analyzing researchers’ papers should balance temporal integrity and quality. A triple mechanism based on year grouping + citation count ranking + conference awards identified by LLM can balance foundational early work and latest frontier breakthroughs within controllable computing costs.
  • Pattern recognition of JSON system errors generated by LLMs: Through 12 or more JSON repair tasks, it was found that unescaped citation errors mainly concentrate on: (1) Academic term citations (e.g., “visual understanding”); (2) Mixed use of Chinese and English quotation marks; (3) Special symbols in technical terms (e.g., + in C++, mathematical symbols like σ_w²). Solutions require multi-layered protection: explicit requirements to follow escaping rules in prompts + output format verification + regular expressions to automatically fix common patterns.
  • The strategy of analyzing by year in layers is better than processing the entire corpus at once (prompt too long/analysis too shallow) or random sampling (missing temporal evolution). This ensures both the integrity of the temporal dimension and the quality of the papers. The “few but deep” iterative pattern is more suitable for academic analysis than the “many but shallow” one-time pattern.
  • When implementing the event scheduler, it is necessary to include events that have already started within the existence check (include_past=True). Otherwise, events from the past become invisible, and each cycle recreates them.
  • In multi-component systems, it is necessary to understand the actual location where the prompt is used. INTENT_PROMT_FRAGMENTS and BASE_SYSTEM_PROMT serve different code paths—the main LLM classification uses BASE_SYSTEM_PROMT, while multi_intent.py uses INTENT_PROMT_FRAGMENTS.
  • The keyword backtracking mechanism is a reliable LLM safety net—even if the LLM cannot return structured data, it can correctly route through detecting keywords in the original input.
  • Project renaming requires a two-tier approach: (1) Code/docs (automated through search and replace), (2) Infrastructure (manual operations: conda, git, hosting, services). Checking these two layers in advance helps avoid partial migration.
  • The “distillation” of daily reports requires the LLM to understand the causal and inclusion relationships of tasks. Treating “implementing functions” → “fixing errors” → “verifying fixes” as different stages of the same work thread, rather than three separate tasks. This ability to understand semantic relationships is key to report readability.
  • pip install -e . (editable mode) is the correct way to import the shared toolkit in multi-tool projects, rather than operating through PYTHONPATH.
  • For Python with a state system, hot reloading is almost impossible (e.g., Discord bots have event loops, singleton caches, background schedulers). A feasible solution is to restart the process level through watchfile.

Practical Learning#

  • robosuite provides official tools (input2action, inputUtils) that can correctly handle the conversion from devices to actions. Manually parsing action_dict will lose the coordinate transformation and controller scaling.
  • Testing infrastructure hardening (pytest-timeout, @pytest.mark.external, skipping logic) can prevent certain unstable tests from causing the entire suite to crash. This pattern can be reused in multiple projects.
  • Using max_per_year=0 as a sentinel value for “infinite papers” is clearer than using a boolean flag or None, as zero is never a valid paper limit case.

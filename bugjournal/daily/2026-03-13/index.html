Daily Report — 2026-03-13#

Today’s Overview#

  • What was done: Added five per-frame variables such as operation progress and target pose in the RoboTwin simulation platform Place Dual Shoes task, and removed critical_region.
  • How it was done: Adopted a post-processing approach: obtained the end-state from the simulator after each move() execution, and retrieved the pickle file for that frame to avoid issues with knowing future states during online collection.
  • What’s useful: Provided richer supervisory signals such as operation progress and target end pose for VLA model training, thereby improving dataset quality and model learning capabilities.

Added five new per-frame data variables for the RoboTwin Place Dual Shoes robot task, and fixed two data collection quality bugs.

Today’s Tasks#

Architecture and Strategy#

  • ✅ Design a new variable data collection architecture — After analyzing the code library, determined the post-processing approach: since target_endpose/target_joint requires the end-state at the end of move(), we used the method of retrieving the patch pickle file after each move() execution. The general recursive design of pkl2hdf5.py does not need to be modified.

Implementation and Fixes#

  • ✅ Implement target_endpose and target_joint variables — Retrieved the end-state of both handenders’ poses and joint states from the simulator after each move(), and used these as the target state for that move, written into all corresponding frames.
  • ✅ Implement manip_progress_distance_left/right variables — Calculated the operation progress of both hands based on formula 1 - |current-final| / |start-final|, and clamped it to [0,1] to avoid out-of-range values caused by curved paths.
  • ✅ Implement manip_progress_time variable — Generated a linear interpolation time progression variable from 0 to 1 during each move(), setting it to 0 at the start of the move and 1 at the end.
  • ✅ Remove critical_region variable — Finally, explicitly pop(‘critical_region’, None) was used during the pickle patch phase to remove this field. Simply removing the subclass override was not enough because the base class get_obs()无条件 writes this field.

Problems and Solutions#

Key Problems#

1. Variables like target_endpose/target_joint require the end-state obtained at the end of move(), but this cannot be known during frame collection.#

Solution: Adopted a post-processing architecture: retrieved the end-state from the simulator after each move() execution, and retrieved the pickle file for all frames during that move.

Key Insight: Frame-level data collection and action execution are in a pipeline relationship. Variables that depend on future states must be post-processed rather than collected online. The general recursive design of pkl2hdf5.py is naturally compatible with the new keys and does not need to be modified.

General Problems#

2. After removing the subclass get_critical_region_label() override, the critical_region field still appears in HDF5.#

Solution: Explicitly called pkl_data.pop(‘critical_region’, None) during the pickle patch phase of move() to remove this field.

Key Insight: The base class _base_task.py’s get_obs() calls get_critical_region_label() and writes it unconditionally at line 510. Whether the subclass method is overridden or not does not affect the appearance of this field. It must be removed actively after data writing.

3. manip_progress_distance_left/right shows negative values in some frames.#

Solution: Used np.clip to force the calculation result to be within [0.0, 1.0].

Key Insight: When the robot end-effector moves along a curved path, the distance between intermediate frames and the final point may be greater than that between the starting frame and the final point, resulting in negative values in the progress formula. The linear progress indicator based on Euclidean distance has fundamental limitations when dealing with non-linear paths.

AI Limitations#

General Limitations#

  • Did not anticipate that the base class get_obs() would write critical_region unconditionally. Mistakenly thought that removing the subclass override would remove this field, resulting in a secondary fix.
  • Did not consider the boundary situation where intermediate frame distances exceed the starting distance due to curved paths when designing the manip_progress_distance calculation formula. Missing the [0,1] clamping treatment.

Today’s Gains#

Core Gains#

  • When data variables depend on the end-state after the action sequence is completed, post-processing (retrieving pickle files) is a more reliable architecture choice than online collection, as long as the downstream HDF5 converter is sufficiently versatile.

Practical Gains#

  • Before modifying data output, it is necessary to check whether the base class unconditionally calls/writes relevant fields. Simply overriding the subclass method may not be enough to prevent the field from appearing in the output.
  • The progress indicator calculated based on the ratio of Euclidean distances (1 - dist_current/dist_start) will produce out-of-range values under non-linear paths. It is necessary to explicitly clamp it to a valid range.

Problems and Solutions#

Key Problems#

1. When the haiku repair fails, JSON parsing crashes, causing original LLM response to be lost and debugging impossible.#

Solution: Implemented a recursive repair chain (haiku → tanka → opus), using try_parse_json helper function to avoid recursion, and saved failed responses to output/logs/json_repair_failed/.txt for debugging.

Key Insight: When using LLM-based repairs, it’s necessary to avoid infinite recursion. Separate pure parsing logic (_try_parse_json) from repair logic (parse_json_response), and retain the original output in case of complete failure.

2. ArXiv only returns the last 100 papers, and Semantic Scholar has 500 papers with citation data, but the merging logic discarded S2 papers that didn’t match ArXiv, resulting in loss of many historically influential works.#

Solution: Reversed the priority of data sources: S2 became the main source (with year/citation/conference/abstract), ArXiv was supplemented with arxiv_id/pdf_url/categories, then added isolated ArXiv papers. Result: comprehensive timeline coverage.

Key Insight: When merging datasets with different biases (ArXiv for the latest, S2 for comprehensiveness), it’s better to base on a more complete data source and supplement with information from professional sources, rather than the other way around.

3. Sorting papers only by citation count loses track of research trends over time, making it impossible to understand researchers’ focus changes across career phases.#

Solution: Implemented annual stratification: grouped by year, sorted within that year by award weight and citation count, selecting ≤10 works per year, and displaying them in chronological order. While preserving the narrative of work trajectories, highlighting the best works from each period.

Key Insight: For longitudinal analysis, temporal structure is usually more valuable than global rankings; stratified sampling can consider both dimensions.

4. The collected data contains negative progress_distance_left/right values and cases where it exceeds 1.#

Solution: Added np.clip(progress_dist, 0.0, 1.0) to limit left/right progress distances within valid [0,1] range.

Key Insight: Curved robot trajectories may cause the current EE position to temporarily exceed the distance from start to goal during interpolation, breaking the assumption of 1 - |current-final|/|start-final|. It’s necessary to set clear boundaries for progress metrics.

5. ArXiv and Semantic Scholar APIs cannot provide award data (best paper, highlight article, etc.), but this is crucial for identifying breakthrough achievements.#

Solution: Users suggested using LLM-trained data: sending paper titles and conference locations to the LLM to recall known award information. This step was implemented as a new analysis phase, with a separate caching mechanism.

Key Insight: LLMs can serve as auxiliary knowledge bases for handling data that cannot be obtained via APIs, especially authoritative but unstructured metadata like conference awards.

6. Even after removing the override method, critical_region still appears in the HDF5 file.#

Solution: Analyzed base class base_task.py:510 and found that get_obs() unconditionally writes pkl_dic[‘critical_region’]. Added post-processing steps to remove this key from the repaired pickle file.

Key Insight: Removing the override method only changes the return value; it’s necessary to trace the code path of the base class to ensure the key is truly removed from the output. Post-processing the pickle file before converting to HDF5 can effectively remove unnecessary keys.

Common Issues#

7. Log message ‘[S2] Get paper: 1736651’ is ambiguous—it seems to mean getting 173,600 papers, but actually refers to the author ID. Users immediately identified this problem.#

Solution: Modified to ‘[S2] Get author 1736651’s papers…’ and ‘[S2] Author 1736651: Retrieved 500 papers’. Clear subject-verb-object structure eliminates ambiguity.

Key Insight: Even when reading log messages without context, it’s necessary to ensure semantic clarity. Numbers should always have clear labels indicating their semantic type.

8. In detailed mode, only the 10 most cited papers are downloaded, but after annual stratification, there are already ≤10 papers per year—all papers should be analyzed in detailed mode.#

Solution: Removed filter ‘sorted(…, reverse=True)[:10]’, replaced with ‘papers_for_download = [p for p in papers if p.arxiv_id]’. Now, all filtered papers will be downloaded in full text.

Key Insight: When quality thresholds are already applied at the upstream stage, downstream shouldn’t re-filter using different criteria—this could lead to inconsistencies between modes.

Limitations of AI#

Key Limitations - Using a smaller dataset (ArXiv 100 latest) as the basis for merging, rather than a comprehensive dataset (S2 500 historical data), focusing more on algorithmic simplicity than semantic completeness#

  • Static code analysis is insufficient to track data flow: Claude removed the override method for get_critical_region_label(), but couldn’t track the unconditional writing behavior of the base class get_obs() to pkl_dic[‘critical_region’], requiring users to test at runtime to find this issue

  • Initially, the traditional data pipeline approach (API scraping, structured datasets) was used to obtain award information, ignoring the fact that LLMs already “know” the awards for famous conferences from training data

  • Implemented a single-stage LLM repair mechanism (only haiku), rather than a multi-stage upgrade mechanism, treating errors as a binary issue, not a progressive cost-benefit issue

  • Lack of physical reasoning for edge cases: Did not anticipate that curved robot trajectories might cause progress_distance to exceed [0,1] range, so did not consider geometric constraints when implementing the formula

  • Over-reliance on method inheritance logic: Assumed that removing the override method would remove the variable from the output, but did not verify all code paths that write to the data structure. Empirical testing is needed to uncover the unconditional writing behavior of the base class### Practical Learning Outcomes

  • ArXiv HTML rendering is now more suitable for full-text extraction than PDF: Faster, without OCR errors, and retains structure. For pre-Html papers, PDF can be used as an alternative. This indicates that the evolution of infrastructure requires regular review of “best practices.”

  • Numerical records without semantic labels can lead to ambiguity: “1736651” could refer to a count, ID, or size. Always use clear labels like “Author 1736651” or “1,736,651 papers.” The extra two words are less costly but prevent misinterpretation across categories.

  • For academic analysis tools, narrative sections prefer Chinese output, while technical terms prefer English: A dual-language approach: write the structure in English (breakthroughs, why not before, impact), and explain it in Chinese.

  • When there is selection logic upstream (yearly stratification → ≤10 years), downstream stages should not use different criteria for re-filtering: This can lead to inconsistent patterns (fast mode sees different papers than detailed mode).

  • Multi-agent exploration pattern: Use the Explore agent to understand the codebase, the Plan agent for design, and user verification for implementation. This clearly separates concerns and allows reviewing plans before execution.

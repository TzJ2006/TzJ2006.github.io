Problems and Solutions#

Key Problems#

1. [[[HYMTPH_18]] returns a function rather than direct results in the [[[HYMTPH_19]] library. The AI’s initial code used it as a direct result, leading to TypeScript type errors.#

Solution: Change [[[HYMTPH_20]] to [[[HYMTPH_21]]

Key Insight: [[[HYMTPH_22]] is a higher-order function that returns a reusable parser function. Calling this function results in [[[HYMTPH_23]]. This is a unique feature of this library compared to common Result monad implementations.

2. The local ccusage.json contains metadata keys like [[[HYMTPH_24]]. After parsing with valibot schema, it is treated as an empty pricing entry, causing the test assertion “Only entries containing token cost fields should be loaded” to fail.#

Solution: Add an additional check for [[[HYMTPH_25]] after schema parsing to filter out entries without pricing data.

Key Insight: The LiteLLM pricing JSON contains both description documents ([[[HYMTPH_26]]) and actual model entries. Business semantic filtering needs to be added outside the schema layer.

General Problems#

3. ESLint rules prohibit using the global [[[HYMTPH_27]] object directly. The AI used [[[HYMTPH_28]] in its new code, causing lint failures.#

Solution: Add [[[HYMTPH_29]] at the beginning of the file.

Key Insight: This project requires all Node.js built-in objects to be explicitly imported through the [[[HYMTPH_30]] protocol. This is a strict requirement of ESLint configuration.

4. The pnpm command does not exist in the ccusage environment. corepack cannot download pnpm due to network permission EPERM issues, resulting in failure to run format/typecheck/test.#

Solution: Users need to install pnpm themselves and then re-run [[[HYMTPH_31]] with elevated permissions. This successfully installed 930 dependencies.

Key Insight: The HPC cluster environment does not have write permissions for the [[[HYMTPH_32]] directory, and the npm registry has limited network access. Users need to install the package manager manually.

AI Limitations#

Important Limitations#

  • Misuse of the [[[HYMTPH_35]] library’s [[[HYMTPH_36]] API: Assuming its behavior is the same as common Result monad implementations (directly returning results), it actually returns a reusable parser function. Additional calls are needed to obtain the result, resulting in two rounds of fixes.
  • When implementing it for the first time, the AI didn’t consider that the LiteLLM pricing JSON contained non-pricing metadata keys like [[[HYMTPH_37]]. This led to test failures until business filtering logic for token cost fields was added later.

General Limitations#

  • The AI couldn’t predict the network and permission restrictions of the HPC cluster environment. Multiple attempts to install pnpm through corepack failed due to EPERM or network fetch issues, resulting in a lot of iterations before it became clear that users needed to solve this problem manually.

Today’s Gains#

Key Gains#

  • The [[[HYMTPH_38]]’s [[[HYMTPH_39]] returns a function (factory pattern), rather than directly executing and returning Result. This is different from common Rust/fp-ts-style Result implementations, so this needs special attention.
  • The offline pricing in ccusage is pre-filtered with the [[[HYMTPH_40]]’s [[[HYMTPH_41]] function during construction, retaining only Claude-related models. Extending support for multiple models requires同步修改这个宏和[[[HYMTPH_42]]。
  • The MIHD project plan is implemented in a strict dependency order: configuration basics (Idea 6 Phase 1) → normalization → UNI2+scGPT experiment → Q-Former/LLaVA → Niche query → batch correction → complete configuration integration. This order was designed to avoid accumulating refactoring costs.

Practical Gains#

  • When using Node.js projects on the HPC cluster, package managers (pnpm) and corepack may fail due to file system permissions or network isolation. It’s necessary to confirm the availability of the toolchain in advance.

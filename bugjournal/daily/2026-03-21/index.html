Issues and Solutions#

Key Issues#

1. Systematic Phase Detection Failure in Multi-Object Tasks Like threading/pick_place: All threading frames were marked as pre_reach, pick_place lacked reach/grasp stages, resulting in inability to find opportunities for 12/13 skills.#

Solution: Root cause: Phase detection relies on the distance between a single object and EEF, which is untenable in special geometries like needle-shaped needles or multi-object scenarios. Fix: Added InteractionSegmenter to segment trajectories based on object interactions (EEF proximity + gripper state + co-motion detection), with each segment having a clear target_object and phase, completely bypassing phase detection defects.

Key Insight: The single-object assumption in phase detection is a structural design flaw that cannot be fixed through parameter tuning; trajectory segmentation is a more fundamental solution, abstracting from the defect rather than repairing it.

2. Ambiguity of Target Objects in v5 Pipeline: OpportunityScanner/skillInject()/validate()/ContextReplayEngine default to objects[0], leading to systematic errors in multi-object scenarios; missing target_pos in trajectory_context makes drop-type skills’ filtering conditions ineffective.#

Solution: From Steps 1-7 onwards, pass through target_object and target_pos throughout the pipeline; fixed the semantics of objects[0] for 13 skills; wrote target_pos after constructing the trajectory context; used error_spec.target in context_replay.py.

Key Insight: In multi-object tasks, objects[0] is the first object in dictionary insertion order, with incorrect semantics that causes systematic errors in judgments based on object state (displacement/gripper/pose).

3. Systematic Mismatch between Injection Parameters and Verification Thresholds in Error Skills, resulting in 41+ subtypes generating 0 scenarios (5 root causes including insufficient displacement, empty grippers, objects not released, insufficient collision force).#

Solution: Identified 5 major root causes and made targeted modifications to 7 files in 24 locations (reducing verification thresholds, increasing steps, adjusting speed direction); applied a slight initial velocity (Z=-0.15m/s) to drop-type skills to assist release.

Key Insight: The lower limit of injection parameters must be matched with the verification threshold; for drop-type skills, it cannot rely solely on gravity, requiring an active initial velocity to overcome friction.

4. Collect script never called segment_interactions(), NPZ lacked segmentation data; in three_piece_assembly, InteractionSegmenter and get_target_object() mistakenly selected base fixtures as targets, causing scan regression from 23 to 4 subtypes.#

Solution: Fixed 5-6 to limit target search to graspable objects (with grasp_geoms configuration); collect script needs to call segment_interactions(); replay_and_label_phases() must pass in target_object; scanner needs to fix label coverage timing issues. (The fixture part has been fixed, but collection integration is pending.)

Key Insight: The segmenter was implemented, but data collection pipelines were not connected. Contract alignment must track and persist on the storage layer—unit tests passing do not guarantee E2E functionality.

5. python summarize/daily_summary.py permanently stuck in @ccusage/codex steps, with no output or errors.#

Solution: subprocess.run(capture_output=True) redirected stdin to DEVNULL; npx installation confirmation prompts were fixed by adding –yes parameter during the first installation; added –yes parameter to all 3 npx calls to skip interaction confirmation.

  • MuJoCo physical simulation (enable_camera=False) is a pure CPU task. The sbatch –wrap command uses /bin/sh by default, rather than the user’s logged-in shell. The number of workers should exactly match the specified –cpus-per-task.
  • General issues:
    • In e02 drop_in_transit, action[-1]=-1.0 indicates “Open gripper” in the comment, but actually it closes the gripper, causing dropped objects to be re-caught and resulting in a failed drop. In e11 validate(), the threshold min_progress * 30 is hardcoded, while inject() reads freeze_steps from config, leading to inconsistent judgment criteria.
    • Solution: Change e02 to action[-1]=1.0 (robosuite OSC 1.0 = open). Add freeze_steps = self.config.get(‘freeze_steps’, 30) in e11 to unify the source.
    • Robosuite gripper action symbols contradict intuition and comments (-1 means closed, 1 means open). The log shows drop metrics as 0, which can be misjudged as a physical execution failure rather than a control command error.

    • Solution: Add double-key fallbacks (first find the new name, then the old one) for reverse_steps. Change the handling of reverse_steps from fixed values to range sampling.
    • Key insight: The evolution of config keys does not update code synchronously. The getdefault mechanism conceals errors—config is ignored without an error message, but the program reports no errors. This is a highly concealed bug.

  • Today’s takeaways:
    • Pipeline contract alignment must track the entire data flow: not only should the code processing logic be modified, but also ensure that data is persisted at the storage level (e.g., segment_interactions() results written to NPZ). Otherwise, data will be lost when reloaded, and repairs will be ineffective for subsequent processes. Passing unit tests does not mean E2E correctness.
    • In multi-object robot operations tasks, target_object must be transmitted throughout the detector/injector/validator stages as a primary citizen. Objects[0] is the first object in dictionary insertion order. In multi-object scenarios, it is semantically incorrect and will not report errors, leading to systematic misjudgments based on object states.
    • Before analyzing complex system problems, all relevant configuration files must be read to verify basic assumptions (e.g., how many objects there are in the task, what type of task). “Configuration file is the map, code is the terrain. We need to look at the map first.”
    • After systematic repairs, before and after comparisons must be performed on all tasks. It cannot be assumed that there are no side effects just because unit tests pass. The simultaneous occurrence of threading +22 and three_piece_assembly -19 illustrates this point.
    • The lower limit of Error skill injection parameters must be adjusted in conjunction with the verification threshold. When reducing the injection lower limit, the verification threshold (min_displacement, min_rotation_deviation, etc.) must also be reduced accordingly. Otherwise, injection will still fail to pass verification. Drop-type errors cannot rely solely on gravity and must have a slight initial downward velocity (Z=-0.15m/s) to assist in releasing objects from friction.
    • subprocess.run(capture_output=True) will redirect stdin to DEVNULL. When calling CLI tools such as npx that may have interactive prompts, parameters like –yes/-y must be explicitly passed to skip the interaction. Otherwise, it will hang indefinitely instead of timing out.
    • Log aggregation statistics (Counter) is more effective at locating systematic bugs than reading each item individually: 1698 times gripper not closed directly points to target_object ambiguity, rather than real gripper execution problems. Data-driven root cause analysis is better than subjective guessing.
    • MuJoCo physical simulation (enable_camera=False) is a pure CPU task. MUJOCO_GL=disabled can completely bypass OpenGL. HPC shared nodes are not suitable for large-scale parallel subprocesses. Slurm exclusive nodes should be used, and the number of workers must exactly match –cpus-per-task. Slurm sbatch –wrap uses /bin/sh by default and needs to be explicitly wrapped with bash -c.
    • Graspable objects and fixtures must be distinguished in multi-object MuJoCo tasks. The grasp_geoms field in task_config can be used as a filter basis to avoid misjudging targets when EEF trajectories pass above the fixture. Other_objects still need to retain the fixture for collision detection, only filtering during target search.
    • Robosuite OSC controller gripper action symbols contradict intuition and comments (-1.0=closed, 1.0=open). The log shows drop-related metrics as 0, which can be misjudged as a physical execution failure rather than a control command error. The naming of robosuite MuJoCo bodies does not match that of task config, requiring fuzzy matching fallbacks.

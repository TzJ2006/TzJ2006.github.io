Problems & Solutions#

Critical Issues#

1. Visualization videos showed no visible error effects despite logged force application and body displacement#

Solution: Root cause: Demo actions during injection caused OSC controller (kp=150) to compensate external force; fix: use neutral actions (zero vector) during injection+settling phases, allowing force to accumulate without controller interference

Key Insight: OSC position controller actively resists external forces when tracking a trajectory; must use neutral actions to observe disturbance effects. Validation phase (drop.py) was correct all along with neutral actions.

2. Visualization video shows robotic arm barely moves, but logs show force applied and body position changed (15mm-117mm)#

Solution: First fix: re-apply force every step during duration window (mimic drop.py), solving MuJoCo’s per-step xfrc_applied clearing. Verification confirmed force re-application (Test 1: 11 times, Test 3: 21 times). Discovered deeper issue: visualization uses demo actions, OSC controller (kp=150) compensates external force effect

Key Insight: Layer 1 problem: MuJoCo physics engine auto-clears xfrc_applied after each physics step (prevent accumulation), must re-apply every step. Layer 2 problem: validation uses neutral actions (zero vector), controller only maintains position; visualization uses demo actions, controller tracks demo trajectory at every substep (20 substeps), generating corrective torques far exceeding external force. True fix requires using neutral actions during force window + settling period

3. Q-Former process crashed/died after completing 6/11 sections, potentially losing hours of GPU compute work#

Solution: Pipeline’s checkpoint recovery mechanism (checking for existing *_embeddings.npz files) automatically detected completed sections and resumed from section 151672, skipping already-finished work

Key Insight: 断点续跑 (checkpoint/resume) mechanism worked as designed; run_pipeline.py’s is_completed() check prevented re-running expensive experiments, validating the recovery architecture from original implementation plan

4. JSON file writes interrupted by Ctrl+C/crash/power loss lead to half-written invalid JSON, causing parse failures on next read#

Solution: Implemented _atomic_write() function: write to temp file (mkstemp), flush, os.replace for atomic rename, cleanup temp on failure. Ensures either old complete file or new complete file exists, never half-written state

Key Insight: Atomic write is standard pattern for preventing file corruption: write temp → rename atomically. Applies to all JSON/config file persistence scenarios

5. Multi-device workflow requires manually copying log files from each device to one machine for merge, tedious operations#

Solution: Export uploads to /logs/, merge uploads to /reports/, added –sync flag to auto-download all device logs from remote (filter with –include ‘{date}_*.json’), dedupe paths then call API to generate report

Key Insight: rclone subdirectory isolation + –sync flag simplifies multi-device collaboration from ‘manual copying’ to ‘one command’, significantly reducing mental burden

6. scgpt_uni2_qformer experiments failed with CUDA out of memory (11/12 failures), tried to allocate 2 MiB but GPU only had 2.31 MiB free, process already using 21.46 GiB#

Solution: Identified root cause: pca_uni2_qformer was monopolizing GPU (21.5 GB), leaving insufficient memory for parallel scgpt_uni2_qformer runs; determined scgpt Q-Former experiments must wait until pca Q-Former completes to avoid GPU contention

Key Insight: Q-Former training requires ~22GB GPU memory; running two Q-Former experiments in parallel on 32GB GPU leads to OOM; sequential execution is required despite being technically independent tasks

7. LLM occasionally returns invalid JSON (markdown code blocks, extra text), causing json.loads to fail#

Solution: For Anthropic use tool_use to force structured output (define submit_report tool schema + tool_choice), for OpenAI use response_format: json_object, keep 3-layer fallback parsing logic as degradation path

Key Insight: Cloud LLM APIs now support structured output; should prioritize SDK features over fragile regex/brace-matching parsing

8. Same device running export –summarize multiple times on same day repeatedly calls API, wasting cost and time#

Solution: Added _merged_devices list in JSON to track already-summarized devices, read existing file on export to check if current device in list, skip API call if present and reuse existing summary

Key Insight: Implementing idempotency through persistent state avoids redundant operations; device_summary changed to {device_name: summary} format makes each device’s contribution clearly visible

9. EGL rendering assertion failed: MUJOCO_EGL_DEVICE_ID must be contained in CUDA_VISIBLE_DEVICES#

Solution: Fixed batch_visualize.py GPU assignment: set MUJOCO_EGL_DEVICE_ID=0 (always) and adjust CUDA_VISIBLE_DEVICES per process to match (e.g., CUDA_VISIBLE_DEVICES=3 → MUJOCO_EGL_DEVICE_ID=3 incorrect; should both use relative ID 0)

Key Insight: MuJoCo EGL uses relative GPU indexing within CUDA_VISIBLE_DEVICES, not absolute system GPU IDs

10. ProximityDetector and CollisionDetector only generate impulse type specs, leading to single-type scene library (all 30 scenes are impulse/tip_over)#

Solution: Modified both detectors’ detect() methods to replace _generate_impulse_specs() with _generate_mixed_specs(), generating impulse+pose_perturb+friction candidates simultaneously, configured max_scenes_per_demo increased to 10

Key Insight: Detector design pattern proposing multi-type specs (GraspPrecondDetector already has this) needs to be extended to all frequently-triggered detectors to avoid single type consuming quota

11. One staig_fusion failure (scgpt_uni_staig_fusion/151676) due to ‘Input X contains NaN’ error, same section also failed for pca_uni_staig_fusion#

Solution: Identified as section-specific data issue (151676) causing model collapse (zero-variance embeddings after training), not a code bug; same section consistently fails across different gene encoders (pca, scgpt), indicating fundamental data quality issue

Key Insight: When same section fails across multiple configurations with NaN/zero-variance symptoms, it’s likely a data issue (missing spatial coordinates, corrupted expression values) rather than code bug; STAIG model collapse is a known failure mode for problematic inputs

12. CalendarPro had 8 instances of duplicate AI provider selection logic scattered across different files (discord_bot: 4, api_routes: 2, executor: 1, planner: 1)#

Solution: Created src/ai/provider_selector.py with unified get_first_available_provider() and generate_with_fallback() functions, replaced all 8 instances with imports, consolidated error handling and timeout logic in one place

Key Insight: When same pattern repeats 8 times it must be extracted; beyond reducing duplication, unified error handling and timeout logic prevents inconsistent behavior; Single Source of Truth principle applies to logic patterns not just data

13. BCPolicy fallback bug: adapter assigned before start_episode() call, leaving broken state on model load failure#

Solution: Refactored _ensure_loaded() to create adapter in local variable, call start_episode(), only assign to self._adapter after both succeed; on exception, set _fallback=True and keep _adapter=None

Key Insight: Lazy loading must be transactional: all initialization steps must succeed before committing state; partial success leads to broken objects

14. collector.py’s _save_episode() had trajectory saving TODO unimplemented, BC policy loading logic also TODO#

Solution: _run_episode() changed to return (summary, trajectory) tuple, _save_episode() saves NPZ (actions/rewards/eef_pos/gripper/objects), 3_collect_data.py added BCPolicy loading logic (read checkpoint path from config.evaluation.models, device from policy_generation)

Key Insight: Trajectory data needs complete state preservation (not just actions/rewards, but also EEF position, gripper state, all object positions), NPZ compressed storage (~64KB each), controlled by config flag

15. Three duplicate _parse_time() implementations in energy_projection, intelligent_scheduler, and essential_scheduler (lines 137-144, 362-369, 251-258 respectively)#

Solution: Created src/core/time_utils.py providing parse_time() and parse_datetime_with_tz() functions, all three files changed to import and use shared implementation

Key Insight: Time parsing is infrastructure-level functionality that should have unified handling rather than per-module implementations; eliminates risk of inconsistent time parsing behavior across modules

Learnings#

Key Learnings#

  • OSC position controllers (kp=150) generate joint torques that far exceed typical external forces (15-20N), actively compensating disturbances when tracking trajectories. Neutral actions disable tracking, allowing force effects to accumulate.
  • Checkpoint/resume mechanisms validate their design when actual crashes occur; MIHD’s is_completed() file-existence check successfully prevented 6 sections (~9 hours of GPU work) from being re-run after process crash
  • Atomic write is standard pattern for preventing file corruption: tempfile + write + rename. Applies to all JSON/config file persistence scenarios
  • Multi-device collaboration tool design pattern: local generation → upload to cloud subdirectories → one-command download+merge, encapsulating complex workflow into single command via –sync flag
  • MuJoCo automatically zeros xfrc_applied after each physics step (0.002s default); must re-apply force every control step for sustained effect. Validation code does this correctly in 10-step loop.
  • Error scene generation pipeline is fully modular: detectors→injectors→validators all follow plugin pattern and can be reused unchanged for policy-based generation by just swapping action source (policy.predict vs demo_actions[i]).
  • Robosuite’s env.step() contains 20 physics substeps (default control_freq=20, each control step 0.04s = 20×0.002s physics steps). OSC controller tracks target pose at every substep (kp=150 stiffness), external force smaller than control torque will be compensated away
  • Error injection pipeline is already policy-agnostic at action level (EnvWrapper.step(action) doesn’t care about action source). Core of extending from demo injection to policy injection is just action source replacement (demo_actions[step] → policy.predict(obs)), detection-injection-validation pipeline reuses as-is
  • Writing plans to .md files and pasting in new sessions is effective way to preserve context across sessions, more robust than depending on tool call chains
  • Q-Former training requires ~22GB GPU memory; on 32GB GPU, parallel Q-Former experiments cause OOM despite being logically independent; memory-intensive experiments require sequential execution even when compute graph is independent
  • Cloud LLM API’s structured output features (Anthropic tool_use, OpenAI response_format) should be prioritized, can significantly reduce JSON parsing failure rate
  • Idempotency design: through _merged_devices list tracking processed state, skip completed operations on re-run, saving API cost
  • MUJOCO_EGL_DEVICE_ID uses relative indexing within CUDA_VISIBLE_DEVICES, not absolute system GPU IDs. For single-GPU processes, always set MUJOCO_EGL_DEVICE_ID=0 regardless of which physical GPU.
  • Lazy loading with fallback requires transactional initialization: create object in local scope, validate all initialization steps, only assign to instance variable after complete success. Partial success creates broken state.
  • Detector design should let high-frequency triggered detectors propose multi-type specs to avoid single type consuming max_scenes_per_demo quota. GraspPrecondDetector has correct design (impulse+pose_perturb+gripper_bias), needs extension to ProximityDetector and CollisionDetector
  • When same data section (e.g., 151676) consistently fails across multiple model configurations with NaN/zero-variance symptoms, root cause is likely data quality issue (missing coordinates, corrupted values) not code bug
  • Code duplication threshold: when same pattern repeats 8 times, it must be extracted into shared utility; beyond reducing repetition, centralized error handling and timeout logic prevents inconsistent behavior
  • Trajectory data saving should include complete state (actions/rewards + EEF position + gripper state + all object positions), NPZ compressed storage (~64KB/episode), controlled by config flag (evaluation.save_trajectories)
  • Infrastructure-level functionality (time parsing, provider selection) should have unified implementations rather than per-module copies; Single Source of Truth principle applies to logic patterns not just data definitions
  • BC policy loading needs to read checkpoint path and device from config, error messages should provide actionable fix guidance (tell user which file to set which field). Pi0/Pi0.5 and other foundation models currently have no robosuite fine-tuned checkpoint, need self fine-tune or integrate LIBERO environment

Practical Learnings#

  • rclone remote configuration requires two steps: install rclone + run rclone config to bind cloud account, rclone_remote in script just specifies ‘where to upload’
  • Hugo deployment commonly uses public/ subdirectory as git repo pushed to GitHub Pages, with source file repo and deployment repo separated as standard practice
  • External tool output formats may change with version/environment, defensive programming should check types and handle gracefully (e.g., ccusage returning list case)
  • Git submodule cleanup requires: git rm –cached , delete .gitmodules, clone fresh copy, remove .git directory, stage as regular files. Also need to update all documentation references and remove .git/modules/ if present.
  • Hugo front matter summary field can directly extract from report blockquote lines for content reuse

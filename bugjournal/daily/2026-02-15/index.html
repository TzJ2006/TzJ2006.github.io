Limitations of Artificial Intelligence#

Main Limitations ### Today’s Summary#

Key Points#

  • The OSC controller continuously applies joint torque during 10 physical substeps per env.step(), which is much greater than the typical external force injection (15-20N). This actually compensates for the force effect. Visual repair must use neutral actions during the force application period, allowing the controller to “maintain the current state” rather than tracking trajectories. Only in this way can the force effect be observed in the video. This is the root cause of differences in behavior during visualization and verification stages.
  • MuJoCo’s xfrc_applied is a temporary buffer that is automatically reset after each physical step. To apply continuous forces, it is necessary to call injector.apply() again at each step. Visual effects cannot be confirmed solely through numerical changes. The controller may immediately correct errors within the same physical substep.
  • Error scenario generation based on demonstration errors is highly efficient (100 scenarios/10 demonstrations), thanks to humans’ prior knowledge of “task-related trajectories.” Strategy deployment, especially random strategies, are less efficient, but their value lies in generating diverse errors that更接近 real deployment scenarios. These two methods complement each other and can balance quantity and quality.
  • VLA and spatial genomics are developing towards the same architectural innovations (diffusion transformers/modal markers/comparing alignments). What is worth transferring to MIHD are flow matching fusion (continuous spatial generation embeddings) and register tokens+AdaLN (unified model across parts).
  • Atomic writing (writing temporary files → os.replace renaming) is a standard method to prevent file corruption. The core three-step process of multi-device cloud synchronization is “uploading to the corresponding sub-directory + unified download entry point + automatically pulling during merging.” These three steps are essential.
  • The diversity of benchmark error types depends on the “proposal diversity” of the detector, not the type of injector. If the detector only proposes pulse specifications, the verifier will never produce other types of errors. The problem lies at the detector level. The PolicyAdapter pattern uses the same process to support any strategy, such as random/robot simulation/VLA.
  • Error detection has a two-tier architecture: a rule classifier (main layer, based on state thresholds, fast and definitive) and a VLM layer (auxiliary layer, generating natural language descriptions and cross-validation after video rendering). The former covers the observable state space, while the latter handles advanced semantics and fuzzy boundaries.
  • Idempotency design: Embedting processed device lists in the exported JSON file (_merged_devices), allowing –summarize to support repeated executions without wasting API calls. Placing the _source_device field at the end of the JSON is the best balance between readability and compatibility (the tail is for quick viewing, merging does not read it, Python 3.7+ dictionaries maintain insertion order).
  • Submodule migration integrity check: File system separation (deleting.git files) is just the first step. It is also necessary to use Grep to remove all hard-coded URL references in the entire library—the lifecycle of code references is much longer than git tracking.

Practical Summary#

  • Correct lazy loading mode: Use local variables to temporarily store objects, only assigning values after the entire initialization chain (init + start_episode/load) is successful. This prevents half-initialization errors, and in case of failure, it will use _fallback to restore to a safe default value.
  • The most reliable way to retain plans: Write Chinese plan documents into.md files (containing nested large/medium/small targets), and paste the plan text into a new session for implementation. This approach is more persistent than relying on tool call chains, allowing users to view and modify plans at any time.
  • After remapping CUDA_VISIBLE_DEVICES to update the number of GPUs, MuJoCo EGL should use the mapped virtual numbers (always 0), rather than the physical GPU numbers. The output format of external tools requires defensive type checking; do not assume a fixed structure, especially between different platforms or versions.

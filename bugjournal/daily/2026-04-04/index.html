Problems & Solutions#

Key Problems#

1. FloatBall window locked at 400x400 on Linux, Tauri set_size() completely ineffective#

Solution: GTK bottom-up size negotiation takes max(window_requested, widget_natural_size); WebKitGTK natural size ≈200x200 causes the window to lock. Fix: call widget.set_size_request() on the GTK main thread via with_webview(), remove set_resizable toggling (since with_webview is async dispatch, set_resizable(false) queued before the callback executes would lock the old size), keep Linux permanently resizable.

Key insight: Tauri’s set_size/set_min_size/set_max_size operate at the GTK window hint level; GTK bottom-up negotiation ignores these hints to satisfy widget natural size. with_webview() returns before its callback executes, and subsequent Tauri API calls are still enqueued FIFO and may deadlock the old size.

2. Linux collapse visual jump: two-phase sequence exposes intermediate state + WM clamping to old width causes systematic coordinate offset#

Solution: Two-phase sequence: macOS has a 250ms animation buffer; after disabling transitions on Linux the intermediate state is jarring. Changed to: Linux path first shrinks the window atomically in Rust to 56px, then sets CSS expanded=false (at which point 100% = 56px, making the change invisible). WM clamp issue: GTK gravity has no effect in this WM (and actually introduces systematic offset); probe + corrective reapply produces double-frame flicker. The fundamental solution is a fixed large-size window + internal CSS (pending implementation).

Key insight: Disabling CSS transitions requires simultaneously adjusting corresponding JS/Rust timing logic; GTK gravity is not supported by all WMs; post-hoc correction is itself a source of visible transition frames; the root solution is to have the WM accept correct coordinates in a single shot.

3. Life-Copilot MCP Server three-layer breakage: Calendar token present but agent still reports connection failure#

Solution: Three simultaneous breakages: FastMCP was instantiated but run() was never called; .claude/mcp_servers.json was missing; butler instructions (including MCP tools list) were never passed to the Claude CLI subprocess. Fixed by creating __main__.py to launch stdio transport, creating mcp_servers.json, and adding --append-system-prompt to ClaudeAdapter. All 19 MCP tools restored.

Key insight: “Code was written” and “code is being called” are two different things; creating an object with a factory pattern does not start a service; external programs need config files to discover services; app-level token validation and CLI tool context injection are two independent layers — passing token validation does not mean the CLI tool received the correct context.

4. Linux Screen Time Tracker initial approach used xdotool, which does not support Wayland (user’s machine runs GNOME 50 Wayland)#

Solution: Feasibility probe confirmed Wayland environment; switched to GNOME Introspect D-Bus (org.gnome.Shell.Introspect.GetWindows()) as the primary path, xprop as XWayland fallback, PID→cmdline as final fallback. All subprocess calls have 5-second timeout + proc.kill().

Key insight: GNOME 41+ provides a standard Introspect D-Bus interface, the correct way to get the active window under Wayland. This issue was proactively discovered by the Critic subagent during adversarial review — missing it would have caused the feature to be completely non-functional on the user’s machine.

5. AionUI cannot connect to a remote headless server’s Claude Code CLI via SSH or any non-URL method#

Solution: AionUI’s remote Agent architecture assumes the agent exposes an HTTP port; CLI processes have no HTTP endpoint, making them fundamentally incompatible. Recommended tmux/screen for managing multiple sessions as a temporary substitute, and filing a feature request with the AionUI team.

Key insight: AionUI’s database already has a remote_agents table (with an allow_insecure field), indicating the development team is building remote Agent capabilities; SSH direct connection may be on the future roadmap.

6. X11 WM refuses to position windows partially off-screen, causing snap and collapse position drift#

Solution: Mutter/GNOME and other X11 WMs clamp undecorated windows within the visible area, rejecting positions where x < 0 or x > screen_width - window_width. Linux uses edge_offset=0; macOS/Windows use edge_offset=ball_size/2.

Key insight: macOS and Windows allow undecorated windows to partially extend beyond screen boundaries; X11 WMs generally do not. The two platforms require different edge offset values.

7. ClaudeCLIAdapter.is_available() missing async causes Claude to silently fall back to Codex and time out#

Solution: Awaiting a non-async method produces no compile error; the runtime TypeError is swallowed by except Exception, causing Claude to be deemed unavailable. Adding the async keyword restored Claude available: True.

Key insight: In Python, awaiting a synchronous method produces a TypeError. If silently caught by a broad except Exception, it causes the logic to return an incorrect False result, manifesting as feature degradation rather than an error — extremely difficult to debug.

8. Bot cannot understand cross-turn references in multi-turn conversations (DualVerifier parameter wiring broken)#

Solution: DualVerifier, SemanticRouter, and KeywordScorer all implement last_assistant_message and last_intent parameters, but _process_input() never passed them in its calls. Fixing the wiring required no changes to any classification logic.

Key insight: “A feature has an implemented parameter interface” does not mean “the parameter is being passed.” The integration layer is the most common wiring breakage point; validation should check the call site, not just the implementation.

9. Codex CLI session resume has a bug; three CLI session mechanisms differ too much to unify#

Solution: Abandoned reliance on native CLI sessions. Maintain a lightweight history ring buffer (last 20 messages) at the Adapter layer; inject history as a prompt prefix when switching CLIs, achieving transparent cross-CLI conversation history transfer.

Key insight: A unified abstraction layer (history injection) is more reliable than adapting to each CLI’s session mechanism; session mechanism inconsistency is the primary risk in a multi-CLI architecture.

10. lark-oapi WebSocket client runs in a blocking manner, incompatible with the FastAPI asyncio main loop#

Solution: Start the Feishu bot in a separate daemon thread; message callbacks bridge to the main event loop via asyncio.run_coroutine_threadsafe().

Key insight: lark-oapi v1.5.3’s Client.start() is fundamentally synchronous and blocking — it must be thread-isolated rather than directly awaited in an asyncio task. This issue was proactively discovered by the Critic agent during adversarial review.

11. WebKitGTK internal ScrolledWindow scrollbar cannot be suppressed with CSS#

Solution: The WebKitGTK scrollbar belongs to the gtk::ScrolledWindow layer, not the WebKit render tree; CSS pseudo-elements cannot reach it. In the with_webview callback, traverse the widget ancestor chain and call set_policy(Never, Never) on each ScrolledWindow.

Key insight: WebKitGTK’s UI chrome belongs to the GTK widget layer and cannot be targeted by CSS selectors; it must be handled at the GTK code level.

12. OpenReview API returns 403 for all unauthenticated access, NeurIPS D&B Track reviewer comment retrieval completely fails#

Solution: Identified two compounding issues: OpenReview recently tightened anonymous access policy (account authentication required); the NeurIPS D&B Track uses a separate venue ID rather than a sub-path of the main conference. Together they cause complete failure. Currently blocked, requires OPENREVIEW_USERNAME/OPENREVIEW_PASSWORD environment variables.

Key insight: In the OpenReview venue ID system, satellite tracks use independent venues rather than parent conference paths, requiring separate ID confirmation; the API policy has been broadly tightened recently.

Today’s Takeaways#

Core Takeaways#

  • Full picture of GTK window size control: GTK bottom-up negotiation takes max(window_requested, widget_natural_size); WebKitGTK natural size ≈200x200 logical — the breakthrough requires calling widget.set_size_request(); Tauri’s set_size/set_min_size/set_max_size are ineffective for WebKitGTK windows; with_webview() returns before its callback executes, subsequent Tauri API calls may deadlock the old size under FIFO ordering; WebKitGTK scrollbars belong to the GTK widget layer and must be disabled with set_policy(Never, Never) at the GTK code level.
  • Architectural constraints of Linux transparent borderless Tauri/WebKitGTK floating windows: WM, GTK, and WebView content area dimensions/positions do not auto-sync — this is a platform constraint, not a bug. The correct architecture is a fixed large native window with pure CSS expand/collapse internally, without relying on frequent resizing of the outer window; GTK gravity is not supported by all WMs, and moving by right-edge coordinates causes systematic offset; probe + corrective reapply cannot eliminate user-visible transition frames; separating the visible rect from the window rect at the native coordinate layer causes drag start points and mouse drift.
  • Runtime logs are the fundamental tool for low-level system debugging: when dealing with GTK, WM, OS, or other low-level system behavior, validate assumptions via logs/probes empirically rather than inferring from code; cross-platform animation strategies must be holistically consistent — disabling CSS transitions requires simultaneously adjusting corresponding JS/Rust timing logic; X11 WMs forcibly clamp undecorated windows within the visible area (edge_offset must be platform-differentiated); macOS/Windows allow partial off-screen positioning.
  • NeurIPS D&B Track submission strategy: “Field correctors” (triggering evaluation paradigm reconstruction by quantifying specific flaws in existing benchmarks, e.g., 57% accuracy rate) are more likely to be accepted and have more lasting impact than “new method proposers”; reviewers value most: multi-dimensional evaluation, fair baseline comparisons, human validation mechanisms (κ≥0.68), and counter-intuitive negative findings; counterfactual entity substitution can eliminate lexical overlap bias; symbolic verification loops (SPARQL/SQL execution comparison + N automatic repair cycles) can improve data correctness from 57% to 96%+; data cleaning can be reframed as unsupervised anomaly detection (Isolation Forest + Z-score cross-language normalization).
  • Radical simplification (swapping the engine) is often more effective than incremental patching: Thin Adapter + MCP Server + Skills three-layer pattern replacing proprietary routing logic achieves cross-CLI universality; large refactors should undergo adversarial review (Critic subagent) before coding, proactively discovering hidden dependencies and architectural risks. Across multiple projects today, the Critic phase discovered high-value issues including Wayland compatibility, lark-oapi thread blocking, ThoughtStore non-singleton, and 4 hidden code dependencies — extremely high ROI.
  • “A feature’s parameter interface is implemented” does not mean “the parameter is being passed”: the integration layer is the most common wiring breakage point; validate at the call site, not just the implementation side. FastMCP’s “create” and “serve” are two separate steps, and mcp_servers.json declarations are additionally required for external programs to discover the service; Python’s TypeError from awaiting a sync method, if swallowed by except Exception, causes silent feature degradation rather than an error.
  • Modern Linux desktops (Ubuntu 22.04+) default to Wayland; cross-platform window tracking solutions must use GNOME Introspect D-Bus as the primary path, with xdotool/xprop only as XWayland fallback. Background sampling tasks should not all depend on CronScheduler (minimum 60s tick) — tasks requiring sub-minute periods need independent asyncio.Task loops. ECL documents structurally persist design decisions and are a key engineering practice for multi-phase complex feature development.
  • Multi-CLI system design best practices: different CLI tools (Claude Code/Codex/Gemini) each have conventional instruction files (CLAUDE.md/AGENTS.md/GEMINI.md); using convention files is simpler and more reliable than runtime injection. Skills are essentially Markdown prompts; cross-platform migration only requires tool name mapping. App state validation and CLI tool context injection are two independent layers — passing token validation does not mean the CLI tool received the correct context. A unified history ring buffer is more reliable than adapting to each CLI’s session mechanism.
  • AionUI remote Agent only supports URL connections (CLI processes have no HTTP endpoint — fundamental architectural limitation), but AionUI’s database already has a remote_agents table, suggesting SSH direct connection may be on the future roadmap. AI should not state technical parameters (commands, model specs) with certainty in fast-moving product domains — it should declare a knowledge cutoff date and recommend users verify.
  • Correct lark-oapi WebSocket architecture: Client.start() is blocking and must use a separate daemon thread + asyncio.run_coroutine_threadsafe() bridge. Google Calendar OAuth scope is granular — token validation should use events.list(calendarId='primary') rather than calendarList.list(). Log level is a zero-cost observability lever — changing provider errors from DEBUG to WARNING immediately and significantly improves operational visibility.

Problems & Solutions#

Critical Issues#

1. FloatBall windows always remain 400x400 under Linux, and Tauri’s set_size() is ineffective#

Solution: GTK bottom-up size negotiation takes max(window_requested, widget_natural_size), and WebKitGTK’s natural size of approximately 200x200 causes the window to become locked. Fix: Called widget.set_size_request() through with_webview() in the GTK main thread. Removed the set_resizable switch (with_webview is asynchronous, so setting set_resizable(false) will queue it before the callback executes, preventing the old size from being locked). Linux will now remain resizable permanently.

Key Insight: Tauri’s set_size/set_min_size/set_max_size functions operate at the GTK window level hint. GTK bottom-up negotiation ignores these hints to meet the widget’s natural size. When with_webview() returns, the callback has not yet executed, so subsequent Tauri API calls are still in FIFO asynchronous queue.

2. Visual glitches when closing Windows under Linux: Two-stage sequence exposes intermediate states, and WM clamping of old widths causes systematic coordinate offsets#

Solution: Two-stage sequence: macOS has a 250ms animation buffer, while Linux disables transitions, making intermediate states glaring. In Linux, the window is first shrunk to 56px using Rust atomic operations, then CSS is set to false (so the change is invisible). Regarding WM clamping issues: GTK gravity does not work in this WM (instead, it introduces systematic offsets). Probe and corrective reapplication cause double-frame flickering. The ultimate solution is to use a fixed large-size window with internal CSS for expansion and collapse.

Key Insight: Disabling CSS transitions requires adjusting JS/Rust timing logic simultaneously. GTK gravity is not supported by all WMs.事后补救措施本身就会导致可见闪烁，根本解决方案是让WM一次性接受正确的坐标。

3. Three-layer breakdowns in Life-Copilot MCP Server: There is a Calendar token, but the agent still reports a connection failure#

Solution: Three areas were broken simultaneously: FastMCP was instantiated but not called run(), .claude/mcp_servers.json was missing, and butler commands (including the list of MCP tools) never reached the Claude CLI subprocess. Created main.py to start stdio transport, new mcp_servers.json, and added –append-system-prompt to ClaudeAdapter. All 19 MCP tools worked normally again.

Key Insight: “Writing code” and “code being called” are two different things. The factory pattern creates objects, not starting services. External programs need configuration files to discover services. App validation layer and CLI tool context injection are two independent layers. Token verification passing does not mean the CLI tool received the correct context.

4. The initial solution for Linux Screen Time Tracker used xdotool, which does not support Wayland (the user’s machine runs GNOME 50 Wayland)#

Solution: Conducted feasibility checks to confirm that Wayland was available. Instead, used GNOME Introspect D-Bus (org.gnome.Shell.Introspect.GetWindows()) as the main path, and xprop as a backup for XWayland. PID→cmdline was used as the final downgrade. All subprocess calls had a 5-second timeout and proc.kill().

Key Insight: GNOME 41+ provides a standard Introspect D-Bus interface, which is the correct way to obtain active windows under Wayland. This issue was actively discovered by Critic’s sub-agent anti-review. If overlooked, the function would be completely unavailable on user machines.

5. AionUI could not connect to the Claude Code CLI on a headless server via URLs or SSH#

Solution: AionUI’s remote agent architecture assumes that the agent exposes an HTTP port. Since the CLI process has no HTTP endpoint, it is fundamentally incompatible. It is suggested to use tmux/screen to manage multiple sessions as a temporary alternative. A feature request should be submitted to the AionUI team.

Key Insight: AionUI’s database has a remote_agents table (with an allow_insecure field), indicating that the development team is building remote agent capabilities. SSH direct connection may be a future roadmap direction.

6. X11 WM refuses to position half-out-of-screen windows, causing吸附 and collapse positions to drift#

Solution: X11 WMs like Mutter/GNOME place non-decorated windows within the visible area, refusing positions where x < 0 or x > screen_width - window_width. Linux uses edge_offset=0, while macOS/Windows use edge_offset=ball_size/2.

Key Insight: macOS and Windows allow non-decorated windows to partially exceed the screen boundary, while X11 WMs usually do not allow it. Different edge offsets are needed for both platforms.

7. ClaudeCLIAdapter.is_available() lacks async functionality, causing Claude to silently degrade to Codex and time out#

Solution: The await synchronous method does not report compilation errors, but the runtime TypeError was ignored by the except Exception. Claude was deemed unavailable. Adding the async keyword restored Claude availability to True.

Key Insight: In Python, await synchronous methods produce TypeError. If caught by the broad except Exception, it can lead to incorrect False results, resulting in a degraded function rather than an error. This is very difficult to troubleshoot.

8. In multi-turn dialogues, the bot cannot understand cross-turn references (DualVerifier parameter connection issue)#

Solution: DualVerifier, SemanticRouter, and KeywordScorer all have last_assistant_message and last_intent parameters, but they are never passed during _process_input() calls.

Problems & Solutions#

Critical Issues#

1. FloatBall windows always remain 400x400 in Linux, Tauri’s set_size() is ineffective#

Solution: GTK bottom-up size negotiation takes max(window_requested, widget_natural_size), and WebKitGTK’s natural size of approximately 200x200 causes window locks. Fix: Called widget.set_size_request() through with_webview() in the GTK main thread, removed set_resizable switch (with_webview is asynchronous, set_resizable(false) will queue up and lock the old size before the callback execution), and made Linux permanently resizable.

Key Insight: Tauri’s set_size/set_min_size/set_max_size operations follow GTK window-level hints, and GTK bottom-up negotiation ignores these hints to meet the widget’s natural size. The callback from with_webview() has not yet executed, so subsequent Tauri API calls are still FIFO asynchronous queues.

2. Visual flicker when folding in Linux: Two-stage sequence exposes intermediate states + WM clamp’s old width causes systematic coordinate offsets#

Solution: Two-stage sequence: macOS has a 250ms animation buffer, while Linux disables transitions, making the intermediate state glaring. In Linux, first shrink the window to 56px using Rust atomic operations, then set CSS expanded to false (at this point 100% = 56px, so the change is invisible). WM clamp issue: GTK gravity does not work on this WM (instead, it introduces systematic offsets), probe + corrective reapplication produces double-frame flickering. The fundamental solution is to use a fixed large-size window + internal CSS for expansion and folding.

Key Insight: Disabling CSS transitions requires synchronizing JS/Rust timing logic; GTK gravity is not supported by all WMs;事后补救措施本身就是可见闪烁的来源,根本解法是让WM一次性接受正确的坐标。

3. Three-layer break in Life-Copilot MCP Server: There is a Calendar token, but the agent still reports a connection failure#

Solution: Three places simultaneously broke: FastMCP was instantiated but not called run(), .claude/mcp_servers.json was missing, and Butler commands (including MCP tool list) never reached the Claude CLI child process. Created main.py to start stdio transport, new mcp_servers.json, and added –append-system-prompt to ClaudeAdapter. All 19 MCP tools worked normally again.

Key Insight: “Code written” and “code called” are two different things; creating an object with the factory pattern does not equal starting a service; external programs need configuration files to discover services; App verification layer and CLI tool context injection are two separate layers; token verification passing does not mean the CLI tool received the correct context.

4. Initial solution for Linux Screen Time Tracker used xdotool, which does not support Wayland (user’s machine runs GNOME 50 Wayland)#

Solution: Conducted feasibility checks to confirm Wayland environment, replaced the main path with GNOME Introspect D-Bus (org.gnome.Shell.Introspect.GetWindows()), used xprop as a backup for XWayland, and PID→cmdline as the final downgrade. All subprocess calls had a 5-second timeout and proc.kill().

Key Insight: GNOME 41+ provides a standard Introspect D-Bus interface, which is the correct way to obtain active windows under Wayland. This issue was actively discovered by Critic sub-agent during review. If overlooked, it would result in the function being completely unusable on user machines.

5. AionUI could not connect to the Claude Code CLI on a remote headless server via URL or other methods (e.g., SSH)#

Solution: AionUI remote agent architecture assumes that the agent exposes an HTTP port, while the CLI process has no HTTP endpoint. This is fundamentally incompatible. It is recommended to use tmux/screen to manage multiple sessions as a temporary alternative, and submit a feature request to the AionUI team.

Key Insight: AionUI database already has a remote_agents table (with allow_insecure field), indicating that the development team is building remote agent capabilities. SSH direct connection may be a future roadmap direction.

6. X11 WMs reject positioning of half-out-of-screen windows, causing drifting during adsorption and folding#

Solution: X11 WMs like Mutter/GNOME place non-decorated windows within the visible area, rejecting positions where x < 0 or x > screen_width - window_width. Linux uses edge_offset=0, while macOS/Windows use edge_offset=ball_size/2.

Key Insight: macOS and Windows allow non-decorated windows to partially exceed the screen boundary, while X11 WMs usually do not allow it. Different edge offsets are required for both platforms.

7. ClaudeCLIAdapter.is_available() lacks async function, causing Claude to silently downgrade to Codex and time out#

Solution: Ordinary await methods do not throw compilation errors, but runtime TypeError is swallowed by except Exception. Claude was deemed unavailable. Adding the async keyword restored Claude availability to True.

Key Insight: In Python, await synchronous methods produce TypeError. If caught by a broad except Exception, it can lead to incorrect False results, resulting in a downgrade in functionality rather than an error message. This is very difficult to troubleshoot.

8. In multiple-round dialogues, the bot cannot understand cross-round references (DualVerifier parameter connection broken)#

Solution: DualVerifier, SemanticRouter, and KeywordScorer all have last_assistant_message and last_intent parameters, but they are never passed during _process_input() calls. The AI analyzed logs and confirmed that the WM clamps the position based on the old expanded width, resulting in a systematic offset of ~305px. Multiple fixes, including those related to GTK gravity (introducing new offsets), GDK move_resize, probe corrections (double-frame flickering), and transparent safety margins (disrupting mouse tracking), did not fully resolve the issue.

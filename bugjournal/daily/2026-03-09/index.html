Problems & Solutions#

Critical Issues#

1. PyTorch 2.9/CUDA 12.9 cuBLAS bug on RTX 2080 Ti causes crashes during backward pass of large matrix operations (3639×3639 logits for contrastive loss). Multiple architecture rewrites attempted before identifying root cause.#

Solution: Redesigned contrastive loss to use mini-batch sampling (512 samples per batch) to avoid full N×N matrix computation. Added gradient clipping and LayerNorm for stability. For cross-sample visualization task, switched to CPU training as fallback when GPU incompatibility confirmed (PyTorch 2.9.0+cu129 vs CUDA 13.1 driver).

Key Insight: GPU hardware/driver compatibility issues cannot be solved through model architecture alone—algorithmic workarounds at the loss function level are necessary. Version mismatches between PyTorch CUDA build and driver version cause sporadic ‘illegal memory access’ errors that pass basic tests but fail in complex training loops. CPU fallback is valid strategy for medium-scale training (3460 spots).

2. MCP server stdout corruption: gadget tools use print() extensively, but MCP uses stdout as JSON-RPC transport#

Solution: Created capture_stdout() context manager to redirect print() output, wrapped all tool invocations with _safe_call() that captures stdout/stderr and converts SystemExit to RuntimeError, returned captured logs in JSON response’s _logs field

Key Insight: When integrating existing CLI tools into protocol-based systems, stdout isolation is critical. Context managers provide clean, reversible redirection without modifying original code

3. research_scout hardcoded arxiv_id throughout codebase, blocking multi-source expansion#

Solution: Introduced paper_id as canonical ID with source field, kept arxiv_id as backward-compatible alias, created _paper_id()/_paper_url() helpers, updated prompts to use paper_id, made merge logic check both fields when matching LLM responses

Key Insight: When generalizing from single-source to multi-source, introduce new abstraction while maintaining old field as alias. Helper functions centralize ID/URL logic, making future source additions trivial

4. RM-IDEAL correlation low for Layer_3 (ρ=-0.21/-0.36) in 151673-151508 cross-sample query, while Layer_1/4/5 show high positive correlation (ρ>0.4)#

Solution: Analysis revealed Layer_3 is the largest DLPFC layer with high internal heterogeneity, making embeddings unable to capture unified spatial structure. Smaller, unique layers with clear boundaries (Layer_1: ρ=0.62, Layer_4: ρ=0.66) perform best.

Key Insight: Tissue layer size and internal heterogeneity critically affect embedding quality for spatial structure capture. Distinct, boundary-clear layers yield better cross-sample matching than large heterogeneous regions.

5. Single-token cross-attention degenerates to linear projection when each spot has only one gene vector and one image vector#

Solution: Used spatial k-NN neighbors to construct K/V sequences, allowing cross-attention to operate on neighborhood context. Reused QFormerFusion’s batched neighbor construction infrastructure.

Key Insight: Spatial transcriptomics data provides natural sequential context through neighborhood structure. Leveraging spatial adjacency enables meaningful attention even without temporal or batch dimensions.

6. RoboTwin demo_clean data (14-DOF, directory-based) incompatible with existing LeRobot conversion script (16-DOF, zip-based)#

Solution: Created new script convert_robotwin_democlean_to_lerobot.py: read directly from data/instructions directories, change ACTION_DIM/STATE_DIM to 14, use LeRobotDataset.create() with root parameter instead of HF_LEROBOT_HOME env var hack

Key Insight: Robot data format variations (DOF count, storage format) require tailored conversion scripts. Directory structure and HDF5 key names are project-specific.

7. Empty motivation/innovation_point fields in research_scout screening stage due to silent LLM JSON parsing failures#

Solution: Added empty-field detection after _screen_papers() and _screen_papers_chunked(), logged warnings with count and DEBUG-level per-paper details, enhanced _try_repair_result() to log raw response preview when repair fails

Key Insight: Silent .get() fallbacks mask data quality issues. Explicit validation and logging at aggregation points (after batch processing) provides actionable debugging information without cluttering per-item logic

8. Initial CrossModalEnhancer design outputting original modality dimension (1536d for image) triggered cuBLAS GEMM failure on 128→1536 projection#

Solution: Revised architecture to output in hidden_dim space (128d) instead, avoiding problematic large-matrix projections. Used projection heads for contrastive loss while keeping core output compact.

Key Insight: Engineering constraints (GPU bugs) can force architectural decisions that differ from ideal semantic design. Practical systems must balance theoretical elegance with implementation robustness.

Learnings#

Key Learnings#

  • RM-IDEAL uses WWL (Wasserstein Weisfeiler-Lehman) graph kernel to compute spatial niche matching: iterative neighbor label hashing → node feature distributions → optimal transport (Wasserstein distance) between subgraphs. Provides topology-based ground truth for evaluating embedding quality.
  • Cross-attention on single tokens degenerates to linear projection. Spatial transcriptomics provides natural solution: k-NN neighbors form K/V sequences that encode neighborhood context. This pattern (spatial neighbors as attention context) is broadly applicable to spatial omics architectures.
  • PyTorch 2.9 + CUDA 12.9 on RTX 2080 Ti has cuBLAS bug causing illegal memory access on large matrix operations (gemm failures at specific shapes like 128→1536, backward on N×N where N>3500). Workaround: mini-batch processing to keep matrices small. PyTorch CUDA build version (e.g., cu129) must match GPU driver CUDA version.
  • scGPT foundation model underperforms traditional methods (PCA, scVI, Harmony) in zero-shot clustering benchmarks (Genome Biology 2025). AvgBIO scores on Pancreas/PBMC datasets show scGPT ~0.52-0.60 vs. HVG baseline ~0.60-0.77. scGPT-spatial requires fine-tuning and only compares to weak baselines (SpaGCN, stLearn), not SOTA (GraphST, STAGATE). Independent benchmarks more reliable than original paper claims.
  • Conditional Flow Matching for robot action generation: learns vector field u_t from noise to actions using linear interpolation path x_t = t·noise + (1-t)·actions, Beta(1.5,1) time sampling for stability near noise endpoint, Euler integration with 10 steps for inference. Key insight: time conditioning via sinusoidal embeddings + MLP (or adaRMS for Pi0.5).
  • MCP tool design principle: tools for Claude Code should return content by default (markdown reports, JSON data), with optional file persistence. This enables in-context use without filesystem round-trips. Summarize/research are ’enhancement skills’, not just file generators.
  • When wrapping existing CLI tools for protocol-based systems (MCP, JSON-RPC), stdout/stderr isolation is a first-class architectural concern, not an afterthought. Use context managers for clean, reversible redirection without modifying original code
  • When generalizing single-source systems to multi-source, introduce new abstraction (paper_id) while keeping old field (arxiv_id) as alias. Helper functions (_paper_id, _paper_url) centralize cross-source logic and make future expansions trivial
  • DLPFC tissue layer characteristics directly impact embedding cross-sample matching: Layer_1 (superficial, thin) and Layer_4 (granular, cell-dense) show Spearman ρ>0.6; Layer_3 (largest, heterogeneous) shows ρ<-0.2. Embedding quality correlates with structural uniqueness and boundary clarity, not layer size.
  • RoboTwin data format: 14-DOF action space (left_arm 6 + gripper 1 + right_arm 6 + gripper 1), stored as HDF5 with joint_action/vector, observations in compressed image format. Demo_clean directory structure: data/episodeX.hdf5, instructions/episodeX.json with seen/unseen language variations.
  • STAIG fusion training for spatial omics: UNI2 (1536-dim vision) + PCA (50-dim gene) → STAIG fusion encoder (64-dim). Key insight: alignment mode uses tau=26, softmax_log edge weights, 300 pseudo-clusters, 550 epochs. Model can collapse (all-zero embeddings) if training fails.
  • UniVLA training pipeline: three stages (LAM stage-1/2, VLA pretraining, per-benchmark finetuning). CALVIN training: DiskCalvinDataset loads auto_lang_ann.npy for episode-language mappings, reads episode_*.npz windows (12-13 frames), extracts rgb_static → LAM → latent_action_idx → VLA input → ActionDecoder → rel_actions.
  • Cross-environment tool invocation (conda run -n ) provides clean isolation for sibling repos with conflicting dependencies. Critical to set cwd to tool repo root, as many Python scripts use relative imports and sys.path manipulation
  • Production logging requires two handlers: StreamHandler (INFO) for user-facing progress, RotatingFileHandler (DEBUG) for diagnostic details. Keep print() in interactive UI commands (list, show) but migrate all operational/diagnostic messages to logger
  • CALVIN evaluation dataset structure: only needs validation/.hydra/merged_config.yaml for environment initialization. Episode .npz files not read during evaluation (online simulation generates observations). Data extraction can reduce dataset size from 1.3GB to 600KB.
  • Zero-dependency API clients (urllib.request + xml.etree) are viable for production tools when avoiding new dependencies is critical. bioRxiv and PubMed both implement with stdlib only, maintaining gadget’s minimal dependency philosophy
  • Python package distribution for MCP servers: standard pattern is PyPI + uvx for stateless tools, but pip install -e . (editable install) required when server depends on repo data directories. Entry point in pyproject.toml [project.scripts], then claude mcp add gadget – gadget-mcp.

Practical Learnings#

  • When integrating new fusion strategies into existing pipeline: (1) register in apply_fusion() dispatcher, (2) add to needs_coords tuple in runner.py and evaluation_planner.py, (3) add config dict entry in runner.py, (4) define experiment in pipeline_config.yaml. This 4-step integration pattern is standard across all trainable fusion methods.
  • When updating documentation, grep for CLI flags and function names to systematically identify gaps rather than relying on manual code review

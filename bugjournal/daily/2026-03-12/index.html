Problems and Solutions#

Key Issues#

1. curobo precompiled .so files are incompatible with torch 2.7.1 ABI (undefined symbol: torchInternalAssertFail). JIT recompilation failed due to lack of ninja and CUDA header files.#

Solution: Install ninja, set CUDA_HOME to the conda environment root directory, set CPATH to envs/RefineVLA/targets/x86_64-linux/include/, and execute pip install -e . to recompile curobo from source. Key insight: The CUDA toolkit headers installed via conda are located in /envs//targets/x86_64-linux/include/ instead of /usr/local/cuda/include/. When compiling CUDA extensions, CUDA_HOME and CPATH must be set to this path.

2. torch 2.7.1+cu126 is not compatible with torchvision version 0.21.0. The torchvision::nms operator registration failed.#

Solution: Upgrade torchvision to 0.22.1+cu126 (pip install torchvision==0.22.1 –index-url [[[HYMTPH_0]]). Key insight: Torch 2.7.x must be paired with torchvision 0.22.x. The link in the torchvision .so file refers to the torch C++ ABI, and a major version upgrade requires a corresponding update.

General Issues#

3. The checkpoint path policy/pi05/checkpoints/pi05_robotwin2/demo_clean/5000/assets/ does not exist. The default deploy_policy.yml checkpoint_id is 5000, but the actual available values are 15000/25000/29999.#

Solution: The interrupted session was not resolved. The checkpoint_id in the deploy_policy.yml needs to be changed to an available value (e.g., 29999) or passed as a parameter to eval.sh. Key insight: The checkpoint_id parameter passed by eval.sh must correspond to the actual training output. The default value of 5000 in the yml is just a placeholder.

4. The output width of gpumon.py exceeds the terminal width, and processes appear repeatedly on multiple GPUs.#

Solution: Rewrite the process table logic to limit the output to 80 columns and filter out noise from sub-processes like multiprocessing worker/wandb-core. Key insight: AI understood the requirement as global deduplication (once per process), but the actual need is to deduplicate for each (process, GPU) pair (each process appears once per GPU).

AI Limitations#

Important Limitations#

  • AI misunderstood gpumon.py’s deduplication requirement, implementing “global deduplication per process” instead of “deduplication for each (process, GPU) pair”. Users had to explicitly correct it to discover the error.

General Limitations#

  • It took multiple attempts to trace CUDA header file paths (pip nvidia package path → system /usr/local/cuda → conda targets directory). AI failed to locate the correct path based on the environment structure at once.
  • When paths were uncertain, AI tended to use find for global search, which was rejected by users. It should prioritize using the known environment structure (conda env layout) to infer paths.

Today’s Gains#

Core Gains#

  • When installing the CUDA toolkit in a conda environment, the header files are located in /envs//targets/x86_64-linux/include/ (not /usr/local/cuda/include/). When compiling CUDA extensions, set CUDA_HOME to <conda_env_root> and CPATH to <targets_include>.
  • After a major version upgrade of torch, all CUDA extensions that depend on the torch C++ ABI (precompiled .so files like torchvision and curobo) need to be recompiled or upgraded. Torch 2.7.x corresponds to torchvision 0.22.x.
  • When curobo’s precompiled .so file is not compatible with the current torch, deleting the .so and then using pip install -e . to rebuild from source is a feasible quick fix. The key is setting the correct CUDA compilation environment.

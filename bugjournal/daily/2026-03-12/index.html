Problems and Solutions#

Key Problems#

1. macOS non-sandbox applications cannot persist file access permissions, causing “Operation Not Allowed” errors for old videos in historical records.#

Solution: Enabled App Sandbox permissions (com.apple.security.app-sandbox and files.user-selected.read-write). Fixed ensureFileAccess to add non-restrictive bookmark rollback parsing. Added NSOpenPanel reauthorization mechanism for showVideo/showImage.

Key Insight: macOS TCC requires App Sandbox to create security-level bookmarks. Non-restrictive bookmarks can only parse URLs but cannot grant access, resulting in inability to access old files. Combining sandbox and downgrading paths is needed to solve this problem.

2. ABI incompatibility between torch 2.7.1 and torchvision 0.21.0 caused failure when registering torchvision::nms operator.#

Solution: Upgraded torchvision to 0.22.1+cu126 to match torch 2.7.1 version.

Key Insight: When compiling custom operators in torchvision, it links to torch’s C++ API. Incompatible versions can cause operator registration failures. torchvision 0.21.0 corresponds to torch 2.6.x, which cannot be used with torch 2.7.x.

3. curobo’s pre-compiled.so file is incompatible with current PyTorch ABI, and JIT compilation couldn’t find cuda_runtime.h.#

Solution: Deleted old.so file, installed ninja compiler, set CUDA_HOME to /path/to/conda.env, and configured CPATH to /path/to/targets/x86_64-linux/include. Recompiled using pip install -e.

Key Insight: After upgrading PyTorch version, C++ extensions need to be recompiled. The Nvidia package installed in the conda environment places the CUDA header file in a non-standard path (targets/x86_64-linux/include/), and environment variables must be correctly configured for JIT compilation to find them.

General Problems#

4. Output of GPU monitoring tools is too wide and process displays are repetitive, making it difficult to read.#

Solution: Limited output to 80 columns, aggregated GPUs per process, and filtered out noise from multi-process work processes.

Key Insight: The original implementation grouped GPUs by device, resulting in repeated display of multiple GPUs on multiple GPUs. It’s necessary to aggregate processes and display all card numbers used in the GPU column.

Lessons Learned#

Key Lessons#

  • macOS TCC permission mechanism: Non-sandbox applications cannot create security-range bookmarks; can only create non-security-range bookmarks (only parsing URLs that don’t require access rights), resulting in historical file access failure after application restart. Application sandbox must be enabled to maintain file access permissions.
  • PyTorch C++ extension ABI compatibility: Custom operators in torchvision are linked to Torch’s C++ API during compilation. Versions must match exactly (torchvision 0.21→torch 2.6, torchvision 0.22→torch 2.7). Incompatibility can cause operator registration failures.
  • Condna environment CUDA toolchain configuration: The Nvidia package installed via pip places the CUDA header file in targets/x86_64-linux/include/, not the standard include/ directory. For JIT compilation of C++ extensions, it’s necessary to set CUDA_HOME to the conda environment and configure CPATH to point to the target path.
  • Defensive programming principles: Combining sandbox and reauthorization is more robust than using a single solution. Sandbox addresses the root cause (persistence permission issues), while NSOpenPanel provides a downgrading approach (temporary authorization). Combined use ensures users can always access files.

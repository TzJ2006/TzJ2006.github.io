问题与解决方案#

关键问题#

1. H&E image-only方法在DLPFC精细分层任务上ARI极低（0.11-0.16），远低于多模态（0.45-0.64）#

解决方案: 调研揭示根本原因：预训练数据以癌症组织为主（脑组织极少），DINOv2学粗粒度语义不学精细空间梯度，patch独立编码无法感知皮层深度位置。近期方案：STEGO（feature correspondence distillation）、域内BYOL适配（STAIG方案）、TTT-MAE可逐步改善

关键洞察: 问题不是encoder质量，而是预训练任务与目标任务根本不对齐；从FGVC到遥感到材料科学，共识解法是域内自监督预训练+特征精炼

2. STEGO对比损失全程NaN：n_spots>3000的dense相似度矩阵在temperature=0.07下float32数值溢出#

解决方案: 先尝试L2归一化输入（无效），最终改用数值稳定的logsumexp实现：sim先减去行最大值（masked_fill）再做log_softmax，loss正常收敛至1.96，ARI=0.209

关键洞察: 大n_spots场景下dense对比矩阵必须使用numerically stable logsumexp；small temperature（≤0.1）使极端值更容易溢出float32

3. BC-RNN策略无法在1c_generate_from_policy.py中直接使用（仅支持vla_server），导致transport/place相位场景无法生成#

解决方案: 在M12计划中将BC-RNN注入模式作为补充场景生成的核心路径，通过–policy bc_rnn参数调用，Makefile添加scale-bc-rnn-inject目标

关键洞察: BC-RNN约60-70%成功率是唯一能到达transport/place后期阶段的策略；VLA（Pi0 SR=0%）无法到达后期阶段，BC-RNN是解决相位空白的关键

4. G4实验恢复训练数据来源不明确（Oracle自动回放 vs 人工遥操作），用户对"恢复训练数据"核心概念不熟悉#

解决方案: AI用具体例子解释（正常demo vs 从错误状态出发→成功的轨迹），用户确认选择人工遥操作（目标≥200条，先50条快速验证）

关键洞察: Oracle回放仅适用于impulse注入场景，人工遥操作覆盖面更广且代表真实恢复策略；研究规划中专业术语需主动用具体例子解释

一般问题#

5. VLA评估耗时5-15小时，crash后全部进度丢失；多组运行结果分散在不同目录无法统一分析#

解决方案: collector.py添加resume支持（启动时重建completed集合跳过已完成episodes）；4_analyze_results.py支持–run_dir多路径输入，新增load_episodes_multi()合并JSONL，输出per-error-type矩阵和summary.json

关键洞察: append-only JSONL天然支持resume；评估数据采集与分析脚本应解耦，分析脚本通过nargs=’+‘支持多次运行合并，比单次全量运行更健壮

6. eval_scan_fusion.py多处调试：load_dlpfc_data返回值解包错误 + load_spatial_coordinates返回(1,2)形状坐标 + staig_fusion缺少raw基因特征参数#

解决方案: 改为不解包元组直接读取adata.obsm[‘spatial’]获取空间坐标；staig_fusion通过adata.X传入原始基因表达矩阵

关键洞察: load_dlpfc_data实际只返回单个adata；直接用obsm[‘spatial’]比调用load_spatial_coordinates更稳健；staig_fusion与其他fusion接口不同，需要原始基因矩阵而非PCA embeddings

7. Windows OpenSSH ControlMaster配置导致VS Code Remote-SSH连接崩溃（Bad file descriptor / nonexistent pipe）#

解决方案: 注释掉~/.ssh/config中的ControlMaster auto、ControlPath、ControlPersist三行，并删除残留stale socket文件（~/.ssh/sockets/*）

关键洞察: Windows OpenSSH的连接复用基于Unix domain socket，产生损坏socket文件导致后续所有连接静默失败

8. MimicGen启动时MUJOCO_EGL断言错误：MUJOCO_EGL_DEVICE_ID=0与CUDA_VISIBLE_DEVICES=3不匹配#

解决方案: 将MUJOCO_EGL_DEVICE_ID改为与CUDA_VISIBLE_DEVICES一致（均设为3），重新启动任务

关键洞察: robosuite要求MUJOCO_EGL_DEVICE_ID必须等于GPU物理设备索引，而非虚拟可见索引0；多卡服务器上分配特定GPU时需同时设置两个环境变量

9. AI反复触发ExitPlanMode请求二次审批（MIHD文档、M12/M13实施、/init命令等），导致至少5次会话中断#

解决方案: 用户多次拒绝后改为直接提供完整实现计划，要求AI直接执行；收到用户完整计划时应跳过plan mode直接实现

关键洞察: AI无法区分"用户自制计划→直接执行"与"AI需先规划→再实现"场景，在已有完整计划时默认进入plan mode是错误行为

10. 大型文档超token上限（项目全景总结.md超25000 token）以及学术论文付费墙（Nature/Oxford等返回403/303）#

解决方案: 分块读取（offset+limit参数）+并行Explore agent收集统计数据；转用PMC全文、bioRxiv预印本、GitHub README、HuggingFace model card等替代来源

关键洞察: 大文件处理需要分块策略+专门agent并行；PMC和preprint是获取方法论细节的最可靠免费来源

11. HPC环境中旧的孤儿CPU进程（运行76分钟未结束）与新GPU后台任务竞争资源，导致后者长时间无输出#

解决方案: kill该进程及其父shell进程，GPU任务随即正常启动

关键洞察: 并行提交多个GPU任务前应先确认无孤儿进程占用GPU显存

AI 局限性#

重要局限#

  • 在已获得用户完整实现计划的情况下，仍默认进入计划模式（通过ExitPlanMode请求二次批准），无法区分"用户自制计划→直接执行"与"AI需先规划→再实现"的场景，导致至少5次会话中断
  • 文献调研中出现误分类（MILWRM被误归为image-only方法，实际基于基因表达）和对象错误（将scGPT可视化误作UNI2分析），体现了在多信息源并行处理时对核心假设验证不足
  • 未主动识别跨任务结构相似性（CRC-100K与MIHD空间域识别的本质等价）；使用专业术语（“恢复训练数据”）未主动用具体例子解释，导致用户困惑
  • 代码生成中的细节失误：STEGO初版存在数值稳定性bug（NaN loss，需两轮调试）、eval_scan_fusion.py错误解包返回值、numpy bool类型断言（is True vs == True），均需额外调试轮次

一般局限#

  • 对image-only范围限定条件需被动纠正两次才能专注；付费论文全文（Nature Medicine等）和图表数值无法访问，部分定量分析缺乏具体数字支撑
  • HPC环境中忽视孤儿进程竞争GPU资源等运行时环境因素；GPU配置细节知识不足（MUJOCO_EGL_DEVICE_ID对齐）需观察错误日志后才能修正

今日收获#

核心收获#

  • benchmark论文的核心论证结构四层递进：大规模数据集（覆盖广度）→分类有效性论证（学术严谨性）→基线性能建立（可比较基准）→Foundation Model有用性证明（应用价值），每层为下层提供基础
  • H&E image-only方法在ST精细分层任务中ARI仅0.11-0.16，根本原因是：预训练数据以癌症组织为主（脑组织极少）、DINOv2学粗粒度语义不学精细空间梯度、patch独立编码无法感知皮层深度位置。这是文献空白——几乎没有方法单独评测vision-only embedding在空间域识别（无监督聚类）上的效果，也是MIHD image-only baseline的合理预期下界
  • SCAN自监督方法（k-NN特征挖掘+聚类一致性损失）在DLPFC 151673上ARI 0.251→0.303（+20.6%）；SCAN优化后的256维视觉embeddings与PCA基因特征做llava_mlp fusion进一步提升至ARI 0.409，优于原始UNI2+PCA baseline，说明SCAN可作为多模态pipeline的有效视觉特征预处理步骤，具有正向叠加效应
  • 域内自监督适配方案：STAIG用BYOL在目标H&E patches上做域内自监督预训练（选BYOL原因：不需要负样本、小batch友好、patch间差异微小时SimCLR会把相似层patch推开）；跨领域共识解法已收敛：域内无监督预训练>域外预训练，小数据场景首选STEGO/SCAN，无训练预算时FM embedding+NMF（F-SEG）是zero-shot baseline
  • HEST-1k公平比较方法：所有encoder embedding统一PCA降至256维再做Ridge回归，避免高维encoder被正则化不公平惩罚；模型大小与性能对数相关（R=0.81）；spEMO显示GPFM和CHIEF在空间域识别最优，UNI2在spot检索最优
  • Training-free VLA错误恢复方法（FailSafe、VLA-SCT）最易集成benchmark评测，无需额外训练即可在π0/OpenVLA上验证；BC-RNN（60-70% SR）是唯一能到达transport/place后期阶段的策略，detector min_step阈值（100→50）直接决定相位覆盖广度
  • 大规模VLA评估必须内置resume机制（append-only JSONL天然支持断点续跑）；采集与分析脚本应解耦；人工遥操作恢复数据比Oracle自动回放更有论文价值（代表真实恢复策略，而非算法构造的理想轨迹）
  • 大n_spots场景（>3000）下dense对比损失矩阵必须使用数值稳定logsumexp（减去行最大值后log_softmax），temperature≤0.1时float32极易溢出；这一数值风险在实现时应主动预防而非被动调试
  • Kruskal-Wallis非参数检验+Dunn事后检验是验证错误分类区分度的统计框架，适用于小样本非正态分布的机器人实验数据；Cohen’s kappa矩阵验证分类器非冗余性（跨category kappa<0.3为标准）
  • 项目文档"追加式增长"是常见反模式——每次只添加章节不整合，导致同一概念被描述2-4次；定期结构性重构（而非持续追加）是必要实践，可操作checklist是文档的核心不可压缩部分
  • UNI2中outlier cluster（棕色，规律性分布）对应血管横截面、大神经元胞体等跨层级视觉相似结构，揭示基于癌症图像训练的PFM在正常脑组织上的迁移局限；UNI原论文34个评测任务横跨ROI分类/Slide分类/细胞分割/图像检索四类，Kather MSI和HEST基因表达预测是形态→分子的跨模态任务，与MIHD最相关

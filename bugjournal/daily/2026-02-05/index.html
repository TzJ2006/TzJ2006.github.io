Problems & Solutions#

Critical Issues#

1. apply_fusion() missing **kwargs parameter caused NameError, preventing staig_fusion_e2e branch execution#

Solution: Added **kwargs to apply_fusion() function signature to accept dynamically passed parameters (e.g., gene_expr_raw)

Key Insight: Python function signatures must be complete - even if parameters are only used in internal branches, they must be declared in signature or received via **kwargs

2. STAIGTrainerE2E hardcoded assumption that input is raw gene expression (3000-dim), incompatible with PCA/MLP/GCN encoder outputs and forcing gene_encoder=‘none’#

Solution: Added gene_preprocessing parameter (auto/standard/clip/none) to automatically select preprocessing strategy based on input dimensionality and source; raw expression uses StandardScaler+Clip, pre-encoded embeddings use Clip only

Key Insight: Flexible preprocessing strategy should be determined by data characteristics (dimensionality, source) rather than hardcoded; auto mode can intelligently choose via dimension threshold (>100). API design should prioritize flexibility over assumptions about data flow - end-to-end training can work with pre-encoded features

3. GCN+STAIG fusion performance extremely poor (ARI 0.05-0.09) and unstable compared to PCA+STAIG (ARI 0.45)#

Solution: Two-phase diagnosis: (1) Compared with original STAIG implementation discovering 4 algorithmic differences (edge dropout, feature clipping, edge weight direction, tau decay), (2) Analyzed original STAIG configuration revealing hyperparameter mismatches (learning_rate, num_layers, num_epochs, weight_decay)

Key Insight: Correct code logic ≠ correct implementation - both algorithmic alignment and hyperparameter configuration are critical for deep learning model performance. Implementation alignment requires thorough line-by-line comparison with reference code, not just reading the paper. Dramatic underperformance (0.05 vs 0.45) signals implementation bugs, not hyperparameter tuning issues

4. MIHD STAIG implementation achieves ARI=0.05, far below original STAIG project’s 0.452#

Solution: Compared original STAIG’s train_img_config.yaml revealing critical hyperparameter inconsistencies; need to change learning_rate to 0.0005, num_layers to 1, num_epochs to 300, weight_decay to 0.00001

Key Insight: Correct code logic does not equal correct implementation - hyperparameters are critical to deep learning model performance, must fully align with original paper/project configuration to reproduce results

5. Adaptive edge dropout implementation differs fundamentally from PyG’s standard dropout_adj - requires custom logic based on edge probabilities#

Solution: Implemented custom adaptive_dropout_adj() function following original STAIG net.py:209-240, where mask = rand() >= edge_prob means higher-weight edges are MORE likely to be dropped

Key Insight: Original STAIG’s counter-intuitive dropout strategy (drop similar edges, keep dissimilar) prevents in-domain overfitting and enhances cross-domain learning

6. UNI2 without parameters causes position encoding resampling error on GPU (shape mismatch)#

Solution: Pass complete timm_kwargs (including img_size, patch_size, depth, num_heads, etc.) to let timm correctly initialize UNI2 model, avoiding reliance on automatic inference

Key Insight: When loading pretrained models, explicitly passing architecture parameters is more reliable than depending on framework automatic inference, especially for shape-sensitive components like position encoding

General Issues#

7. General environment dependency chaos with missing packages and version conflicts#

Solution: Adopted pragmatic strategy: directly run benchmark, install missing packages on-demand when encountering ModuleNotFoundError; use conda install for large conda-managed packages; fix underlying dependencies (e.g., six) first when encountering uninstall errors

Key Insight: Environment repair best practice is incremental fixing rather than comprehensive upfront checking - this quickly identifies actually needed dependencies and avoids over-installation

8. argparse rejected ‘–gene_encoder gcn’ even though GCNGeneEncoder class exists in codebase#

Solution: Added ‘gcn’ to choices list and implemented GCN handling branch in run_evaluation() with spatial graph construction

Key Insight: Implementation completeness requires both the model class and the CLI integration - missing CLI support makes features invisible to users

Learnings#

Key Learnings#

  • When implementing published methods, compare line-by-line with reference code BEFORE claiming alignment - papers omit critical implementation details like adaptive dropout mechanics. Both algorithmic alignment and hyperparameter configuration are critical for deep learning model performance reproduction
  • Deep learning project reproduction key is complete hyperparameter configuration alignment - even with perfectly correct code logic, subtle parameter differences (e.g., learning_rate from 0.0005 to 0.001) can cause cliff-like performance drops (ARI from 0.452 to 0.05)
  • Flexible software architecture requires strategy pattern at key nodes: gene_preprocessing’s auto/standard/clip/none four-mode design demonstrates how to make a module support multiple input types without breaking interfaces. API flexibility matters more than narrow optimization - E2E training should support both raw and pre-encoded inputs, not force one path
  • Dramatic underperformance (0.05-0.09 vs 0.45 ARI) signals implementation bugs, not hyperparameter tuning issues - reference comparison is faster than blind optimization. Ablation studies reveal which changes actually matter - expected improvements should be validated incrementally
  • End-to-end training vs two-stage training choice affects model performance and training efficiency: original STAIG uses end-to-end approach (GCN embeddings trained within), while MIHD initially used two-stage (extract features then train) - this is the core architectural difference
  • When loading pretrained models (like UNI2), explicitly pass architecture parameters rather than relying on framework automatic inference, especially for shape-sensitive components like position encoding - explicit parameters avoid runtime errors

Practical Learnings#

  • Environment dependency management best practice is incremental repair rather than comprehensive upfront checking: directly run target program, install on-demand when encountering ModuleNotFoundError - more efficient than pre-scanning dependencies
  • GPU computation determinism can be resolved by setting CUBLAS_WORKSPACE_CONFIG environment variable, but must be set early at program entry (before imports), with slight performance overhead

Problems and Solutions#

Key Problems#

1. The apply_fusion() function lacks **kwargs parameters, causing a NameError, which prevents the staig_fusion_e2e branch from executing.#

Solution: Added **kwargs parameters to the signature of the apply_fusion() function, allowing dynamic passing of parameters (e.g., gene_expr_raw).

Key Insight: Python function signatures must be complete—even if parameters are only used in internal branches, they must be declared in the signature or received via **kwargs.

2. STAIGTrainerE2E assumes that the input is the original gene expression (3000 dimensions), which is incompatible with the output of PCA/MLP/GCN encoders, forcing gene_encoder=‘none’.#

Solution: Added gene preprocessing parameters (automatic/standard/cropping/no). Automatically selected preprocessing strategy based on input dimensions and source; for original expression, used StandardScaler+Clip, while for predefined embeddings, only Clip was used.

Key Insight: Flexible preprocessing strategies should be determined based on data characteristics (dimensions, sources), rather than being hard-coded. Automatic modes can be intelligently selected based on dimension thresholds (>100). API design should prioritize flexibility over assumptions about data flows—end-to-end training can use pre-encoded features.

3. GCN+STAIG fusion performance is extremely poor (ARI 0.05-0.09) and unstable. In contrast, PCA+STAIG has an ARI of 0.45.#

Solution: Diagnosed in two stages: (1) Compared with the original STAIG implementation, identified 4 algorithm differences (edge discard, feature cropping, edge weight direction, tau decay); (2) Analyzed the original STAIG configuration, identifying hyperparameter mismatch issues (learning rate, number of layers, number of iterations, weight decay).

Key Insight: Correct code logic does not equal correct implementation—algorithm alignment and hyperparameter configuration are crucial for deep learning model performance. Implementing alignment requires line-by-line comparison with reference code, not just reading the paper. The significant performance drop (0.05 vs 0.45) indicates an implementation error, not a hyperparameter adjustment issue.

4. The ARI of MIHD STAIG implementation is 0.05, much lower than the original STAIG project’s 0.452.#

Solution: Compared with the original STAIG’s train_img_config.yaml file, identified key hyperparameter inconsistencies. Need to change the learning rate to 0.0005, number of layers to 1, number of iterations to 300, and weight decay to 0.00001.

Key Insight: Correct code logic does not equal correct implementation—hyperparameters are crucial for deep learning model performance and must be fully aligned with the original paper/project configuration to reproduce results.

5. The implementation of adaptive edge discard differs fundamentally from PyG’s standard dropout_adj—custom logic based on edge probability is required.#

Solution: Implemented a custom adaptiveDropoutAdj function, following the approach in the original STAIG net.py: lines 209-240, where mask = rand() >= edge_prob means edges with higher weights are more likely to be discarded.

Key Insight: The unconventional discard strategy in the original STAIG (discarding similar edges, retaining dissimilar ones) can prevent intra-domain overfitting and enhance cross-domain learning ability.

6. UNI2 causes a position encoding resampling error on GPU without parameters (invalid shape).#

Solution: Passed the complete timmkwargs (including img_size, patch_size, depth, num_heads, etc.) to allow timm to correctly initialize UNI2 model, avoiding reliance on automatic inference.

Key Insight: When loading a pre-trained model, explicitly passing architecture parameters is more reliable than relying on framework automatic inference, especially for shape-sensitive components like position encoding.

Solution: Add ‘gcn’ to the list of options and implement handling for GCN in run_evaluation(), while also dealing with spatial graph construction.

Key Insight: Implementing completeness requires model classes and CLI integration—lacking CLI support makes features invisible to users.

Lessons Learned#

Key Lessons#

  • When implementing published methods, compare line by line with the reference code before claiming consistency—the paper omits key implementation details such as adaptive dropout mechanisms. Algorithm consistency and hyperparameter configuration are crucial for reproducing deep learning model performance.
  • The key to deep learning project replication is achieving complete consistency in hyperparameter configuration—even if the code logic is correct, subtle parameter differences (e.g., learning_rate changing from 0.0005 to 0.001) can lead to a sharp drop in performance (ARI dropping from 0.452 to 0.05).
  • A flexible software architecture requires using strategy patterns at key nodes: the four-mode design of gene_preprocessing (auto/standard/crop/no) shows how modules can support multiple input types without breaking the interface. API flexibility is more important than narrow optimization—end-to-end training should support raw inputs and pre-encoded inputs, rather than forcing one method.
  • A significant performance drop (0.05-0.09 vs 0.45 ARI) indicates an implementation error, not a hyperparameter adjustment issue—comparison with the reference is faster than blind optimization. Ablation studies reveal which changes are truly important—expected improvements should be gradually verified.
  • The choice between end-to-end training and two-stage training affects model performance and training efficiency: the original STAIG used an end-to-end method (training GCN embeddings internally), while MIHD initially used a two-stage method (extracting features first, then training)—this is the core architectural difference.
  • When loading a pre-trained model (e.g., UNI2), explicitly pass architecture parameters rather than relying on framework automatic reasoning, especially for shape-sensitive components like position encoding—clear parameters can avoid runtime errors.

Practical Experience#

  • The best practice for managing environment dependencies is to fix problems incrementally, rather than conducting a comprehensive pre-check: directly run the target program and install missing packages as needed when encountering ModuleNotFoundError—this is more efficient than scanning dependencies in advance.
  • Setting the CUBLAS_WORKSPACE_CONFIG environment variable can solve GPU computing determinism issues, but it must be set early (before importing) and comes with a slight performance penalty.

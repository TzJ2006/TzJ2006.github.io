Problems & Solutions#

Critical Issues#

1. ARI stability analysis would be extremely slow (~30min/section/run) if rerunning entire pipeline (extract+align+fusion+cluster), 10 runs × 11 sections = 330 experiments infeasible#

Solution: Designed architecture to load existing gene/vision embeddings from cache, only rerun fusion+clustering stages. analyze_ari_stability.py directly calls CacheManager to load PCA/UNI2, uses different seeds to initialize STAIGTrainer fusion, then cluster. Single test verified acceleration to ~25 minutes (CPU)/section, GPU estimated 3-5 minutes

Key Insight: Key optimization point for repeated experiments is identifying where randomness exists - MIHD’s vision extraction is deterministic (UNI2 model fixed weights), gene PCA also fixed, only STAIG fusion’s GCN initialization, dropout, mclust random seed affect ARI. Through layered caching (data→gene emb→vision emb→fused emb→clustering) can resume at any stage

2. Collection script requirement ambiguous: ’error scene’ might refer to subtype level (like grasp_misalignment_D0) or specific scene file (like v5_train_xxx_000.json), quota count changed from user’s initial 5 to 10#

Solution: Through ccplan Phase 2 used AskUserQuestion tool for requirement verification: confirmed subtype level, 10 per subtype, pre-recorded MP4 reference videos, preserve complete verification process

Key Insight: Ambiguous requirements must be verified through structured questioning, ccplan’s 9-dimensional intent extraction and AskUserQuestion are key tools, avoid implementing wrong solution

3. MIHD label mixing analysis had predicted labels not matching GT label values, resulting in only 2.6% accuracy#

Solution: Identified that pipeline stored pred_labels are original cluster IDs (not Hungarian matched), while ARI/NMI are permutation invariant so don’t need matching. Applied scipy.optimize.linear_sum_assignment() for Hungarian matching in analyze_label_mixing.py, used matched aligned_pred for visualization functions, accuracy corrected to 49-61%

Key Insight: Permutation invariant metrics like ARI/NMI hide label alignment issues - when generating confusion matrix or per-class metrics must first do Hungarian matching, otherwise will get misleading extremely low accuracy. Pipeline caching design choice to store original cluster IDs is correct (maintains flexibility), but downstream analysis needs to be aware of this

4. TokenMonitor project git pull conflict: local and remote each have 16 commits diverged, 54 file conflicts covering almost all Rust backend modules + core frontend components#

Solution: Through git log –format analyzed both sides’ commit messages and change statistics, identified essence as ‘independent identical modular refactoring + cross-platform support from v0.5.0 common ancestor’. Comparison found remote v0.7.2 contains all functionality of local v0.6.0 plus PR#4 code review fixes, rate limit unification, cache tier pipeline, etc. additional improvements. Decided to use remote and executed git reset –hard origin/main, avoiding manual resolution of 54 file conflicts

Key Insight: When local and remote do large-scale refactoring on same batch of files, merge conflict count doesn’t represent change difference magnitude - might be very similar refactoring. Key at this time is identifying ‘who is superset of whom’ rather than file-by-file comparison. Through version numbers (0.7.2 vs 0.6.0), commit message detail (PR review, bug fix vs simple ‘update’) can quickly judge which side is more complete

5. User wanted skip scene (right-click) or validation failure to not jump to next subtype, should continue collecting in current subtype#

Solution: Modified logic in round-robin loop: changed break (exit current subtype loop) to continue (stay in current subtype, move to next scene)

Key Insight: Round-robin is not just change in traversal order, also involves retry strategy when failed/skipped, need to carefully consider state machine transitions

6. MuJoCo 3.4.0 incompatible with Sawyer robot meshes (volume too small error for nut_assembly and pick_place tasks)#

Solution: Successfully rendered Panda-based tasks (coffee, coffee_preparation, mug_cleanup) first; switched to prepared_source HDF5 files that worked with current environment. Alternative: could use mimicgen_env with older MuJoCo version

Key Insight: Robot model compatibility is environment-dependent; having multiple conda environments with different MuJoCo versions provides fallback options for rendering tasks across different robot types

7. HD RM-IDEAL evaluation needs to support single section data (currently only hd_crop10large), but original benchmark_rm_ideal.py assumes at least 2 sections for cross-section query#

Solution: Check len(sections) before bidirectional cross-section evaluation, if =1 output prompt ‘single section mode: supports within-sample evaluation (query within same section) or cross-dataset evaluation’, don’t execute cross-section loop. Also load_hd_section_string_labels() supports fine_grained parameter to switch between 3 classes (merged CE+Tumor) or 4 classes (original STHD) annotations

Key Insight: Within-sample query is equally valuable in single-sample datasets - can evaluate method’s representation comparability between different niche types within single tissue section. HD data though only has 1 crop, but 17502 spots density is extremely high (8μm bins), sufficient to support statistically significant within-sample RM-IDEAL evaluation

8. Extraction planner currently doesn’t support passing extra_config.source_patch_size to vision extraction job params, limiting HD patch size experiment flexibility#

Solution: Created independent script extract_hd_intermediate_patches.py to bypass pipeline, directly instantiate UNI2VisionEncoder, loop call adata.extract() passing different source_patch_size. Manually manage cache (different patch sizes use different output directories), each size evaluates vision_only/concat/mean three fusion types. This is pragmatic implementation for experimental feature

Key Insight: When pipeline infrastructure doesn’t support certain experimental requirement, independent script vs modifying pipeline trade-off: if one-time exploratory experiment, independent script is faster and doesn’t pollute main pipeline; if recurring requirement, worth investing time in pipeline extension. HD patch size experiment belongs to former - goal is finding optimal size rather than supporting arbitrary size production pipeline

Learnings#

Key Learnings#

  • ccplan skill’s value in complex requirement decomposition: 9-dimensional intent extraction (Task/Input/Output/Constraints/Context/Audience/Memory/Success/Examples) + AskUserQuestion structured questioning + ECL document persistence, can effectively handle ambiguous requirements and ensure cross-session consistency
  • Hungarian matching is key step for converting clustering results to per-class metrics, but easily masked by permutation invariant metrics like ARI/NMI. When generating confusion matrix, per-layer accuracy, spatial misclassification map must first align labels, otherwise will get misleading extremely low accuracy (2.6% vs real 49-61%). Pipeline design should preserve original cluster IDs in cache (maintain flexibility), do matching on-demand in analysis scripts
  • Performance optimization key for repeated experiments is identifying randomness source and layered caching. MIHD’s ARI stability analysis reduced from ~30min/run to ~3-5min/run (GPU), by only rerunning fusion+clustering while reusing vision/gene embeddings. Design principle: deterministic steps (vision extraction) cache results, randomness steps (STAIG GCN initialization, mclust seed) re-execute each time. This applies to any experiments needing parameter sweep/randomness quantification
  • Round-robin collection mode implementation is not just change in traversal order, also involves: (1) state machine design (per-subtype count vs global rounds), (2) failure/skip retry strategy (stay vs move), (3) progress display multi-level aggregation (rounds/subtype/total progress)
  • Error recovery benchmark v5 pipeline architecture: (1) collect_clean_trajectories from demos → (2) scan_opportunities calling skill.can_inject() per frame → (3) schedule_injections sampling from virtual timeline → (4) generate_training_scenes executing injection with InjectionEngine. Each scene stored as JSON metadata + NPZ sim state
  • Large-scale git merge conflict may stem from ‘parallel implementation of identical refactoring’ rather than genuine logic conflicts. Identification method: (1) count similar change magnitude on both sides (23k vs 24k), (2) conflict files all in same modules (Rust commands/platform, Svelte components), (3) commit messages show similar architecture goals (‘modularize backend’, ‘cross-platform support’). At this time key is judging ‘who is superset of whom’ (version number, additional features, bug fix count) rather than file-by-file merge
  • Pipeline extensibility vs independent script decision framework: (1) one-time exploratory experiment (like HD patch size)→independent script, (2) recurring but non-core requirement (like neighborhood query)→optional parameter integration, (3) core functionality (like HD dataset support)→complete pipeline integration. Independent script advantage is quickly verify hypothesis without polluting main code, disadvantage is if later productionized need to rewrite. Key judgment point is ‘will this feature become standard process?’
  • Label mixing analysis revealed spatial omics clustering patterns: Layer_4 (middle cortex) worst accuracy (19.3%) often misclassified as adjacent Layer_3/5, Layer_2 split into Layer_1/3 (40.1%), while outer Layer_1 and WM highest accuracy (83-100%). This reflects biological reality: (1) middle cortical layers have blurred boundaries (neuron type gradients), (2) outer layers have clear anatomical features (Layer_1 is molecular layer, WM is white matter myelin), (3) spot-level embedding based clustering difficult to capture inter-layer gradients. Corresponding solution may need neighborhood-level representation or spatial-aware clustering
  • Robot teleoperation collection interface UX design principles: (1) multi-camera viewpoint assists operation (front+side+wrist), (2) real-time progress feedback (avoid ‘black box’ feeling), (3) reference video reduces learning cost, (4) automatic retry when validation fails (rather than manual restart)
  • Error scene audit importance: before implementing collection process, must audit actual subtype count and scene file count for each task, avoid hardcoding wrong quota or traversal logic
  • MuJoCo EGL rendering pipeline for headless servers requires careful environment configuration: MUJOCO_GL=egl, CUDA_VISIBLE_DEVICES matching MUJOCO_EGL_DEVICE_ID, and compatible robot meshes. MuJoCo 3.4.0 has Sawyer mesh incompatibility; using prepared HDF5 with compatible environments is more reliable than debugging mesh volumes
  • robosuite/mimicgen environment registration: custom _D0 environments (PickPlace_D0, Coffee_D0) require ‘import mimicgen’ before robosuite.make(). State replay from HDF5 uses pre-computed sim states at each frame, enabling exact visual reproduction without action replay or physics determinism concerns
  • id_eligible classification in error scenes: 249/1365 scenes marked as ‘similar to initial distribution’ is by design, not a bug. Error types like drop_in_transit, collision_empty, stuck_no_progress naturally produce table-stable states where object is on table, gripper empty, within workspace. Stack-like tasks have higher id_eligible rates (78/240) due to reset similarity. This enables MimicGen augmentation directly from error scenes
  • Question quality determines analysis depth: in Git conflict case, user asked ‘what problems did local and remote each solve?’ rather than ‘help me merge’ or ‘resolve conflicts’, guided commit history comparison, version evolution analysis, architecture decision trade-off and other deep diagnostics. This demonstrates engineer’s ‘understanding-driven’ mindset - understand problem nature first then choose solution. AI should proactively provide this analysis when encountering ‘direct execution’ type requests, rather than waiting for user to ask

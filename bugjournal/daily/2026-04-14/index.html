Problems and Solutions#

Key Issues#

1. If running the entire process again (extraction + alignment + fusion + clustering), the ARI stability analysis will be very slow (about 30 minutes per part/run). 10 runs × 11 parts = 330 experiments are not feasible.#

Solution: Designed an architecture that loads existing gene/visual embeddings from the cache and only reruns the fusion and clustering stages. analyze_ari_stability.py directly calls CacheManager to load PCA/UNI2, initializes STAIGTrainer with different seeds for fusion, and then performs clustering. Single-test verification has been accelerated to about 25 minutes per part (on CPU), with GPU estimates ranging from 3-5 minutes.

Key Insight: The key optimization for repeated experiments is to identify the presence of randomness—the visual extraction in MIHD is deterministic (UNI2 model fixes weights), and gene PCA also fixes weights. Only the GCN initialization, dropout, and mclust random seeds for STAIG fusion affect ARI. By using a hierarchical cache (data → gene embeddings → visual embeddings → fusion embeddings → clustering), it’s possible to recover at any stage.

2. The requirements for the collection script are unclear: “error scenarios” might refer to sub-type levels (e.g., grasp_misalignment_D0) or specific scenario files (e.g., v5_train_xxx_000.json). The quota has been changed from the user’s initial 5 to 10.#

Solution: Used the AskUserQuestion tool in the second phase of ccplan to verify the requirements: confirmed sub-type levels, 10 for each sub-type, and pre-recorded MP4 reference videos to maintain the full verification process.

Key Insight: Unclear requirements must be verified through structured questioning. The 9-dimensional intent extraction of ccplan and AskUserQuestion are key tools to avoid implementing incorrect solutions.

3. The labels mixed in MIHD analysis do not match the GT label values, resulting in an accuracy of only 2.6%.#

Solution: It was found that the predicted labels stored in the process are the original cluster ID (not Hungarian matching), while ARI/NMI are invariant permutations. Therefore, no matching is required. Applied scipy.optimize.linear_sum_assignment() in analyze_label_mixing.py for Hungarian matching, and used the matched aligned_pred for visualization functions, improving accuracy to 49-61%.

Key Insight: Invariant metrics like ARI/NMI can hide label alignment issues—before generating confusion matrices or metrics for each class, Hungarian matching must be performed. The choice to store the original cluster ID in the process cache was correct (maintained flexibility), but downstream analysis needs to be aware of this.

4. There was a git pull conflict in the TokenMonitor project: there were 16 differences between the local and remote commits, with 54 files conflicting involving almost all Rust backend modules and core frontend components.#

Solution: Analyzed both commit messages and change statistics using git log –format, determining that the issue was essentially “independent modular refactoring with the same base from v0.5.0 onwards”. After comparison, it was found that the remote v0.7.2 included all features of the local v0.6.0, along with code review fixes, rate limiting unification, and additional improvements like the cache layer process. Decided to use the remote version and executed git reset –hard origin/main to avoid manually resolving the 54 file conflicts.

Key Insight: When local and remote make large-scale refactoring changes to the same set of files, the number of merge conflicts does not reflect the extent of the changes—they may be very similar. At this point, it’s important to identify “which one is a subset of the other”, rather than comparing files individually. By using version numbers (0.7.2 vs 0.6.0) and commit details (PR reviews, errors fixed, or simple “updates”), it’s possible to quickly determine which side is more complete.

5. Users want to skip scenarios (right-click) or not jump to the next sub-type if the verification fails. They should continue collecting in the current sub-type.#

Solution: Modified the loop logic: changed the interruption (exiting the current sub-type loop) to continuation (staying in the current sub-type and moving to the next scenario).

Key Insight: Loops involve not just the order of traversal, but also retry strategies after failures—this requires careful consideration of state machine transitions.

6. MuJoCo 3.4.0 is not compatible with Sawyer robot meshes (problems with nut_assembly and pick_place tasks due to small volumes).#

Solution: First, successfully rendered tasks based on Panda (coffee, coffee preparation, cup cleaning); then switched to prepared HDF5 files suitable for the current environment. Another option was to use an older version of MuJoCo’s mimicgen_env.

Key Insight: Robot model compatibility depends on the environment; having multiple conda environments with different MuJoCo versions can provide alternative solutions for different robot tasks. Solution: Checked the value of len(sections) before starting the bidirectional cross-valuation. If it equals 1, output a prompt saying “Single-part mode: supports queries within the same part or cross-dataset evaluation”, and do not execute the cross-valuation loop. Additionally, the load_hd_section_string_labels() function supports fine-grained parameters, allowing switching between 3 types of CE+tumor data annotations and 4 types of original STHD annotations.

8. The current extraction planner does not support passing the extra_config.source_patch_size parameter to the visual extraction job parameters, which limits the flexibility of high-definition patch sizes in experiments.#

Solution: Created a separate script extract_hd_intermediate_patches.py to bypass the pipeline process, directly instantiating UNI2VisionEncoder and passing a different source_patch_size parameter when calling data_extract(). Manually managed the cache (using different output directories for different patch sizes), and performed visual detection/merging/mean fusion for each size. This is a practical implementation method for experimental features.

Key Insight: When the pipeline infrastructure cannot support certain experimental requirements, it’s better to use independent scripts instead of modifying the pipeline process. For one-time exploratory experiments, using independent scripts is faster and does not contaminate the main pipeline process. High-definition patch size experiments belong to the former category—the goal is to find the best size, not to support arbitrary sizes in the production pipeline process.

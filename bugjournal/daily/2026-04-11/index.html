Problems & Solutions#

Critical Issues#

1. All MimicGen augmentation verifications failed (0/10 scenes), cubeA not lifted (height 0.82 desktop)#

Solution: Multiple rounds of debugging revealed the root cause: subtask fragmentation (6 subtasks = 5 independent warp boundaries, cumulative errors) + no post-grasp buffer + gripper detection bug (find continuous closure rather than first-time threshold exceedance) + retract subtask not warp + path parsing bug. Fixed issues one by one: simplified to 2-subtask, added 15-frame buffer, improved gripper detection, merged retract into re_acquire, fixed path parsing

Key Insight: Original MimicGen design wisdom: minimize the number of subtasks (2) + post-grasp buffer + start from home position. Each subtask boundary is a source of error, so should be minimized as much as possible

2. success_demos.hdf5 could not serve as MimicGen source — lacked env_interface_name and env_interface_type attributes#

Solution: Added ensure_env_interface attrs() helper to automatically complete attributes in filter_valid_source, augment_success_demos, generate_success_from_error; patched HDF5 files for 7 tasks

Key Insight: MimicGen DataGenerator relies on specific datagen_info attributes to identify environment interfaces. Missing these attributes causes get_env_interface_info_from_dataset failure; integrity must be ensured at each output point in the generation chain

3. BetterSSH front-end buttons were unresponsive — WebSocket messages were silently discarded#

Solution: Traced back to null return of onMessage() placeholder in index.ts. Instantiated SessionManager, loaded SSH/BetterSSH configurations, connected to the complete message route (get_sessions, create_session, destroy_session, send_input, resize)

Key Insight: The backend had a fully implemented SessionManager (500+ LOC) but never instantiated it — a code skeleton trap, interfaces existed but never connected

4. Token usage data disappeared when deleting JSONL logs#

Solution: Implemented hourly append archiving under {app_data_dir}/usage-archive/, using time boundary partitioning (archive covered [0..frontier], live source covered (frontier..now)) to prevent duplicate counting

Key Insight: Time boundary partitioning eliminated overlap between archiving and real-time data, ensuring zero duplicate statistics with minimal storage overhead (about 1KB/day/provider)

5. Claude Code frequently requested permission confirmation in exploratory projects, affecting development efficiency#

Solution: Simplified the 80+ item-by-item enumeration Bash command permissions into a tool-level wildcard “Bash”, covering all Bash commands

Key Insight: Claude Code supported tool-level wildcards (“Bash” matching all Bash commands), allowing relaxed permissions with minimal configuration. Dangerous operations were still subject to confirmation mechanisms through an ask array

6. Adversarial review found WebSocket PTY writes without authentication (ATK-007)#

Solution: Added NEWREQ-009: binding to 127.0.0.1 + 256-bit session token (randomly generated and printed at startup), all WS upgrades required a query string token

Key Insight: Security issues were least costly to fix during the requirements phase. Red-blue confrontation should be enforced in Phase 8 (after architecture is determined, before coding starts)

7. OpenReview API returned 403 Forbidden, preventing access to NeurIPS paper review data#

Solution: Diagnosed double-layer blockages: (1) OpenReview API now forced all requests to require a authentication token (anonymous access disabled), (2) Frequent probing triggered nginx rate limits. Used the user OpenReview credentials (tongtongtot@163.com) to log in through openreview-py SDK, successfully retrieved 61 reviews (7 oral papers: 29 reviews + 7 AC decisions; 8 Spotlight papers: 32 reviews). Added delays between requests to avoid rate limiting

Key Insight: OpenReview’s authentication strategy had changed — even public content API access required a login token. The /login endpoint remained accessible (returned a 400 verification error) while all data endpoints returned 403, indicating that application-level authentication was enforced. Web front-end work used internal credentials due to Next.js RSC server-side rendering

8. Terminal rendering quality was poor — fonts were blurry, scrolling was slow, middle button pasting failed#

Solution: Identified xterm.js as the industry standard (used by VS Code/Replit), and the problem was lack of plugins. Added WebGL renderer (GPU acceleration), Unicode11 (CJK), improved theme (Catppuccin Mocha), better font stack (Cascadia Code fallback)

Key Insight: xterm.js alone was insufficient — production terminals required a complete plugin ecosystem. WebGL provided a 10-fold rendering improvement and clear fonts. Unicode11 prevented CJK width errors

9. GLM5.1 and DeepSeek-Reasoner displayed a price of $0.00, despite the OpenRouter integration being added#

Solution: Added version field to the PricingCache structure (CACHE_VERSION=2), making outdated caches (only containing LiteLLM data) automatically obsolete when new pricing sources were introduced

Key Insight: The caching system needed explicit version control to handle mode evolution — only relying on TTL allowed old data to exist after code changes

10. Terminal session switching discarded all terminal content — no rollback history#

Solution: Changed the terminal lifecycle: created an xterm instance once in useEffect([], []) and reused across sessions. When switching: term.clear() → subscribe to new session → backend sent RingBuffer回填 → term.write()

Key Insight: Terminal instances were expensive and required saving rollback states — should not be destroyed when switching sessions. The 2MB backend RingBuffer provided a replay buffer

Human vs AI Approaches#

Strategic Level#

MimicGen Augmentation Strategy — Repeated error injection vs Fixed Error Scene + Reset Randomization#

Role Approach
Human Humans identify the fundamental problems in the current logic: Repeating error injection for each target scene results in lack of diversity, which does not conform to the original MimicGen design. An error scene should be treated as a fixed base scenario, and variants should be generated by randomizing objects using env.reset()
AI Initially, AI only sees the details of the code implementation (_replay_with_mimicgen_targets), without questioning the design from the perspective of the augmentation strategy. It is only after humans point out the problems that AI understands that the reset logic from the original MimicGen should be moved to the error scene

Difference Analysis: Humans consider “what is a reasonable way to augment” from the systematic design level, while AI tends to accept the existing implementation and optimize within its framework. This represents a difference at the strategic vs tactical level.

AI Limitations#

Critical Limitations#

  • Did not proactively suggest research: After AI’s design was completed, it was ready to go directly, without suggesting “research existing solutions first”. This requires human reminder. Engineers should have the awareness that “reusing is better than rewriting”.
  • Insufficient demand-digging: When users say “unwilling to switch windows”, AI only focuses on remote SSH scenarios and fails to infer that local CLI also needs unified management. Strong scenario integrity reasoning is needed.
  • It took 6 rounds of diagnosis to locate all MimicGen bugs: Subtask fragmentation → retract not warping → gripper detecting errors → insufficient interpolation steps → parallel path bugs → cache cleanup. AI cannot anticipate all problems at once.
  • Failed to access the OpenReview API upon first attempt, without realizing recent authentication policy changes. All 30+ attempts (API v1/v2, different endpoints, visitor login, various headers) returned a 403, until the user provided credentials. The system lacks the latest knowledge of OpenReview authentication requirements for 2026.
  • Failed to detect that SessionManager was never instantiated during initial architecture scanning — it took a lot of time to debug port conflicts and WebSocket issues before the root cause was discovered to be an empty onMessage handler.
  • AI failed to question the rationality of “repeated error injection” design when analyzing, and required humans to explicitly point out “fixed error scene + reset randomization” to start refactoring.
  • Incorrect path assumption: The Agent initially assumed that files were edited under recovery_collection_portable/, but users had removed one level. All edits were lost and needed to be re-applied in the new path.
  • Did not initially recognize that an outdated pricing cache could prevent OpenRouter data loading — it was only discovered through explicit code tracing using litellm.rs cache logic that there was a version control gap.

General Limitations#

  • Triggered OpenReview nginx rate limiting, resulting in temporary IP bans (152.3.43.42) after approximately 30 consecutive requests during the diagnosis phase. Exponential backoff or diagnostic authentication requirements should have been implemented earlier.
  • Subagent execution process is not transparent: When 5 backend agents are running concurrently, users cannot see intermediate progress (only “waiting for completion notification”), lacking real-time visibility. Progress awareness is a UX pain point for large tasks.

Learnings#

Key Learnings#

  • The power of the Superpowers skill chain: Brainstorming (requirement analysis) → ccplan (ECL-driven implementation) → optimize (quality improvement) forms a complete software lifecycle. ECL documents serve as cross-session persistent “design contracts”, ensuring context consistency across multiple rounds of dialogue, multiple agents, and even different models.
  • The value of adversarial review: Among the 9 issues discovered by the Critic agent, 2 are CRITICAL (no authentication PTY writing, SSH parameter injection) and 4 are HIGH (token brute-force cracking, prototype contamination), which could cause security disasters if exposed in a production environment. Repairs at the requirement stage cost « code stage « production stage.
  • The essence of object-centric warping: EEF relative to object motion patterns is an invariant. This invariant is maintained through coordinate transformation T_eef_new = T_obj_target × T_obj_src⁻¹ × T_eef_source. Two-step understanding: (1) Eliminate the absolute position of the source object, (2) Remap with the new object’s position.
  • The wisdom behind the original MimicGen design: (1) Minimize the number of subtasks (only 2 for Stack), (2) Use a post-grasp buffer of 10-20 frames to ensure stable capture, (3) Reduce pose gaps by using env.reset() to home position, (4) Execute in a quasi-closed-loop manner (calculate delta for target_pose_to_action step by step).
  • The core design of MimicGen is source demo + env.reset() randomization: An error scene should be treated as a fixed “initial state”, and then variants should be generated using placement_initializer, rather than injecting errors repeatedly for each target.
  • Priority of NeurIPS DB Track reviews (based on 61 reviews): The most valued qualities are “comprehensive” (mentioned 17 times), “community” (17 times), “novel” (15 times), “clear” (28 times), and “well-designed” (13 times). The most common criticisms are “lack of comparison” (insufficient baseline), “insufficient detail” (unclear methodology), and “limited scope” (restricted application). The best paper (Artificial Hivemind) has an average score of 4.75 for Oral papers — this exceeds the review scores determined by AC. The average for Oral is 5.00 < Spotlight average is 5.16, confirming that scoring is not the only factor considered.
  • Parallel agent scheduling mode: Assign independent modules (Config/Parser/LocalPTY) to different subagents for parallel implementation, then converge and verify, improving development efficiency by 3-5 times. The key is identifying non-dependent leaves in the DAG, maximizing parallelism.
  • Claude Code –output-format stream-json: Through research, it was found that Claude natively supports structured output (newline-separated JSON), eliminating the need for regular parsing of PTYs. This improves the reliability of the Parser from “heuristic” to “deterministic”. Always research before reinventing.
  • Return on investment for TypeScript strict mode: 195 tests + 0 type errors = production-ready code. The type system is particularly effective when refactoring (after extracting parsed-state-utils, all calling points automatically get type checked). Spending 20% more time on typing in the early stages saves 80% of debugging time later.
  • xterm.js is the industry-standard browser terminal (used by VS Code, Replit, Vercel, ttyd). Its quality depends on a complete plugin stack: WebGL for GPU rendering, Unicode11 for CJK, Fit for resizing, SearchAddon, WebLinksAddon. Just the basic xterm.js is not sufficient for production use.
  • The MimicGen-ready dataset requires a specific metadata structure: Each demo_*/datagen_info must contain env_interface_name and env_interface_type, otherwise DataGenerator cannot initialize the environment correctly.
  • Human demonstration data quality verification should be carried out before the collection phase: Simply having teleop success is not enough; replay stability and augmentability must also be verified. Otherwise, a large amount of data that cannot be used for training will be collected.
  • Time boundary partitioning (archive completion hours, live source coverage current hours) elegantly solves duplicate removal without requiring explicit trace at the record level — zero overlap means zero duplication count.
  • Claude Code’s permission system supports tool-level wildcards (such as “Bash” matching all Bash commands), allowing for minimal configuration instead of dozens of item-by-item rules.
  • The cost of Subtask fragmentation: Each subtask boundary is an independent “check object pose → re-warp → 5-step interpolation”, and errors will accumulate. 6 subtasks = 5 error sources, far more than the original one.
  • OpenReview API authentication architecture (2026): All data endpoints require login with a token; only the /login endpoint allows anonymous access (returning a verification error). The web frontend uses server-side credentials through Next.js RSC. Rate limiting is operated at the nginx level (based on IP), triggered in a short time after approximately 30 requests. The Academic Papers Review API is becoming increasingly locked down, requiring institutional or personal credentials.
  • Terminal resize synchronization requires full-chain coordination: ResizeObserver (frontend) → term.onResize → WebSocket message → SessionManager → Pty.resize(). Mismatches lead to errors in the error column. Damping is needed to prevent flooding when resizing with drag-and-drop.
  • Terminal instances should not be destroyed during session switching — they retain a rollback buffer (10,000 lines). The correct pattern: Create once and backfill with clear() + backend RingBuffer (2MB) for cross-session reuse.
  • A cache system must include explicit version fields when the data source can evolve — just TTL expiration is insufficient to handle changes in patterns/sources.
  • The permission configuration allow/deny/ask three-tier mechanism can balance security and efficiency: Allow implements automation, ask retains confirmation for dangerous operations, deny is used for absolute prohibition.
  • Claude Code configuration does not automatically inherit (each project requires independent configuration), but a standardized initialization process can be implemented through an automated script (such as init-claude.sh).
  • Complex robot system documentation should distinguish between developers and operators: Clearly define the target audience, structure, language, and content retention through structured questions to prevent ambiguity.
  • React closure trap ref pattern: Registering third-party library callbacks in useEffect for xterm.js onData captures expired props is a common bug. The general solution is: props → ref, useEffect synchronizes with ref.current, and the callback reads ref.current. This is the best practice in the React ecosystem.
  • Three key elements for WebSocket robustness: Redial (exponential backoff), backpressure (checking bufferedAmount), and heartbeat (not implemented but should be considered). The network layer of distributed systems must assume that “connections may disconnect at any time”, not “connections should remain stable”.
  • The importance of continuous closure detection: Robot capture may have multiple attempts. If gripper_closed > threshold for the first attempt fails, it must detect continuous closure (≥10 frames) to confirm a successful capture.
  • Complexity of multi-process debugging: (1) Each worker requires an independent env+augmenter instance, (2) Path resolution is more error-prone (relative paths are likely to miss levels), (3) Absolute paths like demos_dir must be explicitly passed in, (4) Robosuite WARNING needs to be suppressed separately in the worker function.
  • Quota counting logic should reflect actual availability: augmentable_only is stricter but more reasonable than replay-only, as a non-augmentable demo cannot expand the training set even if it can be replayed.
  • State of the art ASR models (2026): True streaming models are rare. Moonshine v2 (73-107 ms latency, 34-245 M parameters) and Voxtral-Mini-4B-Realtime (480 ms, 4B parameters) represent the most advanced. Moonshine uses a callback-based API suitable for real-time transcription; Voxtral requires manual chunking but offers higher accuracy. Both support Apple Silicon native acceleration (Moonshine uses CoreML, Voxtral uses MLX).
  • Permission requirements differ between exploratory projects and production projects: Exploratory projects prioritize efficiency (lenient permissions), while production projects prioritize security (strict confirmation).
  • Gradually accumulating configurations can lead to inconsistencies (e.g., 80+ long rules, different project styles), but regularly using templates for standardization can improve maintainability.

Problems & Solutions#

Critical Issues#

1. All MimicGen augmentation verifications failed (0/10 scenes), cubeA not lifted (height 0.82 desktop)#

Solution: Multiple rounds of debugging revealed the root cause: subtask fragmentation (6 subtasks = 5 independent warps, cumulative errors) + no post-grasp buffer + gripper detection bug (find continuous closure instead of first-time threshold exceeded) + retract subtask did not warp + path parsing bug. Fixed each issue: simplified to 2-subtasks, added 15-frame buffer, improved gripper detection, merged retract into re_acquire, and fixed path parsing. Key Insight: The original MimicGen design was smart: minimized the number of subtasks (2) + post-grasp buffer + starting from the home position. Each subtask boundary was a source of error, so this should be minimized.

2. success_demos.hdf5 could not be used as a MimicGen source—missing env_interface_name and env_interface_type attributes#

Solution: Added the ensure_env_interfaceattrs() helper to automatically complete attributes in filter_valid_source, augment_success_demos, and generate_success_from_error. Patched 7 tasks’ HDF5 files. Key Insight: MimicGen DataGenerator relied on specific datagen_info attributes to identify environment interfaces. Missing these attributes caused get_env_interface_info_from_dataset to fail. Integrity must be ensured at every output point in the generation chain.

3. BetterSSH front-end buttons were unresponsive—WebSocket messages were silently discarded#

Solution: Traced back to the onMessage() placeholder in index.ts that returned null. Instantiated SessionManager, loaded SSH/BetterSSH configurations, and connected to the complete message routing (get_sessions, create_session, destroy_session, send_input, resize). Key Insight: The backend had a complete implementation of SessionManager (500+ LOC) but never instantiated it—a code skeleton trap, the interface existed but never connected.

4. Token usage data disappeared when deleting JSONL logs#

Solution: Implemented an hourly aggregate archive under {app_data_dir}/usage-archive/, using time boundary partitioning (archive covered [0..frontier], live source covered (frontier..now)) to prevent duplicate counting. Key Insight: Time boundary partitioning eliminated overlap between archives and real-time data, ensuring zero duplicate statistics and minimal storage overhead (about 1KB/day/provider).

5. Claude Code frequently asked for permission confirmation in exploratory projects, affecting development efficiency#

Solution: Simplified the 80+ Bash command permissions to a tool-level wildcard “Bash”, covering all Bash commands. Key Insight: Claude Code supported tool-level wildcards (“Bash” matching all Bash commands), allowing relaxed permissions with minimal configuration. At the same time, a confirmation mechanism was provided for dangerous operations through the ask array.

6. Adversarial review found WebSocket PTY writes without authentication (ATK-007)#

Solution: Added REQ-009: bind 127.0.0.1 + 256-bit session token (randomly generated and printed at startup), and verified the query string token for all WS upgrades. Key Insight: Security issues can be fixed at the lowest cost during the requirements phase. Blue-Red confrontation should be enforced in Phase 8 (after the architecture is determined and before coding starts).

7. OpenReview API returned 403 Forbidden, preventing access to NeurIPS paper review data#

Solution: Diagnosed double-blocking: (1) OpenReview API now requires authentication tokens for all requests (anonymous access was disabled) and (2) frequent probes triggered nginx rate limiting. Used the user OpenReview credentials (tongtongtot@163.com) to log in through the openreview-py SDK, successfully retrieving 61 reviews (7 oral presentations: 29 reviews + 7 AC decisions; 8 Spotlight papers: 32 reviews). Added delay between requests to avoid rate limiting. Key Insight: OpenReview’s authentication policy has changed—even accessing public content APIs requires a login token. The /login endpoint remains accessible (returning 400 verification error) while all data endpoints return 403, indicating that application-level authentication is enforced. The Web frontend worked using internal credentials due to Next.js RSC server rendering.

8. Terminal rendering quality was poor—font was blurry, scrolling was slow, middle button paste function was ineffective#

Solution: Identified that xterm.js was the industry standard (used by VS Code/Replit), and the problem was the lack of plugins. Added a WebGL renderer (GPU acceleration), Unicode11 (CJK), improved theme (Catppuccin Mocha), and better font stack (Cascadia Code fallback chain). Key Insight: xterm.js alone was insufficient—production terminals required a complete plugin ecosystem. WebGL provided a 10-fold rendering improvement and clear fonts. Unicode11 prevented CJK width errors.

9. GLM5.1 and DeepSeek-Reasoner displayed a price of $0.00, even though OpenRouter integration was added#

Solution: Added a version field to the PricingCache structure (CACHE_VERSION=2), making outdated caches (only containing LiteLLM data) automatically invalid when a new pricing source was introduced. Key Insight: The caching system needed explicit version control to handle changes in the mode of evolution—only relying on TTL allowed outdated data to remain after code changes.

10. Switching sessions in the terminal discarded all terminal contents—no rollback history#

Solution: Changed the terminal lifecycle: created an xterm instance once in the useEffect([], []) and reused across sessions. When switching: term.clear() → subscribe to new session → backend sends RingBuffer to fill the buffer → term.write() Key Insight: Terminal instances are expensive and preserve rollback state—should not be destroyed when switching sessions. The backend RingBuffer (2MB) provides a replay buffer.

MimicGen Augmentation Strategy – Repeated error injection vs Fixed error scene + Reset randomization#

Role Approach
Human Humans identify the fundamental problem with the current logic: re-injecting errors into each target scene results in lack of diversity, which does not conform to the original MimicGen design. An error scene should be treated as a fixed base scenario, and variants should be generated by randomizing objects using env.reset()
AI Initially, AI only focuses on the details of the code implementation (_replay_with_mimicgen_targets), without questioning the design from the perspective of the augmentation strategy. It is only after humans point out the problems that AI understands that the reset logic from the original MimicGen should be applied to the error scenes

Difference Analysis: Humans consider the “reasonable way to augment” from the perspective of system design, while AI tends to accept the existing implementation and optimize within its framework. This represents a difference between strategic and tactical levels of thinking.

  • Insufficient Inferring Implicit Requirements: When users said “unwilling to switch windows”, AI only focused on the remote SSH scenario, failing to infer that local CLI also needed unified management. Stronger scenario integrity reasoning is needed.
  • It took 6 rounds of diagnostics to locate all MimicGen bugs: subtask fragmentation → retract not warp → gripper detects errors → insufficient interpolation steps → parallel path bug → cache cleanup. AI cannot anticipate all problems at once.

  • Triggering OpenReview nginx rate limiting, resulting in temporary IP ban (152.3.43.42) after approximately 30 consecutive requests during the diagnostic phase. Exponential backoff or diagnostic authentication requirements should have been implemented earlier.

Learnings#

Key Learnings#

  • The power of the Superpowers skill chain: Brainstorming (requirement analysis) → ccplan (ECL-driven implementation) → optimize (quality improvement) forms a complete software lifecycle. ECL documents serve as cross-session persistent “design contracts”, ensuring context consistency across multiple dialogues, agents, and even different models.
  • The value of adversarial review: Among the 9 issues identified by the Critic agent, 2 are CRITICAL (no authentication PTY writing, SSH parameter injection) and 4 are HIGH (token brute-force cracking, prototype contamination), exposing them in a production environment would cause a security disaster. Repairs in the requirement phase cost less than in the code phase and production phase.
  • The essence of object-centric warping: EEF relative to object motion patterns is an invariant. This invariant is maintained through coordinate transformation T_eef_new = T_obj_target × T_obj_src⁻¹ × T_eef_source. There are two steps to understand this: (1) eliminating the absolute position of the source object (2) remapping with the new object’s position.
  • The wisdom of the original MimicGen design: (1) minimizing the number of subtasks (only 2) (2) having a post-grasp buffer of 10-20 frames to ensure stable capture (3) reducing the pose gap with env.reset() to home position (4) quasi-closed-loop execution (calculating delta for target_pose_to_action step by step)
  • The core design of MimicGen is source demo + env.reset() randomization: An error scene should be treated as a fixed “initial state”, and then variants should be generated using placement_initializer, rather than injecting errors into each target again.
  • Priority of NeurIPS DB Track reviews (based on 61 reviews): The most valued qualities are “comprehensive” (mentioned 17 times), “community” (17 times), “novel” (15 times), “clear” (28 times), and “well-designed” (13 times). The most common criticisms are “lack of comparison” (insufficient baseline), “insufficient detail” (unclear methodology), and “limited scope” (restricted application). The best paper (Artificial Hivemind) has an average score of 4.75 among Oral papers — AC decisions override review scores. The average for Oral is 5.00 < Spotlight average is 5.16, confirming that score is not the only factor in acceptance.
  • Parallel agent scheduling mode: Independent modules (Config/Parser/LocalPTY) are assigned to different subagents for parallel implementation, then combined for verification, improving development efficiency by 3-5 times. The key is to identify dependent nodes in the DAG that are not dependent, maximizing parallelism.
  • Claude Code –output-format stream-json: Through research, it was found that Claude natively supports structured output (JSON separated by line breaks), eliminating the need for regular expression parsing of PTYs. This improves the reliability of the Parser from “heuristic” to “deterministic”. Always research before reinventing.
  • Return on investment of TypeScript strict mode: 195 tests + 0 type errors = production-ready code. The type system is particularly effective during refactoring (after extracting parsed-state-utils, all call points automatically get type checked). Spending 20% more time on typing in the early stages saves 80% of debugging time later.
  • xterm.js is the industry-standard browser terminal (used by VS Code, Replit, Vercel, ttyd). Its quality depends on the complete plugin stack: WebGL for GPU rendering, Unicode11 for CJK, Fit for resizing, SearchAddon, WebLinksAddon. Basic xterm.js alone is not sufficient for production use.
  • MimicGen-ready datasets require specific metadata structure: Each demo_*/datagen_info must contain env_interface_name and env_interface_type, otherwise DataGenerator cannot correctly initialize the environment.
  • Human demonstration data quality verification should be done before the collection phase: Simply achieving teleop success is not enough; replay stability and augmentability must also be verified. Otherwise, a large amount of data will be collected that cannot be used for training.
  • Time boundary partitioning (archive covers completed hours, live source covers current hours) elegantly solves deduplication without requiring explicit level tracking — zero overlap means zero duplication count.
  • Claude Code’s permission system supports tool-level wildcards (such as “Bash” matching all Bash commands), allowing minimal configuration instead of dozens of individual rules.
  • Cost of Subtask Fragmentation: Each subtask boundary involves independently “checking object pose → rewarp → 5-step interpolation”, and errors can accumulate. 6 subtasks = 5 error sources, far more than the original 1.
  • OpenReview API authentication architecture (2026): All data endpoints require login with a token; only the /login endpoint allows anonymous access (returning a verification error). The web frontend uses server-side credentials through Next.js RSC. Rate limits are operated at the nginx layer (based on IP), triggering approximately 30 requests in a short period. The academic paper review API is becoming increasingly locked, requiring institutional or personal credentials.
  • Terminal resize synchronization requires full-chain coordination: ResizeObserver (frontend) → term.onResize → WebSocket message → SessionManager → PTY.resize(). Mismatches result in lines being broken in the error column. Anti-drag prevents flooding when resizing.
  • Terminal instances should not be destroyed during session switching — they save a rollback buffer (10,000 lines). The correct approach is to create once and reuse it through clear() + backend RingBuffer (2MB) across sessions.
  • Cache systems must include explicit version fields when data sources can evolve — just TTL expiration is not enough to handle changes in patterns/sources.
  • The three-layer permission configuration allow/deny/ask can balance security and efficiency: Allow implements automation, ask retains confirmation for dangerous operations, deny is used for absolute prohibition.
  • Claude Code configuration does not inherit automatically (each project requires independent configuration), but can be standardized through automated scripts (such as init-claude.sh) for standard initialization processes.
  • Documents for complex robot systems should distinguish between developers and operators: Clearly define the target audience, structure, language, and content retention through structured questions to avoid ambiguity.
  • React closure trap ref pattern: Registers third-party library callbacks (xterm.js onData) in useEffect, which captures expired props is a common bug. The general solution is: props → ref, useEffect synchronizes with ref.current, and the callback reads ref.current. This is the best practice in the React ecosystem.
  • Three key elements for WebSocket robustness: Exponential backoff, backpressure (checking bufferedAmount), and heartbeat (not implemented but should be considered). The network layer of distributed systems must assume “connections may disconnect” rather than “connections should be stable”.
  • Importance of continuous closure detection: Robot grabs may attempt multiple times, and the first gripperClosed > threshold may fail. Continuous closure (≥10 frames) must be detected to confirm a successful grab.
  • Complexity of multi-process debugging: (1) Each worker requires an independent env+augmenter instance (2) Path resolution is more prone to errors (relative paths may miss levels) (3) Absolute paths like demos_dir must be explicitly passed (4) Robosuite WARNINGs need to be suppressed separately in worker functions.
  • Quota counting logic must reflect actual availability: augmentable_only is stricter than replay_only but more reasonable, as non-augmentable demos cannot expand the training set even if they can be replayed.
  • State of the art ASR models (2026): Real-time streaming models are rare. Moonshine v2 (73-107ms latency, 34-245M parameters) and Voxtral-Mini-4B-Realtime (480ms, 4B parameters) represent the most advanced. Moonshine uses a callback-based API suitable for real-time transcription; Voxtral requires manual chunking but offers higher accuracy. Both support native acceleration for Apple Silicon (Moonshine uses CoreML, Voxtral uses MLX).
  • Permission requirements differ for exploratory and production projects: Exploratory projects prioritize efficiency (lenient permissions), while production projects prioritize security (strict confirmation).
  • Gradually accumulating configurations can lead to inconsistencies (e.g., 80+ long rules, different project styles), and regularly using templates for standardization can improve maintainability.

Human vs AI Approaches#

Strategic Level#

MimicGen Augmentation Strategy – Repeated error injection vs Fixed error scene + Reset randomization#

Role Approach
Human Humans identify the fundamental problem with the current logic: re-injecting errors into each target scene for diversity, which does not conform to the original MimicGen design. An error scene should be treated as a fixed base scenario, and variants should be generated through env.reset() with random objects.
AI AI initially only examines the code implementation details (_replay_with_mimicgen_targets) and does not question the design from the ‘augmentation strategy’ perspective. It takes humans to point out the problem before AI understands that the original MimicGen’s reset logic should be applied to the error scene.

Difference Analysis: Humans consider ‘what constitutes reasonable augmentation’ from a system design perspective, while AI tends to accept the existing implementation and optimize within its framework. This reflects a difference between strategic and tactical levels of thinking.

Whether to Research Existing Solutions#

Role Approach
Human After completing the design presentation, humans actively suggest ‘I need you to search online for existing solutions’, avoiding duplicating efforts.
AI AI initially prepares to proceed directly to the implementation phase (writing design documents → assigning writing plans), without suggesting research.

Difference Analysis: Humans demonstrate pragmatic engineering thinking (reusing is better than rewriting), while AI tends to fall into the habit of ‘building from scratch’ when not prompted. This is one of AI’s blind spots.

AI Limitations#

Critical Limitations#

  • Failure to Propose Technical Research: AI is ready to implement the design but does not suggest researching existing solutions first. Engineers lack the awareness that ‘reusing is better than rewriting’.
  • Inadequate Inferring Implicit Requirements: When users say ‘unwilling to switch windows’, AI only focuses on remote SSH scenarios and fails to infer that local CLI also requires unified management. Stronger scenario completeness reasoning is needed.
  • It took 6 rounds of diagnostics to locate all MimicGen bugs: subtask fragmentation → retract not warp → gripper detects errors → insufficient interpolation steps → parallel path bugs → cache cleanup. AI cannot anticipate all issues at once.
  • First attempt to access the OpenReview API failed, unaware of recent authentication policy changes. All 30+ attempts (API v1/v2, different endpoints, guest login, various headers) returned a 403, until the user provided credentials. The system lacks the latest knowledge of OpenReview authentication requirements for 2026.
  • Failed to detect that SessionManager was never instantiated during the initial architecture scan—it took a lot of time to debug port conflicts and WebSocket issues before the root cause was identified as an empty onMessage handler.
  • AI initially fails to question the rationality of ‘repeated error injection’ design. Humans need to explicitly state ’error scenes should be fixed + reset randomization’ to start refactoring.
  • Incorrect Path Assumption: The Agent initially assumed files were edited under recovery_collection_portable/, but the user had moved to another level. All edits were lost and need to be reapplied in the new path.
  • Not initially recognizing that outdated pricing caching could prevent OpenRouter data loading—it took the litellm.rs cache logic to explicitly trace the version control gap.

General Limitations#

  • Triggering OpenReview nginx rate limiting, resulting in temporary IP ban (152.3.43.42) after approximately 30 consecutive requests during the diagnostic phase. Exponential backoff or diagnostic authentication requirements should be implemented earlier.
  • Subagent execution process is not transparent: When 5 backend agents run in parallel, users cannot see the intermediate progress (only ‘waiting for completion notification’), lacking real-time visibility. Progress tracking is a UX pain point for large tasks.

Learnings#

Key Learnings#

  • The power of the Superpowers skill chain: Brainstorming (requirement analysis) → ccplan (ECL-driven implementation) → optimize (quality improvement) forms a complete software lifecycle. ECL documents serve as cross-session persistent ‘design contracts’, ensuring context consistency across multiple dialogues, agents, and even different models.
  • The value of adversarial review: Among the 9 issues discovered by the Critic agent, 2 are CRITICAL (no authenticated PTY writing, SSH parameter injection) and 4 are HIGH (token brute-force cracking, prototype contamination), exposing them in a production environment would cause a security disaster. Repairs in the requirement phase cost less than in the code phase and production phase.
  • The essence of object-centric warping: EEF relative to object motion patterns is an invariant. This invariant is maintained through coordinate transformation T_eef_new = T_obj_target × T_obj_src⁻¹ × T_eef_source. Two-step understanding: (1) eliminating the absolute position of the source object (2) remapping with the new object’s position.
  • The wisdom behind the original MimicGen design: (1) minimizing the number of subtasks (only 2) (2) having a post-grasp buffer of 10-20 frames to ensure stable capture (3) reducing pose gaps with env.reset() to home position (4) quasi-closed-loop execution (calculating delta for target_pose_to_action step by step).
  • The core design of MimicGen is source demo + env.reset() randomization: Error scenes should be treated as fixed ‘initial states’, and variants should be generated using placement_initializer, rather than injecting errors into each target again.
  • Priority of NeurIPS DB Track reviews (based on 61 reviews): The most valued qualities are ‘comprehensive’ (mentioned 17 times), ‘community’, ’novel’, ‘clear’, and ‘well-designed’ (mentioned 13 times). The most common criticisms are ’lack of comparison’ (insufficient baseline), ‘insufficient detail’ (inadequate methodology), and ’limited scope’ (restricted application). The best paper (Artificial Hivemind) has an average score of 4.75 among Oral papers—AC decides beyond the review scores. The average for Oral (5.00) < Spotlight average (5.16), confirming that scoring is not the only acceptance factor.
  • Parallel agent scheduling mode: Independent modules (Config/Parser/LocalPTY) are assigned to different subagents for parallel implementation, then combined for verification, increasing development efficiency by 3-5 times. The key is identifying nodes in the DAG that are not dependent on each other, maximizing parallelism.
  • Claude Code –output-format stream-json: Through research, it was found that Claude natively supports structured output (JSON separated by line breaks), eliminating the need for regular expression parsing of PTYs. This improves the reliability of the Parser from ‘heuristic’ to ‘deterministic’. Always research before reinventing.
  • Return on Investment for TypeScript strict mode: 195 tests + 0 type errors = production-ready code. The type system is particularly useful during refactoring (after extracting parsed-state-utils, all call points automatically get type checked). Spending 20% more time on typing in the early stages saves 80% of debugging time later.
  • xterm.js is the industry-standard browser terminal (used by VS Code, Replit, Vercel, ttyd). Its quality depends on the complete plugin stack: WebGL for GPU rendering, Unicode11 for CJK, Fit for resizing, SearchAddon, WebLinksAddon. Basic xterm.js alone is not sufficient for production use.
  • MimicGen-ready datasets require specific metadata structure: Each demo_*/datagen_info must contain env_interface_name and env_interface_type, otherwise DataGenerator cannot correctly initialize the environment.
  • Human demonstration data quality verification should be done before collection: Simply achieving teleop success is not enough; replay stability and augmentability must also be verified. Otherwise, a large amount of data will be collected that cannot be used for training.
  • Time boundary partitioning (archive covers completed hours, live source covers current hours) elegantly solves deduplication without explicit level tracking—zero overlap means zero duplicate counts.
  • Claude Code’s permission system supports tool-level wildcards (such as “Bash” matching all Bash commands), allowing minimal configuration instead of dozens of individual rules.
  • Cost of Subtask Fragmentation: Each subtask boundary involves independently checking object poses, re-warp, and 5-step interpolation, with errors accumulating. 6 subtasks = 5 error sources, far more than the original 1.
  • OpenReview API authentication architecture (2026): All data endpoints require login with a token; only the /login endpoint allows anonymous access (returning a verification error). The web frontend uses server-side credentials through Next.js RSC. Rate limits are operated at the nginx layer (based on IP), triggering approximately 30 requests in a short period. The academic paper review API is increasingly locked down, requiring institutional or personal credentials.
  • Terminal resize synchronization requires full-chain coordination: ResizeObserver (frontend) → term.onResize → WebSocket message → SessionManager → PTY.resize(). Mismatches result in lines being broken in the error column. Anti-drag prevents flooding when resizing.
  • Terminal instances should not be destroyed during session switching—they save a rollback buffer (10000 lines). The correct approach is to create once and reuse it through clear() + backend RingBuffer (2MB) across sessions.
  • Cache systems must include explicit version fields when data sources can evolve—just TTL expiration is not enough to handle changes in patterns/sources.
  • The three-layer permission configuration allow/deny/ask can balance security and efficiency: Allow implements automation, ask retains confirmation for dangerous operations, deny is for absolute prohibitions.
  • Claude Code configuration does not inherit automatically (each project requires independent configuration), but can be standardized through automated scripts (such as init-claude.sh) for standardized initialization processes.
  • Documentation for complex robot systems should distinguish between developers and operators: Clearly define the target audience, structure, language, and content retention through structured questions to avoid ambiguity.
  • React closure trap ref pattern: Registers third-party library callbacks (xterm.js onData) in useEffect, which captures expired props, a common bug. Common solution: props → ref, useEffect synchronizes ref.current, and the callback reads ref.current. This is the best practice in the React ecosystem.
  • Three key elements for WebSocket robustness: Redundancy (exponential backoff), backpressure (checking bufferedAmount), and heartbeat (not implemented but should be considered). Distributed system network layers must assume ‘connections may disconnect’ rather than ‘connections should be stable’.
  • Importance of continuous closure detection: Robot grabs may attempt multiple times, and the first gripperClosed > threshold may fail. Continuous closure (≥10 frames) must be detected to confirm a successful grab.
  • Complexity of multi-process debugging: (1) Each worker requires an independent env and augmenter instance (2) Path resolution is more prone to errors (relative paths may miss levels) (3) Absolute paths like demos_dir must be explicitly passed (4) Robosuite WARNINGs need to be suppressed separately in worker functions.
  • Quota counting logic must reflect actual availability: augmentable_only is stricter than replay_only but more reasonable, as non-augmentable demos cannot be expanded even if they can be replayed.
  • State of the art ASR models (2026): True streaming models are rare. Moonshine v2 (73-107ms latency, 34-245M parameters) and Voxtral-Mini-4B-Realtime (480ms, 4B parameters) represent the most advanced. Moonshine uses a callback-based API suitable for real-time transcription; Voxtral requires manual chunking but offers higher accuracy. Both support native acceleration for Apple Silicon (Moonshine uses CoreML, Voxtral uses MLX).
  • Permission requirements differ for exploratory and production projects: Exploratory projects prioritize efficiency (lenient permissions), while production projects prioritize security (strict confirmation).
  • Gradually accumulating configurations can lead to inconsistencies (e.g., 80+ long rules, different project styles), and regularly using templates can improve maintainability.

Issues and Solutions#

Key Issues#

1. BC-RNN RP(0.003) being significantly lower than the random RP(0.020), making the evaluation results completely unreasonable#

Solution: Code review identified 4 errors that caused metric distortion: demo_len=0 corrupted SPL, trigger_step=0 further corrupted SPL, transport always limited RP to a maximum of 0.5 False, and successful episode RP not being 1.0 led to baseline bias. Fixed field name demo_length→demo_len in collector.py:487, changed collector.py:488 to read trigger_step from scene datasets, env_wrapper.py added get_target_pos() to dynamically obtain target positions, and collector.py added if success: rp=1.0.

Key Insight: The distortion of evaluation metrics is usually not a single error, but a chain reaction of multiple errors. The limit of RP being 0.5 revealed failures in the transmission phase, demo_len=0/trigger_step=0 revealed field propagation errors.

2. WKWebView cannot receive mouse/k keyboard events during browsing mode, even with ignoresMouseEvents=false; the.docdesktopWindow layer causes non-interactive window operations#

Solution: Set window.level=.normal when entering BrowseMode (not the top level but interactive), and restore CGWindowLevelForKey(.desktopWindow) when exiting BrowseMode.

Key Insight: In macOS, the.docdesktopWindow layer is lower than the Finder desktop icon layer. Windows at this layer do not receive any input events at system level. IgnoringMouseEvents=false has no effect on this layer. .normal(level=0) is the lowest level that is both interactive and does not affect other applications.

3. LLaVA base model download was hindered due to proxy bandwidth bottlenecks (about 50KB/s, downloading 14GB of models took several weeks); GPU nodes could not be directly connected; xet CDN overproxy timed out#

Solution: Found nyw format llava-1.5-7b-hf (safetensors) in the cluster, and wrote convert_hf_to_llava.py to convert HF format to the original LLaVA pytorch_model.bin format (including vocabulary truncation from 32064→32000). The entire process was done locally without needing a network.

Key Insight: Don’t assume that downloading is necessary. First search for existing model copies on the cluster, and even different formats can be converted. Formatting HF models involves more than just renaming keys; it also involves semantic differences (such as vocabulary size).

4. GPU utilization for Phoenix diffusion strategy training was only 5-6%: DataLoader num_workers=0 caused GPUs to wait for CPU to read data from NFS HDF5 files randomly#

Solution: Preload all HDF5 data into memory in one go during init method (uint8 type saves 4 times more storage space), close file handles, and then perform all getitem operations on the numpy array. Set DataLoader’s num_workers to 8 with pin_memory=True.

Key Insight: h5py file handling cannot be shared across subprocesses (this is a limitation of PyTorch DataLoader). The only safe solution is to complete loading in the main process and then fork the subprocess. NFS random reading is much slower than sequential reading, and one-time loading is 10 times faster than按需加载.

5. After starting Phoenix MPM training, serious estimation errors were found (actual time was 13.5 days, expected few hours), dataset size estimation error (160K→1M)#

Solution: The user paused the training and needed to re-plan and optimize it (possibly reducing training steps, increasing GPU count, optimizing data loading, etc.)

Key Insight: Before starting large-scale training, it is necessary to verify the actual dataset size, time per step, and total number of steps to avoid wasting resources. Errors in dataset size estimation can lead to weeks of wasted computing resources.

6. Fuzziness in CalendarPro intent classification was caused by pre-checking general_keywords: ‘What time are you scheduled?’, matching general_keywords with ‘what time’ led to forcing GENERAL, resulting in an error#

Solution: Divide general_keywords into always_general (always force GENERAL, such as ‘weather’) and context_dependent (force GENERAL only when there is no active session, such as ‘what time’); add confirmation_words(‘ok’, ‘good’, ‘yes’, etc.) to handle quick paths through_is_confirmation_input(); rewritten_process_input() to have a 6-step process

Key Insight: Context-based intent classification requires combining session state (session_id, last_activity_at) and conversation history. Simple keyword matching leads to fuzziness; when there is an active proposal, confirmation phrases (‘ok’) should be routed directly to that proposal.

7. The current research scout paper evaluation process handles all 50 papers at once, wasting resources and lacking in-depth analysis#

Solution: Designed a two-stage process: the first stage quickly screens all papers (only titles + first 1000 characters of abstracts, classifying high/low relevance), and the second stage deeply analyzes highly relevant papers (up to 20, extracting 3 highlights + detailed scoring + direction suggestions)

Key Insight: Mimicking the paper reading workflow of human researchers (quick screening → deep reading of high-value papers) can significantly improve the efficiency and quality of LLM evaluation, while also generating literature review records for low-relevance papers for future reference

8. Due to the ID (‘robot-manipulation’) in project.json not matching the directory name (’error-recovery-benchmark’), the process could not read overview.md file#

Solution: Updated the id field in project.json to match the directory name; the process uses project[‘id’] to build the overview.md path

Key Insight: The directory name and ID in project.json must be consistent; _extract_current_methods() uses project[‘id’] to locate overview.md file

9. In the second-stage evaluation, the title of the overview.md section ‘## 3. Current technical approach’ did not match the process’s regular expression pattern ‘##\s*Current method and progress’, resulting in the current method getting a result of ‘(no record found)’#

Solution: Changed the title of the overview.md chapter to match the OVERVIEW_TEMPLATE format; removed numbering prefixes and chapter separators

Key Insight: The pipeline uses strict regular expressions for chapter extraction; it must strictly follow the template format

10. FLARE specification statistics calculation failed: After importing openpi in the script, it was not possible to find the installation version of zhaoganlong, nor could the new configuration name of tangzijia be found#

Solution: Set PYTHONPATH=/HOME/…/tangzijia/shared_deps/openpi/src:$PYTHONPATH in the script to优先加载 tangzijia’s openpi source code, bypassing older versions in the conda environment

Key Insight: When multiple users share the same conda environment, their modifications can lead to import priority conflicts. PYTHONPATH is a minimal intervention solution that does not affect other users

11. WKWebView scroll bars were tracking transparent states (CSS injection to set background color was ineffective) + Sandbox applications could not load any network resources#

Solution: Removed webView.setValue(false, forKey: “drawsBackground”) to restore the default opaque background, making scroll bar tracking visible; added com.apple.security.network.client=true in the.configurations file

Key Insight: DrainsBackground=false makes the entire WKWebView rendering layer transparent, while CSS’s translucent style on a transparent layer remains transparent. The fix must occur at the rendering layer rather than the CSS layer. By default, the App Sandbox prevents any external network access

12. LLM output contained too much English content; users expect Chinese; unable to configure output language#

Solution: Implemented a three-tier language system: _DEFAULT_LANGUAGE constant, _language_instruction() generator, and {language_instruction} in all prompts;

Configuration resolution chain is CLI > project.json > config.json > default

Key Insight: Language instructions must be injected at the prompt level, not later in the processing stage; different chapters require different language rules (JSON keys vs. content)

13. Unable to create a project from existing overview.md, unless keywords/categories are manually extracted#

Solution: Added create_project_from_overview() function, which uses LLM to parse overview.md and extract titles, search_keywords, arxiv_categories, open_questions; then writes to project.json and copies the original overview.md file

Key Insight: LLMs can reliably extract structured metadata from unstructured text; reuse existing call_llm() and parse_json_response() tools

14. When generating reports in bulk, the JSON output generated by LLM often had incomplete or syntax errors, leading to parsing failures#

Solution: After the user provided the original text, AI manually extracted and repaired JSON fragments (removing non-JSON prefixes/suffixes, fixing quote escapes, adding missing closing parentheses, etc.)

Key Insight: LLMs generate structured output, especially JSON with many nested objects and long text content, which can have formatting errors; automatic repair and verification steps need to be added to the pipeline

Issues and Solutions#

Major Issues#

1. BC-RNN RP (0.003) is much lower than the random RP (0.020), making the evaluation results completely unreasonable#

Solution: Code review identified 4 errors that caused the metrics to be distorted: demo_len=0 corrupted the SPL, trigger_step=0 further corrupted the SPL, the transport always set the RP upper limit to 0.5, and the successful episode RP not being 1.0 caused a baseline bias. The field name in collector.py:487 was corrected from demo_length to demo_len. Collector.py:488 was changed to read trigger_step from scene_dataset. envwrapper.py added the get_target_pos() function to dynamically obtain target positions. Collector.py added if success: rp=1.0.

Key Insight: Distortions in evaluation metrics are usually not due to a single error, but rather a chain of multiple errors. Setting the RP upper limit to 0.5 revealed a failure in the transmission phase. demo_len=0/trigger_step=0 revealed errors in field propagation.

2. WKWebView cannot receive mouse/k keyboard events in browse mode, even with ignoresMouseEvents=false; the.docdesktopWindow layer makes windows non-interactive#

Solution: Set window.level to .normal in enterBrowseMode (not at the top but still interactive), and restore CGWindowLevelForKey(.desktopWindow) in exitBrowseMode.

Key Insight: In macOS, the.docdesktopWindow layer is below the Finder desktop icon layer. Windows on this layer cannot receive any input events. IgnoringMouseEvents=false has no effect on this layer. .normal(level=0) is the lowest level, which is both interactive and does not override other applications.

3. Downloading the LLaVA base model is limited by proxy bandwidth, at around 50KB/s. Downloading a 14GB model takes several weeks; GPU nodes cannot be directly connected; xet’s CDN timed out through the proxy#

Solution: It was found that there was an HF format llava-1.5-7b-hf (safetensors) in the cluster. A convert_hf_to_llava.py script was written to convert the HF format to the original LLaVA pytorch_model.bin format (including vocabulary truncation from 32064 to 32000). This process can be done locally without a network.

Key Insight: Don’t assume you need to download everything. First, search for existing model copies in the cluster. Even if the models are in different formats, they can be converted. Converting HF models involves more than just renaming keys; it also involves semantic differences like vocabulary size.

4. During Phoenix diffusion strategy training, GPU utilization was only 5-6%. Setting DataLoader num_workers=0 caused the GPU to wait for the CPU to randomly read data from NFS HDF5 files#

Solution: Preload all HDF5 data into memory once during init function (uint8 type saves 4 times more storage space). Close the file handle, then perform all getitem operations from the numpy array. Set DataLoader’s num_workers to 8 and pin_memory to True.

Key Insight: h5py file handling cannot be shared across subprocesses (this is a limitation of PyTorch DataLoader). The only safe solution is to load everything in the main process before forking subprocesses. Random reading from NFS is much slower than sequential reading. Loading everything at once is 10 times faster than loading on demand.

5. After starting Phoenix MPM training, the estimated training duration was severely incorrect (13.5 days instead of hours). There was also an error in estimating the dataset size (160K instead of 1M)#

Solution: Users need to pause the training and re-plan accordingly (possibly reducing training steps, increasing the number of GPUs, optimizing data loading, etc.)

Key Insight: Before starting large-scale training, it’s necessary to verify the actual dataset size, time per step, and total number of steps to avoid wasting resources. Incorrect estimation of dataset size can lead to weeks of wasted computational resources.

6. CalendarPro’s intent classification was ambiguous because general_keywords intercepted it prematurely: ‘What time are you scheduled?’ → ‘What time is it?’ matches general_keywords → forced to use GENERAL → error occurred#

Solution: Divide general_keywords into always_general (always forcing GENERAL, like ‘weather’) and context_dependent (forcing GENERAL only when there’s no active session, like ‘what time is it?’). Add confirmation_words (‘ok’, ‘good’, ‘yes’, etc.) and provide a quick path for is_confirmation_input(). Rewrite_process_input() to divide it into 6 steps.

Key Insight: Context-aware intent classification requires combining session state (session_id, last_activity_at) and conversation history. Simple keyword matching leads to ambiguity. When there’s an active proposal, confirmations like ‘ok’ should be routed directly.

7. The current research scout paper evaluation process handled all 50 papers at once, wasting marks and lacking in-depth analysis#

Solution: Designed a two-stage process: first stage was a quick screening of all papers (only title + first 1000 characters of abstract, classified as high/low relevance), and second stage was an in-depth analysis of highly relevant papers (up to 20 papers, extracting 3 highlights + detailed scoring + direction suggestions)

Key Insight: Mimicking the paper reading workflow of human researchers (quick screening → in-depth reading of high-value papers) can significantly improve the efficiency and quality of LLM evaluations. It can also generate literature reading records for low-relevance papers for future reference.

8. Due to the ID (‘robot-manipulation’) in project.json not matching the directory name (’error-recovery-benchmark’), the pipeline couldn’t read overview.md#

Solution: Updated the ID field in project.json to match the directory name. The pipeline used project[‘id’] to build the overview.md path.

Key Insight: The directory name and the ID in project.json must be consistent. _extract_current_methods() used project[‘id’] to locate overview.md.

9. In the second stage evaluation, the current method was evaluated as ‘(no record available)’, because the heading of the overview.md section ‘## 3. Current technical approach’ didn’t match the pipeline’s regular expression pattern ‘##\s*Current method and progress’#

Solution: Changed the heading of the overview.md section to match the OVERVIEW_TEMPLATE format. Also removed the numbering prefix and section separator.

Key Insight: The pipeline uses strict regular expressions for chapter extraction. It’s necessary to strictly follow the template format.

10. FLARE specification statistics calculation failed: The specification statistics script imported openpi, but couldn’t find the version installed by zhaoganlong, nor could it find the new configuration name of tangzijia#

Solution: Set PYTHONPATH=/HOME/…/tangzijia/shared_deps/openpi/src:$PYTHONPATH in the script to prioritize loading tangzijia’s openpi source code, bypassing the old version in the conda environment.

Key Insight: When multiple users share the same conda environment, their source code modifications can cause import priority conflicts. PYTHONPATH is a minimally invasive solution that doesn’t affect other users.

11. The transparency of WKWebView scroll bars was ineffective via CSS injection of background color. Additionally, sandbox applications couldn’t load any network resources#

Solution: Remove webView.setValue(false, forKey: “drawsBackground”), restoring the default opaque background so that the scroll bar trajectory is visible. Add com.apple.security.network.client=true to.entitlements.

Key Insight: drawsBackground=false made the entire WKWebView rendering layer transparent, but the translucent style on top of the transparency remained transparent. The fix had to be made in the rendering layer, not in the CSS layer.

12. LLM outputs contained too much English text, while users expected Chinese. It wasn’t possible to configure the output language#

Solution: Implemented a three-tier language system: _DEFAULT_LANGUAGE constant, __language_instruction() generator, and {language_instruction} in all prompts. The configuration resolution chain was CLI>project.json>config.json>default.

Key Insight: Language instructions must be injected at the prompt level, not later in processing. Different parts require different language rules (JSON keys and content).

13. It was impossible to create a project from the existing overview.md, unless keywords/categories were manually extracted#

Solution: Added a create_project_from_overview() function. Utilize the LLM (OVERVIEW_EXTRACT_PROMPT) to parse overview.md, extracting titles, search keywords, arxiv classifications, and open questions. Then write it into project.json, and copy the original overview.md.

Key Insight: LLMs can reliably extract structured metadata from unstructured text. Reuse the existing __call_llm() and __parse_json_response() tools.

14. When generating reports in bulk, the LLM often produced incomplete or syntactically incorrect JSON outputs, leading to parsing failures#

Solution: Users provided the original text, and AI manually extracted and repaired JSON segments (removing non-JSON prefixes/suffixes, fixing quotes escapes, completing missing closing parentheses, etc.)

Key Insight: LLMs generate structured outputs, especially JSONs containing many nested objects and long text content, which are prone to formatting errors. Automatic repair and validation steps need to be added in the pipeline.

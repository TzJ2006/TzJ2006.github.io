Problems and Solutions#

Key Issues#

1. No visible displacement in the video after applying 30N force to the Sawyer arm#

Solution: Not resolved yet. Users requested testing with infinite forces to diagnose whether the force injection mechanism doesn’t work at all.

Key insights: Possible reasons include: (1) duration_steps is set to 1, meaning the force is cleared after 20ms, allowing the OSC controller to counteract it; (2) incorrect application of xfrc_applied; (3) the direction of the force being orthogonal to the OSC axis, resulting in rapid decay.

2. Multiple attempts to connect to GPU nodes failed due to pam_slurm_adopt rejecting them (active SLURM job required), wasting a lot of time#

Solution: Users suggested checking if there is already a GPU on the current node using nvidia-smi -L. It was found that there were already 5 A800s available, so no SSH was needed.

Key insights: In a HPC cluster environment, it’s possible to be on the GPU node through a bastion host. Instead of trying to connect directly, it’s better to check for local GPUs first.

General Issues#

3. AI repeatedly missed the cd command in SSH commands, resulting in 15+ occurrences of the same error (file path not found)#

Solution: After multiple corrections from users, AI understood that connecting to a remote node is equivalent to starting a new session. It needed to explicitly use cd to the project directory and activate the conda environment.

Key insights: SSH doesn’t inherit the working directory of the current shell. This is a systematic issue that needs to be recorded in CLAUDE.md.

4. AI added unnecessary proxy settings in visualization commands (source setproxy.sh)#

Solution: Users pointed out that the visualization script only operates on local files (NPZ, HDF5, MuJoCo rendering), so there’s no need for a network connection. Proxy is not necessary.

Key insights: Proxy is only useful for network operations like pip install/conda or accessing external networks. Local computing tasks don’t require it at all.

AI Limitations#

  • After trying all available partitions but failing, AI didn’t consider the simplest possibility of “there being already a GPU on the current node”. It needed human guidance.

General limitations#

Today’s Takeaways#

Key takeaways#

  • The OSC controller has strong resistance to external forces (kp=150, Λ≈5-10kg). A force with duration_steps=1 (about 20ms) is cleared before the controller can react, effectively meaning almost no force is applied. To make disturbances visible, continuous force application over several control cycles or using a force far beyond what the controller can handle is required.
  • When using Claude Code in a HPC cluster, it’s necessary to use nvidia-smi -L to check if there’s already a GPU on the current node before starting any GPU tasks. If there’s already a GPU, no SSH is needed. If SSH is required, the user must explicitly use cd to the project directory and activate the conda environment in the remote command.
  • When debugging force injection mechanisms, it’s better to first test with extreme parameters (such as infinite forces or 1000 consecutive steps) to verify that the mechanism works. Then, adjust the parameters step by step to a reasonable range, rather than starting from reasonable parameters and increasing them.

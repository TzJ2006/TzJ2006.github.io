Lessons Learned#

Main Lesson: Data errors are more隐蔽 and destructive than model errors. MIHD’s coordinate exchange error caused all visual experiment results to fail, but the code logic was normal, and all tests passed. It is necessary to trace anomalies at the macro level back to the data source (CSV column definitions). This requires a complete thought chain from data scientists; unit testing alone cannot detect such errors.#

Differences in naming conventions across frameworks are hidden pitfalls: robosuite’s _main suffix, differences in observation key names for different tasks (object and object state), LiteLLM’s model name-v1 suffix—these details are not obvious in the documentation, but they can lead to runtime errors. The robot simulation framework abstracts the original physics engine (robosuite encapsulates MuJoCo). Direct API calls bypassing the framework layer will cause errors—it is necessary to use the framework’s wrappers to handle naming conventions and state management. Robosuite’s observation key naming rule is that the environment automatically adds -state suffix to grouped modalities (e.g., object → object-state), but training data usually uses the original names. The adapter layer must handle these mismatches. The safest strategy is: (1) Use the framework’s API instead of low-level APIs; (2) Pass the original data structure instead of manually reconstructing it; (3) Print the actual output to verify field names, rather than inferring from the documentation. Always check the working reference code in the same codebase before making changes.

The correctness of the framework and the quality of the model require different verification strategies. Unit/integration testing can confirm code logic and eliminate errors like KeyError. Video visualization can reveal behavior quality issues (robot hovering and capturing). Even if unit testing passes, GPU integration testing remains crucial—BC-RNN passed 79 unit tests, but GPU deployment was needed to detect 0% SR issues and diagnose untrained checkpoints through visualization. Each verification layer has a different role.

The multi-device collaborative daily report tool has achieved a closed loop: each device exports local logs → central hub combines remote logs → unified deployment to Hugo → published on GitHub Pages. This workflow was extensively used on 2026-02-17 (processing 4 historical dates), proving the feasibility of the two-stage architecture. The next step is automation (which has been proposed but not implemented). Tool-based tools (used to analyze the tools used by the tool) need to carefully consider the distribution of the workflow—what happens on each device, how data is synchronized, how partial information is handled. For distributed scenarios, a two-stage architecture (local export + centralized aggregation) is more robust than a single-unit single-stage design.

The expansion of the error recovery benchmark scenario library from 251 to 454 is a successful incremental expansion case: First, the pulse injector (M4) was implemented → it was found that non-pulse types were needed (M5) → the MuJoCo API compatibility was fixed → various types of scenarios were generated through the detector candidate expansion mechanism. This shows the process from a minimum viable product to a fully functional product. The generation of error scenarios is a funnel process: detector triggering → error specification generation → injector application → verification → scenario acceptance. Enabling the injector does not guarantee scenario generation—it depends on the frequency of detector triggering. Depending on the dynamics of the task, the acceptance rates of different types of errors vary greatly. Pose perturbation is more general than friction/clamping anomalies in the pickup and placement task. Programmed specification enhancement (_augment_specs_with_alternatives()) can compensate for insufficient detector coverage.

Third-party tool integration requires defensive programming: the patch cost is equal to 0 (ccusage), field name compatibility (modelName or model), and self-downgrade for future repairs (when LiteLLM makes repairs, backtracking becomes inactive). This “only activates when necessary” patch strategy can minimize maintenance costs. Pricing defects of third-party tools should be resolved through local backtracking mechanisms, rather than waiting for upstream repairs. Design backtracking should only be triggered in cases of abnormal values (cost = 0) to maintain backward compatibility. LiteLLM uses versioned model names (anthropic.{model}-v1, optional: 0 suffix), while Claude Code logs use simplified names, leading to systematic mismatches. Cost tracking errors may lead to over 90% underestimation (observed 13 times), severely affecting financial planning. Regular audits of cost indicators are very important.

Human involvement in specifying the system architecture and strategic decisions, with AI responsible for technical implementation and optimization, yields the best results. In this meeting, users’ architectural insights (two-stage design, multi-device synchronization) were crucial; AI’s technical execution (parsing backtracking, configuration optimization) filled in the gaps. Qualitative judgment from domain experts is essential for identifying systemic problems. AI’s data-driven analysis and human experience and intuition form an effective complementarity.

Remote development on unstable connections requires multiple layers of defense: maintaining connection activity (prevention), session persistence (survival), multiplexing (fast reconnection), and automated scripts (recovery). Single-layer solutions will fail under real network conditions. For shared server accounts, user-specific custom settings should be placed in conditional blocks or separate startup scripts, rather than affecting everyone’s common.bashrc entry. tmux has both state persistence and environmental boundaries. Session-wide proxy configuration (not global bashrc) can prevent affecting users with shared accounts.

Structured data generated by LLMs requires multiple backtracking strategies for defensive parsing. The native JSON schema (Anthropic’s tool_use, OpenAI’s json_object) is helpful, but it cannot eliminate failures—always treat fixing as the last resort. Each backtracking layer handles specific failure modes: code blocks handle markdown format, curly braces matching handle truncation, and fixing handle syntax errors.

The daily summary tool benefits from a gradual evolution of the architecture: starting with a basic list of tasks/issues, adding session summaries to provide context, and adding priority metadata for filtering. Each layer meets different reading needs. The batch document workflow benefits from a two-stage architecture: (1) extracting local context for each source, (2) integrating across sources for global deduplication. This pattern appears in summary tools (export → merge), MIHD processes (first-stage cache embedding → second-stage fusion), and error recovery benchmarks (generating scenarios → collecting data).

MimicGen’s release policy is to only provide datasets, not pre-trained model checkpoints. Formal releases may provide data but not the training model—before planning, ensure the availability of assets. BC-RNN (with RNN behavior cloning) is a simple supervised learning: collecting pairs of human demonstrations (states, actions), training an LSTM to predict actions, and then deploying as a strategy. When there are enough demonstrations, this is a simple but effective manipulation task baseline. When pre-trained checkpoints are unavailable, training from demonstrations becomes the necessary backup plan.

Reviewing work logs (summarizing the situation of the past week) is an important exercise that converts raw dialogue logs into structured knowledge, helping to identify patterns and track progress over time. - Idempotent design patterns (tracking what has already been processed, skipping repetitive work) are crucial for workflows that may be interrupted or run again. _merged_devices tracking function can prevent expensive repeated API calls when reprocessing partially completed aggregated data.

  • Managing large model assets (VLA checkpoints larger than 47GB) requires an organized approach based on symbolic links to avoid duplication while maintaining centralized discoverability. Dispersed checkpoints in the file system can hinder reproducibility and configuration management. Centralized checkpoint directories can improve discoverability.
  • When repairing quality issues in historical data, data correction (exportation) should be separated from downstream use (report generation). Avoid blocking the entire process due to failures in downstream dependencies (e.g., LLM API unavailability). The export phase relies on Node.js tools (ccusage), and the merging phase relies on the LLM API—even if merging is blocked, export can still proceed.

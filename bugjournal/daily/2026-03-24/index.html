Problems & Solutions#

Critical Issues#

1. daily_summary.py is 2930 lines long with mixed responsibilities, exceeding maintainability threshold#

Solution: Split into 8 modules by functional boundaries: config (75 lines), remote (100 lines), parsers (460 lines), usage (460 lines), summarizer (340 lines), formatter (440 lines), daily (750 lines), cli (130 lines). Retain daily_summary.py as re-export shim for backward compatibility

Key Insight: Large-scale refactoring requires migration smoke tests (47 import contract tests) to ensure zero breakage. Parallel agent execution can speed up module extraction

2. Zero test coverage on 2930+ lines of LLM pipeline code with 5 conversation parsers and hierarchical merge logic#

Solution: Create 72 tests across 5 test files: test_imports.py (47 smoke tests), test_config.py (7 tests), test_parsers.py (6 tests), test_formatter.py (6 tests), test_summarizer.py (6 tests)

Key Insight: Write import contract tests BEFORE refactoring (run twice: before and after). This creates a safety net and enables confident large-scale changes

3. code-summarizer skill produces flat technical documentation lacking narrative and context for diverse readers#

Solution: Redesign as paper-style six-section structure (Highlights → Introduction → Architecture → Implementation → Results → Conclusion & Future Work). Add scale adaptation (≤3 files: simplified, 4-10: standard, 11-50: full, 50+: architecture-focused). Implement Results three-tier fallback (read outputs → read docs examples → infer from code)

Key Insight: Documentation needs to match both scale and audience. Critic Agent review exposed contradiction between ‘all readers’ and ‘fixed structure’, resolved by internal layering within sections

4. TokenMonitor user initially wanted to render cost text directly into tray icon pixels for unified cross-platform display, but Windows/Linux tray icons are fixed-size squares (16-32px) where ‘$12.45’ would be illegible#

Solution: Pivoted to hybrid approach after feasibility probing: macOS keeps native set_title() for text-beside-icon display; Windows/Linux use set_tooltip() for hover-based cost display (with optional short-form rendering like ‘$12’ in future)

Key Insight: Platform constraints discovered through agent-based research (tray icon size limitations) prevented architectural dead-end before implementation; feasibility probing phase caught this before wasted effort

5. ccusage backend integration initially planned as direct CLI subprocess calls, but multiple provider support (Claude, Codex, OpenCode, Pi, Amp) would require managing 5 separate npm packages with different binary names#

Solution: Discovered ccusage provides @ccusage/mcp (Model Context Protocol server) that already aggregates all provider CLIs via stdio JSON-RPC; switched architecture to single long-lived MCP process instead of repeated subprocess spawns

Key Insight: Phase 1 hypothesis interrogation surfaced that ‘ccusage’ is actually 5 separate packages; deeper research revealed MCP server as superior integration point, solving performance and multi-CLI coordination issues simultaneously

6. sys.path.insert() hacks in 3 locations (mcp_server.py, monthly_summary.py, weekly_summary.py) create fragile module identity issues#

Solution: Packageize summarize/ by adding init.py, update pyproject.toml packages list, change all imports to ‘from summarize.xxx import …’. Remove all sys.path manipulations

Key Insight: sys.path hacks cause dual module identity issues (_cached_config loaded twice). Proper packaging eliminates this class of bugs entirely

7. ccplan skill workflow stops prematurely at phase boundaries without completing the full spiral process#

Solution: Add CONTINUOUS EXECUTION MANDATE at skill header, insert ‘→ NEXT: Proceed immediately to Phase X’ at every phase boundary (10/10 phases), complete multi-turn protocol sections for all phases (9/9 phases), mark phase transitions as mandatory

Key Insight: Prompt engineering for multi-phase workflows requires explicit continuation instructions at every boundary. Implicit expectations (‘just continue’) don’t work reliably

8. Original TokenMonitor plan included preserving change_stats and subagent_stats features (currently dead code marked #[allow(dead_code)]) after ccusage migration, but ccusage has no equivalent functionality#

Solution: After user confirmation, decided to keep minimal JSONL reader in Rust specifically for these two features while migrating token/cost/pricing logic to ccusage MCP server

Key Insight: Adversarial review (FIND-001 CRITICAL) forced explicit decision on feature scope before migration; avoiding implicit feature loss by documenting retention plan

Learnings#

Key Learnings#

  • ECL methodology successfully catches critical issues before implementation through hypothesis interrogation, adversarial filtering, and feasibility probing. Examples: tray icon size constraints, missing ccusage features, performance implications of subprocess spawning. This validates the ‘requirements are hypotheses’ premise—every stated requirement should be challenged
  • ECL (Evolving Constraint Language) documents enable cross-session persistence and traceability of design decisions. Status field tracks workflow phase, pending_input field maintains continuity across context compaction
  • Migration smoke tests are critical for large refactorings: write import contract tests capturing all external consumers, run before and after. This creates a safety net for confident changes (47 tests × 2 runs = 94 validations)
  • Tauri v2’s tray.set_title() is macOS-only (not documented clearly in TokenMonitor code); Windows/Linux require set_tooltip() as fallback. This is a critical cross-platform gotcha for Tauri menu bar apps
  • Adversarial review (Critic Agent) finds design blind spots: discovered 12+ issues including CRITICAL contradictions across both projects. Red-blue confrontation at requirement stage prevents downstream rework
  • Multi-phase prompt workflows need explicit continuation instructions at EVERY boundary: CONTINUOUS EXECUTION MANDATE (global), ‘→ NEXT:’ transitions (per-phase), multi-turn protocols (per-phase). Implicit expectations don’t work
  • When migrating from monolithic platform-specific code to cross-platform architecture, removing platform-specific features entirely (glass blur, dock icon visibility) can be more maintainable than finding lowest-common-denominator alternatives—simplicity beats feature parity in infrastructure code
  • Parallel agent execution accelerates independent tasks: 5 module extraction agents ran concurrently, reducing sequential bottleneck. Requires clear task boundaries and no shared state
  • Scale-adaptive design improves UX across project sizes: ≤3 files get simplified output (avoid over-documentation), 11-50 files get full treatment, 50+ get architecture-focused grouping (avoid context overflow)

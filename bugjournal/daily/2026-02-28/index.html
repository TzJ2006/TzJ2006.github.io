Issues and Solutions#

Key Issues#

1. In scGPT TransformerModel.init, use_fast_transformer was not saved as an instance attribute. As a result, in load_pretrained(), the key remapping never executed, and 17.7% of attention weights were randomly initialized. When strict=False, it silently skipped mismatches without errors.#

Solution: Added self.use_fast_transformer = use_fast_transformer to line 64 of model.py. After fixing this issue, verification showed that all 186 parameters matched, with an average ARI improvement of 44.4%. Key insight: PyTorch’s strict=False setting has two sides: it allows partial loading while silently ignoring mismatched keys. Key weights may run as random values for a long time. The same bug exists in the official GitHub repository. Weight loading paths in open-source code need to be audited actively, not blindly trusted.

2. The CoffeeMachineBodyObject lacked the get_bounding_box_half_size() method. When MimicGen Coffee task rollout initialization was performed, an AttributeError occurred because this method was called but never implemented in the robosuite base class. This represents a gap in interfaces between MimicGen and current robosuite versions.#

Solution: Traced the entire call chain (coffee_machine.py → CoffeeMachineBodyObject → CompositeBodyObject → MujocoXMLObject). The get_bounding_box_half_size() method was implemented in each of the three levels of the base class. This calculated the half-size of the bounding box based on the geometric data of each type. Key insight: Fixing missing methods in third-party libraries requires tracing the entire inheritance chain. Simply patching at the nearest call point can lead to errors in other subclasses. Different robosuite forks may have incompatible APIs. Dependency management should be pinned to the same commit as model training.

3. The WebSocket client always reported a ConnectionRefusedError, even though the server process was listening on port 8000. Different address formats (localhost/127.0.0.1/0.0.0.0) were tried during debugging, but none worked.#

Solution: It was found that the http_proxy environment variable was set to 127.0.0.1:10087. The websockets library failed because of proxy connection. Connecting successfully after unsetting all proxy variables at the beginning of run_eval.sh resolved the issue. Key insight: HTTP/HTTPS proxy environment variables can transparently block WebSocket connections. When debugging local services on HPC clusters, proxy variables should be checked first. Such issues are common on HPC clusters and should be included in standard troubleshooting lists.

4. In env_robosuite.py of robomimic, the mujoco_py was directly imported at the top level (not installed), and mimicgen was not imported. As a result, MimicGen environment variants (Coffee_D0, etc.) were not registered with robosuite, and the rollout environment initialization failed.#

Solution: Wrapped the import of mujoco_py in try/except blocks. At the same time, importing mimicgen allowed the MimicGen environment to be registered through side effects. It was confirmed that the same version of robosuite as training was used (zhaoganlong’s dependency directory). Key insight: The MimicGen environment is registered through side effects. Any external tool calling this environment must explicitly execute this import. Robomimic itself does not know about this dependency and needs to inject it at the integration level.

5. In theadata_8um.h5ad HD data, the highs-resolution images in uns[‘spatial’] were full-size (6000×3886), while the coordinates were in the cropped fullres pixel space (0-4966, 0-2971). These do not match, causing visualization errors. Moreover, since X has been logged transformed, the original preprocess_data() will execute normalize_total + log1p again, resulting in double transformation.#

Solution: In load_hd_data(), replace uns[‘spatial’] images with cropped_fullres.tif and recalculate the scaling factor. Added a skip_log parameter to preprocess_data(). When HD data was passed with skip_log=True, only HVG filtering was performed.

6. Robomimic initiated an interactive overwrite confirmation when encountering an existing checkpoint directory in a non-TTY environment. After receiving EOF, it directly exited with EOFError, causing multiple parallel task launches to fail (partially successful runs leaving a directory that triggers the next launch’s overwriting check).#

Solution: Thoroughly clean the corresponding checkpoint directory before each (re)launch to avoid triggering overwriting checks. Run using nohup + srun –overlap instead of sbatch to ensure the process does not terminate with the shell.

7. Pi0.5 training was considered unsuccessful because SLURM job 46553 crashed. The AI initially judged that the training was incomplete based on previous conversation caches (record of a crash at step 5000).#

Solution: Directly checked the checkpoint directory and found that complete checkpoints from step 4000 to 99999 were present. It was determined that the training had been recovered after the crash and completed all 100k steps. Key insight: The checkpoint directory is the most reliable evidence of a successful training. It is more authoritative than log files and conversation records. SLURM job crashes do not equal failed training. Checkpoints should be verified directly, not relying on logs.

General Issues#

8. In HD data, when the spot_diameter_fullres was set to 7.3 (vs DLPFC ~144), the dots rendered by sc.pl.spatial(size=1.0) were extremely small and almost invisible. Users could detect this issue through visual inspection.#

Solution: Adjusted the size parameter of visualize_from_cache.py for HD visualization from 1.0 to 4.0, making the dot diameter approximately 29px (≈bin spacing of 29.2 fullres px). Key insight: The size parameter of scanpy spatial is a multiple of spot_diameter_fullres. Different resolution datasets require different multiples. The spot diameter of HD 8µm is about 1/20 of Visium’s, so the size multiple needs to be increased accordingly to fill the bin spacing.

9. Multiple SLURM constraints on Tianhe cluster led to repeated failures in task submission: –account=sysu_gbli2 was missing, –mem was prohibited by the policy, MUJOCO_EGL_DEVICE_ID required a physical GPU number instead of the CUDA_VISIBLE_DEVICES logical number. When allocating GPU memory, insufficient checks were made on the idle status of each card, resulting in coffee_d0 OOM.#

Solution: Used sacctmgr to add –account by querying the account name. Replaced –mem with –gres=gpu:1. Set MUJOCO_EGL_DEVICE_ID to the actual physical GPU number (5 or 7). Real-time checks of compute-apps usage per card were performed, and tasks were reallocated after that. Key insight: EGL device IDs directly correspond to physical GPUs and are not affected by CUDA_VISIBLE_DEVICES redirection. Before submitting a new cluster for the first time, sacctmgr should be used to confirm the account name and QOS restrictions. Check compute-apps per GPU instead of just the memory summary.

10. In run_benchmark.py, there was a local cluster embeddings copy that duplicated functions with those in utils/clustering.py. After modifying the tool library, the local copy was not synchronized. As a result, when n_clusters=-1, KMeans reported an InvalidParameterError.Solution: The automatic Leiden estimation logic should also be added to the localcluster_embeddings function in run_benchmark.py. This consistency ensures that changes made to the utility library are reflected in this function.#

Key Insights: Large scripts often contain local functions that duplicate functionality with tools. When modifying the tool library, it’s necessary to check whether there are duplicates within the script or eliminate them through refactoring.

AI Limitations#

Major Limitations#

  • Dependence on outdated dialogue cache information (such as Pi0.5 step 5000 crash records) lacks active verification awareness. It’s necessary to check the file system to correct incorrect judgments. Over-reliance on outdated memories may lead to misleading conclusions. The actual file state should be used as a reference.
  • HPC environment troubleshooting lacks systematic checklist checks: No proactive checking of proxy environment variables (initial WebSocket connection failures), MUJOCO_EGL_DEVICE_ID is hardcoded as a logical ID rather than a physical ID, and multiple launches of large GPU services during debugging may not clean up old processes leading to OOM. These issues need human guidance to be exposed.
  • Improper management of srun process lifecycle: Multiple misjudgments of “task failure” due to empty logs were corrected by cleaning/restarting repeatedly. There was a lack of pre-judgment of “checking the process state before making decisions.” Processes may have been running without producing visible output, which should be combined with status checks.
  • Experiment scale planning is too extensive. No proactive suggestion for iterative verification using small proxy datasets (planning for 545K HD data requires user correction to crop10large). There is a lack of active optimization awareness regarding experiment iteration efficiency.

General Limitations#

  • Failure to predict consistency issues across file code (local cluster embeddings copies in run_benchmark.py not synchronized) and differences in visualization parameters across datasets (small spot sizes for HD). Users need to observe the output and trigger secondary repairs.

Today’s Achievements#

Core Achievements#

  • PyTorch model.load_state_dict(strict=False) silently ignores mismatched keys, causing key weights to be replaced with random values without errors. Such bugs can remain latent for a long time. In production code, it’s necessary to actively print missing_keys/unexpected_keys and verify parameter values after loading. The same bug exists in the upstream GitHub official repository. It’s necessary to audit the weight loading path of open-source code rather than blindly trusting it.
  • staig_fusion is significantly ahead in RM-IDEAL evaluations (average r=0.396 for 151673, peak r=0.644 for Layer_3), proving that learning-based multi-modal fusion has a much better ability to capture spatial niche structures compared to simple fusion (concat/mean avg r≈0.15) and single-modal baselines (gene/vision avg r≈0.06-0.12).
  • MimicGen environment variants (Coffee_D0, Stack_D1, etc.) are registered with robosuite through the side effects of importing mimicgen. Any external tool (such as robomimic) must execute this import before calling these environments. There is API incompatibility between different robosuite forks. Training and evaluation must be pinned to the same commit.
  • HTTP/HTTPS proxy environment variables (http_proxy/https_proxy) are transparently proxied by Python’s websockets library, causing ws://localhost:xxxx to fail through the proxy. When running local WebSocket services on HPC clusters, it’s necessary to unset the proxy variables or set no_proxy=localhost before starting the client.
  • When fixing missing methods in third-party libraries, it’s necessary to trace the complete class inheritance chain and implement them in all involved base classes. Patching only at the most direct call point will cause errors when other subclasses are called.
  • crop10large (17502 spots) is an ideal proxy dataset for verifying the feasibility of HD methods: It has a scale comparable to DLPFC, with supporting cropped fullres images. The entire pipeline can be verified without modifying architectural assumptions. The Leiden community found that resolution=1.0 as an automatic unsupervised clustering number performs well on HD data (k=17-20, Silhouette=0.302), making it suitable for default clustering strategies for unlabeled HD data.
  • Training BC-RNN on the A800 80GB GPU takes about 2.2GB VRAM per task. Multiple tasks can be processed in high density parallel on a single GPU, but this is actually limited by CPU/IO rather than VRAM. The size parameter of scanpy spatial visualization is a multiple of spot_diameter_fullres. For DLPFC (diameter≈144), size=1.0 should be used. For HD with 8µm diameter (diameter≈7.3), size≈4.0 is required to fill the bin spacing.

Practical Achievements#

  • Special SLURM constraints for Tianhe cluster: It’s necessary to specify –account=sysu_gbli2. The –mem constraint is prohibited by QOS policy, so it needs to be changed to –gres=gpu:1. MUJOCO_EGL_DEVICE_ID corresponds to the physical GPU number rather than the logical CUDA_VISIBLE_DEVICES number. srun –overlap can be used to attach a new step to an existing interactive job to reuse node resources, without having to submit the job again.

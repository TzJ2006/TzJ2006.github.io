Problems & Solutions#

Critical Issues#

1. QFormer training extremely slow (~3.5 hours) with only 5% GPU utilization due to Python for loop processing 2700 spots sequentially#

Solution: Implemented batched forward pass: pre-compute padded neighbor index tensor once at training start, use index_select for batch context construction, add key_padding_mask to transformer layers, process all spots in single GPU-parallel forward call

Key Insight: GPU severely underutilized when operations are serialized in Python loops - batching is essential for transformer models even when batch semantic is ‘spatial locations’ rather than ‘samples’

2. All 3 GPU evaluations (m14_cpu/pi0/pi05) crashed at scene 122/649 with EnvironmentMismatchError due to 130 natural capture scenes having different xml_hash fingerprints#

Solution: Added EnvironmentMismatchError import and scene-level try-except wrapper in collector.py around _run_episode() to log warnings and skip incompatible scenes instead of terminating

Key Insight: Database contains two fingerprint groups (519 impulse/augmented vs 130 natural VLA-generated scenes); robust error handling at iteration boundaries prevents cascade failures while preserving completed work via –resume

3. STAIG edge weight calculation uses O(n²) nested Python loops taking 2-5 seconds for 2700 spots, blocking training start#

Solution: Vectorized with scipy.spatial.distance.cdist for distance matrix computation and scipy.special.softmax(axis=1) for per-row normalization, then used NumPy advanced indexing edge_probabilities[rows, cols] instead of list comprehension

Key Insight: Nested loops with mathematical operations should always be vectorized - 100-500x speedup achievable by using scipy/NumPy primitives; the key is recognizing O(n²) patterns and finding the right library function (cdist for pairwise distances)

4. Unclear whether vision embedding refinement (scan_cluster: 1536d→256d) improves or hurts multimodal fusion performance#

Solution: Systematic ablation: ran all 8 fusion methods with and without refinement on section 151673, discovered that refine helps weak methods (mean +0.056, attention +0.086 ARI) but hurts strong methods (qformer -0.054, concat -0.074 ARI) - the dimensionality reduction trades off information for reduced noise

Key Insight: Dimensionality reduction’s value depends on downstream model capacity: simple aggregation methods (mean/attention) benefit from denoising and avoiding curse of dimensionality, but sophisticated learnable models (qformer/llava_mlp) can exploit the richer information in high-dimensional embeddings - there’s no universal answer

5. Pi0.5 training (job 46551) exited unexpectedly after ~580 steps due to stdout buffering preventing monitoring and unclear exit reason (code 0)#

Solution: Added PYTHONUNBUFFERED=1 + stdbuf -oL for real-time logs, set -eo pipefail + ERR trap for error reporting, background GPU monitoring every 30min, and used –resume to continue from step 3000 checkpoint

Key Insight: Python/JAX buffered stdout can hide training failures; unbuffered output + explicit error trapping + periodic health checks are critical for 50+ hour training jobs on shared clusters

6. Pipeline hardcoded vision_encoder=‘uni’ for STAIG fusion, preventing experimentation with other encoders (HIPT, ResNet50, etc.)#

Solution: Refactored orchestration layer: introduced STAIG_UNI_FAMILY set to identify encoders needing staig_strict preprocessing (256×256 patch, no ImageNet normalization, frequency filtering), removed forced encoder override in run_benchmark.py, updated extraction_planner and evaluation_planner to conditionally apply variant based on encoder family membership

Key Insight: Separation of concerns: model layer (STAIGTrainer) already handled arbitrary dimensions via PCA, but orchestration layer imposed artificial constraints - good architecture makes this kind of generalization straightforward once identified

7. evaluate_mimicgen.py lacked environment seeding, making evaluation results non-reproducible across runs#

Solution: Added env.seed(args.seed) call before each task’s episode loop, matching reference implementation in LIBERO/Phoenix main.py

Key Insight: Reproducibility requires explicit seeding of both policy RNG and environment RNG; comparing against reference implementations reveals missing initialization steps

General Issues#

8. adaptive_dropout_adj transfers edge_attr to CPU, generates random numbers on CPU, then moves back to GPU - causing sync overhead every epoch (~1-2s total across 1100 epochs)#

Solution: Generate random numbers directly on edge_attr.device with torch.rand(…, device=edge_attr.device, dtype=edge_attr.dtype) - eliminated all CPU transfers

Key Insight: Small GPU↔CPU transfers accumulate significantly when repeated many times; always generate random numbers on the same device as the tensor being operated on; CUDA synchronization overhead is non-negligible

9. UNI2 (ViT-H/14 with patch_size=14) failed on STAIG’s 256×256 input: ‘256 not divisible by 14’ assertion error#

Solution: Added automatic resize to 224×224 (divisible by 14) in UNI2VisionEncoder.extract() when STAIG strict mode is detected, before feeding patches to the model

Key Insight: Vision transformers have strict input size requirements based on patch size - when adapting preprocessing pipelines designed for one encoder to another, check divisibility constraints; 224×224 is the safe default for most ViTs

10. discriminability script crashed with ValueError: invalid literal for int() with base 10: ‘False’ when parsing success field from JSONL#

Solution: Added string-to-bool conversion: success_raw = row.get(‘success’, False); success = success_raw if isinstance(success_raw, bool) else success_raw.lower() == ’true’

Key Insight: JSONL serialization inconsistency—success field stored as string ‘False’ instead of boolean; defensive type checking prevents parse errors when schema assumptions break

11. m14_cpu crashed with AssertionError: MUJOCO_EGL_DEVICE_ID (0) must be in CUDA_VISIBLE_DEVICES (5)#

Solution: Changed MUJOCO_EGL_DEVICE_ID from 0 to 5 to match CUDA_VISIBLE_DEVICES=5, satisfying binding_utils.py assertion for device ID string containment

Key Insight: Robosuite’s binding_utils.py requires MUJOCO_EGL_DEVICE_ID as substring of CUDA_VISIBLE_DEVICES; when using single GPU, both must match exactly

12. Section 151676 caused STAIG training to produce NaN loss and all-zero embeddings, making KMeans fallback also fail on NaN-filled data#

Solution: Added np.nan_to_num() in KMeans fallback path before clustering to replace NaN with 0 and inf with large finite numbers

Key Insight: Model collapse (NaN loss) is a known failure mode that needs graceful degradation; always sanitize inputs before passing to external libraries like sklearn that may not handle NaN robustly; document known failure cases

13. discriminability script crashed with scipy.stats.kruskal ValueError when all success rates were identical (degenerate case with all SR=0)#

Solution: Wrapped H, p = scipy_stats.kruskal(*groups) in try-except ValueError, returning H=0.0, p=1.0, significant=False, note=‘all values identical’ on exception

Key Insight: Statistical tests fail on degenerate distributions; explicit edge case handling with informative messages prevents silent failures and aids interpretation

Learnings#

Key Learnings#

  • Batch processing is critical for GPU efficiency: Python loops that repeatedly call small GPU operations (even 2700 calls) create severe underutilization (5% GPU for QFormer) - always look for opportunities to aggregate operations into single GPU kernel launches through batching
  • Dimensionality reduction’s value depends on downstream model capacity: helps simple aggregation methods (mean, attention) by reducing noise and curse of dimensionality (+0.05-0.09 ARI), but hurts sophisticated learnable models (qformer, llava_mlp) that can exploit richer information in high dimensions (-0.05 ARI) - no universal answer, must match preprocessing to model complexity
  • Robust error handling at iteration boundaries (scene-level try-except) prevents cascade failures in long-running batch processes while preserving partial results via resume mechanisms
  • Vectorization delivers 100-1000x speedups for mathematical operations on arrays: replacing Python nested loops with scipy/NumPy primitives (cdist, softmax, advanced indexing) is almost always worth it - the key is recognizing O(n²) patterns and knowing the right library function
  • Unbuffered output (PYTHONUNBUFFERED=1 + stdbuf -oL) is non-negotiable for monitoring long-running ML training jobs on remote clusters—buffering can hide failures for hours and make debugging impossible
  • Reproducibility in RL/robotics evaluation requires explicit seeding of both policy RNG and environment RNG; comparing against reference implementations reveals missing initialization steps that documentation may omit
  • Architecture flexibility through family-based conditional logic: when generalizing from hardcoded assumptions (UNI-only), introduce semantic groupings (encoder families) with conditional behavior rather than removing all constraints - preserves special-case optimizations while enabling flexibility

Practical Learnings#

  • GPU↔CPU transfer elimination matters at scale: small transfers (edge_attr tensor) seem negligible but accumulate across many iterations (1100 epochs × multiple dropout calls) to 1-2 seconds total - always generate/operate on device-local data when possible
  • Vision transformer input requirements are strict: ViTs require input dimensions divisible by patch size (UNI2: H%14=0, W%14=0) - when adapting preprocessing designed for one architecture to another, verify dimensional constraints early; 224×224 is safe default for most ViTs
  • Statistical tests (like scipy.stats.kruskal) fail on degenerate distributions; always add edge case handling with informative error messages rather than letting library exceptions propagate
  • Defensive type checking (isinstance() guards) prevents parse errors when serialization format assumptions break; JSONL fields can have inconsistent types across different data sources
  • Model collapse (NaN loss) requires graceful degradation paths: when known failure modes exist (151676 section), add sanitization steps (np.nan_to_num) in fallback paths and document the issue - better to produce degraded output (ARI=0 with clean data) than crash
  • When debugging multi-GPU Slurm jobs, MUJOCO_EGL_DEVICE_ID and CUDA_VISIBLE_DEVICES must have exact string matching—substring containment check in binding_utils.py requires both to align

Issues and Solutions#

Key Issues#

1. QFormer training is very slow (about 3.5 hours), using only 5% of GPU utilization, as Python loops process 2700 points sequentially.#

Solution: Batch forward pass was implemented: Pre-calculated padding neighbor index tensors at the start of training, using index_select to build batch context, adding key_padding_mask in transformer, and processing all points in parallel calls to forward on a single GPU.Key Insight: When serializing operations in Python loops, GPU utilization is extremely low—even if the batch semantics are “spatial position” rather than “samples”, batch processing is essential for Transformer models.

2. All 3 GPU evaluation nodes (m14_cpu/pi0/pi05) crashed at scenario 122/649 due to inconsistent XML hash fingerprints of 130 natural capture scenarios, causing EnvironmentMismatchError.#

Solution: EnvironmentMismatchError import statement was added to collector.py, and scene-level try-except wrappers were added around _run_episode() function to log warnings and skip incompatible scenarios instead of terminating the program.

Key Insight: The database contains two sets of fingerprint data (519 pulses/enhanced scenarios vs. 130 natural VLA-generated scenarios); robust error handling at the iteration boundary can prevent cascading failures, and the work can continue with –resume.

3. STAIG’s edge weight calculation uses O(n²) nested Python loops, which takes 2-5 seconds to calculate 2700 positions, blocking the training process.#

Solution: Scipy.spatial.distance.cdist was used for distance matrix calculations, scipy.special.softmax(axis=1) for row normalization, and advanced indexing features of NumPy were used to access edge probabilities instead of list comprehensions.

Key Insight: Nested loops containing mathematical operations should always be vectorized—using basic operations of Scipy/NumPy can achieve 100-500 times speedup; the key is identifying O(n²) patterns and finding appropriate library functions (for paired distances, cdist can be used).

4. It’s unclear whether the refinement of visual embeddings (scan_cluster: 1536d→256d) improves multimodal fusion performance or actually reduces it.#

Solution: Systematic ablation experiments were conducted: All 8 fusion methods were tested on section 151673, and it was found that refinement helps weaker methods (mean increases by 0.056, attention mechanism increases by 0.086 ARI), but negatively affects stronger methods (qformer decreases by 0.054, concat decreases by 0.074 ARI)—the downsampling process requires balancing information to reduce noise.

Key Insight: The value of downsampling depends on the capacity of downstream models: Simple aggregation methods (mean/attention) can benefit from noise reduction, avoiding dimensionality disasters, while complex learnable models like qformer/llava_mlp can utilize rich information in high-dimensional embeddings—there is no universal answer.

5. Pi0.5 training task (job number 46551) unexpectedly exited after about 580 steps due to stdout buffering preventing monitoring, and the exit reason was unclear (code 0).#

Solution: PYTHONUNBUFFERED=1 + stdbuf -oL was added to achieve real-time log output, –eo pipefail + ERR trap was added for error reporting, and health checks on background GPUs were performed every 30 minutes. –resume was used to resume training from checkpoint step 3000.

Key Insight: Buffering of stdout in Python/JAX may hide training failures; no-buffered output, explicit error catching, and regular health checks are crucial for 50-hour training tasks on shared clusters.

6. During STAIG fusion, the pipeline hard-coded visionencoder=‘uni’, limiting our ability to try other encoders (such as HIPT, ResNet50, etc.).#

Solution: Orchestration layer was refactored: STAIG_UNI_FAMILY setting was introduced to identify encoders that require staig_strict preprocessing (256×256 patches, no ImageNet standardization, frequency filtering), forced encoder coverage logic in run_benchmark.py was removed, extract_planner and evaluation_planner were updated to conditionally apply different variants based on encoder family membership.

Key Insight: Responsibility separation: The model layer (STAIGTrainer) has already processed data of any size through PCA, but the orchestration layer imposes artificial restrictions—good architecture makes this generalization simple.

7. evaluate_mimicgen.py lacks environment initialization, making evaluation results non-replicable between runs.#

Solution: env.seed(args.seed) call was added before each task’s episode loop to match the reference implementation in LIBERO/Phoenix main.py.

Key Insight: Reproducibility requires clear initialization strategies for RNG and environment RNG; comparing with the reference implementation reveals missing initialization steps.

General Issues#

8. adaptiveDropoutAdj transfers edgeAttr to CPU, generates random numbers on CPU, then returns to GPU—this causes synchronization overhead in each cycle (totaling about 1-2 seconds in 1100 cycles).#

Solution: torch.rand(…), device=edgeAttr.device, dtype=edgeAttr.dtype was used directly on edgeAttr_device to generate random numbers—eliminating all CPU transfers.

Key Insight: Small GPU↔CPU transfers can accumulate significantly over multiple repetitions; always generate random numbers on the same device as the operation tensor; CUDA synchronization overhead cannot be ignored.

9. UNI2 (ViT-H/14, patch_size=14) failed on STAIG’s 256×256 input, with an assertion error stating “256 cannot be divided by 14”.#

Solution: When the strict mode of STAIG was detected, UNI2VisionEncoder.extract() was automatically adjusted to 224×224 (divisible by 14), and then the patch was passed to the model.

Key Insight: Visual transformers have strict input size requirements; adjusting patches based on size requires checking divisibility constraints; for most ViTs, 224×224 is the safe default size.

10. discriminability script encountered ValueError error when parsing JSONL’s success field: invalid decimal integer value “False”.#

Solution: String to boolean conversion was added: success_raw = row.get(‘success’, False); success = success_raw if isinstance(success_raw, bool) else success_raw.lower() == ’true’

Key Insight: JSONL serialization inconsistency—success field is stored as string ‘False’; defensive type checking can prevent parsing errors, especially when pattern assumptions are not met.

11. m14_cpu crashed under AssertionError: MUJOCO_EGL_DEVICE_ID (0) must be in CUDA_VISIBLE_DEVICES (5).#

Solution: MUJOCO_EGL_DEVICE_ID was changed from 0 to 5 to match CUDA_VISIBLE_DEVICES=5, thus satisfying the assertion in binding_utils.py regarding device ID strings.

Key Insight: Robosuite’s binding_utils.py requires MUJOCO_EGL_DEVICE_ID to be a substring of CUDA_VISIBLE_DEVICES; when using a single GPU, both must match perfectly.

12. Section 151676 caused STAIG training to produce NaN losses and zero embeddings, making KMeans fail on NaN-filled data.#

Solution: np.nan_to_num() was added to KMeans backtracking path, replacing NaN with 0 before clustering, and infinity with a larger finite value.

Key Insight: Model crashes (NaN losses) are known failure modes; graceful degradation is needed; input should always be cleaned before passing to external libraries that may not handle NaN gracefully; known failure cases should be documented.

13. discriminability script crashed under scipy.stats.kruskal ValueError due to all success rates being the same (all SR=0 cases).#

Solution: H, p = scipy_stats.kruskal(*groups) was encapsulated in try-except ValueError, returning H=0.0, p=1.0, significant=False, note=‘all values identical’ in case of exceptions.Key Insight: Statistical tests will fail on degenerate distributions; explicit edge cases and informative messages can prevent silent failures and help interpret results

Lessons Learned#

Key Lessons#

  • Batch processing is crucial for GPU efficiency: Python loops that continuously call small GPU operations (even 2700 calls) can lead to severe resource waste (QFormer only used 5% of GPU resources)—always look for opportunities to combine operations into single GPU kernel calls.
  • The value of dimensionality reduction depends on the capacity of downstream models: It helps simplify aggregation methods (e.g., mean, attention mechanisms), thereby reducing the impact of noise and dimensionality disasters (+0.05-0.09 ARI), but it has a negative impact on advanced learning models that can utilize high-dimensional information (e.g., qformer, llava_mlp) (-0.05 ARI). There is no universal solution; preprocessing methods must be adjusted according to the complexity of the model.
  • Conducting robust error handling at the iteration boundary (scene-level try-except handling) can prevent cascading failures during long-running batch processes, while retaining some results through recovery mechanisms.
  • Vectorizing operations can significantly improve array operation speed: Using scipy/NumPy primitives to replace Python nested loops is always worth it—the key is to identify O(n²) patterns and know the correct library functions.
  • For machine learning training tasks that run for long periods on remote clusters, non-buffered output (PYTHONUNBUFFERED=1 + stdbuf -oL) is essential—buffering can hide failures for hours, making debugging impossible.
  • In RL/robotics evaluations, it’s necessary to explicitly initialize strategies RNG and environment RNG—comparing with reference implementations can reveal initialization steps that may be missed in documentation.
  • Achieving flexibility in architecture through conditional logic based on families: From hard-coding assumptions (only UNI) to introducing semantic grouping (encoder family), adopting conditional behavior instead of removing all constraints—this allows for optimization of special cases while maintaining flexibility.

Practical Lessons#

  • In large-scale applications, eliminating data transfer between GPUs and CPUs is important: Small data transfers may seem insignificant, but they can accumulate over multiple iterations (1100 cycles × multiple dropout calls), totaling 1-2 seconds—perform generation and operations locally on the device.
  • Visual transformers have very strict input requirements: ViTs require input sizes to be multiples of Patch Size (UNI2: H%14=0, W%14=0)—when applying preprocessing methods designed for one architecture to another, it’s necessary to verify size constraints in advance. For most ViTs, 224×224 is a safe default value.
  • Statistical tests (e.g., scipy.stats.kruskal) fail on degenerate distributions; always include edge case handling with informative error messages, rather than allowing library exceptions to propagate.
  • Defensive type checking (isinstance() protection) can prevent parsing errors when serialization format assumptions go wrong; JSONL fields may have inconsistent types across different data sources.
  • Model crashes (NaN losses) require graceful downgrading paths: When there are known failure patterns, add sanitization steps (np.nan_to_num) in the downgrading path and record the issues—producing a downgraded output (ARI=0 and clean data) is better than a crash.
  • When debugging multi-GPU Slurm jobs, MUJOCO_EGL_DEVICE_ID and CUDA_VISIBLE_DEVICES must have precise string matching—substrings in binding_utils.py include checks that require both to align.

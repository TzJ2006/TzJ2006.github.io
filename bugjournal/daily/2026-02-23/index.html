Problems and Solutions#

Key Issues#

1. The state of pi0.5 full fine-tuning training (params+optimizer+EMA) requires ~62GB/GPU space. Even with FSDP=4 splitting, the initial phase still results in each GPU temporarily holding the entire model due to replicated_sharding, leading to repeated OOM issues.#

Solution: Switch to LoRA fine-tuning (gemma_2b_lora+gemma_300m_lora). The trainable parameters are reduced by ~90%, and the memory requirement is reduced to ~22.5GB/GPU.

Key insight: JAX FSDP only splits parameter storage but does not reduce the activation memory during forward propagation. A single A800 with 80GB of memory cannot accommodate pi0.5 full fine-tuning. LoRA is the only feasible solution for an A800 cluster. When encountering the warning “Can’t reduce memory use below 62.46GiB,” switch to LoRA immediately instead of continuing with FSDP configuration.

2. All three M14 evaluations crashed at scene 122/649: Natural capture scenarios (~130 natural_*) had inconsistent xml hashes compared to the current mimicgen environment, causing EnvironmentMismatchError without any detection, which led to the entire evaluation process crashing.#

Solution: Add try-except for EnvironmentMismatchError in collector.py’s collect_on_scenes() for each scenario. Record warnings and skip that scenario, then resume the three-part evaluation with –resume after fixing the issue.

Key insight: The 649 scenario database contains two types of environments: ~519 impulse/augmented-compatible scenarios (xml hashes match) and ~130 natural capture scenarios (generated in a VLA environment, with different xml hashes). The actual valid evaluation targets for M14 are ~519 scenarios, so the number of target episodes needs to be adjusted accordingly.

3. UNI2 (ViT-H/14, patch_size=14) is incompatible with STAIG strict mode’s 256×256 patch input (256/14 = 18.28 not an integer, AssertionError).#

Solution: When detecting STAIG mode during UNI2 extraction, automatically resize patches from 256×256 to 224×224 before batch inference (since 224 is divisible by 14) if STAIG mode is detected.

Key insight: UNI v1 supports any input with dynamic_img_size=True. UNI2 is standard for ViT-H/14. The 256×256 patch in STAIG was designed for UNI v1 and needs to be adapted to the patch_size constraints when transplanted to new models.

4. Training logs caused serious inaccuracies due to stdout buffering: Pi0.5 training seemed to only run ~580 steps, but actually reached step 3000, leading to an incorrect determination of training termination.#

Solution: Add PYTHONUNBUFFERED=1 and stdbuf -oL in the training script and sbatch submission scripts to ensure real-time log updates. Also add ERR trap and background GPU memory monitoring.

Key insight: All long-training scripts should standardize with PYTHONUNBUFFERED=1. Otherwise, stdout buffering will cause significant inaccuracies in progress monitoring, leading to unnecessary restarts.

5. Dual problems with Slurm resource management: srun –overlap blocks the command execution on allocated jobs when running commands; long-training processes are terminated by SIGTERM/SIGKILL (exit code 137/143) after an interactive session times out.#

Solution: Use srun –jobid=XXXXX –overlap to resolve resource conflicts and run commands on allocated nodes. For long-training tasks, use sbatch to submit independent batch jobs, decoupling them from the interactive session lifecycle.

Key insight: Interactive development (srun) should be strictly separated from long-training (sbatch). Sbatch jobs have independent resource allocation and lifecycle management, making them the correct way for cluster training.

6. For DLPFC slice 151676, STAIG training loss became NaN from the first epoch, and KMeans fallback crashed due to NaN.#

Solution: Add nan_to_num cleanup in the KMeans fallback path. Training collapses are known issues (high temperature tau=30 + special data distribution) and have no fundamental solution.

Key insight: 151676 is an abnormal slice and may require separate adjustments of tau/dropout rate/graph construction parameters for a dedicated investigation.

General Issues#

7. After multiple training failures, zombie GPU processes (nvitop, previous training) occupied over 70GB of memory, preventing new training from acquiring sufficient memory. All OOM issues were caused by these zombie processes rather than configuration problems.#

Solution: Use fuser /dev/nvidiaX and kill -9 to clean up zombie processes one by one. Pay special attention to nvitop, which holds a CUDA context even when there is no active training.

Key insight: Before starting training, it’s necessary to check and clear all non-essential GPU processes. Monitoring tools like nvitop are common sources of hidden memory usage.

8. Python .pyc cache causes module modifications not to take effect: After renaming constants, phase2_evaluate.py references were omitted (ImportError); after modifying config.py’s batch_size/fsdp_devices, the cache was not cleared (configuration did not take effect).#

Solution: After modifying module constants or configuration, actively clear all pycache and .pyc files to ensure Python is recompiled with the latest code.

Key insight: Python bytecode caching is a common pitfall in rapid iteration development. After renaming constants, it’s necessary to globally search for all references before submitting the modification to avoid omissions.

9. MUJOCO_EGL_DEVICE_ID=0 does not match CUDA_VISIBLE_DEVICES=5, causing an AssertionError crash when starting m14_cpu evaluation.#

Solution: Set MUJOCO_EGL_DEVICE_ID to the same GPU number as the string CUDA_VISIBLE_DEVICES (both should be set to 5).

Key insight: Robosuite’s EGL binding implements assertions through string inclusion. MUJOCO_EGL_DEVICE_ID must be the actual GPU number in CUDA_VISIBLE_DEVICES, not a relative index after mapping.

10. Collection of dataset and script compatibility issues: Datasets 4.4.1 are incompatible with lerobot 0.1.0 (column access API changes), int(‘False’) in scripts/8 causes a crash (JSONL boolean values are serialized as strings), tabulate is not installed.#

Solution: Downgrade datasets to 3.6.0 and re-convert the dataset. When reading success fields, add ‘True’/‘False’ strings to the bool conversion and protect against kruskal degeneration cases. Install tabulate with pip.### Key Insights

The compatibility of Python dependencies in shared cluster environments requires verification one by one during the first run. Boolean values in JSONL may be converted to strings after being serialized using json.dumps, so explicit conversion is necessary when reading them.

AI Limitations#

Major Limitations#

  • Misunderstanding of JAX FSDP memory optimization principles: Believing that FSDP=4 could reduce each GPU’s memory to about 16GB (62/4), but in reality, FSDP only splits parameter storage. The full model still needs to be loaded during initialization, and the forward activation memory does not decrease. LoRA training (gemma_2b_lora) should have reduced memory requirements to ~22.5GB, making it the only feasible solution for the A800 cluster.
  • Failure to pre-check model architecture constraints and data compatibility: During implementing UNI2 compatibility, no check was made on ViT-H/14’s patch_size divisibility requirements. The scenario fingerprint distribution in meta.json was not checked before evaluation. Only fixes were found after runtime crashes, leading to additional debugging cycles.
  • Mistakes in monitoring progress of long-term tasks: PYTHONUNBUFFERED=1 was not set in the training script, resulting in misjudgment of log buffering and progress (580 steps → actual 3000 steps). Training was restarted multiple times without checking whether GPU memory was actually released. Multiple OOMs were caused by zombie processes rather than configuration issues.

General Limitations#

  • Incomplete cache cleanup and reference searching after code modifications: After renaming constants, not all references were searched immediately (scripts/phase2_evaluate.py was missed). After modifying config.py, pycache was not cleared, leading to ImportError or ineffective configuration during runtime. Additional fixes were required.
  • Insufficient understanding of Plan Mode workflow: Multiple attempts to call ExitPlanMode after writing the plan were rejected by the user. It was not understood that the plan is a user decision checkpoint, not an automatic signal from AI.

Today’s Takeaways#

Key Takeaways#

  • For pi0.5 full training state (parameters + AdamW + EMA), ~62GB of memory is required. A single A800 cannot handle full fine-tuning. JAX FSDP only splits parameter storage, so it does not affect the loading of the full model during initialization and the forward activation memory. LoRA fine-tuning (gemma_2b_lora) reduces memory requirements to ~22.5GB, making it the only feasible solution for the A800 cluster.
  • The 649-scenario database consists of two categories: ~519 impulse/pose_perturb/augmented-compatible scenarios (xml_hash matches), and ~130 natural_* natural capture scenarios (generated in a VLA environment with cameras enabled, with different xml hashes). In M14 evaluation, the actual number of valid scenarios was ~519. The number of target episodes and all documentation need to be updated accordingly.
  • Vision Refine has a bimodal effect on Fusion: scan_cluster refine (1536d → 256d dimensionality reduction) is beneficial for weak fusion (attention +0.086 ARI), but harmful for strong fusion (qformer -0.054). Strong fusion (QFormer) can handle high-dimensional inputs, but reducing dimensions actually loses information. The optimal result on that day was qformer+no-refine (ARI=0.4832).
  • Vectorization and batching acceleration are highly effective in scientific computing: O(n²) nested loops → cdist matrix operations are approximately 100-500x faster; Python loop iteration → batch GPU forward is expected to be 20-50x faster. This is the core method for accelerating space group training, and it also verifies the general optimization strategy of “preprocessing instead of calculating per epoch.”
  • Best practices for Slurm cluster training engineering: Long-term training requires using sbatch (not srun) to avoid session timeout due to SIGTERM. All training scripts should include PYTHONUNBUFFERED=1 + stdbuf -oL. Before starting training, use fuser /dev/nvidiaX to clean up zombie GPU processes (monitoring tools like nvitop also hold CUDA context). MUJOCO_EGL_DEVICE_ID must be exactly the same as the number of GPUs in the CUDA_VISIBLE_DEVICES string.
  • STAIG fusion is an end-to-end refine+fuse integrated solution (GCN encodes spatial relationships). Adding scan_cluster refine at the input end leads to functional overlap and information redundancy. Neither should be used together. DLPFC slice 151676 has a stable training collapse problem (NaN from epoch 1), requiring independent adjustment of tau/dropout rate/figure construction parameters.
  • Preliminary analysis conclusion for M13: Random+BC-RNN two CPU baselines have SR=0 in all scenarios, with Fleiss’ kappa = -0.02 (poor), indicating no significant discrimination—because the baseline strategies simply cannot succeed, lacking SR variability.有意义的统计结果需要VLA (Pi0/Pi0.5) data.

Problems and Solutions#

Key Issues#

1. SwiftUI .ultraThinMaterial displays a fixed-color glass in videos, rather than transparent glass#

Solution: SwiftUI materials use the.behindWindow blend mode (displaying the desktop behind the window) instead of the.integralWindow mode (displaying content inside the window). Replace it with NSViewRepresentable using NSVisualEffectView, and explicitly set the.integralWindow mode and clearBackgroundColor in NSHostingView.

Key Insight: SwiftUI’s abstraction hides the details of complex visual effects; for complex visuals like window-level transparency, control needs to be at the NSView level.

2. The STAIG fused embeddings for 151676 were all zero (all embedding similarity values were 0, Spearman r was NaN), causing the model to completely crash#

Solution: Added np.allclose(embeddings, 0) detection in load_fused Embeddings; if true, print a warning and automatically trigger the–recompute_fusion path to reload the gene/visual embeddings from the cache and retrain STAIG (300 cycles, about 50 seconds on RTX 2080 Ti). After fixing, the embeddings returned to normal (64 dimensions, norm 1.0).

Key Insight: Due to unstable training, the STAIG fused embeddings saved in the second stage might go down to zero; active detection and automatic recalculation are needed in the benchmark testing script to avoid manual intervention.

3. When switching videos after selecting a bookmark in NSOpenPanel, the old video still plays#

Solution: Ensure FileAccess parses each bookmark as the old video URL instead of requesting a new one. Added a guard to compare the parsed URL with the requested URL; if different, return the original URL (which has temporary sandbox permissions provided by NSOpenPanel).

Key Insight: Bookmark storage conflicts with multiple video histories; a guard is needed to detect that a bookmark points to an expired file.

4. The success rate for the Coffee task’s BC-RNN was 0%, suspected to be an evaluation errorSolutions: The training logs were checked, and it was confirmed that the BC-RNN training for the coffee task achieved 0% SR at all stages (600 cycles of training were performed). This is not an evaluation error; the task indeed failed during training.#

Key Insights: Before debugging the evaluation process, it is essential to verify the training metrics. A zero value in downstream metrics may indicate a failure in the upstream training process.

CUDA kernels on P100 GPU (sm_60) are incompatible with PyTorch (requires sm_70+ versions). torch.cuda.is_available() returns True, but asynchronous operations fail.#

Solution: Initially, an attempt was made to use try/except to catch the torchzeros(1).cuda() exception, but the asynchronous failures on P100 could not be caught. The user successfully ran the script after replacing it with RTX 2080 Ti (sm_75). Finally, an environment variable CUDA_VISIBLE_DEVICES="" was added to the script to implement a forced CPU fallback logic.

Key Insights: torch.cuda.is_available() only checks whether the driver and libraries exist, without checking whether the GPU’s computing power is compatible with current PyTorch. CUDA operations may fail asynchronously, and cannot be caught by Python’s try/except. The most reliable method is to replace the GPU or use an environment variable to force the use of the CPU.

The JSON generated by the LLM is truncated (… truncation marks), causing daily_summary.py to fail to parse on 2026-02-14 and other dates.#

Solution: The truncated original text was sent to the AI as a separate prompt, with only the required valid JSON portion requested to be repaired/extracted. The AI successfully restored the complete JSON structure.

Key Insights: The extremely long JSON generated by the LLM is at risk of being truncated (especially in scenarios where multiple segments are merged). Truncation detection and automatic JSON repair retries should be added during the merging phase, rather than relying solely on manual discovery and intervention.

The screen saver clock text leaves a 0.2-second smear when the timeout text is updated.#

Solution: CALayer has a default animation effect of about 0.25 seconds for changes to content properties. In CATransaction, blurView.layer?.mask was updated, and setDisableActions(true) was set to enforce immediate replacement.

Key Insights: CALayer has an animation effect for almost all property changes. Explicit transaction control is needed to achieve immediate visual updates.

The BC-RNN strategy evaluation crashed due to “Unable to handle data type <class ’numpy.float64’>”.#

Solution: Robomimic’s TensorUtils.to_tensor() cannot handle numpy scalar types. The_to_robosuite_obs was modified to filter out keys in the raw_obs dictionary that only contain array/image values, which are passed to the robomimic strategy.

Key Insights: Robomimic expects a structured obs dictionary containing array values. NumPy scalars from the robosuite (such as gripper width float64) must be filtered before being passed to TensorUtils.

The visualization script failed multiple times: Bashheredoc does not equal escape character != (SyntaxError), file name prefix does not match (151673_filtered_feature_bc_matrix.h5 versus filtered_feature_bc_matrix.h5), spatial/data path errors (located in Dataset/DLPFC, not directly in the chapter directory), load_dlpfc_data() return value parsing error (returning 1 instead of 2), read_10x_h5 lacks uns[‘spatial’], causing sc.pl.spatial() to fail.#

Solution: An independent script/_viz_rm_ideal.py file was used instead of the heredoc method. The glob “*filtered_feature_bc_matrix.h5” was used to find the correct file name. sc.read_visium("/path/to/Dataset/DLPFC/151673") was used directly for loading (automatically handling spatial/subdirectories). The problem of unpacking load_dlpfc_data into a single return value was fixed. The load_dlpfc_data function was abandoned, and sc.read_visium was used directly to ensure the presence of uns[‘spatial’].

Key Insights: Bash’s heredoc is sensitive to special character escapes. Complex scripts should be written as separate files. The naming of DLPFC data files includes chapter prefixes. Spatial data is located in Dataset/DLPFC/{section}/spatial/ instead of {section}/spatial/. Load_dlpfc_data uses read_10x_h5 instead of read_visium, resulting in the lack of spatial metadata. sc.pl.spatial relies heavily on uns[‘spatial’], so it must be loaded using read_visium.

Common Issues#

There is a gray ghost shadow on the edges of the screen saver clock text.#

Solution: The root cause is that the font weights between ScreensaverClockHighlight (.light) and renderTextMask (.regular) do not match, causing the mask to become thicker and exposing the frosted glass edge of hudWindow. This issue was solved by unifying the font weight to .reasonable.

Key Insights: Visual effects layers must achieve pixel-level alignment. Even minor weight differences can expose the underlying content.

Processing historical data over 13 days requires frequent switching of date parameters, and manually confirming the export/merge status is tedious and prone to errors.#

Solution: The daily_summary.py command export –date YYYY-MM-DD was executed manually, and the merge commands for each date were executed one by one. The integrity of the output files was confirmed through the file system.

Key Insights: The Summarize tool lacks a batch mode (e.g., –date-range 2026-02-07..2026-02-19 automatically iterates). Future improvements should include support for batch processing and progress recovery mechanisms.

When using relative paths in nohup, if the SSH command is executed in the project directory rather than the user’s home directory, problems may occur.#

Solution: The command was changed from ‘ssh an53 “nohup bash scripts/…”’ to ‘ssh an53 “cd $PROJ && nohup bash scripts/…”’, ensuring that the working directory was correctly entered before the script execution.

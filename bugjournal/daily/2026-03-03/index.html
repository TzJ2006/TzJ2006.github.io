Issues and Solutions#

Key Issues#

1. Pi0.5 LoRA training configuration name contains ‘pi05_libero’, users suspect the initialization weights came from libero checkpoint rather than pi05_base#

Solution: Verified through three layers of evidence: ① openpi config.py weight_loader explicitly points to pi05_base/params; ② Training log shows “Restoring checkpoint from …/pi05_base”; ③ zhaoganlong’s configuration also loads pi05_base.

Key Insight: The configuration name in openpi TrainConfig (e.g., pi05_libero) describes data loading configurations, while the weight_loader field is the only basis for model initialization. These two can have different names.

2. BC-RNN had a success rate of 0% across all 5 MimicGen tasks, and Coffee_D0’s SR was always 0 during 600 epochs#

Solution: Not fully resolved yet; confirmed through training logs that it was an inherent training failure rather than an evaluation issue. Further investigation is needed to determine whether the checkpoint quality and dataset paths were correct.

Key Insight: Historical training logs can distinguish between “evaluation issues” and “training failures,” avoiding wasting time on non-existent evaluation problems. BC-RNN’s complete failure under error recovery framework indicates that traditional sequence modeling strategies lack generalizability.

3. An53 node experienced a NODE_FAIL at 08:30, users suspected AI operations (srun) caused the failure#

Solution: Using sacct step timestamps systematically ruled out possibilities: AI operations were completed before 05:07, and srun never executed due to node congestion. The most likely cause of the NODE_FAIL was system OOM/GPU driver failure due to 8 GPUs fully loaded.

Key Insight: Sacct step timestamps are effective tools for tracing operation sequences and determining causal relationships.

4. STAIG fusion had all-zero embeddings on 151676, resulting in a NaN Spearman r during cross-section evaluation#

Solution: Added zero-embedding detection (norm < 1e-6) to load_fused_embeddings(), automatically triggering –recompute_fusion to retrain STAIG (RTX 2080 Ti, 300 epochs, approximately 50 seconds).

Key Insight: STAIG has a risk of training collapse in certain sections. Phase 2 cached embeddings cannot be trusted unconditionally and must be verified at runtime.

5. AI pre-started fresh-start training process when the user did not confirm the intention to resume#

Solution: The user stopped the process promptly, and AI killed the error process and recreated the resume script (–no-overwrite –resume), restarting it via SSH in the background.

Key Insight: Before performing resource-intensive operations (GPU training), it is necessary to confirm with the user whether to use resume/fresh-start. Relying on pre-determined judgments cannot be trusted.

General Issues#

6. P100 GPUs (sm_60) are incompatible with new PyTorch; common SSH remote execution issues (relative paths, uid mapping, variable expansion, multi-line Python heredoc)#

Solution: Switch to RTX 2080 Ti (sm_75) nodes; explicitly use absolute paths in SSH commands; specify -l username; use single quotes for nohup scripts; convert multi-line Python code into temporary script files.

Key Insight: There are fixed patterns and solutions for SSH remote execution issues. CUDA_VISIBLE_DEVICES remapping can affect MuJoCo EGL device numbers (physical GPU 1 should have EGL device id set to 0 under CUDA_VISIBLE_DEVICES=1).

Human Thinking vs AI Thinking#

Strategic Level#

Experimental Design and Key Decisions (Evaluation Framework, Base Model Selection, Resume vs Fresh-Start)#

Role Approach
Human Humans led all core experimental decisions: proposed RM-IDEAL + embedding cosine similarity cross-sample evaluation framework; actively identified potential risks of pi05_libero vs pi05_base; chose “immediate evaluation using current checkpoint” to save waiting time; stopped AI from mis-starting fresh-start in time and requested resume

| AI | AI was responsible for turning designs into code and systematically verifying the correctness of experimental configurations through multiple layers of evidence (config.py + training logs + comparison groups) |

Difference Analysis: Humans identified key risks based on domain knowledge and intuition, while AI provided systematic verification but sometimes required multiple rounds of correction. Humans’ pragmatic decisions (immediate evaluation, not covering checkpoints) saved a lot of computational resources and waiting time for experiments.

Problem Diagnosis (STAIG zero-embedding, NODE_FAIL responsibility determination, BC-RNN training failure identification)#

Role Approach
Human Humans relied on temporal relevance and intuition to make preliminary judgments (e.g., suspecting srun caused NODE_FAIL), sometimes making mistakes

| AI | AI provided more reliable root cause determinations through active debugging (checking embedding norm), structured evidence analysis using sacct, and reviewing historical training logs (confirming that Coffee SR was always 0). This was better than intuitive judgment. |

Difference Analysis: AI had an advantage in diagnosing issues that had already occurred (systematic verification, historical log analysis), but its proactive review of potential experimental design risks was not as effective as domain experts.

Cluster Operation Method Selection (sbatch vs SSH Direct Connection)#

Role Approach
Human Users explicitly requested not to use sbatch and instead used SSH direct connection to an49 in the background, based on a clear understanding of the cluster’s actual architecture (consistent paths, direct connection to compute nodes)

| AI | AI defaulted to using sbatch as the standard HPC scheduling method. After creating .sbatch scripts, it was interrupted; switching to SSH made it possible to execute effectively. |

Difference Analysis: This is a typical “general HPC knowledge vs specific cluster environment knowledge” difference. Users’ specific knowledge is more suitable for this cluster, while AI’s general best practices were inefficient in this scenario.

AI Limitations#

Important Limitations#

  • Lack of proactive review of experimental design rationality: Did not actively check whether the Pi0.5 LoRA fine-tuning base model configuration met expectations. It was only discovered after users raised concerns about potential issues between libero and base.
  • Lack of necessary pre-verification before resource-intensive operations (GPU training): Pre-started fresh-start training scripts before the user explicitly stated their intention to resume/fresh-start. Multiple attempts at ExitPlanMode also showed that it wasn’t accurate enough to determine when confirmation was needed.
  • There were intermediate misjudgments during analysis, requiring multiple rounds of verification to reach correct conclusions: Verified base models went through multiple redundant verifications; visual script debugging experienced 5 failures; auth token expiration caused multiple sub-agents to fail.

General Limitations#

  • Technical limitations of SSH remote execution: Multi-line Python heredoc escaping, no-upward variable expansion failures ($ts not expanded), process management (pkill being unreliable and requiring specifying PID) repeatedly occurred.

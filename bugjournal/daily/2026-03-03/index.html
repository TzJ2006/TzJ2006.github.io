Bug Journal 2026-03-03

March 3, 2026 · 12 min

Issues and Solutions#

Key Issues#

  1. SwiftUI .ultraThinMaterial displays a fixed color instead of transparent glass when showing videos.

    • “Solution: Replace SwiftUI materials with NSViewRepresentable to wrap the NSVisualEffectView, and specify .inWindow mode along with clear backgroundColor in CATransaction.setDisableActions(true).”
    • “Key Insight: SwiftUI’s abstraction layer hides the details of rendering processes. For complex visual effects like window-inner transparency, control at the NS level is necessary.”

  2. In 151676, all STAIG fusion embeddings result in zero values (embedding similarity is 0, Spearman r is NaN), indicating a complete model failure.

    • “Solution: Add a check for np.allclose(embeddings, 0) in load_fused_embeddings. If this condition is met, print a warning and automatically trigger the --recompute_fusion path to re-load the gene/visual embedding data from the cache and retrain STAIG (300 cycles, taking about 50 seconds on RTX 2080 Ti). After fixing, the embedding results return to normal (64 dimensions with a standard deviation of 1.0).”
    • “Key Insight: Due to unstable training, the STAIG fusion embeddings saved in the second phase may become zero. Active detection and automatic re-computation are required in the benchmark script to avoid manual intervention.”

  3. When switching videos after selecting one in NSOpenPanel, the old video is still played.

    • “Solution: Add a mechanism to detect mismatches between bookmark URLs, returning the original URL if there is no match during bookmark parsing, and adding a protection in ensureFileAccess.Solutions: Ensure that FileAccess parses the old video URL as per the bookmark on the screen, rather than requesting a new URL. A protection mechanism has been added to compare the parsed path with the requested path; if they differ, the original URL is returned (this URL has temporary sandbox permissions provided by NSOpenPanel).”

Key insights: There is a conflict between the bookmarks stored on each screen and the multi-video history; a protection mechanism is needed to detect that the bookmarks point to expired files.

The success rate of the Coffee task BC-RNN was 0%, suspected to be an evaluation error.#

Solutions: The training logs were checked, confirming that the Coffee BC-RNN training itself achieved an SR of 0% across all cycles (600 cycles). This was not an evaluation error; the training for this task indeed failed.

Key insights: It is always necessary to verify the training metrics before debugging the evaluation process. A zero value downstream may reflect a failure in upstream training.

The an53 node crashed at 08:30, resulting in the tmux session being lost, and task 46798 was executed.#

Solutions: An investigation was conducted through the sacct log, determining that the crash was caused by resource overload (8 GPUs were used up to 100%, with high CPU usage). The AI operation was ruled out as the cause (the last completion was at 05:07, 3 hours before the crash). The SSH host key mismatch issue was resolved using ssh-keygen -R an53.

Key insights: Slurm reported minimal details of the crash, but there was no root access to the system logs. The “Request node busy” error in the srun command indicated that the node was overloaded, not a command failure. When using Slurm, even without root access, minimal crash information can be obtained.

The CUDA kernel for P100 GPU (sm_60) is incompatible with PyTorch (sm_70+ version required); torch.cuda.is_available() returns True, but asynchronous operations fail.#

Solutions: Initially, an attempt was made to use try/except to catch the exception of torchzeros(1).cuda(), but since the asynchronous operations on P100 could not capture errors. The user successfully ran it after switching to RTX 2080 Ti (sm_75). Finally, the CUDA_VISIBLE_DEVICES="" environment variable was added to force the use of CPU in the script.

Key insights: torch.cuda.is_available() only checks whether the driver and library exist, without checking whether the GPU’s computing power is compatible with the current PyTorch. CUDA operations may fail asynchronously and cannot be caught by Python’s try/except. The most reliable way is to switch GPUs or use environment variables to force the use of CPU.

The JSON generated by the LLM was truncated (…truncated marker), resulting in daily_summary.py failing to parse on 2026-02-14 and other dates.#

Solutions: The truncated original text was sent as a separate prompt to AI, requesting only the repair/extraction of valid JSON sections; AI successfully restored the complete JSON structure.

Key insights: There is a risk of truncation when the LLM generates very long JSON (especially in scenarios where multiple segments are merged); a truncation detection and automatic JSON repair retry logic should be added during the merging stage, rather than relying solely on manual discovery and intervention.

After the screen saver clock text was updated, there was a 0.2-second afterimage left behind.#

Solutions: CALayer has an implicit animation of approximately 0.25 seconds for changes in content properties. When updating wrapped blurView.layer?.mask in CATransaction, set setDisableActions(true) to force immediate replacement.

Key insights: By default, CALayer has animations for almost all property changes; explicit transaction control is needed to achieve immediate visual updates.

The BC-RNN strategy evaluation crashed due to “Unable to process data type <class ’numpy.float64>’”.#

Solutions: Robomimic’s TensorUtils.to_tensor() was unable to handle numpy scalar types. The_to_robosuite_obs was modified, only passing keys that contained array/image values to the robomimic strategy.

Key insights: Robomimic expects a structured obs dictionary containing array values; the numpy scalars returned by robosuite (such as gripper_width_float64) must be filtered before being passed to TensorUtils.

The visualization script failed multiple times: Bash heredoc did not conform to != syntax error, file name prefix did not match (151673_filtered_feature_bc_matrix.h5 and filtered_feature_bc_matrix.h5), spatial/data path errors (located in Dataset/DLPFC, not directly in the section directory), load_dlpfc_data() returned a value error (returned 1 instead of 2), read_10x_h5 lacked uns[‘spatial’], resulting in sc.pl.spatial() failure.#

Solutions: Use an independent script/_viz_rm_ideal.py file instead of heredoc; use glob “*filtered_feature_bc_matrix.h5” to find the correct file name; directly load sc.read_visium("/path/to/Dataset/DLPFC/151673") (automatically handling the spatial subdirectory); fixed the issue where the unpacked result of load_dlpfc_data was a single return value; abandoned using load_dlpfc_data and used sc.read_visium to ensure the presence of uns[‘spatial’].

Key insights: Bash heredoc is very sensitive to special character escapes; complex scripts should be written as files. The DLPFC data file names have section prefixes; spatial data is located in Dataset/DLPFC/{section}/spatial instead of {section}/spatial; load_dlpfc_data used read_10x_h5 instead of read_visium, resulting in a lack of spatial metadata; sc.pl.spatial heavily relies on uns[‘spatial’], and must be loaded using read_visium.

Common Issues#

The screen saver clock text displayed gray ghosting effects on the edges.#

Solutions: The root cause was that the font weights between ScreensaverClockHighlight (.light) and renderTextMask (.regular) did not match, causing the mask to thicken and exposing the frosted glass edge of hudWindow. This issue was resolved by unifying the font weight to .regular.

Key insights: The visual effect layer must maintain pixel-level alignment; even a slight difference in weights can expose underlying content.

Processing historical data over 13 days required frequent switching of date parameters and manual confirmation of export/merge status, which was both tedious and error-prone.#

Solutions: Manually execute the daily_summary.py’s export –date YYYY-MM-DD and merge commands every day, handling each date individually, and confirming the integrity of the output files through the file system.

Key insights: The summarize tool lacks a batch processing mode (e.g., –date-range 2026-02-07..2026-02-19 automatically iterates); in the future, support for batch processing and progress recovery mechanisms should be added.

Using relative paths with nohup failed when starting from the home directory instead of the project directory via SSH.#

Solutions: Change the command from ‘ssh an53 “nohup bash scripts/…”’ to ‘ssh an53 “cd $PROJ && nohup bash scripts/…”’, ensuring that the working directory is correctly entered before the script execution.

Human and Artificial Intelligence Methodology#

Strategic Level#

Discovery of Configuration Errors in Pi0.5 Base Model Setup#

Role Method
Human The user noticed that LoRA fine-tuning used pi0.5_libero instead of pi0.5_base as the base model, which was considered a key architectural decision that required verification and possible retraining.
Artificial Intelligence During multiple evaluations, artificial intelligence did not actively detect this configuration issue. It focused on monitoring progress and success rates, without questioning the choice of the base model.

Difference analysis: Humans used their domain knowledge about model architecture to discover basic training configuration errors. Artificial intelligence ignored this strategic issue while focusing on operational metrics.

Diagnosis of Zero Embedding Issues#

Role Method
Human After discovering that the embedding similarity was 0, immediately asked “Can you show me the visualization results?” shifting from numerical inspection to visual diagnosis; after finding the root cause, requested methods for cross-sample patch queries.
Artificial Intelligence Initially attempted a full benchmark test for zero embeddings, only starting troubleshooting when NaN values appeared, without actively requesting visualizations; it was not until the user asked for visualization that a need for spatial heat maps was realized; documentation requires explicit user requests to begin.

Difference analysis: Humans actively drove the process from numerical inspection to visualization and then to method summary; artificial intelligence tended to solve individual problems step by step, lacking a global perspective and initiative.

Cause Analysis of an53 Node Failure#

Role Method
Human Humans correctly suspected that AI operations might have caused the failure of an53, requested timeline evidence, and mentioned peer reports about the 11:22 failure time.
Artificial Intelligence Artificial intelligence systematically analyzed Slurm logs (sacct), tracked all srun commands (04:55-05:07, all completed successfully), identified a NODE_FAIL issue at 8:30 (3 hours after AI activity), and diagnosed it as possibly due to resource overload.

Difference analysis: Humans used indirect evidence and peer reports to form hypotheses. Artificial intelligence used system logs and timestamps to build a clear timeline, proving that the operation ended 3 hours before the failure.

Design Decisions for Benchmarking#

Role Method
Human Humans explicitly specified testing only the PCA+UNI2+STAIG_FUSION combination, with target chapters being 151673 and 151676. First, they created separate scripts to verify feasibility, then integrated them into subsequent processes. They repeatedly rejected ExitPlanMode, expecting to execute directly when the direction was clear.
Artificial Intelligence The initial design covered multiple fusion methods and more chapters; after creating each plan, it tried using ExitPlanMode for approval; due to repeated rejections, it failed to understand the user’s execution style preferences.

Difference analysis: Humans made strategic decisions (scope control, priority sorting), while artificial intelligence tended to cover everything comprehensively; humans preferred direct execution modes, and there was a tension between the artificial intelligence’s conservative approval process and the user’s expected execution efficiency.

Batch Processing Strategy for Historical Data#

Role Method
Human Humans knew they needed to generate historical reports for 13 days, manually called the tool to process each date, and continued processing the next date when issues occurred.
Artificial Intelligence As a daily report analyst, artificial intelligence passively responded to each individual date request, reprocessing dialogues, extracting information, generating reports each time, without realizing that this was a batch task, nor did it actively suggest workflow optimization.

Difference analysis: Humans had a global intention for batch processing and time planning capabilities, while artificial intelligence could only see individual calls; artificial intelligence lacked the ability to actively identify repetitive patterns and suggest automation improvements.

Screen Saver User Experience in Video-Free Situations#

Role Method
Human Users expect the screen saver to remain active without video loading, displaying a black screen and a clock (consistent with the system’s screen saver).
Artificial Intelligence Artificial intelligence initially only saw the screen saver without video as an intentional design; it did not question the user experience assumption until it was explicitly requested.

Difference analysis: Humans applied a psychological model from the macOS system screen saver; artificial intelligence followed literal code constraints, without inferring broader user experience expectations.

Judgment of Granularity in Merging Multiple Dialogues | Role | Method |#

|——|—-| | Human | Users explicitly request to analyze multiple dialogue segments on the same day separately (possibly from different devices or projects), and then merge these segments into a complete daily report to maintain logical clarity. | | AI | AI strictly follows the instructions, analyzing each segment separately before merging them. However, this leads to redundancy in the merged output (repeating similar issues rather than truly summarizing). Humans need to emphasize “merge and refine, not copy verbatim” in their prompts. |

Difference analysis: Humans understand that “merging” is about information aggregation and deduplication, while AI tends to retain all details, resulting in an output that is lengthy. AI needs explicit instructions to achieve the quality of summary that humans expect.

Artificial Intelligence Limitations#

Key Limitations#

  • During multiple evaluations, failed to actively detect configuration errors in the pi0.5 base model (the difference between libero and the base model), until the user explicitly raised this issue.
  • “Lack of initiative and global perspective: ① Did not proactively propose a visualization solution, which required explicit user requests; ② Did not proactively suggest writing method papers, which also required user requests; ③ After multiple ExitPlanMode failures, still failed to understand the user’s execution style preferences (direct implementation rather than waiting for approval); ④ After discovering zero embedding issues, failed to actively diagnose the root cause (why the embeddings saved in the second phase would crash).”
  • “Error in the design of the compute_embedding_similarity function: It incorrectly called compute Embedding_similarity(embeddings_ref, niche_mask_query=None) in run_cross_section_benchmark, with incorrect parameters and confusing logic; it was only fixed after recreating the compute_cross_embedding_similarity function. This indicates a lack of deep understanding of cross-section similarity calculations.”
  • “Multiple failures in Bash heredoc and path handling: ① Bash heredoc != escaped with != (syntax error); ② Multiple misses of DLPFC file names with partial prefixes; ③ Mistakenly thought that spatial/ was located directly in the sub-directory, not in Dataset/DLPFC/{section}/spatial; ④ Misuse of load_dlpfc_data() resulting in a lack of uns[‘spatial’]. This indicates a lack of understanding of the DLPFC dataset file organization and scanpy API.”
  • “Error in the CUDA compatibility detection strategy: torch.cuda.is_available() only checks drivers and libraries, not GPU computing power compatibility; try/except cannot catch CUDA asynchronous failures; in the P100 asynchronous failure scenario, multi-layer CPU backtracking was ineffective. This indicates a lack of understanding of PyTorch CUDA mechanisms and asynchronous error propagation.”
  • As a daily report analyst, repeatedly called to handle similar tasks (analyzing historical dialogues → generating structured reports), but failed to recognize it as a batch processing scenario, nor did it actively suggest “whether batch processing mode is needed” or “automated scripts” to improve efficiency.
  • The API token expired multiple times during the sub-agent exploration task, hindering investigations of checkpoints and zhaoganlong folders. Manual intervention/log-in was required.
  • Before attempting to resolve the BC-RNN numpy dtype error, multiple debugging iterations were conducted (trying to track backtraces, ssh retries, checkpoint checks), and it was ultimately determined that to_robosuite_obs was the focus for filtering scalar types.
  • When seeing that the SR in the coffee evaluation was 0%, failed to actively check the training logs; before investigating the upstream training failure, a clear issue statement was needed.

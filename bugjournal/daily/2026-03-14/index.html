Human vs AI Approaches#

Strategic Level#

Gradient strategy selection in pi05 experiment design#

Role Approach
Human Directly chose option A (all stop_gradient) based on the reasoning ’ safest, does not affect action prediction, can be adjusted later’ — reflects a conservative starting strategy for iterative experiments
AI AI proposed 3 options (A full block, B full return, C hybrid) and analyzed trade-offs, preferring option A but emphasizing the uniqueness of special tokens

Difference Analysis: Humans make quick decisions based on engineering experience, while AI tends to list options and trade-offs thoroughly, but ultimately humans’ concise decision-making strategy (conservative first, then iterate) is reasonable

Naming and organization of experimental configurations#

Role Approach
Human Requires ‘config switches, easier to start experiments’ — reflects emphasis on experimental efficiency
AI AI designed a unified ProgressConfig class and 4 independent named configurations (exp1~4), allowing direct selection through tyro CLI

Difference Analysis: Humans focus on practicality and convenience, while AI translates this into a config class design and CLI interface, both agreeing on the goal of ’easy experiment switching’

AI Limitations#

Critical Limitations#

  • AI did not proactively anticipate data pipeline issues after modifying the model — should have checked if data loading supported new fields immediately after modification, rather than waiting for user errors
  • AI did not run actual training smoke tests when modifying pi05, only tested configuration imports, leading to missing dependencies in the data conversion script

Learnings#

Key Learnings#

  • Integrity checks for complex model experiments should cover model code, configuration, data pipeline, and training scripts, not just model code verification
  • Using stop_gradient in JAX/Flax for auxiliary tasks is a common multi-task learning pattern, allowing use of features while avoiding interference with the main task
  • Special token mechanism requires 3 steps: tokenizer registers token IDs, model extends embedding layers, and prompts are concatenated during data loading

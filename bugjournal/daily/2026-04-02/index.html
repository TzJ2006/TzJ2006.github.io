Problems & Solutions#

Critical Issues#

1. In Linux/Wayland, Tauri’s set_position() method is completely ignored by the synthesizer, causing the window to appear in the center of the screen#

Solution: Check the WAYLAND_DISPLAY environment variable at the run() entry. If GDK_BACKEND=x11 is forced under Wayland sessions, allow the application to run through XWayland.

Key Insight: Wayland protocol design prohibits clients from controlling the absolute position of windows. Any retry strategy is ineffective. You must revert to the X11 protocol layer.

2. The floating object’s layout is anchored at the screen border rather than the object itself, causing visual displacement when clicked to expand#

Solution: Change expanded_rect_from_ball to ball-anchored mode: When Left is anchored, expand x = ball_rect.x; when Right is anchored, expand x = ball_rect.right - expanded_width. Keep clamped_y stable to maintain the visual position of the object.

3. In ssh_remote.rs, if assert!() is used in host_cache_dir(), panicking occurs when path validation fails, causing the entire program to crash#

Solution: Change the function signature to Result<PathBuf, String>. Propagate the Result throughout the call chain. Use unwrap_or_default() at the calling point.

4. After setting the Linux GTK window to resizable(false), programming-based set_size() calls are rejected by the WM, preventing the floating object from collapsing#

Solution: In the Linux branch, temporarily unlock the window by setting set_resizable(true) before set_size(). Unlock it again after the operation is completed by setting set_resizable(false).

5. FloatBallState does not store the window geometry. Reading outer_position/outer_size from the WM may result in outdated values, leading to layout errors#

Solution: Add FloatBallState.last_rect and create an apply_and_store_rect helper function to synchronize the writing of the window and state records. Add a get_float_ball_position command for use by the front end.

6. sync_ssh_host blocks all readers during the SSH I/O wait, which lasts several seconds while holding a RwLock read lock#

Solution: Add #[derive(Clone)] to SshCacheManager. Clone the manager before waiting, and release the lock in advance.

7. The provider parameter is not validated, allowing for path traversal attacks (e.g., passing ‘../../../etc’ to create malicious paths)#

Solution: Add a validate_provider() function to only allow ‘claude’ and ‘codex’. Return Err for other values.

8. load_cached_records silently returns an empty Vec for all I/O errors (including permission denied errors), hiding the actual error#

Solution: Return a Result instead of an empty Vec. Clearly distinguish between ErrorKind::NotFound (legal, returning Ok(vec![]) and other I/O errors returning Err).

9. ssh_command considers ssh_command to have a non-zero exit code but some stdout as successful, accepting incorrect results#

Solution: Strictly check the exit code first. Only accept stdout when success(). Record stderr to the log when the exit code is non-zero.

10. Using both onMount and $effect in Svelte 5 components causes double requests during mounting#

Solution: Remove the onMount call. Keep $effect (which is executed automatically during mounting in Svelte 5).

11. Asynchronous registration of Tauri event listeners inside FloatBall’s onMount causes race conditions: The component may be destroyed before listen() can be resolved, resulting in cleanup functions not being called and causing memory leaks#

Solution: Introduce a destroyed boolean flag. If the Promise is resolved while the component is marked as destroyed, unlisten() immediately.

Learnings#

Key Learnings#

  • Wayland synthesizer prohibits clients from setting absolute window positions at the protocol level, so any set_position() retry strategy is ineffective; the correct solution is to detect WAYLAND_DISPLAY and force GDK_BACKEND=x11 back to XWayland
  • In Rust production code, assert!() and unwrap() are potential crash points when accepting external inputs (file paths, user parameters, SSH output); the unified approach is to change the function signature to Result and use ? to propagate it, with unwrap_or_default() handling call boundaries
  • GTK’s resizable(false) WM hint prevents both user dragging and programming-based set_size(); for transparent floating windows that need dynamic sizing, resizable must be temporarily unlocked before resizing
  • The geometric state of transparent overlay windows should be managed by the application layer (last_rect), rather than being reread from the WM every time; WM’s outer_position/outer_size may have timing delays after resizing, and reading outdated values during expansion/collapse can lead to layout errors
  • The expansion layout of floating windows should be based on the ball itself (ball-anchored), rather than the screen safe zone boundary, to ensure consistent visual position of the ball before and after expansion, with no user perception of jumping
  • In Rust asynchronous code, locking across awaits is a potential performance/deadlock issue; the solution is to Clone or Arc::clone the required data before awaits, releasing the lock in advance
  • Any user input concatenated to file paths must be whitelisted at the system boundary; “Only known values allowed” is more reliable than “Reject dangerous characters” to effectively prevent path traversal attacks
  • In Svelte 5, $effect executes once when components are mounted, and using it with onMount can lead to double requests; when front-end components asynchronously obtain cleanup handles, they need to use the destroyed flag to handle race conditions where components have shorter lifetimes than Promises

Practical Learnings#

  • git reset --soft <base> is an efficient way to organize commit history: It doesn’t require interactive rebasing and can compress any number of commits into one semantic commit, making it suitable for CI environments and automated tools
  • After adding new directory structures to the rclone sync configuration, these directories won’t be automatically created if data has never been pushed to the remote; when pulling data, it’s important to distinguish between “remote does not exist” (normal, considered successful) and “pull failed” (error), to avoid false positives

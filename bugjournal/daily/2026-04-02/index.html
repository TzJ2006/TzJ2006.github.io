Problems & Solutions#

Critical Issues#

1. In Linux/Wayland, Tauri’s set_position() method is completely ignored by the synthesizer, causing the window to appear in the center of the screen.#

Solution: Check the WAYLAND_DISPLAY environment variable at the run() entry point. If GDK_BACKEND=x11 is forced in Wayland sessions, the application will run through XWayland.

Key Insight: Wayland protocol design prohibits clients from controlling the absolute position of windows. Any retry strategy is ineffective. It’s necessary to revert to the X11 protocol layer.

2. The floating ball’s layout is based on the screen border rather than the ball itself, causing visual displacement when clicked.#

Solution: Change expanded_rect_from_ball to ball-anchored mode: When Left is anchored, expand x = ball_rect.x; when Right is anchored, expand x = ball_rect.right - expanded_width. This ensures that the ball’s visual position remains stable during expansion and contraction.

3. In ssh_remote.rs, if assert!() is used in host_cache_dir(), the program crashes due to a panic when path validation fails.#

Solution: Change the function signature to Result<PathBuf, String>. All call chains should propagate the Result. Call points should use unwrap_or_default().

Key Insight: Using assert!() in a production environment is like having a time bomb. Errors must be propagated through Results when dealing with external paths.

4. After setting the Linux GTK window to resizable(false), the programming-based set_size() function is rejected by the WM, preventing the floating ball from contracting.#

Solution: In the Linux branch, set_resizable(true) first to temporarily unlock the window. After the operation is complete, set_resizable(false) to lock the window again.

5. FloatBallState does not store the window geometry. Reading outer_position/outer_size from the WM may result in outdated values, leading to layout errors.#

Solution: Add FloatBallState.last_rect and create an apply_and_store_rect helper function to synchronize the writing of the window geometry and state. Also add a get_float_ball_position command for the front end.

6. sync_ssh_host blocks all readers when holding a RwLock read lock during SSH I/O waits.#

Solution: Add #[derive(Clone)] to SshCacheManager. Clone the manager before waiting, and release the lock in advance.

7. The provider parameter is not validated, allowing for path traversal attacks (e.g., passing ‘../../../etc’ to create a malicious path).#

Solution: Add a validate_provider() function to only allow ‘claude’ and ‘codex’. Other values should return Err.

8. load_cached_records silently returns an empty Vec for all I/O errors (including permission denied errors), hiding the actual errors.#

Solution: Return a Result instead, clearly distinguishing ErrorKind::NotFound (legal, returning Ok(vec![]) with an empty vector) from other I/O errors (returning Err).

9. ssh_command considers ssh_command to have a non-zero exit code but some stdout as successful, accepting incorrect results.#

Solution: Strictly check the exit code first. Only accept stdout when success(). Record stderr to the log when the exit code is non-zero.

10. Svelte 5 components use both onMount and $effect to trigger data requests, causing double requests during mounting.#

Solution: Remove the onMount calls and keep $effect (which is executed automatically during mounting in Svelte 5).

11. The async registration of Tauri event listeners inside FloatBall’s onMount triggers race conditions. The component may be destroyed before the listen() function can be called, causing memory leaks.#

Solution: Introduce a destroyed boolean flag. If the Promise resolves while the component is marked as destroyed, unlisten() should be called immediately.

General Issues#

12. When sync.py pulls data, 6 directories report ‘directory not found’. Users mistakenly assume the synchronization failed.#

Solution: Confirm that these sub-directories were never pushed to the remote. Consider the remote directory as non-existent in the sync logic. Instead of treating it as a failure, consider it as a case where there is data on the remote but the pull failed.

Learnings#

Key Learnings#

  • Wayland synthesizers prohibit clients from setting absolute window positions at the protocol level, making any set_position() retry strategy ineffective; the correct solution is to detect WAYLAND_DISPLAY and force GDK_BACKEND=x11 back to XWayland
  • In Rust production code, assert!() and unwrap() can be potential crash points when accepting external inputs (file paths, user parameters, SSH output). A unified approach is to change the function signature to Result and use ? to propagate the error, while handling call boundaries with unwrap_or_default()
  • GTK’s resizable(false) WM hint prevents both user dragging and programming-based set_size(); for transparent floating windows that need dynamic sizing, resizable must be temporarily unlocked before resizing
  • The geometric state of transparent overlay windows should be managed by the application layer (last_rect), rather than being read from the WM every time. The outer_position/outer_size of the WM may have timing delays after resizing, and reading outdated values during expansion/collapse can lead to layout errors
  • The expansion layout of floating windows should be based on the ball itself (ball-anchored), rather than the screen safe zone boundary, to ensure consistent visual position of the ball before and after expansion, with no user perception of jumping
  • In Rust asynchronous code, performance and deadlock issues can be detected at the point where locks are crossed await. The solution is to Clone or Arc::clone the required data before await, releasing the lock in advance
  • Any user input concatenated into file paths must be whitelisted at the system boundary. “Only allowed known values” is more reliable than “rejecting dangerous characters”, as it can effectively prevent path traversal attacks
  • In Svelte 5, $effect executes once when a component is mounted, and using it alongside onMount can lead to double requests. When front-end components asynchronously obtain cleanup handles, a destroyed flag needs to be introduced to handle race conditions where components have shorter lifetimes than Promises

Practical Learnings#

  • git reset –soft is an efficient way to organize commit history: It doesn’t require interactive rebasing and can compress any number of commits into one semantic commit, making it suitable for CI environments and automated tools
  • After adding new directory structures to the rclone sync configuration, these directories won’t be automatically created if data has never been pushed to the remote. When pulling data, it’s important to distinguish between “remote does not exist” (which is considered a success) and “pull failed” (which is an error), to avoid false positives

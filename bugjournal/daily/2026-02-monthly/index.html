本月关键成就#

  1. STAIG Strict模式ARI从~0.09突破至0.5435 (2026-02-07, MIHD) — 通过逐行对比MIHD与原始STAIG实现，发现4处关键差异：①UNI encoder跳过ImageNet归一化；②patch尺寸256×256+dynamic_img_size；③训练后去掉额外StandardScaler；④drop_feature()强制CPU生成随机mask。全部修复后ARI=0.5435，超过原始STAIG水平，验证了图像预处理细节对对比学习的级联影响。
  2. 发现并修复scGPT静默checkpoint加载bug，全11个DLPFC切片平均ARI+44.4% (2026-02-28, MIHD) — TransformerModel._init__未将use_fast_transformer保存为实例属性，导致Wqkv→in_proj key重映射永不执行，12层attention的Q/K/V权重（17.7%参数）随机初始化，strict=False静默跳过无任何报错。添加一行self.use_fast_transformer修复后，全部11个DLPFC section平均ARI从0.1177提升至0.1700（+44.4%）。scGPT官方GitHub上存在相同bug，属于可向上游提交的发现。
  3. MIHD两阶段Pipeline架构重构，计算量降低86%-99.6% (2026-02-14, MIHD) — 将embedding提取与fusion评估完全分离：先为每个切片缓存UNI2+基因encoder的embedding，再复用做所有fusion变体评估。UNI2提取次数从77次降至11次（-86%），数据加载从~3000次降至11次（-99.6%）。在RTX 5000 Ada端到端验证通过，ARI最高达0.42（staig_fusion），从根本上将重复计算从O(n×m)降至O(n+m)。
  4. QFormer Enhanced（register tokens+spatial bias）ARI=0.401，非STAIG最优 (2026-02-16, MIHD) — 对比plain qformer（ARI=0.344）提升+17%，仅需50 epochs。实验证明空间先验（spatial bias）的贡献超过复杂架构（register tokens），印证「空间先验>复杂融合架构」原则。同时确认编码器质量决定融合上限：PCA（0.288）远超scGPT（0.115），弱编码器+视觉特征简单融合有害（scGPT+UNI2 concat=0.028）。
  5. 力注入body定位根因修复+PreGrasp检测器6个Bug完整修复 (2026-02-14, ErrorRecoveryBenchmark) — 力注入：_get_eef_body_name()的possible_names列表缺少’gripper0_eef’，fallback匹配到bin2，修复后力正确施加到body_id=21。PreGrasp：删除永远返回False的stub、改用future_demo_actions前瞻夹爪、跳过全局距离门控、per-detector独立冷却、修复target body名称，验证pregrasp触发2次，41个单元测试全部通过。

反复出现的问题#

1. AI调参替代根因分析：面对性能异常时反复调整参数而非先定位系统性根因 (出现 5 次)#

日期: 2026-02-01, 2026-02-05, 2026-02-09, 2026-02-14, 2026-02-16 根本原因: AI缺乏对多系统交互的全局状态感知，倾向从当前可见的参数层面入手；在无明确假设时通过exhaustive search替代定向诊断；对「行为正常但作用对象错误」的silent failure模式（力施加到bin2、stub方法返回False、strict=False静默丢弃权重）缺乏主动验证意识 状态: 🔄 持续中

2. AI过度进入plan mode / ExitPlanMode被反复拒绝：无法区分「用户已规划→直接执行」与「AI需先规划」两种场景 (出现 14 次)#

日期: 2026-02-05, 2026-02-09, 2026-02-12, 2026-02-15, 2026-02-16, 2026-02-19, 2026-02-22, 2026-02-23 根本原因: AI将plan mode作为默认安全机制，即使在用户已提供详细指令的情况下仍走审批流程，将用户计划重新包装为新计划；AI未能从多次被拒中学习用户「信息充足即直接执行」的工作风格；跨会话状态丢失使AI无法累积用户偏好。用户最终采用「Implement the following plan:」直接提供计划文本绕过plan mode UI 状态: 🔄 持续中

3. 模型训练-评估任务不匹配：未在实验规划阶段主动验证checkpoint与目标任务的匹配性 (出现 3 次)#

日期: 2026-02-18, 2026-02-22, 2026-02-26 根本原因: AI默认假设当前使用的checkpoint适合评估任务，未主动建立「训练任务≡评估任务」的检查习惯。具体表现：Pi0 LIBERO checkpoint用于robosuite PickPlace（2/18）、Pi0/Pi0.5通用预训练checkpoint未明确区分zero-shot vs 微调后评估（2/22）、Pi0.5在9个MimicGen任务上训练却在PickPlace上评估（2/26）。三次均由人类通过领域知识或直接质疑发现，AI无一次主动识别 状态: 🔧 有变通方案

4. GPU OOM与并行资源竞争：并行启动多个GPU密集型任务导致进程被杀或静默失败 (出现 4 次)#

日期: 2026-02-09, 2026-02-14, 2026-02-15, 2026-02-16 根本原因: AI缺乏主动资源估算意识，未在启动前预判各进程显存需求；OOM进程不立即退出（长时间占用CPU），难以通过日志区分正常运行与OOM状态；conda run子进程输出全量缓冲至结束，监控手段受限 状态: 🔧 有变通方案

5. 代码接口假设错误：AI对跨环境API名称、参数格式、方法存在性做出错误假设 (出现 4 次)#

  • STAIG复现关键：①图像预处理（无ImageNet归一化、256×256 patch+dynamic_img_size）对对比学习有级联影响；②GCN应与多模态对比学习联合训练而非独立预训练；③超参数精确对齐（lr/layers/epochs/weight_decay）影响ARI可达0.05→0.45；④边权重方向（softmax_neglog vs softmax_log）显著影响信息传播；⑤mclust（高斯混合）显著优于kmeans，相同embedding下ARI差距0.13→0.37+；⑥drop_feature()必须强制CPU生成随机mask (来源: 2026-02-01, 2026-02-05, 2026-02-07)

  • 机器学习评估铁律：任何评估实验开始前必须验证训练任务⊆评估任务，且两者使用完全相同的observation space、action space和环境注册机制。VLA模型跨任务泛化边界严格：Pi0 LIBERO checkpoint在PickPlace上0% SR，Pi0.5在9个MimicGen任务上训练后在PickPlace上0% SR，说明现有VLA模型不能假设「大模型=通用」。 (来源: 2026-02-18, 2026-02-26, 2026-02-27)

调试 (debugging)#

  • PyTorch模型checkpoint加载陷阱：strict=False会静默丢弃不匹配的key，不产生任何警告或错误。每个用于控制模型结构的构造参数（如use_fast_transformer）都必须显式保存为实例属性（self.xxx = xxx），否则getattr的默认值会导致权重加载逻辑走错分支，造成隐蔽的随机初始化。验证方法：加载后检查missing_keys和unexpected_keys，对关键层做统计分布验证。 (来源: 2026-02-28)
  • 高效调试策略三则：①逐步调参无效时，git diff对比历史已知工作commit是快速定位代码回归的高效方法；②物理仿真调试用极端值（inf量级）先验证基础机制是否工作，比先建立精确力学模型更高效；③对silent failure（力施加到错误对象、stub方法永远返回False、strict=False权重随机初始化），必须先验证「作用对象是否正确、路径是否到达」再调整参数大小 (来源: 2026-02-01, 2026-02-14, 2026-02-15)
  • 代码审查最佳实践：①修改函数前先grep全局搜索所有定义位置，确认无同名局部实现覆盖工具模块；②追踪调用链到底层，确认无stub方法（同类中已有正确实现却因stub未被使用）；③Python falsy陷阱：步数等从0起计的变量不能用整数0作未初始化哨兵，应用None+is None判断；④创建独立评估脚本容易遗漏原有pipeline中的关键参数（如STAIG fusion的staig_alignment_config），导致双重归一化等隐蔽错误；复用已验证的run_benchmark.py是更安全的方案 (来源: 2026-02-09, 2026-02-14, 2026-02-16, 2026-02-20)
  • 分布式ML推理连接调试：http_proxy代理环境变量会拦截localhost的WebSocket连接，调试时应第一时间unset所有代理变量（http_proxy/https_proxy/HTTP_PROXY/HTTPS_PROXY），再用原生TCP socket验证端口可达性。robomimic在非TTY环境checkpoint目录已存在时调用input()立即触发EOFError，启动前必须清理旧checkpoint目录。GPU环境变量对齐：MUJOCO_EGL_DEVICE_ID必须等于CUDA_VISIBLE_DEVICES指定的GPU物理索引；nvitop等监控工具持有CUDA context占用每张GPU约14GB显存，启动大显存任务前必须kill清理。 (来源: 2026-02-19, 2026-02-23, 2026-02-28)

架构 (architecture)#

  • 两阶段解耦架构适用场景：当多个下游任务共享相同的上游计算（embedding提取）时，分离「数据收集/提取」与「分析/融合」阶段可将重复计算从O(n×m)降至O(n+m)；gadget/summarize的export/merge两阶段同理（本地无API→跨设备汇总）；策略驱动生成与demo注入在管线层等价（detection→injection→validation不关心action来源）。STAIG fusion是自包含的refine+fuse架构，不应与外部vision refine串联（两阶段融合信息损失陷阱：自监督精炼以聚类判别性为目标，丢失多模态互补所需的细粒度信息）。 (来源: 2026-02-14, 2026-02-15, 2026-02-22, 2026-02-23)
  • 多设备工作流设计原则：①上传/下载路径严格对称（export→logs/，merge→reports/）；②设备追踪字段按设备名索引（dict而非flat）；③幂等性通过_merged_devices持久化标记实现；④跨会话计划传递最健壮方案是将计划写入.md文件，而非依赖工具调用链；⑤os.replace是同一文件系统内的原子操作，配合tempfile完全避免文件损坏。批量评估脚本应在scene级别（而非run级别）捕获异常：append-only JSONL天然支持resume，数据采集与分析脚本应解耦，分析脚本通过nargs=’+‘支持多次运行合并。 (来源: 2026-02-15, 2026-02-19, 2026-02-23)
  • 研究项目文档管理：追加式增长（每次只添加章节不整合）导致同一概念被描述2-4次，文件从1554行增至冗余状态。定期结构性重构是必要实践；可操作checklist是文档核心不可压缩部分；四层文档体系：战略规划/操作指南/研究资料/历史归档；新增工具必须同步更新顶层README和CLAUDE.md。benchmark论文核心论证结构四层递进：大规模数据集→分类有效性论证→基线性能建立→Foundation Model有用性证明；Kruskal-Wallis+Dunn事后检验验证错误分类区分度；Cohen’s kappa矩阵验证分类器非冗余性（跨category kappa<0.3为标准）。 (来源: 2026-02-19, 2026-02-22, 2026-02-23)

Problems and Solutions#

Key Issues#

1. Save_episode Error in LeRobot 0.4.0: observation.dones field shape is (1,), which can only be converted to Python scalar#

Solution: The root cause is LeRobot’s get_hf_features_from_features() method mapping data shaped (1,) to scalar values, while validate_frame() requires ndarray. A loop was added before save_episode() to convert all dones arrays to Python integers.

Key Insight: When framework verification and coding logic conflict, the best insertion point is in the buffer between them, rather than modifying feature definitions or bypassing verification.

2. “Protobuf Parsing Failure” Error When Starting Screenpipe 0.2.13, Loading Wespeaker_en_voxceleb_CAM++.onnx Model#

Solution: Discovered that the Homebrew version was outdated and unable to build. Uninstalled the version, deleted corrupted model cache (~/Library/Caches/screenpipe/models/), switched to npx screenpipe@latest or official desktop application, and updated all documentation.

Key Insight: An outdated Homebrew formula is a serious issue. When encountering underlying dependency problems, check the upstream maintenance status instead of debugging outdated versions.

3. Despite Normal Data, Battery Training Losses Became NaN From First Iteration#

Solution: Systematically traced the entire process and found that due to corrupted rows (Battery 31, 833 loops), there were 4 NaN values in the loop columns. StandardScaler calculated the mean and standard deviation of NaN values, causing all scaled values to become NaN. This corrupted row was removed.

Key Insight: NaN values propagate through normalization: a single NaN in training data can disrupt the statistical properties of StandardScaler, causing all subsequent scaled values to become NaN. Ensure data cleanliness before normalization operations.

4. TokenMonitor Costs Not Consistent with Claude Code’s/cost Output#

Solution: Found that TokenMonitor uses 2.0 times the 1-hour cache token rate, while Claude Code uses a uniform 1.25 times rate. Changed the pricing of cache_write_1h in pricing.rs to match Claude Code’s.

Key Insight: Claude Code’s/cost intentionally simplifies pricing (uniform rate), although the API provides detailed segmentation. The simplicity of the UI justifies the aggregation of data—not every field needs to be displayed.

5. CircuitBreaker HALF_OPEN Concurrency Issues: Outside lock checks allow multiple programs to enter simultaneously#

Solution: Moved state checks into the “async with self._lock” block to ensure atomicity of state checks.

Key Insight: Concurrent state machines must execute state checks and transitions within the same lock. Any outside lock checks will introduce a window of concurrency.

6. MIHD Paper Positioning Unclear—Initially defined as “Fusion Strategy Benchmark,” core contribution description not clear#

Solution: Relocated to “Frozen FM + Unsupervised Fusion Paradigm Evaluation,” answering whether this approach can replace fine-tuning methods. The narrative shifted from method proposal to system boundary exploration.

Key Insight: The benchmark paper evaluates the paradigm, not the specific method. The core value lies in system boundary exploration, not claims of cutting-edge technology. Precise problem definition determines the correct narrative framework.

7. LiteLLM Pricing Database Lacks Major Chinese AI Models (GLM, Qwen, DeepSeek)—There is a systemic gap in the Chinese AI ecosystem#

Solution: Merged OpenRouter API (including 92 Chinese model information) with LiteLLM, providing simultaneous coverage: OpenRouter provides basic coverage, LiteLLM covers when available. Added 7-day auto-refresh function.

Key Insight: Global AI pricing requires multi-source strategies and considers geographical diversity. Western databases systematically lack Chinese models; Chinese sources also lack Western models. Redundancy is necessary rather than optional.

8. Initial Uniform Sampling Used Fixed 0-20 Second Range, Resulting in Invalid IR Values (16, 15, 14…), Instead of Expected 0-0.024 Range#

Solution: Changed strategy: interpolated within the actual price time range for each cycle (usually 10-13 seconds), avoiding forcing data into a fixed global range. This avoids extrapolating beyond observed data points.

Key Insight: Interpolation and extrapolation: Forcing data into a fixed global range beyond observed data points leads to invalid values. Use sample-adaptive ranges to maintain interpolation accuracy.

9. The Term “Zero Sample” Can Be Misunderstood—STAIG Fusion Training Spent 550 Cycles, If Really “Zero Sample,” Reviewers Would Raise Questions#

Solution: Changed to “No Tags/Annotations,” emphasizing the lack of manual annotation required. Clearly stated that “zero sample” refers to the encoder level (frozen FM), while fusion is unsupervised, using spatial coordinates as free signals.

Key Insight: Precise terminology avoids confusion among reviewers. Hierarchical definition (frozen encoder + unsupervised fusion) avoids ambiguity and highlights the actual value (no annotation costs).

10. GitHub Access Failure on Tianjin Cluster: “Unable to Connect to 172.16.31.200 Port 3138: Host Not Found”#

Solution: Diagnosed that the proxy configuration was outdated (172.16.31.200). User provided “source /app/bin/proxy.sh” command. Updated git config http/https.proxy to new proxy address (99.72.0.200:3138), npx openskills installation succeeded.

Key Insight: High-performance cluster network proxies often change. When encountering network errors, verify the validity of the proxy (env | grep proxy, git config –list), rather than assuming the configuration is correct.

Learning Outcomes#

Key Learning Outcomes#

  • In LeRobot 0.4.0, there are contradictions in the shape=(1,) feature processing design: validate_frame requires ndarray, while get_hf_features_from_features maps it to scalar values. The solution is to insert a type conversion layer between validation and encoding.
  • NaN propagation mechanism of StandardScaler: Any row containing NaN in any feature column will cause StandardScaler.fit() to calculate NaN statistics (mean/stddev), and propagate this to all scaled data through (x - NaN) / NaN = NaN. It is necessary to verify for NaN/Inf situations before normalization.
  • Benchmark papers evaluate paradigms rather than methods - Papers in Nature Methods Analysis do not require SOTA, but need systematic answers to “how far this method can go.” The narrative shifts from “Our method is the best” to “What are the boundaries and bottlenecks of this paradigm?”
  • The fact that Homebrew formula is no longer in use is an important signal - When encountering low-level dependency issues, one should first check the maintenance status of upstream versions instead of debugging deprecated versions.
  • Development in graph contrast learning in spatial transcriptomics: From STAIG (node-level) to QueST (subgraph-level), then to Loki (cross-modal), showing a trend from “one method per problem” to “one base model applicable to multiple tasks.”
  • Hierarchical repair scheme for code review: From CRITICAL (safe/competitive) to HIGH (deprecated APIs/private access), then to MEDIUM (types/styles), and finally to LOW (code inspectors). Testing and verification are required after each level to avoid cascading failures.
  • Correctness invariant of Asycio state machines: State checks and transitions must be atomic within the same lock region; any checks outside the lock region will introduce a race condition window.
  • Global AI pricing needs a multi-source strategy with geographical diversity - Western databases like LiteLLM systematically lack Chinese models; Chinese data sources also lack Western models. Redundancy is essential, not optional.
  • Performance patterns of Rust: Avoid unnecessary clone(), use borrowing structures for serialization, prioritize line counting over full deserialization when only metadata is needed.
  • Term selection is important: No-label vs zero-shot - “Zero-shot” can be misinterpreted as “no training at all,” while no-label/no-annotation accurately conveys the core advantage of “no need for manual annotation.” Hierarchical definitions (zero-shot encoder + fusion self-supervised) can avoid ambiguity.
  • Parallel proxy strategies significantly accelerate the generation of large-scale documents - Dividing the 2246-line document into 3 independent sections (paper plan, repository structure, module specifications), each section generated and assembled by a separate proxy. This also accelerated the research: 4 proxies completed in 14 seconds, while sequential processing took 45-60 seconds.
  • “Free signal” framework of self-supervised learning - Spatial transcriptomics coordinates are inherent structured information of the Visium chip. Treating spatial graphs as “free supervision signals” clarifies the value of no-label data and enhances its usefulness (without additional annotation costs).
  • Interpolation and extrapolation in time resampling: Linear interpolation should remain within the range of observed data. Forcing data into a fixed global range will lead to invalid values in extrapolation. Using per-sample adaptive ranges can avoid artifacts caused by extrapolation.
  • End-to-end verification is required for the correctness of multi-stage data processing - Special values like -1 must be consistent across all stages (preprocessing → dataset → loss function → evaluation).
  • In Claude Code, /cost intentionally simplifies pricing display (uniform 1.25x cache rate), although the API provides detailed 50,000/t-hour breakdown. The simplicity of the UI justifies data aggregation - not every field needs to be made public.
  • HPC cluster network proxies often change - Establishing standard inspection workflows: env | grep proxy + git config –list + test connection, rather than assuming the correctness of configurations.
  • Migration mode of Pydantic v2: dict→model_dump(), but test simulations must provide the same interface via auxiliary functions like make_settingsMock().
  • JSONL session logs do not contain all billed attributes - The fast mode level is invisible in logs, although it affects actual costs, limiting which cost tracking tools can reconstruct data from historical data.

Practical Learning Outcomes#

  • Weekly updates to pricing (7-day validity), combined with background checks, achieved the optimal balance for LLM cost tracking: frequency high enough to capture new models and rate changes, yet low enough to avoid API quota issues.

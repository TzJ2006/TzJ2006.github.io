12. The curobo compilation failed, with the message “cuda_runtime_api.h: No such file or directory”, even though the CUDA toolkit was installed in the RefineVLA’s conda environment.#

Solution: The CUDA header file is located in a non-standard location (targets/x86_64-linux/include instead of include/). This issue was resolved by setting the CPLU_INCLUDE_PATH and C_INCLUDE_PATH environment variables before pip installation.

Key insights: The CUDA toolkit installed using Conda may place the header files in sub-directories of a specific architecture instead of the standard path. Always use the find command to check the actual location of the header files before assuming the standard path.

Lessons Learned#

Main Lessons Learned - The combination of EventBus, BackgroundCoordinator, and ContextAssembler is a powerful pattern for building modular AI assistants: EventBus implements service decoupled communication (publishing-consumeing), BackgroundCoordinator manages the lifecycle of scheduled tasks (CronScheduler + conditional loading), and ContextAssembler dynamically loads context layers based on intent types (avoiding full data transfers every time). Together, they support the collaboration of more than 16 independent services.#

  • The Spearman correlation of RM-IDEAL shows significant directional asymmetry: The correlation coefficient ρ between 151508 and 151673 in Layer_4 is 0.657, while the reverse relationship between 151673 and 151508 is only 0.275. This indicates that there are differences in the fused embedding space for the same label across different samples, which may be related to sample point density, noise level, or annotation quality.
  • The topological structure of intermediate layers (Layer_3/6) in the embedding space shows negative correlations (ρ ranges from -0.21 to -0.36): This may be because these layers are confused with adjacent layers (Layer_2/4/5) in the embedding space, resulting in the order of embedding being opposite to the order of RM-IDEAL based on topology. This indicates that boundary layers (Layer_1/WM) and central layers (Layer_5) are easier to distinguish, and intermediate transition layers require stronger discriminative features.
  • Data preprocessing processes need to be clearly verified to ensure that all expected features have standardized statistical information. Silent skipping (strict=False) may lead to scale mismatch issues throughout the training process, without causing errors.
  • For large-scale refactoring projects, referring to mature open-source project architectures (such as openclaw’s ACP spawn/Gateway/plugin SDK, get-shit-done’s wave execution/context engineering/200K window) is more reliable than zero design. However, it is necessary to adapt to the conventions of the target language (such as TypeScript→Python’s async/await, singleton factory functions, dataclass), and cannot be directly copied.
  • Unit testing (isolating components, simulating dependencies, providing quick feedback) and integration testing (real Discord bot interactions, end-to-end experience) are two complementary dimensions: the former ensures code correctness (319 test cases passed), while the latter ensures that the user experience meets expectations (29 manual test checks); 68 unit test passes do not mean the system is ready, and real user scenarios are needed for verification.
  • Dangerous silent exception handling (only handling Exception exceptions) is a technical debt tumor: It makes the system appear to run normally, but in reality, the state is incomplete (situation_monitor’s energy/diet/exercise/ideas/reminders may all be empty), and the source of the error cannot be found during debugging (Schrödinger’s state). Exceptions should always be logged (logger.error/warning), with distinction between critical functions (errors) and minor functions (warnings), and diagnostic information should be retained.
  • Large-scale refactoring should gradually verify key integration points (service startup, event connection, data persistence), rather than implementing all functions first and then testing. The ‘implement all functions first and then test’ approach will result in a system that is technically feasible but cannot run properly (services exist but will never start).
  • Project description narrative strategy: External communication should start with aspects that are relevant to the target audience of the application scenario (such as clinical patch query knowledge transfer), then explain the technical path and key differences (zero-shot vs. end-to-end training), and finally elevate to domain value. Technical details serve the narrative logic, not the other way around. Control each sentence within double short clauses to reduce cognitive burden.
  • Background services need to explicitly call the startup function in application lifecycle hooks. Class definitions and dependency injection are not enough - it is necessary to call coordinator.start_all() in on_ready() or similar functions.
  • Solution to Python loop import problem: Use deferred imports at the package level (only using all, not importing immediately, add explanatory comments), import at the module level inside functions (import deferred until the actual need arises), and do not call the site when testing patch source modules (because deferred imports do not create new bindings at the site call); three-layer defense ensures scalability.
  • Parallel agent execution greatly accelerates the development of independent modules. Four background agents (context_detector, checkin_service, learning_service, energy_projection) complete complex enhancement functions simultaneously, while the main thread handles joint modules. The total time is about 140 seconds, while the sequential method takes about 560 seconds (4 times faster).
  • Effective code auditing requires multi-dimensional analysis: syntax level (TODO/FIXME/NotImplementedError), behavior level (exception handling patterns, empty loops, stub methods), structure level (missing test coverage, unused exports), and operation level (service registration, startup hooks). Auditing from a single dimension will overlook problems across the entire category.
  • Creating large test suites benefits from parallel agent execution and shared fixtures. The reusable simulators in conftest.py (mock_settings, mock_notification) make it possible to write consistent tests across more than 20 files.

Practical Learning#

  • Long-running background task monitoring strategy: For hour-long computing tasks, use run_in_background and monitor progress in the cache file system (file modification time, number of completed sub-tasks), rather than relying on potentially failed standard output redirection. The final result should be read from the persistent output directory, not the temporary log files.
  • UNI/UNI2 standardization mechanism: Both perform L2 standardization at the output layer (F.normalize(features, dim=-1)), ensuring that the L2 norm of the feature vector is 1. The input layer uses ImageNet standardization by default unless skip_normalize=True. This two-layer standardization ensures geometric consistency in the feature space, making it easy to calculate cosine similarity.
  • The CUDA toolkit installed using Conda places the header files in a specific architecture path (targets/x86_64-linux/include), instead of the top-level include/ directory.

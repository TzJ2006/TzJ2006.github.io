Lessons Learned#

Main Lessons – The combination of EventBus, BackgroundCoordinator, and ContextAssembler is a powerful pattern for building modular AI assistants: EventBus enables decoupled communication (publish-subscribe), BackgroundCoordinator manages the lifecycle of scheduled tasks (CronScheduler + conditional loading), and ContextAssembler dynamically loads context layers based on intent types (avoiding full data transfers every time). These three together support collaboration among more than 16 independent services.#

  • The Spearman correlation in RM-IDEAL showed significant asymmetry in directionality: The correlation coefficient ρ between 151508 and 151673 in Layer_4 was 0.657, while the reverse relationship between 151673 and 151508 was only 0.275. This indicates that there are differences in the fusion embedding space for the same label across different samples, which may be related to sample point density, noise level, or annotation quality.
  • The topological structure of intermediate layers (Layer_3/6) in the embedding space showed negative correlations (ρ ranging from -0.21 to -0.36): This might be because these layers are confused with adjacent layers (Layer_2/4/5) in the embedding space, leading to the opposite ranking of RM-IDEAL based on embedding similarity compared to topology-based ranking. This indicates that boundary layers (Layer_1/WM) and central layers (Layer_5) are easier to distinguish, and intermediate transition layers require stronger discriminative features.
  • Data preprocessing processes need to be clearly verified to ensure that all expected features have standardized statistical information. Silent skipping (strict=False) may lead to scale mismatch issues throughout the training process, without causing errors.
  • For large-scale refactoring projects, referring to mature open-source project architectures (such as openclaw’s ACP spawn/Gateway/plugin SDK, get-shit-done’s wave execution/context engineering/200K window) is more reliable than zero design. However, it’s necessary to adapt to the conventions of the target language (such as TypeScript→Python’s async/await, singleton factory functions, dataclass), and cannot be directly copied.
  • Unit testing (isolating components, simulating dependencies, providing quick feedback) and integration testing (real Discord bot interactions, end-to-end experience) are two complementary dimensions: the former ensures code correctness (319 test cases passed), while the latter ensures that the user experience meets expectations (29 manual test checks). Having 68 unit tests passed does not mean the system is ready; real user scenarios are needed for verification.
  • Dangerous silent exception handling (only handling Exception exceptions) is a technical debt tumor: It makes the system appear to run normally, but in reality, the state is incomplete (situation_monitor’s energy/diet/exercise/ideas/reminders may all be empty), and the source of the error cannot be found during debugging (Schrödinger’s state). Exceptions should always be logged (logger.error/warning), with distinction made between critical functions (errors) and minor functions (warnings), and diagnostic information retained.
  • Large-scale refactoring should gradually verify key integration points (service startup, event connection, data persistence), rather than implementing all functions before testing. The approach of “implementing all functions before testing” results in systems that are technically feasible but cannot run properly (services exist but never start).
  • Project description narrative strategy: External communication should start from aspects that matter to the target audience (e.g., clinical patch query knowledge transfer), then explain the technical path and key differences (zero-shot vs. end-to-end training), and finally elevate to domain value. Technical details serve the narrative logic, not the other way around. Control each sentence within double short clauses to reduce cognitive burden.
  • Background services need to explicitly call the startup function in application lifecycle hooks. Class definitions and dependency injection are not enough—they must be called in on_ready() or similar functions (coordinator.start_all()).
  • Methods to solve Python loop import issues: Use delayed imports at the package level (only using all, not immediately importing, add explanatory comments), import at the module level inside functions (delayed until actually needed), and do not call the site when testing patch source modules (since delayed imports do not create new bindings at the site call); three-layer defense ensures scalability.
  • Parallel agent execution greatly accelerates the development of independent modules. Four background agents (context_detector, checkin_service, learning_service, energy_projection) complete complex enhancement functions simultaneously, while the main thread handles joint modules. The total time is about 140 seconds, while the sequential approach takes about 560 seconds (4 times faster).
  • Effective code auditing requires multi-dimensional analysis: syntax level (TODO/FIXME/NotImplementedError), behavior level (exception handling patterns, empty loops, stub methods), structure level (missing test coverage, unused exports), and operation level (service registration, startup hooks). A single-dimensional audit will overlook entire categories of problems.
  • Creating large test suites benefits from parallel agent execution and shared fixtures. The reusable mockers in conftest.py (mock_settings, mock_notification) make it possible to write consistent tests across more than 20 files.

Practical Learnings#

  • Strategies for monitoring background tasks that run for hours: For hour-long computing tasks, use run_in_background and monitor progress in the cache file system (file modification time, number of completed sub-tasks), rather than relying on standard output redirection that may fail. The final results should be read from the persistent output directory, not from temporary log files.
  • UNI/UNI2 standardization mechanism: Both perform L2 normalization at the output layer (F.normalize(features, dim=-1)), ensuring that the L2 norm of feature vectors is 1. The input layer uses ImageNet normalization by default unless skip_normalize=True.

Problems & Solutions#

Critical Issues#

1. Openpi-moe training didn’t error when norm_stats.json lacked prev_actions key, leading to scale mismatch in trajectory VAE (unnormalized hist_actions concatenated with normalized actions)#

Solution: Root cause: transforms.py apply_tree() uses strict=False by default, silently skipping normalization for missing keys. Fixed by modifying compute_norm_stats.py to dynamically detect prev_actions in batches and compute its statistics

Key Insight: Silent failures in data preprocessing pipelines are dangerous. apply_tree’s lenient behavior (strict=False) allows scale mismatches to propagate through training. Always validate that all expected keys have normalization statistics, especially for concatenated features

2. services/init.py eager imports caused circular dependency (services ↔ core.scheduler), tests couldn’t import any service modules#

Solution: Changed all imports to lazy import, keeping only all list and comments, requiring users to directly import (e.g. from src.services.calendar_service import CalendarService), avoiding circular triggers during package initialization

Key Insight: Python package init.py doesn’t have to provide immediate imports; for packages with circular dependencies, lazy import + documentation comments is better choice; package-level interface exposure doesn’t equal mandatory eager load

3. 16 dangerous except Exception: pass swallow all errors, causing incomplete state but system appears to run normally, can’t find error source when debugging#

Solution: Replaced with except Exception as e: logger.error/warning(f"descriptive info: {e}"), preserving error information for diagnosis, distinguishing error (critical functionality) and warning (secondary functionality)

Key Insight: Never use empty pass to handle exceptions; if must ignore, at least log; silent failures in state tracking layers (situation_monitor/life_context) lead to ‘Schrödinger’s state’ - don’t know if truly empty or load failed

4. How to implement massive 31-goal upgrade plan efficiently without sequential bottlenecks?#

Solution: Used parallel background agent execution pattern: launched 4 independent agents for context_detector, checkin_service, learning_service, and energy_projection enhancements, while simultaneously performing direct modifications on 8 other files. All agents completed successfully with no conflicts

Key Insight: Wave-based parallel execution dramatically accelerates large-scale refactoring when dependencies are properly isolated. Key is identifying truly independent work units that can run concurrently

5. AI repeatedly attempted to enter plan mode, but user preferred direct execution#

Solution: User rejected ExitPlanMode tool calls, directly provided plan content for AI to execute. AI read results from previous research agents in plan mode transcript then directly wrote documentation without re-planning

Key Insight: User work style favors rapid iteration over lengthy planning; AI should adaptively reduce plan mode trigger frequency, or learn this preference after multiple user rejections

6. CalendarPro had 6 completely untested source files, 12 modified files with untested new methods, and BackgroundCoordinator never started despite being implemented#

Solution: Created systematic test infrastructure: conftest.py for shared fixtures, 20 new test files following existing patterns (pytest, AsyncMock, patch decorators), fixed on_ready() to actually call coordinator.start_all(), added 9 missing Discord intent handlers

Key Insight: Large-scale features often ship with incomplete integration. Background services need explicit startup calls, not just class definitions. Systematic testing requires shared fixtures and consistent patterns across all test files

7. Test fixtures had incorrect patch paths (patched call site instead of definition site), causing mock failures after lazy import, multiple tests reported AttributeError#

Solution: All singleton factory function patches (get_settings/get_calendar_service etc.) must be at source module (src.config/src.services.calendar_service), not call site (e.g. src.services.gap_analyzer), because lazy import delays name binding

Key Insight: unittest.mock.patch golden rule - patch where it’s looked up, not where it’s defined; lazy import means ’looked up’ location is always source module, won’t create new binding at import site

8. How to discover hidden bugs and incomplete implementations in large codebase after major refactoring?#

Solution: Launched multiple specialized exploration agents: TODO/FIXME searcher, git status analyzer, test gap detector, and exception handler auditor. Systematic multi-angle audit revealed critical issues: incomplete executor loop, stub get_progress, unimplemented meal preference learning, and 50+ dangerous silent exception handlers

Key Insight: Code audits require systematic multi-dimensional search: static analysis (TODOs, NotImplementedError), behavioral analysis (exception handlers, empty loops), and integration analysis (service registration, startup hooks). Single-angle audits miss categories of bugs

Learnings#

Key Learnings#

  • EventBus + BackgroundCoordinator + ContextAssembler combination is powerful pattern for building modular AI assistants: EventBus achieves service decoupling communication (pub-sub), BackgroundCoordinator unified management of scheduled task lifecycle (CronScheduler+conditional loading), ContextAssembler dynamically loads context layers based on intent type (avoiding full data transmission every time), three together support 16+ independent services collaboration
  • RM-IDEAL Spearman correlation exhibits significant directional asymmetry: 151508→151673 Layer_4 ρ=0.657, but reverse 151673→151508 only 0.275. This reveals fusion embedding space quality differences in structural representation of same niche label across different samples, possibly related to sample spot density, noise level or annotation quality
  • Middle layers (Layer_3/6) show negative correlation with topological structure ranking in embedding space (ρ -0.21 to -0.36): possibly because these layers are confused with adjacent layers (Layer_2/4/5) in embedding space, causing embedding-based similarity ranking opposite to topology-based RM-IDEAL ranking. This suggests boundary layers (Layer_1/WM) and central layers (Layer_5) are easier to distinguish, middle transition layers need stronger discriminative features
  • Data preprocessing pipelines need explicit validation that all expected features have normalization statistics. Silent skipping (strict=False) can propagate scale mismatches through entire training runs without errors
  • For large refactoring projects, referencing mature open-source project architecture (openclaw’s ACP spawn/Gateway/plugin SDK, get-shit-done’s wave execution/context engineering/200K window) is more reliable than zero design; but need to adapt to target language idioms (TypeScript→Python async/await, singleton factory functions, dataclass), can’t directly copy
  • Unit tests (isolated components, mock dependencies, rapid feedback) and integration tests (real Discord Bot interaction, end-to-end experience) are two complementary dimensions: former guarantees code correctness (319 tests passing), latter guarantees user experience meets expectations (29 manual test checklist items); 68 unit tests passing ≠ system usable, must have real user scenario validation
  • Dangerous silent exception handling (except Exception: pass) is technical debt toxin: it makes system appear to run normally but state incomplete (situation_monitor’s energy/meals/exercise/thoughts/reminders could all be empty), can’t find error source when debugging (Schrödinger’s state); should always log exceptions (logger.error/warning), distinguish critical functionality (error) and secondary functionality (warning), preserve diagnostic information
  • Large-scale refactoring should incrementally validate critical integration points (service startup, event wiring, data persistence) rather than implementing all features first then testing. ‘Implement everything then test’ approach created functioning but non-operational system (services exist but never start)
  • Project description narrative strategy: external communication should start from application scenario target audience cares about (e.g. clinical patch query knowledge transfer), then explain technical path and key differentiators (zero-shot vs end-to-end training), finally elevate to domain value. Technical details serve narrative logic, not vice versa. Control each sentence within dual short clauses to reduce cognitive load
  • Background services require explicit startup calls in application lifecycle hooks. Class definitions and dependency injection are insufficient - coordinator.start_all() must be called in on_ready() or equivalent
  • Python circular import systematic solution: package-level use lazy import (only all, no eager import, add explanatory comments), module-level use function-internal import (delay until truly needed), when testing patch source module not call site (because lazy import won’t create new binding at call site); three-layer defense ensures extensibility
  • Parallel agent execution dramatically accelerates independent module development. Four background agents (context_detector, checkin_service, learning_service, energy_projection) completed complex enhancements concurrently while main thread worked on coupled modules. Total wall-clock time ~140 seconds vs sequential ~560 seconds (4x speedup)
  • Effective code audits require multi-dimensional analysis: syntactic (TODO/FIXME/NotImplementedError), behavioral (exception handling patterns, empty loops, stub methods), structural (missing test coverage, unused exports), and operational (service registration, startup hooks). Single-dimension audits miss entire categories of issues
  • Large test suite creation benefits from parallel agent execution and shared fixture patterns. conftest.py with reusable mocks (mock_settings, mock_notification) enables consistent test authoring across 20+ files

Practical Learnings#

  • Long-running background task monitoring strategy: for hour-level computational tasks, should use run_in_background and monitor progress through cache file system (file modification time, completed sub-task count) rather than relying on potentially failing stdout redirection. Final results should be read from persistent output directory, not temporary log files
  • UNI/UNI2 normalization mechanism: both do L2 normalization at output layer (F.normalize(features, dim=-1)) ensuring feature vector L2 norm is 1, input layer defaults to ImageNet normalization (unless skip_normalize=True). This dual-layer normalization ensures geometric consistency of feature space, friendly to cosine similarity calculation
  • Conda-installed CUDA toolkits place headers in architecture-specific paths (targets/x86_64-linux/include) rather than top-level include/. Always verify actual header locations before assuming standard paths

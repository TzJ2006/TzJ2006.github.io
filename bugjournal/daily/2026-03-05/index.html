Problems and Solutions#

Key Issues#

1. Openpi-moe Training Does Not Error When norm_stats.json Lacks prevActions Key, Leading to Scale Mismatch in Trajectory VAE (Unnormalized hist Actions Merged with Normalized Actions)#

Solution: The root cause is transforms.py’s apply_tree() defaulting to strict=False, silently skipping normalization of missing keys. Modified compute_norm_stats.py to dynamically detect prevActions in batches and calculate their statistics, solving this issue

Key Insight: Silent failures during data preprocessing are dangerous. The loose behavior of apply_tree(strict=False) allows scale mismatch to propagate during training. Always verify that all expected keys have normalized statistical information, especially for merged features

2. Eager Import in services/init.py Results in Cyclic Dependencies (services ↔ core.scheduler), Making It Impossible to Import Any Service Modules#

Solution: Changed all imports to lazy loading, keeping only the all list and comments, requiring users to import directly (e.g., from srcServices.calendar_service import CalendarService), avoiding cyclic dependencies during package initialization

Key Insight: Python package init.py doesn’t need to provide import statements immediately; for packages with cyclical dependencies, lazy loading combined with documentation comments is a better choice; exposing interfaces at the package level doesn’t equal mandatory eager loading

3. 16 Dangerous Exception Errors: Ignoring All Errors Results in Incomplete States, But System Still Seems Working, With No Ability to Locate Source of Errors During Debugging#

Solution: Replaced code with except Exception as e: logger.error/warning(f"Description: {e}"), keeping error messages for diagnosis, distinguishing between errors (critical functions) and warnings (minor functions)

Key Insight: Never use empty passes to handle exceptions; if errors must be ignored, at least record the error message; silent failures in state tracking layers (situation_monitor/life_context) lead to “Schrödinger states” - unable to determine whether it’s a true null processing or loading failure

4. How To Efficiently Implement a Large-Scale 31-Goal Upgrade Plan Without Creating Sequential Bottlenecks?#

Solution: Used parallel background agent execution mode: Started 4 independent agents simultaneously to execute context_detector, checkin_service, learning_service, and energy_projection enhancements, while directly modifying another 8 files. All agents completed successfully without conflicts

Key Insight: Wave-based parallel execution can significantly accelerate large-scale refactoring when dependencies are correctly isolated. The key is identifying independent work units that can truly run concurrently

5. AI Repeatedly Attempts to Enter Planning Mode, But Users Prefer Direct Execution#

Solution: Users rejected the ExitPlanMode tool call, providing planning content directly to AI execution. AI read results from previous research agents in the planning mode and then wrote directly into the documentation, without needing to re-plan

Key Insight: Users prefer rapid iteration over long-term planning; AI should adaptively reduce the frequency of planning mode triggers or learn this preference after multiple user denials

6. CalendarPro Has 6 Completely Untested Source Files, 12 Modified New Methods, And BackgroundCoordinator Is Implemented But Never Started#

Solution: Created a system’s testing infrastructure: conftest.py for sharing test fixtures, creating 20 new test files (pytest, AsyncMock, patch decorator) based on existing patterns, fixing on_ready() function to actually call coordinator.start_all(), adding 9 missing Discord intent handlers

Key Insight: Large-scale features usually come with incomplete integration. Background services require explicit startup calls, not just class definitions. The system’s testing requires shared test fixtures and consistency in all test files

7. Patch Paths in Test Fixtures Are Incorrect (Patched Call Points Instead of Definition Points), Leading to Mock Failure After Lazy Loading, With Multiple Tests Reporting AttributeError Errors#

Solution: All singleton factory functions for patching (get_settings/get_calendar_service) must be located in the source module (src.config/src.services.calendar_service), not at the call point (e.g., srcservices.gap_analyzer), as lazy loading delays name binding

Key Insight: Gold rule for unittest.mock.patch - patch at the location where it’s being looked up, not at the definition location; Lazy loading means “looking up” locations are always in the source module, not creating new bindings at import locations

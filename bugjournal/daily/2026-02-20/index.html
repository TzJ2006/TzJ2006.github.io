Today’s Overview#

  • What was done: Completed the STAIG fusion special comparison experiment and added support for embedding cache read/write to run_benchmark.py
  • How it was done: Tested staig_fusion through temporary scripts, combined with CacheManager to reuse the existing pipeline cache infrastructure

Conducted special tests on STAIG fusion in the MIHD project. Identified a double normalization bug in eval_scan_fusion.py and introduced a pipeline-level embedding cache mechanism in run_benchmark.py.

Today’s Tasks#

Architecture and Strategy#

  • ✅ STAIG fusion preprocessing difference investigation — Confirmed that staig_fusion has never been officially tested on 151673. Found that eval_scan_fusion.py does not pass staig_alignment_config, causing STAIGTrainer to perform StandardScaler internally again, resulting in a double normalization bug.
  • ✅ Add embedding cache read/write to run_benchmark.py — Introduced CacheManager to check embeddings_cache/ before gene/vision encoding. If the cache is hit, it loads directly (skipping encoder instantiation). If not, it extracts and saves the cache. Supports override for forced re-extraction and allows any vision_encoder name (relaxed argparse choices). Vision caches are available in three variants: standard/freq/staig_strict.
  • ✅ STAIG fusion special comparison experiment (151673) — Interrupted a 2-hour long task to run staig_fusion × {UNI2_raw, SCAN(UNI2) separately. Results showed ARI values of 0.3929 / 0.3880 respectively, almost the same. The reason was that STAIG’s internal StandardScaler+PCA preprocessing counteracted the optimization effects of SCAN.)

Problems and Solutions#

Key Issues#

1. Double normalization bug in staig_fusion in eval_scan_fusion.py (staig_alignment_config not passed)#

Solution: After identifying the root cause, it was decided to directly use run_benchmark.py (which correctly passes staig_alignment_config) instead of fixing the standalone script.

Key insight: Reusing existing implementations is more reliable than fixing bugs. Caching raw embeddings before normalization and performing post-encoder normalization after loading is the correct order.

2. run_benchmark.py re-instantiates encoder to extract embeddings every time, even if the pipeline already has a cache#

Solution: Integrated CacheManager from pipeline/cache_manager.py to load from cache first before encoding and save to cache after extraction.

Key insight: The pipeline’s two-stage cache infrastructure can be reused without reinventing the wheel.

General Issues#

3. Evaluation task ran for 2 hours without completion, unable to obtain results within reasonable time#

Solution: Killed the long-running task and created a temporary script _test_staig_scan.py to test staig_fusion separately. Results were obtained in about 30 seconds.

Key insight: Breaking down the full comparison (18 combinations) into single-method tests allows for quick verification of hypotheses.

Today’s Gains#

Core Gains#

  • STAIG fusion’s internal StandardScaler+PCA preprocessing absorbs external embedding optimization effects (ARI difference between UNI2_raw and SCAN(UNI2) is only 0.005), indicating that STAIG’s robustness to vision embeddings comes from its built-in normalization process.
  • In complex experimental systems, it’s better to reuse existing, verified tool paths (such as run_benchmark.py). Standalone scripts are prone to introducing hidden bugs such as inconsistent preprocessing.
  • Embedding caches should store raw embeddings before normalization. Performing post-encoder normalization after loading is the correct pipeline design pattern to ensure consistent results across different call paths.

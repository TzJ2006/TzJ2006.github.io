Problems & Solutions#

Critical Issues#

1. Co-authorship analysis returned 0 students for all tested professors (Sergey Levine, Pieter Abbeel, Chelsea Finn) despite correct implementation#

Solution: Human recognized faculty homepages explicitly list students; implemented homepage-first discovery with LLM extraction of student names from faculty pages, using co-authorship data only as supplementary enrichment with source tracking for transparency

Key Insight: Faculty homepages are the authoritative source for student lists and more reliable than inferring relationships from co-authorship patterns. When heuristic algorithms fail due to data sparsity, switch to direct information sources (professor webpages) rather than debugging scoring algorithms. Ground truth beats inference.

2. Semantic Scholar matched wrong researchers for 6+ names due to name ambiguity (Ming Yu→explosives chemist, Xiaolong Wang→veterinary scientist/geologist/NLP researcher, 5 out of 6 initial queries failed)#

Solution: Human provided full name clarification (Mingyu Ding); relied on ArXiv paper matching which was more accurate; AI correctly identified data quality issues by recognizing incompatible research domains, returned explicit warnings rather than generating false trajectories; recommended ORCID+institution verification for common Chinese names

Key Insight: Common Chinese names create severe disambiguation problems in academic databases requiring mandatory institution/ORCID matching. Same-name aggregation is a critical data quality issue - need field-based clustering or institution filtering before analysis. Multi-source data integration (homepage + ArXiv + S2) with explicit source tracking provides robustness against individual failures. Better to return ‘data unusable’ than propagate false profiles.

3. Output files scattered across tool-specific subdirectories (summarize/logs/, research/cache/, test/benchmark_results.csv) with hardcoded Path(file).parent patterns created tight coupling and difficult disk space management#

Solution: Designed unified outputs/ structure organized by type (reports/, logs/, cache/, data/) then by tool, introduced canonical path constants in common/paths.py, replaced all hardcoded paths atomically, updated environment variable fallbacks

Key Insight: Type-first organization (reports/, logs/, cache/) is more maintainable than tool-first when multiple tools share similar output patterns. Centralized path management enables atomic refactoring - changing one constant propagates everywhere without breaking code. Phased execution with verification checkpoints prevents cascading failures.

4. Two overlapping research tools in same directory (monolithic research_scout.py + modular profiler package) causing code duplication, unclear single entry point, and undocumented features. Initial /init attempt interrupted by user suggesting AI was heading toward documentation improvements rather than architectural changes.#

Solution: User provided explicit 9-step integration plan with precise function signatures, line counts, and architectural decisions. AI executed plan: kept research_scout.py as stable CLI entry point, refactored modular package as importable library, added profile/citations subcommands, implemented three-stage evaluation with citation analysis, added multi-backend LLM support

Key Insight: For complex architectural changes, detailed written plans prevent scope creep and ensure AI stays on track. Hybrid integration (preserving stable CLI while refactoring internals) reduces migration risk vs full rewrites. Human architectural vision + AI tactical execution works better than letting AI improvise. Gradual migration is safer than big-bang changes.

5. Major new modular tool (researcher profiler with ~15 Python modules) completely undocumented in CLAUDE.md causing confusion about codebase capabilities; AI initially missed ‘New feature’ directory entirely#

Solution: Used Agent exploration to thoroughly map package structure, then rewrote CLAUDE.md to document both tools with separate sections for architecture, usage, and configuration

Key Insight: Documentation rot happens rapidly when new features are added without updating guidance files; requires proactive documentation maintenance or automated detection of undocumented code paths. Update CLAUDE.md immediately after structural changes to prevent drift.

6. JSON parsing failed for Yuke Zhu trajectory analysis after three-tier LLM repair attempts (haiku → sonnet → opus), resulting in incomplete reports#

Solution: Accepted partial results (paper list without trajectory analysis) and documented the failure; analysis completed with degraded output quality

Key Insight: LLM repair escalation has limits; when all three model tiers fail, underlying issue may be prompt/context rather than model capability. Persistent failures indicate need to revisit input structure or question decomposition.

7. arXiv search returned 0 results for robotics leaders despite using correct author names due to sorting/recency issues in search API#

Solution: Switched from arXiv search API to Researcher Profiler which uses both arXiv and Semantic Scholar data sources for more robust multi-source fetching

Key Insight: API search behavior is not always intuitive; multi-source data fetching is more robust than single-API queries. Don’t debug API quirks when better data sources exist.

Learnings#

Key Learnings#

  • For large architectural refactoring, detailed written plans with function signatures, line counts, and architectural decisions dramatically improve AI execution accuracy and prevent scope creep. Human architectural vision + AI tactical execution works better than AI improvisation.
  • Large-scale refactoring should be parallelized by phase/domain: CalendarPro’s 7 phases implemented by 5 concurrent agents each owning a subset of changes, then integrated with test verification. Phased execution with verification checkpoints prevents cascading failures that are hard to diagnose.
  • Academic profiling systems must implement name disambiguation as first-class concern - common Chinese names create severe disambiguation problems requiring mandatory ORCID/institution matching. Same-name aggregation creates entirely false researcher profiles that appear plausible until domain expertise reveals incompatibility. Better to return ‘data unusable’ than propagate false information.
  • Multi-source data integration (homepage scraping + co-authorship + ArXiv + Semantic Scholar) with explicit source tracking provides robustness against individual data source failures or inaccuracies. Faculty homepages are more authoritative than co-authorship analysis for student discovery as they explicitly list current and former students with status.
  • Re-export shims with lazy imports enable large-scale code refactoring while maintaining backward compatibility and avoiding circular dependency issues. Centralized infrastructure modules (paths.py, cache.py, llm.py) enable atomic refactoring - changing one constant propagates everywhere without breaking downstream dependencies.
  • Modular architecture enables clean integration: keeping research_scout.py as monolithic CLI entry point while treating profiler package as library code separates concerns beautifully. Hybrid integration approaches (preserving stable CLI while refactoring internals) reduce migration risk compared to full rewrites.
  • Academic research analysis tools enable rapid domain landscape understanding: researcher profiler quickly reveals field leaders’ trajectories, current directions, and breakthrough works. Systematic framework for tracking cutting-edge research directions.
  • Type-first organization (reports/, logs/, cache/) is more maintainable than tool-first organization when multiple tools share similar output patterns. Centralized path management enables atomic refactoring across entire codebase.
  • Parameter threading through nested call chains requires backward-compatible defaults (backend=‘claude_cli’) and incremental syntax testing (py_compile) to prevent cascading errors. When adding parameters to deeply-nested call chains, default values preserve backward compatibility.
  • Use parallel Agent exploration for thorough codebase understanding before making architectural decisions; sequential exploration misses connections between components. Documentation files like CLAUDE.md become stale rapidly as features are added; need active maintenance or automated detection of undocumented code paths.
  • Documentation-driven development pays off: updating CLAUDE.md and TUTORIAL.md immediately after code changes prevents doc rot and ensures knowledge transfer to future sessions. Documentation synchronization critical for maintainability.
  • Heuristic algorithms depend critically on data quality: student discovery with Semantic Scholar co-authorship failed when data was sparse; need multiple sources or ground truth (homepages). When heuristics fail, switch to direct information sources rather than debugging scoring algorithms.
  • LLM repair escalation (haiku → sonnet → opus) has limits; persistent JSON parse failures may indicate prompt or context issues rather than model capability problems. When all three model tiers fail, revisit input structure or question decomposition.
  • Always ask users about design tradeoffs (scoring weights, automatic vs opt-in features, performance vs convenience) rather than assuming defaults; user context reveals non-obvious priorities like preferring automatic citation analysis over cost optimization.
  • When extracting shared utilities, best-of-breed merging (taking most robust implementation from each source) produces better results than arbitrary selection. API search behavior is not always intuitive; multi-source data fetching is more robust than single-API queries.

Practical Learnings#

  • Hugo content organization benefits from semantic separation: reference material (researcher profiles) and temporal content (daily reports) should go in different sections for better navigation and user experience.
  • Python package entry points matter: ‘python -m package’ is more robust than direct script execution for complex module structures with internal imports. Using -m flag ensures proper module resolution.
  • Polite web crawling (1 request per 2 seconds) with rate limiting is essential when scraping academic homepages at scale to avoid being blocked or overloading servers.

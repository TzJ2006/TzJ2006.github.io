Problems and Solutions#

Key Problems#

1. Co-author analysis showed that all tested professors (Sergey Levine, Pieter Abbeel, Chelsea Finn) had no students, although the implementation was correct.#

Solution: Human-identifiable faculty homepages clearly list student information; implemented a homepage-based discovery mechanism, using LLM to extract student names from faculty pages, using co-author data as a supplement, and increasing transparency through source tracing.

Key Insight: Faculty homepages are authoritative sources of student information, more reliable than inferring from co-author models. When heuristic algorithms fail due to lack of data, one should rely directly on the information source (faculty webpages) rather than debugging the scoring algorithm. Facts speak louder than reasoning.

2. Semantic Scholar matched 6 researchers with ambiguous names (Ming Yu → Explosive Chemist, Xiaolong Wang → Veterinary Scientist/Geologist/Natural Language Processing Researcher). Five of the initial 6 queries failed.#

Solution: Humans provided full names for clarification (Mingyu Ding); relying on ArXiv paper matching was more accurate; AI correctly identified data quality issues, identified incompatible research fields, and returned clear warnings instead of generating incorrect trajectories; suggested verifying common Chinese names with ORCID+institution information.

3. Output files were scattered in tool-specific subdirectories (e.g., summarize/logs/, research/cache/, test/benchmark_results.csv), using a hard-coded Path(file).parent pattern, leading to tight coupling and difficult disk space management.#

Solution: Designed a unified output/structure, organized by type (reports, logs, cache, data) and tool, introduced standardized path constants in common/paths.py, replaced all hard-coded paths, and updated environment variable fallbacks.

Key Insight: When multiple tools have similar output patterns, a type-based structure is easier to maintain than a tool-based one. Centralized path management makes refactoring simpler—changing one constant affects the entire system without breaking the code. Implementing in stages and setting verification checkpoints can prevent cascading failures.

4. Two overlapping research tools existed in the same directory (monolithic research_scout.py and modular profiler package), leading to code duplication, unclear single entry points, and undocumented functionality. Hybrid integration (retaining a stable CLI while restructuring internal logic) reduces migration risk compared to complete rewriting. Human architectural vision combined with AI tactical execution is more effective than letting AI adjust on its own. Gradual migration is safer than one large change.

5. A new modular tool was completely omitted from CLAUDE.md, causing confusion about the capabilities of the codebase; AI initially ignored the “New Features” directory entirely.#

Solution: Used proxy exploration to thoroughly map the package structure, then redrewed CLAUDE.md to document both tools separately, including architecture, usage, and configuration details.

Solution: Accepted partial results (excluding the paper list for trajectory analysis) and documented the reasons for failure; the analysis was completed, but the output quality decreased.

Lessons Learned#

Key Lessons#

  • For large-scale architectural refactoring, a detailed written plan, including function signatures, line counts, and architectural decisions, can significantly improve the accuracy of AI execution and prevent scope creep. Combining human architectural vision with AI tactical execution yields better results than AI improvisation.
  • Large-scale refactoring should be processed in stages or by domain. CalendarPro adopted 7 stages, with 5 agents working on partial changes simultaneously, followed by testing and verification. Phased execution with verification checkpoints prevents difficult-to-diagnose cascading failures.
  • Academic figure analysis systems must prioritize name disambiguation. Common Chinese names cause serious disambiguation issues, requiring mandatory ORCID/institution matching. Aggregating people with the same name results in completely incorrect researcher profiles that appear plausible until domain experts identify the incompatibility. Instead of spreading misinformation, it’s better to return “data unavailable.”
  • Multi-source data integration (web scraping, co-author relationships, ArXiv, Semantic Scholar) combined with clear source tracking can withstand failures or inaccuracies of a single data source. Teacher profiles are more accurate at identifying students than co-author analyses because they explicitly list current and past students along with their status.
  • By delaying imports and re-exports, backward compatibility can be maintained during large-scale code refactoring, avoiding circular dependencies. Centralized infrastructure modules (paths.py, cache.py, llm.py) enable atomic refactoring—changing one constant affects the entire code without breaking downstream dependencies.
  • Modular architecture allows for clear integrations: keeping research_scout.py as a single CLI entry point and the profiler package as an independent library code. This separates different functions effectively. Mixed integration methods (retaining stable CLI while refactoring internal code) are easier to migrate than complete rewriting.
  • Academic research analysis tools can quickly understand the current state of the field: Researchers analyzers can quickly reveal leaders’ trajectories, current directions, and breakthrough achievements. A systematic framework can track the latest research directions.
  • When multiple tools share similar output patterns, organizing by type (reports/, logs/, cache/) is easier to maintain than by tool. Centralized path management makes refactoring the entire code library more atomic.
  • Passing parameters in nested call chains requires backward-compatible default values (backend=‘claude_cli’) and step-by-step syntax testing (py_compile) to prevent cascading errors. Default values can maintain backward compatibility when adding parameters to deeply nested call chains.
  • Before making architectural decisions, use parallel agents for exploration to fully understand the code library. Sequential exploration would miss connections between components. Documentation like CLAUDE.md becomes outdated quickly as features increase, so continuous maintenance or automatic detection of unrecorded code paths is needed.
  • Documentation-driven development is very effective: Updating CLAUDE.md and TUTORIAL.md immediately after code changes prevents documentation obsolescence and ensures knowledge is passed down for future conferences. Document synchronization is crucial for maintainability.
  • Heuristic algorithms rely heavily on data quality: Student discovery using Semantic Scholar fails when data is insufficient. Multiple sources or real data (e.g., teacher profiles) are needed. When heuristic algorithms fail, direct information sources should be used instead of debugging scoring algorithms.
  • The upgrade process of LLM repairs (from haiku to sonnet to masterpiece) has limitations. Continuous JSON parsing failures may indicate a problem with hints or context, rather than model capabilities. When all three levels of models fail, the input structure or problem decomposition needs to be reexamined.
  • Always ask users about design trade-offs (scoring weights, automatic features vs. active selection features, performance vs. convenience). Assume default settings only when necessary. User context reveals some implicit priorities, such as preference for automatic citation analysis over cost optimization.
  • When extracting shared common features, it’s better to use the best implementation merging method (choosing the most robust implementation from each source). This is better than arbitrary selection. API search behavior isn’t always intuitive. Multi-source data acquisition is more reliable than single API queries.

Summary of Practical Experience#

  • Hugo’s content organization benefits from semantic separation: Reference materials and time-based content should be placed in different sections for better navigation and user experience.
  • The input point for Python packages is important: “python -m package” is more reliable than running scripts directly, especially for complex module structures with internal imports. Using the -m flag ensures correct module resolution.
  • Polite web scraping (requesting every 2 seconds) with rate limits is crucial when scraping academic homepages on a large scale, to avoid being blocked or causing server overload.

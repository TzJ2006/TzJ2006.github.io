General Limitations#

  • After large-scale refactoring, it is impossible to automatically test end-to-end functions—import operations were verified, but actual tool commands were not run using the new paths to confirm correctness.
  • It is not possible to reliably verify conference awards (best papers, focus awards, oral awards) solely based on training data—real-time access to official conference websites or paper metadata APIs is required. Instead of guessing, an empty array with a disclaimer is returned.
  • No optimization solutions for citation analysis were proposed (e.g., threshold for referring counts, selective flags); the rule of “always analyzing the top 5” was blindly implemented without considering the impact on API costs.
  • English documentation is used by default, even though the code library belongs to a Chinese user project; clear instructions are needed to switch TUTORIAL.md to Chinese.
  • It is impossible to automatically summarize the final state of multiple parallel agents; manual verification is required to ensure that all 230 CalendarPro tests passed after 5 agents completed their work.
  • When pip installation fails due to Windows file locking, no alternative solutions were proposed, such as stopping the MCP server process or using the –user flag, until action was taken after the first failure.
  • By default, citation analysis is optional (with the –citations flag), rather than being considered as part of the usage pattern, which tends to be automated for ease of use.

Lessons Learned#

Key Lessons#

  • For large-scale architectural refactoring, a detailed written plan, including function signatures, line counts, and architectural decisions, can significantly improve the accuracy of AI execution and prevent scope creep. The combination of human architectural vision and AI tactical execution yields better results than AI’s improvisation.
  • Large-scale refactoring should be processed in stages or by domain: CalendarPro adopted 7 stages, with 5 agents working simultaneously on different parts, followed by testing and verification. Phased execution with defined checkpoints can prevent difficult-to-diagnose cascading failures.
  • Academic figure analysis systems must prioritize name disambiguation—common Chinese names cause serious disambiguation issues, requiring mandatory ORCID/institution matching. Aggregating researchers with the same name results in completely incorrect profiles that appear plausible until domain experts identify the incompatibility. Instead of spreading misinformation, it is better to return “data unavailable”.
  • Multi-source data integration (web scraping, co-author relationships, ArXiv, Semantic Scholar) combined with clear source tracking can withstand failures or inaccuracies of a single data source. Teacher homepages are more accurate at identifying students than co-author analysis, as they clearly list current and past students along with their status.
  • By adjusting import and re-export timing, backward compatibility can be maintained during large-scale code refactoring, avoiding circular dependencies. Centralized infrastructure modules (paths.py, cache.py, llm.py) enable atomic refactoring—changing one constant affects the entire code without breaking downstream dependencies.
  • Modular architecture allows for clear integration: keeping research_scout.py as a single CLI entry point and the profiler package as an independent library code allows for better separation of different functions. Mixed integration methods (retaining a stable CLI while refactoring internal code) are easier to migrate than complete rewriting.
  • Academic research analysis tools can quickly understand the current state of the field: Researchers analyzers can quickly reveal leaders in the field, current directions, and breakthrough achievements. A systematic framework can track the latest research directions.
  • When multiple tools share similar output patterns, organizing by type (reports/, logs/, cache/) is easier to maintain than organizing by tool. Centralized path management makes refactoring the entire code library more atomic.
  • Passing parameters in nested call chains requires backward-compatible default values (backend=‘claude_cli’) and step-by-step syntax testing (py_compile) to prevent cascading errors. Default values can maintain backward compatibility when adding parameters to deeply nested call chains.
  • Before making architectural decisions, use parallel agents for exploration to fully understand the code library; sequential exploration would miss connections between components. Documentation like CLAUDE.md becomes outdated quickly as features increase, requiring continuous maintenance or automatic detection of unrecorded code paths.
  • Documentation-driven development is very effective: updating CLAUDE.md and TUTORIAL.md immediately after code changes prevents documentation obsolescence and ensures that knowledge is passed down for future conferences. Document synchronization is crucial for maintainability.
  • Heuristic algorithms rely heavily on data quality: when data is insufficient, using Semantic Scholar for student discovery fails; multiple sources or real data (e.g., teacher homepages) are needed. When the heuristic algorithm fails, direct information sources should be used instead of debugging the scoring algorithm.
  • The upgrade process of LLM repairs (from haiku to sonnet to masterpiece) has limitations; persistent JSON parsing failures may indicate a problem with prompts or context, rather than model capabilities. When all three levels of models fail, the input structure or problem decomposition needs to be reexamined.
  • Always ask users about design trade-offs (scoring weights, automatic features vs. active selection features, performance vs. convenience), rather than assuming default settings. User context reveals some implicit priorities, such as preferring automatic citation analysis over cost optimization.
  • When extracting shared common features, it is better to adopt the method of merging the best implementations (choosing the most robust implementation from each source), which is better than arbitrary selection. API search behavior is not always intuitive; multi-source data acquisition is more reliable than single API queries.

Summary of Practical Experience#

  • Hugo’s content organization benefits from semantic separation: reference materials and time-sensitive content should be placed in different sections for better navigation and user experience.
  • The input point for Python packages is important: ‘python -m package’ is more reliable than executing scripts directly, suitable for complex module structures with internal imports. Using the -m flag ensures correct module resolution.
  • Polite web crawling (requiring a rate limit every 2 seconds) is crucial when scraping academic homepages on a large scale, to avoid being blocked or causing server overload.

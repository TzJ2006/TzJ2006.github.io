Problems and Solutions#

Key Issues#

1. Collaborator analysis showed zero students for all testing professors (Sergey Levine, Pieter Abbeel, Chelsea Finn), despite the implementation being correct.#

Solution: Professor homepages clearly listed student information; implemented homepage-first discovery functionality using LLMs to extract student names from professor pages. Use collaborator data as supplementary enrichment while retaining source tracking for transparency.

Key Insight: Professor homepages are authoritative sources of student information, making them more reliable than inferring relationships based on collaborators. When heuristic algorithms fail due to sparse data, direct information sources (professor webpages) should be used instead of debugging the scoring algorithm. Practical verification is better than reasoning.

2. Semantic Scholar matched over 6 researchers with ambiguous names (Ming Yu → Explosive Chemist, Xiaolong Wang → Veterinary Scientist/Geologist/NLP Researcher). The first 6 queries failed 5 times.#

Solution: Humans provided full name clarification information (Mingyu Ding); relied on ArXiv paper matching for more accurate results. AI can identify data quality issues by returning clear warnings when incompatible research fields are identified, instead of generating incorrect trajectories. It is recommended to verify common Chinese names with ORCID + institution verification.

Key Insight: Common Chinese names lead to serious ambiguity in academic databases. Therefore, mandatory institution/ORCID matching is required. Aggregating data for the same name is an important data quality issue—field-based clustering or institution filtering is needed before analysis. By clearly tracking sources, integrating multi-source data (homepages, ArXiv, S2) can improve fault tolerance. It’s better to return “data unavailable” than spread incorrect information.

3. Output files were scattered in tool-specific subdirectories (e.g., summarize/logs/, research/cache/, test/benchmark_results.csv). Hardcoding the Path(file).parent pattern led to tight coupling and difficulty managing disk space.#

Solution: Designed a unified output structure, categorized by type (reports, logs, cache, data) and tool. Introduced standardized path constants in common/paths.py, replaced all hard-coded paths, and updated environment variable backends.

Key Insight: When multiple tools have similar output patterns, organizing by type (reports, logs, cache, etc.) is easier to maintain than by tool. Centralized path management allows for atomic refactoring—changing one constant will take effect throughout the system without causing code errors. Phasing out and setting up verification checkpoints can prevent cascading failures.

4. Two overlapping research tools in the same directory (monolithic research_scout.py and modular analysis package) led to code duplication, unclear single entry points, and no documented functionality. The initial/init attempt was interrupted due to users suggesting AI focus on improving documentation rather than architecture.#

Solution: Users provided a detailed 9-step integration plan, including precise function signatures, line counts, and architectural decisions. AI executed the plan: kept research_scout.py as a stable CLI entry point, refactored modular packages into importable libraries, added analysis subcommands, implemented three-stage evaluation, and added support for multiple backend LLMs.

Key Insight: For complex architectural changes, a detailed written plan can prevent scope creep and ensure that AI follows the plan. Hybrid integration (retaining a stable CLI while refactoring internal code) is easier to migrate than complete rewriting. Human architectural vision combined with AI tactical execution is more effective than letting AI adjust on its own. Phasing out is safer than making one-off changes.

5. A new modular tool (researcher analyzer, containing about 15 Python modules) was completely unrecorded in CLAUDE.md, leading to misunderstandings of code capabilities. AI initially ignored the “new feature” directory entirely.#

Solution: Used proxy exploration to thoroughly map package structures and then re-wrote CLAUDE.md to record the architectures, usage, and configuration information of these two tools separately.

Key Insight: When adding new features, if the guide document is not updated, the documentation will quickly become messy. Therefore, active maintenance or automatic detection of unrecorded code paths is required. Update CLAUDE.md immediately after structural changes to prevent information bias.

6. Despite using correct author names, Yuke Zhu’s trajectory analysis still couldn’t parse JSON after three LLM fixes, resulting in incomplete reports.#

Solution: Accepted partial results (excluding the paper list of trajectory analysis) and recorded failures; although the analysis was completed, the output quality declined.

Key Insight: LLM fixes have limitations. When all three model levels fail, the problem may lie in prompts/context rather than model capabilities. Continuous failures indicate that the input structure needs to be re-examined or the problem needs to be decomposed.

7. Despite using correct author names, when searching on arXiv, zero results were found for robotics leaders due to sorting/time issues with the search API.#

Solution: Switched from arXiv search API to Researcher Profiler, which uses arXiv and Semantic Scholar data sources for more powerful multi-source access.

Key Insight: API search behavior is not always intuitive; multi-source data acquisition is more reliable than single API queries. When better data sources exist, don’t debug the quirks of the API.

Limitations of Artificial Intelligence#

Key Limitations#

  • Failed to proactively detect semantic ambiguity issues in Semantic Scholar names during preliminary researcher analysis—acceptingly received incorrect match results (such as explosion chemistry, veterinary science, geology, etc.), and required human-observe symptoms or clear instructions to trigger error detection. Despite domain mismatches, no doubts were raised about the matching results.
  • Required a clear 9-step integration plan to execute complex architectural changes; initial attempts were interrupted as users believed AI was more suitable for document improvement rather than structural restructuring. For cross-cutting changes, architectural guidance was needed.
  • The original design only considered student collaborative research, without considering that teacher homepage is an authoritative and explicit source of student lists; human advice was needed to switch methods.
  • Ignored the entire “new feature” directory during the first analysis of the code base; this issue needed to be pointed out through clear user prompts before discovering an unrecorded analysis tool containing approximately 15 Python modules.
  • Even after fixing the three-layer LLM, it was still unable to parse Yuke Zhu’s trajectory analysis JSON files, resulting in incomplete reports with only a list of papers without analytical insights.
  • Required explicit user corrections to understand that researcher information should be stored separately from daily research reconnaissance reports, lacking semantic understanding of content classification (reference content vs. chronological order).
  • Used citation counts as part of the comprehensive relevance score, but did not realize that popularity and project-specific relevance are independent dimensions; confused “important papers” with “project-related”.

General Limitations#

  • Unable to automatically test end-to-end functionality after large-scale restructuring—verified import operations, but did not run actual tool commands using the new paths to confirm correctness.
  • Unable to reliably verify conference awards (best papers, spotlight, oral reports) solely based on training data—required real-time access to official conference websites or paper metadata APIs. Returned an empty array with a disclaimer instead of guessing results.
  • Did not propose cost optimization solutions for reference analysis (e.g., citation count thresholds, selective flags); blindly implemented the rule of “always analyzing the top 5”, without considering the cost impact of the API.
  • Defaulting to English documentation, even though the code base belongs to Chinese user projects; required explicit instructions to switch tutorial.md to Chinese.
  • Unable to automatically summarize the final state of multiple parallel agents; after five agents completed their work, manual verification was needed to ensure all 230 CalendarPro tests had passed.
  • When pip installation failed due to Windows file locking, no alternative methods were suggested, such as stopping the MCP server process or using the –user flag. These measures were not taken until after the first failure.
  • By default, reference analysis is optional (with the –citations flag), rather than considering usage patterns, allowing for convenient automatic execution.

Lessons Learned#

Key Lessons ### Practical Experience Summary#

  • Hugo content organization benefited from semantic separation: Reference materials (researcher profiles) and dynamic content (daily reports) should be placed in different sections to facilitate navigation and improve user satisfaction.
  • Python package entry points are important: For complex module structures with internal imports, ‘python -m package’ is more reliable than executing scripts directly. Using the –m flag ensures proper module parsing.
  • Polite web crawlers (requesting once every 2 seconds) combined with rate limits are crucial when scraping academic homepages on a large scale to avoid being blocked or causing server overload.

Issues and Solutions#

Key Issues#

1. QCVLA: The second round of Dagger only improved by 1 percentage point (60%→61% SR), indicating it was saturated. The bridge could only see the error distribution it naturally produced, limiting its diversity.#

Solution: Proposed perturbation and bridging data augmentation methods: During Dagger promotion, randomly inject short action sequences, followed by reverse recovery actions. This captures larger deviations that the bridge would never naturally produce, expanding the training distribution. Train the bridge (performing feature prediction in a perturbed state) and action heads (recovering action sequences) simultaneously.

Key Insight: When the model only sees its own error patterns, self-supervised training becomes saturated. Injecting synthetic but real deviations (inspired by FLARE’s perturbation strategy) can expand the distribution. Compared to Gaussian noise per frame, coherent action perturbations are more effective in robot control.

2. VLA Inference: The task was completed, but due to the current_prompt not being reset, it led to an infinite loop, sending completion messages, keeping chunk_count non-zero, and the model continued outputting high scores.#

Solution: Set current_prompt = "" after the task is completed, and call publish_state(“idle”) to prevent the next round from triggering immediately. Add a None check before task_completed.item() to avoid crashes in non-pi05 models.

Key Insight: The state machine needs to explicitly reset all related variables at each termination node; otherwise, unpredictable loops may occur. Multi-device compatibility (pi05 and non-pi05) requires handling boundary conditions in advance.

3. MIHD: The NEEDS_COORDS_FUSIONS tuple is duplicated in pipeline/runner.py and pipeline/evaluation_planner.py, and both lack adaln_attention (potential error), making it difficult to maintain consistency when adding new fusion strategies.#

Solution: Centralize NEEDS_COORDS_FUSIONS as a constant in models/Fusion.py, imported by both files. Add missing adaln_attention to the tuple, fixing the vulnerability where this fusion strategy cannot receive spatial coordinates.

Key Insight: Constant duplication across modules leads to inconsistencies and missed updates. A single source can prevent potential errors and simplify maintenance work.

4. MIHD: refine_labels_spatial_majority in utils/clustering.py uses scipy.spatial.distance.cdist, which has an O(N²) memory complexity, causing OOM on Visium HD data with over 17K points.#

Solution: Replace cdist with scipy.spatial.cKDTree.query, which has an O(N*k) complexity, suitable for k-nearest neighbor searches. Switch from calculating the full distance matrix to querying neighbors based on radius.

Key Insight: For large point clouds, spatial queries need to use tree-based structures (like KDTree, BallTree) to avoid memory explosion. For high-definition spatial data (10K+ points), algorithm selection is more important than implementation details.

5. During training, combining combined_action_loss and task_loss into one value and recording it in wandb prevents separate monitoring. Solution: Modified the compute_loss function to return a tuple (combined_loss, {“action_loss”: …, “task_loss”: …}), allowing the training loop to use nxn.value_and_grad(… with has_aux=True) to parse auxiliary metrics and add them to the info dictionary.#

Key Insight: JAX/Flax’s has_aux=True mechanism allows for elegant extraction of additional monitoring metrics from loss functions without affecting gradient calculation. All compute_loss implementations (pi0 and pi0_fast) need to be synchronized to maintain consistency in the interface.

6. Error benchmark thread tasks can only inject collision_empty errors; other types of errors cannot be generated.#

Solution: Analysis revealed that all thread-based clean trajectory phase_labels were ‘pre_reach’, lacking grasp/lift/transport stages, resulting in errors dependent on these stages (e.g., drop_in_transit) not being injected. The get_task_phase logic or thread task definition needs to be checked.

Key Insight: Error injection depends on the segmentation of task stages; incomplete stage identification can make certain error types detectable. It’s necessary to verify all tasks cover expected stages during opportunity graph generation.

7. gpumon.py counts bash/ffmpeg/claude processes as GPU processes inheriting CUDA_VISIBLE_DEVICES, and displays DDP worker processes separately from the parent process.#

Solution: Modified the get_proc_gpus() function to require both CUDA_VISIBLE_DEVICES environment variable and open/dev/nvidia* file descriptors. Added parent chain traversal in get_gpu_procs(), merging child processes with the same GPU set with their ancestors.

Key Insight: Detecting GPU usage in processes requires checking both environment variables and actual device file descriptors to avoid false positives.

8. All 317 RoboCasa tasks encountered errors downloading MimicGen data, unclear whether it’s a code issue or data availability problem.#

Solution: System analysis showed that 257 tasks did not register mg_path (yellow warning), while 60 tasks had mg_path but box_links_ds.json contained 0 MimicGen download links (red error). Confirmed this was an upstream data publishing issue, not a code defect.

Key Insight: When all downloads fail systemically, it’s necessary to check data availability before attempting code fixes. Using grep to search box_links_ds.json for ‘/mg/’ yielded zero results, confirming the absence of download links.

9. CALVIN RLDS dataset conversion failed, causing KeyError: ‘image’. The actual field names should be ‘rgb_static’ and ‘rgb_gripper’.#

Solution: Read features.json to confirm the actual schema, modify the key names for frame extraction, and remove missing environment_state and relative_action field definitions from the dataset.

Key Insight: Before converting the dataset format, it’s necessary to verify the schema of the source data. RLDS features.json is the authoritative source for determining the actual field names.

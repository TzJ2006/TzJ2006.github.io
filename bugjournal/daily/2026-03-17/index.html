Problems & Solutions#

Critical Issues#

1. QCVLA: DAgger Round 2 only improved by +1pp (60%→61% SR), indicating saturation. Bridge only sees its own naturally-produced error distribution, which is limited in diversity#

Solution: Proposed perturbation & bridging data augmentation: during DAgger rollout, randomly inject short action sequences followed by reverse recovery actions. Captures coherent large-magnitude deviations bridge would never naturally produce, expanding training distribution coverage. Trains both bridge (feature prediction on perturbed states) and action head (recovery action sequences).

Key Insight: Self-supervised training saturates when model only sees its own error modes. Injecting synthetic but realistic deviations (inspired by FLARE’s perturbation strategy) expands the distribution. Coherent action perturbations are more effective than per-frame Gaussian noise for robotic manipulation.

2. VLA inference: task completed but infinite loop sending done messages because current_prompt was not reset, causing chunk_count to remain non-zero and model to continuously output high scores#

Solution: Set current_prompt = "" after task completion and call _publish_state(“idle”) to prevent next round from triggering immediately; moved None check before task_completed.item() to avoid crashes on non-pi05 models

Key Insight: State machines require explicit reset of all related variables at each termination node, otherwise unpredictable loops occur; multi-device compatibility (pi05 vs non-pi05) needs early boundary condition handling

3. MIHD: NEEDS_COORDS_FUSIONS tuple was duplicated in pipeline/runner.py and pipeline/evaluation_planner.py, with adaln_attention missing from both (latent bug), making it hard to maintain consistency when adding new fusion strategies#

Solution: Centralized NEEDS_COORDS_FUSIONS as a constant in models/Fusion.py, imported by both files. Added missing adaln_attention to the tuple, fixing a bug where this fusion strategy would fail to receive spatial coordinates.

Key Insight: Duplicate constants across modules lead to inconsistency and missed updates. Single source of truth prevents latent bugs and simplifies maintenance.

4. MIHD: refine_labels_spatial_majority in utils/clustering.py used scipy.spatial.distance.cdist with O(N²) memory complexity, causing OOM on Visium HD data with 17K+ spots#

Solution: Replaced cdist with scipy.spatial.cKDTree.query which has O(N*k) complexity for k-nearest neighbor search. Changed from computing full distance matrix to on-demand radius neighbor queries.

Key Insight: Spatial queries on large point clouds require tree-based structures (KDTree, BallTree) to avoid quadratic memory explosion. For HD spatial data (10K+ points), algorithmic choice matters more than implementation details.

5. Training combined action_loss and task_loss into single value logged to wandb, unable to monitor each separately#

Solution: Modified compute_loss to return (combined_loss, {“action_loss”: …, “task_loss”: …}) tuple, training loop uses nnx.value_and_grad(…, has_aux=True) to unpack auxiliary metrics and add to info dict

Key Insight: JAX/Flax’s has_aux=True mechanism elegantly returns additional monitoring metrics from loss function without affecting gradient computation; requires synchronizing all compute_loss implementations (pi0 and pi0_fast) to maintain interface consistency

6. Error benchmark threading task can only inject collision_empty, other error types cannot be generated#

Solution: Analysis found all clean trajectory phase_labels for threading are ‘pre_reach’, lacking grasp/lift/transport phases, causing errors dependent on these phases (like drop_in_transit) to have no injection opportunities; need to check get_task_phase logic or threading task definition

Key Insight: Error injection depends on task phase segmentation; incomplete phase recognition causes some error types to be uncoverable; need to verify all tasks cover expected phases during opportunity map generation

7. gpumon.py counted bash/ffmpeg/claude processes with inherited CUDA_VISIBLE_DEVICES as GPU processes, and showed DDP worker processes separately instead of grouping with parent#

Solution: Modified _get_proc_gpus() to require both CUDA_VISIBLE_DEVICES env var AND open /dev/nvidia* file descriptors; added parent chain traversal in get_gpu_procs() to collapse child processes with identical GPU sets to their ancestor

Key Insight: Process GPU usage detection requires both environment variable inheritance check and actual device file descriptor verification to avoid false positives

8. All 317 RoboCasa tasks failed to download MimicGen data with error messages, unclear if code bug or data availability issue#

Solution: Systematic analysis revealed: 257 tasks have no mg_path registration (yellow warnings), 60 tasks have mg_path but box_links_ds.json contains 0 MimicGen download URLs (red errors). Confirmed this is upstream data publishing issue, not code defect.

Key Insight: When download failures are systematic across all items, investigate data availability before attempting code fixes. box_links_ds.json grep for ‘/mg/’ returned zero matches, conclusively proving missing download links

9. CALVIN RLDS dataset conversion failed with KeyError: ‘image’, actual field names are ‘rgb_static’ and ‘rgb_gripper’#

Solution: Read features.json to confirm actual schema, modified frame extraction key names, removed non-existent environment_state and relative_action field definitions from dataset

Key Insight: Dataset format conversion must verify source data schema before assuming field names; RLDS features.json is authoritative source for determining actual field names

10. How to implement interactive keyboard navigation in a monitoring tool that needs to refresh periodically without blocking on input#

Solution: Used select.select() with 0 timeout for non-blocking keyboard input detection, termios/tty for raw mode terminal control, stored selected_idx and scroll_offset state across renders, applied reverse video (\033[7m) for highlighting

Key Insight: Non-blocking I/O with select() enables real-time interaction in monitoring tools while maintaining refresh intervals. Terminal raw mode requires careful state management to capture arrow keys without echo

Learnings#

Key Learnings#

  • Large-scale code refactoring should use parallel Agent-based multi-dimensional review (reuse/quality/efficiency) to systematically discover issues. This uncovered 9 problems including latent bugs, performance bottlenecks, and maintainability issues that single-pass review would miss
  • When DAgger-style training saturates, expand data distribution via structured perturbations (coherent action sequences + recovery) rather than per-frame noise. For robotic manipulation, temporal coherence matters—physics and inertia make sequential perturbations more realistic than i.i.d. noise
  • State machine implementations must explicitly reset all related variables at each termination node; implicit assumptions that “next loop will naturally change” easily lead to unpredictable behaviors like infinite loops
  • Code review should be proactive rather than reactive; even when user only asks “how it’s implemented”, expand to “are there problems”. Proactively discovering potential bugs is more valuable than fixing them after users encounter issues
  • Cross-project technical transfer requires distinguishing ‘directly applicable’ from ‘conceptually adaptable’. FLARE→QCVLA transfer was mostly conceptual (perturbation thinking, expert mixing) rather than code-level, because FLARE addresses policy robustness while QCVLA addresses feature prediction efficiency
  • Spatial queries on large point clouds (10K+ spots) must use tree-based structures. O(N²) distance matrices cause OOM. cKDTree with O(N*k) neighbor queries scales to HD spatial omics data
  • JAX/Flax’s value_and_grad(…, has_aux=True) elegantly returns additional monitoring metrics from loss function without affecting gradient computation, the standard pattern for multi-metric monitoring
  • Error injection benchmark quality depends on task phase segmentation completeness; incomplete phase recognition causes some error types to be uncoverable, need to verify all tasks cover expected phases during opportunity map stage
  • Process GPU usage detection requires both environment variable inheritance check and actual device file descriptor verification to avoid false positives from shell processes
  • Download failure diagnosis should follow data-first approach: when systematic failures occur, verify data availability (grep for expected patterns in config files) before deep-diving into code logic
  • Dataset format conversion must read schema files (like RLDS features.json) to confirm actual field names before assuming common conventions or guessing
  • Non-blocking terminal I/O pattern: select.select() with timeout=0 + termios raw mode + state persistence across renders enables interactive features in real-time monitoring tools without sacrificing refresh rate
  • Parent process chain traversal is effective for deduplicating hierarchical process structures (DDP workers, PyTorch subprocesses) in monitoring tools by collapsing children to ancestors when GPU sets match

Practical Learnings#

  • Git local repository config > global config > environment variables priority can override unavailable global proxy on shared servers; git config –local is effective for resolving shared server configuration conflicts
  • Server environment automation workflow: SSH key→package manager (fnm/nvm)→runtime (Node.js)→tools (Claude/Codex)→project transfer. User handles interactive auth, AI handles repeatable installation steps

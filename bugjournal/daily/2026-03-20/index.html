Problems and Solutions#

Key Problems#

1. Personal data files (dialogue logs, AI summaries, blog content, research cache, benchmark test results) are tracked and pushed to GitHub through git, making the boundary between code repository and data storage unclear and posing a privacy risk#

Solution: Designed a three-layer cleanup mechanism: (1) Current working directory: Used git rm –cached to remove tracking records; (2) Rewrote all 24 commit history records using git filter-repo; (3) After forced push, used rclone to uniformly manage personal files (remote path gdrive:gadget). Reduced repository size from 22MB to 516KB

Key Insight: The responsibility boundaries of Git and rclone must be clearly defined: Git manages public code and copyable configuration settings, while rclone is responsible for personal data and runtime output. Mixing the two can lead to repository expansion (94MB of personal data), privacy leakage, and collaboration difficulties. A clear separation can establish data privacy boundaries

2. When users perform merge operations on other devices to generate final reports, local merge–sync-all still considers all dates as non-final states, leading to repeated calls of LLM#

Solution: AI discovered that CMD_merge_sync_all only called_rclone_download_logs() and did not call reports, resulting in local final check failures. Implemented the_rclone_download_reports() function to pull existing reports from remote before synchronizing all final state checks

Key Insight: In a distributed collaboration scenario, state synchronization must synchronize inputs (logs) and outputs (reports) simultaneously; otherwise, repeated processing logic will lead to resource waste (repeated calls of LLM)

3. The CPU BLAS matrix_size recorded in test/CLAUDE.md was incorrectly marked as 8192 (single-core) and 4096 (all cores), while GPU was set to 2048#

Solution: Confirmed the actual code: CpuSingleCoreBLASBenchmark.MATRIX_SIZE=2048, CpuAllCoresBenchmark.MATRIX_SIZE=4096, GpuBenchmark.MATRIX_SIZE=8192. Updated documentation to match the actual situation

Key Insight: Differences between documentation and code will accumulate over time; systematic line-by-line verification using parallel agents can detect numerical errors that may be overlooked when reading casually. Systematic and comprehensive verification is necessary to ensure correctness

General Problems#

4. git filter-repo removed.gitkeep files and the directory after filtering, destroying the preservation of the directory structure#

Solution: Recreated.gitkeep files (research/cache, research/project, research/reports), used touch and git add -f operations, and submitted them as a separate “directory structure restoration” submission after filter-repo

Key Insight: The –invert-paths option of git filter-repo will delete the entire directory tree including structural files; restoration of.gitkeep must be submitted after filter-repo, not before filtering

5. In research/CLAUDE.md, line 74 mentions a “two-stage evaluation process,” but the title in line 79 is “three-stage evaluation process” — there is an internal contradiction#

Solution: Confirmed that the code implemented 3 stages (filtering → in-depth analysis → citing impact); updated line 74 to “three-stage,” and batch replaced all 7 occurrences of “two-stage” in research_scout.py with “three-stage”

Key Insight: Contradictions in documentation indicate that historical evolution has not been fully coordinated; batch corrections using grep can ensure no residual inconsistencies

AI Limitations#

Key Limitations#

  • In designing rclone synchronization logic, AI failed to anticipate the dependence of final checks on reports in sync-all scenarios, resulting in users needing to explicitly state “Reports also need to be downloaded.” Lack of foresight in maintaining integrity of distributed collaboration state synchronization.
  • AI initially did not suggest line-by-line document verification until users explicitly requested it; default mode was targeted/sample-based verification, which might overlook numerical errors (such as matrix_size, doctoral degree duration).

General Limitations#

  • AI initiated 3 parallel exploration agents to scan the code repository (list of personal files, rclone logic, git history), although this approach is comprehensive, efficiency may not be ideal—this information can be quickly obtained through several grep/ls/git commands, and the overhead of agents might be excessive.
  • AI failed to predict that.gitkeep files would be removed by git filter-repo –invert-paths (directory structure preservation issues); users needed to manually restore them after post-filter-repo.

Key Takeaways### Practice Learning Outcomes#

When modifying remote paths, it’s necessary to check the integrity of configuration migrations: updating (1) summarize/daily_summary.py config.json reading, (2) research/research_scout.py configuration, (3) test/sync logic, (4) website/Hugo integration, (5) sync.py global configuration, and (6) existing remote data migration scripts.

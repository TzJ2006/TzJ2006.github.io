Problems and Solutions#

Key Issues#

1. drop_in_transit_D0 generated very few valid scenes. It was diagnosed that a min_hold_height of 0.85 was too low, causing objects on the desktop (z≈0.88) to be misjudged as airborne. An 80% chance of false positives occurred.#

Solution: Increase min_hold_height from 0.85 to 0.93 (desktop height 0.80 + object height 0.08 + margin 0.05). Simultaneously modify the five error skill files and configs. Eliminate false positives, increasing D0 success rates from 10% to 50%.

Key Insight: In the pick_place task, the z-height of objects on the desktop can exceed the set minimum height. The design threshold for grasping detection must consider the combination of “desktop height + object height” with enough margin.

2. The training scene generation script was completely serial. The success rate for collision_eef_object subtype was extremely low (only 5 scenes in 10 minutes). It would take over 6 hours to complete 6 tasks.#

Solution: Add –subtypes filtering parameters to the script and create a ThreadPoolExecutor parallel launcher (32 workers, 4 GPUs in round-robin allocation). All 973 scenes were completed in 41 minutes.

Key Insight: MuJoCo EGL rendering is a CPU-intensive task. Up to 32 independent processes can run on the same node without conflict. Subprocess-based parallelism is more stable than multi-threading and can fully utilize 48-core CPUs.

3. After the Slurm interactive job expired, all training processes and data generation processes were killed via SSH. This wasted time during training.#

Solution: Formal jobs must be submitted via sbatch, or processes should be guaranteed to persist using tmux within an allocated job. The pam_slurm_adopt policy will kill all related processes when the job ends. SSH nohup cannot bypass this.

Key Insight: HPC cluster Slurm resource isolation mechanisms will forcibly clean up all processes within a job, including subprocesses connected via SSH.

4. After 128/130 workers were completed, there were still 29 task×subtype combinations that were completely missing. Threading only generated 3/29 subtypes.#

Solution: Cross-analyze opportunity maps with generation logs to distinguish two types of failures: no opportunity points found during the opportunity scan phase versus validation failures during generation. Prioritize fixing gripper_closed_norm=0.00 anomalies (P0 level, which can unlock 8 combinations).

Key Insight: Scene generation failures are divided into two stages: opportunity scan (can_inject is always False) and validator rejection. The former requires modifying the can_inject condition, while the latter requires adjusting thresholds or injection parameters. These must be processed separately.

5. The ARI difference between STAIG hyperparameter sweep baseline (0.23) and clustering ablation ARI (0.58) was unusually large. The experiment name ‘pca_uni2_staig_fusion’ was misleading: raw HVG was actually used instead of UNI2, and the ablation script used raw HVG instead of PCA input.#

Solution: Modify the ablation script to add –gene_encoder and –vision_variant parameters. Run sweeps with PCA+UNI/UNI2 again, resulting in an ARI average of about 0.47 (within a reasonable range), establishing fair benchmarks.

Key Insight: Experiment naming conventions must strictly correspond to implementation. Misleading names can lead to long-term misunderstandings. It is necessary to verify the actual implementation by reading logs rather than just looking at the experiment name.

6. The PickPlace task did not have MimicGen D1 variants, so it was unable to generate a D0+D1 merged dataset as planned.#

Solution: Directly generate 2000 D0 items to replace the D0+D1 merged scheme.

Key Insight: The number of variants in MimicGen varies for different robosuite tasks. The data strategy must confirm available variants for each task individually. Total equivalence is a feasible engineering decision.

General Issues#

7. Images generated by Visium sc.pl.spatial only show tissue H&E background, with no gene expression points superimposed.#

Solution: Change the way sc.pl.* is saved from plt.savefig to native scanpy save parameters (sc.settings.figdir + save parameters), allowing the internal pipeline of scanpy to handle spot rendering.

Key Insight: Scanpy’s spatial plot must use its internal saving mechanism. Bypassing matplotlib directly will skip the spot rendering step.

8. MUJOCO_EGL_DEVICE_ID=0 did not match CUDA_VISIBLE_DEVICES=1, causing import failures.#

Solution: Change MUJOCO_EGL_DEVICE_ID to match CUDA_VISIBLE_DEVICES (both set to 1).

Key Insight: MUJOCO_EGL_DEVICE_ID must be the global physical GPU ID, not the number mapped from CUDA_VISIBLE_DEVICES.

Today’s Achievements#

Core Achievements- The pam_slurm_adopt policy in the Slurm HPC cluster will强制终止所有相关进程 when the job ends; SSH nohup cannot bypass this limitation. Long training tasks must be submitted as formal jobs using sbatch, and should not rely on interactive jobs.#

  • The design of the height threshold for robot operation scenarios must consider the combined range of “desktop height + object’s own height”. Simply setting a fixed min_height without considering the actual stacking height of objects will result in many false positives, leading to very few valid scenarios.
  • MuJoCo EGL simulation is a CPU-intensive task. Up to 32 independent processes can run on the same node without conflict. Failure in training scenario generation occurs in two stages: can_inject is always False during the opportunity scan stage, and validation rejects the process. It’s necessary to distinguish between these issues: modify the can_inject condition for the former, and adjust the threshold or injection parameters for the latter.
  • The naming convention for experiments must strictly correspond to the actual implementation. Inconsistent naming (e.g., ‘pca_uni2’ actually uses UNI) can lead to misunderstandings in long-term experiments. It’s important to confirm the actual configuration by reading logs rather than just looking at the experiment name.
  • The number of D0/D1 variants for MimicGen varies depending on the task (e.g., PickPlace has only D0). When designing a multi-task data strategy, it’s necessary to confirm the available variants for each task. An equivalent total amount (e.g., 2000 D0 instead of 1000 D0 and D1) is a feasible engineering decision.
  • When diagnosing problems with generated data, it’s important to distinguish between “current running meta.json statistics” and “disk history accumulation file count”. If there is a significant difference between the two, it indicates that the problem lies in the current running logic, not historical issues. For parallel task logs, it’s recommended to write separate files for each worker so that failure can be quickly identified using grep.

Practical Achievements#

  • Spatial transcriptomics visualization functions like sc.pl.spatial in Scanpy must be saved using the sc.settings.figdir + save parameters. Using plt.savefig directly will cause incomplete images due to internal layer rendering issues.

Learnings#

Key Learnings#

  • Git vs rclone responsibility boundary must be clearly defined: git manages reproducible public code and configuration (requirements.txt, scripts, CLAUDE.md), rclone manages personal data and runtime outputs (logs, reports, cache, test results, website content). Mixing causes repo bloat (94MB personal data), privacy risks, and collaboration obstacles. Clear separation establishes proper data privacy boundary
  • Documentation-code drift is inevitable and accumulates silently over time; periodic exhaustive line-by-line verification (using parallel Explore agents for speed) is necessary to catch numeric values, path references, and internal contradictions that casual reading misses. Systematic verification prevents accumulation of small errors
  • Distributed collaboration state synchronization design principle: must sync not only input data (logs), but also intermediate outputs (reports), otherwise logic depending on output state (like finalized checks) will fail across different devices. Complete state sync prevents resource waste (repeated LLM calls)
  • git filter-repo is the correct tool for purging large files/sensitive data from git history (22MB→516KB repo size reduction), but requires full workflow: backup .git, run filter-repo, restore structural files (.gitkeep), force-push, update all clones. Risk management checklist: (1) must backup on all devices, (2) requires force-push, (3) all other clones will break and need re-clone. Should use explicit user confirmation process (AskUserQuestion) to ensure understanding of destructive consequences

Practical Learnings#

  • When building centralized sync infrastructure (sync.py), supporting –dry-run and –category flags from day one enables safe incremental testing and selective sync without full data upload; rclone remote path migration (gdrive:summarize/ → gdrive:gadget/) is low-risk with atomic move operations. Configuration migration completeness checklist when modifying remote paths: must simultaneously update (1) summarize/daily_summary.py config.json reading, (2) research/research_scout.py configuration, (3) test/ sync logic, (4) website/ Hugo integration, (5) sync.py global config, (6) existing remote data migration script

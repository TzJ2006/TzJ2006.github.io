Issues and Solutions#

Key Issues#

1. npx ccusage and all Node.js CLI tools cannot access external data even when httpproxy/https proxy environment variables work properly and the local proxy runs at 127.0.0.1:10087.#

Solution: It was found that Node 24’s –use-env-proxy function only reads uppercase HTTP_PROXY/HTTPS_PROXY/NO_PROXY, regardless of lowercase forms. The.bashrc and.bashrcy files were modified to output both uppercase and lowercase variables in the proxy_on/net_on functions, and NODE USE_ENV_PROXY=1 was set to enable Node’s native fetch() proxy support.

Key Insight: Node.js 24 and later versions provide built-in environment variable proxy support through the –use-env-proxy flag/NODEUSEENV_PROXY environment variable, but strictly require uppercase variable names. This is a simpler solution than injecting an undici ProxyAgent or packaging individual CLI tools. It’s a universal solution for all Node CLI tools, not a workaround for specific tools.

2. After converting LLM prompts and Markdown rendering to English, verification identified 6 categories of residual Chinese text that would disrupt the English-dominated process: upper and lower text strings in LLMs, matplotlib chart labels, DEEP_EVAL comments, project templates, JSON fix hints, configuration default values.#

Solution: Used 3 parallel verification proxies (one for each module) to systematically search for Chinese patterns and verify the consistency of the entire process. All 6 types of issues were fixed in the summarize, research, and website modules.

Key Insight: LLM prompt translation not only requires checking main PROMPT constants but also: (1) upper and lower text strings injected during execution, (2) non-text output (charts), (3) comments appearing in prompts, (4) generated templates, (5) error recovery hints. Parallel proxy verification (divided by subsystem) is more thorough than single-pass grep and can detect strings injected at runtime.

3. Incorrect date filtering on remote SSH servers in different time zones - Timestamps marked as 2024-01-01T23:00:00+00:00 in the records were mistakenly considered to be January 2nd due to local timezone UTC+8.#

Solution: Added a parse_remote_ts_to_local_date() helper function to convert FixedOffset timestamps to local time before extracting dates. Modified 4 calling points to use conversions considering local time. Fixed iso_filter in Codex to include the UTC offset for correct cross-timezone string comparisons.

Key Insight: DateTime::date_naive() extracts dates from server timezone, not local time. It must be converted to local time using .with_timezone(&chrono::Local) before calling date_naive(). This is important for any distributed system with cross-timezone data.

4. The meeting-helper requires Chinese-English mixed transcription, but Whisper’s language detection occurs for each piece of text (about 30 seconds), resulting in poor quality for code switching within sentences like “We need to implement this feature”.#

Solution: Multi-layer mitigation measures: (1) Added a custom vocabulary to screenpipe-settings.json to include common bilingual terms, (2) Added a retranscription workflow in CLAUDE.md to guide users to use bilingual hints POST /audio/retranscribe API when transcription quality is poor, (3) Added a step 4 to the pipeline to check transcription quality and trigger retransmission of mixed fragments, (4) Clearly documented this limitation to manage user expectations.

Key Insight: Whisper’s architecture limitations (language detection for each piece of text) cannot be fully solved with model parameters. A more effective strategy is to use a custom vocabulary + secondary transcription + user feedback loop to gradually improve quality while honestly disclosing the limitations in the documentation, rather than overpromising.

5. LifeCopilot’s background coordinator manages more than 18 background services, but there is no health check mechanism. When services crash, only errors are recorded without automatic restarting, resulting in system partial failures that users may not notice.#

Solution: Migrated the health monitoring mode from OpenClaw: created health_manager.py file and implemented ServiceHealthManager (4-state FSM: HEALTHY→DEGRADED→FAILED→STOPPED, with a 10-time per hour restart budget and exponential backoff times of 1→2→4→8→60 seconds). Modified BackgroundCoordinator to integrate health management: 5-minute health monitoring, automatic restart when failed, marked as degraded when the budget ran out, notified via EventBus. Added GET /api/health endpoint to expose all service statuses.

Key Insight: Migrating mature models from production-verified open-source projects like OpenClaw’s health monitoring is more reliable than designing from scratch. The key is understanding the principles of the original design (why use a 4-state FSM? Why limit the number of restarts? Why adopt exponential backoff times?) and adapting them to the context of the target project.

6. The floating ball folding animation on Windows has visual flickering when the direction is ’left’, but no such flickering occurs when the direction is ‘right’.#

Solution: The root cause is that SetWindowPos moves and adjusts the window size simultaneously, but WebView2 rendering is asynchronous. When the sphere position jumps during size adjustment (x=96→x=0), a 1-2 frame interval will show old content incorrectly cropped. The fix is to use ShowWindow(SW_HIDE) before SetWindowPos and ShowWindow(SW_SHOWNOACTIVATE) afterward, so that WebView2 won’t render intermediate frames. This only applies to the collapsed state (!expanded) to minimize side effects.Key insight: Native window geometry changes (SetWindowPos) are synchronous, but WebView2 content rendering is asynchronous. When the left edge of the window moves (collapsed in the ’left’ direction), the content coordinate system of WebView changes, causing visual flicker during the 1-2 frame rendering delay. There is no flicker when moving to the ‘right’ direction, because the left edge remains at x=0—only the right edge shrinks.

7. Reporting script errors caused pickplace and coffee to read the outdated full_run_200attempts.log file, which showed error counts of 398/260 instead of the actual 168/85#

Solution: Bypass log parsing and directly use h5py to check HDF5 files to get the actual demonstration count.

Key insight: Reports based on logs may deviate from actual data; HDF5 files are an authoritative source for data verification.

8. The floating ball collapse animation causes the sphere to move left then right instead of staying still#

Solution: Root cause: Immediately setting expanded=false removes the .shell.expanded class, causing CSS justify-content to shift from flex-end/flex-start to center. During the 250 milliseconds width transition, the sphere moves towards the center. Fix: Remove the expanded condition from the direction-based justify-content rules so it applies throughout the transition.

Key insight: CSS transitions interact with native window size adjustments. Front-end CSS transitions (250 milliseconds) complete before backend window size adjustments (t=260 milliseconds). If CSS rules change immediately based on the expanded state, the sphere will float during the transition. Direction-based positioning must remain constant throughout the transition to fix the sphere’s position.

Issues and Solutions#

Key Issues#

1. Risk of broken import when mcp_server.py directly imports 15 functions after research_scout.py refactoring (CRITICAL level)#

Solution: Reduced research_scout.py to about 80 lines of thin shim, ensuring zero changes to mcp_server.py through re-export. No modifications were needed in the calling party.

Key Insight: The risk of MCP import breakage discovered by Critic agent is the highest-risk point in modular refactoring; shim re-export is the only non-invasive migration path.

2. Code review found ~1300 lines of dead MCP modules declared in lib.rs but never called externally, wasting compilation time and causing confusion#

Solution: Removed 4 mod declarations and related Phase A comments from lib.rs. The files were retained for future integration.

Key Insight: Three parallel agents’ code reviews detected “self-consistent but isolated” module groups—internal dependencies without being referenced by any external modules. Single-linear reviews could not detect this.

3. macOS-specific APIs (Keychain, /bin/zsh path, set_dock_icon_visible) cause compilation issues on Windows/Linux during cross-platform migration#

Solution: Used #[cfg(target_os)] conditional compilation to separate platform code; macOS Keychain returned Err instead of panic on other platforms for graceful degradation; set_dock_icon_visible was restored using cfg on macOS, kept noop for non-macOS.

Key Insight: “Deleting macOS code” and “changing macOS code to noop” are two different operations that must be decided function by function. macOS-only APIs can return Err gracefully on other platforms without completely removing features.

4. Three runtime issues with Windows tray applications: windows not aligned with taskbar, CLI pop-up window, progress bar too short#

Solution: Used TrayBottomCenter instead of TrayCenter; added .creation_flags(0x0800_0000) to all Command calls; defined 1.5x progress bar width for Windows/Linux using #[cfg].

Key Insight: tauri_plugin_positioner already has TrayBottomCenter built-in; tokio::process::Command exposes creation_flags() natively in Windows, no additional trait import needed (different from std::process::Command).

5. In lerobot2rlds.py, ‘observation.state’ missed joint_state fields, causing RLDS data to lack joint states#

Solution: Changed it to startswith(‘observation.’)+exclusion of image/dones exact combinations, while handling v[’names’] KeyError.

Key Insight: String containment matches can lead to unexpected results (‘observation.state’ is a substring of ‘observation.joint_state’); using startswith+exclusion combination is more precise.

6. Optimize skill single-file solution estimated 500-700 lines. Critic found that multi-language prompts dilute Claude’s attention when handling single language#

Solution: Adopted hub+spoke architecture: placed common frameworks (133 lines) in the hub, with each of the 4 languages having ≤80 lines in the spoke, combined with single-language focused instructions.

Key Insight: In multi-language prompt design, physical file isolation (speech files) is more effective than section separation to prevent prompt dilution.

7. resizeDebug tool was considered simple to archive, but actually contained 8 files called 100+ times in depth#

Solution: Adjusted the plan after a full grep scan: moved to lib/debug/ directory instead of archiving completely, avoiding large-scale call cleanup.

Key Insight: Grep must be used to evaluate actual call depth before archiving. Fewer files does not mean less impact; ccplan Review Gate requires this evaluation step.

8. Global skill library has over 50 irrelevant skills that load about 15000-20000 tokens per session#

Solution: Divided by project granularity: moved Python/research/coding-related skills to gadget project levels, removed 36 irrelevant skills.

Key Insight: Claude Code loads all skill indexes per session, should be managed according to proximity principles. Only truly general skills should remain in the global library.

General Issues#

9. Loading LeRobot Parquet files with datasets 3.6.0 reported ValueError: Feature type ‘List’ not found#

Solution: The new version of datasets removed List type and replaced it with Sequence; guided use of fix_parquet_metadata.py to batch replace metadata using positional parameters.

Key Insight: LeRobot dataset is tightly coupled with the datasets library version. Cross-version upgrades require checking Parquet metadata feature type definitions.

10. In Git Bash environment, /usr/bin/link overwrites MSVC link.exe, causing Rust MSVC target linking failures#

Solution: Install Visual Studio 2022 Build Tools, activate MSVC environment via cmd.exe before running cargo build.

Key Insight: Git Bash path priority can override Windows native linker. Rust MSVC targets must bypass Git Bash environment using cmd.exe.

Human Thinking vs AI Thinking#

Strategic Level#### Complementarity of Runtime Bug Awareness (Users) vs Static Code Analysis (AI)#

Role Approach
Humans Users perceive 3 specific bugs during real Windows testing: incorrect window position, terminal pop-up, short progress bars. The descriptions are precise at the phenomenon level.
AI AI identifies issues such as dead modules, feature regressions, and redundant IPC calls during code review on /simplify. This is a static analysis rather than runtime awareness.

Difference Analysis: The two perspectives complement each other. Users provide runtime bug reports, while AI helps identify code structure quality issues. In cross-platform scenarios, the user’s real-world experience cannot be covered by AI’s static analysis.

Productization Thinking: Users’ Perspective Beyond AI’s “Function Completion”#

Difference Analysis: Users consider productization experience for packaging and distribution earlier than AI. AI only focuses on functional implementation while neglecting distribution convenience.

  • Cross-platform code changes cannot be locally run on the target platform (Windows) using cargo build/check for real-world compilation verification. It relies on user confirmation offline, increasing the friction of cross-platform migration verification.

  • Inaccurate understanding of API differences between tokio::process::Command and std::process::Command: First, unnecessary CommandExt trait imports are added. Only after the compiler reports an unused import is the issue corrected.
  • The default commit behavior does not align with user preferences (automatically adding Co-Authored-By lines; incorrect reference to old SSH keys). This needs to be corrected after users notice it.

Today’s Takeaways#

Key Takeaways#

  • ECL (Evolving Constraint Language) documentation can preserve the complete architecture decision context across sessions (assumption verification history, reasons for scheme selection, Critic feedback, dependency DAG), allowing subsequent agents to continue without re-analysis. This is a key persistence mechanism for large multi-session refactoring.
  • The Critic agent and parallel multi-angle Agent review (reuse/quality/efficiency) are effective mechanisms to prevent destructive changes. They can systematically identify key dependencies missed by humans (MCP import breakage, prompt dilution, 100+ calls to resizeDebug), and adversarial reviews should be enforced at every architecture decision.
  • The difference between OpenVLA-OFT and OpenVLA: OFT uses independent Action Head (L1/Diffusion) to output continuous actions rather than discrete tokens. It also supports FiLM language conditions, ontology awareness, Action Chunking, and multi-image inputs. Both have RLDS data formats, but OFT’s RLDSBatchTransform additionally processes wrist camera and ontology fields.
  • The Hub+spoke architecture is suitable for designing multi-language prompt skills: The hub maintains a common framework and philosophy, while the spoke focuses on language-specific checks (≤80 lines). This helps prevent attention dilution when Claude handles single languages. Physical file isolation is more effective than section separation.
  • When migrating across platforms, “deleting macOS code” and “changing macOS code to noop” are two different operations that must be decided function by function. The macOS-only API can gracefully downgrade in other platforms without completely removing features.
  • The core ideas of CrossFormer (time segmentation, time×dimension dual attention, Router routing, multi-scale prediction) can be mapped to battery foundation models: Charge-discharge cycles = segmented units, multiple sensors = dimension attention, large-scale battery packs = routing requirements, fine-to-medium-thick three-gradation = multi-scale prediction.
  • Rust’s cross-platform conditional compilation #[cfg(target_os] is a clean solution for handling platform-specific APIs. tokio::process::Command has native support for creation_flags() in Windows, eliminating the need for additional trait imports (different from std::process::Command).
  • The hierarchical management rules of Claude Code skills: ~/.claude/skills/ is loaded globally, and project-level .claude/skills/ is only loaded for that project. The more skill index entries there are, the more context tokens are consumed. They should be managed according to proximity principles. Unrelated skills should be moved to project-level or removed.
  • The ccplan Review Gate is valuable in tasks like file reorganization: Before making archiving/deletion decisions, one must use grep to evaluate the actual call depth. “Small number of files” does not equal “small impact range”. This avoids incorrectly archiving frequently called tools.

Practical Takeaways#

  • RLDS data format uses is_first/is_last/is_terminal to represent episode boundaries. It should not contain explicit dones fields. When converting from LeRobot to RLDS, the dones fields need to be skipped, and the episode boundaries should be used to derive them automatically.

Problems & Solutions#

Critical Issues#

1. After research_scout.py refactor, mcp_server.py directly imports 15 functions — risk of breakage (CRITICAL)#

Solution: Reduced research_scout.py to a ~80-line thin shim with re-exports, guaranteeing zero changes to mcp_server.py and no modification required to any callers.

Key insight: The MCP import breakage identified by the Critic agent is the highest-risk point in modular refactoring; shim re-export is the only non-invasive migration path.

2. Code review found ~1,300 lines of dead MCP modules declared in lib.rs but never called externally, wasting compile time and creating confusion#

Solution: Removed the 4 mod declarations and related Phase A comments from lib.rs; the files themselves were kept for future integration.

Key insight: Three parallel Agent code reviews can detect “self-consistent but isolated” module clusters — groups with internal dependency chains but no external callers — which are difficult to spot in a linear review.

3. During cross-platform migration, macOS-specific APIs (Keychain, /bin/zsh path, set_dock_icon_visible) blocked compilation on Windows/Linux#

Solution: Used #[cfg(target_os)] conditional compilation to isolate platform-specific code; macOS Keychain returns Err instead of panic on other platforms for graceful degradation; set_dock_icon_visible restored via cfg for macOS, non-macOS retains a noop.

Key insight: In cross-platform migration, “delete macOS code” and “convert macOS code to noop” are two distinct operations that must be decided function by function. macOS-only APIs can return Err on other platforms for graceful degradation without fully removing the functionality.

4. Three Windows tray application runtime issues: window not aligning to the bottom of the taskbar, CLI probing popping up a terminal window, progress bar displaying too short#

Solution: Switched to TrayBottomCenter instead of TrayCenter; added .creation_flags(0x0800_0000) to all Command calls; used #[cfg] to define 1.5x progress bar width for Windows/Linux.

Key insight: tauri_plugin_positioner already has TrayBottomCenter built in. tokio::process::Command natively exposes creation_flags() on Windows without requiring extra trait imports (unlike std::process::Command).

5. Substring match for ‘observation.state’ in lerobot2rlds.py missed the joint_state field, causing RLDS data to lack joint state#

Solution: Changed to startswith(‘observation.’) + exclusion of image/dones, a precise combination, while also handling the v[’names’] KeyError.

Key insight: Substring matching is error-prone (‘observation.state’ is a substring of ‘observation.joint_state’); a startswith+exclusion combination is more precise.

6. Single-file approach for optimize skill was estimated at 500–700 lines; Critic found that multi-language content would dilute Claude’s attention when processing a single language#

Solution: Changed to hub+spoke architecture: hub holds the general framework (133 lines), 4 language spokes ≤80 lines each, paired with single-language focus instructions.

Key insight: In multi-language prompt design, physical file separation (spoke files) prevents prompt dilution more effectively than section dividers.

7. The resizeDebug utility was assumed to be easily archivable, but was actually deeply embedded across 8 files with 100+ call sites#

Solution: After a full grep scan, adjusted the plan: moved to a lib/debug/ subdirectory rather than full archiving, avoiding a large-scale call-site cleanup.

Key insight: Before archiving, a grep scan must assess actual call depth — a small file count does not imply a small impact radius. The ccplan Review Gate mandates this assessment step.

8. A global skill library with 50+ unrelated skills loads approximately 15,000–20,000 tokens of index per session#

Solution: Split by project granularity: Python/research/coding-related skills moved to the gadget project level; 36 unrelated skills deleted.

Key insight: Claude Code loads all skill indexes at each session start; skills should be managed by proximity — only truly universal skills belong in the global scope.

General Issues#

9. datasets 3.6.0 raises ValueError: Feature type ‘List’ not found when loading LeRobot Parquet files#

Solution: The new version of datasets has replaced the List type with Sequence. Guided the use of the existing fix_parquet_metadata.py to batch-replace metadata using positional arguments.

Key insight: LeRobot datasets are tightly coupled to the datasets library version; cross-version upgrades require checking Parquet metadata feature type definitions.

10. /usr/bin/link in Git Bash overrides MSVC link.exe, causing Rust MSVC target linking to fail#

Solution: Installed Visual Studio 2022 Build Tools and ran cargo build via cmd.exe after activating the MSVC environment.

Key insight: Git Bash’s PATH priority can shadow the Windows native linker; Rust MSVC targets must use cmd.exe to bypass the Git Bash environment.

Today’s Takeaways#

Core Takeaways#

  • ECL (Evolving Constraint Language) documentation preserves the complete architectural decision context across sessions (hypothesis validation history, rationale for solution choices, Critic feedback, dependency DAG), allowing subsequent agents to continue execution without re-analysis. It is the key persistence mechanism for large multi-session refactors.
  • The Critic agent and parallel multi-perspective Agent reviews (reuse/quality/efficiency) are effective mechanisms against destructive changes: they systematically discover critical dependency chains that humans overlook (MCP import breakage, prompt dilution, resizeDebug 100+ call depth). Adversarial review should be mandated at every architectural decision.
  • Core difference between OpenVLA-OFT and OpenVLA: OFT uses an independent Action Head (L1/Diffusion) to output continuous actions instead of discrete tokens, and additionally supports FiLM language conditioning, proprioception, Action Chunking, and multi-image input. Both use RLDS as the data format, but OFT’s RLDSBatchTransform additionally processes wrist camera and proprioception fields.
  • Hub+spoke architecture is suitable for multi-language prompt skill design: the hub maintains the general framework and philosophy; spokes focus on language-specific checks (≤80 lines), paired with “single-language focus instructions” to prevent Claude’s attention from diluting when processing a single language. Physical file separation is more effective than section dividers.
  • In cross-platform migration, “delete macOS code” and “convert macOS code to noop” are two distinct operations that must be decided function by function. macOS-only APIs can return Err on other platforms for graceful degradation without fully removing the functionality.
  • CrossFormer’s core ideas (time segmentation / time×dimension dual-attention / Router routing / multi-scale prediction) can be mapped to a battery foundation model: charge/discharge cycles = segmentation units, multi-sensor = dimension attention, large-scale battery packs = routing requirement, fine/medium/coarse granularity = multi-scale prediction.
  • Rust cross-platform conditional compilation #[cfg(target_os)] is a clean solution for handling platform-specific APIs. tokio::process::Command natively supports creation_flags() on Windows without extra trait imports (unlike std::process::Command).
  • Claude Code skill layering rules: ~/.claude/skills/ loads globally; project-level .claude/skills/ loads only for that project. The more skill index entries there are, the more context tokens are consumed — skills should be managed by proximity; unrelated skills should be moved to the project level or deleted.
  • ccplan Review Gate is especially valuable for file reorganization tasks: archiving/deletion decisions must be preceded by a grep scan to assess actual call depth. “Few files” does not mean “small impact radius.” Avoids incorrectly archiving heavily-used utilities.

Practical Takeaways#

  • RLDS data format uses is_first/is_last/is_terminal to express episode boundaries and should not include an explicit dones field. When converting LeRobot→RLDS, the dones field must be skipped; episode boundaries are derived automatically.

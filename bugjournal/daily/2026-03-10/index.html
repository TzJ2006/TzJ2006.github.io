Problems & Solutions#

Critical Issues#

1. Error Recovery Benchmark v4’s physics-force injection cannot generate VLA-compatible temporal context - errors injected as initial states lack the behavioral history VLA models require#

Solution: Designed v5.0 architecture with offline two-stage pipeline: Stage 0 collect clean trajectories (human + 10×VLA policy success runs) → Stage 1 offline scan opportunities per frame → Stage 2 quota scheduler (100 per task×category) → Stage 3 context replay (replay first N-1 frames to build VLA context window, inject error at frame N, let VLA freely infer from N onward). Shifted from force-based injection to action-based Error Skill modules (EEF movement/gripper control), each independently handling detect/inject/validate for one error type

Key Insight: VLA models have temporal context dependencies - cannot simply initialize with error states. Errors must ‘grow naturally’ from correct trajectories. Offline scan+quota approach efficiently achieves dataset balance while preserving temporal realism

2. MIHD spatial transcriptomics patch query visualization showing misaligned coordinates - query patches don’t match expected spatial locations#

Solution: Identified 3 root causes: (1) Dataset/DLPFC/ path vs top-level path mismatch in adata loading, (2) position-based assignment without barcode mapping (cache order ≠ adata order after scanpy operations), (3) marker overlay using cache coords instead of adata.obsm[‘spatial’]. Fixed by using consistent paths, bc_to_cache_idx dictionary for explicit barcode alignment, and barcode-based coordinate lookup

Key Insight: Spatial transcriptomics workflows require explicit barcode-based alignment at every junction (cache→load→render) - implicit position matching fails when adata gets reloaded from different sources or reordered by scanpy. Never assume index preservation across file I/O boundaries

3. Robobrain Pi training loss oscillates significantly - suspected dimension processing issue in action prediction#

Solution: Deep analysis revealed 14-dim real actions (right arm 6DOF+gripper + left arm 6DOF+gripper) padded to 32-dim for model architectural compatibility, but MSE loss computed over all 32 dimensions. This means 56% of loss (18/32) comes from zero-padded dimensions where model learns trivial noise→zero mapping, severely diluting gradient signals. Recommended: action_loss = mean(square(v_t[…, :14] - u_t[…, :14]))

Key Insight: Zero-padding for architectural compatibility severely dilutes gradient signals when included in loss computation - always mask loss to real dimensions only. In this case, 56% of gradient signal was wasted on learning a trivial mapping. This is a subtle but critical bug affecting training efficiency

4. STAIGTrainerE2E.py file deleted but still referenced in Fusion.py and pipeline_config.yaml, causing ImportError when staig_fusion_e2e strategy is invoked in MIHD pipeline#

Solution: User chose ‘cleanup all references’ strategy. Removed staig_fusion_e2e from Fusion.py, pipeline_config.yaml, data_preparer.py, pipeline runner, STAIG utils, MEMORY.md, and 8 other files across codebase. Also fixed 6 README dimension/description errors exposed during audit

Key Insight: Orphaned references are critical production bugs. Systematic grep-based audit is essential before file deletion. Automated import tests would catch these earlier. The code-documentation consistency audit (via parallel exploration agents) successfully identified 1 critical + 6 medium discrepancies

5. RoboTwin dataset conversion always uses first instruction from JSON, reducing training diversity for language-conditioned policies#

Solution: Replaced instruction_data.get(‘seen’, [’’])[0] with np.random.choice(seen_instructions) to randomly sample one instruction per episode, matching pattern in reference implementation convert_aloha_data_to_lerobot_robotwin.py

Key Insight: Dataset conversion should incorporate augmentation at source level - random instruction sampling per episode increases training diversity without requiring multiple dataset versions, directly impacting model generalization to instruction paraphrasing. This is a one-line change with major training impact

6. Robobrain Pi task completion binary classifier may learn trivial ‘always predict 0’ solution despite task_loss_weight=0.1#

Solution: Root cause: severe class imbalance (only last 20 frames of ~200-frame episodes are positive = 9:1 negative:positive ratio) combined with task_pos_weight=1.0 (no balancing). Solution requires independent tuning of two hyperparameters: keep task_loss_weight=0.1-0.15 (prevent auxiliary task from overpowering 50-step action prediction), but increase task_pos_weight to 5.0 (give positive class sufficient gradient signal via per-class weighting)

Key Insight: For imbalanced auxiliary classification, per-class weighting (pos_weight) is more effective than overall loss scaling when the task shouldn’t dominate the main objective. loss_weight and pos_weight serve different purposes: loss_weight balances task importance, pos_weight balances class contributions within that task. Must tune independently

7. Node.js ccusage tool cannot fetch latest model pricing from LiteLLM (1.4MB JSON), falls back to cached 46 models instead of 2696#

Solution: Diagnosed via layered testing: (1) curl through campus proxy takes 28 seconds for 1.4MB download, (2) Node.js fetch shows ‘getaddrinfo EBUSY’ hostname resolution failure, (3) discovered Node.js v24’s built-in fetch (undici) doesn’t respect HTTPS_PROXY environment variable by default unlike curl/axios. Solution: NODE_OPTIONS="–use-env-proxy" npx ccusage@latest

Key Insight: Node.js v24’s fetch differs fundamentally from traditional HTTP clients (curl/axios/requests) - it ignores environment proxy variables by default. This is a compatibility trap requiring explicit –use-env-proxy flag or global-agent library. Critical for restricted network environments

9. Kubernetes container nvidia-smi shows [Not Found] for all GPU processes despite GPU being actively used#

Solution: Root cause: PID namespace isolation. nvidia-smi queries kernel driver returning host PID (e.g., 3076643), but container PID namespace has different numbering (e.g., 990465). Solution: developed gpumon.py tool combining psutil (reads container-internal /proc for process info: CPU/memory/cmdline/runtime) with nvidia-smi (GPU hardware state: VRAM/utilization/temperature), visualized via rich TUI

Key Insight: Container namespace isolation breaks cross-namespace process ID mapping. Tools must collect information within their respective namespaces then aggregate. This is why standard monitoring tools fail in containerized environments - they assume unified PID space

10. Openpi-moe serve_policy.py crashes with ‘MUJOCO_EGL_DEVICE_ID must be between 0 and 0, got 6’#

Solution: CUDA_VISIBLE_DEVICES=6 restricts process to single GPU remapped as device 0 in CUDA context, but EGL renderer attempted to use original hardware index 6. Added os.environ.setdefault(‘MUJOCO_EGL_DEVICE_ID’, ‘0’) at script startup to align with CUDA remapping

Key Insight: CUDA_VISIBLE_DEVICES remaps GPU indices to 0-N range for isolated process; dependent libraries (MuJoCo EGL, rendering contexts) must use remapped indices not original hardware IDs. This is subtle but critical interaction between CUDA device isolation and rendering frameworks. Always set rendering device IDs to 0 when using CUDA_VISIBLE_DEVICES with single GPU

11. CALVIN calvin_agent module not found despite install.sh completing successfully#

Solution: Discovered pip install executed in different conda environment than active univla environment. Editable installs create .egg-info directories but package availability is scoped to Python environment executing the install. Reinstalled with pip install -e . –no-deps in correct environment to avoid overwriting PyTorch

Key Insight: Editable pip installs are environment-scoped - artifacts may exist but imports fail if installation ran elsewhere. Always verify ‘which python’ matches intended environment before troubleshooting missing modules. –no-deps flag allows installing package structure without touching existing dependencies

Learnings#

Key Learnings#

  • VLA model temporal context dependency requires fundamental benchmark redesign: Cannot simply initialize episodes with error states - VLA needs behavioral history to make contextually appropriate recovery decisions. Solution: replay first N-1 frames to build context window, inject error at frame N, let VLA infer freely from N onward. This shifts error injection from ‘initial state perturbation’ to ’trajectory interruption’, matching real-world failure modes where errors emerge during execution
  • Offline scan + quota scheduler architecture for balanced error datasets: Stage 0 collect clean trajectories (human + 10×VLA policy success runs) → Stage 1 offline scan opportunities per frame → Stage 2 quota allocation (100 per task×error category) → Stage 3 context replay. More efficient than online generation, guarantees dataset balance, enables reuse of expensive clean trajectory collection across multiple error injection experiments
  • Error taxonomy deformation dimension classification: D0 (position only), D1 (rotation only), D2 (position+rotation) systematically describes spatial deformations across all error types. More principled than phase-based classification (reach/grasp/lift/transport/place), enables clearer validator logic by decomposing 6DOF changes into orthogonal components
  • Zero-padding architectural compatibility trap: Padding for model compatibility (14→32 dims) can severely dilute gradient signals (56% waste) if included in loss computation. Always mask loss to cover only meaningful dimensions. In multi-dimensional outputs, this is subtle but critical - the model converges but learns inefficiently because majority of gradient signal optimizes trivial zero→zero mappings
  • Independent hyperparameter tuning for imbalanced auxiliary tasks: loss_weight (balances task importance) and pos_weight (balances class contributions within task) serve different purposes and must be tuned independently. For auxiliary classification under severe imbalance: keep loss_weight low (0.1-0.15) to avoid overpowering main objective, increase pos_weight (e.g., 5.0) to address class imbalance. Overall scaling vs per-class weighting are orthogonal controls
  • Spatial transcriptomics barcode-based alignment requirement: Explicit barcode mapping required at every pipeline junction (cache→load→render). Position-based matching breaks when adata reloaded from different sources or reordered by scanpy operations. Never assume index preservation across file I/O boundaries. Solution: maintain bc_to_cache_idx dictionaries throughout pipeline, use barcode lookup for all coordinate assignments
  • MIHD architecture deep understanding: 3-stage pipeline (Encoders → Fusion → Clustering/Evaluation), 5 gene encoders (PCA 50d, MLP 128d random init, scGPT 512d, GCN 50d, none), 4 vision encoders (ResNet50, HIPT, UNI, UNI2), 10 fusion strategies (concat/mean/element_wise_sum + attention variants + STAIG/QFormer). Two-phase: Phase 1 extraction (cache to embeddings_cache/), Phase 2 evaluation (load + fuse + cluster + metrics)
  • Dataset conversion augmentation at source level: Random instruction sampling per episode (np.random.choice from available paraphrases) increases training diversity without requiring multiple dataset versions. One-line change with major impact on language-conditioned policy generalization. Shows importance of pattern matching against reference implementations in same codebase
  • CUDA_VISIBLE_DEVICES GPU index remapping: Restricting to subset remaps indices to 0-N range within process CUDA context. Dependent libraries (MuJoCo EGL, rendering frameworks) must use remapped indices not original hardware IDs. Always set rendering device IDs to 0 when using single-GPU isolation. Subtle but critical for multi-GPU setups where different jobs run on isolated device subsets
  • Injection vs Natural Capture applicability: Physics-based injection (impulse/friction/pose_perturb) reliably generates 7 error types (DROP, COLLISION, POSITIONING, STALL partially).

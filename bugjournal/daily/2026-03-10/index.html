问题与解决方案#

关键问题#

1. 修复夹爪归一化后机会扫描仍显示 drop/collision_holding 机会为 0。根因是三层嵌套 bug 链断裂：get_task_completion_stages() 中 grasp_closed 阈值为 0.7，而 Sawyer 实际持握时归一化值仅约 0.08，导致所有 lift/transport 帧均被 can_inject 拒绝#

解决方案: grasp_closed 阈值降至 0.05，增加 EEF-to-object 距离（<8cm）的 lift 阶段 fallback；更新 6 个任务 YAML；修复 scanner 输出路径（加 /{task} 子目录）和 max_per_trajectory（10→50）

关键洞察: bug 是三层嵌套的：夹爪归一化 → grasp 阈值 → 相位检测 → can_inject 全拒绝。每层修复必须带动对下游依赖的系统性检查

2. 训练 action loss 被 padding 维度稀释：模型 action_dim=32 但实际动作仅 14 维，56% MSE loss 来自零填充维度，真实动作梯度严重不足；同时 task_pos_weight=1.0 在类别不平衡（末 20 帧正类，约 1:9）下分类头可能学 trivial 的全负类预测#

解决方案: loss 计算只对前 14 维求均值（action_loss = jnp.mean(jnp.square(v_t[…, :14] - u_t[…, :14]))）；task_pos_weight 提升至 5.0（约等于 (L-20)/20），task_loss_weight 保持 0.1

关键洞察: padding 维度归一化后梯度信号接近零，系统性稀释真实动作维度的学习；类别不平衡不加权时负类 loss 占优势，正类梯度近乎为零

3. Sawyer 夹爪 get_gripper_closed_norm() 恒返回 0，导致所有基于夹爪状态的 can_inject 判断全部失效。根因：两指对称夹爪 qpos=[0.04,-0.04]，mean 恒为 0，除零 bug#

解决方案: 改用第一根手指绝对位置归一化：norm = abs(qpos[0]) / abs(close_qpos[0])；同步更新 7 个 error skill 文件的夹爪阈值（0.5/0.7 → 0.05/0.15/0.20）

关键洞察: 两指对称夹爪的 mean 归一化天然无效；实际持握时 norm≈0.08，全开时≈0.025，与直觉相反

4. Drop 技能注入后物体不掉落（delta_z≈0），validate 永远失败。根因：robosuite OSC 控制器在零动作下主动维持 EEF 位置，打开夹爪后物体仍停在手指上#

解决方案: 直接操作 MuJoCo sim.data.qpos（物体位置偏移 2cm）和 sim.data.qvel（设置 -0.5m/s 下落速度），绕过 OSC 控制器；验证 DROP D0/D2 和 COLLISION_HOLDING delta_z > 2.5cm 通过

关键洞察: robosuite OSC 是位置控制器，零动作≠无力；需直接物理状态操纵实现自由落体，绕过控制器补偿

5. CUDA_VISIBLE_DEVICES=6 运行时 EGL 报错：MUJOCO_EGL_DEVICE_ID must be between 0 and 0, got 6#

解决方案: 在 main.py 入口添加 os.environ.setdefault(‘MUJOCO_EGL_DEVICE_ID’, ‘0’)，将 EGL 设备 ID 与 CUDA 重映射后的逻辑编号对齐

关键洞察: CUDA_VISIBLE_DEVICES 重映射 CUDA 设备编号但不影响 EGL 物理设备枚举，两套编号体系独立，必须手动同步

6. K8s Pod 内 nvidia-smi 显示 GPU 进程 [Not Found] 无法 kill；serve_policy.py 同时出现 traj_proj PyTree 不匹配警告和 socket DNS crash，难以区分严重性#

解决方案: K8s：nvidia-smi 报告宿主机 PID，容器 PID 命名空间不同；用 ps aux | grep python 查容器内进程并 kill。serve_policy.py：traj_proj 不匹配为无害警告（allow_partial=True 自动合并），真正 crash 是 DNS 解析失败，需检查网络配置或改用 IP

关键洞察: containerd K8s Pod 有独立 PID 命名空间，NVIDIA 驱动暴露宿主机 PID；多错误并发时需通过读实现代码（allow_partial 参数）区分严重性，避免被无害警告误导

一般问题#

7. QuotaScheduler 只调度 5/17 子类型（collision_empty 等早期高频 subtype 把 trajectory quota 吃光）；ccusage 无法从 LiteLLM 获取最新定价（仅 46 个模型）#

解决方案: max_per_trajectory 10→50，使 total_slots 与 trajectory 数量解耦。ccusage：NODE_OPTIONS=’–use-env-proxy’ npx ccusage@latest 让 undici 读取 HTTPS_PROXY

关键洞察: 配额分配中 per-trajectory 上限是子类型覆盖的硬瓶颈，需满足 n_traj × max_per_traj ≥ n_subtypes × target；Node.js v24 undici 不自动读取代理环境变量

8. Windows SSH/tmux 内中文字符显示为下划线；conda 环境中 egg-info 存在但 import 失败；共享服务器 wandb login 影响其他用户#

解决方案: 中文渲染：chcp 65001 或改用 Windows Terminal（根因是本地 code page 非 UTF-8）。calvin_agent：在目标 conda 环境中 pip install -e . –no-deps 重装验证。wandb：不修改 ~/.netrc，改用 WANDB_API_KEY 环境变量传递凭证，仅对当前进程有效

关键洞察: 三个问题均属「看似服务器端但实为客户端/环境隔离」的典型；「同环境他人是否正常」是快速定位根因侧的关键对照实验

AI 局限性#

重要局限#

  • 未预判 bug 的传播链：修复夹爪归一化后没有系统检查所有依赖它的下游逻辑（get_task_completion_stages、grasp 相位、can_inject），需要多次运行失败才逐层发现；参数推荐（task_loss_weight）初期也未充分追踪维度链路，需用户纠正后重做
  • 对 robosuite OSC 控制器行为认知不足：假设「打开夹爪就会掉落」，未预判 OSC 的位置维持特性，导致多次无效的阈值调整才最终发现需要绕过控制器直接操作物理状态

一般局限#

  • 在规划类会话中过度依赖交互工具（多次尝试 ExitPlanMode/AskUserQuestion 被用户拒绝），未根据上下文做出合理推断后继续推进；临时调试脚本（test_drop_fix.py 等）未整合进正式单元测试，清理后诊断知识丢失

今日收获#

核心收获#

  • VLA 训练必须 mask padding 维度：flow matching 模型若 action 存在 zero-padding，只对真实维度计算 loss，否则 padding 维度系统性稀释梯度信号（本例 14/32 维只贡献 43.75% loss）。机器人数据分类标签末尾 N 帧为正类会导致严重不平衡，pos_weight 必须根据实际数据分布设置，而非默认值 1.0
  • Sawyer 两指夹爪 qpos=[0.04,-0.04]，mean 恒为 0，必须用单指绝对值归一化（持握 norm≈0.08，空手≈0.025，与直觉相反）。robosuite OSC 控制器 zero-action 维持 EEF 位置，打开夹爪≠物体下落；需直接操作 MuJoCo sim.data.qpos/qvel 绕过控制器实现自由落体。MuJoCo 接触力是瞬态的，不适合 post-settle 验证，应改用位移/速度等持续性指标
  • CUDA_VISIBLE_DEVICES 和 MUJOCO_EGL_DEVICE_ID 是两套独立的 GPU 编号体系，前者对 CUDA 运行时重映射物理 GPU，后者枚举 EGL 物理设备；在指定非零 GPU 时必须手动同步，显式设置 MUJOCO_EGL_DEVICE_ID=0。JAX/Flax allow_partial=True 机制允许在已有 checkpoint 上扩展模型架构，缺失键从初始化填充，多余键丢弃
  • 相位检测是多个 error skill 正常工作的隐含前提；底层传感器修复后必须系统检查所有依赖它的上游判断逻辑。QuotaScheduler 的 max_per_trajectory 需满足 total_slots = n_traj × max_per_traj ≥ n_subtypes × target，否则早期高频 subtype 会吃光所有 quota
  • K8s Pod 有独立 PID 命名空间，NVIDIA 驱动暴露宿主机 PID 给所有容器，导致 nvidia-smi 进程名无法解析；需用 psutil+pynvml 在容器内自行实现进程监控。排查渲染/显示类问题时，「同环境其他用户是否正常」是最有效的早期过滤问题，能快速区分客户端与服务器端根因

实践收获#

  • conda egg-info 目录存在不代表当前 Python 环境已安装该包，多环境共享文件系统时必须在目标环境中通过 pip show 和 import 验证。Node.js v24 内置 fetch（undici）默认不读取 HTTPS_PROXY，需 –use-env-proxy 显式开启。wandb 凭证存储在用户级 ~/.netrc，应通过 WANDB_API_KEY 环境变量传递凭证，仅对当前进程有效

Problems and Solutions#

Key Problems#

1. Error Recovery Benchmark v4’s physical force injection couldn’t generate a time context compatible with VLA – because initial state errors lacked the behavior history required by VLA modelsSolution: Designed v5.0 architecture with an offline two-stage process: Stage 0 collects clean trajectories (human data + 10×VLA strategy successful runs) → Stage 1离线 scans each frame opportunity → Stage 2 quota scheduler (100 opportunities per task × error type) → Stage 3 context replay (replay N-1 frames to build VLA context window, inject error at frame N, allowing VLA to freely infer from N frames). Shifted from force-based injection to behavior-based Error Skills module (EEF motion/claw control), handling detection/injection/verification independently for each error type.#

Key Insight: VLA models depend on time context – errors must “naturally occur” from correct trajectories. Offline scanning + quota method efficiently balances datasets while maintaining temporal realism.

2. MIHD spatial transcriptomics patch query visualization showed incorrect coordinate alignment – query patches didn’t match expected spatial positions.#

Solution: Identified 3 root causes: (1) Data loading inconsistencies between dataset/DLPFC/path and top-level paths; (2) Location-based allocation without barcode mapping (cache order after scanpy operation didn’t match data loading order); (3) Using cache coordinates instead of data.obsm[‘spatial’] for marking. Resolved these issues with consistent pathing, bc_to_cache_idx dictionary for precise barcode alignment, and barcode-based coordinate searching.

Key Insight: Spatial transcriptomics workflow requires explicit barcode alignment at each node (cache→load→render). Implicit location matching fails when data is reloaded from different sources or re-sorted by scanpy. Never assume index preservation between I/O boundaries.

3. Robobrain Pi training loss fluctuated significantly – suspected issue with dimension processing in action prediction.#

Solution: In-depth analysis revealed that 14-dimensional real actions (right arm 6DOF+claw+left arm 6DOF+claw) were filled to 32 dimensions for model compatibility, but MSE loss was calculated across all 32 dimensions. This meant 56% of losses (18/32) came from zero-filled dimensions, where the model learned trivial noise-to-zero mappings, severely diluting gradient signals. Suggestion: action_loss = mean(square(v_t[…, :14] - u_t[…, :14]))

Key Insight: Zero filling for compatibility often dilutes gradient signals in loss calculations. 56% of gradients were wasted learning trivial mappings—this was a subtle but critical error affecting training efficiency.

4. STAIGTrainerE2E.py file was deleted, but still referenced in Fusion.py and pipeline_config.yaml, causing ImportError when calling staig_fusion_e2e strategy in MIHD pipeline.#

Solution: Users chose “remove all references” strategy. Removed staig_fusion_e2e from Fusion.py, pipeline_config.yaml, data_preparer.py, pipeline runner, STAIGutils, MEMORY.md, and 8 other files. Also fixed 6 dimensional/description errors found during audit.

Key Insight: Isolated references are serious production-level errors. Systematic grep audits before deleting files are necessary. Automated import testing can detect these issues earlier. Code-document consistency audits using parallel exploration agents successfully identified 1 serious and 6 moderate issues.

5. RoboTwin dataset conversion always used first instruction in JSON, reducing training diversity for language conditional policies.#

Solution: Replaced instruction_data.get(‘seen’, [’’]) with np.random.choice(seen_instructions) to randomly select instructions per episode, consistent with reference implementation convert_aloha_data_to_lerobot_robotwin.py.

Key Insight: Dataset conversion should be enhanced at the source level. Randomizing instructions per episode increases training diversity without requiring multiple dataset versions, directly affecting model generalization to instruction interpretation. This is a simple change but has significant impact on training.

6. Robobrain Pi task completion binary classifier might learn a trivial “always predict 0” solution, despite task_loss_weight=0.1Solutions: The root cause is a severe category imbalance (only the last 20 frames out of approximately 200 are positive, with a ratio of 9:1 for positives vs negatives). Additionally, task_pos_weight=1.0 means there is no balance between the two categories. The solution requires adjusting two hyperparameters separately: keep task_loss_weight=0.1-0.15 (to prevent auxiliary tasks from overriding the 50-step action prediction), but increase task_pos_weight to 5.0 (to give enough gradient signal to the positive category by weighting each category).#

Key Insight: For unbalanced auxiliary classification tasks, when the task should not dominate the main goal, weighted scaling by category is more effective than overall loss scaling. Loss_weight and pos_weight have different purposes: loss_weight balances the importance of the task, while pos_weight balances the contribution of each category within the task. Both need to be adjusted independently.

7. The Node.js ccusage tool cannot obtain the latest model price information from LiteLLM (1.4MB JSON), instead it reverts to using 46 cached models, rather than 2696 models#

Solution: The problem was diagnosed through hierarchical testing: (1) downloading the 1.4MB file via the campus proxy took 28 seconds, (2) Node.js encountered “getaddrinfo EBUSY” error due to hostname resolution failure, (3) It was found that Node.js v24’s built-in retrieval method (undici) does not follow the HTTPS_PROXY environment variable by default, unlike curl/axios. Solution: NODE_OPTIONS="–use-env-proxy" npx ccusage@latest

Key Insight: Node.js v24’s retrieval method is completely different from traditional HTTP clients like curl/axios/requests—it ignores the environment proxy variables by default. This is a compatibility issue that requires explicitly using the –use-env-proxy flag or using a global proxy library. This is particularly important in restricted network environments.

8. The MIHD documentation is overly detailed: 6 detailed reference guides were created (48.9KB), while users prefer a simpler document structure#

Solution: Users requested a simplification of the document structure after completing the 6 guides. A new plan was developed to re-integrate the guides into the README (including all functions), tutorials (how to use), plans (past/present/future), and results (all experiments). This has not yet been implemented.

Key Insight: AI tends to adopt a “comprehensive classification” approach rather than a “minimal use-oriented” document structure. Research code changes quickly—detailed guides will soon become outdated. A better approach is to have concise integrated documents and well-documented code. The user’s preferred granularity should be confirmed before creating a detailed structure. The user’s first indication (reducing from 25+ to 4) should always be a constraint.

9. The Kubernetes container nvidia-smi shows that all GPU processes are [not found], even though GPUs are being actively used**#

Solution: The root cause is PID namespace isolation. nvidia-smi returns the host PID (e.g., 3076643) when querying the kernel driver, but the container’s PID namespace has a different numerical ID (e.g., 990465). Solution: The gpumon.py tool was developed, combining psutil (to read the /proc files inside the container for process information: CPU/memory/command line/duration) and nvidia-smi (for GPU hardware status: memory usage/temperature), and visualizing the results with a rich user interface.

Key Insight: Container namespace isolation disrupts the mapping of process IDs across namespaces. The tool must collect information from each namespace and then aggregate it. This is why standard monitoring tools fail in containerized environments—they assume the PID spaces are unified.

10. The Openpi-moe serve_policy.py crashes because MUJOCO_EGL_DEVICE_ID must be between 0 and 0, but actually gets 6#

Solution: CUDA_VISIBLE_DEVICES=6 limits the process to use only one GPU as device 0, but the EGL renderer tries to use the original hardware index 6. Add os.environ.setdefault(‘MUJOCO_EGL_DEVICE_ID’, ‘0’) at the start of the script to align with CUDA’s remapping.

11. Despite the successful completion of install.sh, the CALVIN’s calvin_agent module still cannot be found#

Solution: It was found that the pip installation was performed in a conda environment different from the activated univla environment. An editable installation creates an.egg-info directory, but the availability of packages is limited to the Python environment where the installation was performed. Reinstall in the correct environment using pip install -e . –no-deps to avoid overwriting PyTorch.

General Issues#

12. In the tmux session, Chinese characters appear as underscores, but other users on the same server can see the correct characters#

Solution: The issue was traced throughout the chain: the server’s locale was correct (C.UTF-8), tmux was configured for UTF-8, but the Windows client terminal did not pass UTF-8 encoding via SSH. The root cause was on the client side: Windows PowerShell SSH did not correctly negotiate UTF-8 encoding. Solution: Enable “Beta: Use Unicode UTF-8” in the Windows locale settings, or run chcp 65001 before the SSH connection.

Key Insight: Terminal encoding issues can propagate through multiple layers (client terminal → SSH protocol → tmux → shell). When one user encounters a problem while others do not, the client configuration should be investigated first. By comparing observations (“others can use it”), the problem can be narrowed down from the server to the client.

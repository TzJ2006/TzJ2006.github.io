Key Issues#

1. Error Recovery Benchmark v4’s physical force injection couldn’t generate a VLA-compatible time context — Because the errors used as initial states lacked the behavior history required by the VLA model. Solution: Designed v5.0 architecture with an offline two-stage process: Phase 0 collects clean trajectories (human data and 10 times more successful runs for each VLA strategy) → Phase 1 performs offline scanning for injection opportunities → Phase 2 quota scheduler (100 opportunity points per task category) → Phase 3 context playback (playback N-1 frames to build VLA context window, introducing errors at frame N so VLA could freely infer from N). Changed mandatory injection method to behavior-based error skill module (EEF motion/claw control), each module handling detection, injection, and verification of a specific error.#

Key Insight: VLA models have time context dependencies — cannot be initialized with arbitrary errors. Errors must “naturally” occur from correct trajectories. Offline scanning + quota methods efficiently balance datasets while maintaining temporal realism.

2. MIHD spatial transcriptomics patch query visualization showed coordinate misalignment issues — The queried patches didn’t match expected spatial positions#

Solution: Identified 3 root causes: (1) Dataset/DLPFC path mismatch during data loading; (2) Location-based assignments not using barcode mapping (scanning py operation cache order inconsistent with data loading order); (3) Using cache coordinates instead ofadata.obsm[‘spatial’] for marker overlay. Resolved these issues by using consistent paths, bc_to_cache_idx dictionary for barcode alignment, and barcode-based coordinate retrieval.

Key Insight: Spatial transcriptomics workflow requires explicit barcode-based alignment at every step (cache → loading → rendering). Implicit position matching fails when data is reloaded from different sources or re-sorted by scanning py. Never assume indexes remain consistent between I/O boundaries.

3. Robobrain Pi training loss fluctuations — Suspected to be due to dimension processing issues in action prediction#

Solution: In-depth analysis showed that actual actions were 14-dimensional (right arm 6DOF+claw + left arm 6DOF+claw), but for model compatibility, it was expanded to 32 dimensions. However, MSE losses were calculated across all 32 dimensions. This meant 56% of losses (18/32) came from zero-filled dimensions, and the model learned only trivial noise on these dimensions → zero mapping, severely weakening gradient signals. Suggested define action_loss as mean(square(v_t[…, :14] - u_t …,…:14]).

Key Insight: Zero filling for compatibility purposes severely weakens gradient signals in loss calculations — always retaining losses up to the actual dimension. In this case, 56% of gradient signals were wasted learning trivial mappings. This was a subtle but critical error that affected training efficiency.

6. The Robobrain Pi task completion binary classifier may learn an trivial solution of “always predicting 0,” despite task_loss_weight = 0.1.#

Solution: Root cause: Severe class imbalance (only the last 20 frames out of approximately 200 frames in the drama are positive, i.e., a 9:1 positive-negative ratio) plus task_pos_weight = 1.0 (no balance). The solution requires adjusting two hyperparameters separately: keep task_loss_weight = 0.1-0.15 (to prevent the auxiliary task from overwhelming the 50-step action prediction), but increase task_pos_weight to 5.0 (providing enough gradient signals for the positive class through weighted by class).

Key Insight: For unbalanced auxiliary classification, when the task should not dominate the main goal, weighted loss scaling is more effective than overall loss scaling per class. Loss_weight and pos_weight have different purposes: loss_weight balances the importance of the task, while pos_weight balances the category contributions within the task. These two parameters must be adjusted separately.

9. The Kubernetes container nvidia-smi shows that all GPU processes are [unfound], even though GPUs are being actively used Solution: The root cause is an isolation issue in the PID namespace. nvidia-smi returns the host PID (e.g., 3076643) when querying the kernel driver, but the container’s PID namespace has a different numerical ID (e.g., 990465). The solution involved developing the gpumon.py tool to combine psutil with nvidia-smi to read process information within the container: CPU/memory/command line/time elapsed; visualizing this through a rich user interface.#

Key Insight: Isolation in the container namespace disrupts the mapping of process IDs across namespaces. Tools must collect information from each namespace separately before aggregating it. This is why standard monitoring tools fail in containerized environments—they assume that the PID space is unified.

10. The Openpi-moe serve_policy.py crashed due to ‘MUJOCO_EGL_DEVICE_ID must be between 0 and 0,’ but actually got 6.#

Solution: CUDA_VISIBLE_DEVICES=6 limits the process to using only a single GPU as device 0, but the EGL renderer tries to use the original hardware index 6. The os.environ.setdefault(‘MUJOCO_EGL_DEVICE_ID’, ‘0’) was added at the start of the script to align with CUDA’s remapping.

Key Insight: CUDA_VISIBLE_DEVICES reassigns GPU indices within the range of 0-N so that isolated processes can be created; dependent libraries (such as MuJoCo EGL, rendering contexts) must use the remapped indices instead of the original hardware IDs. This is a subtle but crucial interaction between CUDA device isolation and the rendering framework. When using CUDA_VISIBLE_DEVICES with a single GPU, always set the rendering device ID to 0.

11. Despite install.sh successfully completing, the CALVIN calvin_agent module was not found.#

Solution: It was discovered that the pip installation was performed in a conda environment different from the active univla environment. The editable installation creates an.egg-info directory, but package availability is limited to the Python environment where the installation was executed. Reinstalling in the correct environment using pip install -e . –no-deps prevents overwriting PyTorch.

Key Insight: Editable pip installations are environment-dependent—there may be issues if installed elsewhere, leading to import failures. Before investigating missing modules, always verify whether ‘which python’ matches the expected environment. The –no-deps flag allows for installing package structures without affecting existing dependencies.

Lessons Learned#

Key Takeaways#

  • VLA model’s temporal context dependency requires redesigning benchmarks: It cannot be initialized with error states simply. VLA needs behavioral historical data for proper recovery decisions. The solution is to replay the first N-1 frames to build a context window, then insert an error in the Nth frame, allowing VLA to proceed with reasoning from that point onward. This transforms errors from “initial state perturbation” into “trajectory interruption,” which matches how errors occur in real-world scenarios.
  • Offline scanning + quota scheduling architecture for balancing error datasets: Phase 0 collects clean trajectories (human data and 10 times more successful runs than VLA’s strategy); Phase 1 performs offline scanning per frame; Phase 2 allocates quotas (100 quotas per task × error category); Phase 3 conducts context replay. This approach is more efficient than online generation, ensures dataset balance, and allows reuse of expensive clean trajectory collection data in multiple error injection experiments.
  • Error classification based on deformation dimensions: D0 only targets position, D1 only targets rotation, D2 considers both position and rotation. This classification systematically describes the spatial deformation of all types of errors. Compared to stage-based classification (reach/grasp/raise/traffic/placement), this approach is more systematic, making the验证逻辑 clearer by breaking down 6 degrees of freedom into orthogonal components.
  • Compatibility issues with zero-filling architecture: Filling to increase model compatibility from 14 dimensions to 32 dimensions could severely dilute gradient signals (wasting 56%). Therefore, only meaningful dimension information should be retained. In multi-dimensional outputs, this is a subtle but critical issue—the model may converge, but since most gradient signals optimize simple zero-to-zero mappings, the learning effect is not ideal.
  • For imbalanced auxiliary tasks, hyperparameters need to be adjusted independently: loss_weight balances task importance, pos_weight balances category contribution within tasks. They have different roles and must be adjusted separately. For severely imbalanced auxiliary classification tasks, keep loss_weight low (0.1-0.15) to avoid overemphasizing the main goal; increase pos_weight (e.g., 5.0) to address category imbalance issues. Overall scalability and class weight allocation are independent control factors.
  • Spatial transcriptomics requires precise barcode alignment: Clear barcode mappings are needed at each process node (cache → load → render). Location-based matching fails when reloading data from different sources or reordering after scanpy operations. Never assume that the index remains consistent between file I/O boundaries. The solution is to maintain a bc_to_cache_idx dictionary throughout the process and use barcode lookup for all coordinate assignments.
  • Deep understanding of MIHD architecture: 3-stage process (encoder → fusion → clustering/ evaluation), 5 gene encoders (PCA 50d, MLP 128d random initialization, scGPT 512d, GCN 50d, none), 4 visual encoders (ResNet50, HIPT, UNI, UNI2), 10 fusion strategies (connection/ average/ element sum + attention variants + STAIG/QFormer). Divided into two stages: extraction in the first stage (cache to embedding cache/), evaluation in the second stage (load + fusion + clustering + metrics).
  • Transforming and enhancing datasets at the source level: Randomly selecting instruction samples from each set can increase training diversity without using multiple dataset versions. This simple change has a significant impact on the generalization ability of language-conditioning strategies. This highlights its importance in pattern matching with reference implementations within the same code base.
  • CUDA_VISIBLE_DEVICES GPU index remapping: Limit indices to the range 0-N to process within the CUDA context. Dependent libraries (MuJoCo EGL, rendering framework) must use the remapped indices instead of the original hardware ID. When using single-GPU isolation, always set the rendering device ID to 0. This is a subtle but critical issue for multi-GPU setups, as different tasks run on independent devices.
  • Applicability of injection and natural capture: Physics-based injections (pulses/frictions/pose perturbations) can reliably generate 7 types of errors (falling, collision, positioning, partial stagnation). GRASP_FAILURE/sequence/trajectory/regression errors require strategic behavioral responses—only through natural capture can these insights be obtained. A mixed approach is needed to fully cover errors.
  • RM-IDEAL spatial niche matching algorithm: (1) Build a spatial adjacency graph (k-NN, n_neighbors=6), (2) WWL messaging (K=3 iterations summarizing neighbor tags), (3) calculate Hamming distance cost matrix, (4) Wasserstein optimal transport, (5) RM = 1 - W ∈ [0,1]. Cross-section evaluation: bidirectional queries, comparing RM-IDEAL scores with embedding cosine similarity, evaluated using Spearman ρ, P@K, SL@K metrics.
  • Kubernetes PID namespace isolation can disrupt monitoring: nvidia-smi returns host PID when querying kernel drivers, but containers have independent PID namespaces. The solution is to collect information in their respective namespaces and then aggregate it. For GPU monitoring, use psutil to obtain process information (/proc parsing), use nvidia-smi to get hardware status, and visualize using a rich TUI. General principle: namespace isolation requires localized data collection.
  • Research code library’s documentation philosophy: Users prefer a “minimal use-oriented” structure (what/ how/ plan/ result) rather than a “comprehensive classification” (detailed function guides). Reason: Research code changes quickly, and detailed guides become outdated soon. Instead of having a thick reference manual, it’s better to have concise integration documents and well-documented code. AI should default to minimal documentation and ask users before creating a comprehensive structure.
  • Code vs. documentation consistency audit workflow: (1) Start parallel exploration proxies for each module, (2) Compare statements in the README with actual code (class names, dimensions, parameters, training process), (3) Classify differences (serious errors/ moderate errors/minor omissions), (4) Users choose a repair strategy (update documentation or fix code). In MIHD projects, 1 serious ImportError and 6 dimension mismatches were found.
  • AI behavior during exploration phase: When users’ thinking is still developing (with multiple statements of “new ideas”), avoid forcing decisions through AskUserQuestion. Provide information, analyze trade-offs, and wait for autonomous decision-making. Continuously rejecting ExitPlanMode/AskUserQuestion calls is a clear “pause” signal. Users need to explore the space rather than prematurely deciding on implementation paths.
  • Node.js v24 compatibility issues with proxy acquisition: The built-in acquisition method (undici) does not follow HTTPS_PROXY/HTTP_PROXY environment variables, unlike curl/axios/requests. A explicit NODE_OPTIONS="–use-env-proxy" or global proxy library is required. This is very important for restricted network environments. This represents a change in behavior compared to traditional HTTP clients.
  • Editable pip installation environment range: Package availability is limited by the Python environment where the installation command is executed. Although some files (.egg-info) may exist, importing them elsewhere will fail. Always verify that “which python” matches the expected environment. Use –no-deps to install package structures without overwriting existing dependencies (this is important for avoiding PyTorch version conflicts).
  • MIHD clustering methods: KMeans (default option, fast), mclust (implementing R language GMM algorithm through rpy2, suitable for STAIG scenarios with zero-variance backoff mechanism → using KMeans); Leiden method (used for HD data without real data, automatically performing cluster estimation). The choice logic is based on the fusion strategy and data characteristics.
  • Terminal encoding troubleshooting: Problems can propagate through multiple layers (client terminal → SSH protocol → tmux → shell). When one user encounters a problem but others do not on the same server, check for problems in the client side first. By observing whether “others can also use it,” you can quickly narrow down the problem range from shared components to specific user configurations.

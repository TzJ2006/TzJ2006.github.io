Problems and Solutions#

Key Problems#

1. CalendarPro robots cannot understand subsequent messages, such as “What time is your schedule?” Solution: Identified three levels of problems: (1) general_keywords pre-checking executed before the AI saw the dialogue history; (2) pre-checking used overly broad keywords like ‘what time’; (3) the AI provider’s call did not receive historical data parameters. The designed solution is to eliminate pre-check bypass behavior and pass the complete dialogue history to the AI.#

Key Insight: Context loss is usually a multi-level problem: data not collected → data not passed → data not used. It must be tracked from storage to the provider interface to prompt construction.

2. SAPIEN encounters ‘rendering error’ in headless Docker containers - RuntimeError: Rendering device not found#

Solution: Discovered that NVIDIA Vulkan/OpenGL rendering libraries (libGLX_nvidia.so, etc.) were missing. The container had the CUDA computing library but no rendering stack. Downloaded the matching NVIDIA driver version 535.104.12 to match the kernel driver, extracted the library files, and configured VK_ICD_FILENAMES and LD_LIBRARY_PATH environment variables in eval.sh.

Key Insight: Docker containers with NVIDIA GPUs require CUDA runtime and Vulkan/OpenGL libraries for graphical simulations. The driver version must strictly match the kernel module and user space library; otherwise, the Vulkan ICD will not load.

3. Phoenix training checkpoints exist in zhaoganlong’s directory, unclear whether to reuse or retrain#

Solution: Users indicated that existing checkpoints should be checked and used via symbolic links only when available, and only training the missing parts on an53 idle GPUs.

Key Insight: Before designing a training plan, it is necessary to inventory existing resources (checkpoints, datasets, configurations) to avoid wasting computing resources. Establish a checkpoint reuse strategy first.

4. Users believe standardized data comes from the STHD process, but data characteristics do not match the process code#

Solution: By tracing the code, it was found that STHD does not perform standardization—it directly uses raw counts. The standardized data users saw came from external scanpy preprocessing (normalize_total + log1p).

Key Insight: It is necessary to distinguish between processing within the process and external preprocessing. Code inspection is more reliable than assumptions based on data appearance.

General Problems#

5. Missing warm-up training configurations for 3 out of 9 MimicGen tasks in OpenPI configuration#

Solution: Used add_finetune_config and add_inference_config helper functions to add fine-tuning and protection model configurations for coffee_task_D1, stack_three_task_D1, and three_piece_assembly_task_D1.

Key Insight: Use configuration generation functions to maintain consistency across task configurations instead of manually repeating the same steps.

6. FrozenInstanceError: Unable to assign field ‘asset_id’ to field in policy_config.py#

Solution: DataConfig was defined as @dataclasses.dataclass(frozen=True), making it unchangeable. Use dataclasses.replace(data_config, asset_id=value) to replace direct assignment (data_config.asset_id = value), allowing for the creation of a new instance.

Key Insight: Frozen data classes enforce immutability at the language level—modifications require using dataclasses.replace() to create a new instance, rather than modifying it in place.

7. FileNotFoundError, path duplication: assets/norm_stats.json/norm_stats.json#

Solution: When robotwin_repo_id was provided, asset_id was incorrectly set to ’norm_stats.json’ (file name instead of directory). The modified code directly used normalize.load() from the assets/ directory without adding an asset_id sub-directory.

Key Insight: Path construction errors are usually caused by mixing directory identifiers with file names. Explicit handling for special cases (local files vs remote repositories) can prevent path duplication.

8. MulticoreTSNE build failed due to CMake version incompatibility (system 3.26.4 not compatible with <=3.18)#

Solution: It was found that MulticoreTSNE is only used for optional TSNE visualization, not the core training/evaluation. Recommended to comment out dependencies to proceed with installation (user refused this approach).

Key Insight: It is necessary to distinguish between critical dependencies and optional visualization tools. In research code, non-essential dependencies that cause build problems can usually be skipped.

AI Limitations#

Key Limitations#

  • Did not check existing checkpoints when designing the training plan - Need to remind users to consider reuse strategies. Before implementing, resources should always be inventoried.
  • Initially tried to install NVIDIA graphics libraries (libnvidia-gl-535) without first checking kernel driver version compatibility, resulting in version mismatch (535.288.01 vs 535.104.12). Multiple iterations were required to find the correct download match for the driver and extract the library.

General Limitations#

  • Although the exploration of dialogue history records was very comprehensive, it required 3 parallel agents to identify the core problem where LLM calls did not receive historical parameters - If the initial exploration had been more targeted, the problem could have been identified faster.
  • Directly creating a tutorial file of 1674 lines without first previewing the structure to confirm user opinions - This method should be verified on larger works.
  • Modified test_render.py by replacing the bare ’except:’ with ’except Exception as e:’, but this pattern (swallowing exceptions without logging) exists in the original code library and was not introduced by AI. AI should have identified this anti-pattern earlier.
  • Multiple runs/init commands conducted similar explorations and analyses, but did not remember previous evaluation results - Lack of session memory functionality.
  • Spent a lot of time conducting systematic explorations (finding commands, vulkaninfo checks), which could have been shortened if AI had a more direct understanding of common Docker+NVIDIA Vulkan setup patterns.
  • Ultimately completed successfully, but the steps were more than what experts would require.

Learning Outcomes#

Key Learnings### Practical Learning Outcomes#

  • Python frozen data classes provide guaranteed compile-time immutability, but explicit modifications using dataclasses.replace() are required. Direct assignment will trigger FrozenInstanceError. This pattern forces configuration objects to adopt a functional programming style.

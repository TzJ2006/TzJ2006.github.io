Issues and Solutions#

Key Issues#

1. Error recovery robots never approached objects during demo replays (minimum distance was 0.22 meters, tasks require contact).#

Solution: Analyzed robosuite source code, HDF5 metadata, and MuJoCo’s body mapping relationships. It was found that demonstrations recorded using PickPlace_D0+controller_configs used pure PickPlace during replays. The same operation produced completely different trajectories. Key Insight: Environmental configurations (kp, damping coefficient, ramp_ratio in controller_configs) fundamentally changed the robot’s dynamics. Always use environmental metadata completely matching the HDF5 dataset for replays—parameter adjustments could mask the underlying problem.

2. CalendarPro’s semantic router and LLM conflicted, causing the system to always use SR, leading to incorrect routing (e.g., “leetcode” keyword → using manage_recurring instead of update).#

Solution: Modified dual_verify.py to prioritize LLM intentions when mismatches occurred. The LLM had complete dialogue context, while SR only matched keywords. Logs showed “→using LLM intentions” for monitoring. Key Insight: Keyword-based classifiers would fail when dealing with ambiguous expressions. When there was a disagreement between semantic routing and LLM, the more contextually supported model (LLM) should be trusted, rather than the faster model (SR). SR would filter obvious cases, while the LLM acted as an arbitrator.

3. Missing data in 4 MIHD sections caused pipeline failures (missing spatial files, empty frequency filters, STAIG model crashes).#

Solution: Extracted spatial files from Dataset.zip; added boundary checks (min_w/min_h >= 10); added backtracking mechanism from mclust to KMeans, using try/except wrappers for handling. Key Insight: Spatial transcriptomics data required robust fault tolerance: multi-path candidate searches, image processing boundary checks, statistical model backtracking strategies to handle distribution anomalies.

4. Gadget’s original single-step workflow couldn’t support multi-device collaboration (required manual copying of logs to each device).#

Solution: Refactored into a two-stage process: (1) Exporting sub-commands to generate local JSON logs with device information (no API key needed); (2) Merging sub-commands, aggregating multiple logs, and calling API to generate final reports; keeping the old mode for backward compatibility. Key Insight: Multi-device workflows required portable intermediate formats (versioned JSON logs) and clear phase division (local parsing and remote merging). Main designs included: device information collection, version support for future expansion, optional device summaries to prevent information loss during merging.

5. All error detectors failed after adding proximity gates—iterative threshold adjustments (0.05→0.15→0.25→0.30 meters) didn’t produce any trigger signals.#

Solution: Written diagnostic scripts to measure actual EEF-object distances, finding that the issue was environmental mismatch, not threshold problems. The root cause was playback configuration issues, not detector sensitivity issues. Key Insight: When all detectors failed, real-world measurements should be taken through diagnostic logging, followed by threshold adjustment. Symptoms without trigger signals often masked underlying mismatches (e.g., incorrect controller dynamics).

6. CalendarPro calendar API errors (403 unauthorized access, HttpError) were caught and returned as None via print() function, resulting in users receiving a general “failed” message, causing confusion.#

Solution: Created CalendarServiceError exception; refactored all calendar_service methods to provide detailed context when errors occurred; updated all callers to capture and display user-friendly messages with technical details. Key Insight: Silent error handling (print + returning None) violated error handling conventions. Users couldn’t fix problems they couldn’t see. Always pass contextual errors to the presentation layer, then format them appropriately.

7. CalendarPro AI provider lacked timeout handling mechanism—network hangs could cause the entire robot to freeze indefinitely.#

Solution: Wrapped all API calls with asyncio.wait_for(timeout=30.0), and added httpx.Timeout(30.0) in the client constructor, returning an AIResponse with error fields when timed out. Key Insight: Timeout protection required dual defense: asyncio coroutine level (wait_for) and HTTP client level (httpx.Timeout), to catch different failure modes.

8. CalendarPro’s JSON parsing failed when AI returned prose before JSON, multiple blocks, or truncated JSON—resulting in intent classification failures.#

Solution: Implemented extract_json() function, extracting the first complete object by balancing parentheses count; added three-stage backtracking mechanism (direct parsing→tail trailing comma correction→general intent); removed ValueError exception for logging and backtracking purposes. Key Insight: Production-grade AI systems needed graceful degradation—for malformed outputs, they must never crash, always providing safe backtracking intentions.

9. Plan documentation lacked self-contained nature—users repeatedly rejected ExitPlanMode, demanding “writing every sentence of the plan into the plan document.”#

Solution: Redrew the plan documentation to make it fully self-contained: added user decision records, code snapshots with exact line numbers, QueST reference code details, complete implementation code, with “inserted after line X” locations for each stage. Key Insight: Plan mode documentation must be fully self-contained to be restored later. Users’ rejections indicated insufficient documentation details—all exploration results (exact locations, line numbers, function signatures, existing code) needed to be written into the plan, rather than relying on external context.

10. MuJoCo’s speed switching caused false instability triggers (angular velocity was misinterpreted as linear).#### 11. The CLAUDE.md file has 615 lines, with a lot of repetition between sections. Some content also belongs to the README file.#

Solution: Rewritten documentation, focusing on: (1) Unimportant information (server restrictions, common pitfalls); (2) Consolidating commands into one block; (3) Removing easily noticeable content (file tree, method signatures); (4) Keeping only the architectures that require multi-file reading.

Key Insight: Documentation for AI agents should focus on content that tools cannot discover, rather than being a comprehensive reference. Simplicity improves usability—101 lines vs. 615 lines, making it clearer.

12. When CalendarPro updates duplicate events, it modifies the entire series instead of individual instances, even though singleEvents=True is used in searches.#

Solution: Modified the update_event() function to detect duplicate instances (with recurringEventId), and remove duplicate/reoccurringEventId before calling the API. Google Calendar creates exceptions for such instances separately.

Key Insight: Implicit behavior of Google Calendar API: using singleEvents=True in searches gives instance IDs, but updates/deletes require explicitly removing relevant fields to avoid modifying the entire series. Always test duplicate operations on a test calendar.

13. CalendarPro’s asyncio.create_task() for message queues lacks exception handling callbacks—tasks may silently terminate when the queue is full, and user messages will never be processed.#

Solution: Added done_callback=_on_queue_task_done to all create_task calls; implemented cleanup handlers to record exceptions and remove dead tasks; added close() method for closing.

Key Insight: asyncio background tasks are “after-triggered forgotten,” so done callbacks are always needed for monitoring/cleanup, especially for long-running tasks.

General Issues#

14. Manually querying token usage is cumbersome—users want automatic daily statistics.#

Solution: Integrated ccusage by calling npx ccusage@latest daily –json, parsed output; automated collection at all three entry points; displayed total tables and model details in token usage section; consolidated data across devices.

Key Insight: Automate data collection using existing tools instead of reinventing them. Design considerations include: fault tolerance of subprocesses (skip silently if unavailable), multi-device aggregation strategy (summarizing tokens/costs, combining model details), and readable presentation (tables > lists).

15. MIHD GPU peak statistics always remain zero (ResourceMonitor never calls update_peak_stats()).#

Solution: Refactored ResourceMonitor: start_monitoring() starts a daemon thread that samples every second, stop_monitoring() terminates the thread and samples the final peak, added threading.Event lifecycle control.

Key Insight: Monitoring peak statistics requires continuous sampling during operation, not just startup/stop snapshots. Background daemon threads are a standard solution, but proper lifecycle management is needed to avoid resource leaks.

16. The gadget test/directory is a nested Git repository with its own remote address, causing conflicts during git add operations.#

Solution: Used git submodules: moved test folder to a backup location, ran ‘git submodule add git@github.com:TzJ2006/test.git test’ for cloning, removed backup, .gitmodules automatically records configuration.

Key Insight: Git submodules are suitable for nested repositories with independent remote repositories. Key points: submodules record commit hashes rather than branches; git submodule update --init --recursive is required during cloning; updating requires entering the submodule directory, executing git pull, then modifying the commit hash.

17. CalendarPro’s start.bat cannot detect conda environments (cmd doesn’t support ‘conda activate’ without conda init, new cmd window doesn’t have CONDA_PREFIX).#

Solution: Created start.ps1 script that supports native PowerShell conda functionality; PowerShell can directly execute ‘conda activate calendarpro’. Also updated start.bat to allow it to call PowerShell under -ExecutionPolicy Bypass policy; added PID tracking and stop.ps1 script.

Key Insight: Windows batch files perform poorly in conda integration. For Windows-based conda workflows, PowerShell is a better choice. Always test environment activation in a new shell, rather than using inherited conda sessions.

18. CalendarPro’s simulated patch failed because generate_with_fallback was imported inside the method, not at the module level.#

Solution: Changed the patch target from ‘src.thoughts.thought_organizer.generate_with_fallback’ to ‘src.aiProvider_selector.generate_with_fallback’ to apply patches to the source module.

Key Insight: During simulation, patches should affect the function definition, not the import location—local imports create new references, thus bypassing module-level patches.

19. After creating test_thoughts.py, CalendarPro’s pytest async tests encountered ‘coroutine never awaited’ warnings.#

Solution: Installed the pytest-asyncio package, which can automatically detect async test functions and run them in the event loop.

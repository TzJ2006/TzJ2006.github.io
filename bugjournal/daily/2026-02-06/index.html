Issues and Solutions#

Key Issues#

1. Public users cannot push data to the repository, but need to contribute benchmark results#

Solution: Designed a three-tier architecture: users submit data to relay endpoints → relays validate and forward data to repository_dispatch → GitHub Actions queue/ingest/publish. Relays hold write tokens, while users have no repository access rights.

Key Insight: Crowdsourcing data pipelines requires an untrusted ingestion path; GitHubActions combined with validation workflows can provide audit trails and abuse controls without granting contributors write access.

2. Bar charts show opposite lengths (bars for higher scores are shorter), even though CSV values are correct#

Solution: The root cause was mismatching binary encoded chart data (bdata) and CDN versions. Solutions include: (1) fixing Plotly JS to match installed Python version, (2) converting all chart x/y/text to plain Python lists instead of Pandas Series, (3) forcing flops_gflops to be numerical types upon loading, (4) setting yaxis rangemode=‘tozero’.

Key Insight: Binary serialization formats in client plotting libraries may cause silent rendering errors when CDN versions change; explicit list conversion and version fixing ensure visual consistency.

3. Automatically uploading benchmark data without user consent raises ethical issues#

Solution: Added interactive prompts after saving runs (default: no upload), added explicit –upload/–no-upload flags for automation, required configuring relay URL before prompts, allowing non-interactive runs to skip prompts unless –upload is specified.

Key Insight: Telemetry and data collection require explicit consent and clear disclosures; default no-upload prompts and automated modes with flags respect user preferences.

4. HOSTNAME regex is too broad, causing timestamps and version strings in valid data to be corrupted#

Solution: Narrowed down HOSTNAME_RE range from matching any token-separated string to requiring TLD suffix (.[A-Za-z]{2,}\b), avoiding misidentification of ‘13.0’ or ‘2026-02-07’ patterns.

Key Insight: Privacy disinfection rules require precise boundaries; overly aggressive regex patterns may damage legitimate technical metadata sharing surface syntax, leading to PII leakage.

General Issues#

5. Trend charts were incomplete, leaving outdated method calls that caused AttributeError crashes#

Solution: Added compatibility backports create_trend_chart(), returning empty graphs so old or outdated call paths could gracefully fail instead of causing AttributeError.

Key Insight: Defensive programming with no-op templates prevents fragile failures during feature removal; compatibility layers allow smooth migration of paths.

Learning Outcomes#

Major Takeaways#

  • Crowdsourcing data pipelines require a three-tier architecture (untrusted submissions → trusted gateways → protected data storage) with validation/audit mechanisms to prevent abuse without granting write permissions
  • Ethical data collection requires explicit consent confirmation, defaulting to “no” option; automated scenarios require explicit flags rather than relying on interactivity detection
  • Visual errors in client visualization often stem from serialization format mismatches rather than data correctness issues; explicit list conversion and version fixing can prevent silent rendering failures
  • Privacy disinfection rules require precise boundary conditions; overly aggressive regex patterns may damage legitimate technical metadata sharing surface syntax, leading to PII leakage

Practical Gains#

  • Feature removal refactoring should include compatibility stubs for gradual migration; no-op backports can prevent fragile AttributeError crashes due to old call paths

Problems & Solutions#

Critical Issues#

1. Public users cannot push to repository but need to contribute benchmark results#

Solution: Designed three-tier architecture: users submit to relay endpoint → relay validates and forwards to repository_dispatch → GitHub Actions queue/ingest/publish. Relay holds write token, users have zero repo permissions

Key Insight: Crowdsourced data pipelines require untrusted ingestion paths; GitHub Actions with validation workflows provide audit trail and abuse control without granting contributor write access

2. Bar charts rendered with inverse lengths (higher scores showed shorter bars) despite correct numeric values in CSV#

Solution: Root cause was binary-encoded plot data (bdata) and CDN version mismatch. Fixed by: (1) pinning Plotly JS to installed Python version, (2) converting all chart x/y/text to plain Python lists instead of Pandas Series, (3) forcing flops_gflops to numeric at load time, (4) setting yaxis rangemode=‘tozero’

Key Insight: Binary serialization formats in client-side plotting libraries can cause silent rendering bugs when CDN versions drift; explicit list conversion + version pinning ensures deterministic visualization

3. Automatic benchmark uploads without user consent raised ethical concerns#

Solution: Implemented interactive prompt (default No) after saved runs, added explicit –upload/–no-upload flags for automation, required relay URL configuration before prompting, made non-interactive runs skip prompt unless –upload specified

Key Insight: Telemetry and data collection require explicit opt-in with clear disclosure; default-No prompts and automation-requires-flags patterns respect user agency

4. Hostname sanitization regex was too broad, corrupting timestamps and version strings in valid data#

Solution: Narrowed HOSTNAME_RE from matching any dot-separated tokens to requiring a TLD suffix (.[A-Za-z]{2,}\b), preventing false positives on ‘13.0’ or ‘2026-02-07’ patterns

Key Insight: Privacy sanitization rules need precise boundaries; overly aggressive regex patterns can corrupt legitimate technical metadata that shares surface syntax with PII

General Issues#

5. Trend chart removal was incomplete, leaving stale method calls that caused AttributeError crashes#

Solution: Added compatibility fallback create_trend_chart() returning empty figure, so legacy/stale call paths fail gracefully instead of raising AttributeError

Key Insight: Defensive programming via no-op stubs prevents brittle failures when refactoring removes features; compatibility layers smooth migration paths

Learnings#

Key Learnings#

  • Crowdsourced data pipelines require three-tier architecture (untrusted submit → trusted gateway → protected storage) with validation/audit trails to prevent abuse without granting write access
  • Ethical data collection requires explicit opt-in consent with default-No prompts; automation scenarios need explicit flags rather than relying on interactivity detection
  • Client-side visualization bugs often stem from serialization format mismatches rather than data correctness; explicit list conversion + version pinning prevents silent rendering failures
  • Privacy sanitization rules need precise boundary conditions; overly aggressive regex patterns can corrupt legitimate technical metadata sharing surface syntax with PII

Practical Learnings#

  • Feature removal refactors should include compatibility stubs for gradual migration; no-op fallbacks prevent brittle AttributeError crashes from stale call paths

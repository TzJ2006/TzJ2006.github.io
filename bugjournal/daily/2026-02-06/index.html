Issues and Solutions#

Key Issues#

1. Public users cannot push results to the repository but need to contribute benchmark test results.#

Solution: Designed a three-tier architecture: user submits on the relay side → relay verifies and forwards results to repository_dispatch → GitHub Actions queues/pulls/publishes. The relay holds the write token, and users have no repository access rights.

Key Insight: Crowdsourcing data pipelines require an untrusted pull path; GitHub Actions with verification workflows can provide audit trails and abuse controls without granting contributors write permissions.

2. Bar charts show inverted lengths (higher scores have shorter bars), even though the values in CSV are correct.#

Solution: The root cause is mismatching binary-encoded chart data and CDN versions. Fixes include: (1) fixing Plotly JS to match installed Python version, (2) converting all chart x/y/text to plain Python lists instead of Pandas Series, (3) forcing flops_gflops to a numerical type at load time, (4) setting yaxis rangemode=‘tozero’.

Key Insight: Binary serialization formats in client plotting libraries can cause silent rendering errors when CDN versions change; explicit list conversion and version fixation ensure visualization certainty.

3. Automatically uploading benchmark tests without user consent raises ethical issues.#

Solution: Added interactive prompts after saving runs (disabled by default), added explicit –upload/–no-upload flags for automation, required configuring relay URLs before the prompt, allowing non-interactive runs to skip prompts unless the –upload option is specified.

Key Insight: Telemetry and data collection require clear choices and transparent disclosures; disabling prompts and requiring flags respects user preference.

4. The hostname purification regex was too broad, causing timestamps and version strings in valid data to be corrupted.#

Solution: Narrowed HOSTNAME_RE from matching any dot-separated tokens to requiring a TLD suffix (.[A-Za-z]{2,}\b) to prevent misjudgments of ‘13.0’ or ‘2026-02-07’ patterns.

Key Insight: Privacy purification rules require precise boundaries; overly aggressive regex patterns can damage valid technical metadata shared with PII having similar surface syntax.

General Issues#

5. Trend charts were not removed thoroughly, leaving outdated method calls that caused AttributeError errors.#

Solution: Added compatibility fallbacks for create_trend_chart() function, returning empty charts so old or outdated call paths could fail gracefully without causing AttributeError errors.

Key Insight: Defensive programming to prevent brittle failures through non-operating interfaces avoided issues when removing features during refactoring; compatibility layers smoothed the migration path.

AI Limitations#

Key Limitation - Did not proactively identify the impact of “public users pushing to GitHub” on security/access control until a human explicitly asked about it.#

  • Initially, binary-encoded chart data (bdata) was recommended, which led to rendering mismatches between browsers and CDN Plotly versions.
  • The initial hostname cleanup regex was too broad, causing legitimate version strings and timestamps to be corrupted until it was discovered through manual testing.

General Limitations#

  • Incomplete trend charts were not removed, leaving outdated method calls that required compatibility fallbacks. Users raised this issue after reporting AttributeError errors.

Learnings#

Key Takeaways#

  • Crowdsourcing data pipelines require a three-tier architecture (untrusted submission → trusted gateway → protected storage) with verification/audit trails to prevent abuse while not granting write permissions.
  • Ethical data collection requires explicit consent, which should not be prompted by default; automated scenarios require explicit flags rather than relying on interactive detection.
  • Client visualization errors usually stem from mismatching serialization formats rather than data correctness issues; explicit data list conversion and version fixation can prevent silent rendering failures.
  • Privacy cleanup rules require precise boundary conditions; overly aggressive regex patterns may damage legitimate technical metadata shared with PII having similar surface syntax, leading to PII leakage.

Practical Takeaways#

  • Feature removal refactoring should include compatibility stubs for gradual migration; non-operating fallbacks can prevent brittle AttributeError crashes caused by outdated call paths.

Issues and Solutions#

Key Issues#

1. Context Replay’s policy_adapter feeds during the replay loop (clean trajectory phase), forcing the strategy to see a clean state before injection rather than a stable state after errors, which does not match actual deployment scenarios.#

Solution: Move the call to policy_adapter.predict() to Step F (completed collect_rollout_stats, environment stabilized), ensuring the strategy receives post-error observation with a stable environment.

Key Insight: The timing of policy_adapter feeding must align with actual deployment scenarios—the strategy should only see the state after errors occur and the environment stabilizes; feeding clean frames during the replay phase is meaningless.

2. The context_window parameter name carries two meanings (VLA observation window size vs rendering start frame offset), leading to incorrect documentation descriptions like “VLA requires timing context window,” which does not match the actual VLA open-loop reasoning mechanism.#

Solution: Rename the parameter to render_window, clarifying its sole purpose as controlling rendering range; batch-replace 22 errors in 7 code files with “MuJoCo deterministic simulation state replay” for correct descriptions, grep verification zero remaining.

Key Insight: VLA is an open-loop reasoning system that accepts single frames without maintaining a recurrent state, so it doesn’t need a “timing context window.” Naming is the cheapest form of documentation; incorrect narratives are more damaging than code bugs and must be systematically cleaned up.

3. TokenMonitor Dashboard jitters vertically at a rate of 1-2Hz across multiple layers: data refresh → re-rendering → window Resize → data refresh again, with oscillations in the is_active state at the 30-minute boundary.#

Solution: Four-layer concurrent fixes: ①RESIZE_SETTLE_DELAY_MS increased from 16 to 100ms to stabilize the window; ②shallowPayloadEqual skipped meaningless store updates; ③resize throttling (up to 3 times within 500ms); ④is_active check added a 2-minute grace period to eliminate time boundary oscillations.

Key Insight: Oscillating jitter bugs involving multiple layers of positive feedback loops must be fixed at all stages simultaneously; any single fix will only weaken the problem, not completely eliminate it.

4. BOSS evaluation script reports KeyError: ‘potato’ during runtime, preventing task environment loading; in boss_44 evaluation results, 7 LIVING_ROOM tasks had 0% success rates, suggesting either model generalization or training data issues.#

Solution: Copied 5 missing object assets (corn/egg/lemon/onion/potato) from the BOSS repository to the corresponding LIBERO directories and registered them as new classes; checked form_boss_44_dataset.py to confirm that boss_44’s allowed_files whitelist contained only KITCHEN_SCENE files—7 LIVING_ROOM tasks were intentionally excluded as zero-shot generalization tests, with 0% success rate being a deliberately designed benchmark behavior.

Key Insight: BOSS is an extended benchmark of LIBERO, introducing new objects from the standard repository; its core testing concept (Out-of-Suppositional-Set) evaluates the model’s zero-shot generalization ability in completely unseen scenarios, so a 0% failure cannot be misjudged as a model failure.

5. cchypothesis’s current Phase 3 pure READ-ONLY parallel architecture cannot verify runtime assumptions (timing races, data flow states, dynamic behavior), leaving debugging capabilities vulnerable.#

Solution: Designed an intelligent dual-track architecture: parallel READ-Only subagents for static assumptions; upgrading inconclusive static analysis results to serial instrumentation detection ([DEBUG Hx] tag logs + each git restore cleanup); Git Safety Checkpoint as the entry point for Phase 3, protecting users’ work environment.

Key Insight: Instrumentation debugging should be a step-up mechanism when static analysis cannot provide enough information, rather than an alternative. It allows maintaining parallel processing speed while gaining runtime detection capabilities.

6. Rust project had 42 compilation errors (unresolved import crate::change_stats/crate::integrations), and the coexistence of old commands.rs (2222 lines) with new commands/ directory caused module path ambiguity.#

Solution: Rewrote 2222 lines of old commands.rs into about 80 thin modules, correctly declaring 6 submodules under the commands/ directory; removed all duplicate functions and outdated imports moved to submodules; also fixed 4 pre-stored clippy warnings.Key Insights: In the Rust module system, commands.rs and commands/mod_rs serve as mutually exclusive module roots. When both exist simultaneously, old files prevent new sub-modules from being recognized. Incremental refactoring must be done in a last-minute manner to replace or delete old entries.

7. The TokenMonitor pop-up appears in the middle-left part of the screen rather than the system tray area. During resizing, the bottom edge moves instead of staying fixed at the top (top-bottom inversion). Dynamic detection using detect_vertical_anchor is prone to false positives under initialization timing and concurrency conditions.#

Solutions: ① Use the Win32 API (Shell_TrayWnd → TrayNotifyWnd) to obtain precise tray coordinates instead of tauri_plugin_positioner. ② Remove the dynamic detection for detect_vertical_anchor, and always use work.bottom - height to calculate the Y coordinate for aligned_window_origin. ③ Change the front-end setSize() method to call the back-end set_window_size_and_align IPC atomic command to update both size and position.

Key Insights: The Windows support for tauri_plugin_positioner is unreliable. The system tray pop-up always anchors at the bottom, so no dynamic detection is required. Tauri’s setSize() is a pure size API, and resizing needs to be handled through the back-end IPC atomic command to process both size and position.

Today’s Takeaways#

Core Takeaway#

  • The VLA (Vision-Language-Action) model is an open-loop reasoning mechanism: Each step accepts single-frame observation outputs of actions without maintaining a recurrent hidden state. Therefore, there is no need to “pre-replay N frames to establish a context window” - this assumption is a fundamental misunderstanding of the VLA working mechanism.
  • The MuJoCo simulation state is deterministic but not reproducible as a snapshot: It is necessary to execute actions frame by frame from the initial state to obtain the correct intermediate physical state. Directly setting sim_state_flat to the target frame is a potential alternative (physical consistency needs to be evaluated).
  • UI jitter-related bugs are often the result of multiple layers of positive feedback loops. Single-layer repairs can only weaken, not eliminate, these problems. It is necessary to interrupt all links simultaneously (measurement delay, equality check, throttling valve, data boundary grace period).
  • The correct architecture mode for poking debugging: Use static parallel analysis as the main path and poking detection as the upgrade path for inconclusive results. Each assumption is independently completed through the instrument→run→analyze→cleanup cycle. Git Safety Checkpoint protects the user’s work area at the entry point. In multi-stage debugging, use git restore. rather than git stash to avoid stacking conflicts.
  • Naming is the cheapest way to express design intentions (context_window → render_window). Incorrect documentation narratives are more harmful than code bugs - they do not affect the current running results, but they cause systematic misunderstandings during project handover, paper writing, and code review. It is necessary to actively identify and systematically clean up such issues.
  • BOSS benchmark design mechanism: The boss_44 training set intentionally covers 37 KITCHEN tasks through the allowed_files white-list but excludes 7 LIVING_ROOM tasks for zero-sample generalization evaluation. BOSS has expanded the standard LIBERO object assets (corn/egg/lemon/onion/potato). Before using them, you need to obtain them separately from the BOSS repository.
  • The system tray pop-up should always anchor at the bottom (work.bottom - height). There is no need for dynamic detection (dynamic detection is prone to false positives under initialization timing and concurrency conditions). The Windows support for tauri_plugin_positioner is unreliable. It is necessary to directly obtain precise coordinates using Win32 API (FindWindowW/FindWindowExW). Tauri’s setSize() is a pure size API. Resizing needs to be handled through the back-end IPC atomic command to handle both size and position.
  • The safe order for Rust incremental module refactoring: First create new file structures and re-export old entry files. After verification, replace or delete old entries in the last step. The coexistence of commands_rs and commands/mod_rs causes module path ambiguity. Old files prevent new sub-modules from being recognized.
  • The shallowPayloadEqual mode (reference equality check when cache hits + shallow comparison of fields during background refresh) is an effective way to avoid unnecessary Svelte store rendering. It is especially suitable for high-frequency data polling scenarios.

Problems and Solutions#

Key Issues#

1. cchypothesis lacks runtime debugging capabilities — All hypotheses are verified only through static code analysis, making it impossible to detect dynamic state/time-line/data flow issues.#

Solution: Implemented a third-stage intelligent classification: static hypotheses remain in parallel read-only mode; runtime hypotheses use sequential debugging mode with [DEBUG Hx] marked logs and Git security checkpoints, automatically cleaned up via git restore.

Key Insight: Dual-track investigation retains speed (parallel static processing) when needed, while increasing depth (debugging mode detection) — avoiding slowing down all debugging sessions.

2. Vertical dashboard jitter that persists at 1-2Hz is caused by a three-layer positive feedback loop: data refresh → UI re-rendering → window size adjustment → data refresh.#

Solution: Multi-layer interrupt mechanism: (1) Window: 16ms to 100ms settling delay + limited speed of 500ms/3 operations; (2) Data: shallow PayloadEqual, skipping unnecessary.set() functions; (3) Parser: 2-minute active timeout to prevent five_hour_cost oscillations.

Key Insight: Self-oscillation errors require intervention at all feedback layers simultaneously; fixing one layer alone only worsens the problem without eliminating cycles.

3. Git storage stack conflict: ccplan proposal added a third-stage storage above the existing fourth-stage storage, causing confusion in pop order and potentially losing uncommitted user work.#

Solution: Removed the fourth-stage Git storage and used non-destructive git restore for cleanup; now only the third stage uses storage (single storage model).

Key Insight: Multiple storage layers in automated workflows make state management fragile — prefer non-stated cleanup (git restore) over stated undo (storage).

4. TokenMonitor window appears in the center of the screen instead of near the system tray; size adjustments cause the window to extend down into the taskbar instead of maintaining a bottom edge.#

Solution: Implemented direct Win32 API control of tray position (FindWindowW → TrayNotifyWnd); changed front-end size adjustment from webviewWindow.setSize() to backend set_window_size_and_align IPC for atomic size and position updates.

Key Insight: Tauri’s cross-platform positioning plugin abstracts platform differences, but accuracy is reduced; Windows system tray positions require platform-specific Win32 API for precise alignment.

5. 42 Rust compilation errors: command.rs includes incorrect imports of deleted modules (change_stats, integrations, parser), as well as duplicate code from new commands/submodules.#

Solution: Rewrote command.rs into a simple 80-line module structure, declaring 6 common modules, retaining only AppState/UsageDebugReport and 4 shared helper functions, fixing visibility and type issues.

Key Insight: Incomplete refactoring (using old and new code together) is worse than single-unit code; either complete the transition or undo the refactoring.

6. Window anchor detection is unreliable: visible:false initially, there’s a competition between position_near_tray and GetWindowRect, continuous size adjustments cause anchors to flip.#

Solution: Completely removed the dynamic detection feature; tray pop-up windows always use position_near_tray to work.bottom, so the bottom anchor behavior was hardcoded.

Key Insight: For known types of windows, static configuration is more reliable than dynamic detection; runtime detection adds complexity and failure modes but offers no additional benefits.

General Issues#

7. Critical agent found that git restore does not clean up untracked files (e.g., .debug/hypothesis.log created during debugging).#

Solution: Explicitly perform rm -rf .debug/ operation after each debugging cycle; move log output directory from.claude/ (configuration directory) to.debug/ (temporary project directory) and guide use of gitignore.

Key Insight: Git restore only affects tracked files — automated cleanup workflows must combine Git commands with explicit file system operations to achieve thorough cleanliness.

Learning Outcomes#

Key Takeaways#

  • Multi-stage planning workflow (ccplan) is highly effective when dealing with complex integrations with conflicting issues — adversarial review stages identified critical git stash conflicts that could lead to data loss
  • Cascade feedback loops require multi-layer intervention: Fixing only window size adjustment issues (16ms→100ms) is insufficient; also solving data layer (load equality) and computation layer (timeout) issues
  • Indicated debugging requires careful lifecycle management: Each hypothesis needs independent metrics → execution → analysis → cleanup cycles to avoid cross-contamination and residual debugging code
  • Incomplete module refactoring creates hidden dependency hells: commands.rs still imports crate::parser, while new commands/calendar.rs imports crate::usage::parser — these two paths cannot coexist
  • Dynamic runtime detection adds failure modes without benefits when static configuration is sufficient: Tray window always uses position_near_tray, so bottom anchor should be hardcoded
  • Cross-platform window positioning in Tauri: Plugin abstraction sacrifices accuracy for portability; perfect pixel positioning (especially system tray alignment) requires platform-specific native APIs

Practical Learning Outcomes#

  • Comprehensive test suites can catch refactoring issues: 218 Rust tests and 191 front-end tests verified that fixes did not break existing behavior
  • Automated cleanup workflows must distinguish between tracked and untracked files: Git restore handles tracked files; untracked files (logs, temporary directories) require explicit rm/rmdir operations

Issues and Solutions#

Key Issues#

1. ccplan’s automatic execution phases (4-6) progressed too quickly due to shallow analysis, motivated by “continuous execution instructions”#

Solution: Added explicit depth requirements: Phase 4 expanded to 10 dimensions, with minimum detection thresholds set; Phase 5 became a 4-level analysis (chain/cross-cutting/flow/contract); Phase 6 required at least one detection and structured risk assessment.

Key Insight: Automatic execution phases needed quantitative depth requirements and minimum output thresholds. Qualitative instructions like “thoroughly” were not enough to motivate efficiency improvements.

2. When the WebSearch/WebFetch tools returned tags, the ccplan workflow was interrupted, causing the skill list to reload and losing phase context#

Solution: Added a tool call state preservation protocol, including a 3-step recovery mechanism: Recording the current phase before the tool call, re-anchoring to the ECL state field after the tool returned, ignoring system reminder metadata; added inline comments in phases 0/3/7 where searching was possible.

Key Insight: Complex multi-phase workflows required clear state preservation protocols, especially when calling external tools that return a lot of metadata. Implicit context continuity could not be relied upon.

3. Function regression issues during fixes - Claude Code forgot some features due to context compression, resulting in accidental deletion#

Solution: Implemented persistent ECL documentation with functional protection sections. Developed a secure guard-check.py CLI tool (with interactive confirmation of command execution) that implemented automatic generation for Phase 10 and added a pre-modification check protocol.

Key Insight: Memory and context are temporary, but disk data is permanent. Guards written to ECL files could remain unchanged during sessions and compression. Portability (carried with the skill directory) was better than specific project solutions.

4. Float ball drag failure - The startDragging() function was called in the setTimeout callback, resulting in loss of pointer event context required by Windows#

Solution: Manually capturing the pointer instead: Capturing the pointer and recording the starting position during onPointerDown events, checking a 180ms threshold during onPointerMove events, and calling move_float_ball_toIPC function; deciding whether to expand the window based on the threshold during onPointerUp events.

Key Insight: Windows requires that drag operations must be initiated within the original pointer context. setTimeout/async would break this rule.

5. Adjusting window size during CSS max-height transitions caused the bottom edge to lose taskbar alignment, due to multiple rapid ResizeObserver→IPC calls#

Solution: Changed the CSS transition for detail to ‘opacity 0.8s ease, max-height 0s linear 0.8s’ — first fading out opacity without changing layout, then folding the height immediately after the delay, triggering a single resize adjustment.

Key Insight: CSS transitions that affect layout size would trigger ResizeObserver every frame; delaying layout changes until after the animation was complete could prevent IPC call storms and maintain window atomic repositioning.

6. When the floating window approaches the edge, its expanded panel overflows the screen; Windows window resizing always takes the bottom as a reference, regardless of actual position.#

Solution: Added an EXPAND_MARGIN=8.0 constant to ensure that when expanding, it stays at least 8 pixels away from the edge; implemented dynamic anchor detection with a threshold of 5 pixels — using the top anchor when touching the top edge, and the bottom anchor when touching the bottom edge; added the SWP_NOCOPYBITS flag to prevent copying old client area content.

Key Insight: Expand/collapse states require different positioning strategies; anchors should be determined dynamically based on the current window position, not hardcoded; Windows’s default resizing behavior would retain old content, causing visual drift.

7. Due to the coexistence of both window systems (main window global CSS + floating window page), the floating window’s transparent frame has artifacts, and hover delays also cause visual problems.#

Solution: Removed global CSS reuse from the floating window page, added Windows-native region clipping functionality (SetWindowRgn), and rewritten the window geometry state machine using a unified rectangle calculation method to ensure that the bottom edge was always aligned with the work area’s bottom.

Key Insight: Coexistence of window systems leads to style and container stacking conflicts; complete separation is required, not patching.

8. Chart hover details coverage solution caused visual overlap between elements, violating user experience requirements.#

Solution: Used a block layout and coordinated window height animations through the onDetailToggle callback (following the existing AccordionToggleDetail pattern). Implemented handleChartDetailToggle functionality in App.svelte, and added observer suppression mechanisms.

Key Insight: When dealing with native window operations, passive observation should be suppressed during coordinated animations to prevent double triggering and visual artifacts.

9. The current cost calculation uses a local hardcoded pricing table, which does not match the results calculated by ccusage.#

Solution: Added a ccusage CLI adapter (ccusage.rs), implemented silent execution on Windows using CREATE_NO_WINDOW; implemented a priority-based mechanism (trying ccusage first, failing to fallback to parser); UsagePayload included usage_source and usage_warning fields, with clear warnings displayed on the frontend.

Key Insight: A transparent fallback mechanism is very important: Do not disguise the degraded result as the normal state; clearly inform users of the data source and limitations.

10. On non-MacOS platforms, since glassEffect is applied to all platforms without native blur support, the visual effect becomes too transparent.#

Solution: Added an isMacOS() check in applyGlass(), forcing the implementation of opacity effects on non-MacOS platforms; changed the default value to false and updated test snapshots.

Key Insight: Differences between platforms (macOS has native blur, Windows/Linux do not) result in different visual outcomes for the same configuration; platform detection and compensation are required at the application level.

General Issues#

11. The edit tool’s replace_all function needs to read the file first; batch path updates across multiple files failed, resulting in an “file not yet read” error.#

Solution: Used sed for cross-file batch replacements (crate::parser → crate::usage::parser for multiple.rs files).

Key Insight: For batch path updates across multiple files, shell tools like sed and grep perform better than individual tool calls under pre-read editing restrictions.

Learning Outcomes#

Key Learnings - Complex Multi-Stage Workflows Require Clear State Saving Protocols When Invoking External Tools That Return a Large Amount of Metadata Results. Implicit Context Continuity Cannot Be Relyed Upon. Tools Returning the Tag May Damage the Workflow’s State.#

  • The automatic execution phase of the workflow requires quantitative depth requirements (at least N discovery points, M layers of analysis, at least one probing). This prevents “fast but shallow” behavior. Qualitative instructions like “thoroughly” are not sufficient for efficiency incentives.
  • Persistence protection mechanisms (ECL feature_guard) and automated checks (guard-check.py) can prevent functionality from degrading between sessions. Memory and context are temporary, but disk is persistent. Testing is ideal, but protection mechanisms provide a lighter security net for incremental development.
  • Best practices for silently executing child processes in Windows GUI applications: Use std::os::windows::process::CommandExt to create flags and set CREATE_NO_WINDOW (0x0800_0000) to avoid console window pop-ups.
  • Control of visual anchors during Windows window size adjustments: The default behavior copies old client content to the upper left corner, causing visual drift. SWP_NOCOPYBITS needs to be used to disable copying and force a full redraw, while dynamically detecting _vertical_anchor is needed to fix the appropriate edges.
  • CSS transition effects that affect layout sizes trigger ResizeObserver every frame. Delaying layout changes until after the visual transition is completed (e.g., “opacity 0.8s ease, max-height 0s linear 0.8s”) can prevent IPC call storms and maintain the atomicity of window repositioning.
  • Combining waveform-based execution with verification checkpoints (cargo check, cargo test, vitest after each waveform) can detect issues early in large-scale refactoring, thereby avoiding cascading failures.
  • Using directory mtime for intelligent cache invalidation is more efficient than unconditional invalidation. Existing infrastructure (DirectoryStamp) can be used for low-cost change detection.
  • When dealing with native window operations, passive observation (ResizeObserver) must be suppressed during coordinated animations to prevent double triggering and visual artifacts.
  • A transparent fallback mechanism is essential: Do not disguise degraded results as normal ones. Inform users clearly through data model fields (source/warning) and UI warnings.
  • Tip engineering skills should decompose multi-intentions before intention extraction. In practical use, assuming a single intent is a common error pattern. First step: decomposing multi-intentions allows each trace to be processed independently.
  • Adopting a research-first approach and setting quality limits (e.g., >1k GitHub stars) yields better designs than isolated inventions. Leveraging proven solutions can reduce blind spots.
  • The Tauri v2 feature system requires clear permission declarations for all window APIs. Missing permissions result in silent failures (returning null), rather than visible errors. Requirements include: core:window:allow-outer-position, allow-scale-factor, and allow-current-monitor to implement dragging functionality.
  • Large file decomposition should maintain the single responsibility of each module while keeping test code within the same file. Using 987 lines of Query.rs is acceptable because most of it is test code.
  • A mixed tool strategy (parallel proxies for complex logic, sed/bash for batch string replacements, direct editing for targeted fixes) optimizes correctness and speed in refactoring within the code base.
  • Following existing code base patterns (e.g., AccordionToggleDetail callback mechanism) leads to maintainable solutions rather than introducing new communication paradigms.
  • Cross-platform UI consistency issues usually stem from differences in platform APIs (fuzzy effects on macOS versus no fuzzy effects on Windows/Linux). Application-level platform detection and compensation are needed.
  • Fixed UI behavior should not have runtime configuration dependencies. If the behavior is constant, the code should reflect this by removing conditional logic, improving clarity, and maintainability.

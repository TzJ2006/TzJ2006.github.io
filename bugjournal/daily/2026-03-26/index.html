Problems & Solutions#

Critical Issues#

1. ccplan auto-execution phases (4-6) progressed too quickly with shallow analysis, incentivized by ‘continuous execution mandate’#

Solution: Added explicit depth requirements: Phase 4 expanded to 10 dimensions with minimum findings threshold; Phase 5 became 4-layer analysis (chain/cross-cutting/data flow/contracts); Phase 6 requires minimum 1 probe with structured risk assessment

Key Insight: Auto-execution phases need quantitative depth requirements and minimum output thresholds; qualitative instructions like ‘be thorough’ are insufficient against efficiency incentives

2. ccplan workflow breaks when WebSearch/WebFetch tools return tags that reload skill list and lose phase context#

Solution: Added Tool Invocation State Preservation protocol with 3-step recovery: record current phase before tool call, re-anchor to ECL status field after tool return, ignore system-reminder metadata; added inline notes in Phases 0/3/7 where search is likely

Key Insight: Complex multi-phase workflows need explicit state preservation protocols when invoking external tools that return metadata-heavy results; can’t rely on implicit context continuity

3. Feature regression during bug fixes - Claude Code forgets features due to context compression, leading to accidental removal#

Solution: Implemented Feature Guard Protocol with persistent ECL documents containing feature_guard sections, guard-check.py CLI tool with security hardening (interactive confirmation for command execution), Phase 10 auto-generation, and pre-modification check protocol

Key Insight: Memory and context are ephemeral, but disk is persistent. Guards written to ECL files survive across sessions and compactions. Portability (traveling with skill directory) is superior to project-specific solutions.

4. Float ball drag failed because startDragging() was called in setTimeout callback, losing pointer event context required by Windows#

Solution: Replaced with manual pointer capture: onPointerDown captures pointer and records start position, onPointerMove checks 180ms threshold and invokes move_float_ball_to IPC, onPointerUp toggles expansion if under threshold

Key Insight: Windows requires drag operations to be initiated within the original pointer event context; setTimeout/async breaks this contract

5. Window resizing during CSS max-height transitions caused bottom edge to lose taskbar alignment due to multiple rapid ResizeObserver→IPC calls#

Solution: Changed .detail CSS transition to ‘opacity 0.8s ease, max-height 0s linear 0.8s’ - opacity fades first without layout change, then max-height collapses instantly after delay, triggering single resize

Key Insight: CSS transitions that affect layout dimensions trigger ResizeObserver on every frame; delaying layout change until animation completes prevents IPC call storms and preserves atomic window repositioning

6. Float ball expanded panel overflows screen when near edge; Windows window resize always anchored to bottom regardless of actual position#

Solution: Added EXPAND_MARGIN=8.0 constant to ensure ≥8px distance from edges when expanded; implemented dynamic anchor detection with 5px threshold - top-anchored when touching top edge, bottom-anchored when touching bottom; added SWP_NOCOPYBITS flag to prevent old client area content from being copied back

Key Insight: Expand and collapse states require different positioning strategies; anchor point should be determined dynamically based on current window position rather than hardcoded; Windows default resize behavior preserves old content causing visual drift

7. Float ball transparent box artifacts and hover lag caused by coexisting window systems (main window global CSS + float ball page)#

Solution: Removed global CSS reuse from float-ball page, added Windows native region clipping (SetWindowRgn), rewrote window geometry state machine with unified rect calculation ensuring bottom edge always aligns with work area bottom

Key Insight: Coexisting window systems caused style and container stacking conflicts; required complete separation rather than patchwork fixes

8. Chart hover detail overlay solution caused visual overlap between elements, violating UX requirements#

Solution: Pivoted to block layout with explicit window height animation coordinated via onDetailToggle callback (following existing AccordionToggleDetail pattern), implemented handleChartDetailToggle in App.svelte with observer suppression mechanism

Key Insight: When native window operations are involved, passive observation (ResizeObserver) must be suppressed during coordinated animations to prevent double-triggering and visual artifacts

9. Current cost calculation uses local hardcoded pricing tables, inconsistent with ccusage official calculations#

Solution: Added ccusage CLI adapter (ccusage.rs) with Windows CREATE_NO_WINDOW silent execution, implemented priority-based mechanism (try ccusage first, fallback to parser on failure), UsagePayload includes usage_source and usage_warning fields, frontend displays clear warnings

Key Insight: Transparent fallback mechanisms are essential: don’t disguise degraded results as normal; explicitly inform users of data source and limitations

10. Non-macOS platforms too transparent because glassEffect applied to all platforms without native blur support#

Solution: Added isMacOS() check in applyGlass() to force opaque for non-macOS, changed default to false, updated test snapshots

Key Insight: Platform differences (macOS has native blur, Windows/Linux don’t) cause same config to produce different visual results; requires application-layer platform detection and compensation

General Issues#

11. Edit tool’s replace_all requires reading files first; bulk path updates across many files failed with ‘File has not been read yet’ errors#

Solution: Switched to sed for cross-file bulk replacements (crate::parser → crate::usage::parser across multiple .rs files)

Learnings#

Key Learnings#

  • Complex multi-phase workflows need explicit state preservation protocols when invoking external tools that return metadata-heavy results; can’t rely on implicit context continuity. Tools returning tags can break workflow state.
  • Auto-execution phases in workflows require quantitative depth requirements (minimum N findings, M layers of analysis, at least 1 probe) to prevent ‘fast but shallow’ behavior; qualitative instructions like ‘be thorough’ are insufficient against efficiency incentives
  • Persistent guards (ECL feature_guard) + automated checking (guard-check.py) prevent feature regression across sessions. Memory and context are ephemeral, but disk is persistent. Tests are ideal but guards provide lighter-weight safety net for incremental development.
  • Windows GUI应用静默执行子进程的最佳实践: use std::os::windows::process::CommandExt creation_flags with CREATE_NO_WINDOW (0x0800_0000) to avoid console window popup
  • Windows窗口resize的视觉锚点控制: default behavior copies old client content by top-left causing visual drift; need SWP_NOCOPYBITS to disable copy and force full redraw, plus dynamic detect_vertical_anchor to fix appropriate edge
  • CSS transitions that affect layout dimensions trigger ResizeObserver on every frame; delaying layout change until after visual transition completes (e.g., ‘opacity 0.8s ease, max-height 0s linear 0.8s’) prevents IPC call storms and preserves atomic window repositioning
  • Wave-based execution with verification checkpoints (cargo check, cargo test, vitest after each wave) prevents cascading failures in large refactors by catching issues early
  • Smart cache invalidation using directory mtime is more efficient than unconditional invalidation; existing infrastructure (DirectoryStamp) can be leveraged for low-overhead change detection
  • When native window operations are involved, passive observation (ResizeObserver) must be suppressed during coordinated animations to prevent double-triggering and visual artifacts
  • Transparent fallback mechanisms are essential: don’t disguise degraded results as normal; explicitly inform users via data model fields (source/warning) and UI warnings
  • Prompt engineering skills should decompose multi-intent inputs before intent extraction; single-intent assumption is a common failure mode in real usage. Step 1: Multi-Intent Decomposition enables per-track independent processing.
  • Research-first approach with quality constraints (e.g., >1k GitHub stars) produces better designs than isolated invention; borrowing from battle-tested solutions reduces blind spots
  • Tauri v2 capability system requires explicit permission declarations for all window APIs; missing permissions cause silent failures (null returns) rather than visible errors. Needed: core:window:allow-outer-position, allow-scale-factor, allow-current-monitor for drag functionality.
  • Large file decomposition should maintain single responsibility per module while keeping test code in same file; usage_query.rs at 987 lines is acceptable because most are tests
  • Hybrid tooling strategy (parallel agents for complex logic, sed/bash for bulk string replacements, direct Edit for targeted fixes) optimizes for both correctness and speed in codebase-wide refactors
  • Following existing codebase patterns (e.g., AccordionToggleDetail callback mechanism) leads to more maintainable solutions than introducing new communication paradigms
  • Cross-platform UI consistency issues often stem from platform API differences (macOS blur vs Windows/Linux no blur); requires application-layer platform detection and compensation
  • Fixed UI behavior should not have runtime configuration dependencies. If behavior is constant, code should reflect that by removing conditional logic, improving clarity and maintainability.

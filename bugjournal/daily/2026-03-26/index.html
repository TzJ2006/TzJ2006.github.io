Key Limitations of Artificial Intelligence### General Limitations#

  • During the intention calibration phase in stage 0 of the ccplan, multiple uses of AskUserQuestion were required to eliminate ambiguity. It was not possible to infer the user’s exact intentions directly from the context.
  • There was an over-reliance on planning documents, while the user’s actual concerns (such as “no terminal pop-up windows”) were simpler technical details. When users needed direct constraints on verification, time was spent explaining the rationale behind MCP.

Lessons Learned#

Key Lessons#

  • Complex multi-stage workflows require a clear state preservation protocol when invoking external tools that return a large amount of metadata. Implicit context continuity should not be relied upon. Tools that return tags may disrupt the state of the workflow.
  • The automatic execution phase of the workflow requires a certain level of quantitative depth (at least N discovery points, M layers of analysis, at least one probe) to prevent “fast but shallow” behavior. Qualitative instructions like “thoroughly” are insufficient for improving efficiency.
  • Persistence protection mechanisms (ECL feature protection mechanism) and automated checks (guard-check.py) can prevent feature degradation between different sessions. Memory and context are temporary, but disk is persistent. Testing is ideal, but protection mechanisms provide a lighter security net for incremental development.
  • In Windows GUI applications, it is best practice to set CREATE_NO_WINDOW (0x0800_0000) when creating flags using std::os::windows::process::CommandExt to avoid console window pop-ups.
  • Visual anchors for controlling window size: The default behavior copies the upper left corner content of the old client, causing visual drift. SWP_NOCOPYBITS needs to be used to disable copying and force a full repaint. Dynamic detection of vertical anchors is also required to fix edge issues.
  • CSS transition effects that affect layout dimensions trigger ResizeObserver every frame. Delaying the layout change until after the visual transition is completed (e.g., “opacity 0.8s ease, max-height 0s linear 0.8s”) prevents IPC call storms and maintains the atomic repositioning of window positions.
  • Wave-based execution methods, with validation checkpoints after each wave, allow early detection of problems, thereby avoiding cascading failures during large-scale refactoring.
  • Using directory file modification times for intelligent cache invalidation is more efficient than unconditional invalidation. Existing infrastructure (DirectoryStamp) can be utilized for low-cost change detection.
  • When dealing with native window operations, passive observation (ResizeObserver) must be suppressed during coordinated animations to prevent double triggering and visual artifacts.
  • A transparent backtracking mechanism is crucial: Do not disguise degraded results as normal ones. Clearly inform users through data model fields and UI warnings.
  • Hint engineering skills should decompose multi-intentions before intent extraction. Assuming a single intent is a common error pattern in real use cases. The first step is to decompose multi-intentions so that each tracking can be processed independently.
  • Adopting a research-first approach and setting quality constraints (e.g., >1k GitHub stars) yields better designs compared to isolated inventions. Leveraging proven solutions can reduce blind spots.
  • Tauri v2 functional system requirements require explicit permission declarations for all window APIs. Missing permissions result in silent failures (returning null), rather than visible errors. Adding core: window: allow-outer-position, allow-scale-factor, allow-current-monitor is necessary to implement drag-and-drop functionality.
  • The decomposition of large files should maintain the single responsibility principle for each module. Test code should be placed in the same file. Lines 987 of usage_query.rs are acceptable because most of it is test code.
  • A hybrid tool strategy (parallel proxies for complex logic, sed/bash for batch string replacements, direct Edit for targeted repairs) can optimize both correctness and speed during code repository-level refactoring.
  • Following existing code repository patterns (e.g., AccordionToggleDetail callback mechanism) makes the solution easier to maintain than introducing new communication paradigms.
  • Cross-platform UI consistency issues often stem from differences in platform APIs (fuzzy effects on MacOS versus no fuzzy effects on Windows/Linux). Platform detection and compensation need to be performed at the application layer.
  • Fixed UI behaviors should not rely on runtime configurations. If the behavior is constant, the code should reflect this by removing conditional logic, thereby improving code clarity and maintainability.

Problems and Solutions#

Key Issues#

1. Tauri decorations:false on Windows 11 cannot completely eliminate window borders (there are two sources: CSS box-shadow and Windows DWM system details)#

Solution: Remove the box-shadow inset from app.css; call DwmSetWindowAttribute(hwnd, DWMWA_BORDER_COLOR=34, &DWMWA_COLOR_NONE=0xFFFFFFFE) in window.rs to eliminate the DWM system border.

Key Insight: Tauri window borders have two independent sources that need to be addressed simultaneously; DWMWA_COLOR_NONE value is 0xFFFFFFFE instead of 0.

2. Tauri v2 capability system defaults to rejecting all undeclared APIs: floating ball windows’ outerPosition()/scaleFactor() calls silently fail, making dragging completely ineffective; floating balls in multi-window applications are not declared in capabilities#

Solution: Add three missing permissions (allow-outer-position/allow-scale-factor/allow-current-monitor) to capabilities/default.json, and add float-ball to the windows array.Key Insights: The capabilities of Tauri v2 are on a whitelist, and any window API must be explicitly declared. Silent failures without error messages are the most difficult to troubleshoot.

3. When startDragging() is called in a setTimeout callback or async context, it fails due to Win32 restrictions, resulting in ineffective floating ball dragging.#

Solution: Use Pointer Capture for manual dragging: onPointerDown captures the starting point of the pointer, and onPointerMove, if the movement exceeds a threshold of 5px, it enters dragging mode. Move the window through move_float_ball_toIPC, and if the onPointerUp does not exceed this threshold, it is considered a click.

Key Insight: startDragging() must be called within synchronous pointer event handlers. Any asynchronous or delayed calls will cause Win32 to reject it. Pointer Capture allows for precise distinction between dragging and clicking.

4. When the CSS max-height animation transition ends during the CSS resizeObserver, the native flicker occurs due to continuous window repositioning. The overlay fix eliminates the flicker but introduces content occlusion.#

Solution: Use a layered CSS transition delay: opacity fades out after 1s (without triggering layout changes and without ResizeObserver); then instantly reset max-height to zero after 0.8-1s (exactly triggering one resize). Replace passive observers with explicit callbacks on onDetailToggle and the suppressResizeObserver flag.

Key Insight: Decoupling visual animations (opacity) from structural animations (max-height) is key. To fix visual bugs, it’s necessary to verify both the intended solution and side effects (the overlay fixes flicker but disrupts layout semantics).

5. ccusage is marked as status: verified in the planning document, leading users to believe it has been implemented in the code. However, the actual code still uses a hard-coded pricing table in pricing.rs.#

Solution: Confirm through code review that it has not been implemented and inform users clearly. Choose silent CLI access (instead of an ongoing MCP process) and define fallbacks based on scenarios: for ccusage, implement day views and Codex 5h granularity to backtrack to the old parser.

Key Insight: The status field in the planning document does not indicate that the code has been implemented. It is necessary to directly verify it by reading the source code. Third-party tools may not cover all granularities, so it’s important to determine the fallback strategy based on scenarios when integrating them.

6. AI misunderstands the issue of cross-sample queries: It generalizes comparisons at the query protocol level to analyze differences in training architectures (GIN vs GCN, loss functions, batch effects, etc.), which is seriously off track from what users actually care about.#

Solution: Users clearly correct the direction, and AI focuses on the query process. It identifies four differences in query protocols: query granularity (entire layer centroid vs single spot K-hop subgraph), candidate representation (spot emb vs niche emb), niche type definition (single-layer label vs cross-layer boundary types), and evaluation metrics (Spearman vs PCC+NCJS).

Key Insight: There are multiple levels of ‘method differences’ in the same paper (training architecture/推理协议/evaluation system). AI tends to start from the most macroscopic level. Researchers usually have a clear focus, and AI should confirm the analysis granularity before responding for the first time.

7. The Error Recovery Benchmark has two problems: There are ~60 lines of repetitive object detection logic in Drop skills; CLAUDE.md has long-term documentation drift after the classification system was重构 (29 subtypes/D2 degree historical errors not updated).#

Solution: Extract 6 shared helper methods from BaseErrorSkill and call them directly for each subclass to eliminate duplication. Verify the entire CLAUDE.md file and correct it to 26 subtypes/D0+D1 levels through code verification.

Key Insight: Abstract base classes should provide both shared tool methods and definitions. After code refactoring, document verification should be carried out immediately. The constants of the classification system should be based on authoritative sources (error_taxonomy_v5.py) rather than being manually maintained in the documentation.

8. There is a systematic tendency to ‘skip over’ in ccplan Phases 4-6: Qualitative descriptions (‘careful analysis’) cannot enforce analysis depth; Phase 0 does not consider scenarios with multiple intentions, and single-intention extraction fails under complex prompts. The system reminder tag returned by WebSearch can interrupt the workflow context.#

Solution: Increase the minimum discovery threshold in Phases 4-6 to max(3, item_count/2) and remove the skip all option. Add a multi-intention decomposition step in Phase 0 Step 1 (coupled/related/independent classification + parallel tracks). Add a section on preserving tool invocation states (externalizing states + inline reminders for three high-risk phases).

Key Insight: Quantitative constraints (at least N discoveries) are better than qualitative descriptions (‘careful analysis’). AI will find reasons to skip qualitative requirements. The ECL document serves as an externalized state anchor, which is more effective than relying on context memory in preventing tool calls interrupting the workflow.

9. In the initial version of guard-check.py, there are two critical security issues: Reading command fields from YAML and directly passing them to subprocess.run(shell=True) (shell injection); bare except Exception silences all exceptions, making the guard ineffective without a prompt.#

Solution: Confirm interactively before execution with a yes/no option (skip directly in non-interactive mode). Only catch the expected json.JSONDecodeError; write other exceptions to stderr for visibility.

Key Insight: If the security hook fails silently, it’s equivalent to having no hook at all. Even if YAML is a trusted source, there must be a confirmation gate when executing in bulk. Failure paths must leave observable traces.

10. When ccplan Phase 4-6 resize, due to the default behavior of Win32 SetWindowPos, old client content is copied back in the upper left corner, causing visual position drift.#

Solution: Add the SWP_NOCOPYBITS flag to the SetWindowPos call to prevent the copy of old content. Also introduce detect_vertical_anchor to distinguish between top and bottom anchors based on a threshold of 5px.

Key Insight: The Win32 API (InvalidateRect/RedrawWindow, etc.) may not have a one-to-one binding in the Windows crate. It’s necessary to use an equivalent but more conservative path (SWP_NOCOPYBITS).

General Issues#

11. HPC environment engineering challenges: gpu-common partition is fully loaded, git-lfs is missing, causing failure to clone large files; UCE model file download from Figshare fails (0 bytes); scGPT’s official API cannot be used due to pyarrow compatibility issues; Geneformer V2 CUDA OOM occurs on a 48GB single card.#

Solution: Filter out available nodes for scavenger-gpu (majoroslab RTX 6000 Ada); fix git-lfs to allow cloning large files with conda install; UCE blocking requires proxy or scp pre-transmission; directly implement the underlying reasoning logic to bypass package compatibility issues; reduce the batch size of Geneformer to 10 to solve OOM issues.

Key Insight: HPC computing node system tools are often missing, and it’s necessary to supplement them through conda user-level installation. The availability of model checkpoints should be verified as a risk point during planning. Priority should be given to bypassing package compatibility issues rather than forcing downgrades.

12. The Pi0.5 evaluation script failed in three rounds of debugging: ① It used Python from openpi .venv, but the openpi package was linked to an incorrect user copy; ② PYTHONPATH lacked the full project directory prefix; ③ MUJOCO_EGL_DEVICE_ID specified GPU was not in CUDA_VISIBLE_DEVICES.#

Solution: Use the openpi05 conda environment; correct PYTHONPATH to absolute paths; modify the eval process to include both server GPU and EGL GPU in CUDA_VISIBLE_DEVICES.

Key Insight: On a shared file system, packages with the same name for multiple users must specify a conda environment explicitly. The GPU used for MuJoCo EGL rendering must be a subset of devices visible to CUDA_VISIBLE_DEVICES. This is a rigid requirement of Robosuite.

AI Limitations#

Important Limitations#

  • Lack of depth guarantee in skill design: There is a systematic tendency to ‘skip over’ in the designs of Phases 4-6. This was discovered and fixed only after users reported that ‘progress too quickly’. Phase 0 did not actively consider scenarios with multiple intentions until explicitly indicated by users.
  • Insufficient security code generation: When writing guard-check.py, the risk of shell injection was not considered. It took a professional code review sub-agent to discover this issue. This indicates that AI lacks security awareness when generating code involving external commands.
  • Lack of proactive comparison between planning documents and actual code: It was not actively reported that ccusage is only marked as verified in the planning document but has not been implemented in the code. This was revealed only after users asked questions. This indicates a lack of initiative in comparing planning status with actual code.
  • Lack of product intuition: The floating ball product positioning requires more than three rounds of clarification to determine whether it is ‘a secondary entrance rather than the main channel’. The floating ball clarification options did not cover the behavior that users actually want, such as ‘吸附缩入边框’, forcing users to manually add None of the above.
  • Default analysis granularity for tools invocation: The system reminder tag returned by WebSearch can interrupt the ccplan workflow. AI only discovered and fixed this issue after users reported it. This indicates a lack of pretection against context amnesia risk when handling tool invocation scenarios.
  • Incomplete evaluation of UI side effects: When recommending an overlay fix to eliminate flicker, it did not actively evaluate scenarios where content would be occluded. When implementing the horizontal layout of the floating ball, it did not anticipate the need to leave the screen edge after expansion. This was only discovered after users provided feedback after actual use.

General Limitations#

  • Omissions in HPC planning: The network accessibility of HPC nodes to Figshare was not pre-verified, resulting in UCE being blocked. Verifying the availability of model checkpoints should be a necessary item during planning.
  • Bias in judgment regarding research scenarios: It is recommended to use smaller models (UCE 4-layer, Geneformer V1) to quickly run through the process. However, researchers require ensuring academic comparability according to paper/HuggingFace default settings. AI prioritizes engineering feasibility over experimental reproducibility standards.
  • Incomplete understanding of Tauri/Win32 API details: decorations:false cannot completely eliminate Windows borders (it requires calling DwmSetWindowAttribute separately). The way Win32 return values is different in different versions of the windows crate (comparing SetWindowRgn return values was incorrect). It took multiple failed API import attempts to find an available path.

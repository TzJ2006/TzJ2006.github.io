Problems and Solutions#

Key Problems#

1. BasicContrastive.py Contrast Loss Bottleneck: Python loop iterations per node reached 3000-4000 times, with tensor cloning operations performed during each iteration.#

Solution: Vectorized loss calculations using batch logsumexp operations to eliminate loops; masking self-similarity through fill_diagonal_(-inf); using Boolean masks for positive neighbor filtering.

Key Insight: Even mature deep learning code may contain non-vectorized loops during the prototype phase; analysis and vectorization can improve performance by 10-50 times without changing the algorithm.

2. Crossformer LIPM Version (MSE=9, MAE=1.3) has a performance ratio of 18 times lower than the main version, despite having a patch mechanism designed for irregular battery data.#

Solution: Diagnosed architectural mismatch issues: Crossformer’s DSW embedding assumption of continuous time steps combined with position encoding, but LIPM patch methods produced irregular intervals that filled in zeros, disrupting time continuity, wasting model capacity, and losing time interval information. It is recommended to use interpolation to uniform grids, time window aggregation, or irregular time series models (GRU-D, BRITS, NAOMI).

Key Insight: For sequence models with position encoding, zero filling is not neutral—it actually misleads the model to learn “zero patterns” rather than temporal dependencies. Architectural assumptions must match data preprocessing; forcing irregular data into patch+zero format will destroy semantic meaning.

3. 12 error-prone skill files duplicated over 150 lines: Frame availability checks (22 times), direction normalization (8 times), list(dict.keys())[0] pattern (14 times).#

Solution: Extracted 3 helper methods into BaseErrorSkill: check_remaining_frames, normalize_direction, random_direction; replaced list(keys())[0] with next(iter()); added 10 unit tests for the new helper functions.

Key Insight: Code review found that “specific skill thresholds” (e.g., e08 using 30 frames while the standard is 20) were actually errors; duplicate data hid errors because the fixes were not propagated.

4. PairedDataset expects {clean_ep_idx: [{aug_ep, overlap}]} format, but episode_mapping.json uses {task_name: {bddl_idx: [{aug_demo_idx, orig_ep_idx}]} format; loss of source file/demonstration index association during HDF5→LeRobot conversion.#

Solution: Modified the conversion script to record conversion_index.json = [{lerobot_ep, source_file, source_demo_idx, length}]; created build_paired_mapping.py that parses clean/aug conversion indices, uses orig_ep_idx from episode_mapping as a bridge, aggregates all bddl_idx, and outputs a format compatible with PairedDataset, with up to 10 aug variants per clean episode.

Key Insight: Data sources must be consistent throughout every conversion step; three-way association requires explicit recording of bridging keys during generation; 1-to-N mapping requires aggregating various variant indices.

5. Users need to specify the boundaries for recovery demonstration data: Does training include an error injection process, or only the recovery part?#

Solution: Confirmed through code archaeology that recovery demonstrations only recorded remote operation behavior after injection; actions_list/states_list started after scene.load() or replay completed; verified with sample NPZ checks; the first state of the recovery demonstration equaled the post_sim_state of the scenario; training conversion filter conditions were set to success=true, and only recovery actions were used.

Key Insight: Error recovery datasets deliberately excluded actions that caused errors, preventing the model from learning failure patterns; only corrective human demonstrations were used for imitation learning.

6. Previous discussions concluded that 0% enhancement failures were due to “deformation unable to adapt,” but code checks found certain sub-types had dominant NO_WARP parts (position_error_D1: 150/192 steps in correct_position), and these parts did not undergo deformation at all.#

Solution: Corrected the diagnostic results: failures were not due to deformation limitations, but rather from incomplete deformation coverage and fundamental architectural mismatch between action-replay (current) and Waypoint execution methods.

Key Insight: Making conclusions without checking the execution path can lead to attribution errors in root cause analysis; repository evidence directly refutes the cited reasons: hard-coded NO_WARP labels mean the dominant part is still open-loop.

7. Current enhancers use source action replay and deformation targets, while MimicGen actually uses source target pose extraction + waypoint trajectory generation + interpolation execution—fundamental architectural mismatch.#

Solution: Implemented faithful transfer: added target_poses records to the collection, created object-pose tracking compatible with DatagenInfo, specified object references for each subtask in the configuration, interpolated waypoints through MimicGen’s WaypointTrajectory, and implemented target pose-to-action execution in the enhancer.

Key Insight: Surface similarity (“both using object poses”) may mask deep structural differences; MimicGen’s success depends on closed-loop correction for each subtask, not just better deformation of open-loop actions.

8. 7 HDF5 files were opened without a context manager; exceptions during processing could lead to resource leaks.#

Solution: Wrapped all h5py.File() calls in 4 script files with ‘with’ statements (_merge_rounds.py, _check_valid_source.py, parallel_mimicgen_generate.py, augment_success_demos.py).

Key Insight: Resource leaks in HPC batch jobs can accumulate over long runs, leading to mysterious OOM crashes; Python’s context manager can prevent such errors.

10. MimicGen generation reports errors (398 successful demonstrations), which is due to log parsing errors that count all SUCCESS strings in full_run_200attempts.log, regardless of task association.#

Solution: Verified the actual results of HDF5 files and generation_report.json, corrected the total number to 739/1365 (success rate 54.1%), identified 626 specific failed scenarios using regular expressions, and developed a manual collection plan.

Key Insight: Always verify summary metrics against real files; log parsing errors can lead to statistical errors when reading data ranges incorrectly in backtracking logic.

11. AI misinterpreted “uniform sampling” as balance of sample distributions (per battery/per cycle), rather than uniformity of time intervals.Solution: The user specified that equal time intervals should be maintained between consecutive measurements. The AI turned to using scipy.interpolate for time-based resampling, analyzed the chargetime distribution (35.83% at intervals of 1, with others being irregular), and generated a uniform grid dataset through linear interpolation.#

Key Insight: Domain-specific terms require context support: “uniform sampling” generally indicates temporal regularity in a时间序列 context, rather than balance in data distribution.

12. The existing recovery demonstration lacks the target_poses field required by the new enhancer.#

Solution: The backfill_recovery_target_poses.py script was created. This script replayes each saved demonstration from the recorded initial state and reconstructs the controller’s target pose through env_wrapper.action_to_target_pose(). This allows reuse of the existing demonstration corpus without the need to recall it.

Key Insight: When execution is deterministic, data format migration can be non-destructive: saving the state + recording the actions equals reconstructing the missing fields.

13. Bash commands in git* will match all Git commands, including destructive operations like git clean -f, which poses a security risk.#

Solution: The user chose a lax strategy for exploratory environments and accepted this risk. The AI suggested splitting the commands into specific secure subcommands, but the user rejected this suggestion.

Key Insight: Permission configuration requires a balance between convenience and security. Exploratory environments can be more lenient.

14. The first Ollama major patch failed due to Windows command length limits (CreateProcessAsUserW failed: 206).#

Solution: Splitting a major patch into multiple minor patches was implemented: First, the shared module was deployed, and then the callers were migrated separately.

Key Insight: There are hard limits on the length of Windows command arguments. Major patches need to be submitted in batches.

General Issues#

15. The website translation workflow needs to handle 74 files distributed across directories. It is necessary to keep the original Chinese files and generate English versions.#

Solution: translate_site_batch.py was used, combined with state tracking, to translate only the changed files step by step. The language direction was switched (Chinese source files → .zh.md, English output → .md), making English the primary language. Key Insight: For bilingual websites where English is the primary language but the content originates from Chinese, changing the file naming convention can eliminate the need to update all internal links.

16. When everything-claude-code locally deletes 18 skill files, modifications or deletions may occur during the merge with the remote version.#

Solution: Using git checkout –ours was used to retain the remote version (third-party repository). The same strategy was applied to package.json/package-lock.json. The merge was successfully completed.

Key Insight: For third-party repositories, it is better to keep up-to-date upstream versions rather than making local modifications.

17. In the zh_path function name in common/bilingual.py, there is a conflict with local variables, leading to an UnboundLocalError error.#

Solution: The imported zh_path was renamed translation_zh_path to avoid naming conflicts.

Key Insight: In Python, assignment within functions makes variables local variables, even when importing from the module level.

18. Conda runs cause temporary file conflicts when executed in parallel.#

Solution: Executing in serial was chosen instead of using conda run. Using system Python instead of conda run provides greater stability for scripts that do not depend on Condria dependencies.

19. The clock times differ after each restart on both Windows and Ubuntu systems. Windows regards RTC as local time, while Ubuntu considers it to be UTC time.#

Solution: It was found that Ubuntu is configured to use RTC with the local TZ. It was suggested to restore Ubuntu to using UTC (timedatectl set-local-rtc 0), and set Windows to use UTC through the registry (RealTimeIsUniversal=1).

Key Insight: The correct solution is to have both operating systems use UTC as RTC, although Windows defaults to using local time. Using local RTC on Linux leads to daylight saving issues.

20. An attempt was made to create a project-wide.codex/config.toml file, but.Codex is a read-only Lustre file system mount point, which prevents mkdir/mv operations.#

Solution: The documentation policy was changed to AGENTS.md in the repository root directory. This method allows for global configuration inheritance and propagation through Codex’s project documentation to all subdirectories, eliminating the need to modify the file system.

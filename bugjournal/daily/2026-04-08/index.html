Problems & Solutions#

Critical Issues#

1. BasicContrastive.py contrastive loss bottleneck: per-node Python loop over 3000-4000 spots with tensor cloning each iteration#

Solution: Vectorized loss computation using batch logsumexp operations eliminating the loop; masked self-similarity with fill_diagonal_(-inf); used boolean masks for positive neighbor filtering

Key Insight: Even mature deep learning code often contains un-vectorized loops from prototyping phase; profiling + vectorization can yield 10-50x speedups with no algorithmic changes

2. Crossformer LIPM version (MSE=9, MAE=1.3) performs 18x worse than master version (MSE≈0.5) despite patch mechanism designed for irregular battery data#

Solution: Diagnosed fundamental architecture mismatch: Crossformer’s DSW_embedding assumes continuous seg_len time steps with positional encoding, but LIPM patch method creates irregular gaps filled with zeros that break temporal continuity, waste model capacity, and lose time interval information. Recommended: interpolation to uniform grid, time window aggregation, or irregular time-series models (GRU-D, BRITS, NAOMI)

Key Insight: Zero-padding is NOT neutral for sequence models with positional encodings - it actively misleads model to learn ‘zero patterns’ instead of temporal dependencies. Architecture assumptions must match data preprocessing; forcing irregular data into patch+zero format breaks semantic meaning

3. 12 Error Skill files duplicating 150+ lines: frame availability checks (22 occurrences), direction normalization (8 occurrences), list(dict.keys())[0] pattern (14 occurrences)#

Solution: Extracted 3 helper methods to BaseErrorSkill (check_remaining_frames, normalize_direction, random_direction); replaced list(keys())[0] with next(iter()); added 10 unit tests for new helpers

Key Insight: Code review revealed ‘skill-specific thresholds’ (e.g., e08 using 30 frames vs. standard 20) were actually bugs; duplication hides bugs because fixes aren’t propagated

4. PairedDataset expects {clean_ep_idx: [{aug_ep, overlap}]} but episode_mapping.json uses {task_name: {bddl_idx: [{aug_demo_idx, orig_ep_idx}]}} format; HDF5→LeRobot conversion loses source file/demo index association#

Solution: Modified conversion scripts to record conversion_index.json = [{lerobot_ep, source_file, source_demo_idx, length}]; created build_paired_mapping.py that parses clean/aug conversion indices, uses episode_mapping’s orig_ep_idx as bridge, aggregates across all bddl_idx, outputs PairedDataset-compatible format with up to 10 aug variants per clean episode

Key Insight: Data provenance must be tracked through every conversion step; three-way association requires explicit bridge keys recorded at generation time; 1-to-N mappings need aggregation across variant indices

5. User needed clarification on recovery demo data boundaries: whether training includes error injection process or only recovery segment#

Solution: Through code archaeology, confirmed recovery demos record only post-injection teleop actions; actions_list/states_list begin after scene.load() or replay completes; verified via sample NPZ inspection that recovery demo’s first state equals scene’s post_sim_state; training conversion filters for success=true and uses only recovery actions

Key Insight: Error recovery datasets intentionally exclude error-causing actions to prevent models from learning failure modes; only corrective human demonstrations are used for imitation learning

6. Previous conversation concluded 0% augmentation failures were due to ‘warping can’t adapt’, but code inspection revealed some subtypes had dominant NO_WARP segments (position_error_D1: 150/192 steps in correct_position) that aren’t warped at all#

Solution: Corrected diagnosis: failures aren’t due to warping limitations but due to incomplete warping coverage and fundamental architecture mismatch between action-replay (current) vs target-pose waypoint execution (MimicGen standard)

Key Insight: Attribution errors in root cause analysis persist when conclusions are drawn without inspecting execution paths; repo evidence directly contradicted stated cause: hardcoded NO_WARP labels meant dominant segments were still open-loop

7. Current augmenter used source-action replay with warped targets, while MimicGen actually uses source target-pose extraction + per-subtask waypoint trajectory generation + interpolated execution - fundamental architectural mismatch#

Solution: Implemented faithful port: added target_poses recording in collection, created DatagenInfo-compatible object pose tracking, per-subtask object-ref specs in config, waypoint interpolation via MimicGen’s WaypointTrajectory, target-pose-to-action execution in augmenter

Key Insight: Surface-level similarity (‘both use object poses’) can mask deep architectural differences; MimicGen’s success depends on waypoint-level closed-loop correction per subtask, not just better warping of open-loop actions

8. 7 HDF5 file handles opened without context managers; exceptions during processing cause resource leaks#

Solution: Wrapped all h5py.File() calls with ‘with’ statements in 4 script files (_merge_rounds.py, _check_valid_source.py, parallel_mimicgen_generate.py, augment_success_demos.py)

Key Insight: Resource leaks in HPC batch jobs accumulate over long runs causing mysterious OOM kills; Python’s context managers prevent this entire class of bugs

9. Ollama service dropped connection during translation (chunk 10/26)#

Learnings#

Key Learnings#

  • Vectorization of tensor operations eliminates Python loop overhead and enables GPU parallelism: BasicContrastive contrastive loss refactor from per-node loop to batch operations yielded 10-50x speedup with identical mathematical behavior
  • Zero-padding is NOT neutral for sequence models with positional encodings where it actively breaks semantic meaning: Crossformer’s DSW_embedding assumes continuous seg_len time steps; zero-padded patches with irregular gaps mislead model to learn ‘zero patterns’ instead of temporal dependencies
  • Code duplication hides bugs because fixes don’t propagate: 12 Error Skill files had inconsistent frame thresholds (e08: 30 vs. standard 20) that code review revealed as unintentional; extracting to shared helpers forces consistency
  • Data provenance tracking must be explicit through every conversion pipeline step: HDF5→LeRobot conversion loses source_file+demo_idx mapping; solution is recording conversion_index.json during conversion to enable bridge-key linkage with generation-time metadata
  • For irregular time-series with non-uniform sampling intervals: (1) interpolation to uniform grid preserves continuity, (2) time window aggregation avoids zero-padding, (3) time-aware models (GRU-D, BRITS, NAOMI) handle gaps natively, (4) adding time delta as feature helps uniform models; never use zero-padding for models with segment continuity assumptions
  • ‘Uniform sampling’ in time-series contexts typically means equal temporal spacing (constant delta t), not balanced sample distribution across categories; critical for sequence models like Transformers that assume regular timesteps
  • Error Recovery Benchmark implements 5-stage pipeline: (1) clean demo context-replay, (2) error skill injection with post-stabilization, (3) human recovery teleop (recording only this stage), (4) trajectory segmentation by RBG heuristics, (5) MimicGen object-centric warping; training data excludes error-injection actions to prevent learning failure modes
  • Faithful algorithm implementation requires understanding full execution model, not just data transformation: MimicGen’s success came from waypoint-level closed-loop correction per subtask, which can’t be replicated by better open-loop warping alone
  • Ollama service can start on-demand and auto-close through process management, suitable for non-persistent use cases; HTTP API connectivity detection can serve as startup signal
  • Multi-round requirement clarification (request_user_input) effectively narrows open-ended tasks to clear technical choices, but each round should focus on key decision points rather than minutiae
  • Local LLM services (Ollama) require robust retry mechanisms for production batch processing due to transient failures
  • Adversarial code review catches issues missed in forward implementation: critic subagent identified in-place mutation semantics, pattern variations, and defensive check locations that main agent overlooked
  • 1-to-N augmented dataset mapping requires aggregation across variant indices: for each clean episode, gather all bddl_idx (0-9 OSS variants) where orig_ep_idx matches, then convert aug_demo_idx to aug_lerobot_ep via conversion_index; some variants fail generation so actual N ≤ 10
  • Always cross-validate aggregated metrics with ground truth: log parsing bugs can propagate incorrect statistics; verify against direct file counts (HDF5 demo_N groups) and structured reports (generation_report.json)
  • MimicGen-style data augmentation for robot manipulation uses object-centric coordinate transforms: source demo trajectories are warped relative to target object poses, enabling generalization across scene configurations while preserving action semantics
  • When single demo source has only 1 successful trajectory per error subtype, even perfect augmentation will have limited scene coverage; 0% results reflected insufficient source diversity + algorithm mismatch, not just one root cause
  • Data format migrations are safer when execution is deterministic: saved (initial_state, actions) → replay in sim → reconstruct missing fields (target_poses, object_orientations); enables backward-compatible augmentation without recollecting demos
  • Git merge conflict ours/theirs semantics are reversed in different scenarios: git merge theirs is merged branch, git stash pop theirs is stash version, easily confused
  • Claude Code’s settings.local.json supports wildcard permissions (like Bash(git:*)), can configure permissive strategy for specific projects without affecting global settings
  • Python’s transformers library is large and requires GPU dependencies; switching to Ollama local service can reduce website extra from torch+transformers+accelerate to just Pillow
  • Windows sandbox command length has hard limit; apply_patch tool fails on patches exceeding certain size (CreateProcessAsUserW error 206), need to split into multiple small patches
  • Marker_single OCR workflow: –force_ocr for full OCR (ignore embedded text), –disable_multiprocessing and –disable_tqdm for cleaner output in automation
  • Config-driven behavior boundaries (generator_mode: faithful_mimicgen vs legacy_replay) enable architecture changes without abandoning old paths, useful for A/B validation and gradual migration
  • Markdown translation needs to preserve structural elements including frontmatter, code blocks, images, URLs, DOI, page markers; also need to detect source text language to avoid redundant translation
  • For scripts without heavy conda dependencies, system Python is more stable than conda run, especially for concurrent or long-running tasks

Practical Learnings#

  • Third-party upstream repository local modifications should prioritize retaining remote version during pull; locally deleted files in conflicts use git checkout –ours to restore remote content
  • Filesystem symlinks provide zero-overhead bidirectional sync for dual-boot systems: replacing ~/Desktop/algorithms with symlink to Windows partition eliminates sync lag and guarantees single source of truth
  • Lustre filesystem mountpoints can appear as regular files (size 0, mode 0444) but behave as immutable filesystem boundaries; when direct configuration files are blocked, hierarchical inheritance mechanisms (AGENTS.md, .gitignore-style configs) provide robust alternatives

Issues and Solutions#

Key Issues#

1. Contrast Loss Bottleneck in BasicContrastive.py: Each node needs to iterate through 3000-4000 points, with tensor cloning at each iteration#

Solution: Vectorized loss calculation using batch logsumexp operations, eliminating loops; using fill_diagonal_(-inf) to mask self-similarity; using boolean masks for positive neighbor filtering

Key Insight: Even mature deep learning code often contains non-vectorized loops during prototyping; performance can be improved by 10-50 times without changing the algorithm

2. Crossformer LIPM version (MSE=9, MAE=1.3) is 18 times slower than master version, despite having a patch mechanism for irregular battery data#

Solution: Identified architectural mismatch: Crossformer’s DSWembedding assumes continuous time steps, but LIPM patch method generates irregular gaps filled with zero, breaking time continuity, wasting model capacity, and losing time interval information. It is recommended to use interpolation to convert data to uniform grids or use irregular time series models (GRU-D, BRITS, NAOMI)

Key Insight: Zero filling is not neutral for sequence models with position encoding—it actually misleads the model to learn “zero patterns” rather than time dependencies. Architectural assumptions must match data preprocessing; forcing irregular data to patch+zero format destroys semantic meaning

3. 12 Error Skill files contain over 150 duplicate lines: Frame availability checks (22 times), direction normalization (8 times), list(dict.keys())[0] pattern (14 times)#

Solution: Extracted 3 helper methods into BaseErrorSkill: check_remaining_frames, normalize_direction, random_direction; replaced list(keys())[0] with next(iter()); added 10 unit tests for new helper functions

Key Insight: Code review found that “skill-specific thresholds” were actually errors; duplicate data hid errors because fixes were not propagated

4. PairedDataset expects format {clean_ep_idx: [{aug_ep, overlap}]}, but episode_mapping.json uses {task_name: {bddl_idx: [{aug_demo_idx, orig_ep_idx}]} format; during HDF5-to-LeRobot conversion, correlation information between source files/demonstration indices was lost#

Solution: Modified conversion script to record conversion_index.json = [{lerobot_ep, source_file, source_demo_idx, length}]; created build_paired_mapping.py script to parse clean/aug conversion indices, using episode_mapping’s orig_ep_idx as bridge, summarizing all bddl_idx information, outputting format compatible with PairedDataset, allowing up to 10 enhancement variants per clean episode

Key Insight: Data sources must be tracked at each conversion step; three-way association requires clear bridge keys at generation time; 1-to-N mapping requires summarizing index information of each variant

5. Users need to clarify the boundaries of recovery demonstrations: Does training include error injection process, or only recovery parts#

Solution: Through code archaeology analysis, confirmed that recovery demonstrations only record post-injection operations; actions_list/states_list start after scene.load() or replay completes; verified with sample NPZ checks, first state of recovery demonstrations equals scene’s post_sim_state state; training conversion filter conditions are success=true, and only use recovery operations

Key Insight: Error recovery datasets intentionally exclude operations that cause errors, preventing model from learning failure modes; only corrective human demonstrations are used for imitation learning

6. Previous discussions suggested 0% enhancement failures were due to “deformation unable to adapt,” but code checks showed some sub-types had dominant NO_WARP parts (e.g., position_error_D1: 150/192 correct positions), which were not deformed at all#

Solution: Corrected diagnostic results: failures were not due to deformation limitations, but due to incomplete deformation coverage, and fundamental architectural mismatch between current animation replay and target pose path execution

Key Insight: Attribution errors remain when conclusions are drawn without checking execution paths; evidence in repositories directly refutes the cited reasons: hard-coded NO_WARP labels mean dominant parts are still open-loop

7. Current enhancer uses source animation replay and deformation targets, while MimicGen actually uses source target pose extraction, path trajectory generation for each sub-task, and interpolation execution—this is a fundamental architectural mismatch#

Solution: Achieved faithful transfer: added target pose records to collections, created object pose tracking compatible with DatagenInfo, specified object reference specifications for each sub-task in configuration, interpolated paths using MimicGen’s WaypointTrajectory, implemented target pose to action execution in enhancer

Key Insight: Surface similarity (“both use object poses”) may hide deeper structural differences; MimicGen’s success depends on path-level closed-loop correction for each sub-task, not just better open-loop animation deformation

8. 7 HDF5 files were opened without a context manager; exceptions during processing could lead to resource leaks#

Solution: Wrapped all h5py.File() calls in 4 script files with with statements (_merge_rounds.py, _check_valid_source.py, parallel_mimicgen_generate.py, augment_success_demos.py)

Key Insight: In high-performance computing batch jobs, resource leaks can accumulate over long runs, leading to mysterious OOM crashes; Python’s context managers can prevent such errors

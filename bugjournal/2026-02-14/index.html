日报 — 2026-02-14#

实现 MIHD 两阶段 Pipeline 架构和 CalendarPro OAuth 修复

今日任务#

  • ✅ MIHD benchmark 测试和超时调整 — 完成 pca+uni2 系列实验，发现 basic_contrastive 超时问题并调整超时时间从 1 小时到 2 小时；staig_fusion_e2e 被移除
  • ✅ MIHD 两阶段 Pipeline 架构设计与实现 — 创建了 pipeline 包含 8 个核心模块（cache_manager、data_preparer、extraction/evaluation_planner、gene/vision_extractor、runner）和 3 个入口脚本，将 embedding 提取和 fusion 评估分为两个独立阶段，实现去重和缓存复用
  • ✅ Pipeline 端到端测试（151508） — 测试了 gene_only、core_multimodal、pca+uni2 多种组合，修复了 ExperimentLogger、mclust fallback 等 bug，验证了缓存机制正常工作
  • ✅ Vision encoding 耗时分析 — 分析了 vision encoding 的性能瓶颈：Patch 提取串行循环、GPU 推理频繁调用 empty_cache、模型加载等阶段；设计了三项优化方案（encoder 级缓存、patch 提取并行化、GPU 推理优化）
  • ✅ CalendarPro Google Calendar 403 错误修复 — 修复了 get_user_timezone() 使用 calendarList API 导致的权限不足问题，改用 settings().get(setting=‘timezone’) API
  • ✅ Gadgets 工具集文档更新 — 更新 CLAUDE.md 和 README.md，移除已删除的 Video/Audio/Image/Papers/OCR 等工具，保留 summarize 和 test 两个核心工具

问题与解决方案#

1. basic_contrastive fusion 超时（3600s 限制）#

解决方案: 将超时时间从 1 小时增加到 2 小时，因为训练型 fusion（basic_contrastive、staig_fusion）每个 section 需要 30-80 分钟

关键洞察: 训练型 fusion（550 epochs）本质上需要更长时间，不能简单减少 epoch 否则影响结果质量

2. Q-Former 每个 section 需要约 5.7 小时（200 epochs × ~100s/epoch）#

解决方案: User 选择将 epochs 从 200 减少到 50（加速 4 倍），通过修改 pipeline_config.yaml 的 extra_config.qformer.epochs

关键洞察: 不同 fusion 策略的计算开销差异巨大：concat/mean/attention 秒级，staig/llava/qformer 分钟到小时级

3. ExperimentLogger 没有 save_comparison_csv() 方法导致 pipeline 崩溃#

解决方案: 移除调用，ExperimentLogger 在 log_experiment 内部自动调用 _update_comparison_csv()

关键洞察: CSV 汇总是自动的，不需要手动触发

4. mclust 聚类失败（rpy2 未安装导致 ImportError）#

解决方案: 在 runner.py 中添加 ImportError 捕获（原本只捕获 ValueError），fallback 到 KMeans

关键洞察: staig_fusion 的 mclust 依赖 R 和 rpy2，General 环境不一定有，需要 robust fallback

5. 每次切换 fusion 策略都重新计算相同的 encoder embedding，UNI2 被算 55+ 次#

解决方案: 设计两阶段架构：Phase 1 提取并缓存所有 encoder embedding（每个 encoder×section 只算一次），Phase 2 从缓存加载执行 fusion+聚类+评估

关键洞察: 缓存粒度应该在 encoder 级别，而不是最终的 fusion 结果级别

6. CalendarPro get_user_timezone() 使用 calendarList API 导致 403 insufficientPermissions#

解决方案: 改用 settings().get(setting=‘timezone’)，该 API 在 calendar.events scope 下可用

关键洞察: Google Calendar API 不同端点的权限要求不同，settings API 比 calendarList 权限要求更宽松

人类思路 vs AI 思路#

Pipeline 架构设计#

角色 思路
人类 User 提出核心思路：每个 slide 先提取所有 encoder embedding 并缓存到文件，然后对相同 embedding 做不同 fusion，避免重复计算
AI AI 扩展设计为完整的两阶段架构，包含 8 个模块、去重逻辑、环境切换、scGPT subprocess 调用、版本控制等工程细节

差异分析: User 抓住了核心瓶颈（重复编码），AI 负责工程化和边界情况处理（scGPT 环境、STAIG 严格模式、缓存版本等）

Q-Former epochs 处理#

角色 思路
人类 面对 Q-Former 耗时过长（5.7h/section），User 选择减少 epochs 到 50 而不是跳过或单独运行
AI AI 提供了三个选项（跳过、减少 epochs、继续），让 User 决策

差异分析: User 在时间和结果质量之间做了务实的权衡

Pipeline 重写范围#

角色 思路
人类 User 要求「完全重写 pipeline」而非增量改进
AI AI 设计时仍保留了向后兼容性（不修改旧版 run_benchmark.py），创建全新 pipeline/ 包

差异分析: AI 在「完全重写」和「保持兼容性」之间找到了平衡，避免破坏现有工作流

AI 局限性#

  • 初次实现 pipeline 时调用了不存在的 save_comparison_csv() 方法，说明对 ExperimentLogger 内部实现细节理解不够深入
  • mclust fallback 最初只捕获 ValueError 而忘记 ImportError，对 Python 异常层级处理不够全面
  • 对 Google Calendar API 不同端点的权限要求不熟悉，需要通过错误日志才能定位到 calendarList vs settings 的差异

今日收获#

  • 两阶段架构的核心价值：将数据提取（重）和计算逻辑（轻）分离，通过缓存避免重复工作
  • 缓存设计要点：版本控制、完整性验证、去重逻辑、环境隔离（scGPT 需要 subprocess）
  • STAIG 系列 fusion 的特殊性：需要 staig_strict 模式的 vision embedding（256px、频域滤波、无 ImageNet 归一化）和额外的 staig_gene_feat
  • Google Calendar API scope 限制：calendar.events 可以访问 settings 端点但不能访问 calendarList，需要选择合适的 API 端点
  • 性能优化的多维度思考：去重（减少计算次数）、并行化（提高单次效率）、缓存（复用结果）

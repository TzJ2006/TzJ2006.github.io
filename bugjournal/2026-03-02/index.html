问题与解决方案#

关键问题#

1. Desktop Video 沙盒应用中 security-scoped 书签从未成功创建（日志显示 ‘saved non-scoped bookmark’），所有启动走 fallback URL，导致 Code 257 权限拒绝。#

解决方案: 补全 Desktop_Video.entitlements 中 com.apple.security.files.bookmarks.app-scope 权限；修复 restoreFromBookmark 和 ensureFileAccess 中的非作用域回退路径，使旧版本书签也能解析并升级。

关键洞察: Xcode ENABLE_USER_SELECTED_FILES=readonly 不自动注入书签权限，必须手动写入 .entitlements。errorLog 的并发写入 bug 同时导致书签创建失败的日志被丢弃，两个 bug 互相遮蔽，大幅增加排查难度。

2. STAIG two-stage fusion 的 GCN 直接使用原始 HVG 表达矩阵而非 gene encoder 输出，导致 scGPT+UNI2+STAIG 与 PCA+UNI2+STAIG 结果完全相同。#

解决方案: 代码追溯确认 Fusion.py 中 train_staig_fusion 接收的是原始 HVG 特征，在周报中注明该结论，提示 PI 无需重跑该组合。

关键洞察: gene encoder 选择对 STAIG fusion 无影响，因为 STAIG 的图神经网络完全绕过了 gene encoder 的输出。

3. Pi0.5 评估 pipeline 的 GPU 利用率仅 ~10%——action chunking 机制使每次推理输出 50 步动作，400 步 trial 仅需约 8 次 GPU 调用，GPU 实际工作时间约 20s / 200s 总时长。#

解决方案: 设计并实现 BatchedVLAServer（多 worker 并行持有 MuJoCo 仿真环境，批量收集请求后单次 forward pass），Stack_D0 实测 4.35s/trial（串行方式估计约 15 分钟，约 4x 提升）。

关键洞察: VLA 推理是内存带宽瓶颈，batch size 从 1 增到 4-8 几乎不增加延迟但线性提升吞吐；核心是将 GPU 空闲时间（CPU 仿真步）隐藏到并行 worker 的下一请求准备中。

4. WebSocket 客户端在 JAX 首次 JIT 编译期间（30-60s）触发默认 20s ping timeout，导致所有 episode 失败，SR 误报 0%。同时多个 JAX 进程并发 JIT 编译时争抢 CPU/内存带宽/PCIe，编译时间从 ~30s 膨胀至 2-8 分钟。#

解决方案: 修改 openpi_client 库的 websocket_client_policy.py，添加 ping_timeout=None；改为错开顺序启动（staggered startup），每个 server 完成 JIT warmup 后再启动下一个。最终架构确认为每个 GPU 独立运行一对 server+client（VLA server 推理严格串行，多 client 共享一 server 反而变慢）。

关键洞察: JAX JIT 编译时间远超 WebSocket 默认超时，此类隐性冲突需要 end-to-end 测试才能暴露；VLA 并行评估的正确粒度是多 GPU 独立 server+client 对，而非多 client 共享同一 server。

5. BatchedVLAServer 启动时 cuBLAS 无法初始化（BLAS OOM），因为 JAX 默认预分配全部 80GB GPU 内存。#

解决方案: 启动命令前加 XLA_PYTHON_CLIENT_MEM_FRACTION=0.85，限制 JAX 使用 85% 内存，留出约 12GB 给 cuBLAS。

关键洞察: JAX 批量推理服务器与 cuBLAS 共存时必须显式限制内存占比，80GB A800 推荐 0.80-0.85。

一般问题#

6. 集群节点访问受限：SSH 到 an49 被 PAM slurm_adopt 拒绝，srun 到高负载节点持续超时，an53 因 CPU 过载连新 srun step 也拒绝。#

解决方案: 组合策略：有活跃 Job 的节点先尝试 SSH；失败则用 srun –overlap –jobid=；节点 CPU 过载时直接 sbatch 新 Job 到空闲节点（an46）。通过共享文件系统读日志绕过节点连接问题。

关键洞察: PAM slurm_adopt 决定 SSH 可达性（目标节点需有活跃 Job）；节点过载时应提交新 Job 而非复用繁忙节点；共享文件系统读日志比 srun 连接更可靠。

7. kill launcher PID 意外同时杀死所有 6 个子训练进程（进程生命周期绑定于 launcher，而非孤儿进程）。#

解决方案: 重新用独立的后台 srun –overlap 命令启动每个任务，使其生命周期不依赖任何父 shell，恢复全部 9 个训练任务。

关键洞察: Slurm 环境中通过 srun 启动的进程生命周期绑定于 job step；持久化后台进程须从 launcher 外部用独立 srun –overlap 启动。

人类思路 vs AI 思路#

战略层面#

GPU 利用率低的发现与诊断#

角色 思路
人类 用户自行完整分析出 GPU 利用率低的根因（action chunking 机制、每 trial 约 8 次 GPU 调用、GPU 工作时间占比约 10%），直接将结论提供给 AI 并要求研究解决方案；同时直觉判断 eval client 显存占用小（实测 ~270MB）可并行 trials。
AI AI 在用户分析前曾给出「GPU 推理占 99% 时间」的错误说法；接收用户分析后才设计 BatchedVLAServer 方案；对并行化的工程障碍（JIT 编译竞争）则主动发现并修复。

差异分析: 系统性能诊断由人类主导，AI 负责方案设计和工程实现；人类性能直觉准确，AI 在运行时问题预判上有明显短板。

科研方向规划与模型设计理解#

角色 思路
人类 PI 主动提出下一步实验方向（cross-sample query：用 layer5 patch 的 zero-shot embedding 跨样本检索，对 HD patches 为新颖应用），并从模型设计哲学角度快速理解 scGPT 单模态不如 PCA（单 spot embedding 不需要邻域信息）。
AI AI 从数值角度分析（Avg ARI 0.170 vs 0.261），用优化目标解释（PCA 直接最大化方差，天然适合 KMeans），对新研究方向无主动贡献。

差异分析: 科研方向的创新提出完全来自人类，AI 依赖数据验证后才给出类似解释；PI 的直觉判断速度和方向感超越 AI 的系统性分析。

评估目标与实验设计原则（MimicGen vs pick_place）#

角色 思路
人类 明确指出应在 MimicGen 对应环境上评估各自的 fine-tuned checkpoint，checkpoint 与环境必须严格匹配，不允许用通用 checkpoint 替代。
AI AI 自动关联到已有 error_recovery_benchmark 框架，提议在 pick_place 场景上评估或使用 pi05_base 通用 checkpoint，需用户两次纠正才明确目标。

差异分析: 人类有清晰的实验设计原则（matched-pair evaluation），AI 倾向于寻找能运行的替代方案而非坚守实验严格性。

集群资源调度（SSH 优先 + 主动利用空闲节点）#

角色 思路
人类 提出应先 SSH 到节点再用 Slurm 命令（更直接高效）；发现 an53 有空闲 GPU 后主动要求将排队任务迁移过去，而非等待 an49 释放（节省约 55 小时）。
AI 受旧 CLAUDE.md 规则（‘Never SSH’）约束默认用 Slurm 方式，不主动感知集群空闲资源，只被动汇报排队状态和预计等待时间。

差异分析: 人类凭实际集群操作经验主动优化资源分配；AI 受规则约束，缺乏主动资源感知与调度能力。

实现层面#

Liquid Glass 效果的概念设计与技术实现#

角色 思路
人类 人类提出「文字本身是玻璃、透过数字看到背后视频」的视觉概念，参考 macOS 原生屏保效果，明确无背景卡片的需求。

差异分析: 概念创意来自人类，技术路径研究来自 AI；实现后的细节验证仍需用户反馈，AI 缺乏主动感知能力。

AI 局限性#

重要局限#

  • 未充分探索现有工具就动手新建实现：在可视化任务中新建脚本而非使用现有 visualize_from_cache.py；在评估任务中默认关联 error_recovery_benchmark 框架而非用户指定的 MimicGen 评估体系；均需用户明确纠正。
  • 无法预判运行时隐性冲突：未在设计阶段识别 JAX JIT 编译时间与 WebSocket 默认超时的冲突、多进程并发 JIT 的资源争抢、kill launcher 会同时终止所有子训练进程——均在运行后才暴露并修复，浪费多轮调试时间。
  • 科研与运维主动性不足：不主动提出下一步实验方向（cross-sample query 完全由 PI 提出）；不主动发现并利用集群空闲 GPU 节点；受旧规则约束（‘Never SSH’）未能灵活应变，需用户明确指出才更新策略。

一般局限#

  • 诊断效率偏低：an53 SSH 超时后尝试约 8 种方案才确定提交新 Job 到 an46；eval 脚本 boolean flag 格式错误尝试三个变体才找到正确写法；排查思路缺乏系统性，倾向于穷举重试而非快速定位根因。
  • 工具在网络文件系统（/XYFS01）上系统性失效（Glob/Grep 无结果），需手动用 Read 工具逐文件访问补偿，大幅降低天河服务器上的代码探索效率。

今日收获#

核心收获#

  • Action chunking 是 VLA 评估 GPU 利用率偏低的结构性根因：Pi0.5 每次推理输出 50 步动作，400 步 trial 仅需约 8 次 GPU 调用（~20s GPU 时间 / ~200s 总时长）。BatchedVLAServer（多 worker 并行持有仿真环境 + 批量推理）可将吞吐提升约 4 倍，Stack_D0 实测 4.35s/trial。
  • WebSocket 长连接与 JAX JIT 编译的隐性冲突：JAX 首次推理需 30-60s 编译，超过 WebSocket 默认 20s ping 超时，导致 SR 误报 0%。修复：ping_timeout=None。多 JAX 进程并发 JIT 是资源炸弹（编译时间膨胀至 2-8 分钟），必须错开启动并等待 warmup 完成。VLA server 推理严格串行，有效并行策略是多 GPU 各自独立运行一对 server+client。
  • macOS 沙盒应用中，Xcode ENABLE_USER_SELECTED_FILES=readonly 不会自动添加安全作用域书签权限；com.apple.security.files.bookmarks.app-scope 必须手动写入 .entitlements 文件，否则书签创建调用静默失败。日志系统 Bug 可以掩盖功能 Bug，日志系统的正确性应被视为基础设施。
  • STAIG two-stage fusion 的 GCN 直接使用原始 HVG 表达矩阵而非 gene encoder 输出，因此 gene encoder 选择（PCA/scGPT）对 staig_fusion 结果无任何影响。scGPT 单模态（Avg ARI 0.170）显著低于 PCA（0.261），但 QFormer 非线性融合 UNI2 视觉特征后提升至 0.381（+124%），说明 scGPT 的语义表示需要非线性融合才能发挥价值。
  • JAX 与 cuBLAS 共存时必须显式限制内存占比（XLA_PYTHON_CLIENT_MEM_FRACTION=0.85），JAX 默认独占所有 VRAM；80GB A800 推荐设置 0.80-0.85 给 Pi0.5 推理。
  • Pi0.5 MimicGen fine-tuned checkpoints（step 14000-15000）在匹配环境上的表现：Stack_D0=100%、Stack_D1=95%、StackThree_D0=80%、StackThree_D1=90%、Coffee_D0=45%、Coffee_D1=15%、Threading_D0=45%。Stack 系列显著优于 Coffee 系列，反映任务操作精度和视觉复杂度对 VLA 微调效果的显著影响。
  • 集群环境关键操作规范：SSH 可达性由 PAM slurm_adopt 决定（目标节点需有活跃 Job）；节点 CPU 过载时应直接提交新 sbatch Job 到空闲节点；通过共享文件系统读日志比 srun 连接节点更可靠；Slurm 持久化后台进程须从 launcher 外部用独立 srun –overlap 启动，避免生命周期绑定。

实践收获#

  • Bash 内联 Python 脚本安全写法：含 f-string 或复杂逻辑的脚本应外化为独立 .py 文件，通过参数传递而非代码嵌入，避免 single-quote/double-quote 嵌套错误（如 warmup 脚本 IndentationError 教训）。

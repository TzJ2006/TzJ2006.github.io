问题与解决方案#

关键问题#

1. S2 中大量历史高引论文丢失：原始设计只从 ArXiv 取最近 100 篇，通过标题匹配用 S2 数据丰富，导致历史论文大多被忽略#

解决方案: 将 S2 作为主数据源（有 year/citation/venue/abstract），ArXiv 仅用于丰富 arxiv_id、pdf_url 和 categories，无 S2 匹配的 ArXiv 新 preprint 作为补充

关键洞察: 数据源主次关系应由数据完整性决定，而非获取便利性；S2 有历史数据和引用量，ArXiv 有全文和分类，两者互补

2. 新变量 target_endpose/target_joint/manip_progress_time 需要 move() 结束时的「未来状态」，而数据收集是逐帧进行的#

解决方案: 设计后处理方案：在 move() 完成后，读取并重写该 move 覆盖的所有 pkl 文件，将常量字段填入每一帧；pkl2hdf5.py 的通用递归转换器无需修改即可自动支持新 key

关键洞察: 后处理写入常量比实时计算更简洁，通用转换器零成本支持新字段是良好的架构设计

3. LLM 生成长段 JSON 分析结果时频繁出现格式错误（未转义引号、中文引号混用、响应截断），分析 Pieter Abbeel 时触发 4-5 次修复请求#

解决方案: 实现 haiku→sonnet→opus 逐级修复链；将「纯解析」（_try_parse_json）和「LLM 修复」分离，避免递归调用导致无限循环；全部失败时保存原始响应便于调试

关键洞察: LLM 修复 JSON 需要 escalation chain 而非单次尝试；分离职责是防止递归的关键设计决策

4. critical_region 在删除子类 override 后仍然出现在 HDF5 数据中#

解决方案: 在 move() 的 pickle 后处理循环里调用 pkl_data.pop(‘critical_region’, None)，直接从已保存的 pkl 中删除该 key

关键洞察: 基类 _base_task.py 的 get_obs() 无条件写入 critical_region，删除子类 override 只是让其值变为0而非不写入；修改数据格式时必须追踪所有上游写入点

5. 获奖论文信息（Best Paper/Spotlight/Oral）无法通过 ArXiv 或 S2 API 直接获取，手动维护奖项表成本高#

解决方案: 利用 LLM 的训练数据知识识别奖项（发送论文标题+年份+venue，让 LLM 返回已知奖项）

关键洞察: LLM 训练数据包含大量学术会议奖项信息，可作为轻量级识别方案，无需额外 API 或人工维护

一般问题#

6. manip_progress_distance_left/right 出现负值#

解决方案: 使用 np.clip(…, 0.0, 1.0) 将距离进度字段强制限制在 [0, 1] 范围

关键洞察: 末端执行器实际轨迹为曲线，可能在到达终点前先偏离起点方向，导致 1 - |current-final|/|start-final| < 0 或 > 1

今日收获#

核心收获#

  • 学术数据双源策略：S2（历史数据/引用量/指标）作为主数据源，ArXiv（全文/分类/新 preprint）作为补充，按标题匹配融合，能覆盖从历史经典到最新 preprint 的完整论文列表
  • 「未来知识」变量（move 完成后才能确定的值）应使用后处理 pkl 文件的模式：move() 结束后批量读写对应帧范围的 pkl 写入常量字段；pkl2hdf5 的通用递归转换器无需修改即可支持新 key
  • LLM JSON 修复需要 escalation chain（haiku→sonnet→opus），并需要分离「纯解析」与「LLM 修复」职责（前者不触发 LLM 调用）避免递归；同时应将失败响应持久化到日志文件以供事后调试
  • 修改数据格式时必须追踪所有上游写入点（get_obs、save_traj_data 等），而不只是子类 override 层——删除 override 并不等于删除字段的写入
  • 利用 LLM 训练数据识别学术奖项信息是可行的轻量方案，无需外部 API，适用于 NeurIPS/ICML/CVPR 等主流会议 Best Paper、Spotlight、Oral 的识别

实践收获#

  • 基于直线距离的操作进度公式（1 - |cur-final|/|start-final|）在曲线末端轨迹下会产生越界值，需要 clamp 到 [0,1]；ArXiv 已支持 HTML 格式论文页面，比 PyMuPDF PDF 解析更轻量且零额外依赖

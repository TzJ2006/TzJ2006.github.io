Learnings#

Architecture (architecture)#

  • Foundation model embeddings (scGPT 512d, UNI2 1536d) are NOT drop-in replacements for dimensionality reduction (PCA 50d) in clustering tasks. Zero-shot foundation models fundamentally underperform PCA by 54% gap (ARI 0.170 vs 0.261) because FM optimizes for cross-tissue generalization capability (pretraining objective is language modeling/contrastive learning) not variance maximization. PCA does task-aligned dimensionality reduction, maximizing variance≈maximizing cluster separability. Validated by Microsoft and 3+ papers + MIHD own data. STAIG graph-based fusion completely ignores gene encoder outputs, using raw HVG matrix instead. QFormer achieved 2× ARI improvement (0.370) by learning gene-vision alignment
  • Action chunking in VLA models fundamentally changes GPU utilization profile and system bottlenecks. Pi0.5 outputs 50-step sequences, so 400-step trials require only ~8 inference calls (~20s GPU work) vs 200s total time = ~10% GPU utilization. Bottleneck shifts from GPU compute to pipeline serialization (CPU preprocessing, WebSocket I/O, simulation steps). Solution requires batched evaluation implementation with timer-based request collection rather than sequential processing
  • EventBus + BackgroundCoordinator + ContextAssembler combination is powerful pattern for building modular AI assistants. EventBus achieves service decoupling through pub-sub pattern, BackgroundCoordinator provides unified management of scheduled task lifecycle using CronScheduler with conditional loading, ContextAssembler dynamically loads context layers based on intent type. Three together support 16+ independent services collaboration with proper lifecycle management requiring explicit startup calls in application hooks. Critical to verify integration points: BackgroundCoordinator must be started in setup_hook() or main.py for background services to actually run
  • Multi-source data collection architectures should be explicit in design and documentation. Error scene generation benefits from diverse sources with different strengths: demo injection for reproducibility, policy rollout injection for realism under learned behavior, natural failure capture for authenticity from real deployment. Pipeline diagrams must show all sources as first-class citizens with clear convergence points (e.g., ErrorSceneDatabase). Missing this conceptual clarity causes documentation-code mismatch and incomplete implementations
  • Context-aware intent classification needs to combine session state (session_id, last_activity_at) and conversation history, pure keyword matching causes ambiguity. Split keywords into always_general (like ‘天气’ weather always force GENERAL) and context_dependent (like ‘几点了’ what time depends on active session). Unit tests (isolated components, mock dependencies, rapid feedback) and integration tests (real interaction, end-to-end experience) are complementary dimensions: 319 unit tests passing ≠ system usable. Must verify critical integration points (service startup, event wiring, data persistence) during implementation not after
  • Two-stage evaluation pipeline mimicking human researcher workflow dramatically improves efficiency and quality. Stage 1 fast screening (title+abstract first 1000 chars, high/low relevance classification) → Stage 2 deep analysis (≤20 high-relevance papers, extract 3 highlights + detailed scoring + direction suggestions). Avoids wasting tokens on low-value papers while generating literature reading records for future reference. Applies to any large-scale filtering problem: coarse-grained batch filtering → fine-grained detailed analysis
  • Symlink-based dependency sharing can reduce disk usage by 99% (78GB→37MB) for large research projects while maintaining independent project structure. Critical to symlink data directories rather than copy, use relative paths for portability. However, creates complex dependency graphs requiring comprehensive mapping before refactoring (65 symlinks × multiple import patterns = high breakage risk). Document all symlinks with purpose, target, and dependents before attempting removal

Debugging (debugging)#

  • Evaluation metric distortion is often not a single bug but cascading chain of multiple bugs where each amplifies the next. BC-RNN case revealed 4 cascading bugs: demo_len=0 broke SPL baseline, trigger_step=0 further broke SPL calculation, transport always False capped RP at 0.5, successful episode RP not 1.0 caused baseline deviation. Need end-to-end validation not just isolated unit tests. Anomalous metric values (RP ceiling of 0.5, systematic zeros) are diagnostic clues revealing chain breakage points. When diagnosing anomalous data, prioritize suspicion of code bugs over rationalizing results. User intuition observing normal behavior during training more accurate than AI explaining ’error recovery inherently difficult’
  • Data preprocessing pipelines need explicit validation that all expected features have normalization statistics. Silent skipping (strict=False in apply_tree) can propagate scale mismatches through entire training runs without errors. apply_tree’s lenient behavior allows scale mismatches in concatenated features (unnormalized prev_actions + normalized actions). Always validate expected keys have norm stats, especially for concatenated features that combine multiple sources. Dynamic detection during data loading catches configuration drift
  • Context loss bugs require tracing complete call chain across multiple layers: data collection (conversation_history.json storage), data passing (function call signatures and parameter threading), data usage (prompt construction and template rendering). Missing any layer breaks context. Log archaeology (reading full log files) can reveal patterns invisible in code: Desktop Video’s ‘saved non-scoped bookmark’ repeated 20× across sessions exposed missing entitlements. Concurrent write bugs (errorLog FileHandle race) silently destroy diagnostic information. High-frequency timer logs (879 updateClockLabels) obscure real errors → need state-based logging not tick-based
  • Effective code audits require multi-dimensional analysis using specialized exploration: syntactic dimension (TODO/FIXME/NotImplementedError grep), behavioral dimension (exception handling pattern analysis, empty loops, stub methods with pass), structural dimension (test coverage gaps, unused exports, dead code), operational dimension (service registration, startup hooks, lifecycle management). Dangerous silent exception handling (except Exception: pass) makes system appear to run normally but state incomplete (Schrödinger’s state). Should always log exceptions (logger.error/warning), distinguish critical functionality (error) and secondary functionality (warning), preserve diagnostic information
  • Documentation synchronization pattern for large codebases: Use parallel exploration agents (one per major directory: error_framework/, scripts/, configs/, outputs/), verify all numbers from actual files (summary.json, filesystem stats, code line counts), then systematically update document section by section. Don’t trust document claims without code/data ground truth verification. Training-vs-evaluation mismatch debugging checklist: When policy performs well in training but fails in evaluation, systematically check (1) observation dictionary keys, (2) environment initialization parameters, (3) action space bounds, (4) data normalization, (5) camera rendering setup. BC-RNN 0% SR on normal tasks + 0.28% on error recovery suggests env mismatch not policy failure
  • Large-scale refactoring should incrementally validate critical integration points rather than implementing all features first then testing. Python circular import solutions: package-level lazy import (only all in init.py), module-level function-internal import where needed, test patch source module not call site after lazy import. When diagnosing resource estimation errors, always verify actual dataset size, per-step time and total steps before launching large-scale training. Phoenix MPM case: dataset scale estimation error (160K→1M) caused 13.5-day training vs expected hours

Domain Knowledge (domain)#

  • RM-IDEAL cross-sample evaluation complete workflow: (1) load string labels aligned by cache barcode, (2) load fusion embeddings from Phase 2 outputs or recompute if zeros detected, (3) compute RM-IDEAL scores using WWL graph kernel + Wasserstein distance, (4) compute embedding cosine similarity (niche centroid vs all spots), (5) evaluate correlation with Spearman r / Precision@K / Same-label rate, (6) generate spatial heatmap visualization. WWL kernel: K rounds of WL message passing, shared hash tables for label encoding, POT library ot.emd2() solves Wasserstein distance (10× speedup over naive implementation). QueST evaluation framework: when no ground truth niche labels exist, construct synthetic ground truth using WWL kernel over cell type annotations, then validate model’s similarity rankings with Pearson correlation. Metrics: Best Match Accuracy (top-1 correctness), Overall Similarity Accuracy (correlation with WWL ground truth)
  • Cross-sample niche query evaluation reveals layer-specific performance patterns and directional asymmetry. Spearman r≈0.44 (p<1e-150) indicates moderate positive correlation between fusion embedding cosine similarity and RM-IDEAL topology scores. P@K close to 0 indicates large head differences (embedding similarity and topology measure different dimensions). Low same-label rate suggests embedding spaces across sections not fully aligned. Layer_1/5 showed best performance (ρ 0.42-0.66), Layer_3/6 exhibited negative correlation (ρ -0.21 to -0.36) possibly confused with adjacent layers in embedding space, causing embedding-based similarity ranking opposite to topology-based ranking. Boundary/central layers easier to distinguish than middle transition layers. Directional asymmetry example: 151508→151673 Layer_4 ρ=0.657, but reverse only 0.275. Reveals fusion embedding space quality differences in structural representation of same niche label across different samples, possibly related to spot density, noise level or annotation quality
  • VLA systematic failures at task-critical phases (imprecise pressing, failed grasping/placing) can be addressed by training binary classifier to predict ‘critical regions’ using distance-based heuristics with gripper-state filtering. Enables cls token classifier training for high-stakes regions requiring extra caution. Must verify actual dataset size, per-step time and total steps before launching large-scale training to avoid resource waste. Phoenix MPM case: dataset scale estimation error (160K→1M) caused 13.5-day training vs expected hours when expected hours. Multi-task ML pipelines built via incremental prototyping (start with 1 task) leave scattered single-task configurations (commented-out task lists, hardcoded paths). Scaling to production requires systematic audit of: data prep scripts, dataset mapping JSONs, checkpoint directories, and evaluation configs
  • Project description narrative strategy for external communication: start from application scenario target audience cares about (clinical patch query cross-patient knowledge transfer), then explain technical path and key differentiators (zero-shot vs end-to-end training), finally elevate to domain value. Technical details serve narrative logic not vice versa. Control each sentence within dual short clauses to reduce cognitive load. In spatial transcriptomics, adata.X can be either normalized values (scanpy pipeline) or raw counts (STHD pipeline) - must check code rather than assume from data type. Checkpoint inventory best practices: Record full path, size, last modified date, and usability status for all model checkpoints. Large checkpoint directories (180GB) indicate many saved steps - document which step to use for evaluation. Date-based naming helps track training timeline but needs explicit step-to-date mapping

AI Usage Notes#

Effective Patterns:

  • ✓ Parallel exploration agents for rapid codebase understanding: launched 3-4 independent agents to simultaneously explore conversation history management, intent classification, AI provider call chains, service registration patterns across large codebases
  • ✓ Multi-dimensional code auditing with specialized agents: TODO/FIXME search, git analysis, exception auditing, test gap detection to systematically discover hidden bugs invisible in normal code review

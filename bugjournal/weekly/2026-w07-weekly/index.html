Problems & Solutions#

1. Visualization videos showed no visible error effects despite logged force application and body displacement (15mm-117mm) [Error Recovery Benchmark] (2026-02-15)#

Solution: Root cause: OSC position controller (kp=150) actively compensated external forces when tracking demo trajectory. Fix: use neutral actions (zero vector) during injection+settling phases, allowing force to accumulate without controller interference. Validated with 90mm displacement in test cases.

2. MuJoCo automatically clears xfrc_applied after each physics step, causing force effects to disappear [Error Recovery Benchmark] (2026-02-15)#

Solution: Re-apply force every control step during duration window (mimicking validation code drop.py), ensuring sustained force effect across all 20 physics substeps per control step

3. Q-Former process crashed after completing 6/11 sections, potentially losing ~9 hours of GPU compute work [MIHD] (2026-02-15)#

Solution: Pipeline’s checkpoint recovery mechanism (checking for existing *_embeddings.npz files) automatically detected completed sections and resumed from section 151672, validating the 断点续跑 architecture

4. JSON file writes interrupted by Ctrl+C/crash/power loss lead to half-written invalid JSON causing parse failures [gadget/summarize] (2026-02-15)#

Solution: Implemented _atomic_write() function using tempfile + os.replace pattern at 5 file write points, ensuring either old complete file or new complete file exists, never half-written state

5. Multi-device workflow required manually copying log files from each device to one machine for merge [gadget/summarize] (2026-02-15)#

Solution: Export uploads to /logs/, merge uploads to /reports/, added –sync flag to auto-download all device logs from remote, simplifying to one-command workflow

Learnings#

Architecture (architecture)#

  • Error injection pipeline is already policy-agnostic at action level. Extending from demo injection to policy injection only requires action source replacement (demo_actions[step] → policy.predict(obs)); detection-injection-validation pipeline reuses as-is with no modifications needed.
  • Atomic write pattern prevents file corruption: write to temp file, flush, os.replace for atomic rename, cleanup temp on failure. Ensures either old complete file or new complete file exists, never half-written state. Apply to all JSON/config file persistence.
  • Multi-device collaboration tool design pattern: local generation → upload to cloud subdirectories → one-command download+merge. Encapsulate complex workflow into single command via –sync flag, reducing mental burden from ‘manual copying’ to ‘one command’.
  • Checkpoint/resume mechanisms validate their design during actual crashes. File-existence checks (is_completed()) successfully prevented re-running expensive experiments (~9 hours GPU work for 6 sections), proving the recovery architecture works in production.
  • Lazy loading with fallback requires transactional initialization: create object in local scope, validate all steps, only assign to instance variable after complete success. Partial success creates broken state.
  • Code duplication threshold: when same pattern repeats 8+ times, extract into shared utility. Beyond reducing repetition, centralized error handling/timeout logic prevents inconsistent behavior. Single Source of Truth applies to logic patterns, not just data.
  • Dependency analysis is critical for multi-phase plans: normalization must precede fusion experiments, config foundation enables all subsequent work, batch correction requires normalized fusion output. Document dependencies explicitly.

Debugging (debugging)#

  • OSC position controllers (kp=150) generate joint torques that far exceed typical external forces (15-20N), actively compensating disturbances when tracking trajectories. Neutral actions disable tracking, allowing force effects to accumulate. Validation uses neutral actions; visualization must too.
  • MuJoCo automatically zeros xfrc_applied after each physics step (0.002s default) to prevent accumulation. Must re-apply force every control step for sustained effect. Robosuite’s env.step() contains 20 physics substeps; OSC controller tracks target pose at every substep with kp=150 stiffness.
  • In large codebases with modular utilities, always grep for all occurrences of a function name before modifying. Local copies can shadow imports and cause silent modification failures (e.g., run_benchmark.py shadowing utils/visualization.py).
  • When same data section consistently fails across multiple model configurations with NaN/zero-variance symptoms, root cause is likely data quality issue (missing coordinates, corrupted values) not code bug. STAIG model collapse is known failure mode for problematic inputs.
  • MuJoCo EGL uses relative GPU indexing within CUDA_VISIBLE_DEVICES, not absolute system GPU IDs. For single-GPU processes, always set MUJOCO_EGL_DEVICE_ID=0 regardless of which physical GPU (adjust CUDA_VISIBLE_DEVICES per process).

Tools (tools)#

  • Cloud LLM APIs now support structured output - prioritize SDK features over fragile parsing. Use Anthropic tool_use with schema definition, OpenAI response_format: json_object. Keep 3-layer fallback parsing as degradation path for legacy/non-compliant responses.
  • Idempotency design through persistent state avoids redundant operations. Track processed state (e.g., _merged_devices list), check on re-run, skip if already completed. Saves API cost and time while making each contributor’s work visible (per-device dict format).
  • Experiment management requires explicit caching strategy: –override for full re-run, –skip_cached for incremental completion, no flags to fail on duplicates. Always check existing results before re-running experiments.
  • Writing plans to .md files and pasting in new sessions effectively preserves context across sessions, more robust than depending on tool call chains. Plan text pattern overcomes session boundaries and context window limits.
  • External tool output formats may change with version/environment. Defensive programming should check types and handle gracefully (e.g., ccusage returning list vs dict). Add type checks after parsing external tool JSON.

Domain Knowledge (domain)#

  • Q-Former training requires ~22GB GPU memory. On 32GB GPU, parallel Q-Former experiments cause OOM despite being logically independent. Memory-intensive deep learning experiments require sequential execution even when compute graph is independent.
  • Deep learning benchmarks with large spatial transcriptomics data (4000+ spots, 1024-dim embeddings, STAIG training) require exclusive GPU access. Parallelize only CPU-bound tasks like RM-Ideal WWL graph kernels.
  • Detector design pattern: let high-frequency triggered detectors propose multi-type specs to avoid single type consuming max_scenes_per_demo quota. Extend ProximityDetector and CollisionDetector to generate mixed specs (impulse+pose_perturb+friction) like GraspPrecondDetector.
  • STAIG fusion requires spatial coordinate validation and image integrity checks as preprocessing steps. Missing coordinates or corrupted images cause runtime failures. Robust pipelines should validate prerequisites and fail gracefully with clear error messages.
  • Normalization inconsistency across encoders (UNI2/scGPT L2-normed, HIPT/ResNet50/MLP/GCN not) hurts downstream clustering. Systematic 3-point normalization (post-encoder, pre-fusion, post-fusion) with configurable methods provides fine-grained control.
  • Trajectory data saving should include complete state: actions/rewards + EEF position + gripper state + all object positions. Use NPZ compressed storage (~64KB/episode), controlled by config flag (evaluation.save_trajectories).

问题与解决方案#

1. MuJoCo data.cvel中linvel/angvel读反：env_wrapper.py将cvel[:3]当作线速度，实际布局为[angular(3), linear(3)]，导致instability detector系统性误触发，angvel=0.445 rad/s被误判为linvel=0.445 m/s超阈值 [Error Recovery Benchmark] (2026-02-13)#

解决方案: 交换env_wrapper.py中:3和3:赋值，修正注释，同步修改state_extractor.py/filters.py/impulse.py，消除系统性误判

2. Demo回放中物体完全不动，EEF与物体距离始终≥22cm：demo录制使用PickPlace_D0+OSC_POSE controller_configs，但生成脚本用裸PickPlace且无控制器配置，相同action在不同控制器下产生完全不同轨迹 [Error Recovery Benchmark] (2026-02-13)#

解决方案: 修改create_env()从HDF5 data.attrs[’env_args’]读取完整环境配置，将MimicGen变体名映射回基础名，pop不兼容的kwargs

3. 可视化视频机械臂无位移：OSC控制器（kp=150）在每个物理子步产生的校正力矩远大于15-20N外力，将机械臂强制拉回demo轨迹；xfrc_applied每步自动清零 [Error Recovery Benchmark] (2026-02-15)#

解决方案: 在力注入窗口改用neutral actions（零动作向量），增加settle_steps让物体自由运动，与validators/drop.py对齐，物体位移达90mm+

学习收获#

调试 (debugging)#

  • MuJoCo data.cvel布局为[angular_velocity(3), linear_velocity(3)]，与直觉相反；MuJoCo xfrc_applied每物理步自动清零，单次施力冲量极小；set_sim_state_flat()只恢复qpos/qvel，不恢复model.geom_friction和data.xfrc_applied。调试前应优先验证底层数据合理性（物体是否运动），而非直接调参。
  • robosuite环境复现必须从HDF5 data.attrs[’env_args’]读取完整配置（包含controller_configs）。MimicGen变体名（PickPlace_D0）需映射回基础名，不兼容的kwargs需pop。相同action在不同控制器参数下产生完全不同轨迹，环境配置不匹配是最常见的复现失败原因。
  • 机器人OSC控制器闭环增益（kp=150）产生的关节力矩远大于外力注入，demo actions执行期间注入外力几乎无效。正确方案：注入窗口切换为neutral actions+settle_steps，与validators保持一致。数值日志的position变化不能替代目视检查，两者必须一致才算验证成功。
  • Python mock中patch路径必须指向「名字被查找的模块」而非「函数定义的模块」；函数体内的lazy import将名字绑定到执行模块命名空间，应patch已绑定的实例方法。大型脚本中重要工具函数可能同时存在本地定义和模块导入两版本，应用Grep搜索所有def出现位置。
  • 从混合文本中提取JSON必须用字符级平衡括号计数，不能使用贪婪正则{[\s\S]*}——后者在多JSON块时合并相邻对象产生无效JSON。

架构 (architecture)#

  • 错误注入可视化必须展示完整因果链：demo回放（手臂靠近物体）→detector触发→注入错误→物理效果。execute step但不返回渲染帧的函数（collect_rollout_stats类型）不能用于可视化，需手动逐步渲染。
  • 多设备协作工作流核心模式：解耦「解析」（纯本地操作）和「AI汇总」（需API），以JSON为中间媒介；用_merged_devices列表实现操作幂等性；pull-before-merge是防止多设备覆盖报告的必要机制；目录分层（logs/ vs reports/）让不同类型产物互不干扰。
  • 注册表模式的调用方也必须同步改为registry查找（不能硬编码）；注入器注册表键与检测器生成的ErrorSpec.type字符串必须严格一致，最佳实践是在统一常量文件中定义错误类型枚举；「写好但未接入」是代码库中最常见的隐性风险，需端到端集成测试才能发现。
  • 策略驱动误差注入的核心只是替换action来源（demo_actions[step]→policy.predict(obs).action），检测→注入→验证管线完全复用；适配器模式（PolicyAdapter）统一Random/BC-RNN/VLA的接口差异，配合TYPE_CHECKING避免循环导入。
  • AI API调用需要双层超时保护：asyncio.wait_for（协程级，防止await卡死）+httpx.Timeout（连接级，防止TCP握手卡死）；asyncio.create_task创建的后台任务必须设置done_callback做异常监控。Anthropic tool_use的tool_choice参数可强制模型返回结构化JSON，比prompt描述格式更可靠。
  • 双重验证分类器不一致时应优先信任LLM：SR做嵌入相似度匹配易被关键词牵引，LLM理解完整句意，mismatch场景下LLM准确率更高。惰性初始化必须在所有步骤全部成功后才提交状态，不能在初始化过程中提前暴露半初始化对象。

工具 (tools)#

  • HPC/服务器两个关键规范：①conda activate在非交互式bash中失效，必须用环境Python的完整路径；②subprocess调用需显式设置cwd，输出路径必须是绝对路径，断点续跑依赖文件系统状态时尤为关键。
  • scanpy的sc.pl.spatial()使用save=参数时返回None，无法二次标注；正确做法是传入ax=参数配合显式plt.subplots()，再用fig.text()添加指标文本。
  • os.replace在同一文件系统上是原子操作，配合tempfile.mkstemp写入同目录可完全消除文件损坏风险；JSONL格式（每行一条JSON）是快速追加写入的最佳存储选择，无需加锁、天然支持流式读取；定时任务用last_scheduled_date字段做每日幂等性保护是简洁有效的防重复策略。
  • GPU监控必须使用后台线程持续采样（每秒一次）而非开始/结束时快照，否则操作期间GPU已空闲读数为0。JSONL反序列化bool时需显式转换：bool(str(v).lower() == ’true’)，Python str()会将False→‘False’，bool(‘False’)=True。

领域知识 (domain)#

  • QueST的niche查询机制：通过k-hop空间子图聚合组织微环境（niche），用余弦相似度跨样本检索相似niche，可发现TLS等疾病特征；批次效应校正使用对抗训练，编码器被训练生成批次不可分辨的嵌入。
  • VLA与空间组学架构正在独立趋同：两个领域收敛到相同技术创新（Diffusion Transformer、模态tokens、对比对齐）；最值得迁移的是Flow Matching融合和Register Tokens+AdaLN。
  • 随机策略rollout yield极低（~10%）：随机策略缺乏到达操作关键区域的能力，demo回放天然覆盖关键物理交互区域；随机策略适合验证管线正确性，不适合生成有代表性的错误场景。HPC实验调度应将慢速任务（Q-Former ~90min/section）独立后台运行，不与快速实验混合串行。
  • L2归一化具有幂等性（对已归一化向量再归一化结果不变），可安全统一应用于所有编码器输出；三点归一化（post-encoder/pre-fusion/post-fusion）解决多模态融合管线的一致性问题。CLAUDE.md应只包含「读多个文件才能理解的大局架构」和「会造成困惑的运行时约束」，不应重复README或列出工具可直接发现的文件结构。

AI 使用备注#

有效模式:

  • ✓ 并行多子智能体探索复杂代码库（MIHD、CalendarPro多Agent并行分析显著加速理解）
  • ✓ 以结构化计划文档（含精确文件路径和行号）驱动大型任务分步实施，可中断恢复
  • ✓ 测试设计能力强：在实现计划书未明确说明的边界情况下主动补充测试（如bool反序列化、MetricsWithCI格式化）
  • ✓ 文献综合能力强：系统调研7篇论文并构建25种错误类型分类体系，主动提炼VLA→空间组学迁移框架
  • ✓ 实现细节优化：主动将贪婪正则替换为括号计数算法等更稳健方案

局限性:

  • ✗ 接受异常数据为既成事实而不质疑：面对linvel=0.445 m/s和EEF距物体22cm等明显异常，倾向从代码逻辑合理化解释而非追问数据来源
  • ✗ SSH命令反复缺少cd或绝对路径，同一模式失败20+次未能主动切换策略
  • ✗ 过度工程化倾向：无详细计划时设计复杂抽象层（PolicyAdapter多层级），需人类指出「就是换action来源」才回归最小改动
  • ✗ ExitPlanMode时机判断与用户不同步：多次被拒绝后未主动询问原因，被动等待用户新指令
  • ✗ 对设计层面的抽象反馈处理不足：将「视频三阶段相同」（因果链缺失）连续两次误诊为帧捕获技术问题
  • ✗ 初始实现不够稳健：_extract_json用贪婪正则、mock patch路径错误、subprocess相对路径等基础错误需测试失败后才修正

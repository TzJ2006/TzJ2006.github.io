Problems & Solutions#

1. Visualization videos showed no visible error effects despite logged force application and body displacement [Error Recovery Benchmark] (2026-02-15)#

Solution: Root cause: Demo actions during injection caused OSC controller (kp=150) to compensate external force. Fix: use neutral actions (zero vector) during injection+settling phases, allowing force to accumulate without controller interference

2. Q-Former process crashed after completing 6/11 sections, potentially losing hours of GPU compute work [MIHD Spatial Transcriptomics Benchmark] (2026-02-15)#

Solution: Pipeline’s checkpoint recovery mechanism (checking for existing *_embeddings.npz files) automatically detected completed sections and resumed from section 151672, skipping already-finished work

3. JSON file writes interrupted by Ctrl+C/crash/power loss lead to half-written invalid JSON, causing parse failures on next read [gadget/summarize Tool] (2026-02-15)#

Solution: Implemented _atomic_write() function: write to temp file (mkstemp), flush, os.replace for atomic rename, cleanup temp on failure. Ensures either old complete file or new complete file exists

4. Multi-device workflow requires manually copying log files from each device to one machine for merge, tedious operations [gadget/summarize Tool] (2026-02-15)#

Solution: Export uploads to /logs/, merge uploads to /reports/, added –sync flag to auto-download all device logs from remote, dedupe paths then call API to generate report

5. scgpt_uni2_qformer experiments failed with CUDA out of memory (11/12 failures), tried to allocate 2 MiB but GPU only had 2.31 MiB free [MIHD Spatial Transcriptomics Benchmark] (2026-02-15)#

Solution: Identified root cause: pca_uni2_qformer was monopolizing GPU (21.5 GB), leaving insufficient memory for parallel scgpt_uni2_qformer runs; determined scgpt Q-Former experiments must wait until pca Q-Former completes

6. LLM occasionally returns invalid JSON (markdown code blocks, extra text), causing json.loads to fail [gadget/summarize Tool] (2026-02-15)#

Solution: For Anthropic use tool_use to force structured output (define submit_report tool schema + tool_choice), for OpenAI use response_format: json_object, keep 3-layer fallback parsing logic as degradation path

7. Same device running export –summarize multiple times on same day repeatedly calls API, wasting cost and time [gadget/summarize Tool] (2026-02-15)#

Solution: Added _merged_devices list in JSON to track already-summarized devices, read existing file on export to check if current device in list, skip API call if present and reuse existing summary

8. EGL rendering assertion failed: MUJOCO_EGL_DEVICE_ID must be contained in CUDA_VISIBLE_DEVICES [Error Recovery Benchmark] (2026-02-15)#

Solution: Fixed batch_visualize.py GPU assignment: set MUJOCO_EGL_DEVICE_ID=0 (always) and adjust CUDA_VISIBLE_DEVICES per process to match relative ID

9. ProximityDetector and CollisionDetector only generate impulse type specs, leading to single-type scene library (all 30 scenes are impulse/tip_over) [Error Recovery Benchmark] (2026-02-15)#

Solution: Modified both detectors’ detect() methods to replace _generate_impulse_specs() with _generate_mixed_specs(), generating impulse+pose_perturb+friction candidates simultaneously, configured max_scenes_per_demo increased to 10

Learnings#

Architecture (architecture)#

  • OSC position controllers (kp=150) generate joint torques that far exceed typical external forces (15-20N), actively compensating disturbances when tracking trajectories. Neutral actions disable tracking, allowing force effects to accumulate. Robosuite’s env.step() contains 20 physics substeps with OSC tracking at every substep.
  • Error injection pipeline is policy-agnostic at action level - EnvWrapper.step(action) doesn’t care about action source. Core of extending from demo injection to policy injection is just action source replacement (demo_actions[step] → policy.predict(obs)), detection-injection-validation pipeline reuses as-is.
  • Checkpoint/resume mechanisms validate their design when actual crashes occur. MIHD’s is_completed() file-existence check successfully prevented 6 sections (~9 hours of GPU work) from being re-run after process crash, validating the recovery architecture from original implementation plan.
  • Multi-device collaboration tool design pattern: local generation → upload to cloud subdirectories → one-command download+merge, encapsulating complex workflow into single command via –sync flag.
  • Dependency analysis is critical for multi-phase plans - normalization must precede fusion experiments, config foundation enables all subsequent work, and QueST batch correction requires normalized fusion output.
  • Infrastructure-level functionality (time parsing, provider selection) should have unified implementations rather than per-module copies. Single Source of Truth principle applies to logic patterns not just data definitions. Code duplication threshold: when same pattern repeats 8 times, it must be extracted.

Debugging (debugging)#

  • MuJoCo automatically zeros xfrc_applied after each physics step (0.002s default); must re-apply force every control step for sustained effect. Validation code does this correctly in 10-step loop.
  • Q-Former training requires ~22GB GPU memory; on 32GB GPU, parallel Q-Former experiments cause OOM despite being logically independent. Memory-intensive experiments require sequential execution even when compute graph is independent.
  • When same data section (e.g., 151676) consistently fails across multiple model configurations with NaN/zero-variance symptoms, root cause is likely data quality issue (missing coordinates, corrupted values) not code bug. STAIG model collapse is a known failure mode for problematic inputs.
  • In large codebases with modular utilities, always grep for all occurrences of a function name before modifying - local copies can shadow imports and cause silent modification failures.

Tools (tools)#

  • Atomic write is standard pattern for preventing file corruption: write to temp file (mkstemp), flush, os.replace for atomic rename, cleanup temp on failure. Ensures either old complete file or new complete file exists, never half-written state. Applies to all JSON/config file persistence scenarios.
  • Cloud LLM API’s structured output features (Anthropic tool_use, OpenAI response_format) should be prioritized over fragile regex/brace-matching parsing, can significantly reduce JSON parsing failure rate.
  • Idempotency design: through _merged_devices list tracking processed state, skip completed operations on re-run, saving API cost and time. Device_summary changed to {device_name: summary} format makes each device’s contribution clearly visible.
  • MUJOCO_EGL_DEVICE_ID uses relative indexing within CUDA_VISIBLE_DEVICES, not absolute system GPU IDs. For single-GPU processes, always set MUJOCO_EGL_DEVICE_ID=0 regardless of which physical GPU.
  • Lazy loading with fallback requires transactional initialization: create object in local scope, validate all initialization steps, only assign to instance variable after complete success. Partial success creates broken state.
  • Experiment management requires explicit caching strategy: –override for full re-run, –skip_cached for incremental completion, no flags to fail on duplicates.
  • Writing plans to .md files and pasting in new sessions is effective way to preserve context across sessions, more robust than depending on tool call chains.

Domain Knowledge (domain)#

  • Detector design should let high-frequency triggered detectors propose multi-type specs to avoid single type consuming max_scenes_per_demo quota. GraspPrecondDetector has correct design (impulse+pose_perturb+gripper_bias), needs extension to ProximityDetector and CollisionDetector.
  • Trajectory data saving should include complete state (actions/rewards + EEF position + gripper state + all object positions), NPZ compressed storage (~64KB/episode), controlled by config flag (evaluation.save_trajectories).
  • Documentation improvement is high-leverage: MIHD’s CLAUDE.md was reduced 67% while adding missing content by eliminating redundancy, discoverable info, and implementation history.

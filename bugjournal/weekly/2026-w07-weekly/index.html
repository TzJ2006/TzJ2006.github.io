Issues and Solutions#

1. Complete E2E verification of error_recovery_benchmark is hindered by GPU node dependencies (EGL/robosuite runtime constraints) [error_recovery_benchmark] (2026-02-10)#

Solution: Define a clear testing strategy: perform CPU unit tests and smoke tests first. Delay complete E2E verification until the GPU node (an53) is available to avoid blocking development

2. @praha/byethrow’s Result.try() API was misused: AI used it directly as an result, while it actually returns a reusable parser function [ccusage] (2026-02-12)#

Solution: Store the return value of Result.try({…}) as a function, and explicitly call this function to obtain Result<T, E>

3. The LiteLLM pricing JSON contained metadata keys like sample_spec, which were treated as empty pricing entries after parsing with valibot schema, leading to test assertion failures [ccusage] (2026-02-12)#

Solution: After schema parsing, additionally check that input_cost_per_token or output_cost_per_token is not null. Filter out metadata entries without pricing data

4. Multiple attempts to SSH to the GPU node were rejected by pam_slurm_adopt (active SLURM job required), resulting in significant wasted time [error_recovery_benchmark] (2026-02-14)#

Solution: Users suggested checking whether the current node has a GPU using nvidia-smi -L. It was found that there were already 5 A800 GPUs on the current node, so no SSH transfer was required

Learnings#

Architecture#

  • Using external summary tools (ccusage) to export historical sessions is an effective engineering approach for solving AI context limitations: externalizing session summaries into files allows AI to quickly restore project context in new sessions, avoiding repeated background description
  • Testing strategies for large robot RL projects should be clearly hierarchical: CPU unit tests, CPU smoke tests, GPU E2E testing. Avoid blocking the entire test process due to GPU unavailability

Domain Knowledge#

  • The OSC controller (kp=150, Λ≈5-10kg) has strong resistance to external forces. An impulse with duration_steps=1 (about 20ms) is canceled out within the controller’s response cycle, effectively equivalent to no force application. To make disturbances visible, continuous force application must be maintained for several control cycles or use forces far beyond what the controller can handle
  • MIHD enhancement plan is implemented in a strict dependency order: configuration basics (Idea 6 Phase 1) → normalization → UNI2+scGPT experiments → Q-Former/LLaVA → niche query → batch correction → full configuration integration. This order is designed to avoid cumulative refactoring costs

Tools#

  • @praha/byethrow’s Result.try({try, catch}) returns a factory function (which must be called explicitly), rather than directly executing and returning a result. This differs from common Rust/fp-ts implementations of the Result monad. Special attention is needed
  • In HPC clusters, use nvidia-smi -L to check whether the current node has a GPU before executing GPU tasks. If you need to SSH to remote nodes, you must explicitly change directory to the project folder and activate the conda environment in the remote command (SSH does not inherit the current shell working directory)
  • ccusage’s offline pricing is pre-filtered using the isClaudeModel function in _macro.ts during construction. Extending multi-model support requires simultaneous modifications to both macro and _pricing-fetcher.ts
  • ESLint rules for this project require all Node.js built-in objects to be explicitly imported via the node: protocol (e.g., import process from ’node:process’). Global objects cannot be used directly

Debugging#

  • When debugging force injection or other physical mechanisms, it is important to first verify whether the mechanism itself works using extreme parameters (such as infinite force, 1000 consecutive steps), and then gradually adjust the parameters to a reasonable range. This should not be done starting from reasonable parameters

  • ✗ There are incorrect assumptions about library API behavior (Result.try() factory pattern). These can only be corrected after testing fails
  • ✗ Failed to identify existing data assets (local ccusage.json pricing files). The default approach is to modify code logic rather than reuse existing files
  • ✗ Mechanically following historical command templates (proxy settings) without verifying their applicability. Human intervention is needed to indicate that the task does not require network access

Issues & Solutions#

1. Full E2E validation for error_recovery_benchmark blocked by GPU node dependency (EGL/robosuite runtime constraints) [error_recovery_benchmark] (2026-02-10)#

Solution: Established a clear test layering strategy: CPU unit tests and smoke tests first, full E2E validation deferred until GPU node (an53) becomes available, preventing testing from blocking development

2. @praha/byethrow’s Result.try() API misused: AI treated it as returning a result directly, but it actually returns a reusable parser function [ccusage] (2026-02-12)#

Solution: Store the return value of Result.try({...}) as a function, then explicitly call that function to obtain the Result<T, E>

3. LiteLLM pricing JSON contains metadata keys like sample_spec that, after valibot schema parsing, are treated as empty pricing entries, causing test assertion failures [ccusage] (2026-02-12)#

Solution: After schema parsing, additionally check that input_cost_per_token or output_cost_per_token is not null, filtering out metadata entries with no pricing data

4. Multiple SSH attempts to the GPU node rejected by pam_slurm_adopt (requires an active SLURM job), causing significant time waste [error_recovery_benchmark] (2026-02-14)#

Solution: User suggested running nvidia-smi -L first to check whether the current node already has GPUs; discovered the current node had 5× A800 GPUs, making SSH unnecessary

Lessons Learned#

Architecture#

  • Using an external summary tool (ccusage) to export historical sessions is an effective engineering pattern for working around AI context limitations: externalizing session summaries as files lets the AI read them in a new session to quickly restore project context, avoiding repeated background explanations
  • Testing strategy for large robotics RL projects should have explicit tiers: CPU unit tests, CPU smoke tests, GPU E2E tests — preventing GPU unavailability from blocking the entire test pipeline

Domain Knowledge#

  • The OSC controller (kp=150, Λ≈5-10kg) is highly resistant to external forces; a duration_steps=1 (~20ms) impulse is cancelled within one control cycle, equivalent to no force at all. To produce visible perturbation, force must be applied for at least several control cycles, or must far exceed the controller’s compensation capacity
  • The MIHD enhancement plan must be implemented in strict dependency order: config foundation (Idea 6 Phase 1) → normalization → UNI2+scGPT experiments → Q-Former/LLaVA → Niche queries → batch correction → full config consolidation; the ordering is designed to prevent accumulated refactoring costs

Tools#

  • @praha/byethrow’s Result.try({try, catch}) returns a factory function (which must be called again), not an immediately-executed Result; this differs from common Result monad implementations like Rust or fp-ts — note carefully
  • On HPC clusters, run nvidia-smi -L first to check whether the current node already has GPUs before submitting GPU jobs; if SSH to a remote node is needed, the remote command must explicitly cd to the project directory and activate the conda environment (SSH does not inherit the current shell’s working directory)
  • ccusage’s offline pricing is pre-filtered at build time via isClaudeModel in _macro.ts; extending multi-model support requires updating both the macro and _pricing-fetcher.ts
  • This project’s ESLint rules require all Node.js built-ins to be explicitly imported via the node: protocol (e.g., import process from 'node:process'); global objects cannot be used directly

Debugging#

  • When debugging force injection or other physical mechanisms, first verify the mechanism itself works using extreme parameters (e.g., infinite force, 1000 sustained steps), then tune down to reasonable ranges — rather than starting from reasonable values and incrementally increasing

AI Usage Notes#

Effective patterns:

  • ✓ Reconstructing cross-session project history by reading Markdown files from the ccusage summaries/ directory is an effective engineering solution to AI context window limitations
  • ✓ Static code analysis (reading Makefiles, test code, existing docs) can effectively generate contributor guidelines even without git history
  • ✓ The binary extreme-testing approach (first verify the mechanism with infinite force, then tune parameters) is more efficient than systematic code analysis when debugging physics simulations

Limitations:

  • ✗ Repeated the same error within a single session (SSH missing cd command) 15+ times, with no ability to self-correct from repeated failures
  • ✗ Made incorrect assumptions about unfamiliar library API behavior (Result.try() factory pattern) — only corrected after test failures
  • ✗ Failed to proactively identify existing data assets (local ccusage.json pricing file); defaulted to modifying code logic rather than reusing the existing file
  • ✗ Mechanically applied historical command templates (proxy settings) without verifying their applicability; required human to explicitly point out the task needed no network access

Problems & Solutions#

1. User prompts too general causing AI to misinterpret intent, exceed scope, and ignore constraints [Unified Agentic Workflow System] (2026-05-11)#

Solution: Designed mandatory Spec Template with five required fields (Goal, Scope, Constraints, Success Criteria, Non-goals) forcing structured specification before implementation. Agent must complete and get user confirmation.

2. Lack of verification mechanism causing uncertainty whether code changes actually solved the problem [Unified Agentic Workflow System] (2026-05-11)#

Solution: Implemented Verification Gate where success_criteria (executable commands) are defined in spec. After completion, automatically runs verification: pass proceeds to review, fail pauses into debug mode with structured hypotheses.

3. PyTorch CUDA packages (torch+cu130, torchvision+cu130) not available on standard PyPI causing pip installation failures [Conda Environment Migration] (2026-05-11)#

Solution: Added –extra-index-url https://download.pytorch.org/whl/cu130 to pip dependency sections in all yml files to access PyTorch’s CUDA-specific package index.

4. Source-installed packages (mimicgen, robosuite, gadget) not available on PyPI causing conda env create to fail during pip phase [Conda Environment Migration] (2026-05-11)#

Solution: Implemented two-phase strategy: create conda-only yml files (removing pip section), install conda packages first, then pip install from filtered requirements.txt files, excluding unavailable packages.

5. AI blindly retries or continues modifying on test failure without reasoning process or user confirmation [Unified Agentic Workflow System] (2026-05-11)#

Solution: Designed Debug Mode that PAUSEs on verification failure, generates structured hypotheses (Because X → Therefore Y with evidence and confidence), outputs terminal summary + HTML detailed report, waits for user direction confirmation.

6. Missing cross-session context recording preventing next session or other agents from knowing what was done and why [Unified Agentic Workflow System] (2026-05-11)#

Solution: Implemented dual-format review logs: human version .md (narrative suitable for review/PR descriptions) and agent version .agent.json (structured JSON with files_modified, rationale, reasoning_chain for program reading).

7. Summarize reports not being deployed to GitHub Pages - 72 days of reports missing from staging [Summarize Deployment Pipeline] (2026-05-12)#

Solution: Modified _cmd_merge_sync_all in daily.py to deploy all non-staged reports from entire reports directory, not just dates processed in current run, fixing historical backlog deployment.

8. GitHub Pages website not updating despite local builds succeeding - update.sh Step 9 silently failing [Summarize Deployment Pipeline] (2026-05-12)#

Solution: Re-cloned tzj2006.github.io repository into website/public/ directory after discovering it lacked independent .git, fixing silent push operation failures.

Learnings#

Architecture (architecture)#

  • Structured specs with mandatory fields (Goal, Scope, Constraints, Success Criteria, Non-goals) prevent AI misunderstanding better than natural language prompts. Success criteria should be executable definitions of ‘what success looks like’ - conceptually identical to verification gates, not separate steps.
  • Community mature tools (Spec Kit 95K stars, Entire CLI 4.3K stars, git-cliff 11.8K stars) are more reliable through large-scale validation than custom builds. Prioritize adoption, only build custom for gaps (like debug mode) and glue layer (like AGENTS.md protocol).
  • Cross-session context transfer needs dual format: human version (narrative, suitable for review/PR) and agent version (structured JSON, suitable for program reading). Single format cannot serve both needs. Recording original plan in review log enables ‘intent vs result’ comparison for reflection and improvement.
  • Subagent-Driven Development excels at parallel execution of independent tasks: dispatch fresh subagent for each task (independent context), do two-stage review after completion (spec compliance → code quality), avoiding main session context pollution.
  • AGENTS.md as cross-agent protocol is more universal than SKILL.md, readable by Codex/Cursor/Claude/Copilot. .cursorrules is Cursor-specific supplement - combining both covers all agents.

Debugging (debugging)#

  • AI’s all judgments must have transparent reasoning chains (‘because X → therefore Y’ with evidence and confidence levels). Pausing for user confirmation on failure is more efficient than blind retries. Structured hypotheses force AI to make reasoning explicit.
  • When debugging deployment pipelines, verify the entire infrastructure chain (code logic + file paths + git repos + remote endpoints), not just code logic. AI can find code bugs but may miss infrastructure-layer issues without end-to-end verification.
  • Gitignored directories that should be independent git submodules need explicit verification during debugging. Absence of .git in expected locations is a critical diagnostic signal for silent deployment failures.

Domain Knowledge (domain)#

  • PyTorch CUDA packages require explicit index URL configuration (–extra-index-url https://download.pytorch.org/whl/cu130). Standard PyPI only hosts CPU versions, and +cu130 version strings will fail pip resolution.

Tools (tools)#

  • Conda environment migration with mixed conda/pip dependencies is fragile. Two-phase approach (conda packages first, then pip separately with error tolerance) provides more control and recovery options than integrated conda env create. Source-installed packages need manual exclusion and reinstallation.
  • Windows path manipulation in bash/sed is error-prone due to backslash escaping complexity. For complex string replacements, delegating to native PowerShell commands (Get-Content | .Replace() | Set-Content) is more reliable.

AI Usage Notes#

Effective Patterns:

  • ✓ Subagent-Driven Development for parallel independent tasks with isolated contexts
  • ✓ Dual-format review logs (human .md + agent .json) for cross-session context transfer
  • ✓ Structured hypothesis reasoning in debug mode forcing transparent ‘because→therefore’ chains
  • ✓ Root cause analysis through systematic code tracing combined with infrastructure verification
  • ✓ Two-phase problem solving: identify code bugs first, then verify infrastructure layer

Limitations:

  • ✗ Background task outputs (run_in_background: true) frequently truncated or lost, requiring fallback to foreground execution
  • ✗ Initially underestimated community tool maturity, leaned toward custom build for control rather than prioritizing proven solutions
  • ✗ Tendency toward over-layered design, splitting conceptually unified concepts into separate independent components

  • ✗ Did not proactively verify end-to-end deployment path before declaring fixes complete
  • ✗ Didn’t proactively include original plan in review logs for intent vs result comparison

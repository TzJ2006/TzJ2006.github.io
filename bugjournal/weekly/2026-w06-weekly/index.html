问题与解决方案#

1. STAIG Strict 模式 ARI 仅 ~0.09，与原始 STAIG（~0.45+）差距巨大 [MIHD] (2026-02-07)#

解决方案: 定位并修复 4 个关键差异：UNI encoder 跳过 ImageNet 归一化（原始 STAIG 只用 ToTensor）、patch 尺寸改为 256×256、去除 projection head 后的 StandardScaler、drop_feature 随机 mask 强制在 CPU 生成。修复后 ARI=0.5435。

2. apply_fusion() 函数签名缺少 **kwargs，staig_fusion_e2e 策略运行时抛出 NameError [MIHD] (2026-02-05)#

解决方案: 在函数签名末尾添加 **kwargs，在 staig_fusion_e2e 分支通过 kwargs.get(‘gene_expr_raw’) 读取传入的原始基因表达。

3. mclust 在 12 个 section 循环调用中随机报错 ‘Error in svd(data, nu=0): a dimension is zero’ [MIHD] (2026-02-07)#

解决方案: 用 try/finally 配对 numpy2ri.activate()/deactivate()，防止转换器栈积累损坏；同时过滤零方差列、替换 NaN/Inf，强制转为 float64 连续数组。

4. experiments_log.md 仅记录 ~14 条实验，36 条 CSV 数据大量缺失，最佳结果 ARI 0.528 完全未被记录 [MIHD] (2026-02-03)#

解决方案: 从 experiment_comparison.csv 完整重建 experiments_log.md，覆盖所有 36 条记录，并新建 BENCHMARK_RESULTS_SUMMARY.md 汇总关键发现。

5. SameLabel@K 指标全部为 0，评估结果明显异常 [MIHD / QueST] (2026-02-08)#

解决方案: 修正计算逻辑：同标签检索率应在全体有效 spot（含 niche 成员）上计算，而非将 niche 成员排除后计算（排除仅适用于 Precision@K 的自匹配去偏）。

6. WWL 算法中 query 和 reference 子图节点 ID 冲突，共享哈希表时 key 碰撞 [MIHD] (2026-02-08)#

解决方案: 为 reference 子图节点添加偏移量 offset = adata_q.n_obs，保证两组节点 ID 全局唯一。

7. UNI2 加载时出现 shape invalid 错误，GPU 非确定性导致实验结果跨 run 不可复现 [MIHD] (2026-02-05)#

解决方案: UNI2 需显式传递完整 timm 架构参数（img_size、patch_size、embed_dim 等）；在所有关键模块入口统一添加 CUBLAS_WORKSPACE_CONFIG=’:4096:8’。

8. POT（最优传输库）在当前 conda 环境中未安装，阻塞 RM-Ideal 实现 [MIHD] (2026-02-08)#

解决方案: 实现 POT → scipy.optimize.linprog 的回退策略，无需新增依赖。

学习收获#

领域知识 (domain)#

  • 原始 STAIG 三个核心机制：①自适应边 Dropout（rand() >= edge_prob）使高相似度边更容易被丢弃，强迫模型学习跨域通用表示；②Tau 从高温度（26）衰减模拟退火；③GCN 以 feat=concat(gene_expr, img_emb) 为节点特征端到端训练，而非分阶段独立编码后融合——这三点共同构成 STAIG 性能的核心保障。
  • 原始 STAIG 使用 UNI 提取图像 patch 时不做 ImageNet 归一化（只有 Resize+ToTensor），这与大多数视觉编码器的标准做法相反；复现论文时必须严格对照原始预处理代码，不能依赖常识。
  • RM-Ideal（WWL 图核 + Wasserstein 距离）是衡量空间 niche 结构相似度的 ground truth 指标，与 embedding 余弦相似度相关但不等价（Spearman r~0.42），说明 embedding 仅中等程度地捕获了 niche 精细结构。SameLabel@K（cell type 检索率）与 Precision@K（与 RM-Ideal 排序的秩相关）是两个独立维度，不能用同一 mask 逻辑处理。
  • MIHD 与原始 STAIG 超参数不对齐（lr 0.001 vs 0.0005、num_layers 2 vs 1、num_epochs 550 vs 300、weight_decay 1e-4 vs 1e-5）是当前性能差距的主要根因；算法对齐后必须逐项对齐超参数才能复现论文最优结果。
  • PCA+UNI2+STAIG 是 MIHD 当前最优且最稳定的组合（ARI 0.45+）；GCN 独立预训练后融合效果差（ARI 0.09-0.37）的根本原因是 GCN 训练阶段缺乏多模态对齐信号，解决方案是实现多模态拼接的端到端训练。

调试 (debugging)#

  • rpy2 的 numpy2ri.activate() 在循环调用时必须配对 deactivate()，否则多次调用会损坏转换器栈，导致 numpy→R 数组维度错误（‘svd dimension is zero’）；应用 try/finally 确保清理。原始 STAIG 单次调用不暴露此问题，多 section 循环场景才会触发。
  • CPU 和 GPU 上的随机数生成器即使设定相同 seed，生成的序列也不相同；在要求精确复现训练过程时，随机 mask 的设备选择会影响最终结果，必须严格对齐。
  • 实验日志与实际运行结果之间容易出现时间戳或覆盖不一致问题；实验记录应与实验执行脚本耦合，自动写入而非手动补录，当前项目存在 ‘CSV 有数据但 markdown 无记录’ 的系统性风险。

架构 (architecture)#

  • WWL 算法要求 query 和 reference 子图共享哈希表并使用无碰撞的全局节点 ID，实现时需引入节点偏移量（offset = adata_q.n_obs）；Hamming 距离需按维度归一化（除以 K+1）才能保证 W ∈ [0,1]。

工具 (tools)#

  • CUBLAS_WORKSPACE_CONFIG=’:4096:8’ 必须在所有 GPU 模块入口统一设置，否则跨 run 结果不可复现；这是复现研究实验的隐性前提，应在项目初始化阶段统一配置。
  • timm 创建 UNI2 模型必须显式传递完整架构参数（img_size、patch_size、depth、num_heads、embed_dim 等），否则位置编码重采样时维度不匹配导致预训练权重加载失败。
  • 大图的 PIL→numpy 转换是高代价操作，在逐 spot 循环中重复执行是严重的性能反模式；应在循环外一次性转换，循环内用 numpy 切片（零拷贝），可实现 10-50 倍加速。
  • 在 HPC 多 conda 环境中，优先用已有依赖（scipy linprog）作为可选库（POT）的回退，而非强制安装新包，可提升代码在不同环境中的可移植性。

AI 使用备注#

有效模式:

  • ✓ 逐行对比原始论文源码定位实现偏差，系统性修复多个协同问题（STAIG 4 项对齐）
  • ✓ 并行启动多个子任务同时分析不同代码模块（STAIG ARI 差距根因定位）
  • ✓ POT → scipy 回退策略：在环境受限的 HPC 中设计无额外依赖的算法实现
  • ✓ 预分配输出张量 + numpy 零拷贝切片的性能优化模式

局限性:

  • ✗ 未能主动验证输出指标的业务合理性（SameLabel@50=0 通过代码检查但数值明显异常，需用户介入发现）
  • ✗ 混淆方法论本质差异：将 staig_fusion（原始两阶段）和 staig_fusion_e2e（自研端到端变体）概念混用，向用户推荐了错误命令
  • ✗ 未能主动识别隐性工程需求（CUBLAS 确定性配置缺失）
  • ✗ 在算法架构设计上理解不足，E2E 框架初始实现未考虑多模态特征在 GCN 输入层融合的根本设计
  • ✗ 未能主动同步更新实验文档，导致 experiments_log.md 滞后于 CSV 数据

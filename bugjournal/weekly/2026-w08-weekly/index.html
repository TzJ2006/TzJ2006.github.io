Issues and Solutions#

1. STEGO training loss was NaN throughout the process. A dense similarity matrix of 3639×3639 caused overflow when temperature=0.07 [MIHD] (2026-02-19)#

Solution: Two-step fix: Normalize the input image_emb; replace InfoNCE with a numerically stable version (reduce maximum value before taking logsumexp, increase temperature to 0.1)

2. MILWRM was incorrectly classified as an Image-Only method. The first AI summary mixed multi-modal methods with image-only results. The conclusions of the first research need significant revision [MIHD] (2026-02-19)#

Solution: Use WebFetch to read the full PMC text to confirm that MILWRM actually relies on gene expression. Specifically extract image-only data points from ablation experiments in articles like SpaConTDS

3. There was a double normalization bug in staig_fusion in eval_scan_fusion.py (StandardScaler used again within STAIGTrainer) [MIHD] (2026-02-20)#

Solution: Abandon fixing the independent script and directly use run_benchmark.py, which correctly passes staig_alignment_config. Reuse the verified path

4. In run_benchmark.py, the encoder was re-instantiated to extract embeddings every time, lacking pipeline-level caching support [MIHD] (2026-02-20)#

Solution: Integrate CacheManager from pipeline/cache_manager.py. Load cache before encoding and write it back after extraction. Skip encoder instantiation if hit

5. Lerobot 0.1.0 was incompatible with datasets>=4.0 (torch.stack reported TypeError: Column object replaced list) [Error Recovery Benchmark / Phoenix pi0.5] (2026-02-22)#

Solution: Downgrade datasets from 4.4.1 to 3.6.0; for datasets 4.0, change the return value of dataset[‘column’] from list to Column object. Lerobot expects a list/tuple of tensors

6. HuggingFace official download failed due to proxy (Squid 503), and Python download script couldn’t connect [Error Recovery Benchmark / Phoenix pi0.5] (2026-02-22)#

Solution: Use hf-mirror.com + wget instead. URL format: https://hf-mirror.com/datasets/{repo_id}/resolve/main/{file_path}，通过集群 HTTP proxy available at 40–200MB/s

7. Unstable access to Slurm GPU nodes: srun without –overlap caused commands to hang, direct SSH connection was unreliable, multi-partition submissions were rejected [Error Recovery Benchmark / Phoenix pi0.5] (2026-02-22)#

Solution: Standardize the workflow: source set-XY-I.sh → squeue → srun –jobid= –overlap; prohibit direct SSH to GPU nodes

8. In eval_scan_fusion.py, there was an anomaly in spatial coordinate dimensions (became (1,2)), causing errors for multiple fusion methods [MIHD] (2026-02-19)#

Solution: Abandon calling load_spatial_coordinates() (barcode matching failed). Instead, directly read coordinates from adata.obsm[‘spatial’]. Fix the error in unpacking return values of load_dlpfc_data

Learnings#

Domain Knowledge### Architecture#

  • SCAN is the most effective method for image-only spatial transcriptomics tasks, with an ARI of 0.303 and a gain of +20.6%. Its core advantage lies in offline feature k-NN mining, which decouples embedding learning from clustering. The 256-dimensional optimized embeddings, when fused with gene features, show strong complementarity (mean fusion +0.065 ARI).
  • Two-stage fusion is not necessarily superior to one-stage fusion: After compressing the visual embedding from 1536d to 256d using scan_cluster, the ARI of multi-modal fusion (0.313) is lower than that of direct concatenation (0.387). This indicates that self-supervised compression may lose the original feature diversity required for fusion tasks. The goals of self-supervised clustering optimization and downstream fusion tasks are inconsistent.
  • STAIG fusion uses StandardScaler+PCA preprocessing to absorb the effects of external embedding optimization (the difference in ARI between SCAN and UNI2_raw is only 0.005). Embeddings should be saved as raw embeddings before normalization, and post-encoder normalization should be performed after loading to ensure consistent results across different call paths.
  • VLA baseline evaluations must clearly state the source of checkpoints: Models fine-tuned on non-target datasets (pi0_libero, pi05_base) are evaluated for zero-shot cross-domain recovery capabilities. The paper must explicitly state this experimental setup, and additional comparative experiments after fine-tuning are needed to fully demonstrate the usefulness of the dataset.
  • In complex experimental systems, it is advisable to reuse existing, validated tool paths (such as run_benchmark.py). Independent scripts can easily introduce hidden bugs due to inconsistent preprocessing. Before calling internal functions within the project, the source code should be read in real-time to verify function signatures, rather than relying on memory.

Debugging#

  • When calculating large-scale dense similarity matrices (n>3000) using InfoNCE with comparative loss, it is necessary to use numerically stable log-sum-exp (minus the row maximum). Otherwise, exponential operations with temperature=0.07 under float32 precision may overflow and cause NaN values. This is a key engineering constraint for implementing large-scale comparative learning.

Tools#

  • Lerobot 0.1.0 has strict constraints on dataset versions; it should be pinned to <4.0 (recommended 3.x). Dataset 4.0 changes the return value of dataset[‘column’] to a Column object, causing a TypeError in torch.stack. MimicGen is fully compatible with LIBERO obs/action format (84×84 images, 8D states, 7D actions), and can directly reuse OpenPI’s LeRobotLiberoDataConfig.
  • The standard way to access HuggingFace from Chinese mainland HPC clusters is via hf-mirror.com. The URL format is: https://hf-mirror.com/datasets/{repo_id}/resolve/main/{file_path}，通过 wget + HTTP proxy with a speed of 40–200MB/s. This should be the default approach.
  • OpenPI’s JAX training natively supports multi-GPU data parallelism. Just specify the list of GPUs using CUDA_VISIBLE_DEVICES, and JAX will automatically build a 2D mesh parallel without modification to TrainConfig. Slurm –overlap is a key parameter for running new commands on existing interactive jobs, and it is a core technique for calling cluster GPU nodes in the Claude Code environment.

Notes on AI Usage#

Effective Patterns:#

  • ✓ Parallel sub-tasks are used to explore project structures and develop cleanup plans (MIHD organizes tasks), reducing the time spent on serial exploration.
  • ✓ Three methods (STEGO/BYOL/SCAN) are executed in parallel on GPUs to compare and verify, significantly reducing the total experiment time.
  • ✓ Using .npz caches allows for batch re-generation of visualizations, completely decoupling inference from visualization. This worked well with 11/11 sections.
  • ✓ Independent temporary scripts (_test_staig_scan.py) isolate tests for individual fusion methods, reducing the task duration from 2 hours to 30 seconds.
  • ✓ Independent exploration of JAX’s multi-GPU mechanism (no need to modify config) and alternative download solutions via hf-mirror.com are independent tasks that do not require user intervention.
  • ✓ The minimum invasive code modification (about 60 lines) is made in run_benchmark.py by inserting the Vision Refinement stage, maintaining the stability of the architecture.

Limitations:#

  • ✗ Inaccurate classification of literature: MILWRM was incorrectly classified as an image-only method. Initial research results were mixed with multi-modal results, requiring two interventions from users to converge to the correct research boundary.
  • ✗ Lack of awareness to actively expose key experimental assumptions: The paper did not explicitly state how VLA evaluation using LIBERO fine-tuned checkpoints affects the validity of the experiment. This topic was only discussed after user inquiries.
  • ✗ API usage tends to rely on memory rather than real-time verification: eval_scan_fusion.py had multiple errors in function signatures/return values. The source code should be read in before calling.
  • ✗ Slurm permission issues led to multiple attempts to access partitions (5+ times). The correct account/partition information was not quickly obtained from users.
  • ✗ Insufficient status monitoring of background tasks: Data conversion was still ongoing when LeRobot dataset verification was triggered, resulting in false timestamp violation errors. Frequent sleep polling was interrupted multiple times.
  • ✗ In the HPC environment, CPU is used for model validation by default, hiding real performance issues and causing additional interaction problems.

问题与解决方案#

关键问题#

1. Sawyer夹爪get_gripper_closed_norm()返回值始终接近0，导致所有夹持检测和相位判断链式失效（pipeline静默运行但输出为空）#

解决方案: 定位到open_qpos=[0,0]与close_qpos=[0.04,-0.04]均值恒为0的除零根因；改用第一手指绝对位置归一化，并将全部7个skill文件和2个validator的夹持阈值从0.5/0.7降至0.05/0.10/0.15；grasp_closed相位阈值从0.7降至0.05并添加近距+高度双条件fallback

关键洞察: 对称双指夹爪不能用手指均值判断开合；相位检测「静默失败」（无报错只是输出为空）是最难调试的类型，需要专项debug脚本逐帧打印相位结果才能发现链式断点

2. Drop Skill注入后物体不下落（delta_z≈0），成功率0%#

解决方案: 诊断出robosuite OSC控制器在action=zeros时主动维持EEF位置，物体因摩擦悬停；改用直接修改MuJoCo sim.data.qpos（夹爪关节+物体位置偏移）和sim.data.qvel（向下初速度）绕过控制器

关键洞察: 机器人仿真中「打开夹爪」≠「物体脱落」；实现Drop类error必须直接操纵MuJoCo物理状态，不能靠控制器命令；MuJoCo接触力是瞬态的，碰撞类error的验证应改用位移量（EEF+物体移动距离）而非接触力

3. VLA评估中直接快照恢复让模型从空白历史开始，评估结果无意义（v4方法论根本性缺陷）#

解决方案: 设计ContextReplayEngine：Step A重置→Step B回放N-1帧建立VLA上下文窗口→Step C注入错误→Step D VLA自由推理

关键洞察: VLA是有状态的时序模型，评估框架必须感知其历史依赖；这是v4→v5最根本的架构重构动机，不是优化而是必要性修正

4. MIHD Patch Query结果坐标标注与实际patch位置不一致#

解决方案: 追踪数据流定位三处独立bug：统一加载路径与cache创建一致；通过barcode dict显式对齐RM-IDEAL分数；从adata.obsm[‘spatial’]获取坐标而非cache coords

关键洞察: cache中barcode顺序与scanpy重新加载的adata顺序不能假设一致，必须通过barcode显式映射，不能依赖位置对齐

5. MIHD代码文档双重不一致：STAIGTrainerE2E.py已删除但6个文件仍引用（运行时ImportError）；README中6处维度描述与实际代码不符#

解决方案: 清理所有14处staig_fusion_e2e引用（选择清理而非恢复文件）；系统性对照models/Fusion.py和GeneEncoders.py源码验证每处维度，以代码实测值为准重写README

关键洞察: AI辅助生成的文档易出现维度值「合理化」错误（写看起来合理的数而非实际值）；孤立的文件删除在多模块项目中会产生隐蔽的运行时错误，不会被语法检查捕获

6. RoboBrain Pi训练loss震荡：MSE loss在全32维（含18维零填充）取均值稀释真实动作梯度56%；task_pos_weight=1.0无法应对5~20%正类率#

解决方案: 建议将loss限制在前14维真实动作维度；task_pos_weight调整至5.0（接近负正比）。确认动作维度处理和任务完成分类代码整体逻辑正确

关键洞察: flow matching的MSE loss若对含padding的全维度取均值会严重稀释真实动作梯度；类别不平衡在pos_weight=1.0时会使模型学到全预测负类的trivial solution

7. 24种错误类型中约8种无法通过物理力injection可靠生成（wrong_object/joint_limit/repeated_failure等）#

解决方案: 分层处理：7种高可行性类型直接injection生成；6种中等类型改造注入；8种不可行类型依赖natural_capture（policy rollout + PolicyErrorMonitor自动检测）

关键洞察: 基于物理动作的错误（掉落/位置偏移/卡住）可injection；基于策略决策和时序行为的错误（错抓物体/关节极限/反复失败）只能natural_capture；v5 Error Skill框架将两种来源统一到同一抽象层

8. Kubernetes容器内nvidia-smi显示GPU进程PID为[Not Found]，无法通过PID kill进程#

解决方案: 根因是Kubernetes PID命名空间隔离：nvidia-smi通过驱动直接看到宿主机PID，但这些PID在容器内/proc中不存在。通过ps aux查看容器内自己的PID；开发gpumon.py用psutil+pynvml+rich提供可视化监控

关键洞察: nvidia-smi能看到显存占用（直接查询驱动）但无法解析进程名（依赖宿主机PID命名空间），这是容器安全隔离设计而非bug

9. 文档体系膨胀：AI按功能模块创建6个专项guide后docs/文件数翻倍，超出用户维护意愿；ccusage无法获取LiteLLM最新定价（fallback到46个模型）#

解决方案: 用户明确只保留4个文档，将6个新建guide归档并内容合并至4个核心文档。ccusage：Node.js v24内置fetch不自动读取HTTPS_PROXY，添加NODE_OPTIONS=’–use-env-proxy’解决

关键洞察: 个人/小团队项目的文档组织应以「用户查找路径」为核心而非功能分类；curl（libcurl）与Node.js fetch（undici）对代理环境变量的读取行为不同

一般问题#

10. CUDA_VISIBLE_DEVICES=6重映射后MUJOCO_EGL_DEVICE_ID越界（got 6, expected 0-0）#

解决方案: 在main.py顶部添加os.environ.setdefault(‘MUJOCO_EGL_DEVICE_ID’, ‘0’)，使EGL设备ID与CUDA重映射后的实际可见GPU索引（0）一致

关键洞察: CUDA_VISIBLE_DEVICES仅对CUDA设备索引重映射，EGL渲染器独立读取物理设备ID，两者需分别配置

11. Windows/tmux终端中文渲染为下划线（多个会话均遇到，本地echo与tmux内均异常）#

解决方案: 根因是Windows本地终端code page非UTF-8，而非服务器或tmux配置问题。解决：chcp 65001或Windows Terminal；服务器端locale/tmux配置无需修改

关键洞察: 中文乱码需区分三层（本地终端编码/SSH传输/服务器locale），「同服务器他人正常但本机异常」是快速定位到客户端侧的关键横向对比

  • 在v5 pipeline仿真调试中多次从表象（阈值过高）入手修改，而未能一开始就系统性追查Sawyer夹爪归一化这一根本bug，导致多轮GPU测试时间浪费；quat_rotate_vector实现也引入了归一化工具函数的隐蔽副作用（中间量[0,v]被错误归一化）
  • 在分析24种错误类型可行性时将8种标注为「不可行」但未主动提出新的injection机制覆盖；初始pipeline规划以100/subtype为默认目标，未主动建议先小规模MVP验证；代码实现完成后未主动推进到端到端运行验证，需人类追问才意识到执行gap

一般局限#

  • ExitPlanMode时机判断在多个会话中被反复拒绝（MIHD、Error Recovery Benchmark、RoboBrain Pi等均有），AI将「计划写入文件」与「用户审核就绪」视为等价；过度依赖AskUserQuestion进行多选项提问，被拒绝后未及时调整为自主假设推进
  • serve_policy.py问题诊断时优先追问非致命warning（traj_proj）而非用户真正关心的socket崩溃问题，优先级判断偏差导致用户中断；Glob工具文件搜索多次超时或失败，需回退到bash find命令

今日收获#

核心收获#

  • robosuite OSC控制器在action=zeros时会主动维持EEF位置，「打开夹爪」不等于「物体脱落」；实现Drop类error必须直接操纵MuJoCo物理状态（sim.data.qpos/qvel），不能靠控制器命令。MuJoCo接触力是瞬态的，碰撞类error的验证应改用位移量作为证据
  • VLA评估的核心约束：必须通过ContextReplay（回放N-1帧）建立VLA观测历史窗口，不能直接从快照恢复；否则VLA的时序注意力机制无法工作，评估结果无意义。这是v4整体方法论的根本性缺陷，也是v5.0架构重构的核心动机
  • 对称双指夹爪不能用手指均值判断开合状态（均值始终为0），必须用单指绝对位移；相位检测的「静默失败」（pipeline正常运行、无报错、只是输出为空）需要专项debug脚本逐帧打印相位结果才能发现链式断点
  • scGPT checkpoint loading关键bug：TransformerModel未保存use_fast_transformer属性导致Wqkv→in_proj_权重重映射未执行，17.7%的attention权重随机初始化。修复后DLPFC 11 section平均ARI提升+44%（0.316→0.454），验证了checkpoint加载细节对Foundation model性能的决定性影响
  • Error Skill自包含范式（can_inject+inject+validate三合一）相比v4分离的Detector/Injector/Validator，实现新错误类型时只需一个文件、一个注册表，内聚性大幅提升；D0/D1/D2分类维度（仅位置/仅旋转/两者）为机器人操作错误提供了统一几何描述框架，便于结构化复用validator逻辑
  • 大规模数据生成任务应先做10/subtype的MVP验证跑：确认每个skill在真实环境中能找到injection机会、能产生有效错误、validate能正确判定，再扩展到100/subtype，避免在错误的pipeline上浪费GPU时间
  • flow matching的MSE loss若对含padding的全维度取均值会严重稀释真实动作梯度（本项目14/32=44%有效）；任务完成分类的类别不平衡（episode末尾固定N帧为正类）在pos_weight=1.0时会使模型学到全预测负类的trivial solution，pos_weight应设为接近负正比
  • Foundation model在下游任务中的悖论：scGPT Spatial（512d）在聚类任务中系统性不如PCA（50d），原因是Foundation model训练目标是跨组织/跨物种的可迁移表示，优化方向与最大化当前数据方差相反；「更大的预训练模型」不等于「更好的下游聚类效果」
  • Error注入可行性分层规律：DROP/部分COLLISION/POSITIONING/STALL中的stuck_contact可通过impulse/friction/pose_perturb物理注入可靠生成；GRASP_FAILURE/SEQUENCING/TRAJECTORY/REGRESSION类错误依赖policy决策和时序行为，只能通过natural_capture（policy rollout + PolicyErrorMonitor检测）获取
  • CUDA_VISIBLE_DEVICES仅对CUDA设备索引重映射，EGL/MuJoCo渲染器独立维护自己的设备ID（MUJOCO_EGL_DEVICE_ID应设为重映射后的逻辑ID，通常为0）；Kubernetes/containerd容器的PID命名空间隔离机制影响所有依赖/proc查询进程信息的监控工具，需用psutil+pynvml绕过
  • STAIG fusion直接使用原始HVG表达矩阵（staig_gene_feat），完全不使用gene encoder输出，因此在staig_fusion实验中设置不同gene_encoder对结果无实质影响；RM-IDEAL算法核心：WWL消息传递→Hamming距离代价矩阵→Wasserstein最优传输→RM=1-W∈[0,1]，无需squidpy/torch_geometric依赖

实践收获#

  • pip install -e.（editable install）是conda环境隔离的，egg-info目录存在不代表当前激活环境已安装；多人共享Linux账户的服务器上，工具凭证（wandb等）存储在用户级配置（~/.netrc），需通过进程级环境变量（WANDB_API_KEY）实现账号隔离；Node.js v24内置fetch需–use-env-proxy才能读取HTTPS_PROXY，而curl/wget默认自动读取
  • 四元数旋转v’=q*[0,v]*q^{-1}中的中间量[0,v]不是单位四元数，不能使用自动归一化的乘法工具函数；需独立实现不含归一化的raw Hamilton积。调度算法的per-trajectory上限在轨迹数量少时会成为覆盖瓶颈，需根据实际轨迹数量动态调整

问题与解决方案#

关键问题#

1. NSVisualEffectView 无法为 AVPlayerView 硬件加速视频提供毛玻璃效果：.withinWindow 无法采样 Metal/CoreVideo 渲染内容，SwiftUI .ultraThinMaterial 内部使用 .behindWindow 采样桌面而非窗口内视频#

解决方案: 将时钟移至独立 NSWindow，使用 .behindWindow 混合模式，由窗口服务器在合成层面处理模糊，可正确透视所有后方窗口内容（包括 Metal/CoreVideo 渲染）

关键洞察: AVPlayerView 通过 Metal/CoreVideo 在独立 compositing surface 渲染，不参与同窗口内的内容采样；.behindWindow 在窗口服务器层面工作，是唯一能突破此限制的方式

2. NSHostingView.cacheDisplay 生成空白位图，renderTextMask 始终返回 nil，时钟完全不可见#

解决方案: 改用 NSAttributedString + NSImage 绘制路径，完全绕开 SwiftUI/CALayer 渲染机制

关键洞察: NSHostingView 完全通过 CALayer 渲染，不实现 draw(:)；cacheDisplay 依赖 draw(:)，对 NSHostingView 永远返回空白 bitmap

3. 视频切换后仍播放旧视频：书签按屏幕 ID 存储，解析出旧视频 URL，allowReuse 判定复用旧播放器#

解决方案: 在书签解析后对比解析结果与请求 URL，不一致则返回原始 URL（NSOpenPanel URL 自带临时沙盒访问权限）

关键洞察: NSOpenPanel 返回的 URL 自带临时沙盒权限，无需书签；书签只在 saveBookmark() 中更新，应信任 Panel 来源的 URL

4. URL 校验 guard 引入回归：重启后历史记录 URL 无沙盒权限，guard 直接返回无权限 URL 导致黑屏#

解决方案: 需区分 URL 来源：NSOpenPanel 来源可直接使用，历史记录来源必须走书签恢复路径（即使路径不同也应尝试书签获取权限）

关键洞察: 两类 URL 的沙盒权限模型完全不同，不能用统一逻辑处理；修复切换 bug 时过度简化了判断逻辑，未考虑持久化场景

5. reloadAndPlayVideo 后时钟窗口被视频窗口覆盖，时钟消失#

解决方案: 视频重载完成后遍历 clockWindows，调用 clockWin.order(.above, relativeTo: wallpaperWin.windowNumber) 重排层叠顺序

关键洞察: showVideo → ensureWallpaperController → orderFrontRegardless() 会将壁纸窗口无条件置顶，独立时钟窗口不会自动跟随，需手动维护层叠关系

6. 每秒更新遮罩时 CALayer 隐式动画（~0.25s 交叉淡出）产生灰色数字残影#

解决方案: 在 contents/frame 赋值前后用 CATransaction.begin() + setDisableActions(true) + commit() 禁用隐式动画

关键洞察: CALayer 默认对 contents 属性执行约 0.25s 交叉淡出；旧遮罩在新遮罩渐入过程中以灰色形式显现，且字重不一致会进一步扩大毛玻璃溢出范围

7. 三处字重不一致（Highlight .light vs Mask .regular）导致毛玻璃从文字边缘溢出#

解决方案: 将三处时间字重统一为 .regular，保持遮罩形状与高光文字完全对齐

关键洞察: 遮罩决定 NSVisualEffectView 的裁剪边界；遮罩比高光更粗时，灰色毛玻璃会从高光文字边缘露出

一般问题#

8. 历史视频缩略图在沙盒环境无读取权限，ThumbnailGenerator 静默返回 nil#

解决方案: 在读取文件前从 BookmarkStore 获取 per-URL 书签并激活安全作用域访问，通过 defer 释放

关键洞察: 沙盒 app 读取历史文件需显式激活安全作用域书签；AVAsset 和 NSImage(contentsOf:) 不会自动获取权限

AI 局限性#

重要局限#

  • 修复视频切换 bug 时未充分考虑 NSOpenPanel URL 与历史记录 URL 的沙盒权限模型差异，一个修复破坏了另一功能（历史视频黑屏回归），暴露了对跨场景副作用评估不足的问题
  • 对 AVPlayerView 硬件加速渲染与 NSVisualEffectView 兼容性预判不足，先后尝试 .withinWindow 和 SwiftUI material 等无效方案，需多轮失败迭代后才确定正确架构
  • 使用 NSHostingView.cacheDisplay 快照 SwiftUI 视图，未预见 NSHostingView 不实现 draw(_:) 的基础限制，导致 renderTextMask 始终返回 nil 并引入「时钟完全不可见」的严重 bug
  • 主动提出的 CABasicAnimation crossfade 方案未达预期，反而引入灰色重影新 bug；过度信任理论分析，需依赖人类实时视觉反馈才能发现问题
  • 获得明确实现计划时仍反复启动多 Agent 重新规划，过度分析而非执行；且集群训练状态调查在 session 内未能完整输出结构化结论，用户需求未完全闭环

今日收获#

核心收获#

  • AVPlayerView 通过 Metal/CoreVideo 在独立 compositing surface 渲染，不参与同窗口内容采样。为硬件加速视频添加毛玻璃效果的正确模式：独立透明 NSWindow + .behindWindow NSVisualEffectView，由窗口服务器在合成层面处理，可突破所有渲染层隔离限制
  • NSHostingView 完全通过 CALayer 渲染，不实现 draw(:)；cacheDisplay 对其永远返回空白 bitmap。需要 bitmap 快照时须改用 NSAttributedString 绘制或 CALayer.render(in:)，不能依赖任何基于 draw(:) 的快照机制
  • 沙盒权限有两种来源，必须分别处理：NSOpenPanel 返回的 URL 自带会话级临时访问权限（可直接使用）；历史记录/重启场景必须通过 security-scoped bookmark 恢复持久权限。用统一逻辑处理两类来源是常见的回归来源
  • CALayer 的 contents/frame 属性默认有约 0.25s 隐式动画；需要即时更新遮罩/内容时，必须用 CATransaction.begin() + setDisableActions(true) + commit() 包裹赋值操作，否则产生视觉残影
  • macOS 独立覆盖窗口架构中，任何调用 orderFrontRegardless() 的操作都会打乱层叠顺序；需在每次视频重载/窗口刷新后主动调用 order(.above, relativeTo:) 重排所有覆盖层，不能假设层叠顺序会自动保持
  • Pi0.5 在 Stack 类任务上泛化良好（step 17000 时 Stack D0 100%、D1 95%），但对 Coffee 和 ThreePieceAssembly 等高精度任务仍需更多训练步数或更大数据集；BC-RNN 全部 9 任务 epoch 600 已完成，可作为基线对比

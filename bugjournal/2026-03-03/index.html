问题与解决方案#

关键问题#

1. Pi0.5 LoRA 微调 base model 疑为 pi0.5_libero 而非 pi0.5_base，影响已完成约 36k 步训练的有效性；调查 subagent 因 OAuth token 过期（401）全部失败#

解决方案: 需后续手动核查 training/config.py weight_loader；token 过期通过 /login 命令重新获取；长会话前应提前验证 token 状态

关键洞察: VLA 微调前必须验证 base checkpoint 路径；libero 域与 MimicGen 域存在显著分布差异，错误初始化权重可能使整批训练失效，应将此列为标准流程必检项

2. 151676 STAIG fusion Phase 2 输出嵌入全为零（GCN 训练坍塌），导致 Spearman r 为 NaN；同时 P100 GPU (sm_60) CUDA kernel 不兼容当前 PyTorch，torch.cuda.is_available() 返回 True 但实际 CUDA 操作失败#

解决方案: 添加零嵌入检测（norm < 1e-6 比例检查）自动触发 recompute；添加 try/except 包裹 CUDA 测试；用户切换到 RTX 2080 Ti (sm_75) 重训 300 epochs 获得有效 64 维嵌入

关键洞察: STAIG GCN 训练存在模型坍塌风险，Phase 2 npz 文件存在 ≠ 嵌入有效；应将 norm 质量检查作为嵌入加载标准验证步骤。torch.cuda.is_available() 不能代替实际 CUDA 操作验证；sm_60 (Pascal) 不被新版 PyTorch 支持

3. NSHostingView.cacheDisplay 生成空白 CGImage（NSHostingView 通过 CALayer 渲染，不调用 draw()），时钟毛玻璃效果完全不显示；reloadAndPlayVideo 后时钟窗口被壁纸窗口覆盖#

解决方案: 改用 NSAttributedString 绘制到 NSBitmapImageRep 生成 CGImage，完全绕过 SwiftUI/NSHostingView 快照路径；在视频重载后将 clockWindows 重新 order(.above, relativeTo:) 到壁纸窗口上方

关键洞察: NSHostingView 通过 CALayer 树渲染 SwiftUI 内容，NSView.cacheDisplay 无法捕获；独立 NSWindow overlay 需在每次底层窗口 orderFrontRegardless() 后主动重新维护 z-order

4. SwiftUI .ultraThinMaterial 锁定为 .behindWindow 无法透视 AVPlayerView 视频内容；ThumbnailGenerator 在沙盒下未激活安全作用域书签，缩略图静默返回 nil；视频切换后书签仍指向旧视频#

解决方案: 用 NSViewRepresentable 封装 NSVisualEffectView 并指定 .withinWindow；新增 activateBookmark() 在读取前激活 per-URL 书签；ensureFileAccess() 中比较 resolved.path 与 url.path，不一致时返回原始 URL

关键洞察: .withinWindow 由 macOS compositor 处理，可透视 Metal/CoreVideo 硬件加速视频层；沙盒书签违规在 AVAsset 层面静默失败不抛异常；NSOpenPanel URL 自带临时沙盒权限无需书签

5. BC-RNN 在所有 9 个 MimicGen 任务上 clean baseline 成功率全部为 0%（各 50 次试验），而 pi0.5_libero base 在 Stack_D0 已有 24%，评估流水线疑存在系统性问题#

解决方案: 记录并标记为可疑异常，尚未深入排查根因（可能为 task_registry 配置错误、checkpoint 路径问题或任务初始化 bug）

关键洞察: 专门训练的 BC-RNN 反而低于通用预训练 VLA，强烈指向评估流水线 bug 而非模型能力不足；应优先排查后再使用此数据作为 baseline

6. 屏保时钟 CALayer mask 边缘灰色重影：SwiftUI 字重与 NSAttributedString mask 字重不一致，NSVisualEffectView 材质从边界外溢出；CALayer crossfade 动画新旧 mask 共存期间额外暴露毛玻璃材质；CALayer contents 属性隐式动画导致 0.2s 时钟数字残留#

解决方案: 统一 SwiftUI 与 NSAttributedString 字重（均为 .regular）；移除 crossfade 动画，改回即时替换；用 CATransaction.setDisableActions(true) 包裹 mask 更新禁用隐式动画

关键洞察: NSVisualEffectView 毛玻璃在 mask 边界之外仍可见，字形必须精确匹配；CALayer mask 不适合 crossfade 动画；CALayer 对 contents 有默认约 0.25s 隐式动画，每秒刷新场景必须显式禁用

7. gadget 日报流水线三类工程问题：LLM 输出截断导致 JSON 不完整；Claude CLI 嵌套 session 检测拒绝执行；多项目合并报告反复触发 32000 token 输出上限#

解决方案: 截断 JSON 通过 AI 修复工具恢复；subprocess 环境中移除 CLAUDECODE/CLAUDE_CODE_ENTRY 环境变量绕过嵌套保护；拆分多项目合并（先按设备分段，再分步合并）规避 token 上限

关键洞察: LLM 生成超长 JSON 需加入截断检测和修复重试逻辑；Claude CLI 嵌套 session 场景需显式解除保护；32000 token 上限是日报合并流水线工程瓶颈，应引入自动分段策略或调整 CLAUDE_CODE_MAX_OUTPUT_TOKENS

一般问题#

8. an53 节点 08:30 发生 NODE_FAIL，tmux 会话全部丢失；Tianhe 集群 an49/an53 SSH 受 pam_slurm_adopt 限制，无 active job 时拒绝访问#

9. Visium 可视化脚本 4 轮调试失败（Bash heredoc 转义、H5 文件名缺少 section 前缀、Dataset/DLPFC 路径层级错误、load_dlpfc_data() 返回值被误解包为元组）#

关键洞察: Visium 数据完整目录（含 spatial/）位于 Dataset/DLPFC/{section_id}/，H5 文件名带 section ID 前缀；复杂 Python 代码应写入临时文件而非内联传递

批量历史补报的流程编排与执行节奏控制#

角色 思路
人类 规划「一次性补全多天欠账」批量化策略；设计两阶段分段提交→合并去重流程；遇到 token 上限时主动拆分重试；对 AI 提出的 ExitPlanMode 方案多次拒绝（调整执行粒度）
AI 执行具体分析任务，识别会话主题，提炼 tasks/problems/learnings；在合并阶段进行内容去重归纳；遇到 token 上限时被动中断，无法主动预警内容量超限；被多次拒绝 ExitPlanMode 后未能有效调整方案粒度

差异分析: 流程编排和执行节奏控制完全依赖人类；AI 的 token 上限是客观约束，需在流程设计层面提前规避；AI 在感知用户当前执行意图和调整方案范围上存在不足

AI 局限性#

重要局限#

  • 对 VLA 微调配置缺乏语义理解：在统计 Pi0.5 训练状态时未识别 weight_loader 配置中 pi0.5_libero 与 pi0.5_base 的关键差异，仅关注步数和成功率指标，遗漏了最重要的配置正确性验证——这是整个训练计划中最重要的决策点
  • 对明显异常结果缺乏主动质量检查意识：BC-RNN 9 个任务全部 0% 时未立即标记为可疑并建议排查评估流水线，直接将其纳入结果汇总
  • macOS 渲染管线知识不足：错误使用 NSHostingView.cacheDisplay 快照 SwiftUI 内容（NSHostingView 通过 CALayer 渲染，不调用 draw()），导致生成空白遮罩需重写；添加 CALayer mask crossfade 动画时未预见新旧遮罩共存会暴露 NSVisualEffectView 底色，引入新 bug
  • LLM 单次输出存在 32000 token 上限，多项目合并报告时反复触发，无法主动预警内容量超限；超长 JSON 输出截断时不主动告知用户，需用户手动发现

一般局限#

  • 当被用户多次拒绝 ExitPlanMode 后，未能有效调整方案粒度（Desktop Video 历史视频黑屏、Tianhe 评估策略等多个场景均出现），说明在感知用户执行节奏和调整方案范围上存在系统性不足
  • 领域数据路径知识不足：对 Visium 数据目录结构不熟悉，可视化脚本需 4 轮调试；SSH 多行 Python 代码引号转义问题反复出现，多次失败后才改用临时 .py 文件；nohup SSH 命令需 3 次才成功（未预判登录到 home 目录导致相对路径失效）

今日收获#

核心收获#

  • macOS .behindWindow 混合模式由窗口 compositor 处理，可透视包括 AVPlayerView（Metal/CoreVideo 硬件加速）在内的所有窗口内容；.withinWindow 只能采样同一窗口层级 Cocoa 内容，无法透视硬件加速视频层；NSHostingView 通过 CALayer 渲染，NSView.cacheDisplay 无法捕获 SwiftUI 内容，应使用 NSAttributedString 直接绘制或 SwiftUI ImageRenderer
  • Pi0.5 LoRA 微调在 Stack 类任务上表现极佳（Stack_D0 100%、Stack_D1 95%），在精细操作任务上表现差（Coffee_D1 15%、ThreePiece 30%）；VLA 微调前必须验证 weight_loader base checkpoint 路径（libero vs MimicGen base），错误的域初始化权重可能使整批训练失效，应列为标准流程必检项
  • STAIG fusion 嵌入在 Layer_3 跨 section 评估中与 RM-IDEAL 拓扑分数呈中等正相关（r≈0.44），确认融合嵌入保留空间 niche 组织结构信息；但 Precision@K 接近 0 说明局部排序差异显著——RM-IDEAL 对拓扑精确匹配敏感，嵌入相似度更反映全局连续分布。Phase 2 npz 文件存在 ≠ 嵌入有效，应将 norm 质量检查列为标准加载步骤
  • macOS 沙盒安全作用域书签违规在 AVAsset/NSImage 层面静默失败（返回 nil，不抛异常）；书签通常按屏幕存储，历史记录文件需额外 per-URL 书签机制持久化权限，但两套书签之间存在冲突风险需整合设计
  • BC-RNN 在 MimicGen 任务上 clean baseline 全部 0%，而 pi0.5_libero base 在 Stack_D0 已有 24%，说明通用 VLA 预训练模型在某些操作任务上可能优于专门训练的 BC-RNN；Coffee 任务 BC-RNN 600 epoch 全程 0% SR 训练失败，说明 Coffee 场景复杂度超出 BC-RNN 能力边界，论文需特别说明

实践收获#

  • CALayer mask 文字字形必须与上层 SwiftUI overlay 精确匹配（字重、大小），任何偏差都会造成 NSVisualEffectView 材质溢出产生灰色轮廓；CALayer 对 contents 等属性有默认约 0.25s 隐式动画，每秒刷新场景必须用 CATransaction.setDisableActions(true) 显式禁用；独立 NSWindow overlay 需在每次底层窗口 orderFrontRegardless() 后重新维护 z-order
  • 批量历史日报处理揭示多项目并行工作全貌；export→merge 两阶段流水线关键约束是每次 merge 的 JSON 长度，需对超大报告引入分页或截断检测；Claude CLI 模型参数（sonnet vs opus）对日报生成质量影响有限，半结构化分析任务优先使用 Sonnet 是合理的成本优化
  • 天河超算 yhbatch/yhrun 等 yh* 前缀命令均为对应 Slurm 命令的 exec wrapper；pam_slurm_adopt 在计算节点本地 PAM 配置中启用，目标节点无活跃作业时拒绝所有 SSH 登录，srun –overlap 是访问非直连节点的标准方式；8 GPU 全满载时应监控系统内存，避免 NODE_FAIL

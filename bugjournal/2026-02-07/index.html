日报 — 2026-02-07#

在DCC服务器上对QueST和MIHD两个空间转录组项目进行了多项工程改进：为QueST生成了CLAUDE.md，修复了MIHD的mclust SVD崩溃bug和patch提取性能瓶颈，并通过定位4处关键差异成功将STAIG Strict模式的ARI从~0.09提升至0.5435。

今日任务#

架构与策略#

  • ✅ 对齐STAIG Strict模式与原始STAIG实现（4处修复） — 逐行对比MIHD与原始STAIG代码，发现4个导致ARI差距的关键差异并全部修复：①UNI encoder跳过ImageNet归一化（原始STAIG仅ToTensor）；②patch尺寸改为256×256并添加dynamic_img_size=True；③训练后去掉额外的StandardScaler；④drop_feature()强制在CPU生成随机mask。修复后ARI从~0.09大幅提升至0.5435。
  • ✅ 修复mclust SVD维度为零的崩溃bug — 诊断并修复utils/clustering.py中的mclust聚类崩溃问题：在12个DLPFC切片循环调用时numpy2ri.activate()未配对deactivate()导致转换栈损坏，以及缺少对零方差列的预处理。在try/finally块中配对activate/deactivate，新增NaN/Inf检查和零方差列移除。
  • ✅ 修复patch提取性能瓶颈 — 发现_extract_patches_from_image()在循环内对每个spot都将整张大图（~20000×20000）转换为numpy数组（~3500次），改为循环外单次转换，同时用numpy数组切片替代PIL crop，预分配输出张量，transforms对象只实例化一次。预估频域滤波路径提速10-50x。
  • 🔄 RM Ideal Score集成到QueST — 用户希望将RM_Ideal_implementation_notes.md中的实现集成到MIHD项目。AI开始探索codebase和graphcompass库API，但会话在探索阶段中断，未实际完成集成。

  • 🔄 对比224×224与256×256 patch尺寸对ARI的影响 — 用户在验证256×256 ARI=0.5435后，希望对比224×224的结果。由于–staig_alignment参数期望YAML文件路径而非内联JSON，首次尝试失败；后续遭遇API限额中断，实验未能完成。

  • ✅ 将STAIG对齐修复写入tutorial文档 — 在THOROUGH_TUTORIAL.md的Section 6中新增了关于4处对齐修复的详细说明，包括每处差异的原因、修复方式和验证结果。

问题与解决方案#

关键问题#

1. STAIG Strict模式ARI (~0.09) 远低于原始STAIG (0.45+)，尽管声称已完全对齐#

解决方案: 通过逐行对比代码发现4处差异：①原始STAIG用256×256且无ImageNet归一化，MIHD用224×224且有归一化；②MIHD在训练后多了一个StandardScaler；③drop_feature()在GPU而非CPU生成随机mask导致随机序列不同。全部修复后ARI=0.5435，超过原始STAIG水平。

关键洞察: 图像预处理的细节（归一化方式和尺寸）对下游所有步骤（PCA降维、伪标签生成、边权重计算、GCN训练）都有级联影响，单一差异就能让结果差距极大。

2. mclust报错’a dimension is zero’，section 151676的ARI为0%#

解决方案: 根本原因：在12个切片的循环中，每次调用cluster_with_mclust()都调用numpy2ri.activate()但从未deactivate()，导致rpy2转换栈积累损坏，到第151676个切片时numpy→R的矩阵转换出错。修复：用try/finally确保activate/deactivate配对，同时新增NaN/Inf清洗和零方差列移除（直接防止SVD看到0维矩阵）。

关键洞察: 原始STAIG只调用一次mclust_R()，而benchmark在循环中调用12次——相同的代码模式在不同使用场景下会产生完全不同的问题。

3. patch提取速度极慢，每个section需要数分钟#

解决方案: 发现在对每个spot的循环内调用np.array(pil_img)将完整高分辨率图像转numpy（~3500次）。将此操作移到循环外，并用numpy数组切片替代PIL crop操作，预分配输出张量，transforms只实例化一次。

关键洞察: 循环内的全图类型转换是最典型的Python性能反模式之一，但因为被嵌套在业务逻辑中，不容易被发现。

一般问题#

4. 在QueST代码库和论文中均无法找到’RM-ideal score’的定义#

解决方案: 通过搜索HTML版论文确认该论文使用的评估指标为’Best Niche Match Accuracy’和’Overall Similarity Accuracy (Pearson correlation)’，以及WWL图核作为ground truth。论文中的’RM score’实际指Region Matching score（基于Wasserstein Weisfeiler-Lehman图核），而非’RM-ideal’。

关键洞察: ‘RM-ideal’可能是用户对论文中某个指标的误记或混淆。AI无法访问PDF正文内容（只能访问摘要），限制了进一步确认。

今日收获#

核心收获#

  • 图像预处理的细节（归一化方式、输入尺寸）对基于对比学习的空间转录组方法有级联影响——它改变了视觉嵌入的分布，进而影响PCA降维、伪标签质量、空间图的边权重，最终决定GCN是否能学到有效表示。复现论文时必须逐行对比原始代码。
  • rpy2的numpy2ri.activate()在多次调用场景下必须配对deactivate()，否则在循环处理多个样本时会积累状态损坏。原始单次调用的代码在benchmark场景下会产生完全不同的问题。

实践收获#

  • 在Python图像处理循环中，将整图类型转换（PIL→numpy）放在循环内是一个极易被忽视的性能陷阱，对于高分辨率组织切片图像（~20000×20000像素）影响尤为显著。
  • QueST论文中评估niche查询性能使用的是WWL图核（Wasserstein Weisfeiler-Lehman）作为ground truth，以Best Niche Match Accuracy和Pearson相关系数（Overall Similarity Accuracy）作为评估指标，而非某个叫做’RM-ideal’的指标。

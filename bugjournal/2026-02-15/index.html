问题与解决方案#

1. website 目录不是 git repo，无法用 git push 推送源文件#

解决方案: deploy 通过 update.sh 脚本完成，脚本将构建产物 public/ 目录 push 到 GitHub Pages；源文件本身不需要独立版本管理

关键洞察: Hugo 网站常用的部署模式是只托管 public/ 构建产物，源文件可以不进 git

2. Q-Former 进程在 6/11 sections 后意外停止（未记录明确错误）#

解决方案: 检查进程状态（0 个运行中）后立即重启，pipeline 自动检测已完成的 6 个 sections 跳过，从第 7 个继续

关键洞察: 断点续跑机制的价值：进程异常终止后无需从头开始，通过检查 embeddings.npz 文件存在性即可恢复

3. scGPT Q-Former 评估出现 11 个 CUDA OOM 错误#

解决方案: pca Q-Former 占用约 21.5GB GPU，scGPT Q-Former 同时运行导致 OOM；需等 pca Q-Former 完成后单独重跑 scgpt_uni2_qformer

关键洞察: GPU 内存管理：单个 Q-Former 实例接近满载，两个并行必然 OOM；应按 GPU 资源串行调度训练型 fusion

4. 多设备 rclone 同步 — export log 和 merge 报告混在同一 flat 目录，多台 merge 会覆盖#

解决方案: 设计分层目录结构（remote/logs/ 存 export，remote/reports/ 存 merge 结果），新增 –sync 参数触发 rclone 下载，默认只 export 不 merge

关键洞察: 分层目录 + opt-in sync 是多设备工作流的正确抽象；覆盖风险来自扁平结构而非并发写入

5. rclone 配置提示示例使用 gdrive:summarize，用户的 remote 名可能不同#

解决方案: 将提示改为通用格式 <remote名>:<目录>，不预设 remote 命名

关键洞察: 配置提示中的示例值应通用化，避免用户误以为 remote 名必须叫 gdrive

6. VLA 研究中用户博客页面结构不标准，直接抓 /posts/ 返回 404#

解决方案: 先抓首页获取所有链接，发现 /bugjournal/ 路径（小写），再逐一访问各 bugjournal 页提取 VLA 内容

关键洞察: Hugo 生成的静态站路径大小写敏感，需先通过首页链接发现真实路径

人类思路 vs AI 思路#

部署流程的主导方式#

角色 思路
人类 主动在第二次会话中提供完整计划文本，绕过了第一次被中断的 ExitPlanMode 流程，直接驱动实施
AI 按照人类提供的计划逐步执行：读文件、生成 front matter、写入目标目录、运行 deploy 脚本

差异分析: 人类通过提前写好计划并在新会话粘贴来解决跨会话状态丢失问题；关键决策由计划文本预先锁定，AI 执行层面较顺畅

默认行为设计#

角色 思路
人类 明确提出’默认不要 merge’——无子命令时只 export，不调 API；这与最小惊讶原则一致（不加参数不触发付费操作）
AI 最初设计的 cmd_legacy 无子命令时会调 API 做完整总结，需要人类纠正

差异分析: 人类对默认行为更敏感（避免意外 API 调用），AI 倾向于提供’最完整’的默认体验

Q-Former 调度策略#

角色 思路
人类 观察到 Q-Former 阻塞所有后续实验后，主动要求停掉并重新排序：先跑快速实验，最后跑 Q-Former
AI 原始调度按顺序运行，遇到 Q-Former（~90 分钟/section）时才发现阻塞问题

差异分析: 人类主动识别资源调度问题并提出重排策略；AI 是响应式的，需要人类发现问题后才调整

VLA 研究范围#

角色 思路
人类 选择同时深入 Flow Matching、Register Tokens+AdaLN、空间组学论文三个方向
AI 列出多个候选方向后等待用户选择；用户选择后并行启动三个子任务

差异分析: 研究方向选择由人类决策，AI 负责并行执行；但并行任务执行效率是 AI 的优势

AI 局限性#

  • 第一次会话中 AI 调用 ExitPlanMode 时未预见用户可能中断，导致计划需要在第二次会话重新实施
  • 对 website 不是独立 git repo 的情况处理较被动，只说明现状，未主动询问是否需要为源文件建立版本管理
  • VLA 研究中访问用户博客时因路径错误（大小写、404）需要多次重试，未能一次准确定位内容
  • Q-Former 进程意外停止时 AI 未能给出明确的停止原因（日志中无明显错误），只能检查进程状态后重启
  • 默认 CLI 行为设计（cmd_legacy 调 API）与用户期望（默认只 export）不符，需要人类纠正

今日收获#

  • Hugo 部署常用 public/ 子目录作为 git repo 推送到 GitHub Pages，源文件 repo 与部署 repo 分离是标准实践
  • 将计划写入 .md 文件后在新会话粘贴，是跨会话保留上下文的有效方式，比依赖工具调用链更健壮
  • GPU 密集型训练任务（Q-Former）应单独调度，不与其他 GPU 任务并行，避免 OOM
  • 多设备文件同步应使用分层目录结构（按类型/设备分子目录），而非 flat 目录；sync 操作应为显式 opt-in 而非默认行为
  • VLA 与空间组学正在独立趋同到相同架构创新（Diffusion Transformer、模态 tokens、对比对齐），Flow Matching 和 Register Tokens+AdaLN 是最值得迁移的技术

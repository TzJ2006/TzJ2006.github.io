日报 — 2026-03-07#

问题与解决方案#

关键问题#

1. CalendarPro 对话历史 50 条硬限制静默丢弃旧消息，daily summaries 生成后从未注入 prompt#

解决方案: 参考 OpenClaw 四层策略：session 感知存储、自动压缩（超阈值时 AI 总结旧消息）、idle/daily session 重置存档、消息软截断；session_summaries 集成进 build_prompt

关键洞察: context window 管理需要多层策略协同：token 预算控制 + 渐进式压缩 + session 边界识别 + 摘要持久化，单靠截断会丢失重要历史

2. 浏览模式 WKWebView 无法接收鼠标/键盘事件，即使设置 ignoresMouseEvents=false 也无效；.floating 层级导致窗口始终置顶#

解决方案: enterBrowseMode 中设 window.level = .normal（不置顶且可交互），exitBrowseMode 恢复 CGWindowLevelForKey(.desktopWindow)

关键洞察: macOS .desktopWindow 层级位于 Finder 桌面图标层下方，该层级窗口在系统层面不接收任何输入事件，ignoresMouseEvents=false 对此层级无效。.normal（level=0）是既能交互又不覆盖其他应用的最低层级

3. intent 分类器把"几点了"、“ok”、“好的"等词无条件归为闲聊，忽略对话上下文#

解决方案: 关键词拆分为两层：始终闲聊词无论上下文强制归 GENERAL；上下文相关词只在无活跃 session 时归 GENERAL；确认词走快捷通道跳过关键词检查直接进入双重验证

关键洞察: 同一个词在对话开始和进行中语义完全不同，“ok” 可以是确认也可以是打招呼，必须以 session 状态为前提做分类

4. Phoenix 扩散策略训练 GPU 利用率仅 5-6%：DataLoader num_workers=0 导致 GPU 等待 CPU 从 NFS HDF5 文件随机读取数据#

解决方案: 在 init 时一次性将所有 HDF5 数据预加载至内存（uint8 节省 4× 存储），关闭文件句柄，之后 getitem 全部从 numpy 数组切片；DataLoader 改为 num_workers=8，pin_memory=True

关键洞察: h5py 文件句柄不能跨 fork 子进程共享（PyTorch DataLoader 已知限制），唯一安全方案是在主进程完成加载后再 fork，即预加载为 numpy 再释放文件句柄

5. LLaVA-v1.5-7b 无法从 HuggingFace 下载（集群无公网 DNS，代理限速 ~50KB/s，xet CDN 超时）；HF 格式权重与原始 LLaVA 格式不匹配（键名前缀/词表大小差异）#

解决方案: 搜索发现集群 nyw 目录存有 HF safetensors 格式副本；编写转换脚本剥离 language_model. 前缀，映射 mm_projector 键名，截断词表（embed_tokens/lm_head 最后 64 行，32064→32000）

关键洞察: HF 上传版本对分词器进行了扩展（32000→32064），转换时必须与 config.json 的 vocab_size 对齐；集群大型模型获取应优先搜索本地副本而非尝试下载

6. LLaVA MPM 训练预估 13.5 天（ZeRO-3 + 5 epoch + 1M 样本），远超计划预期 12-24 小时；ZeRO-3 单卡内存仅用 36%#

解决方案: 切换 ZeRO-3→ZeRO-2，减少到 1 epoch，batch 16→32，训练时长压缩至约 2 天

关键洞察: LoRA 仅更新约 4% 的参数（300M/7B），ZeRO-3 在单 GPU 内存充足时因不必要的参数广播降低吞吐；训练前必须准确估算数据集规模和步数

7. BC-RNN RP（0.003）异常低于 Random RP（0.020），评测结果完全不合理#

解决方案: 代码审查发现 4 个 bug：demo_length→demo_len 字段名不一致导致 demo_len=0、trigger_step 读取源错误导致 trigger_step=0、transport 阶段依赖不存在的 target_pos 字段永为 False（RP 上限 0.5）、成功 episode RP 未设为 1.0。逐一修复后 RP 最高达 0.800。robosuite PickPlace 目标位置在每次 reset 后随机化，必须通过运行时 env.target_bin_placements 动态获取，不能依赖静态配置

关键洞察: 指标管线的低级 bug 能产生比算法失败更隐蔽的误导性结论，必须在相信评测结果前先验证指标计算管线本身的正确性

8. WKWebView 滚动条轨道透明（CSS 注入设置 background-color 无效）+ 沙盒应用无法加载任何网络资源#

解决方案: 移除 webView.setValue(false, forKey: “drawsBackground”) 恢复默认不透明背景后滚动条轨道可见；在 .entitlements 添加 com.apple.security.network.client = true

关键洞察: drawsBackground=false 使 WKWebView 整个渲染层透明，CSS 在透明层之上叠加半透明样式仍然透明，修复必须在渲染层级而非 CSS 层。App Sandbox 默认阻断所有出站网络，WKWebView 不例外

9. FLARE norm stats 脚本 import openpi 时加载了 conda 环境中旧版本，找不到 tangzijia 新增的 config name#

解决方案: 执行时设置 PYTHONPATH=/tangzijia/…/openpi/src:$PYTHONPATH，优先加载 tangzijia 的源码副本

关键洞察: 多用户共享 conda env 时各自的源码修改会产生 import 优先级冲突，PYTHONPATH 是最小侵入性的隔离方案，不影响其他用户且可按需激活

10. natural_capture 模式下 coffee/stack 等 5 个任务崩溃：KeyError: ‘agentview_image’#

解决方案: _capture_single_rollout 中增加策略类型判断：非 VLA 策略使用 env._get_observations() 获取含图像的 raw robosuite obs，与 _generate_from_single_rollout 路径对齐

关键洞察: pick_place 的 BC-RNN 使用 low-dim obs，其余 5 个任务使用 rgb obs（agentview_image + robot0_eye_in_hand_image），两条代码路径对 obs 格式的期望不同，共用脚本需做 per-policy 路由判断

11. 2_visualize_scene.py 渲染 natural 场景时因 INJECTOR_REGISTRY 查找失败直接退出#

解决方案: 增加 is_natural 守卫：natural 场景跳过注入阶段，Phase 2/3 改为渲染中性动作帧；load_demo_actions 扩展 source_type=‘natural_capture’ 路径

关键洞察: Natural 场景的错误已内嵌在仿真状态快照中，无需重放注入动作；渲染逻辑需区分 synthetic injection 和 natural observation 两条路径

AI 局限性#

重要局限#

  • macOS WKWebView 渲染管线理解不足：未能预判 drawsBackground=false 会使 CSS 样式失效，先提出 CSS 注入表层补丁而非分析根本原因；窗口层级系统有认知盲区，对 .desktopWindow 层级不接收输入事件这一硬性约束不了解，两次提交错误方案才在用户指导下得到正确值
  • 未预检关键前置条件：实现 WKWebView 时未检查 App Sandbox 网络权限；训练前未验证数据集实际样本数（预期 160K 实际 1M，时长估算偏差 6 倍）；启动训练后才通过 nvidia-smi 发现跑在了错误 GPU 上
  • 在异常结果出现后未主动与计划预期对比：接受 13.5 天的训练时长而未主动告警，需要用户主动叫停；在多次下载失败后才想到本地格式转换方案，反应路径偏长
  • 默认最小改动策略，缺乏跨项目架构视角：CalendarPro intent 修复中未主动识别 OpenClaw compaction 机制可复用；injection 模式对 image-obs 任务产出 0 个场景，未提前建议调低 detector 阈值或增加任务特定配置

一般局限#

  • 任务阶段理解不清晰：将"错误场景生成（Phase 1: 1c_generate_from_policy.py）“与"恢复策略评测（Phase 3/4: 3_collect_data.py）“混淆，需用户明确澄清后才正确理解行动方向；SSH 多行脚本传递不稳定，先尝试 heredoc 失败后才切换到脚本文件方案

今日收获#

核心收获#

  • OpenClaw 四层 context window 管理策略：Session Pruning（soft-trim 旧 toolResult）→ Pre-Compaction Memory Flush（接近阈值写 memory 文件）→ Auto-Compaction（AI 总结旧对话生成摘要条目）→ Overflow Recovery（溢出时压缩重试）。适用于任何需要长期对话历史的系统
  • macOS 窗口层级系统：.desktopWindow（-2147483621）位于桌面图标层下方且不接收任何输入事件；.normal（0）与普通应用同层可交互；.floating 浮于所有应用之上。浏览模式 .normal 是既能交互又不覆盖其他应用的正确选择
  • 关键词意图分类必须考虑会话状态：同一个词"ok"在新会话开始时是打招呼，在跟进对话中是确认。正确设计：始终闲聊层（无论上下文）+ 上下文相关层（仅在无活跃 session 时归 GENERAL）+ 确认词快捷通道
  • h5py 与 PyTorch DataLoader 兼容性：h5py 文件句柄不能跨 fork 子进程共享，使用 num_workers > 0 时必须先在主进程完成数据加载（转为 numpy），才能安全并行预取。这是机器人学训练代码中 GPU 低利用率的常见成因
  • HF llava-1.5-7b-hf 与原始 liuhaotian/llava-v1.5-7b 的格式差异：weight key 有 language_model. 前缀需剥离，mm_projector 键名需映射，词表从 32000 扩展到 32064 需截断 embed_tokens 和 lm_head。转换脚本可复用于同类 LLaVA 变体；集群大型模型获取应优先搜索已有本地副本
  • ZeRO-3 对单 GPU LoRA 微调是反优化：LoRA 仅训练约 4% 参数（300M/7B），ZeRO-3 的全参数广播在单卡内存充足时只增加无效开销。单 GPU LoRA 应始终用 ZeRO-2；DeepSpeed –include “localhost:N” 才能精确绑定 GPU，–num_gpus 与 CUDA_VISIBLE_DEVICES 是独立机制
  • 评测管线的低级 bug（字段名不一致、读取源错误）能产生比算法失败更隐蔽的误导性结论，必须在相信结果前先验证指标计算管线。robosuite PickPlace 目标位置在每次 reset 后随机化，必须通过运行时 env.target_bin_placements 获取，task_config 只定义采样范围
  • 跨平台脚本调度应在 Python 层做平台检测（sys.platform），通过统一函数封装实现 Windows/UNIX 透明兼容，而非在文档里让用户手动改命令
  • PYTHONPATH 是共享集群多版本源码隔离的关键手段：同一 conda env 被多人使用时，显式 prepend PYTHONPATH 是最小侵入性的隔离方案；模仿学习训练通常无内置在线验证，training loss 只是必要非充分指标，真实性能需完整闭环 rollout 评估
  • 同一 benchmark 的不同任务 BC-RNN checkpoint 可能使用完全不同的 obs modality（pick_place 用 low-dim，其他 5 个用 rgb），共用 rollout 脚本需做 per-policy obs 路由；robosuite MUJOCO_EGL_DEVICE_ID 检查是字符串包含，必须使用与 CUDA_VISIBLE_DEVICES 完全匹配的物理 GPU ID；Natural 场景的错误内嵌在状态快照中，渲染时需与 synthetic injection 路径分开处理；ErrorSceneDatabase 有自动去重逻辑，增加 rollout 数量可增量扩充数据库无需清空重跑
  • Phoenix 完整评估依赖 MPM+MCM+扩散策略三个模型协同，论文未公开 LLaVA checkpoint，需自行训练，单 A800 资源可行（DeepSpeed ZeRO-2 + LoRA）；集群（sysu HPC）GPU 节点无公网 DNS，登录节点代理限速 ~50KB/s，大型模型获取应优先搜索集群内已有副本

实践收获#

  • macOS WKWebView 渲染：drawsBackground=false 使整个渲染层透明，CSS 在透明底层之上无效；SwiftUI ScrollView 在 macOS 上默认透明背景，需显式添加 windowBackgroundColor；App Sandbox 沙盒应用的网络权限必须在 .entitlements 显式声明

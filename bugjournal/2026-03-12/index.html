问题与解决方案#

关键问题#

1. macOS 非沙盒应用无法创建 security-scoped bookmark，历史记录打开旧视频时被 TCC 拒绝（Operation not permitted, Code=257）#

解决方案: 启用 App Sandbox（entitlements 添加 app-sandbox + files.user-selected.read-write），同时修复 ensureFileAccess 使其能解析旧的 non-scoped bookmark，并在权限失败时通过 NSOpenPanel 让用户重新授权

关键洞察: App Sandbox 是 security-scoped bookmark 能工作的前提条件；non-scoped bookmark 本身只能解析出 URL，不能恢复 TCC 权限——权限持久化必须靠 security-scoped bookmark

2. 升级 torchvision 后 curobo 预编译 .so 与新 torch ABI 不兼容，JIT 编译因 CUDA 头文件路径非标准而失败#

解决方案: 删除旧 .so，安装 ninja，设置 CPATH=/HOME/…/envs/RefineVLA/targets/x86_64-linux/include 后从源码重编 curobo

关键洞察: conda 内通过 pip 安装的 nvidia CUDA 包将头文件放在 targets/x86_64-linux/include/ 而非标准的 $CUDA_HOME/include/，需要显式设置 CPATH

一般问题#

3. torch 2.7.1+cu126 与 torchvision 0.21.0 C++ ABI 不兼容，导致 torchvision::nms 算子注册失败#

解决方案: 升级 torchvision 至 0.22.1+cu126

关键洞察: PyTorch 生态中 torch/torchvision 需要精确的版本对应关系（2.7.x → 0.22.x），预编译 wheel 通过算子注册接口与 torch 紧耦合

4. gpumon.py 进程去重逻辑错误：AI 将跨 GPU 的同一进程全局合并为一行，但实际需求是每个进程在每个 GPU 上各显示一次#

解决方案: 用户中途纠正了需求，AI 尚未重新实现

关键洞察: 对于多卡分布式训练，进程在不同 GPU 上的显存占用和利用率可能不同，全局去重会丢失有价值的信息；正确做法是以 (pid, gpu_id) 为粒度去重

5. eval.sh 默认 checkpoint_id=5000，但实际 checkpoint 目录中只有 15000/25000/29999#

解决方案: 需修改 deploy_policy.yml 中的 checkpoint_id 为现有值（如 29999），未完成

关键洞察: deploy_policy.yml 的默认值与实际训练产出的 checkpoint 编号不一致，评估脚本需要对参数范围做验证或提供更友好的报错

角色 思路

AI 局限性#

重要局限#

  • RoboTwin 环境修复缺乏整体预见性：每修复一层依赖（torchvision→curobo→checkpoint 路径）才发现下一个问题，没有在开始时全面检查环境依赖链

一般局限#

  • gpumon.py 进程去重逻辑错误：AI 将「每个进程只显示一次」实现为全局去重，与实际需求「每个进程在每个 GPU 上显示一次」不符，需要用户纠正
  • 面对复杂 CUDA 环境问题时，AI 尝试了多种 workaround（CPATH、符号链接、CUDA_HOME 设置）才找到正确方案，过程中产生了一些冗余操作（创建了部分多余符号链接）
  • 工具调用格式错误：在第一个 Desktop Video 会话中，AI 连续两次调用 Agent 工具时忘记传 description 参数，导致 InputValidationError，需要人类等待重试

今日收获#

核心收获#

  • macOS TCC 与 App Sandbox 的交互机制：非沙盒应用无法创建 security-scoped bookmark（即使有 bookmarks.app-scope entitlement），访问受保护目录（Documents/Desktop）的权限无法在 app 重启后自动持久化，必须通过 App Sandbox + security-scoped bookmark 或每次通过 NSOpenPanel 重新授权
  • PyTorch 生态依赖链：torch → torchvision → curobo 通过 C++ ABI 紧耦合，升级 torch/torchvision 后，依赖 torch C++ API 的第三方库（如 curobo）必须重新编译，不能仅靠 Python 层面的版本升级解决
  • conda 环境中 CUDA 头文件的非标准位置：通过 pip 安装的 nvidia-cuda-runtime 等包将头文件放在 envs//targets/x86_64-linux/include/ 而非标准的 $CONDA_PREFIX/include/，编译 CUDA 扩展时需要显式设置 CPATH 或 CUDA_HOME

实践收获#

  • macOS App Sandbox 对现有功能的兼容性：CGWindowLevelForKey(.desktopWindow) 窗口级别设置、SMAppService 开机启动、~/Library/Logs 日志写入、WKWebView 网络请求在沙盒下均可正常工作，不需要额外适配
